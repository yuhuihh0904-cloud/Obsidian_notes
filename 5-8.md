## 1.2.1 Host 端启动代码

PPT 先回顾普通矩阵乘法 kernel 的启动方式：

~~~cpp
dim3 dimGrid(
    ceil((1.0 * Width) / BLOCK_WIDTH),
    ceil((1.0 * Width) / BLOCK_WIDTH),
~~~

含义：

|---|---|

| `dimGrid.y` | y 方向 block 数，对应输出矩阵行方向 |
| `dimBlock.x` | 每个 block 中 x 方向线程数 |
| `dimBlock.y` | 每个 block 中 y 方向线程数 |
| `BLOCK_WIDTH` | 每个 block 覆盖的 tile 宽度 |

## 1.2.2 普通 MatrixMulKernel

对应页码：p4

~~~cpp
__global__
void MatrixMulKernel(float* d_M, float* d_N, float* d_P, int Width)
{
    int Row = blockIdx.y * blockDim.y + threadIdx.y;
    int Col = blockIdx.x * blockDim.x + threadIdx.x;

    if ((Row < Width) && (Col < Width)) {
        float Pvalue = 0;

        for (int k = 0; k < Width; ++k) {
            Pvalue += d_M[Row * Width + k] * d_N[k * Width + Col];
        }

        d_P[Row * Width + Col] = Pvalue;
    }
}
~~~

每个线程负责计算一个输出元素：

```text
P[Row][Col]
```

计算公式：

```text
P[Row][Col] = Σ M[Row][k] * N[k][Col]
```

# 1.3 问题：大量访问 Global Memory

对应页码：p4-p7

## 1.3.1 普通矩阵乘法的问题

普通矩阵乘法中，每个线程都要反复访问：

~~~cpp
d_M[Row * Width + k]
d_N[k * Width + Col]
~~~

这两个数组都在 global memory 中。

问题是：

```text
global memory 很慢，访问延迟高，带宽有限。
```

虽然 GPU 计算单元很多，但如果数据供应不上，计算单元会等待内存。

---

## 1.3.2 每个 FLOP 需要多少数据？

对应页码：p5

每次循环中，一个线程读取：

```text
M 的一个 float：4B
N 的一个 float：4B
```

总共：

```text
8B
```

同时做一次 multiply-add：

```text
Pvalue += M * N
```

这通常算作：

```text
2 FLOP
```

所以：

```text
8B / 2 FLOP = 4B/FLOP
```

也就是说，每完成 1 个浮点操作，需要从内存读约 4 字节。

---

## 1.3.3 性能上限计算

对应页码：p6

PPT 给了一个例子：

```text
GPU 计算能力：1000 GFLOP/s
Memory bandwidth：150 GB/s
```

如果每个 FLOP 需要 4B：

```text
150 GB/s ÷ 4B/FLOP = 37.5 GFLOP/s
```

所以虽然理论计算能力是 1000 GFLOP/s，但由于内存带宽限制，实际可能只能达到几十 GFLOP/s。

实际执行时可能更低，例如 PPT 提到约 25 GFLOP/s。

---

## 1.3.4 核心结论

```text
普通矩阵乘法的瓶颈不是计算，而是 global memory 访问。
```

考试可以这样写：

> 基础矩阵乘法 kernel 中，每个线程为了计算一个 P 元素，需要从 global memory 反复读取 M 的一行和 N 的一列。由于 global memory 延迟高、带宽有限，每个 FLOP 需要较多字节的数据传输，导致程序受到 memory bandwidth 限制，无法发挥 GPU 的峰值计算能力。

# 1.4 解决思路：Reuse Memory Accesses

对应页码：p7-p10

## 1.4.1 为什么要数据复用？

PPT 明确提出：

```text
To get closer to 1,000 GFLOPs, we need to drastically cut down accesses to global memory.
```

也就是说：

```text
想提高性能，就要显著减少 global memory 访问。
```

方法：

```text
Reuse memory accesses
```

也就是一次从 global memory 读入的数据，不要只用一次，而是让多个线程重复使用。

---

## 1.4.2 Shared Memory Tiling

对应页码：p8-p10

核心思想：

```text
矩阵 M 和 N 很大，放不进 shared memory；
但是可以把它们切成小 tile；
每次把一个 tile 读入 shared memory；
block 内线程反复使用这个 tile；
用完后再加载下一个 tile。
```

---

## 1.4.3 为什么 shared memory 有用？

| 内存 | 特点 |
|---|---|
| Global Memory | 容量大，但慢 |
| Shared Memory | 容量小，但快，block 内线程共享 |

所以 tiling 的本质是：

```text
用 shared memory 作为 global memory 的高速缓存/暂存区。
```

---

# 1.5 Tiling 的通用编程策略

对应页码：p9-p14

## 1.5.1 通用步骤

PPT 给出的通用策略：

```text
1. 把大矩阵切成 tiles
2. 每个 block 处理一个输出 tile
3. block 内线程协作，把 M 和 N 的 tile 从 global memory 加载到 shared memory
4. 所有线程同步
5. 在线程内部用 shared memory 数据计算
6. 所有线程同步
7. 进入下一个 tile 阶段
8. 最终把结果写回 global memory
```

---

## 1.5.2 为什么只能 block 内共享？

对应页码：p10

因为 shared memory 的作用域是：

```text
thread block
```

所以：

```text
同一个 block 内的线程可以共享 shared memory；
不同 block 之间不能共享 shared memory。
```

因此每个 block 都要加载自己需要的 tiles。

---

## 1.5.3 将 global memory 访问转为 on-chip memory 访问

对应页码：p13

PPT 总结为：

```text
Identify a tile of global data
    ↓
Load the tile from global memory into on-chip memory
    ↓
Have threads access data from on-chip memory
    ↓
Move on to the next tile
```

这里的 on-chip memory 主要指 shared memory。

# 1.6 Shared Memory 数组声明

对应页码：p11

在 tiled matrix multiplication kernel 中，需要声明两个 shared memory tile：

~~~cpp
__shared__ float subTileM[TILE_WIDTH][TILE_WIDTH];

~~~

含义：

| 数组 | 含义 |
|---|---|
| `subTileM` | 当前阶段加载进来的 M 的 tile |
| `subTileN` | 当前阶段加载进来的 N 的 tile |

特点：

```text
这两个数组对同一个 block 内所有线程可见。
```

# 1.7 Tiled Multiply 基本思想

对应页码：p15-p16

## 1.7.1 把计算分成 phases

普通矩阵乘法中，一个线程计算：

```text
P[Row][Col] = M[Row][0] * N[0][Col]
            + M[Row][1] * N[1][Col]
            + ...
```

Tiling 后，把 k 方向分成多个阶段：

```text
phase 0：使用 M 的第 0 个 tile 和 N 的第 0 个 tile
phase 1：使用 M 的第 1 个 tile 和 N 的第 1 个 tile
phase 2：使用 M 的第 2 个 tile 和 N 的第 2 个 tile
...
```

每个 phase 中：

```text
先加载 tile
再用 tile 计算部分和
```

---

## 1.7.2 blockIdx 和 threadIdx 的含义

PPT 中用：

```text
bx = blockIdx.x
by = blockIdx.y
tx = threadIdx.x
ty = threadIdx.y
```

对应：

| 变量 | 含义 |
|---|---|
| `bx` | 当前 block 在 x 方向编号 |
| `by` | 当前 block 在 y 方向编号 |
| `tx` | 当前线程在 block 内 x 方向编号 |
| `ty` | 当前线程在 block 内 y 方向编号 |

输出元素坐标：

~~~cpp
int Row = by * TILE_WIDTH + ty;
int Col = bx * TILE_WIDTH + tx;
~~~

---

# 1.8 Loading a Tile

对应页码：p16-p20

## 1.8.1 所有线程共同加载

PPT 强调：

```text
All threads in a block participate.
```

在基础 tiling 代码中：

```text
每个线程加载一个 M 元素
每个线程加载一个 N 元素
```

也就是说，一个 `TILE_WIDTH × TILE_WIDTH` 的 block 有 `TILE_WIDTH²` 个线程，刚好可以协作加载一个 `TILE_WIDTH × TILE_WIDTH` 的 M tile 和一个 N tile。

---

## 1.8.2 加载 M tile

~~~cpp
subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
~~~

解释：

| 部分 | 含义 |
|---|---|
| `Row` | 当前线程负责的输出行，也是 M 的行 |
| `m * TILE_WIDTH + tx` | 当前 phase 中 M tile 的列 |
| `subTileM[ty][tx]` | 存到 shared memory 中的位置 |

---

## 1.8.3 加载 N tile

~~~cpp
subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];
~~~

解释：

| 部分 | 含义 |
|---|---|
| `m * TILE_WIDTH + ty` | 当前 phase 中 N tile 的行 |
| `Col` | 当前线程负责的输出列，也是 N 的列 |
| `subTileN[ty][tx]` | 存到 shared memory 中的位置 |

---

## 1.8.4 Coalesced Access

对应页码：p16

PPT 提到加载 tile 时要尽量让同一个 warp 内线程访问连续地址：

```text
Assign the loaded element to each thread such that accesses within each warp are coalesced.
```

这句话在 Lecture 7 会展开。

简单理解：

```text
线程 0 访问地址 A[0]
线程 1 访问地址 A[1]
线程 2 访问地址 A[2]
...
```

这种连续访问更适合 DRAM burst，也能合并内存事务，提高带宽利用率。

# 1.9 使用 Shared Tile 计算

对应页码：p18-p23

加载完 tile 后，每个线程计算当前 phase 的部分和。

核心代码：

~~~cpp
for (int k = 0; k < TILE_WIDTH; ++k) {
    Pvalue += subTileM[ty][k] * subTileN[k][tx];
}
~~~

解释：

```text
当前线程负责 P[Row][Col]
它需要 M tile 中第 ty 行
以及 N tile 中第 tx 列
做点积的一部分。
```

# 1.10 `__syncthreads()` 的作用

对应页码：p24-p30 左右，代码页 p34

Tiled kernel 中有两个同步点：

~~~cpp
__syncthreads();

for (int k = 0; k < TILE_WIDTH; ++k) {
    Pvalue += subTileM[ty][k] * subTileN[k][tx];
}

__syncthreads();
~~~

---

## 1.10.1 第一个 `__syncthreads()`

放在加载 tile 之后。

作用：

```text
确保 block 内所有线程都已经把 M 和 N 的 tile 加载到 shared memory。
```

如果没有这个同步，某些线程可能开始计算时，其他线程还没有完成加载，那么读取到的 shared memory 可能是旧值或未定义值。

---

## 1.10.2 第二个 `__syncthreads()`

放在计算完当前 phase 之后。

作用：

```text
确保所有线程都已经使用完当前 shared memory tile。
```

因为下一轮 phase 会覆盖 `subTileM` 和 `subTileN`。

如果没有第二个同步，某些线程可能还在用当前 tile，另一些线程已经开始加载下一个 tile 并覆盖 shared memory，导致错误。

---

## 1.10.3 考试重点

如果题目问为什么 tiled matrix multiplication 需要两次同步，可以写：

> 第一次同步保证 shared memory tile 已经被 block 内所有线程完整加载后再开始计算；第二次同步保证所有线程都完成了当前 tile 的计算后，才允许进入下一轮加载并覆盖 shared memory。否则可能出现读到未初始化数据或 tile 被提前覆盖的问题。

# 1.11 Tiled Matrix Multiplication Kernel

对应页码：p34

这是本章最重要代码。

~~~cpp
__global__
void MatrixMulKernel(float* M, float* N, float* P, int Width)
{
    __shared__ float subTileM[TILE_WIDTH][TILE_WIDTH];
    __shared__ float subTileN[TILE_WIDTH][TILE_WIDTH];

    int bx = blockIdx.x;
    int by = blockIdx.y;
    int tx = threadIdx.x;
    int ty = threadIdx.y;

    int Row = by * TILE_WIDTH + ty;
    int Col = bx * TILE_WIDTH + tx;

    float Pvalue = 0;

    for (int m = 0; m < Width / TILE_WIDTH; ++m) {
        subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
        subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];

        __syncthreads();

        for (int k = 0; k < TILE_WIDTH; ++k) {
            Pvalue += subTileM[ty][k] * subTileN[k][tx];
        }

        __syncthreads();
    }

    P[Row * Width + Col] = Pvalue;
}
~~~

---

## 1.11.1 代码分段解释

### 第 1 部分：声明 shared memory

~~~cpp
__shared__ float subTileM[TILE_WIDTH][TILE_WIDTH];
__shared__ float subTileN[TILE_WIDTH][TILE_WIDTH];
~~~

用于缓存当前 phase 的 M tile 和 N tile。

---

### 第 2 部分：计算全局输出坐标

~~~cpp
int Row = by * TILE_WIDTH + ty;
int Col = bx * TILE_WIDTH + tx;
~~~

当前线程计算：

```text
P[Row][Col]
```

---

### 第 3 部分：遍历所有 phase

~~~cpp
for (int m = 0; m < Width / TILE_WIDTH; ++m)
~~~

如果矩阵大小为 `Width × Width`，tile 宽度为 `TILE_WIDTH`，那么沿 k 方向需要：

```text
Width / TILE_WIDTH
```

个 phase。

注意：Lecture 5 的这个版本默认 `Width` 能被 `TILE_WIDTH` 整除。

---

### 第 4 部分：协作加载 tile

~~~cpp
subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];
~~~

每个线程加载一个 M 元素和一个 N 元素。

---

### 第 5 部分：使用 shared memory 计算

~~~cpp
for (int k = 0; k < TILE_WIDTH; ++k)
    Pvalue += subTileM[ty][k] * subTileN[k][tx];
~~~

每个线程使用当前 tile 贡献一部分点积。

---

### 第 6 部分：写回结果

~~~cpp
P[Row * Width + Col] = Pvalue;
~~~

最终只写一次 global memory。

# 1.12 Tiled Kernel 与 Basic Kernel 对比

对应页码：p34-p35

## 1.12.1 Basic Kernel

特点：

```text
每次 k 循环都直接访问 global memory。
```

代码核心：

~~~cpp
Pvalue += M[Row * Width + k] * N[k * Width + Col];
~~~

---

## 1.12.2 Tiled Kernel

特点：

```text
先把 global memory 中的 tile 加载到 shared memory；
再从 shared memory 中反复读取。
```

代码核心：

~~~cpp
subTileM[ty][tx] = M[...];
subTileN[ty][tx] = N[...];

__syncthreads();

Pvalue += subTileM[ty][k] * subTileN[k][tx];
~~~

---

## 1.12.3 性能差异

Basic kernel：

```text
global memory 访问次数多，数据复用差。
```

Tiled kernel：

```text
global memory 访问减少，shared memory 访问增加，但 shared memory 快得多。
```

# 1.13 Tiling 的收益分析

## 1.13.1 数据复用

在一个 `TILE_WIDTH × TILE_WIDTH` 的输出 tile 中：

- M tile 中的每个元素会被同一行多个线程使用。
- N tile 中的每个元素会被同一列多个线程使用。

因此每个加载到 shared memory 的数据不再只用一次，而是被多个线程复用。

---

## 1.13.2 对 global memory traffic 的影响

粗略理解：

```text
TILE_WIDTH 越大，每个 global memory 元素被复用的次数越多。
```

因此每个 FLOP 需要的 global memory 字节数下降。

但是 TILE_WIDTH 不能无限大，因为受限于：

- 每个 block 最大线程数。
- shared memory 大小。
- register 使用量。
- occupancy。
- block 调度限制。

# 2.5 边界问题 1：加载 tile 时越界

对应页码：p5-p11

## 2.5.1 为什么会越界？

假设：

```text
Width = 3
TILE_WIDTH = 2
```

那么需要：

```text
ceil(3 / 2) = 2
```

个 tile。

第二个 tile 中有些位置对应的全局坐标超出了矩阵范围。

例如 M 或 N 的某些元素并不存在。

如果仍然执行：

~~~cpp
subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];
~~~

就可能访问越界。

---

## 2.5.2 undefined tile values

PPT 中多次强调：

```text
Tile values of M are not defined.
Tile values of N are not defined.
```

也就是说，边界外的 tile 位置不对应真实矩阵数据。

如果不处理，这些 shared memory 值可能是：

- 随机旧值。
- 未定义值。
- 越界读取结果。
- 导致最终 P 错误。

---

# 2.6 边界问题 2：输出 P 越界

对应页码：p8-p11

对于最后一块输出 tile，有些线程对应的输出元素不在矩阵范围内。

例如：

```text
Width = 3
TILE_WIDTH = 2
```

block(1,1) 覆盖：

```text
P[2][2], P[2][3], P[3][2], P[3][3]
```

但真实矩阵只有：

```text
P[0..2][0..2]
```

所以只有：

```text
P[2][2]
```

是有效的。

其他线程不能写入 P。

---

# 2.7 正确处理方式：越界加载为 0，越界输出不写

这是 Lecture 6 最重要的结论。

---

## 2.7.1 处理 M tile 边界

原始加载：

~~~cpp
subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
~~~

应改成：

~~~cpp
if (Row < Width && m * TILE_WIDTH + tx < Width)
    subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
else
    subTileM[ty][tx] = 0.0f;
~~~

含义：

```text
如果对应的 M 元素在矩阵范围内，就加载真实值；
否则把 shared memory 中该位置设为 0。
```

为什么设为 0？

因为矩阵乘法中：

```text
0 * x = 0
```

不会影响最终结果。

---

## 2.7.2 处理 N tile 边界

原始加载：

~~~cpp
subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];
~~~

应改成：

~~~cpp
if (m * TILE_WIDTH + ty < Width && Col < Width)
    subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];
else
    subTileN[ty][tx] = 0.0f;
~~~

---

## 2.7.3 处理 P 输出边界

原始写回：

~~~cpp
P[Row * Width + Col] = Pvalue;
~~~

应改成：

~~~cpp
if (Row < Width && Col < Width)
    P[Row * Width + Col] = Pvalue;
~~~

如果当前线程对应的输出坐标越界，就不写。

---

# 2.8 phase 循环也要改

原始代码：

~~~cpp
for (int m = 0; m < Width / TILE_WIDTH; ++m)
~~~

如果 Width 不能整除 TILE_WIDTH，这样会漏掉最后一个不完整 tile。

应该使用向上取整：

~~~cpp
for (int m = 0; m < ceil(Width / (float)TILE_WIDTH); ++m)
~~~

或者整数写法：

~~~cpp
for (int m = 0; m < (Width + TILE_WIDTH - 1) / TILE_WIDTH; ++m)
~~~

---

# 2.9 边界版 Tiled Matrix Multiplication Kernel

考试中可以写这个版本。

~~~cpp
__global__
void MatrixMulKernel(float* M, float* N, float* P, int Width)
{
    __shared__ float subTileM[TILE_WIDTH][TILE_WIDTH];
    __shared__ float subTileN[TILE_WIDTH][TILE_WIDTH];

    int bx = blockIdx.x;
    int by = blockIdx.y;
    int tx = threadIdx.x;
    int ty = threadIdx.y;

    int Row = by * TILE_WIDTH + ty;
    int Col = bx * TILE_WIDTH + tx;

    float Pvalue = 0.0f;

    int numTiles = (Width + TILE_WIDTH - 1) / TILE_WIDTH;

    for (int m = 0; m < numTiles; ++m) {
        int mCol = m * TILE_WIDTH + tx;
        int nRow = m * TILE_WIDTH + ty;

        if (Row < Width && mCol < Width)
            subTileM[ty][tx] = M[Row * Width + mCol];
        else
            subTileM[ty][tx] = 0.0f;

        if (nRow < Width && Col < Width)
            subTileN[ty][tx] = N[nRow * Width + Col];
        else
            subTileN[ty][tx] = 0.0f;

        __syncthreads();

        for (int k = 0; k < TILE_WIDTH; ++k) {
            Pvalue += subTileM[ty][k] * subTileN[k][tx];
        }

        __syncthreads();
    }

    if (Row < Width && Col < Width)
        P[Row * Width + Col] = Pvalue;
}
~~~

# 2.10 为什么不能只在最终写 P 时判断？

一个常见错误：

```text
只判断 P 是否越界，但加载 M/N 时不判断。
```

这是不够的。

原因：

```text
即使最终 P 是有效位置，计算它时也可能需要最后一个不完整 tile。
最后一个 tile 中部分 M/N 元素越界。
如果加载时不处理，就会读到错误数据。
```

所以必须：

1. M tile 加载时判断。
2. N tile 加载时判断。
3. P 写回时判断。

---

# 2.11 为什么不能跳过越界线程？

有些线程虽然不负责有效 P 输出，但仍然可能需要参与 `__syncthreads()`。

注意：

```text
同一个 block 内所有线程必须执行到同一个 __syncthreads()。
```

如果某些线程提前 return，而其他线程进入 `__syncthreads()`，可能导致死锁。

所以更安全的写法是：

```text
所有线程都参与加载和同步；
越界位置加载 0；
最后只有有效线程写 P。
```

# 3.2 Global Memory Bandwidth：理想和现实

对应页码：p3

PPT 用图片表示：

```text
Ideal：像大坝一样水流很大
Reality：像吸管一样很窄
```

含义：

```text
理论带宽很高，但实际程序不一定能充分利用。
```

原因可能包括：

- 访问不连续。
- 不 coalesced。
- DRAM bank 利用不足。
- burst 传输浪费。
- 线程访问模式不友好。

# 3.3 RAM 和 DRAM

对应页码：p4

## 3.3.1 RAM

RAM = Random Access Memory。

含义：

```text
访问任意地址理论上需要相同时间。
```

这里的 random access 是和磁盘顺序访问相对的概念。

---

## 3.3.2 DRAM

DRAM = Dynamic RAM。

特点：

1. bit 存储在 capacitor 上。
2. 通过 transistor 连接到 bit line 进行读写。
3. capacitor 上的电荷会逐渐泄漏。
4. 数据过一段时间会消失。
5. 所以必须周期性刷新 refresh。
6. 因此叫 dynamic。

PPT 提到电荷大约几十毫秒会泄漏，需要重写。

---

# 3.4 DRAM Cell 和 Bit Line

对应页码：p4-p6

## 3.4.1 一个 bit 的结构

DRAM 一个 bit 大致由：

```text
1 个 capacitor + 1 个 transistor
```

组成。

优势：

```text
每 bit 只需要 1 个 transistor，所以密度高、容量大。
```

劣势：

```text
读写慢，需要刷新。
```

---

## 3.4.2 Many cells per bit line

对应页码：p5

PPT 提到：

```text
大约 1000 个 cells 连接到每条 bit line。
```

通过 SELECT line 选择其中一个 cell 连接到 bit line。

---

## 3.4.3 DRAM is slow but dense

对应页码：p6

DRAM 慢的原因：

- bit 本身电容很小。
- bit line 的电容很大。
- 读写时信号弱，需要 sense amplifier 放大。
- 结构密集但速度不如 SRAM。

总结：

```text
DRAM 容量大、成本低、密度高，但速度慢。
```

---

# 3.5 DRAM Bank Organization

对应页码：p7

DRAM bank 由多个部分组成：

| 结构 | 作用 |
|---|---|
| Core Array | 存储大量 bit |
| Row Decoder | 根据 row address 选择一行 |
| Sense Amps | 放大 bit line 上的微弱信号 |
| Column Latches | 暂存列数据 |
| Mux | 根据 column address 选择输出 |
| Pin Interface | 和芯片外部传输数据 |

---

## 3.5.1 宽内部，窄外部

PPT 图中体现了：

```text
DRAM 内部一次可以读出较宽的一行数据；
但对外 pin interface 比较窄。
```

所以一次 row access 读出的很多数据，不一定都能同时送出芯片。

这为 burst mode 提供了基础。

---

# 3.6 DRAM Interface Clock

对应页码：p8

PPT 提到：

```text
DRAM cells are not clocked.
DRAM interfaces are clocked.
```

含义：

- 存储单元本身不按时钟工作。
- 对外接口按时钟传输数据。

不同 DDR 类型中，core speed 和 interface speed 的比例不同：

| 类型 | Core speed 与 interface speed |
|---|---|
| DDR | core = 1/2 interface |
| DDR2 / GDDR3 | core = 1/4 interface |
| DDR3 / GDDR4 | core = 1/8 interface |

简单理解：

```text
外部接口越来越快，但内部 DRAM core 没有同比例变快。
```

所以需要 burst、bank 并行等方式提高有效带宽。

---

# 3.7 DRAM Read Operation

对应页码：p9-p13

## 3.7.1 读取一部分数据时，会顺带读出附近数据

PPT 用 8×2-bit DRAM 举例。

当选择某条 SELECT line 时，不只是目标 bit 可用，同一行上的其他 bit 也会被 sense amps 读出来。

所以 PPT 说：

```text
The other bits on the line are "free".
```

意思是：

```text
既然已经付出了 row access 的代价，附近连续数据也能顺带获得。
```

---

## 3.7.2 Burst

对应页码：p11-p13

Burst 的思想：

```text
一次行访问后，不只传输一个小数据，而是连续传输多个相邻数据。
```

PPT 示例中通过改变 MUX input bit 获取更多 bits，形成 4-bit burst。

现代 DRAM 通常总是以 burst mode 访问。

---

## 3.7.3 Burst 的关键结论

对应页码：p13

PPT 说：

```text
Modern DRAM systems are designed to be always accessed in burst mode.
Burst bytes are transferred but discarded when accesses are not to sequential locations.
```

中文解释：

```text
现代 DRAM 默认会按 burst 方式传输一段连续数据。
如果程序访问的不是连续地址，那么 burst 读出来的很多字节可能用不上，被丢弃。
```

因此：

```text
连续访问地址非常重要。
```

---

# 3.8 Multiple DRAM Banks

对应页码：p14-p16

## 3.8.1 为什么需要多个 bank？

单个 DRAM bank 在访问时有 core array delay。

如果只有一个 bank，接口可能出现空闲时间。

多个 bank 可以交错访问：

```text
Bank 0 正在等待 core access
Bank 1 可以开始工作
Bank 2 可以准备下一次访问
...
```

这样可以减少接口空闲，提高带宽。

---

## 3.8.2 Multi-bank burst

对应页码：p15-p16

PPT 对比：

```text
Single-bank burst timing：interface 有 dead time
Multi-bank burst timing：reduced dead time
```

含义：

```text
多个 bank 可以并行/交错提供数据，减少等待时间。
```

---

# 3.9 Memory Access Coalescing

对应页码：Lecture 7 后半部分

这是本章和 CUDA 编程最相关的重点。

---

## 3.9.1 什么是 coalescing？

Memory coalescing 指的是：

```text
同一个 warp 内多个线程的 global memory 访问，如果访问的是连续或相邻地址，硬件可以把这些访问合并成更少的内存事务。
```

例如：

~~~cpp
A[threadIdx.x]
~~~

线程访问：

```text
thread 0 -> A[0]
thread 1 -> A[1]
thread 2 -> A[2]
...
thread 31 -> A[31]
```

这就是连续访问，容易 coalesce。

---

## 3.9.2 为什么 coalescing 重要？

因为 DRAM 以 burst 为单位传输连续数据。

如果 warp 内线程访问连续地址：

```text
一次 burst 传回的数据大部分都能被使用。
```

如果访问分散地址：

```text
每个线程可能触发不同内存事务；
burst 读出的很多数据被浪费；
有效带宽下降。
```

---

## 3.9.3 合并访问 vs 非合并访问

### 合并访问

~~~cpp
int i = blockIdx.x * blockDim.x + threadIdx.x;
float x = A[i];
~~~

线程访问连续地址。

---

### 非合并访问

~~~cpp
int i = blockIdx.x * blockDim.x + threadIdx.x;
float x = A[i * stride];
~~~

如果 stride 很大，线程访问地址分散，不容易 coalesce。

---

# 3.10 Row-major 与 coalescing

对于二维矩阵，C/C++ 是 row-major：

~~~cpp
index = Row * Width + Col;
~~~

如果一个 warp 内线程的 `threadIdx.x` 连续变化，访问：

~~~cpp
A[Row * Width + Col]
~~~

那么 Col 连续，地址连续，有利于 coalescing。

如果访问：

~~~cpp
A[Col * Width + Row]
~~~

那么地址跨度为 Width，可能不连续，不利于 coalescing。

---

# 3.11 矩阵乘法中的 coalescing

## 3.11.1 加载 M tile

~~~cpp
subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
~~~

对于同一行、不同 tx 的线程：

```text
tx 连续变化
M 地址连续
```

所以 M tile 加载通常有利于 coalescing。

---

## 3.11.2 加载 N tile

~~~cpp
subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];
~~~

对于同一 ty、不同 tx 的线程：

```text
Col 连续变化
N 地址连续
```

所以 N tile 加载也可以做到 coalescing。

# 3.12 Lecture 7 高频答题模板

## 3.12.1 解释 DRAM 为什么慢但容量大

> DRAM 使用电容存储 bit，并通过 transistor 连接到 bit line。由于每 bit 只需要一个 transistor，所以存储密度高、容量大、成本较低。但电容电荷会泄漏，需要周期性刷新；同时 bit line 电容较大，读写信号需要 sense amplifier 放大，因此访问速度较慢。

---

## 3.12.2 解释 burst mode

> DRAM 在一次 row access 后，实际上会得到一段连续数据。Burst mode 利用这一点，在一次访问中连续传输多个相邻数据，从而摊薄 core array access delay，提高有效带宽。如果程序访问连续地址，burst 中的数据大多有用；如果访问分散地址，burst 传输的数据可能被丢弃，带宽利用率下降。

---

## 3.12.3 解释 multiple banks

> Multiple banks 通过把 DRAM 分成多个可独立工作的 bank，使不同 bank 的访问可以交错进行。当一个 bank 等待 core array access 时，另一个 bank 可以传输或准备数据，从而减少接口空闲时间，提高整体带宽。

---

## 3.12.4 解释 memory coalescing

> Memory coalescing 是指同一个 warp 内多个线程访问 global memory 时，如果访问地址连续或相邻，硬件可以将多个访问合并成较少的内存事务。这样能够更好利用 DRAM burst，提高实际带宽。若线程访问地址分散，则会产生更多内存事务，导致带宽浪费和性能下降。

# 4.2 Convolution 是什么？

对应页码：p3-p4

## 4.2.1 定义

Convolution 是一种数组操作：

```text
输入数组 N
    ↓
通过 mask/filter/kernel 加权求和
    ↓
输出数组 P
```

每个输出元素都是输入附近元素的加权和。

---

## 4.2.2 mask / filter / kernel

PPT 中说：

```text
Definition of nearby and weights together form a convolution mask.
```

也就是说：

```text
mask 定义了：
1. 使用哪些邻近输入元素；
2. 每个邻近元素的权重是多少。
```

mask 也叫：

- filter
- kernel
- coefficients
- weights

注意这里的 kernel 不是 CUDA kernel，而是卷积核。

---

## 4.2.3 same mask

PPT 强调：

```text
Generally, the same mask is used for all input elements.
```

也就是说：

```text
通常所有输出位置都使用同一个 mask。
```

这也是后面把 mask 放入 constant memory 的原因。

# 4.4 1D Convolution

对应页码：p6-p9

## 4.4.1 1D 卷积常用于音频处理

PPT 以一维数组为例，适合理解音频信号处理。

---

## 4.4.2 Mask_Width 和 Mask_Radius

对应页码：p6

| 概念 | 含义 |
|---|---|
| `Mask_Width` | mask 中元素个数 |
| `Mask_Radius` | 中心点左右各取多少个元素 |

通常 `Mask_Width` 是奇数，方便对称。

例如：

```text
Mask_Width = 5
Mask_Radius = 2
```

表示计算 P[i] 时使用：

```text
N[i-2], N[i-1], N[i], N[i+1], N[i+2]
```

---

## 4.4.3 1D 卷积公式

如果 mask 宽度是 `Mask_Width`，则：

```text
P[i] = Σ N[i - Mask_Radius + j] * M[j]
```

其中：

```text
j = 0 到 Mask_Width - 1
Mask_Radius = Mask_Width / 2
```

---

## 4.4.4 边界问题

对应页码：p8

当 i 靠近边界时，比如：

```text
i = 0
```

需要访问：

```text
N[-2], N[-1], N[0], N[1], N[2]
```

其中 `N[-2]` 和 `N[-1]` 不存在。

这些不存在的位置叫：

- ghost elements
- ghost cells
- halo cells
- apron cells

常见处理策略：

1. 填 0。
2. 复制边界值。
3. 镜像。
4. 忽略越界位置。
5. 只计算 valid 区域。

PPT 中的 basic kernel 使用的是：

```text
越界位置当作 0。
```

# 4.5 1D Convolution Basic Kernel

对应页码：p9

~~~cpp
__global__
void convolution_1D_basic_kernel(
    float *N,
    float *M,
    float *P,
    int Mask_Width,
    int Width)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;

    float Pvalue = 0;

    int N_start_point = i - (Mask_Width / 2);

    for (int j = 0; j < Mask_Width; j++) {
        if (N_start_point + j >= 0 &&
            N_start_point + j < Width) {
            Pvalue += N[N_start_point + j] * M[j];
        }
    }

    P[i] = Pvalue;
}
~~~

---

## 4.5.1 代码解释

### 当前线程负责的输出位置

~~~cpp
int i = blockIdx.x * blockDim.x + threadIdx.x;
~~~

每个线程负责计算一个输出元素：

```text
P[i]
```

---

### mask 对应的输入起点

~~~cpp
int N_start_point = i - (Mask_Width / 2);
~~~

如果 `Mask_Width = 5`，则：

```text
Mask_Width / 2 = 2
N_start_point = i - 2
```

---

### 边界判断

~~~cpp
if (N_start_point + j >= 0 &&
    N_start_point + j < Width)
~~~

如果越界，就不加，相当于越界值为 0。

# 4.6 2D Convolution

对应页码：p10-p13

## 4.6.1 2D 卷积场景

二维卷积常用于：

- 图像模糊。
- 图像锐化。
- 边缘检测。
- CNN 中的 feature map 计算。

---

## 4.6.2 2D 卷积思想

对于每个输出像素：

```text
P[Row][Col]
```

取输入图像 N 中以 `(Row, Col)` 为中心的一块邻域，与 mask M 对应位置相乘并求和。

---

## 4.6.3 2D 卷积公式

假设 mask 宽度为 `Mask_Width`，半径为 `Mask_Radius`：

```text
P[Row][Col]
= Σ Σ N[Row + i][Col + j] * M[i + r][j + r]
```

其中：

```text
i, j 从 -Mask_Radius 到 +Mask_Radius
r = Mask_Radius
```

---

## 4.6.4 2D 边界处理

对应页码：p11-p13

二维卷积在边界处也会访问图像外部。

例如左上角像素，需要访问左边和上边不存在的元素。

PPT 中使用：

```text
0 ghost cells
```

也就是把边界外的 ghost cells 视为 0。

# 4.7 2D Convolution Kernel 思路

PPT 主要讲原理，考试可以写如下代码思路。

~~~cpp
__global__
void convolution_2D_basic_kernel(
    float *N,
    float *M,
    float *P,
    int Mask_Width,
    int Width,
    int Height)
{
    int Col = blockIdx.x * blockDim.x + threadIdx.x;
    int Row = blockIdx.y * blockDim.y + threadIdx.y;

    if (Row < Height && Col < Width) {
        float Pvalue = 0.0f;
        int Mask_Radius = Mask_Width / 2;

        for (int i = 0; i < Mask_Width; i++) {
            for (int j = 0; j < Mask_Width; j++) {
                int curRow = Row + i - Mask_Radius;
                int curCol = Col + j - Mask_Radius;

                if (curRow >= 0 && curRow < Height &&
                    curCol >= 0 && curCol < Width) {
                    Pvalue += N[curRow * Width + curCol] *
                              M[i * Mask_Width + j];
                }
            }
        }

        P[Row * Width + Col] = Pvalue;
    }
}
~~~

# 4.8 Access Pattern for M

对应页码：p14

这是 Lecture 8 的关键优化动机。

PPT 说：

```text
Elements of M are called mask coefficients.
Calculation of all output P elements needs M.
M is not changed during grid execution.
M elements are accessed in the same order when calculating all P elements.
M is a good candidate for Constant Memory.
```

中文解释：

卷积 mask M 有几个特点：

1. 所有线程都要读 M。
2. M 在 kernel 执行期间不变。
3. M 是只读的。
4. 所有线程访问 M 的顺序很相似。
5. M 通常比较小。

所以 M 非常适合放在 constant memory 中。

## 4.10.3 Locality

对应页码：p18

程序的内存访问通常具有局部性：

### Spatial Locality 空间局部性

```text
访问 X 后，很可能很快访问 X+1、X+2 ...
```

例如遍历数组。

---

### Temporal Locality 时间局部性

```text
访问 X 后，很可能很快再次访问 X。
```

例如循环中重复使用同一变量或同一 mask。

```text

```

即最近最少使用的缓存行。

# 4.11 GPU Constant Cache

对应页码：p19

## 4.11.1 GPU 有 L1 cache 和 constant cache

PPT 说：

- L1 cache 支持 global memory access。
- L1 cache 支持写，所以需要处理 dirty line / writeback。
- constant / texture memory cache 是只读的特殊情况。
- 因为只读，所以可以更高效。

---

## 4.11.2 Constant Memory 的特点

Constant memory：

1. 存在于 device 端。
2. 所有线程可读。
3. 只读。
4. 有专门的 constant cache。
5. 当 warp 内线程读取相同 constant 地址时，非常高效。
6. 适合存放小而只读、所有线程共享的数据。

卷积 mask M 正好符合这些特征。

---

# 4.12 Constant Memory 版本的卷积

## 4.12.1 声明 constant memory

可以把 mask 声明为：

~~~cpp
__constant__ float M[MAX_MASK_WIDTH];
~~~

或者二维版本：

~~~cpp
__constant__ float M[MAX_MASK_WIDTH * MAX_MASK_WIDTH];
~~~

---

## 4.12.2 Host 端拷贝到 constant memory

使用：

~~~cpp
cudaMemcpyToSymbol(M, h_M, Mask_Width * sizeof(float));
~~~

如果是二维 mask：

~~~cpp

~~~

---

## 4.12.3 Kernel 中使用 constant memory

1D 卷积中：

~~~cpp
Pvalue += N[N_start_point + j] * M[j];
~~~

此时 `M[j]` 来自 constant memory。

---

## 4.12.4 为什么 constant memory 对卷积有用？

因为对于不同输出元素：

```text
线程 0 需要 M[0], M[1], M[2], ...
线程 1 也需要 M[0], M[1], M[2], ...
线程 2 也需要 M[0], M[1], M[2], ...
```

所有线程都重复访问同一个 mask。

所以 constant cache 命中率高，可以减少 global memory traffic。

---

# 4.13 Constant Memory 适用条件

适合 constant memory 的数据：

| 条件 | 说明 |
|---|---|
| 只读 | kernel 执行过程中不修改 |
| 小 | 能放入 constant memory/cache |
| 所有线程都访问 | 广播效果好 |
| 访问模式相似 | 同一 warp 读同一地址时收益大 |
| 重复使用 | temporal locality 好 |

卷积 mask 满足以上条件。

---

# 4.14 Constant Memory 不适合什么？

不适合：

1. 每个线程访问完全不同地址的大数组。
2. 需要写入的数据。
3. 体积很大的数据。
4. 没有复用的数据。
5. 访问模式高度随机的数据。

如果 warp 内每个线程访问不同 constant 地址，constant memory 的优势会下降。

# 4.15 Lecture 8 高频答题模板

## 4.15.1 解释卷积

> 卷积是一种数组计算模式，每个输出元素由输入数组中邻近元素的加权和得到。邻近范围和权重由卷积 mask/filter/kernel 决定。通常同一个 mask 会应用到所有输出位置，因此卷积广泛用于信号处理、图像处理、科学计算和深度学习。

---

## 4.15.2 解释 ghost cells

> 当卷积计算靠近数组或图像边界的输出元素时，mask 会覆盖到输入范围之外的位置，这些不存在的元素称为 ghost cells、halo cells 或 apron cells。常见处理方式包括补 0、复制边界值或只计算 valid 区域。PPT 中 basic kernel 使用的是补 0 策略，即越界元素不参与累加，相当于值为 0。

---

## 4.15.3 为什么 convolution mask 适合 constant memory？

> 卷积 mask 在 kernel 执行期间不会改变，是只读数据；所有输出元素的计算都需要访问同一个 mask；而且不同线程通常以相同顺序访问 mask 元素。因此 mask 具有高复用性和良好的时间局部性，适合放入 constant memory，并通过 constant cache 提高访问效率，减少 global memory 带宽压力。

---

## 4.15.4 解释 spatial locality 和 temporal locality

> Spatial locality 指访问某个地址后，很可能很快访问相邻地址，例如顺序遍历数组。Temporal locality 指访问某个地址后，很可能在短时间内再次访问同一地址，例如卷积中反复访问同一个 mask。Cache 可以利用这两种局部性减少慢速 global memory 访问。

## 6.1 Lecture 5：Tiled Matrix Multiplication

~~~cpp
__global__
void MatrixMulKernel(float* M, float* N, float* P, int Width)
{
    __shared__ float subTileM[TILE_WIDTH][TILE_WIDTH];
    __shared__ float subTileN[TILE_WIDTH][TILE_WIDTH];

    int bx = blockIdx.x;
    int by = blockIdx.y;
    int tx = threadIdx.x;
    int ty = threadIdx.y;

    int Row = by * TILE_WIDTH + ty;
    int Col = bx * TILE_WIDTH + tx;

    float Pvalue = 0;

    for (int m = 0; m < Width / TILE_WIDTH; ++m) {
        subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];
        subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];

        __syncthreads();

        for (int k = 0; k < TILE_WIDTH; ++k) {
            Pvalue += subTileM[ty][k] * subTileN[k][tx];
        }

        __syncthreads();
    }

    P[Row * Width + Col] = Pvalue;
}
~~~

## 6.2 Lecture 6：带边界判断的 Tiled Matrix Multiplication

~~~cpp
__global__
void MatrixMulKernel(float* M, float* N, float* P, int Width)
{
    __shared__ float subTileM[TILE_WIDTH][TILE_WIDTH];
    __shared__ float subTileN[TILE_WIDTH][TILE_WIDTH];

    int tx = threadIdx.x;
    int ty = threadIdx.y;

    int Row = blockIdx.y * TILE_WIDTH + ty;
    int Col = blockIdx.x * TILE_WIDTH + tx;

    float Pvalue = 0.0f;

    int numTiles = (Width + TILE_WIDTH - 1) / TILE_WIDTH;

    for (int m = 0; m < numTiles; ++m) {
        int mCol = m * TILE_WIDTH + tx;
        int nRow = m * TILE_WIDTH + ty;

        if (Row < Width && mCol < Width)
            subTileM[ty][tx] = M[Row * Width + mCol];
        else
            subTileM[ty][tx] = 0.0f;

        if (nRow < Width && Col < Width)
            subTileN[ty][tx] = N[nRow * Width + Col];
        else
            subTileN[ty][tx] = 0.0f;

        __syncthreads();

        for (int k = 0; k < TILE_WIDTH; ++k) {
            Pvalue += subTileM[ty][k] * subTileN[k][tx];
        }

        __syncthreads();
    }

    if (Row < Width && Col < Width)
        P[Row * Width + Col] = Pvalue;
}
~~~

# 7. 最可能考试题型

## 7.1 题型一：解释为什么 tiled matrix multiplication 更快

答题模板：

> 普通矩阵乘法中，每个线程直接从 global memory 反复读取 M 和 N 的元素，导致每个 FLOP 需要较多内存带宽，性能受 global memory bandwidth 限制。Tiled matrix multiplication 将 M 和 N 分成 tile，每个 block 协作把 tile 加载到 shared memory 中，然后 block 内多个线程重复使用 shared memory 中的数据。由于 shared memory 比 global memory 快得多，且减少了 global memory 访问次数，所以 tiled 版本性能更好。

---

## 7.2 题型二：解释 `__syncthreads()` 为什么需要两次

答题模板：

> 第一次 `__syncthreads()` 位于 tile 加载之后，用于保证所有线程都完成了 M 和 N tile 的加载，避免某些线程在数据未加载完成时就开始计算。第二次 `__syncthreads()` 位于当前 tile 计算之后，用于保证所有线程都完成了对当前 shared memory tile 的使用，再允许下一轮加载覆盖 shared memory。否则会出现读到未初始化数据或数据被提前覆盖的问题。

---

## 7.3 题型三：处理 tile 边界

答题模板：

> 当矩阵大小不能被 TILE_WIDTH 整除时，最后一个 tile 可能不完整。解决方法是在加载 M 和 N tile 时进行边界判断，如果对应全局坐标在矩阵范围内就加载真实值，否则向 shared memory 写入 0；在最终写回 P 时也要判断 Row 和 Col 是否有效，只有有效输出位置才写回。phase 循环次数应使用向上取整，即 `(Width + TILE_WIDTH - 1) / TILE_WIDTH`。

---

## 7.4 题型四：解释 DRAM burst 和 coalescing 的关系

答题模板：

> DRAM 通常以 burst mode 传输连续地址上的一段数据。如果 warp 内线程访问连续 global memory 地址，硬件可以进行 memory coalescing，把多个线程的访问合并成较少的内存事务，并充分利用 burst 传回的数据。如果线程访问地址分散，则会产生更多内存事务，burst 中很多数据被丢弃，导致实际带宽下降。因此 CUDA 程序中应尽量让同一 warp 内线程访问连续地址。

---

## 7.5 题型五：为什么 convolution mask 适合 constant memory

答题模板：

> 卷积 mask 是只读数据，在 kernel 执行期间不会改变；所有线程计算输出元素时都需要访问同一个 mask；并且不同线程通常以相同顺序访问 mask 元素，因此具有很强的时间局部性和广播特性。将 mask 放入 constant memory 后，可以利用 constant cache 加速访问，减少 global memory 带宽压力。

Lecture 5-8 的核心是 CUDA 程序性能优化：基础矩阵乘法受 global memory 带宽限制，因此 Lecture 5 使用 shared memory tiling 提高数据复用；Lecture 6 解决 tile 不整除时的边界问题；Lecture 7 从 DRAM 硬件角度解释 burst、bank 和 coalescing 为什么影响带宽；Lecture 8 介绍卷积这种重要并行模式，并说明只读且高复用的 convolution mask 适合放入 constant memory，通过 constant cache 降低内存访问开销。
