
## 一、面试标准回答

面试官问：**什么是 Function Calling？它的原理是什么？**

可以这样回答：

Function Calling 是大模型调用外部工具的一种标准机制。

它的核心不是让模型自己去执行代码、访问网络或者查数据库，而是让模型根据开发者提供的工具描述，判断当前问题是否需要调用工具。如果需要，模型不会直接输出自然语言答案，而是输出一段结构化的 `tool_calls` JSON，里面包含要调用的函数名和参数。真正执行函数的是我们的后端代码，代码执行完后再把工具结果作为 `role=tool` 的消息塞回对话历史，让模型基于工具结果生成最终回答。

所以 Function Calling 的核心分工是：

> **模型负责决策，代码负责执行。**

完整流程一般是：

~~~text
用户问题 + 工具 schema
    ↓
模型判断是否需要工具
    ↓
如果需要，输出 tool_calls JSON
    ↓
后端代码解析函数名和参数
    ↓
代码真正执行工具
    ↓
把工具结果作为 tool message 塞回上下文
    ↓
再次调用模型
    ↓
模型生成最终答案
~~~

这里最关键的是工具定义，也就是 JSON Schema。开发者需要清楚描述工具的 `name`、`description` 和 `parameters`，尤其是 `description`，因为模型会根据这些描述判断要不要调用工具、调用哪个工具、参数怎么填。

另外，Function Calling 也支持并行工具调用。比如用户问“北京和上海天气分别怎么样”，模型可以一次返回多个 `tool_calls`，后端代码并行执行这些工具，再把所有结果一起返回给模型。这样可以减少多轮模型调用和工具调用带来的延迟。

一句话总结：

> **Function Calling 是一种让大模型输出结构化工具调用请求，再由宿主程序执行工具，最后把结果交给模型生成答案的机制。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **Function Calling 的本质不是模型执行工具，而是模型输出结构化调用指令，由代码真正执行工具。**

很多人容易误解为：

~~~text
大模型自己联网
大模型自己查数据库
大模型自己调用 API
大模型自己执行代码
~~~

这是不准确的。

大模型本身只会生成文本，它不会真正执行函数，也不会直接访问外部系统。

Function Calling 做的事情是：

~~~text
让模型输出标准化 JSON
    ↓
让后端代码按 JSON 执行函数
    ↓
再把函数结果返回给模型
~~~

所以它解决的核心问题是：

> **把以前不稳定的“解析自然语言调用工具”，变成稳定的“解析结构化 JSON 调用工具”。**

## 三、Function Calling 解决了什么问题？

在 Function Calling 出现之前，如果想让模型调用工具，通常只能让模型输出类似这样的自然语言：

~~~text
我需要查询北京的天气。
~~~

然后后端代码再写一堆规则去解析：

~~~python
if "天气" in model_output:
    call_weather_api()
~~~

这种方式很脆弱。

因为模型可能换一种说法：

~~~text
我帮你查一下北京今天的气温。
~~~

或者：

~~~text
需要调用天气服务获取北京天气。
~~~

这时候原来的规则就可能匹配不上。

Function Calling 的改进是：

> **模型不再输出自然语言调用意图，而是输出结构化 JSON。**

例如：

~~~json
{
  "name": "get_weather",
  "arguments": {
    "city": "北京"
  }
}
~~~

这样后端代码就可以稳定解析：

~~~text
函数名：get_weather
参数：city=北京
~~~

这比解析自然语言可靠得多。

## 四、Function Calling 中的三个角色

可以把 Function Calling 理解成一个任务委托流程。

### 1. 开发者：定义工具

开发者负责告诉模型：

~~~text
有哪些工具
每个工具叫什么
每个工具能做什么
每个工具需要哪些参数
参数有什么限制
~~~

这部分通过 JSON Schema 完成。

---

### 2. 模型：决定调用

模型负责判断：

~~~text
当前问题需不需要工具
需要调用哪个工具
工具参数应该怎么填
~~~

注意：

> **模型只负责输出调用请求，不负责真正执行。**

---

### 3. 宿主代码：真正执行

后端代码负责：

~~~text
解析 tool_calls
找到对应函数
执行真实 API / 数据库 / 搜索 / 计算逻辑
拿到结果
把结果返回给模型
~~~

所以三者分工可以记成：

| 角色 | 作用 |
|---|---|
| 开发者 | 写工具说明书 |
| 模型 | 判断要调用哪个工具 |
| 后端代码 | 真正执行工具 |

最重要的一句话：

> **模型是决策者，不是执行者。**

---

## 五、工具定义：JSON Schema

Function Calling 的工具定义，本质上是一份结构化的工具说明书。

例如：

~~~python
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询指定城市的实时天气，包含气温、天气状况、风向风速，仅支持中国大陆城市",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，如北京、上海，不要带省份前缀"
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "温度单位，默认用摄氏度"
                    }
                },
                "required": ["city"]
            }
        }
    }
]
~~~

---

## 六、Schema 中几个关键字段

### 1. name：工具名

`name` 是工具的唯一标识。

模型输出 `tool_calls` 时，会用这个名字告诉后端：

~~~text
我要调用哪个函数。
~~~

例如：

~~~text
get_weather
search_news
query_stock_price
retrieve_document
~~~

---

### 2. description：工具描述

`description` 非常关键。

因为模型判断是否调用工具、调用哪个工具，主要就看这个描述。

如果写得太模糊：

~~~text
获取天气
~~~

模型可能会误用。

更好的写法是：

~~~text
查询指定城市的实时天气，包含气温、天气状况、风向风速，仅支持中国大陆城市
~~~

这样模型就更清楚：

- 工具能做什么；
- 工具不能做什么；
- 适合什么场景；
- 不适合什么场景。

所以记住：

> **description 写得越清楚，模型调用工具越准确。**

---

### 3. parameters：参数结构

`parameters` 用来告诉模型函数需要哪些参数。

包括：

~~~text
参数名
参数类型
参数含义
是否必填
可选值范围
示例格式
~~~

例如：

~~~json
{
  "city": {
    "type": "string",
    "description": "城市名称，如北京、上海"
  }
}
~~~

如果参数描述不清楚，模型就可能填错参数。

---

### 4. required：必填参数

`required` 表示哪些参数必须提供。

例如：

~~~json
"required": ["city"]
~~~

表示调用天气工具时，必须提供 `city` 参数。

---

## 七、完整调用流程：两轮对话 + 中间执行

Function Calling 的运行流程可以理解为：

> **两轮模型对话，中间夹一次工具执行。**

---

### 第一轮：模型判断是否需要工具

开发者把用户问题和工具列表一起传给模型。

例如用户问：

~~~text
北京今天天气怎么样？
~~~

模型发现自己不能凭空知道实时天气，于是输出：

~~~json
{
  "tool_calls": [
    {
      "id": "call_001",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\": \"北京\"}"
      }
    }
  ]
}
~~~

这表示：

> **模型还没有最终答案，它需要调用工具。**

---

### 中间执行：代码真正调用工具

后端代码拿到 `tool_calls` 后，做几件事：

~~~text
解析函数名
解析参数
找到对应函数
执行函数
拿到结果
~~~

例如：

~~~text
调用 get_weather(city="北京")
返回：北京今天晴，15°C，东北风 3 级
~~~

注意：

> **这一步不是模型执行的，而是后端代码执行的。**

---

### 第二轮：模型基于工具结果生成答案

后端把工具结果作为 `role=tool` 的消息加入对话历史：

~~~json
{
  "role": "tool",
  "tool_call_id": "call_001",
  "content": "北京今天晴，15°C，东北风 3 级"
}
~~~

然后再次调用模型。

模型看到工具结果后，生成最终回答：

~~~text
北京今天晴朗，气温 15°C，东北风 3 级，适合外出。
~~~

---

## 八、为什么需要 tool_call_id？

当模型输出工具调用时，每个调用都会有一个 `tool_call_id`。

它的作用是：

> **把模型发出的工具调用请求，和后端返回的工具执行结果对应起来。**

尤其是并行工具调用时，这个字段非常重要。

例如模型一次返回两个调用：

~~~text
call_001：查北京天气
call_002：查上海天气
~~~

后端执行完后，需要分别返回：

~~~text
tool_call_id = call_001 的结果
tool_call_id = call_002 的结果
~~~

这样模型才能知道哪个结果对应哪个工具请求。

---

## 九、并行工具调用

Function Calling 支持一次返回多个 `tool_calls`。

比如用户问：

~~~text
帮我查北京和上海今天的天气。
~~~

模型可以一次返回：

~~~json
{
  "tool_calls": [
    {
      "id": "call_001",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\": \"北京\"}"
      }
    },
    {
      "id": "call_002",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\": \"上海\"}"
      }
    }
  ]
}
~~~

后端代码可以同时执行：

~~~text
get_weather("北京")
get_weather("上海")
~~~

然后把两个结果一起塞回模型。

---

## 十、为什么需要并行工具调用？

如果没有并行调用，就只能串行：

~~~text
第一轮：模型说要查北京天气
    ↓
代码查北京天气
    ↓
第二轮：模型说还要查上海天气
    ↓
代码查上海天气
    ↓
第三轮：模型生成最终答案
~~~

这样会增加很多延迟。

并行调用可以变成：

~~~text
第一轮：模型一次性输出北京和上海两个 tool_calls
    ↓
代码并行查询北京和上海天气
    ↓
第二轮：模型基于两个结果生成最终答案
~~~

好处是：

> **减少模型调用轮数，降低整体延迟。**

后端可以用：

~~~text
asyncio.gather
多线程
协程
任务队列
~~~

来并行执行多个互不依赖的工具。

---

## 十一、什么时候能并行，什么时候不能并行？

并行调用的前提是：

> **多个工具调用之间没有依赖关系。**

### 可以并行的例子

用户问：

~~~text
查一下北京、上海、广州今天的天气。
~~~

这三个查询互不依赖：

~~~text
查北京天气
查上海天气
查广州天气
~~~

所以可以并行。

---

### 不能并行的例子

用户问：

~~~text
帮我查一下我的订单物流。
~~~

可能流程是：

~~~text
先根据用户信息查订单号
    ↓
再根据订单号查物流
~~~

第二步依赖第一步结果，所以不能并行。

这种情况下应该串行调用：

~~~text
第一轮：查订单号
第二轮：根据订单号查物流
第三轮：生成答案
~~~

## 十四、Function Calling 和自然语言解析工具调用的区别

| 对比项 | 自然语言解析工具调用 | Function Calling |
|---|---|---|
| 模型输出 | 自然语言 | 结构化 JSON |
| 后端解析 | 靠关键词、正则、if/else | 按 schema 解析 |
| 稳定性 | 低 | 高 |
| 标准化 | 差 | 好 |
| 参数提取 | 容易出错 | 更明确 |
| 工程可维护性 | 差 | 更好 |

Function Calling 的核心改进是：

> **把“猜模型想干什么”，变成“解析模型明确输出的 JSON 指令”。**

## 十五、常见误区

### 1. 误区一：模型自己调用 API

错误理解：

~~~text
模型自己访问网络，自己调用 API。
~~~

正确理解：

~~~text
模型只输出调用请求，真正调用 API 的是后端代码。
~~~

---

### 2. 误区二：Function Calling 等于执行代码

错误理解：

~~~text
模型可以直接执行函数。
~~~

正确理解：

~~~text
模型只是输出函数名和参数，执行函数的是宿主程序。
~~~

---

### 3. 误区三：工具描述随便写就行

错误理解：

~~~text
只要给模型一个函数名，它就知道怎么用。
~~~

正确理解：

~~~text
description 和 parameters 写得越清楚，模型越能正确选择工具和填写参数。
~~~

---

### 4. 误区四：所有工具都能并行调用

错误理解：

~~~text
多个工具调用都可以并行。
~~~

正确理解：

~~~text
只有互不依赖的工具调用才能并行；如果后一个工具依赖前一个工具结果，就必须串行。
~~~

## 十六、结合 Agent 项目怎么说

如果面试官问：

> **你在 Agent 项目里怎么理解 Function Calling？**

可以这样回答：

在 Agent 项目里，Function Calling 本质上是 LLM 和外部工具之间的标准接口。

比如我的金融分析 Agent 中，模型本身并不会真正访问股票行情、新闻搜索或者向量数据库。它只是根据工具 schema 判断当前问题是否需要调用工具。如果需要，它会输出结构化的工具调用请求，比如调用新闻搜索工具、股票行情工具或者 RAG 检索工具。后端代码解析这个请求后，真正去执行 API 调用或向量检索，然后把结果返回给模型，让模型基于这些外部信息继续推理和生成答案。

所以在工程实现里，我会把它拆成几个部分：

~~~text
工具注册
    ↓
工具 schema 生成
    ↓
模型选择工具
    ↓
解析 tool_calls
    ↓
执行具体函数
    ↓
把 Observation / tool result 写回上下文
    ↓
模型继续生成
~~~

这和 ReAct 思路很像，都是让模型进行工具选择和多步推理。区别是 Function Calling 更标准化，模型输出的是结构化 JSON，而不是让开发者从自然语言里解析 Action。

## 十七、面试官可能继续追问

### 1. Function Calling 中模型到底做了什么？

模型做的是决策：

~~~text
判断要不要调用工具
判断调用哪个工具
根据 schema 填写工具参数
输出 tool_calls JSON
~~~

它不负责执行工具。

---

### 2. 工具是谁执行的？

工具由宿主程序执行，也就是我们的后端代码。

模型输出函数名和参数后，代码根据函数名找到真实函数，然后调用 API、查数据库、检索向量库或者执行计算逻辑。

---

### 3. 为什么 Function Calling 比自然语言解析更稳定？

因为 Function Calling 输出的是结构化 JSON，函数名、参数字段都有明确格式。

自然语言解析则依赖关键词、正则或 if/else，模型稍微换一种说法就可能解析失败。

---

### 4. description 为什么重要？

因为模型判断是否调用工具、调用哪个工具、参数怎么填，主要依赖工具的 `description` 和参数说明。

如果描述模糊，模型就容易误调用、漏调用或者参数填错。

---

### 5. Function Calling 的调用流程是什么？

可以概括为：

~~~text
第一轮：模型输出 tool_calls
中间：代码执行工具
第二轮：模型基于工具结果生成最终答案
~~~

也就是：

> **两轮对话 + 中间执行。**

---

### 6. 什么是并行工具调用？

并行工具调用是指模型一次返回多个 `tool_calls`。

如果这些工具之间没有依赖关系，后端可以并行执行，然后把所有结果一起返回给模型。

这样可以减少多轮调用带来的延迟。

---

### 7. 哪些工具调用不能并行？

有依赖关系的工具不能并行。

例如：

~~~text
先查订单号
再根据订单号查物流
~~~

第二个工具依赖第一个工具结果，所以必须串行。

---

## 十八、记忆版总结

这篇文章可以压缩成一句话：

> **Function Calling 的本质是：模型输出结构化工具调用请求，代码真正执行工具，再把结果返回给模型生成最终答案。**

进一步记：

~~~text
模型：负责决策
代码：负责执行
Schema：负责描述工具
tool_calls：模型输出的结构化调用请求
role=tool：工具执行结果返回给模型
两轮对话：先要工具，再基于工具结果回答
并行调用：一次输出多个 tool_calls，后端并行执行
~~~

最重要的一句话：

> **Function Calling 不是模型自己调用函数，而是模型告诉代码“该调用哪个函数、参数是什么”。**
