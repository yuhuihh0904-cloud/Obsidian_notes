
## 一、面试标准回答

面试官问：**MCP 的通信方式是什么？Transport 是什么？**

可以这样回答：

MCP 不只是定义“工具长什么样”，还要定义 **MCP Client 和 MCP Server 之间怎么通信**。这个通信层就叫 **Transport**。

Transport 主要解决：

~~~text
消息怎么发送
消息怎么接收
连接怎么建立
流式结果怎么返回
本地 Server 和远程 Server 怎么通信
~~~

MCP 常见通信方式主要有：

~~~text
stdio
SSE
Streamable HTTP
~~~

其中：

- **stdio** 主要用于本地 MCP Server；
- **SSE** 是早期远程通信方案；
- **Streamable HTTP** 是后续更推荐的远程通信方案；
- SSE 因为单向流、请求和响应分裂、状态管理复杂，逐渐被 Streamable HTTP 替代。

简单理解：

> **stdio 适合本地工具，Streamable HTTP 适合远程工具，SSE 是早期过渡方案。**

一句话总结：

> **MCP 的 Transport 负责 Client 和 Server 之间的消息传输；本地用 stdio，远程用 Streamable HTTP。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **MCP 不只是工具协议，它还必须解决通信问题。**

因为 MCP Client 和 MCP Server 之间需要不断交换消息：

~~~text
列出工具
调用工具
返回工具结果
读取资源
获取 prompt
流式输出
错误处理
取消请求
状态同步
~~~

所以 MCP 必须定义一层 Transport。

可以这样理解：

~~~text
MCP 协议：规定说什么
MCP Transport：规定怎么说
~~~

---

## 三、什么是 MCP Transport？

MCP 可以拆成两层：

~~~text
上层：协议内容
- tools
- resources
- prompts
- sampling
- JSON-RPC 消息格式

下层：Transport
- stdio
- SSE
- Streamable HTTP
~~~

Transport 的作用是：

> **负责把 MCP 的 JSON-RPC 消息从 Client 传到 Server，再把结果从 Server 传回 Client。**

所以：

~~~text
JSON-RPC 是消息格式
Transport 是消息通道
~~~

---

## 四、第一种 Transport：stdio

`stdio` 是 MCP 最常见的本地通信方式。

它本质上是：

~~~text
Client 启动一个本地 MCP Server 进程
    ↓
Client 通过 stdin 给 Server 发消息
    ↓
Server 通过 stdout 给 Client 返回消息
~~~

结构：

~~~text
AI Client
    ↓
stdin / stdout
    ↓
本地 MCP Server
    ↓
真实工具 / API / 文件系统
~~~

例如配置：

~~~json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/me/docs"]
    }
  }
}
~~~

这里 AI Client 会启动一个本地子进程，然后通过标准输入输出通信。

---

## 五、stdio 的优点和缺点

### 1. 优点

stdio 的优点是：

- 实现简单；
- 不需要 HTTP 服务；
- 不需要开放端口；
- 不需要网络通信；
- 本地性能高；
- 天然支持双向通信；
- 很适合本地开发和桌面客户端。

可以记成：

> **本地 MCP Server 首选 stdio。**

---

### 2. 缺点

stdio 的问题是：

> **它只适合本地进程，不适合远程服务。**

因为它要求：

- Client 能启动本地子进程；
- MCP Server 和 Client 在同一台机器；
- 不方便多个客户端共享；
- 不适合云端部署；
- 不适合远程企业服务。

所以：

~~~text
本地工具：stdio 很合适
远程工具：stdio 不合适
~~~

---

## 六、第二种 Transport：SSE

SSE 全称是：

~~~text
Server-Sent Events
~~~

它是 HTTP 的一种流式推送方式。

结构大概是：

~~~text
Client 发起连接
    ↓
Server 保持连接不断开
    ↓
Server 持续向 Client 推送事件
~~~

SSE 的特点是：

> **服务端可以持续向客户端推送数据。**

例如：

~~~text
data: message 1

data: message 2

data: done
~~~

这很适合：

- 日志流；
- 事件流；
- token streaming；
- 服务端持续推送。

---

## 七、为什么早期 MCP 会用 SSE？

因为 MCP 远程通信需要：

- 基于 HTTP；
- 支持流式返回；
- 实现相对简单；
- 浏览器和代理兼容性较好。

SSE 看起来天然适合：

~~~text
Server 持续返回工具执行过程和结果
~~~

所以早期 MCP 远程方案曾经使用过：

~~~text
HTTP + SSE
~~~

---

## 八、为什么 SSE 被弃用？

SSE 最大的问题是：

> **它本质上是单向流，只支持 Server 向 Client 推送。**

但 MCP 需要的是复杂双向通信：

~~~text
Client -> Server：发请求、调用工具、取消任务
Server -> Client：返回结果、流式输出、错误信息
~~~

SSE 会带来几个问题。

---

### 1. 请求和响应流分裂

SSE 通常会变成：

~~~text
HTTP POST：Client 发送请求
SSE GET：Server 返回流式结果
~~~

也就是说：

~~~text
请求走一个连接
响应流走另一个连接
~~~

这会导致：

- 状态难关联；
- 任务生命周期难管理；
- 取消请求复杂；
- 重连恢复复杂。

---

### 2. 不是真正的双向协议

SSE 只能：

~~~text
Server -> Client
~~~

不能在同一个连接里：

~~~text
Client <-> Server
~~~

所以 MCP 这种需要多轮交互的协议，用 SSE 会越来越别扭。

---

### 3. 长连接状态管理复杂

远程 MCP 可能涉及：

- 长任务；
- 多轮工具调用；
- 流式输出；
- 任务取消；
- 网络断开重连；
- 超时处理。

SSE 在这些场景下会让实现变复杂。

---

### 4. HTTP 语义不统一

SSE 模式下经常是：

~~~text
POST 负责提交请求
GET SSE 负责接收结果
另一个请求负责取消任务
~~~

整体协议被拆成多个通道，不够统一。

所以 SSE 逐渐被更统一的远程方案替代。

---

## 九、第三种 Transport：Streamable HTTP

Streamable HTTP 可以理解为：

> **用现代 HTTP 流式能力来统一完成请求、响应和流式输出。**

它的目标是解决 SSE 的问题。

核心思想：

~~~text
Client 通过 HTTP 发送请求
    ↓
Server 可以在同一个 HTTP 响应中流式返回 MCP 消息
    ↓
请求和响应生命周期更统一
~~~

相比 SSE，它更适合：

- 远程 MCP Server；
- 云部署；
- 长任务；
- 流式结果；
- 统一 HTTP 语义；
- 更自然的请求 / 响应模型。

---

## 十、stdio、SSE、Streamable HTTP 对比

| Transport | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| stdio | 本地 MCP Server | 简单、高性能、无需网络、天然双向 | 只适合本地，不适合远程 |
| SSE | 早期远程方案 | 支持服务端流式推送，实现简单 | 单向流，请求和响应分裂，状态管理复杂 |
| Streamable HTTP | 推荐远程方案 | HTTP 语义更统一，适合远程和流式输出 | 实现比 stdio 复杂 |

记忆方式：

~~~text
本地：stdio
早期远程：SSE
推荐远程：Streamable HTTP
~~~

---

## 十一、MCP 通信流程

以工具调用为例，通信过程是：

~~~text
MCP Client
    ↓
通过 Transport 发送 JSON-RPC 请求
    ↓
MCP Server 接收请求
    ↓
执行对应工具
    ↓
通过 Transport 返回 JSON-RPC 响应
    ↓
MCP Client 把结果交给模型
~~~

这里 Transport 可以是：

~~~text
stdio
Streamable HTTP
~~~

但上层消息格式仍然是 MCP 的 JSON-RPC。

所以：

> **Transport 可以变，但 MCP 消息语义不变。**

## 十六、结合 Agent 项目怎么说

如果面试官问：

> **你怎么理解 MCP 的通信层？**

可以这样回答：

我理解 MCP 不只是工具 schema 标准，它还定义了 Client 和 Server 怎么通信，也就是 Transport 层。

如果是本地 Agent，比如 Claude Desktop 或本地开发环境，很多 MCP Server 会用 `stdio`。Client 启动本地 MCP Server 子进程，然后通过 stdin/stdout 交换 JSON-RPC 消息。这种方式简单、高性能，也不需要网络端口。

如果是远程 MCP Server，比如企业内部服务、云端工具服务，就不能依赖 stdio。这类场景需要基于 HTTP 的远程通信。早期 MCP 用过 SSE，因为它天然支持服务端流式推送，但 SSE 本质是单向流，会导致请求和响应分裂、状态管理复杂，所以后来逐渐被 Streamable HTTP 替代。

所以在 Agent 系统设计里，我会这样判断：

~~~text
本地工具、本地文件系统、本地脚本：stdio
远程工具、企业服务、云端 API：Streamable HTTP
~~~

## 十七、面试官可能继续追问

### 1. MCP 为什么需要 Transport？

因为 MCP 不只是定义工具和消息格式，还需要解决：

~~~text
Client 和 Server 怎么连接
消息怎么传
结果怎么返回
流式数据怎么处理
~~~

这些属于通信层问题，所以需要 Transport。

---

### 2. stdio 为什么适合本地 MCP？

因为 stdio 通过本地进程的标准输入输出通信：

~~~text
Client 写 stdin
Server 读 stdin
Server 写 stdout
Client 读 stdout
~~~

它简单、快、不需要网络，很适合本地工具。

---

### 3. stdio 为什么不适合远程？

因为 stdio 要求 Client 能启动并管理本地子进程。

远程服务在另一台机器或云端，无法通过本地 stdin/stdout 通信，所以需要 HTTP 类 Transport。

---

### 4. SSE 最大问题是什么？

SSE 最大问题是：

> **它是单向流，不适合 MCP 这种复杂双向交互。**

它通常需要 POST 发请求、SSE GET 收结果，导致请求和响应流分裂。

---

### 5. 为什么 Streamable HTTP 更适合远程 MCP？

因为它能在统一 HTTP 语义下处理请求和流式响应，更适合云部署、长任务、流式结果和远程服务。

---

## 十八、记忆版总结

这篇文章可以压缩成一句话：

> **MCP Transport 解决 Client 和 Server 怎么通信；本地用 stdio，远程用 Streamable HTTP，SSE 因为单向流和状态管理复杂逐渐被弃用。**

进一步记：

~~~text

stdio：
~~~

> **本地 MCP Server 用 stdio，远程 MCP Server 用 Streamable HTTP。**
