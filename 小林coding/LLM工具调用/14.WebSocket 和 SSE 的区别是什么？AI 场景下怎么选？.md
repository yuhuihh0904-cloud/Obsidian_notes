
## 一、面试标准回答

面试官问：**WebSocket 和 SSE 通信有什么区别？各自有什么局限？**

可以这样回答：

WebSocket 和 SSE 都能实现服务端持续向客户端发送数据，但它们的底层机制和适用场景不一样。

**SSE，全称 Server-Sent Events，是基于 HTTP 的服务端单向推送机制。**  
客户端发起一个普通 HTTP 请求，服务端保持连接不关闭，然后不断往响应体里追加数据。它适合模型流式输出这种场景：模型持续生成 token，服务端持续推给前端，用户只需要接收和展示。

**WebSocket 是基于 TCP 的全双工通信协议。**  
它先通过 HTTP Upgrade 完成握手，然后升级成 WebSocket 连接。连接建立后，客户端和服务端都可以随时主动发消息，适合真正需要双向实时交互的场景，比如实时打断、多人协同、游戏同步等。

所以两者最核心的区别是：

~~~text
SSE：服务端 -> 客户端，单向推送
WebSocket：客户端 <-> 服务端，全双工通信
~~~

但不能简单说 WebSocket 更强就一定更好。  
在 LLM 文字流式输出场景里，大多数情况下是模型持续往前端推 token，用户只是看；用户下一轮输入可以重新发一个普通 POST 请求，所以 SSE 已经够用，而且它是 HTTP 原生机制，运维简单、代理兼容性好、扩展也更容易。

WebSocket 只有在真正需要客户端随时主动发消息时才值得引入，比如用户要在模型说话中途实时打断，或者多人协同编辑这种高频双向通信场景。

一句话总结：

> **单向流式输出用 SSE，真正需要双向实时交互才用 WebSocket。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **SSE 和 WebSocket 不是“简单版”和“高级版”的关系，而是通信模型不同，适用场景不同。**

常见误区是：

~~~text
WebSocket 功能更强，所以一定比 SSE 好
SSE 是 WebSocket 的简化版
AI 对话必须用 WebSocket
~~~

这些理解都不准确。

更合理的判断是：

~~~text
如果只是服务端持续推送文本：SSE 足够
如果客户端和服务端都要随时主动发消息：WebSocket 更合适
~~~

在大多数 LLM 文字对话场景里，SSE 反而是更常见的选择。

---

## 三、先理解普通 HTTP 的问题

普通 HTTP 是典型的：

~~~text
客户端请求
    ↓
服务端响应
    ↓
连接结束或复用
~~~

它是“一问一答”模型。

但 AI 对话不适合等模型生成完整答案后再一次性返回。

因为模型可能要生成几秒甚至十几秒，如果用户一直看空白页面，体验很差。

所以更好的体验是：

~~~text
模型生成一个 token
    ↓
服务端推送一个 token
    ↓
前端实时显示
~~~

这就需要一种能让服务端持续推送数据的通信方式。

SSE 和 WebSocket 都是在解决这个问题，但方式不同。

---

## 四、SSE：基于 HTTP 的单向流式推送

SSE 全称：

~~~text
Server-Sent Events
~~~

它本质上还是 HTTP。

客户端发起请求时带上：

~~~text
Accept: text/event-stream
~~~

服务端收到后不立刻关闭连接，而是持续往响应体里写数据。

可以理解成：

> **服务端往客户端打开了一根单向水管。**

数据格式通常是：

~~~text
data: {"token": "你"}

data: {"token": "好"}

data: [DONE]
~~~

浏览器可以用 `EventSource` 接收 SSE 数据。

SSE 的特点：

- 基于 HTTP；
- 服务端单向推送；
- 只支持文本；
- 实现简单；
- 代理和防火墙兼容性好；
- 很适合 LLM 文本流式输出。

---

## 五、WebSocket：真正的双向通信

WebSocket 是一个独立协议，建立在 TCP 之上。

它的建立过程是：

~~~text
客户端先发 HTTP 请求
    ↓
请求头里带 Upgrade: websocket
    ↓
服务端同意后返回 101 Switching Protocols
    ↓
连接升级为 WebSocket
~~~

升级之后，这条连接就不再是普通 HTTP 的一问一答，而是一条全双工通道：

~~~text
客户端可以随时发消息
服务端也可以随时发消息
~~~

WebSocket 的特点：

- 全双工；
- 双方都能主动发消息；
- 支持文本和二进制；
- 适合高频实时交互；
- 但连接有状态，扩展和运维更复杂。

---

## 六、SSE 和 WebSocket 的核心区别

| 对比项 | SSE | WebSocket |
|---|---|---|
| 协议基础 | HTTP 原生机制 | 独立协议，基于 TCP |
| 通信方向 | 服务端单向推送 | 双向全双工 |
| 客户端发消息 | 需要另发 HTTP 请求 | 可以在同一连接中发送 |
| 数据格式 | 文本 | 文本 + 二进制 |
| 实现复杂度 | 低 | 高 |
| 代理兼容性 | 好 | 可能被代理 / 防火墙拦截 |
| 横向扩展 | 更容易 | 更麻烦 |
| 适合场景 | LLM 文本流式输出 | 实时双向通信 |

最核心一句：

> **SSE 是服务端单向推，WebSocket 是双方都能主动说话。**

---

## 七、SSE 的局限

### 1. 单向性导致双通道架构

SSE 只能：

~~~text
Server -> Client
~~~

如果用户要发消息，需要另起一个普通 HTTP 请求：

~~~text
用户发消息：POST /chat
模型回消息：SSE /stream
~~~

也就是说，一个对话可能有两条通道：

~~~text
上行：HTTP POST
下行：SSE
~~~

这会带来一定状态管理复杂度，需要通过：

~~~text
conversation_id
request_id
message_id
~~~

把两条通道关联起来。

---

### 2. HTTP/1.1 下有连接数限制

浏览器对同一域名的 HTTP/1.1 连接数有限制，常见是 6 条。

SSE 是长连接，会一直占用连接。

如果用户同时打开多个标签页，就可能导致后面的 SSE 请求排队，表现为页面卡住。

不过 HTTP/2 有多路复用，可以缓解这个问题。

---

### 3. 只能传文本

SSE 只支持文本。

如果要传：

~~~text
图片
音频
二进制数据
~~~

通常要先 Base64 编码。

Base64 会带来：

- 数据体积膨胀；
- 编码 / 解码开销；
- 延迟增加。

所以 SSE 不适合实时语音、视频、图片流这类场景。

---

### 4. 断线续传需要额外设计

SSE 有自动重连机制，但如果要做到断线后不丢消息，需要服务端支持：

~~~text
Last-Event-ID
~~~

客户端重连时带上最后收到的事件 ID，服务端从后续位置继续推送。

如果没有设计好，断线后可能丢失中间 token 或结果。

---

## 八、WebSocket 的局限

### 1. 有状态，横向扩展麻烦

WebSocket 最大的问题是：

> **连接是有状态的。**

一条 WebSocket 连接建立后，会绑定到某一台后端服务器。

后续这个用户的消息都必须发到同一台服务器，因为连接状态在那台机器上。

这会导致横向扩展麻烦：

~~~text
用户 A 连接在 Server 1
用户 B 连接在 Server 2
新增 Server 3
老连接无法自动迁移到 Server 3
~~~

如果要解决，通常需要：

- sticky session；
- Redis 保存连接状态；
- Pub/Sub 消息转发；
- 网关层维护连接映射。

这些都会增加架构复杂度。

---

### 2. 代理和防火墙可能拦截

WebSocket 需要 HTTP Upgrade 握手。

但很多企业代理、老旧 CDN、安全网关可能不支持或不允许：

~~~text
Upgrade: websocket
```

结果就是连接建立失败。

SSE 因为始终是普通 HTTP 请求，兼容性通常更好。

---

### 3. 没有内置请求-响应配对机制

HTTP 天然是：

~~~text
一个请求对应一个响应
~~~

但 WebSocket 里只是不断收发消息。

如果客户端发了多个请求，服务端返回多个结果，就需要自己设计：

~~~text
request_id
response_id
callback map
timeout
retry
~~~

否则客户端不知道某条响应对应哪个请求。

---

### 4. 断线重连和未完成请求处理复杂

WebSocket 断线后，需要自己处理：

- 重连；
- 连接恢复；
- 未完成请求怎么办；
- 消息是否重发；
- 是否会重复执行；
- 服务端状态如何恢复。

这些都比普通 HTTP / SSE 更复杂。

---

## 九、AI 场景下怎么选？

核心原则：

> **单向推用 SSE，真正双向实时才用 WebSocket。**

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| LLM 文字流式输出 | SSE | 模型单向推 token，简单稳定 |
| 普通多轮对话 | SSE + POST | 用户发消息走 POST，模型回复走 SSE |
| 用户中途打断模型输出 | WebSocket | 客户端需要在同一连接中实时发停止指令 |
| 多人协同编辑 | WebSocket | 高频双向消息 |
| 游戏 / 实时同步 | WebSocket | 双向实时通信 |
| 实时语音对话 | WebRTC | 音频更需要低延迟，适合 UDP |
| MCP 远程 Server | Streamable HTTP | 保留 HTTP 兼容性，同时支持流式输出 |

## 十、为什么很多 LLM API 用 SSE 而不是 WebSocket？

因为 LLM 文字输出通常是：

~~~text
服务端持续生成 token
客户端持续接收 token
~~~

这是典型单向推送。

用户下一轮输入可以重新发 POST 请求，不需要和模型输出共享同一条双向连接。

SSE 的优势更明显：

- 基于 HTTP；
- 实现简单；
- 运维简单；
- 代理兼容性好；
- 易于横向扩展；
- 足够满足文本流式输出。

所以在大多数文字对话场景里：

> **WebSocket 的全双工能力用不上，反而会引入额外复杂度。**

---

## 十一、什么时候必须考虑 WebSocket？

当你的场景需要客户端随时主动发消息时，WebSocket 更合适。

典型场景：

~~~text
模型说话中途，用户实时打断
多人协同编辑
实时白板
游戏状态同步
实时金融行情交互
双向控制类应用
~~~

比如“中途打断模型输出”：

### 用 SSE

~~~text
关闭当前 SSE 流
    ↓
另发一个 POST 请求告诉服务端停止
    ↓
服务端处理取消
~~~

会有一点割裂。

### 用 WebSocket

~~~text
同一条连接里直接发送 stop 消息
    ↓
服务端立刻停止生成
~~~

体验更自然。

## 十五、结合 Agent / LLM 项目怎么说

如果面试官问：

> **你的 Agent 或 LLM 项目里，流式输出会用 SSE 还是 WebSocket？**

可以这样回答：

如果只是普通 LLM 文本流式输出，我会优先用 SSE。

因为这个场景本质上是服务端持续推 token 给前端，客户端不需要在同一条连接里高频发消息。用户下一轮问题可以走普通 POST 请求，模型回复再走 SSE。这样实现简单，基于 HTTP，部署和代理兼容性都更好。

但如果产品要求用户可以在模型生成过程中实时打断，或者需要多人协同、实时控制这类双向通信能力，我会考虑 WebSocket。

不过 WebSocket 会带来状态连接、负载均衡、断线重连、请求响应配对这些复杂问题，所以只有当双向实时能力真的必要时才使用。

如果是实时语音对话，我不会优先用 SSE 或 WebSocket，而是考虑 WebRTC，因为语音更关注低延迟，不能接受 TCP 重传带来的卡顿。

## 十六、面试官可能继续追问

### 1. SSE 和 WebSocket 最本质区别是什么？

最本质区别是通信方向：

~~~text
SSE：服务端单向推送
WebSocket：双向全双工通信
~~~

---

### 2. 为什么 LLM 文字流式输出常用 SSE？

因为 LLM 文字输出通常只是服务端不断推 token 给客户端，单向推送就够了。

SSE 基于 HTTP，实现简单，代理兼容性好，运维成本低。

---

### 3. SSE 有哪些坑？

主要有三个：

~~~text
单向性导致上行 POST + 下行 SSE 的双通道架构
HTTP/1.1 下同域名连接数限制
只能传文本，二进制要 Base64 编码
~~~

此外，断线续传需要服务端支持 Last-Event-ID。

---

### 4. WebSocket 有哪些坑？

主要有三个：

~~~text
有状态，横向扩展麻烦
可能被代理、防火墙、老旧 CDN 拦截
没有内置请求-响应配对机制
~~~

此外，断线重连和未完成请求恢复也要自己处理。

---

### 5. 为什么 WebSocket 横向扩展更麻烦？

因为 WebSocket 连接建立后，会绑定到某台后端服务器。

后续消息必须回到这台服务器，所以需要 sticky session、连接状态管理、Redis Pub/Sub 等机制。

这比普通 HTTP 请求更难扩展。

---

### 6. 为什么 WebSocket 不适合所有 AI 对话？

因为很多 AI 文字对话只是服务端推 token，不需要客户端随时发消息。

这时 WebSocket 的全双工能力用不上，反而会带来部署和扩展复杂度。

---

## 十七、记忆版总结

这篇文章可以压缩成一句话：

> **SSE 是 HTTP 上的服务端单向流，适合 LLM 文本流式输出；WebSocket 是全双工长连接，适合真正双向实时交互。**

进一步记：

~~~text
SSE：
HTTP 原生
Server -> Client
简单、轻量、代理友好
适合 LLM token streaming
局限：单向、文本、连接数、双通道状态管理

WebSocket：
TCP 长连接
Client <-> Server
全双工、可传二进制
适合中途打断、协同编辑、实时同步
局限：有状态、扩展麻烦、代理可能拦截、请求响应要自己配对
~~~

选型口诀：

~~~text
单向推 token：SSE
真正双向实时：WebSocket
实时音视频：WebRTC
远程 MCP：Streamable HTTP
~~~

最重要的一句话：

> **不要因为 WebSocket 功能更强就默认选它，通信方案要看场景：LLM 文字流式输出用 SSE 通常更合适。**
