
## 一、面试标准回答

面试官问：**Function Calling、MCP、Skill 这三个概念有什么区别？**

可以这样回答：

Function Calling、MCP、Skill 不是三个平行的竞争方案，而是 Agent 工具系统里的三层架构。

**Function Calling 是最底层的调用语言**，解决的是：**模型怎么触发函数调用**。  
模型根据工具 schema 输出结构化的 `tool_calls` JSON，告诉宿主程序要调用哪个函数、传什么参数。真正执行函数的是后端代码。

**MCP 是中间层的工具标准化协议**，解决的是：**工具怎么暴露给模型和 Agent**。  
它把数据库、文件系统、GitHub、浏览器、API 等外部能力封装成 MCP Server，AI Client 连接后可以发现这些工具，并把它们提供给模型使用。

**Skill 是最上层的任务流程封装**，解决的是：**拿到工具之后，Agent 应该按什么流程完成任务**。  
Skill 通常包含 `SKILL.md`、脚本、模板、参考资料等，它不是一个原子工具，而是一套可复用的任务 SOP。

所以三者关系是：

~~~text
Skill：定义任务流程
MCP：提供标准化工具
Function Calling：让模型触发具体工具调用
~~~

一句话总结：

> **Function Calling 是调用语言，MCP 是工具箱，Skill 是操作手册。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **Function Calling、MCP、Skill 不是替代关系，而是从底到顶的三层协作关系。**

很多人容易把它们理解成：

~~~text
Function Calling 是一种工具调用方案
MCP 是另一种工具调用方案
Skill 也是一种工具调用方案
三者是竞争关系
~~~

这个理解不准确。

更准确的理解是：

~~~text
Function Calling：模型怎么发起工具调用
MCP：工具怎么标准化暴露和复用
Skill：复杂任务怎么按流程完成
~~~

也就是：

> **Function Calling 管调用，MCP 管工具，Skill 管流程。**

---

## 三、第一层：Function Calling

Function Calling 是最底层。

它解决的问题是：

> **模型怎么告诉外部程序：我要调哪个函数、参数是什么。**

模型不会真正执行函数，它只是输出结构化调用请求。

例如：

~~~json
{
  "tool_calls": [
    {
      "name": "query_database",
      "arguments": {
        "sql": "SELECT * FROM sales"
      }
    }
  ]
}
~~~

然后宿主程序解析这个 JSON，真正执行数据库查询。

Function Calling 的核心分工是：

~~~text
模型：负责判断调用哪个函数、参数怎么填
代码：负责真正执行函数
~~~

所以它可以理解为：

> **模型和工具之间的调用语言。**

---

## 四、第二层：MCP

MCP 在 Function Calling 之上。

它解决的问题是：

> **工具怎么标准化暴露给不同 Agent 或 AI Client。**

没有 MCP 时，每个应用都要自己写工具 schema、自己接 API、自己管理工具逻辑。

有了 MCP 之后，可以把工具封装成独立 MCP Server：

~~~text
AI Client / Agent
    ↓
MCP Client
    ↓
MCP Server
    ↓
数据库 / 文件系统 / GitHub / 浏览器 / API
~~~

MCP Server 可以告诉客户端：

~~~text
我有哪些工具
每个工具叫什么
每个工具需要什么参数
每个工具能返回什么结果
~~~

所以 MCP 的核心作用是：

> **让工具从应用内代码，变成可复用、可发现、可标准化接入的外部服务。**

---

## 五、第三层：Skill

Skill 在最上层。

它解决的问题是：

> **Agent 有了工具之后，应该按什么流程完成复杂任务。**

比如用户说：

~~~text
帮我做一份销售数据分析报告。
~~~

这个任务不是调用一个工具就能完成的，而是需要一套流程：

~~~text
1. 查询销售数据
2. 清洗数据
3. 分析趋势
4. 找出下滑产品线
5. 生成图表
6. 按模板写报告
~~~

Skill 就是把这类流程封装起来。

一个 Skill 可能包含：

~~~text
SKILL.md
scripts/
templates/
reference_docs/
assets/
~~~

所以 Skill 更像：

> **一份可复用的任务 SOP。**

它不是告诉 Agent “调用哪个具体函数”，而是告诉 Agent：

~~~text
做这类任务时，应该先做什么、后做什么、按什么标准输出。
~~~

---

## 六、三者的层级关系

三者从下到上是：

~~~text
Skill
  ↑
MCP
  ↑
Function Calling
~~~

更完整地说：

~~~text
Skill：定义任务流程和执行标准
    ↓
MCP：提供流程中需要用到的工具
    ↓
Function Calling：让模型触发具体工具调用
~~~

为什么 Function Calling 在最底层？

因为它是模型触发工具调用的基础格式。

为什么 MCP 在中间？

因为 MCP 负责管理和标准化工具，但最终工具还是要被模型通过类似 Function Calling 的方式触发。

为什么 Skill 在最上层？

因为 Skill 不关心单个工具怎么暴露，而是关心复杂任务怎么组织、怎么执行、怎么输出。

---

## 七、做菜类比

可以用做菜来理解三者：

~~~text
Function Calling：手
MCP：厨房
Skill：菜谱
~~~

### 1. Function Calling 像“手”

它让你能执行具体动作：

~~~text
拿刀
切菜
点火
翻锅
~~~

没有手，就算有厨房和菜谱，也做不了菜。

### 2. MCP 像“厨房”

厨房里有各种工具和资源：

~~~text
刀
锅
调料
食材
冰箱
燃气灶
~~~

MCP 让 Agent 知道有哪些工具可以用。

### 3. Skill 像“菜谱”

菜谱告诉你：

~~~text
先切菜
再热锅
再放油
再下主料
最后调味
~~~

没有菜谱，你虽然有厨房和手，但不知道怎么把菜做好。

所以：

> **Function Calling 让模型能调用，MCP 让模型有工具，Skill 让模型知道流程。**

## 八、一个完整故事串联三者

用户对 Agent 说：

~~~text
帮我分析最近三个月的销售数据，找出下滑的产品线，并给出改进建议。
~~~

### 1. Skill 层先起作用

Agent 发现这个任务属于“数据分析报告”类型，于是加载对应 Skill。

Skill 里定义了流程：

~~~text
第一步：从数据库取最近三个月销售数据
第二步：用 Python 做趋势分析
第三步：找出下滑产品线
第四步：生成图表
第五步：按报告模板输出建议
~~~

这里 Skill 负责：

> **告诉 Agent 该按什么流程做。**

### 2. MCP 层提供工具

执行流程时，Agent 需要工具。

比如：

~~~text
query_database：查询数据库
run_python：执行数据分析代码
generate_chart：生成图表
~~~

这些工具由 MCP Server 提供。

MCP 负责：

> **告诉 Agent 有哪些工具可用，并提供标准化调用入口。**

### 3. Function Calling 层触发调用

当模型决定要查数据库时，会输出类似：

~~~json
{
  "tool_calls": [
    {
      "name": "query_database",
      "arguments": {
        "sql": "SELECT product, revenue, date FROM sales WHERE date >= '2026-01-01'"
      }
    }
  ]
}
~~~

宿主程序或 MCP Client 接收到调用请求后，把它路由给对应 MCP Server 执行。

Function Calling 负责：

> **让模型用结构化格式发出具体工具调用请求。**

### 4. 最终形成完整链路

完整链路是：

~~~text
用户提出复杂任务
    ↓
Agent 加载相关 Skill
    ↓
Skill 给出任务流程
    ↓
流程中需要外部能力
    ↓
MCP 提供可用工具
    ↓
模型通过 Function Calling 触发工具调用
    ↓
工具返回结果
    ↓
Agent 按 Skill 模板生成最终报告
~~~

## 九、三者对比表

| 对比项 | Function Calling | MCP | Skill |
|---|---|---|---|
| 所在层级 | 底层 | 中间层 | 上层 |
| 解决问题 | 模型怎么调用函数 | 工具怎么标准化暴露 | 任务怎么按流程完成 |
| 核心作用 | 输出结构化调用请求 | 工具发现、工具复用、工具服务化 | 流程编排、标准沉淀、模板复用 |
| 典型形式 | `tool_calls` JSON | MCP Server / MCP Client | `SKILL.md` + 脚本 + 模板 |
| 粒度 | 单次函数调用 | 一组标准化工具 | 一个完整任务流程 |
| 类比 | 调用语言 / 手 | 工具箱 / 厨房 | 操作手册 / 菜谱 |
| 关键词 | 调哪个函数，传什么参数 | 有哪些工具，怎么连接 | 先做什么，后做什么 |

## 十一、结合 Agent 项目怎么说

如果面试官问：

> **你在自己的 Agent 项目里怎么理解 Function Calling、MCP 和 Skill？**

可以这样回答：

在我的理解里，Function Calling、MCP 和 Skill 对应 Agent 系统的三个层次。

在金融分析 Agent 中，Function Calling 或类似机制负责让模型发起工具调用，比如调用行情查询、新闻搜索、RAG 检索等工具。模型只输出工具名和参数，真正执行的是后端工具代码。

MCP 更适合把通用工具标准化，比如 GitHub、文件系统、数据库、浏览器搜索、企业知识库等。如果这些工具未来要被多个 Agent 或多个项目复用，就可以封装成 MCP Server，而不是每个项目重复写一套工具接入逻辑。

Skill 则更适合沉淀复杂任务流程。比如“财报分析 Skill”可以规定：

~~~text
第一步：识别文档类型
第二步：提取公司、季度、核心财务指标
第三步：检索当前 PDF 证据
第四步：结合 Research Memory
第五步：生成结构化研报
第六步：输出风险点和后续关注问题
~~~

所以在我的项目里可以这样分工：

~~~text
Function Calling：模型触发具体工具调用
MCP：把通用工具服务化、标准化、可复用
Skill：把财报分析这类复杂任务流程沉淀下来
~~~

这三者不是互相替代，而是从底到顶协作。

## 十二、面试官可能继续追问

### 1. Function Calling、MCP、Skill 是竞争关系吗？

不是。

它们是不同层次的能力：

~~~text
Function Calling：底层调用格式
MCP：工具标准化协议
Skill：任务流程封装
~~~

三者可以在同一个 Agent 系统里同时使用。

---

### 2. 为什么说 Function Calling 是最底层？

因为模型要调用工具，必须先有一种方式表达：

~~~text
我要调用哪个函数
参数是什么
~~~

Function Calling 提供的就是这种结构化表达方式。

---

### 3. 为什么说 MCP 在 Function Calling 之上？

因为 MCP 主要负责工具的标准化暴露和发现。

但 MCP 暴露出来的工具，最终仍然需要以某种结构化调用格式交给模型触发，Function Calling 就是常见的底层调用语言。

---

### 4. 为什么说 Skill 在最上层？

因为 Skill 不关心单个函数怎么调用，而是关心完整任务怎么完成。

一个 Skill 里可能会包含多个步骤，每个步骤可能调用多个 MCP 工具，而每个工具调用底层又可能通过 Function Calling 触发。

---

### 5. 有了 MCP 还需要 Skill 吗？

需要。

MCP 只是提供工具，不会告诉 Agent 复杂任务应该怎么拆解、按什么顺序执行、按什么标准输出。

这些任务流程和标准由 Skill 提供。

---

### 6. 有了 Skill 还需要 MCP 吗？

需要。

Skill 只是流程和知识，不一定提供真实外部能力。

如果流程里需要查数据库、读文件、跑代码、搜索网页，这些具体动作仍然需要 MCP Tool 或其他工具系统来执行。

---

## 十三、记忆版总结

这篇文章可以压缩成一句话：

> **Function Calling 是调用语言，MCP 是工具箱，Skill 是操作手册；三者不是竞争关系，而是 Agent 工具系统从底到顶的三层架构。**

进一步记：

~~~text
Function Calling：
最底层
模型输出 tool_calls
解决“怎么调函数”

MCP：
中间层
工具标准化、服务化、可复用
解决“有哪些工具、怎么暴露工具”

Skill：
最上层
任务流程、脚本、模板、SOP
解决“拿到工具后怎么完成任务”
~~~

完整链路：

~~~text
Skill 定义流程
    ↓
MCP 提供工具
    ↓
Function Calling 触发调用
    ↓
工具执行并返回结果
    ↓
Agent 按 Skill 输出最终结果
~~~

最重要的一句话：

> **面试时不要把三者说成并列方案，要强调它们是底层调用、中层工具、上层流程的协作关系。**
