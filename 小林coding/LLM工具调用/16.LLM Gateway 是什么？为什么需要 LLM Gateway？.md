
## 一、面试标准回答

面试官问：**什么是 LLM Gateway？为什么需要它？**

可以这样回答：

LLM Gateway 本质上是：

> **大模型系统里的统一模型接入层。**

它的作用类似传统互联网架构里的：

- API Gateway；
- 数据库连接池；
- 服务代理层。

以前应用直接调用模型：

~~~text
业务代码
    ↓
OpenAI API
~~~

但当系统越来越复杂后，会出现很多问题：

- 模型供应商越来越多；
- 不同模型 API 格式不统一；
- Prompt 管理混乱；
- Token 和成本难统计；
- 限流和重试逻辑重复；
- 模型切换困难；
- 灰度和 fallback 难做；
- 权限和审计不好统一管理。

于是就需要一个统一中间层：

~~~text
业务系统
    ↓
LLM Gateway
    ↓
OpenAI / Claude / Gemini / Qwen / DeepSeek ...
~~~

LLM Gateway 负责：

~~~text
统一模型接口
模型路由
负载均衡
fallback
限流
鉴权
日志
监控
缓存
Prompt 管理
Token 成本统计
~~~

这样业务代码就不用关心底层模型差异。

一句话总结：

> **LLM Gateway 就是大模型系统里的统一模型代理层。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **当系统从“调用一个模型”演化到“管理很多模型”时，就需要 LLM Gateway。**

早期 Demo 通常：

~~~python
client.chat.completions.create(...)
~~~

直接调模型。

但真正线上系统会遇到：

- 多模型；
- 多供应商；
- 多业务；
- 多环境；
- 多权限；
- 成本控制；
- 稳定性要求。

这时：

> **模型调用不能散落在业务代码里。**

而应该统一收口到：

~~~text
LLM Gateway
~~~

---

## 三、为什么直接调用模型会越来越乱？

假设项目一开始只有：

~~~text
OpenAI GPT-4
~~~

业务代码可能这样写：

~~~python
response = openai.chat.completions.create(...)
~~~

看起来没问题。

但后来：

~~~text
有些任务想用 Claude
有些任务想用 DeepSeek
有些任务想用本地 Qwen
有些任务想降成本
有些任务想提高速度
~~~

业务代码会变成：

~~~python
if task == "code":
    use claude

elif task == "summary":
    use gpt4

elif task == "cheap":
    use deepseek
~~~

再加上：

- retry；
- timeout；
- fallback；
- token 统计；
- 日志；
- 限流；
- cache；

代码会越来越混乱。

所以：

> **模型调用逻辑不能散落在业务代码里。**

---

## 四、LLM Gateway 的核心思想

LLM Gateway 的核心思想是：

> **把“调用模型”抽象成统一服务。**

业务系统只管：

~~~text
我要一个回答
~~~

至于：

~~~text
用哪个模型
是否 fallback
是否缓存
是否重试
是否限流
如何统计 token
~~~

全部交给 Gateway。

架构会变成：

~~~text
业务代码
    ↓
LLM Gateway
    ↓
不同模型 Provider
~~~

---

## 五、LLM Gateway 的核心能力

### 1. 统一模型接口

不同模型 API 格式不同：

~~~text
OpenAI API
Anthropic API
Gemini API
本地 vLLM API
~~~

参数和响应格式都不一样。

Gateway 可以统一：

~~~text
/chat/completions
~~~

业务代码不用关心底层差异。

---

### 2. 模型路由

不同任务走不同模型：

~~~text
代码生成 → Claude
长文本总结 → GPT-4
低成本任务 → DeepSeek
本地隐私任务 → 本地 Qwen
~~~

Gateway 负责自动路由。

---

### 3. fallback

例如：

~~~text
GPT-4 超时
    ↓
自动切 Claude
~~~

或者：

~~~text
主模型失败
    ↓
切备用模型
~~~

提高稳定性。

---

### 4. 负载均衡

如果有：

~~~text
多个 vLLM 实例
多个 GPU 服务
多个 Region
~~~

Gateway 可以做：

- 轮询；
- 权重；
- 最小连接数；
- 健康检查。

---

### 5. Token 和成本统计

Gateway 可以统一统计：

~~~text
prompt token
completion token
总费用
每个用户消耗
每个业务消耗
~~~

这在线上非常重要。

---

### 6. 限流和权限控制

例如：

~~~text
普通用户：
每天 100 次

VIP：
每天 10000 次
~~~

或者：

~~~text
某业务只能调便宜模型
某业务才能调 GPT-4
~~~

这些都适合在 Gateway 做。

---

### 7. Prompt 管理

很多公司会把：

~~~text
System Prompt
模板
few-shot 示例
```

统一托管在 Gateway。

这样：

~~~text
修改 Prompt
不用重新发版业务代码
~~~

---

### 8. Cache

很多问题是重复的：

~~~text
公司介绍
FAQ
固定文档总结
~~~

Gateway 可以做：

- prompt cache；
- semantic cache；
- response cache。

降低成本。

---

## 六、LLM Gateway 很像 API Gateway

可以类比传统微服务：

### 没有 Gateway

~~~text
业务系统直接调每个服务
~~~

### 有 Gateway

~~~text
业务系统
    ↓
API Gateway
    ↓
各种服务
~~~

LLM Gateway 也是：

~~~text
业务系统
    ↓
LLM Gateway
    ↓
各种模型 Provider
~~~

所以它本质上是：

> **模型层的 API Gateway。**

## 八、为什么 Agent 系统更需要 Gateway？

Agent 系统通常会：

- 多轮调用；
- 调很多模型；
- 有推理模型；
- 有 embedding 模型；
- 有 rerank 模型；
- 有小模型；
- 有本地模型。

例如：

~~~text
Planning 用 Claude
RAG Summary 用 GPT-4
Embedding 用 BGE
Rerank 用 bge-reranker
工具选择用小模型
~~~

如果没有 Gateway：

~~~text
业务代码到处写模型逻辑
~~~

会非常混乱。

所以：

> **Agent 系统几乎天然需要 LLM Gateway。**

## 十一、结合 Agent 项目怎么说

如果面试官问：

> **你的 Agent 项目里会不会考虑 LLM Gateway？**

可以这样回答：

如果只是 Demo，直接调模型 API 就够了。

但如果是正式 Agent 系统，我会考虑加一层 LLM Gateway，因为 Agent 通常会涉及很多模型：

~~~text
主推理模型
embedding 模型
rerank 模型
小模型
本地模型
云模型
~~~

而且 Agent 的调用链很长：

~~~text
Planning
Tool Selection
RAG Summary
Memory Extraction
Reflection
~~~

不同阶段可能适合不同模型。

所以我会把：

~~~text
模型路由
fallback
retry
token统计
日志
限流
cache
~~~

统一放到 Gateway，而不是散落在业务代码里。

这样：

- 模型切换更方便；
- 成本更可控；
- 稳定性更好；
- 后续扩展更多模型也更容易。

## 十二、面试官可能继续追问

### 1. LLM Gateway 和 API Gateway 有什么区别？

本质思想类似。

区别是：

~~~text
API Gateway 管微服务
LLM Gateway 管模型服务
~~~

LLM Gateway 还会额外处理：

- token；
- prompt；
- model routing；
- fallback；
- semantic cache。

---

### 2. 为什么不能直接在业务代码里写模型调用？

因为：

- 多模型会导致逻辑散乱；
- retry/fallback 重复；
- token 统计困难；
- 模型切换困难；
- Prompt 管理困难。

所以应该统一抽象。

---

### 3. fallback 为什么重要？

因为：

~~~text
模型超时
限流
provider 故障
GPU OOM
~~~

都可能发生。

Gateway 可以自动切备用模型，提高可用性。

---

### 4. Agent 为什么更需要 Gateway？

因为 Agent 会：

- 多轮调用；
- 调不同模型；
- 调 embedding/rerank/LLM；
- 调本地和云模型。

复杂度远高于普通 Chat。

---

### 5. Gateway 能做缓存吗？

可以。

例如：

~~~text
prompt cache
semantic cache
response cache
~~~

减少重复调用，降低成本。

---
