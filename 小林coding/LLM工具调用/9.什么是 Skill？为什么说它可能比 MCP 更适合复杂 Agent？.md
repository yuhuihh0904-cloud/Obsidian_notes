
## 一、面试标准回答

如果面试官问：**什么是 Skill？它和 MCP 有什么关系？**

可以这样回答：

Skill 可以理解成：

> **比 Tool 更高层的一种“可复用任务能力”。**

传统 Function Calling 或 MCP Tool 更偏向：

```text
执行一个原子操作
```

例如：

```text
读文件
查数据库
调用 API
发送请求
```

而 Skill 更像：

> **把“完整任务流程”封装成一个能力模块。**

例如：

```text
代码审查 Skill
需求分析 Skill
SQL 优化 Skill
Bug 排查 Skill
生成周报 Skill
```

一个 Skill 内部可能包含：

```text
Prompt
推理逻辑
工具调用
工作流
记忆
子任务拆分
结果总结
```

也就是说：

```text
Tool 是原子能力
Skill 是任务级能力
```

Anthropic 最早提出 Skill 的核心原因，是因为：

> **单纯 MCP + Tool Calling 在复杂推理 Agent 场景下不够用。**

尤其是推理模型会遇到：

```text
thinking 被打断
工具调用太碎
上下文越来越长
状态难维护
```

所以后来开始往：

```text
更高层抽象
更长生命周期能力
更完整任务封装
```

这个方向发展。

一句话总结：

> **MCP 更像“工具协议”，Skill 更像“可复用智能能力模块”。**

---

## 二、为什么 Tool 不够用了？

早期 Agent 的思路很简单：

```text
LLM + Tool Calling
```

模型决定：

```text
调用哪个工具
传什么参数
拿到结果后继续思考
```

这个模式在简单任务下很好用。

例如：

```text
查天气
读文件
搜网页
算数学
```

但复杂任务会出现问题。

比如：

```text
分析一个大型代码仓库
完成复杂需求开发
做长链路调试
自动生成完整报告
```

这类任务往往需要：

```text
多轮推理
多个工具协作
中间状态管理
长期上下文
复杂 Prompt
工作流控制
```

如果仍然只靠：

```text
LLM 一步一步临时决定调用 Tool
```

就会导致：

```text
推理链很碎
上下文越来越长
错误容易累积
Agent 不稳定
```

---

## 三、Tool 的本质问题

Tool 的本质是：

> **一次原子操作。**

例如：

```text
read_file()
search_web()
query_sql()
```

Tool 并不知道：

```text
当前整体任务是什么
前面已经完成了什么
后面还需要做什么
```

所以 Tool 天然缺少：

```text
任务级状态
长期上下文
复杂工作流
推理结构
```

这意味着：

> **所有复杂任务逻辑都被迫塞进 LLM 上下文。**

而这正是很多 Agent 不稳定的根源。

---

## 四、Skill 是什么？

Skill 本质上是：

> **把“完成某类任务的方法”封装起来。**

它不是单纯工具，而是：

```text
Prompt
工具调用
工作流
记忆
推理策略
```

的组合。

例如一个“代码审查 Skill”可能内部包含：

```text
1. 读取代码文件
2. 分析依赖关系
3. 调用静态分析工具
4. 用固定 Prompt 做代码审查
5. 生成结构化报告
```

对于上层 Agent 来说：

```text
不用关心里面具体调用了哪些 Tool
只需要调用：
run_code_review_skill()
```

这就是 Skill 和 Tool 最大的区别：

```text
Tool：原子操作
Skill：完整任务能力
```

---

## 五、为什么 Skill 更适合复杂 Agent？

因为复杂 Agent 最大的问题不是：

```text
缺工具
```

而是：

```text
缺稳定的任务执行结构
```

Skill 可以把：

```text
Prompt
推理流程
工具调用顺序
状态管理
错误恢复
```

封装起来。

这样：

```text
LLM 不需要每一步都重新规划
```

而是：

```text
调用一个已经验证过的任务能力模块
```

这会带来几个好处。

---

## 六、Skill 的几个核心优势

### 1. 降低上下文压力

如果所有步骤都让 LLM 临时规划，那么上下文会越来越长。

Skill 可以把内部逻辑封装掉。

上层只需要：

```text
调用 code_review_skill
```

而不是把：

```text
读文件
分析依赖
静态检查
总结结果
```

全部显式写进上下文。

---

### 2. 提高稳定性

Tool Calling 很容易：

```text
一步错
后面全错
```

因为每一步都依赖 LLM 临时决策。

Skill 则更像：

```text
已经验证好的工作流模板
```

稳定性更高。

---

### 3. 提高复用性

Skill 天然适合：

```text
团队沉淀经验
跨项目复用
```

例如：

```text
SQL 优化 Skill
代码审查 Skill
需求分析 Skill
```

这些能力可以长期积累。

---

### 4. 更适合推理模型

推理模型最大的问题是：

```text
thinking 不适合频繁中断
```

而 Tool Calling 会频繁：

```text
停下来
等工具
继续思考
```

Skill 可以把：

```text
多步工具调用
```

封装成：

```text
一次高层任务调用
```

从而减少推理链碎片化。

---

## 七、Skill 和 MCP 的关系

很多人误以为：

```text
Skill 会替代 MCP
```

其实不是。

它们抽象层次不同。

---

### MCP 解决什么？

MCP 解决的是：

```text
工具怎么标准化接入
```

例如：

```text
GitHub Tool
文件系统 Tool
数据库 Tool
浏览器 Tool
```

MCP 更像：

> **工具基础设施层。**

---

### Skill 解决什么？

Skill 解决的是：

```text
复杂任务怎么稳定执行
```

它更像：

> **任务能力层。**

---

## 八、可以把 Skill 理解成“Tool 的组合体”

例如：

```text
code_review_skill
```

内部可能调用：

```text
read_file Tool
grep Tool
git_diff Tool
llm_summary Tool
```

甚至还可能使用：

```text
Prompt 模板
长期记忆
工作流状态
错误恢复逻辑
```

所以：

```text
MCP 提供 Tool
Skill 组织 Tool
```

关系类似：

```text
MCP：零件标准
Skill：完整机器
```

---

## 九、为什么说 Skill 正在走向开放标准？

最开始 Skill 更像 Anthropic 内部概念。

但后来大家发现：

> **复杂 Agent 都会逐渐走向“能力模块化”。**

因为纯：

```text
LLM + Tool Calling
```

很难稳定支撑复杂任务。

所以很多框架都开始出现类似思想：

```text
LangGraph：节点工作流
OpenAI Agents：高层 Agent 抽象
AutoGen：Agent 协作
CrewAI：角色化任务能力
```

虽然名字不同，但核心方向一致：

> **把复杂任务封装成可复用能力模块。**

这其实就是 Skill 思想。

---

## 十、Skill 和传统 Workflow 的区别

很多人会问：

```text
Skill 不就是 Workflow？
```

它们确实很像，但 Skill 更强调：

```text
让 LLM 主导
```

传统 Workflow 通常是：

```text
固定 DAG
固定步骤
固定规则
```

而 Skill 更像：

```text
半结构化任务能力
```

它可能内部仍然允许：

```text
动态推理
动态工具选择
动态任务拆分
```

只是：

```text
把高层能力边界封装好了
```

所以 Skill 是：

```text
Workflow + LLM 推理能力
```

的结合。

---

## 十一、未来 Agent 的可能方向

文章最后其实在强调一个趋势：

未来 Agent 很可能会从：

```text
纯 Tool Calling
```

逐渐走向：

```text
Skill-based Agent
```

也就是：

```text
底层：MCP / Tool
中层：Skill
上层：Agent / Planner
```

形成分层架构。

原因是：

```text
Tool 太底层
复杂任务太容易碎片化
LLM 上下文压力太大
推理链太不稳定
```

所以需要：

```text
更高层任务抽象
```

来提高稳定性和复用性。

---

## 十二、容易踩的坑

### 1. 误以为 Skill = Tool

错误理解：

```text
Skill 就是高级 Tool
```

正确理解：

```text
Tool 是原子操作
Skill 是完整任务能力
```

---

### 2. 误以为 Skill 替代 MCP

错误理解：

```text
有 Skill 就不需要 MCP
```

正确理解：

```text
MCP 提供底层工具能力
Skill 组织这些工具完成复杂任务
```

---

### 3. 误以为 Skill 就是固定 Workflow

错误理解：

```text
Skill 就是写死流程
```

正确理解：

```text
Skill 内部仍然可能包含动态推理和动态工具选择
```

---

## 十三、记忆版总结

Tool 的问题是：

```text
太底层
缺任务状态
缺长期上下文
复杂任务容易碎片化
```

Skill 的核心思想是：

> **把完成某类任务的方法整体封装起来。**

Skill 内部可能包含：

```text
Prompt
Tool Calling
Workflow
Memory
Reasoning
错误恢复
```

所以：

```text
Tool：原子操作
Skill：任务级能力
```

MCP 和 Skill 的关系：

```text
MCP：工具基础设施层
Skill：任务能力层
```

未来 Agent 可能逐渐形成：

```text
底层：MCP / Tool
中层：Skill
上层：Agent / Planner
```

最终一句话：

> **MCP 解决“工具怎么接”，Skill 解决“复杂任务怎么稳定完成”。**
