
## 一、面试标准回答

面试官问：**MCP 和 Agent Skill 有什么区别？**

可以这样回答：

MCP 和 Agent Skill 不是同一类东西，也不是竞争关系，而是互补关系。

**MCP 解决的是 Agent 怎么获得外部能力。**  
它把数据库、API、文件系统、浏览器、GitHub 等外部能力封装成标准化的 MCP Server，Agent 连接后就能调用这些工具。简单说，MCP 是让 Agent 有“手”，能真正操作外部世界。

**Skill 解决的是 Agent 拿到能力之后，该怎么完成某类任务。**  
它不是简单的 prompt 模板，而是一个完整能力包，通常包含 `SKILL.md`、脚本、模板、参考文档等。Skill 更像 SOP，告诉 Agent 做任务时应该按什么步骤、什么标准、什么格式来完成。

所以两者的区别可以概括为：

> **MCP 提供工具能力，Skill 提供任务流程。**

举个例子，如果用户让 Agent 做代码审查：

- MCP 提供 `read_file`、`search_code`、`run_command` 这类具体工具；
- Skill 告诉 Agent：先读取代码，再检查安全问题，再检查性能问题，最后按模板输出审查报告。

所以它们的配合方式是：

> **Skill 负责规划和编排流程，MCP 负责执行流程里的具体动作。**

一句话总结：

> **MCP 是 Agent 的工具箱，Skill 是 Agent 的操作手册。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **MCP 和 Skill 都是在增强 Agent，但增强的是不同层次。**

可以这样记：

~~~text
MCP：让 Agent 能做事
Skill：让 Agent 知道怎么做事
~~~

MCP 偏向工具层，解决外部工具和数据访问问题。

Skill 偏向流程层，解决复杂任务应该怎么执行的问题。

所以二者不是替代关系，而是协作关系。

---

## 三、MCP：让 Agent 有“手”

MCP 的核心作用是：

> **把外部工具标准化暴露给 Agent。**

比如：

- 文件系统；
- 数据库；
- 搜索引擎；
- GitHub；
- 浏览器；
- 企业内部 API；
- 向量数据库。

MCP Server 暴露出来的工具通常是**原子操作**：

~~~text
read_file(path)
query_database(sql)
search_web(query)
create_issue(title, body)
~~~

每个工具只做一件明确的事。

所以 MCP 的特点是：

- 粒度小；
- 偏执行；
- 工具有明确 schema；
- 负责连接外部系统；
- 解决“Agent 能不能做”的问题。

---

## 四、Skill：给 Agent 一份“操作手册”

Skill 的核心作用是：

> **告诉 Agent 完成某类任务时应该怎么做。**

Skill 不只是 prompt，它通常是一个文件夹，里面可能包含：

~~~text
SKILL.md
scripts/
assets/
templates/
reference_docs/
~~~

其中最核心的是 `SKILL.md`，它会描述：

- Skill 名字；
- Skill 适用场景；
- 执行步骤；
- 判断标准；
- 输出格式；
- 需要用到的脚本或模板。

比如一个代码审查 Skill 可能会写：

~~~text
第一步：读取待审查代码
第二步：理解代码改动范围
第三步：检查安全问题
第四步：检查性能问题
第五步：按照报告模板输出审查结果
~~~

所以 Skill 的特点是：

- 粒度大；
- 偏流程；
- 可以包含脚本、模板、文档；
- 负责指导 Agent 怎么完成任务；
- 解决“Agent 应该怎么做”的问题。

---

## 五、Skill 和普通 Prompt 的区别

很多人会把 Skill 理解成高级 Prompt，但这不准确。

普通 Prompt 通常只是一段指令文本。

Skill 是一个完整的任务能力包。

| 对比项 | 普通 Prompt | Skill |
|---|---|---|
| 形式 | 一段文本 | 一个文件夹 / 能力包 |
| 内容 | 指令 | 指令 + 脚本 + 模板 + 文档 |
| 加载方式 | 手动写入上下文 | Agent 可自动发现、按需加载 |
| 复用性 | 一般 | 更强 |
| 适合场景 | 简单任务约束 | 复杂任务流程 |

可以理解为：

> **Skill 是可复用的任务 SOP，不只是提示词。**

---

## 六、Skill 的渐进式加载

Skill 有一个重要机制：

> **渐进式加载。**

也就是 Agent 不会一开始把所有 Skill 的完整内容都塞进上下文。

一般流程是：

~~~text
启动时：只加载 Skill 名字和简短描述
    ↓
用户任务来了：Agent 判断哪个 Skill 相关
    ↓
匹配到相关 Skill：加载完整 SKILL.md
    ↓
执行中需要脚本 / 模板 / 参考文档：再按需加载
~~~

这样做的好处是：

- 节省上下文窗口；
- 避免无关 Skill 干扰模型；
- 可以管理大量 Skill；
- 只在需要时加载具体流程和资源。

## 七、MCP 和 Skill 的核心区别

| 对比项 | MCP | Skill |
|---|---|---|
| 解决问题 | Agent 怎么获得外部能力 | Agent 该怎么完成任务 |
| 定位 | 工具协议 / 工具服务 | 任务流程 / 能力包 |
| 粒度 | 原子操作 | 完整工作流 |
| 典型形式 | MCP Server 暴露 Tools | Skill 文件夹 + SKILL.md |
| 内容 | 工具 schema、资源、接口 | 指令、步骤、脚本、模板、文档 |
| 作用 | 让 Agent 能做事 | 让 Agent 知道怎么做事 |
| 类比 | 工具箱 / 外部软件 | 操作手册 / SOP |
| 是否竞争 | 否 | 否 |

## 八、两者怎么配合工作？

以代码审查为例。

用户说：

~~~text
帮我审查一下这次提交的代码。
~~~

Agent 的执行过程可能是：

~~~text
1. 判断任务属于代码审查
2. 加载 code-review Skill
3. Skill 告诉 Agent 审查流程
4. Agent 根据流程调用 MCP 工具读取代码
5. Agent 调用 MCP 工具查看 diff
6. Agent 执行 Skill 自带的安全检查脚本
7. Agent 加载 Skill 里的报告模板
8. 输出结构化代码审查报告
~~~

这里的分工是：

~~~text
Skill：定义做什么、按什么顺序做、按什么标准做
MCP：提供 read_file、git_diff、query_database 等具体工具
~~~

所以完整配合关系是：

> **Skill 编排流程，MCP 执行动作。**

## 十、一个类比：员工做任务

MCP 像公司给员工配的：

~~~text
电脑
账号权限
办公软件
数据库权限
内部系统权限
~~~

它让员工有能力做事。

Skill 像公司给员工发的：

~~~text
操作手册
工作流程
审查标准
报告模板
SOP 文档
~~~

它告诉员工怎么把事情做好。

所以：

~~~text
没有 MCP：Agent 没工具，只能空谈
没有 Skill：Agent 有工具，但不知道流程和标准
~~~

一句话：

> **缺 MCP，Agent 做不了；缺 Skill，Agent 做不好。**

## 十一、结合 Agent 项目怎么说

如果面试官问：

> **你的 Agent 项目里怎么理解 MCP 和 Skill 的配合？**

可以这样回答：

在 Agent 项目里，我会把 MCP 理解成工具接入层，把 Skill 理解成任务流程层。

比如在金融分析 Agent 中，MCP 或工具系统可以提供行情查询、新闻搜索、文件读取、向量检索等具体能力。但这些工具本身只是一组原子能力，不能告诉 Agent 应该如何完成一份金融研报。

如果我要设计一个“财报分析 Skill”，它可以规定完整流程：

~~~text
第一步：识别文档类型
第二步：提取公司和财报周期
第三步：检索关键财务指标
第四步：结合历史 Research Memory
第五步：生成结构化分析报告
第六步：输出风险点和后续关注问题
~~~

执行这些步骤时，Agent 会调用具体工具：

~~~text
PDF 解析工具
Milvus RAG 检索工具
Research Memory 检索工具
新闻搜索工具
行情查询工具
~~~

所以我的理解是：

> **MCP / Tool 负责提供能力，Skill 负责组织这些能力完成复杂任务。**

这样 Agent 不只是“有很多工具”，而是“知道如何按流程使用工具完成任务”。

## 十二、面试官可能继续追问

### 1. MCP 和 Skill 是竞争关系吗？

不是。

MCP 解决工具和数据访问，Skill 解决任务流程和执行标准。

两者是上下层协作关系。

---

### 2. Skill 是不是 Prompt 模板？

不只是。

Prompt 模板通常只是一段提示词，而 Skill 是一个完整能力包，可以包含：

~~~text
指令
脚本
模板
参考文档
资源文件
~~~

而且 Skill 可以被 Agent 自动发现和按需加载。

---

### 3. MCP 的粒度为什么比 Skill 小？

因为 MCP Tool 通常是一个原子操作，比如：

~~~text
read_file
query_database
search_web
~~~

每次调用只做一件具体事。

而 Skill 面向的是一个完整任务流程，比如：

~~~text
代码审查
数据分析
报告生成
财报解读
~~~

一个 Skill 内部可能会调用多个 MCP Tool。

---

### 4. 为什么有了 MCP 还需要 Skill？

因为有工具不代表会做复杂任务。

Agent 拿到一堆工具后，仍然需要知道：

- 先做什么；
- 后做什么；
- 什么情况下走分支；
- 用什么标准判断；
- 最后按什么格式输出。

这些就是 Skill 提供的。

---

### 5. 为什么有了 Skill 还需要 MCP？

因为 Skill 只定义流程，不一定提供真实外部能力。

如果流程里写着“读取文件”“查询数据库”“搜索网页”，那具体执行还要靠 MCP Tool 或其他工具系统。

所以：

> **Skill 再详细，没有 MCP 也只是纸上谈兵。**

---

## 十三、记忆版总结

这篇文章可以压缩成一句话：

> **MCP 提供工具能力，Skill 提供任务流程；MCP 让 Agent 能做事，Skill 让 Agent 知道怎么做事。**

进一步记：

~~~text
MCP：
工具层
原子操作
外部能力
让 Agent 有手
典型工具：read_file、query_database、search_web

Skill：
流程层
完整任务
知识和 SOP
让 Agent 有方法
典型任务：代码审查、数据分析、财报分析

~~~

最重要的一句话：

> **缺 MCP，Agent 没有执行能力；缺 Skill，Agent 没有任务方法。**
