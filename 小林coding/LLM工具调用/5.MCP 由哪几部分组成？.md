## 一、面试标准回答

如果面试官问：**MCP 由哪几部分组成？**

可以这样回答：

MCP 不能简单理解成只有 Client 和 Server，它可以从三层来理解：

1. **角色层：Host / Client / Server**
2. **能力层：Tools / Resources / Prompts**
3. **协议层：JSON-RPC 2.0 + 传输方式**

**第一层是角色层。**

MCP 里有三个角色：

- **Host**：宿主应用，比如 Claude Desktop、Cursor、Windsurf；
- **Client**：Host 内部负责和 MCP Server 通信的模块；
- **Server**：工具提供方，实现具体工具、资源和提示词模板。

这里最容易混淆的是 **Host 和 Client**。

Host 是整个 AI 应用本身，负责管理多个 MCP Client；  
Client 是 Host 内部的连接模块，一个 Client 通常对应一个 MCP Server 连接。

所以一个 Host 可以连接多个 Server，例如同时连接文件系统 Server、GitHub Server、数据库 Server。

**第二层是能力层。**

MCP Server 可以暴露三类能力：

| 能力 | 作用 | 特点 |
|---|---|---|
| Tools | 执行操作 | 有副作用，会改变外部状态 |
| Resources | 提供数据 | 只读，无副作用 |
| Prompts | 提供提示词模板 | 用来复用标准化任务模板 |

例如：

- Tools：创建文件、提交代码、调用 API；
- Resources：读取日志、读取文档、查询数据库记录；
- Prompts：代码审查模板、SQL 优化模板、需求分析模板。

简单记忆：

```text
Tools：改变世界
Resources：观察世界
Prompts：结构化表达
```

**第三层是协议层。**

MCP 的消息格式统一使用 **JSON-RPC 2.0**，用 JSON 表示：

- 调用哪个方法；
- 参数是什么；
- 请求 ID 是多少；
- 返回结果或错误信息。

传输方式主要有两种：

- **stdio**：适合本地 MCP Server，通过标准输入输出通信；
- **Streamable HTTP**：适合远程 MCP Server，通过 HTTP 连接通信。

所以总结一句话：

> **MCP 的完整组成是：角色层 Host / Client / Server，能力层 Tools / Resources / Prompts，协议层 JSON-RPC 2.0 + stdio / Streamable HTTP。**

## 二、为什么不能只说 MCP = Client + Server？

很多人会把 MCP 理解成：

```text
Client 发请求
Server 返回结果
```

这不算错，但不完整。

因为 MCP 里除了 Client 和 Server，还有一个很重要的角色：

> **Host。**

Host 是真正的 AI 应用本身，例如：

```text
Claude Desktop
Cursor
Windsurf
Agent 框架
```

Client 不是一个完整应用，而是 Host 里面负责通信的模块。

正确关系是：

```text
Host 管理多个 Client
一个 Client 连接一个 Server
Server 暴露工具、资源和提示词模板
```

---

## 三、角色层：Host / Client / Server

### 1. Host：宿主应用

Host 是用户实际使用的 AI 应用。

它负责：

- 管理 MCP Client；
- 决定连接哪些 MCP Server；
- 控制连接生命周期；
- 把 Server 暴露的能力组织给模型使用。

可以理解为：

> **Host 是整个 MCP 系统的调度中心。**

---

### 2. Client：通信模块

Client 是 Host 内部的连接模块。

它主要负责：

- 和某个 MCP Server 建立连接；
- 查询 Server 暴露了哪些能力；
- 把模型的工具调用请求转发给 Server；
- 把 Server 的执行结果返回给 Host / LLM。

一个 Client 通常对应一个 Server。

例如：

```text
Claude Desktop
  ├── GitHub MCP Client → GitHub MCP Server
  ├── Filesystem MCP Client → Filesystem MCP Server
  └── PostgreSQL MCP Client → PostgreSQL MCP Server
```

---

### 3. Server：能力提供方

Server 是工具提供方实现的独立进程或服务。

它负责暴露：

```text
Tools
Resources
Prompts
```

Server 不需要关心上层到底是 Claude Desktop、Cursor，还是其他 Agent 框架。

只要它遵守 MCP 协议，任何支持 MCP 的 Host 都可以使用它。

这也是 MCP 的核心价值：

> **Server 写一次，多端复用。**

---

## 四、能力层：Tools / Resources / Prompts

MCP Server 暴露的不只是工具，而是三类能力。

---

## 五、Tools：执行操作

Tools 是最接近 Function Calling 的部分。

它表示模型可以调用的具体操作。

例如：

```text
创建文件
修改文件
提交代码
发送消息
调用第三方 API
修改数据库
```

Tools 的核心特点是：

> **有副作用，会改变外部世界状态。**

所以 Tools 通常需要更谨慎的权限控制和用户确认。

例如删除文件、发送邮件、提交代码这类操作，不能让模型随便执行。

---

## 六、Resources：读取数据

Resources 是只读数据。

例如：

```text
读取日志文件
读取数据库记录
读取文档内容
查看仓库文件
读取配置文件
```

Resources 的核心特点是：

> **只读，无副作用。**

它只是把信息提供给模型看，不会改变外部状态。

Resources 可以理解成：

> **模型可访问的资料室。**

---

## 七、Prompts：提示词模板

Prompts 是预定义的提示词模板。

它解决的是：

> **重复 Prompt 的标准化和复用问题。**

例如团队内部有一个固定的代码审查模板：

```text
请你作为资深工程师审查以下 {language} 代码：
1. 是否有逻辑 bug
2. 是否有性能问题
3. 是否有安全风险
4. 是否符合团队规范

代码：
{code}
```

这个模板可以放在 MCP Server 中。

当 Client 需要使用时，只要传入参数：

```text
language = Python
code = 具体代码
```

MCP Server 就返回填充后的完整 Prompt，Client 再把它放进 LLM 上下文中使用。

Prompts 的价值是：

```text
沉淀团队最佳实践
统一任务处理标准
减少重复手写 Prompt
让多个 MCP Client 复用同一套模板
```

---

## 八、Tools、Resources、Prompts 的区别

| 能力 | 本质 | 是否有副作用 | 例子 |
|---|---|---|---|
| Tools | 执行操作 | 有 | 创建文件、提交代码、调用 API |
| Resources | 读取资料 | 无 | 读取日志、查询文档、查看数据库 |
| Prompts | 提示词模板 | 无 | 代码审查模板、SQL 优化模板 |

最简单的记忆方式：

```text
Tools：改变世界
Resources：观察世界
Prompts：结构化表达
```

---

## 九、协议层：JSON-RPC 2.0

MCP 的消息格式统一使用 **JSON-RPC 2.0**。

也就是说，Client 和 Server 通信时，消息都是标准 JSON 结构。

例如 Client 查询工具列表：

~~~json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}
~~~

Server 返回工具列表：

~~~json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "read_file"
      }
    ]
  }
}
~~~

Client 调用工具：

~~~json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": {
      "path": "/tmp/log.txt"
    }
  }
}
~~~

JSON-RPC 的好处是：

```text
格式简单
容易调试
语言无关
请求和响应可以通过 id 匹配
```

---

## 十、传输方式：stdio 和 Streamable HTTP

JSON-RPC 规定的是消息格式。

但消息具体怎么传，还需要传输方式。

MCP 常见传输方式有两种：

```text
stdio
Streamable HTTP
```

---

### 1. stdio

stdio 是标准输入输出。

这种方式下，MCP Server 作为本地子进程运行。

Client 通过：

```text
stdin 发送请求
stdout 接收响应
```

适合本地工具，例如：

```text
本地文件系统
本地 Git 工具
本地脚本
```

优点：

```text
不需要网络
延迟低
启动简单
没有端口占用问题
```

---

### 2. Streamable HTTP

Streamable HTTP 适合远程 MCP Server。

这种方式下，Server 作为 HTTP 服务部署，Client 通过 HTTP 访问。

适合：

```text
远程数据库工具
团队共享工具服务
服务器上的业务系统工具
```

它的特点是：

```text
使用单个 /mcp 端点通信
短请求可以直接返回 JSON
长请求可以通过 SSE 流式返回中间结果
```

早期 MCP 的远程通信方案是 **HTTP + SSE 双端点**，后来演进为 **Streamable HTTP 单端点**。

单端点方案更适合：

```text
负载均衡
Serverless 部署
远程服务共享
```

---

## 十一、消息格式和传输方式是解耦的

MCP 一个很重要的设计是：

> **JSON-RPC 2.0 负责消息长什么样，stdio / Streamable HTTP 负责消息怎么传。**

二者是解耦的。

也就是说，同样一条 JSON-RPC 消息，既可以通过 stdio 传，也可以通过 HTTP 传。

这样 MCP Server 可以灵活部署：

```text
本地工具 → stdio
远程服务 → Streamable HTTP
```

但上层工具调用逻辑不需要变化。

这也是 MCP 灵活性的来源。

## 十二、完整流程理解

一个 MCP 工具调用流程可以理解为：

```text
1. Host 启动 MCP Client
2. Client 连接 MCP Server
3. Client 查询 Server 暴露的 Tools / Resources / Prompts
4. Host 把可用能力组织进 LLM 上下文
5. LLM 判断是否需要调用工具
6. Client 把调用请求转发给 Server
7. Server 执行工具或返回资源 / Prompt
8. Client 把结果返回给 Host
9. Host 再把结果交给 LLM 继续生成回答
```

注意：

> **Server 只负责提供能力，真正组织调用流程的是 Host / Client。**

## 十三、面试容易踩的坑

### 1. 把 Host 和 Client 混为一谈

错误理解：

```text
Host 就是 Client
```

正确理解：

```text
Host 是 AI 应用本身
Client 是 Host 内部负责通信的模块
```

---

### 2. 认为 Server 只暴露工具

错误理解：

```text
MCP Server 只提供 Tools
```

正确理解：

```text
MCP Server 可以暴露 Tools、Resources、Prompts 三类能力
```

---

### 3. 混淆 Tools 和 Resources

错误理解：

```text
读文件、改文件都算工具
```

正确理解：

```text
读取数据：Resources，无副作用
修改数据：Tools，有副作用
```

---

### 4. 忽略协议层

MCP 不只是角色架构，还包括通信协议。

需要说清楚：

```text
消息格式：JSON-RPC 2.0
传输方式：stdio / Streamable HTTP
```

---

## 十四、记忆版总结

MCP 由三层组成：

```text
角色层：Host / Client / Server
能力层：Tools / Resources / Prompts
协议层：JSON-RPC 2.0 + stdio / Streamable HTTP
```

角色层：

```text
Host：AI 宿主应用，例如 Claude Desktop、Cursor
Client：Host 内部负责连接 Server 的模块
Server：工具、资源、提示词模板的提供方
```

能力层：

```text
Tools：有副作用，改变外部状态
Resources：只读数据，无副作用
Prompts：提示词模板，用于复用标准任务流程
```

协议层：

```text
JSON-RPC 2.0：定义消息格式
stdio：适合本地 Server
Streamable HTTP：适合远程 Server
```

最重要的一句话：

> **MCP 不是简单的 Client-Server，而是由角色架构、能力类型、通信协议共同组成的一套 AI 工具接入标准。**
