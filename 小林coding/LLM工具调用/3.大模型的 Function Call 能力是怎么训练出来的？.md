# 大模型的 Function Call 能力是怎么训练出来的？

## 一、面试标准回答

如果面试官问：**大模型的 Function Call 能力是怎么训练出来的？**

可以这样回答：

Function Call 能力不是模型在预训练阶段自然学会的，预训练主要让模型学习语言和代码中的统计规律，本质目标是预测下一个 token。模型可能知道“我需要调用 API”，但不一定能稳定输出程序可解析的结构化工具调用格式。

真正让模型具备 Function Call 能力，主要依赖两个阶段：

1. **SFT：教模型怎么调用工具**
2. **RLHF / RLAIF：教模型什么时候该调用工具**

**SFT 阶段**会构造大量工具调用样本，例如：

- system 中给出工具定义和参数 schema；
- user 提出问题；
- assistant 输出结构化工具调用；
- tool 返回执行结果；
- assistant 根据工具结果生成最终回答。

通过这些样本，模型学会：**看到工具定义和用户问题后，应该选择哪个工具、填写什么参数、输出什么格式。**

但 SFT 主要解决“会不会调”的问题，可能导致模型过度调用工具。

所以还需要 **RLHF**。  
RLHF 通过人类偏好数据训练奖励模型，再优化主模型，让模型学会判断：

- 哪些问题必须调用工具；
- 哪些问题应该直接回答；
- 哪些工具调用是多余的；
- 哪些工具调用参数是错误的。

一句话总结：

> **SFT 教会模型怎么调工具，RLHF 教会模型什么时候该调工具。**

---

## 二、为什么预训练不能直接学会 Function Call？

预训练的目标是：

> **根据上下文预测下一个 token。**

模型可能在预训练语料中见过 API 文档、代码、JSON 等内容，所以它可能知道工具调用的大致形式。

但 Function Call 要求模型稳定输出类似下面这种可被程序解析的结构化内容：

~~~json
{
  "tool_calls": [
    {
      "name": "get_weather",
      "arguments": {
        "city": "北京"
      }
    }
  ]
}
~~~

这不是普通自然语言，而是严格格式化的调用请求。

预训练语料里通常没有大量标准化的：

```text
工具定义 + 用户问题 + 正确工具调用 + 工具返回结果 + 最终回答
```

所以 Function Call 能力需要后续专项训练。

---

## 三、SFT：让模型学会“怎么调”

SFT 的核心是：

> **用大量正确示例让模型模仿工具调用流程。**

一条典型训练样本包括：

```text
system：工具定义、工具名称、参数 schema
user：用户问题
assistant：结构化工具调用
tool：工具执行结果
assistant：最终自然语言回答
```

例如：

```text
用户：北京今天天气怎么样？
模型：调用 get_weather(city="北京")
工具：返回天气信息
模型：根据工具结果回答用户
```

SFT 主要让模型学会三件事：

1. **选择正确工具**
2. **生成正确参数**
3. **输出符合协议的结构化格式**

---

## 四、SFT 数据需要覆盖哪些场景？

Function Call 训练数据不能只覆盖简单调用，还要覆盖真实使用中的复杂情况。

### 1. 单工具调用

最基础场景。

```text
用户：北京天气怎么样？
模型：调用 get_weather(city="北京")
```

---

### 2. 多工具并行调用

用户一次询问多个对象时，模型需要能并行调用多个工具。

```text
用户：帮我查北京和上海今天的天气。
模型：同时调用北京天气工具和上海天气工具
```

---

### 3. 工具调用失败处理

真实环境中工具可能失败，例如：

- API 超时；
- 参数错误；
- 权限不足；
- 返回空结果。

模型需要学会根据工具返回的错误信息调整策略，而不是一直重复错误调用。

---

### 4. 不需要工具时直接回答

这是非常关键的一类数据。

```text
用户：1 + 1 等于几？
模型：2
```

如果训练数据全是“调用工具”的正例，模型容易形成错误习惯：

> **遇到任何问题都调用工具。**

---

### 5. 多轮对话中的工具调用

模型还需要理解上下文。

```text
用户：北京天气怎么样？
模型：调用天气工具并回答

用户：那上海呢？
模型：理解用户仍然在问天气，然后调用上海天气工具
```

---

## 五、训练数据来源

Function Call 训练数据主要有两类来源。

### 1. 人工标注

优点：

- 质量高；
- 格式稳定；
- 适合构造核心种子数据。

缺点：

- 成本高；
- 规模难以扩大。

---

### 2. 模型自动生成

也就是用更强的大模型批量生成工具调用样本，常见方法包括：

- **Self-Instruct**
- **Distillation**

优点：

- 成本低；
- 规模大；
- 扩展快。

缺点：

- 强模型生成的错误会被弱模型学进去；
- 容易出现“幻觉传递”。

所以自动生成数据后，仍然需要人工抽查和质量过滤。

---

## 六、RLHF：让模型学会“该不该调”

SFT 后，模型知道了工具调用格式，但它不一定知道：

> **什么时候应该调用工具，什么时候不应该调用工具。**

例如：

```text
北京今天的天气怎么样？
```

需要实时信息，应该调用天气工具。

```text
1 + 1 等于几？
```

不需要工具，直接回答即可。

RLHF 解决的是工具调用的边界问题。

---

## 七、RLHF 的基本流程

RLHF 大致分为四步：

### 1. 生成多个候选回答

对同一个问题，让模型生成多个不同回答，例如：

- 直接回答；
- 调用正确工具；
- 调用错误工具；
- 工具参数填错；
- 调用工具后正确总结。

---

### 2. 人类标注偏好

标注员判断哪个回答更好。

例如：

```text
问题：北京今天的天气怎么样？
更好答案：调用天气工具后回答
```

```text
问题：1 + 1 等于几？
更好答案：直接回答 2
```

---

### 3. 训练奖励模型

奖励模型学习人类偏好。

它不直接回答问题，而是给模型输出打分，相当于一个“裁判”。

---

### 4. 用 PPO 等方法优化主模型

主模型根据奖励模型的分数进行调整。

目标是让模型更倾向于输出人类认为更好的回答。

PPO 中通常会加入 KL 约束，防止模型为了追求奖励而偏离原有语言能力。

---

## 八、RLAIF：用 AI 反馈代替人类反馈

RLHF 的人工标注成本很高，所以也可以使用 **RLAIF**：

> **Reinforcement Learning from AI Feedback**

也就是让更强的 AI 模型代替人类进行偏好打分。

优点：

- 成本低；
- 速度快；
- 可以大规模生成偏好数据。

缺点：

- AI 评委本身也可能有偏见；
- 如果评分标准设计不好，会把错误偏好教给模型。

实际工程中常见做法是：

> **核心数据用人工保证质量，大规模数据用 AI 提高效率。**

---

## 九、SFT 和 RLHF 的区别

| 阶段 | 解决问题 | 核心作用 |
|---|---|---|
| SFT | 会不会调 | 学会工具调用格式和流程 |
| RLHF / RLAIF | 该不该调 | 学会工具调用边界和偏好 |

简单理解：

```text
SFT：教模型怎么调用工具
RLHF：教模型什么时候调用工具
```

只有 SFT，模型可能乱调工具。  
只有 RLHF，模型可能连工具调用格式都不会稳定输出。

所以 Function Call 能力通常需要二者配合。

---

## 十、记忆版总结

Function Call 能力不是预训练自然学会的，而是靠后续专项训练得到的。

核心分两个阶段：

```text
SFT：通过工具调用样本，让模型学会输出结构化调用请求。
RLHF：通过偏好反馈，让模型学会判断什么时候该调工具。
```

训练数据需要覆盖：

```text
单工具调用
多工具并行调用
工具失败处理
无需工具直接回答
多轮对话工具调用
```

数据来源主要有：

```text
人工标注：质量高，但成本高
模型生成：规模大，但要防止错误和幻觉传递
```

最终一句话：

> **SFT 教会模型怎么调，RLHF 教会模型什么时候调。**
