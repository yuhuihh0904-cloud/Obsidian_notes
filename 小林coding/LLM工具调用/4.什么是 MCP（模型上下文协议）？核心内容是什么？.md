## 一、面试标准回答

如果面试官问：**什么是 MCP？它的核心内容是什么？**

可以这样回答：

MCP，全称是 **Model Context Protocol，模型上下文协议**，它是 Anthropic 提出的一个开放协议，主要解决的是 **AI 工具接入碎片化** 的问题。

在 MCP 出现之前，如果想让大模型调用 GitHub、数据库、文件系统、浏览器等工具，通常需要为每个工具、每个客户端单独写适配代码。这样会导致工具接入成本高、复用性差，并且和具体模型或应用强绑定。

MCP 的核心思路是：

> **把工具接入标准化，让工具实现一次，就可以被多个支持 MCP 的 AI 客户端复用。**

它采用 **Client-Server 架构**：

- **MCP Server**：工具提供方，负责封装具体工具能力；
- **MCP Client**：AI 应用侧，比如 Claude Desktop、Cursor、Agent 框架等；
- 一个 Client 可以连接多个 Server，从而获得多种工具能力。

MCP Server 可以暴露三类核心能力：

1. **Tools**：工具操作，通常有副作用，比如创建文件、提交代码、调用 API；
2. **Resources**：只读资源，比如读取文件、查询日志、读取数据库记录；
3. **Prompts**：提示词模板，用来复用团队沉淀的标准 prompt。

底层通信上，MCP 使用 **JSON-RPC 2.0**，传输层常见两种方式：

- **stdio**：适合本地工具，Client 通过标准输入输出和 Server 通信；
- **Streamable HTTP**：适合远程工具服务，一个 HTTP 端点即可支持请求和流式返回。

一句话总结：

> **Function Calling 解决的是模型怎么输出工具调用请求，MCP 解决的是工具怎么标准化接入。**

---

## 二、MCP 解决了什么问题？

MCP 主要解决的是：

> **AI 工具接入碎片化、重复开发、难复用的问题。**

没有 MCP 时，如果要接入一个 GitHub 工具，需要自己处理：

- GitHub API 调用；
- OAuth / Token 认证；
- 参数格式转换；
- 返回结果解析；
- 错误处理；
- 和具体 AI 客户端的适配。

而且如果换一个客户端，比如从 Claude Desktop 换到 Cursor，可能又要重新适配一遍。

这会导致：

```text
每个工具都要单独接
每个客户端都要重新适配
工具实现和具体模型强绑定
工具生态难以复用
```

MCP 的作用就是给 AI 工具接入定义一套统一标准。

## 三、MCP 的核心思想

MCP 可以类比成 **AI 工具领域的 USB 接口**。

USB 出现之前，不同外设接口不同，连接成本很高。  
USB 出现之后，只要设备遵守 USB 标准，就可以插到不同电脑上使用。

MCP 做的是类似的事情：

```text
工具提供方实现 MCP Server
AI 客户端实现 MCP Client
Client 按协议连接 Server
模型自动发现并使用工具能力
```

这样工具只需要实现一次，就可以被多个 AI 应用复用。

## 四、MCP 的 Client-Server 架构

MCP 采用标准的 **Client-Server 架构**。

### 1. MCP Server

Server 是工具能力的实现方。

例如 GitHub MCP Server 可以封装：

```text
列出 PR
创建 Issue
搜索仓库
查看 Diff
提交代码
```

文件系统 MCP Server 可以封装：

```text
读取文件
创建文件
修改文件
列出目录
```

数据库 MCP Server 可以封装：

```text
查询表结构
执行 SQL
读取数据库记录
```

---

### 2. MCP Client

Client 是 AI 应用侧。

例如：

```text
Claude Desktop
Cursor
Windsurf
各种 Agent 框架
```

Client 连接 MCP Server 后，就可以发现并使用 Server 暴露的工具能力。

---

### 3. 一个 Client 可以连接多个 Server

例如一个 AI 客户端可以同时连接：

```text
文件系统 MCP Server
GitHub MCP Server
PostgreSQL MCP Server
浏览器自动化 MCP Server
```

这样模型就同时拥有：

```text
操作本地文件
读写代码仓库
查询数据库
操作浏览器
```

这些能力通常只需要在配置文件中配置 Server 地址或启动命令，不需要每个工具都重新写对接代码。

## 五、MCP 的三类核心能力

MCP Server 可以暴露三类能力：

```text
Tools
Resources
Prompts
```

---

## 六、Tools：有副作用的工具操作

**Tools** 是 MCP 中最接近 Function Calling 的部分。

它表示模型可以调用的具体操作。

例如：

```text
创建文件
修改文件
提交代码
发送 Slack 消息
创建 GitHub Issue
调用第三方 API
```

Tools 的特点是：

> **执行后会改变外部世界的状态。**

所以 Tools 通常是有副作用的，执行时要更谨慎，很多场景需要用户确认授权。

例如：

```text
删除文件
提交代码
发送邮件
修改数据库
```

这些操作不能让模型随便执行。

---

## 七、Resources：只读资源

**Resources** 表示只读数据。

它不会改变外部状态，只是把信息提供给模型。

例如：

```text
读取日志文件
读取数据库记录
获取文档内容
查看仓库文件
读取配置文件
```

Resources 的特点是：

> **只读、无副作用。**

可以把 Resources 理解成模型可以查看的“资料室”。

模型可以读取信息，但不能修改信息。

---

## 八、Prompts：可复用提示词模板

**Prompts** 是预定义的提示词模板。

它解决的是：

> **重复 prompt 的标准化和复用问题。**

例如团队内部有一个固定的代码审查 prompt：

```text
请基于以下规范审查这段 {language} 代码：
1. 是否有明显 bug
2. 是否有性能问题
3. 是否符合团队编码规范

代码内容：
{code}
```

这个 prompt 可以封装成 MCP Prompt。

使用时只需要传入参数：

```text
language = Python
code = 具体代码内容
```

就可以自动展开成完整 prompt。

Prompts 适合沉淀团队经验，比如：

```text
代码审查模板
SQL 优化模板
需求分析模板
测试用例生成模板
文档总结模板
```

---

## 九、Tools、Resources、Prompts 的区别

| 能力 | 作用 | 是否有副作用 | 例子 |
|---|---|---|---|
| Tools | 执行操作 | 有 | 创建文件、提交代码、发送消息 |
| Resources | 读取数据 | 无 | 读取日志、查询文档、查看数据库记录 |
| Prompts | 复用提示词模板 | 无 | 代码审查 prompt、总结 prompt |

简单记忆：

```text
Tools：让模型做事
Resources：让模型看资料
Prompts：让模型复用提示词
```

## 十、MCP 的底层通信：JSON-RPC 2.0

MCP 底层使用 **JSON-RPC 2.0**。

JSON-RPC 是一种轻量级远程调用协议，用 JSON 表示：

```text
调用哪个方法
传入什么参数
请求 ID 是多少
返回结果或错误信息
```

一个简单的 JSON-RPC 请求类似：

~~~json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}
~~~

服务端返回：

~~~json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": []
  }
}
~~~

JSON-RPC 的优点是：

```text
格式简单
容易调试
语言无关
适合跨进程和跨网络调用
```

## 十一、MCP 的传输方式

MCP 常见传输方式主要有两种：

```text
stdio
Streamable HTTP
```

---

### 1. stdio

stdio 指标准输入输出。

这种方式下，MCP Server 作为本地子进程运行。

Client 通过：

```text
stdin 发送请求
stdout 接收响应
```

适合本地工具，例如：

```text
本地文件系统工具
本地 Git 工具
本地脚本工具
```

优点：

```text
不需要网络
启动简单
延迟低
适合本地开发工具
```

---

### 2. Streamable HTTP

Streamable HTTP 适合远程 MCP Server。

这种方式下，Server 作为 HTTP 服务部署，Client 通过 HTTP 请求访问。

适合：

```text
远程数据库工具
团队共享工具服务
部署在服务器上的业务系统工具
```

Streamable HTTP 的特点是：

```text
一个 /mcp 端点即可通信
短请求可以直接返回 JSON
长请求可以通过 SSE 流式返回中间结果
```

早期 MCP 使用的是 **HTTP + SSE 双端点方案**，后来逐步演进为 **Streamable HTTP 单端点方案**，部署和使用更简单。

## 十二、MCP 和 Function Calling 的区别

很多人容易把 MCP 和 Function Calling 混为一谈。

它们不是同一层面的东西。

| 对比项 | Function Calling | MCP |
|---|---|---|
| 关注点 | 模型如何输出工具调用请求 | 工具如何标准化接入 |
| 所在层面 | 模型能力 / API 能力 | 工具接入协议 |
| 解决问题 | 让模型生成结构化调用参数 | 让工具实现一次、多端复用 |
| 典型形式 | JSON 函数调用 | Client-Server + JSON-RPC |
| 复用性 | 通常依赖具体模型厂商 | 任何支持 MCP 的客户端都可接入 |

一句话区分：

```text
Function Calling：模型会不会调工具
MCP：工具怎么统一接入 AI 应用
```

## 十三、为什么 MCP 生态发展很快？

MCP 生态发展快，主要有两个原因。

### 1. 实现门槛低

MCP 有清晰的协议规范和 SDK。

开发者可以比较快地实现一个 MCP Server。

只要遵守协议，AI 客户端就能发现和调用工具。

---

### 2. 头部工具快速跟进

很多常用工具都已经有 MCP Server，例如：

```text
GitHub
Slack
PostgreSQL
Puppeteer
Google Maps
文件系统
浏览器自动化
```

开发者很多时候不需要自己从零开发，只需要配置已有 Server 即可使用。

## 十四、记忆版总结

MCP 是 **Model Context Protocol，模型上下文协议**。

它不是 Agent 框架，也不是 Function Calling 本身，而是：

> **用于标准化 AI 工具接入的开放协议。**

MCP 要解决的问题是：

```text
工具接入碎片化
重复写适配代码
工具和客户端强绑定
工具能力难以复用
```

MCP 的核心架构是：

```text
MCP Client：AI 应用侧
MCP Server：工具实现侧
一个 Client 可以连接多个 Server
```

MCP Server 暴露三类能力：

```text
Tools：有副作用的操作，例如创建文件、提交代码
Resources：只读数据，例如读取日志、查询文档
Prompts：提示词模板，例如代码审查 prompt
```

底层通信：

```text
JSON-RPC 2.0
stdio：适合本地工具
Streamable HTTP：适合远程工具
```

最重要的一句话：

> **Function Calling 解决模型怎么发起调用，MCP 解决工具怎么标准化接入。**
