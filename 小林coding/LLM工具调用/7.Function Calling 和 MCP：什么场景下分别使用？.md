
## 一、面试标准回答

面试官问：**Function Calling 和 MCP 都能做工具调用，那具体什么场景下该选哪个？**

可以这样回答：

Function Calling 和 MCP 都属于工具调用方案，但它们解决的问题层次不一样。

**Function Calling 更适合轻量、临时、单应用内使用的工具调用场景**。  
比如做 Demo、快速原型，或者某个工具只服务当前应用，不需要跨项目复用，这时直接在应用代码里写工具 schema 和执行逻辑就够了。它简单直接，不需要额外启动 MCP Server，也方便在代码里做权限校验、参数处理、异常处理等精细控制。

**MCP 更适合工具需要独立管理、跨项目复用、跨客户端复用的场景**。  
如果同一套工具要被多个 Agent、多个项目、多个团队使用，或者社区已经有现成 MCP Server，比如 GitHub、Slack、PostgreSQL、Google Maps 这类工具，那就应该优先用 MCP。MCP 把工具封装成独立 Server，客户端只需要按协议连接，工具可以独立维护、独立部署、一次实现到处复用。

所以我的判断方法是：

> **只在当前应用里用、用完就结束、不需要复用，就用 Function Calling；只要工具需要复用、独立维护、规模变复杂，或者在做正式 Agent 系统，就优先考虑 MCP。**

但也不能绝对化。比如部署环境不允许启动子进程，或者工具执行逻辑需要和业务代码深度绑定，那 Function Calling 反而更合适。

一句话总结：

> **Function Calling 是应用内嵌工具调用，MCP 是工具服务化和标准化复用。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **Function Calling 和 MCP 不是谁替代谁，而是适用场景不同。**

不能简单说：

~~~text
小项目用 Function Calling
大项目用 MCP
~~~

也不能简单说：

~~~text
工具少用 Function Calling
工具多用 MCP
~~~

这些判断都太单一。

更合理的判断维度包括：

- 有没有现成 MCP Server；
- 工具是否需要跨项目复用；
- 工具数量和复杂度；
- 团队规模和维护成本；
- 是否在做正式 Agent 系统；
- 部署环境是否支持 MCP Server；
- 是否需要对工具执行逻辑做深度定制。

最终判断核心是：

> **这个工具是不是只服务当前应用？如果不是，就应该考虑 MCP。**

---

## 三、先建立直觉：内嵌 vs 独立

Function Calling 和 MCP 最大的区别是：

~~~text
Function Calling：工具内嵌在应用代码里
MCP：工具独立成一个 Server
~~~

---

### 1. Function Calling：内嵌式工具调用

Function Calling 的工具定义和执行逻辑通常写在当前项目代码里：

~~~text
应用代码
    ├── 工具 schema
    ├── 工具函数实现
    ├── 参数校验
    └── 调用逻辑
~~~

特点是：

- 简单；
- 直接；
- 上手快；
- 和当前应用绑定紧；
- 工具复用性较差。

适合：

> **这个工具只为当前应用服务。**

---

### 2. MCP：独立式工具服务

MCP 会把工具封装成独立运行的 MCP Server：

~~~text
AI Client / Agent
    ↓
MCP Client
    ↓
MCP Server
    ↓
真实工具 / API / 数据库 / 文件系统
~~~

特点是：

- 工具独立部署；
- 工具和应用解耦；
- 支持跨客户端复用；
- 支持工具自动发现；
- 维护成本更集中。

适合：

> **这个工具可能被多个项目、多个团队、多个客户端复用。**

---

## 四、什么时候用 Function Calling？

简单说：

> **轻量、临时、不需要复用，就用 Function Calling。**

---

### 1. 快速原型和 Demo

如果只是想快速验证一个想法，比如：

~~~text
让模型调用一个天气接口
让模型查一个本地函数
让模型调一个简单搜索 API
~~~

直接用 Function Calling 最快。

因为你只需要：

~~~text
写工具 schema
写函数实现
把 tools 传给模型
解析 tool_calls
执行函数
~~~

不需要额外启动 MCP Server，也不需要配置协议。

---

### 2. 工具只服务单一应用

如果这个工具只在当前项目里用，不会被其他项目复用，那么 Function Calling 更清晰。

例如：

~~~text
查询本项目内部数据库
调用当前业务系统的私有接口
执行当前应用独有的业务逻辑
~~~

这种工具和业务绑定很深，单独抽成 MCP Server 反而增加复杂度。

---

### 3. 需要精细控制工具执行逻辑

Function Calling 的执行逻辑完全在应用代码里，所以很适合做细粒度控制：

- 权限校验；
- 参数二次处理；
- 异常兜底；
- 调用链路追踪；
- 业务日志记录；
- 特殊重试逻辑；
- 和当前业务状态联动。

如果这些逻辑本来就依赖当前应用上下文，直接写在 Function Calling 的执行层里更方便。

---

### 4. 部署环境受限

有些平台不允许启动额外子进程，或者部署环境不适合跑独立 MCP Server。

比如：

- 某些 Serverless 平台；
- 受限云环境；
- 只能运行单进程的服务；
- 本地部署条件很简单的 Demo。

这种情况下，用 Function Calling 把工具逻辑写在主进程里更稳妥。

---

## 五、什么时候用 MCP？

简单说：

> **只要工具不是“只自己用一次”，就应该考虑 MCP。**

---

### 1. 工具需要跨项目或跨团队复用

如果同一个工具会被多个地方使用，比如：

~~~text
Claude Desktop 要用
Cursor 要用
团队内部 Agent 要用
CI/CD Agent 要用
另一个项目也要用
~~~

那就很适合 MCP。

因为 MCP 可以把工具封装成一个独立 Server：

~~~text
GitHub MCP Server
    ↓
多个客户端都可以连接使用
~~~

这样不用每个项目都写一遍 schema 和调用代码。

---

### 2. 社区已经有现成 MCP Server

如果社区已经有成熟 MCP Server，通常就没必要自己用 Function Calling 重写一遍。

例如：

- GitHub；
- Slack；
- PostgreSQL；
- Puppeteer；
- Google Maps；
- 文件系统；
- 浏览器操作。

这类工具直接配置 MCP Server 就能用。

例如：

~~~json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"]
    }
  }
}
~~~

这种情况下继续手写 API 封装，属于重复造轮子。

---

### 3. 工具数量和复杂度上来了

MCP 不一定只看工具数量，而是综合看：

~~~text
工具数量
工具复杂度
团队规模
接口变更频率
维护成本
~~~

如果工具 schema 和调用逻辑越来越多，散落在应用代码各处，就会出现：

- 新增工具要改主应用；
- 工具 bug 要改主应用；
- 工具 schema 不好统一管理；
- 不同项目重复维护；
- 调用链路越来越乱。

MCP 的好处是：

> **工具集中在 Server 侧管理，主应用只负责连接和使用。**

---

### 4. 正式 Agent 系统更适合 MCP

Agent 系统通常会接入很多工具：

- 文件系统；
- 数据库；
- 搜索引擎；
- 浏览器；
- GitHub；
- 代码执行；
- 企业内部 API；
- 向量数据库。

如果全部用 Function Calling，Agent 代码里会塞满工具 schema 和调用逻辑。

MCP 可以让工具来源模块化：

~~~text
Agent 核心逻辑
    ↓
连接多个 MCP Server
    ↓
按需使用不同工具
~~~

这样 Agent 的核心逻辑和工具管理解耦，架构更清晰。

---

## 六、实用判断方法

遇到“用 Function Calling 还是 MCP”的选择，可以按下面顺序判断。

---

### 1. 社区有没有现成 MCP Server？

如果有，优先用 MCP。

因为：

~~~text
能配置就不要重写
能复用就不要重复造轮子
~~~

---

### 2. 工具是否需要跨项目复用？

如果需要复用，优先 MCP。

例如：

~~~text
这个工具不止当前应用用
以后别的 Agent 也可能用
团队其他成员也要用
多个客户端都要用
~~~

这种场景把工具封装成 MCP Server 更长期。

---

### 3. 工具规模是否开始变复杂？

不是简单看工具数量，而是看有没有维护压力：

~~~text
schema 越来越多
调用逻辑越来越散
接口经常变化
多人同时维护
不同项目重复实现
~~~

如果出现这些问题，就应该考虑 MCP。

---

### 4. 是正式 Agent 系统还是 Demo？

如果只是 Demo：

~~~text
Function Calling 更快
~~~

如果是正式 Agent 系统：

~~~text
MCP 更利于长期维护
~~~

---

### 5. 部署环境是否支持 MCP？

如果部署环境不允许启动子进程，或者不方便维护独立 Server，那么 Function Calling 更稳妥。

---

## 七、选择口诀

可以这样记：

~~~text
只自己用、只用一次、工具简单：Function Calling
要复用、要解耦、工具复杂：MCP
社区已有 MCP Server：直接用 MCP
正式 Agent 系统：优先 MCP
部署环境受限：Function Calling
执行逻辑强依赖业务代码：Function Calling
~~~

---

## 八、Function Calling 和 MCP 对比表

| 对比项 | Function Calling | MCP |
|---|---|---|
| 工具形态 | 内嵌在应用代码中 | 独立 MCP Server |
| 复用能力 | 弱 | 强 |
| 上手成本 | 低 | 中等 |
| 适合场景 | Demo、单应用工具 | 多项目、多客户端、Agent 系统 |
| 维护方式 | 跟随主应用维护 | 工具独立维护 |
| 工具发现 | 手动传入 schema | MCP 协议发现工具 |
| 部署复杂度 | 低 | 更高 |
| 精细业务控制 | 更方便 | 需要额外设计 |
| 长期扩展性 | 一般 | 更好 |

## 十、结合 FinReportAgent 项目怎么说

如果面试官问：

> **你的金融 Agent 项目里，工具调用会选 Function Calling 还是 MCP？**

可以这样回答：

在我的 FinReportAgent 项目里，如果只是当前项目内部使用的工具，比如文档类型识别、当前 PDF 的 RAG 检索、Session Memory 写入、Research Memory 检索，这些和项目内部状态绑定很深，我会更倾向于用 Function Calling 或框架内置 Tool 机制，因为它们需要直接访问当前应用的上下文、数据库连接和业务状态。

但如果是通用工具，比如 GitHub、文件系统、浏览器搜索、数据库查询、企业知识库访问，或者后续希望多个 Agent 都能复用同一套工具，我会考虑把它们封装成 MCP Server。这样不同客户端或不同 Agent 都可以通过 MCP 协议连接，而不用在每个项目里重复写工具 schema 和调用逻辑。

所以我的判断不是简单地说项目大就用 MCP，而是看：

~~~text
这个工具是不是项目强绑定？
这个工具会不会被其他 Agent 复用？
有没有现成 MCP Server？
部署环境是否允许独立 Server？
长期维护成本是否值得抽象成 MCP？
~~~

在项目早期，我会优先用 Function Calling 快速验证；当工具稳定下来并且有复用需求时，再逐步抽成 MCP Server。

## 十一、面试官可能继续追问

### 1. Function Calling 和 MCP 是竞争关系吗？

不是。

Function Calling 更像是模型调用工具的底层能力，MCP 更像是工具服务标准化和复用协议。

很多时候，MCP Server 暴露工具，客户端再通过 Function Calling 或类似机制让模型决定调用哪个工具。

---

### 2. 为什么不能简单说“大项目用 MCP，小项目用 Function Calling”？

因为项目规模不是唯一维度。

一个大项目里的工具如果只服务当前业务、不需要复用，也可以用 Function Calling。

一个小项目如果刚好需要使用现成 GitHub MCP Server，也可以直接用 MCP。

更关键的是：

~~~text
工具是否复用
工具是否独立维护
是否已有 MCP Server
部署环境是否支持
是否做正式 Agent 系统
~~~

---

### 3. Function Calling 最大优势是什么？

最大优势是：

> **简单直接，和应用代码集成紧密。**

它适合快速原型、单应用私有工具、业务逻辑强绑定场景。

---

### 4. MCP 最大优势是什么？

最大优势是：

> **工具标准化、独立化、可复用。**

一个 MCP Server 可以被多个 AI 客户端和多个 Agent 使用，减少重复开发和重复维护。

---

### 5. 如果工具执行逻辑很复杂，选哪个？

要看复杂在哪里。

如果复杂是因为它强依赖当前业务上下文，比如权限、用户状态、业务数据库事务，那 Function Calling 更方便。

如果复杂是因为工具本身功能多、要被多人复用、接口会持续演进，那 MCP 更适合独立维护。

---

### 6. 为什么正式 Agent 系统更适合 MCP？

因为 Agent 系统通常工具多、来源杂、长期维护压力大。

MCP 可以让 Agent 只关注核心推理和任务流程，而把工具接入、工具发现、工具维护交给独立 Server。

这样架构更清晰。

---

## 十二、记忆版总结

这篇文章可以压缩成一句话：

> **Function Calling 适合应用内嵌、轻量、单次使用的工具；MCP 适合独立服务、跨项目复用、正式 Agent 系统的工具。**

进一步记：

~~~text
Function Calling：
轻量、临时、Demo、单应用、强业务绑定、部署受限

MCP：
复用、解耦、工具多、社区已有 Server、正式 Agent、长期维护
~~~

最实用判断：

~~~text
先看有没有现成 MCP Server：有就用 MCP
再看是否需要复用：需要就用 MCP
再看工具是否复杂到难维护：复杂就 MCP
再看是否正式 Agent 系统：正式系统优先 MCP
最后看部署是否受限、是否强业务绑定：是就 Function Calling
