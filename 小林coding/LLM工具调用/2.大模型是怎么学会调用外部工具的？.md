
## 一、面试标准回答

面试官问：**大模型是怎么学会调用外部工具的？**

可以这样回答：

大模型的工具调用能力不是靠参数量自然涌现出来的，而是经过专门训练得到的。

预训练阶段的大模型主要学习的是文本续写能力，它只是在海量文本中学习“下一个 token 怎么预测”。但是工具调用要求模型能理解工具 schema，并输出可被程序解析的结构化 JSON，例如 `tool_calls`。这种能力不是普通预训练语料里自然就有的，所以需要后续训练。

一般可以分成两个关键阶段：

第一阶段是 **SFT，监督微调**。  
通过大量工具调用示范数据，让模型学会完整流程：看到工具描述，理解用户问题，判断是否需要工具，如果需要就输出规范的 JSON 调用请求，工具返回结果后再生成最终回答。  
SFT 主要解决的是：**模型会不会调工具。**

第二阶段是 **RLHF，人类反馈强化学习**。  
SFT 后的模型虽然会调工具，但可能边界感不好，比如用户问 `1+1`，它也去调用计算器。RLHF 通过人类偏好数据训练奖励模型，再用强化学习优化主模型，让模型学会什么时候该调工具，什么时候不该调工具。  
RLHF 主要解决的是：**模型该不该调工具。**

训练完成后，运行时就是 Function Calling 机制：应用代码把工具 schema 和用户问题一起发给模型，模型如果判断需要工具，就输出 `tool_calls` JSON；后端代码解析 JSON，真正执行工具，然后把工具结果作为 `tool` 消息放回上下文，再让模型生成最终答案。

所以一句话总结：

> **SFT 教模型“怎么调工具”，RLHF 教模型“什么时候调工具”，运行时由模型负责决策、代码负责执行。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **工具调用能力不是大模型天生具备的，而是通过 SFT 和 RLHF 训练出来的。**

很多人容易有两个误解：

第一个误解是：

> 模型参数量够大，自然就会调用工具。

这是不准确的。大模型在预训练阶段主要学的是文本预测，不会真正执行工具，也不一定会输出规范的 JSON 调用格式。

第二个误解是：

> 只要做 SFT，模型就能很好地调用工具。

这也不完整。SFT 只能让模型学会“如何调用”，但不一定能让模型学会“什么时候不该调用”。

所以完整理解应该是：

~~~text
SFT：让模型学会工具调用格式和流程
RLHF：让模型学会工具调用的边界
Function Calling：训练好之后的运行时机制
~~~

## 三、为什么原始 LLM 不会调用工具？

预训练阶段的大模型，本质上是在做：

~~~text
给定前面的文本，预测下一个 token
~~~

它学到的是语言规律、知识表达和推理模式。

但是工具调用不是普通文本续写，而是要求模型输出一种程序能解析的结构化格式，例如：

~~~json
{
  "tool_calls": [
    {
      "name": "get_weather",
      "arguments": {
        "city": "北京"
      }
    }
  ]
}
~~~

没经过专门训练的模型，遇到用户问：

~~~text
北京今天天气怎么样？
~~~

它可能会说：

~~~text
我需要查询天气 API 才能回答。
~~~

但它不一定会输出：

~~~json
{
  "name": "get_weather",
  "arguments": {
    "city": "北京"
  }
}
~~~

所以工具调用能力不是“模型聪明了就自然会”，而是需要专门训练。

---

## 四、第一阶段：SFT 让模型学会“怎么调工具”

SFT 全称是 **Supervised Fine-Tuning**，也就是监督微调。

它的核心思路是：

> **给模型看大量标准工具调用示例，让模型模仿。**

一条完整的工具调用训练样本通常包括：

~~~text
System：工具说明书，也就是 tools schema
User：用户问题
Assistant：模型输出 tool_calls JSON
Tool：工具返回结果
Assistant：基于工具结果生成最终回答
~~~

例如：

~~~text
System：
你有一个 get_weather 工具，可以查询指定城市天气。

User：
北京今天天气怎么样？

Assistant：
{
  "tool_calls": [
    {
      "name": "get_weather",
      "arguments": {
        "city": "北京"
      }
    }
  ]
}

Tool：
北京今天晴，15°C，东北风 3 级。

Assistant：
北京今天晴朗，气温 15°C，东北风 3 级，适合外出。
~~~

模型通过大量这种样本，学会：

- 识别工具 schema；
- 判断用户问题是否需要工具；
- 选择合适工具；
- 填写工具参数；
- 输出规范 JSON；
- 基于工具结果生成最终回答。

所以 SFT 解决的问题是：

> **让模型从“没见过工具调用”，变成“会模仿工具调用流程”。**

---

## 五、SFT 的短板：会调，但不一定知道该不该调

SFT 的问题是：

> **它主要教模型“怎么调”，但没有充分教模型“什么时候不该调”。**

因为工具调用训练集中，大量样本都是“需要调用工具”的场景。

模型可能学到一种倾向：

~~~text
只要看到问题，就优先尝试调用工具。
~~~

这会导致一些不合理行为。

比如用户问：

~~~text
1 + 1 等于几？
~~~

模型其实可以直接回答：

~~~text
2
~~~

但 SFT 后的模型可能会去调用计算器工具。

再比如用户问：

~~~text
什么是 RAG？
~~~

这是一个普通知识问题，模型可以直接回答，不一定需要检索工具。

所以 SFT 后的模型可能有两个问题：

- 对简单问题也调用工具；
- 工具调用失败时不知道如何处理；
- 对“能直接回答”和“必须用工具”的边界判断不稳定。

这就需要 RLHF 来补充。

---

## 六、第二阶段：RLHF 让模型学会“什么时候调工具”

RLHF 全称是 **Reinforcement Learning from Human Feedback**，也就是基于人类反馈的强化学习。

如果说 SFT 是让模型看标准答案学习模仿，那么 RLHF 就是：

> **通过人类偏好反馈，让模型学会什么行为更合理。**

它主要解决的是：

> **工具调用的边界感。**

也就是：

~~~text
什么时候应该调工具
什么时候不应该调工具
什么时候应该拒答
什么时候应该直接回答
工具调用失败后该怎么处理
~~~

---

## 七、RLHF 的基本流程

RLHF 大致可以分成四步。

### 1. 生成多个候选回答

对同一个问题，让模型生成多种不同回答。

例如用户问：

~~~text
1 + 1 等于几？
~~~

模型可能生成：

~~~text
回答 A：2。
回答 B：我需要调用 calculator 工具计算。
回答 C：让我查询一下数据库。
~~~

---

### 2. 人类标注偏好

标注员判断哪个回答更合理。

对于 `1 + 1` 这种问题，显然：

~~~text
直接回答 2
~~~

比调用工具更好。

但如果用户问：

~~~text
北京今天实时天气怎么样？
~~~

那么调用天气工具就比直接编一个天气更好。

---

### 3. 训练奖励模型

把人类偏好数据拿来训练一个奖励模型。

奖励模型不负责回答问题，只负责打分：

~~~text
这个回答是否符合人类偏好？
这个工具调用是否合理？
这个参数是否正确？
这个场景是否应该直接回答？
~~~

可以把奖励模型理解成：

> **一个专门给回答质量打分的裁判。**

---

### 4. 用强化学习优化主模型

最后用奖励模型的分数来优化主模型。

主模型会逐渐倾向于生成高分行为：

- 该调工具时调用工具；
- 不该调工具时直接回答；
- 工具参数更准确；
- 工具失败时能合理处理；
- 不会乱调用无关工具。

所以 RLHF 解决的问题是：

> **让模型从“会调用工具”，进一步变成“会合理调用工具”。**

---

## 八、SFT 和 RLHF 的区别

| 阶段 | 解决问题 | 核心作用 |
|---|---|---|
| SFT | 会不会调工具 | 学会工具调用格式和基本流程 |
| RLHF | 该不该调工具 | 建立工具调用边界感 |
| RLAIF | 低成本反馈替代 | 用 AI 反馈替代部分人工反馈 |

简单记：

~~~text
SFT：教动作
RLHF：教判断
~~~

或者：

~~~text
SFT 让模型会调工具
RLHF 让模型少乱调工具
~~~

---

## 九、RLAIF 是什么？

RLAIF 全称是 **Reinforcement Learning from AI Feedback**。

它和 RLHF 类似，只是把“人类打分”换成“AI 打分”。

也就是：

~~~text
RLHF：人类标注员判断哪个回答更好
RLAIF：另一个更强的 AI 判断哪个回答更好
~~~

RLAIF 的好处是：

- 成本更低；
- 速度更快；
- 可以批量生成反馈数据；
- 适合大规模训练。

但它也有风险：

- AI 裁判本身可能有偏差；
- 如果评价标准不好，会把主模型优化到错误方向；
- 高质量场景仍然需要人工抽查。

所以可以理解为：

> **RLAIF 是 RLHF 的低成本替代或补充。**

## 十、运行时：训练好之后怎么用？

训练完成后，工具调用能力通常通过 Function Calling 的形式使用。

运行时流程是：

~~~text
用户问题 + 工具 schema
    ↓
发送给模型
    ↓
模型判断是否需要工具
    ↓
如果需要，输出 tool_calls JSON
    ↓
后端代码解析 JSON
    ↓
后端真正执行工具
    ↓
把工具结果作为 tool message 塞回上下文
    ↓
再次调用模型
    ↓
模型生成最终回答
~~~

关键点是：

> **模型只负责决策，不负责执行。**

比如用户问：

~~~text
北京今天天气怎么样？
~~~

模型输出：

~~~json
{
  "tool_calls": [
    {
      "name": "get_weather",
      "arguments": {
        "city": "北京"
      }
    }
  ]
}
~~~

然后是后端代码真正执行：

~~~text
get_weather(city="北京")
~~~

拿到结果后再塞回模型：

~~~text
北京今天晴，15°C，东北风 3 级
~~~

最后模型生成自然语言回答。

## 十一、模型决策，代码执行

这是理解工具调用最关键的一句话：

> **LLM 负责判断该调用什么工具，宿主程序负责真正执行工具。**

原因是：

- LLM 擅长理解用户意图；
- LLM 擅长根据上下文选择工具；
- 但 LLM 不应该直接拥有执行系统资源的权限；
- 后端代码可以做权限控制、参数校验、异常处理和日志记录。

所以工具调用架构通常是：

~~~text
LLM：判断意图，输出 tool_calls
后端代码：解析参数，执行函数
工具系统：访问 API / 数据库 / 搜索引擎 / 向量库
LLM：基于工具结果生成最终答案
~~~

这样的分工既灵活，也更安全。

## 十四、结合 Agent 项目怎么说

如果面试官问：

> **你的 Agent 项目里，模型是怎么学会调用工具的？**

可以这样回答：

在 Agent 项目中，模型调用工具的能力本质上来自模型训练阶段的 SFT 和 RLHF。

SFT 让模型见过大量工具调用样本，所以它知道看到工具 schema 后，应该输出类似 `tool_calls` 的结构化调用请求，而不是只说“我需要查一下”。RLHF 则进一步让模型学会边界，比如用户问实时行情、新闻、数据库内容时应该调用工具；但用户问普通概念问题时，可以直接回答，不需要无意义调用工具。

在我的金融分析 Agent 中，模型本身不会真的访问行情 API、新闻搜索或者 Milvus。运行时我会把这些工具的 schema 提供给模型，模型根据用户问题判断是否需要调用工具。如果需要，它输出工具名和参数，后端代码再真正执行行情查询、新闻搜索或 RAG 检索，然后把结果返回给模型生成最终答案。

所以 Agent 工程里的关键分工是：

~~~text
模型负责理解意图和选择工具
工具注册系统负责暴露工具 schema
后端代码负责执行工具
模型再基于工具结果继续推理和回答
~~~

这也是 Function Calling 和 ReAct 类 Agent 的共同核心：

> **LLM 负责决策，外部程序负责执行。**

## 十五、面试官可能继续追问

### 1. 工具调用能力是模型涌现出来的吗？

不是。

参数量变大可能带来更强的理解和推理能力，但工具调用要求模型输出稳定、可解析的结构化 JSON，这需要专门的工具调用训练数据。

所以工具调用不是简单靠涌现，而是靠 SFT、RLHF 等训练阶段塑造出来的。

---

### 2. SFT 在工具调用训练里解决什么问题？

SFT 解决的是：

> **模型会不会调工具。**

它通过大量工具调用示范样本，让模型学会：

- 识别工具 schema；
- 判断用户问题；
- 选择工具；
- 填写参数；
- 输出规范 JSON；
- 基于工具结果回答。

---

### 3. RLHF 在工具调用训练里解决什么问题？

RLHF 解决的是：

> **模型该不该调工具。**

它通过人类偏好反馈，让模型学会：

- 简单问题直接回答；
- 实时信息调用工具；
- 需要外部操作时调用工具；
- 不乱调用无关工具；
- 工具失败时合理处理。

---

### 4. 为什么只做 SFT 不够？

因为 SFT 主要是模仿学习。

如果训练数据里大多数都是工具调用样本，模型可能形成“过度调用工具”的倾向。

例如：

~~~text
1 + 1 等于几？
~~~

这种问题可以直接回答，但模型可能也去调用计算器。

所以还需要 RLHF 来建立边界感。

---

### 5. RLAIF 和 RLHF 有什么区别？

区别在于反馈来源不同：

~~~text
RLHF：人类给偏好反馈
RLAIF：AI 给偏好反馈
~~~

RLAIF 成本更低、速度更快，但评价标准也可能有偏差，所以通常需要人工抽查或配合人工反馈使用。

---

### 6. 训练好之后，运行时是谁在执行工具？

是后端代码执行工具，不是模型执行工具。

模型只输出：

~~~text
工具名
参数
调用请求 JSON
~~~

真正访问 API、数据库、搜索引擎、向量库的是宿主程序。

---

### 7. 为什么要让代码执行，而不是模型直接执行？

因为这样更安全、可控。

代码可以做：

- 参数校验；
- 权限控制；
- 日志记录；
- 失败重试；
- 超时控制；
- 沙箱隔离；
- API 密钥管理。

模型只负责决策，代码负责执行，是主流工具调用架构的核心设计。

---

## 十六、记忆版总结

这篇文章可以压缩成一句话：

> **LLM 工具调用能力不是天生的，而是 SFT 教会“怎么调”，RLHF 教会“该不该调”，运行时通过 Function Calling 由模型决策、代码执行。**

进一步记：

~~~text
预训练：只学文本预测，不会真正工具调用
SFT：看大量工具调用样本，学会输出 tool_calls JSON
SFT 短板：容易过度调用，边界感弱
RLHF：通过人类偏好反馈，学会什么时候该调、什么时候不该调
RLAIF：用 AI 反馈替代部分人工反馈，降低成本
运行时：模型输出工具调用 JSON，后端代码真正执行工具
核心分工：模型负责决策，代码负责执行
~~~

最重要的一句话：

> **SFT 解决“会不会调”，RLHF 解决“该不该调”。**
