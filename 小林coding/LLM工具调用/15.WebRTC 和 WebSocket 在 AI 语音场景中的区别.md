
## 面试标准回答

面试官问：**为什么 AI 实时语音要用 WebRTC？它和 WebSocket 的核心差异是什么？**

可以这样回答：

AI 实时语音场景优先用 WebRTC，核心原因不是它支持 P2P，而是它底层基于 **UDP**，更适合实时音视频传输；而 WebSocket 底层基于 **TCP**，更适合文字或普通数据的可靠传输。

语音场景和文字场景的网络要求完全不同。

文字传输更关注：

~~~text
完整性
顺序性
不能丢字符
~~~

所以 TCP 的可靠传输很适合文字。

但是语音传输更关注：

~~~text
低延迟
连续播放
不要卡顿
~~~

语音可以容忍少量丢包，但不能容忍延迟堆积。  
如果用 WebSocket 传语音，一旦网络抖动导致丢包，TCP 会强制重传，后续音频数据全部被阻塞，延迟会不断累积，用户就会感觉通话卡顿。

WebRTC 基于 UDP，丢包后不会等待重传，而是通过 **丢包隐藏 Packet Loss Concealment** 等机制，用前后音频帧插值补偿，用轻微音质损失换取稳定低延迟。它通常可以把语音延迟控制在比较低的范围，更适合实时对话。

另外，WebRTC 不只是一个传输协议，而是一整套实时音视频协议栈，内置了很多语音场景必需能力：

~~~text
回声消除 AEC
噪声抑制 NS
自动增益控制 AGC
自适应码率 ABR
RTP / SRTP 媒体传输
ICE / STUN / TURN NAT 穿透
~~~

这些能力如果用 WebSocket 实现，都需要自己额外开发，工程复杂度很高。

所以总结来说：

> **WebSocket 适合文字和普通数据的实时双向通信；WebRTC 适合实时语音、视频这类低延迟媒体流。**

如果是 LLM 文字流式输出，用 SSE 或 WebSocket 都可以；但如果是 AI 实时语音助手，尤其是要求用户和 AI 可以同时说话、低延迟响应、随时打断，就应该优先选择 WebRTC。

## 对比总结

| 对比项 | WebSocket | WebRTC |
|---|---|---|
| 底层协议 | TCP | UDP |
| 核心目标 | 可靠双向数据通信 | 低延迟实时音视频 |
| 丢包处理 | TCP 重传，后续数据等待 | 不等待重传，用丢包隐藏补偿 |
| 延迟特点 | 网络抖动时容易堆积延迟 | 更稳定低延迟 |
| 音频处理 | 需要自己实现 | 内置 AEC、NS、AGC、ABR |
| 连接方式 | 客户端到服务器 | P2P，失败时可 TURN 中转 |
| 建连复杂度 | 简单 | 较复杂，需要 SDP + ICE/STUN/TURN |
| 适合场景 | 文本流、业务消息、实时数据 | AI 语音、视频通话、实时音视频 |

## 关键加分点

面试时可以补一句：

> **WebRTC 和 WebSocket 不是完全替代关系。WebRTC 传输真正的音频流，但建连前仍然需要一个信令通道交换 SDP，这个信令通道可以用 WebSocket 或 HTTP 实现。**

也就是说：

~~~text
WebSocket / HTTP：负责信令交换
WebRTC：负责真正的音频媒体传输
~~~

最重要的一句话：

> **语音场景的核心原则是：可以丢一点包，但不能等重传；WebRTC 用 UDP 和音频补偿机制换低延迟，而 WebSocket 基于 TCP，丢包重传会导致语音卡顿。**
