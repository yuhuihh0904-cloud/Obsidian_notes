## 一、面试标准回答

如果面试官问：

> 你是怎么给 LLM 加规划能力的？CoT、ToT、GoT 和 Plan-and-Execute 分别是什么？

可以这样回答：

我理解给 LLM 加规划能力，本质上是为了让模型不要直接“一口气生成答案”，而是把复杂任务拆成多个步骤，让推理过程显式化，并且在必要时能够纠偏。

最基础的方法是 **CoT，也就是 Chain of Thought，思维链**。它的做法是在 prompt 里引导模型一步步思考，比如加一句“请一步一步分析”。这样模型会先生成中间推理步骤，再基于这些步骤继续生成后续答案。因为 LLM 是按 token 顺序生成的，前面写出来的推理过程会进入上下文，影响后续生成，所以 CoT 可以减少模型跳步推理带来的错误。

但是 CoT 有一个明显问题：它只有一条推理路径。如果一开始方向错了，后面就会沿着错误方向继续推下去，没有纠偏机制。

为了解决这个问题，就有了 **ToT，也就是 Tree of Thoughts，思维树**。ToT 不再只生成一条推理链，而是让模型同时生成多条候选思路，然后对这些思路进行评估、打分和剪枝，只保留更有希望的路径继续深入。它的核心过程是：

```text
生成多个候选思路 → 评估每条路径 → 剪掉低质量路径 → 继续展开高质量路径
```

所以 ToT 解决的是 CoT “一条路走到黑”的问题。但是它的缺点也很明显：需要多次 LLM 调用，成本通常是 CoT 的 3 到 5 倍，复杂情况下甚至更高。

再进一步是 **GoT，也就是 Graph of Thoughts，思维图**。GoT 把推理结构从树变成图，允许不同推理路径之间的中间结果被复用和合并。比如一个任务需要分别研究竞品 A 和竞品 B，然后再做综合对比，GoT 可以让“研究 A”和“研究 B”的结果共同流入“综合对比”这个节点。它的表达能力比 ToT 更强，但实现复杂，目前更多还是学术思路，生产环境中很少真正落地。

而在真实 Agent 工程里，更常用的是 **Plan-and-Execute，也就是先规划再执行**。它不是只关注模型内部怎么推理，而是把复杂任务拆成两个阶段：

```text
先由 Planner 生成完整计划
再由 Executor 按步骤执行
执行过程中由 Re-planner 根据结果动态调整后续计划
```

Plan-and-Execute 和 ReAct 不是替代关系，而是互补关系。ReAct 更偏每一步内部的“思考、行动、观察”，Plan-and-Execute 更偏整个任务层面的全局规划和步骤编排。

所以总结来说：

> CoT 解决“要不要显式推理”的问题；ToT 解决“走错方向怎么纠偏”的问题；GoT 解决“不同推理路径的中间结果能不能复用”的问题；Plan-and-Execute 则是工程里更常见的任务级规划方式，负责先拆解任务，再逐步执行，并在执行过程中动态调整计划。

## 二、文章核心观点

这篇文章的核心观点是：

> 给 LLM 加规划能力，不只是加一句“请一步步思考”。CoT 只是最基础的方式，真正理解规划能力，需要知道 CoT、ToT、GoT 和 Plan-and-Execute 分别解决什么问题。

它们之间的关系可以概括成：

```text
CoT：一条链
ToT：一棵树
GoT：一张图
Plan-and-Execute：工程里的任务规划与执行框架
```

---

## 三、关键知识点

### 1. 为什么 LLM 需要规划能力？

普通 LLM 默认是直接生成答案。

这种方式在简单问题上没什么问题，但复杂任务通常需要多步推理。如果模型不显式写出中间过程，而是直接跳到答案，中间某一步隐式出错后，后面就会继续沿着错误方向生成。

所以规划能力的核心作用是：

```text
把隐式推理显式化；
把复杂任务拆成步骤；
让执行过程可以检查和纠偏。
```

---

### 2. CoT：Chain of Thought

CoT 的核心是：

```text
让模型一步一步写出推理过程。
```

常见 prompt：

```text
请一步一步思考。
```

它的优点是：

```text
实现简单；
成本低；
只需要改 prompt；
适合作为默认增强方式。
```

它的缺点是：

```text
只有一条推理路径；
如果一开始方向错了，后面容易一路错下去。
```

一句话记忆：

> CoT 让模型“把推理写出来”，解决的是显式推理问题。

---

### 3. ToT：Tree of Thoughts

ToT 的核心是：

```text
同时探索多条推理路径，边探索、边评估、边剪枝。
```

基本流程：

```text
生成多个候选思路
↓
评估每个思路的可行性
↓
剪掉低质量路径
↓
保留高质量路径继续深入
```

它解决的是：

```text
CoT 单路径推理，一旦走错就无法纠偏的问题。
```

它的优点是：

```text
可以探索多个方向；
有评估和剪枝机制；
复杂推理任务更稳。
```

它的缺点是：

```text
调用成本更高；
通常是 CoT 的 3 到 5 倍；
不适合所有任务默认使用。
```

一句话记忆：

> ToT 让模型“多想几条路，再选好的路继续走”。

---

### 4. GoT：Graph of Thoughts

GoT 的核心是：

```text
把推理结构从树变成图，让不同路径的中间结果可以复用和合并。
```

它解决的是：

```text
ToT 不同分支之间结果不能自然复用的问题。
```

例如：

```text
研究竞品 A
研究竞品 B
研究竞品 C
   ↓
综合对比分析
   ↓
最终报告
```

这种多个中间结果汇聚到同一个后续节点的结构，用图表示更自然。

它的优点是：

```text
表达能力更强；
适合复杂依赖关系；
可以复用中间结果。
```

它的缺点是：

```text
实现复杂；
调度复杂；
状态管理复杂；
目前工程落地较少。
```

一句话记忆：

> GoT 让模型的推理不只是树状分叉，还能让不同分支汇合和复用。

---

### 5. Plan-and-Execute：工程里更常用的规划模式

Plan-and-Execute 的核心是：

```text
先规划，再执行。
```

它通常包含三个角色：

```text
Planner：负责生成计划
Executor：负责按计划执行
Re-planner：负责根据执行结果调整后续计划
```

标准流程：

```text
用户任务
↓
Planner 生成步骤清单
↓
Executor 执行当前步骤
↓
观察执行结果
↓
Re-planner 判断是否需要修改计划
↓
继续执行下一步
```

它解决的问题是：

```text
复杂 Agent 任务不能只靠模型边想边做；
需要先有全局计划，再逐步执行。
```

一句话记忆：

> Plan-and-Execute 是工程里常用的任务级规划方法，适合长流程、多步骤、需要工具调用的复杂任务。

---

## 四、Plan-and-Execute 和 ReAct 的关系

ReAct 的流程是：

```text
Thought → Action → Observation
```

它强调的是：

```text
每一步怎么思考、怎么行动、怎么根据观察继续决策。
```

Plan-and-Execute 强调的是：

```text
整个任务先做什么、后做什么、计划是否需要调整。
```

所以二者不是替代关系，而是组合关系：

```text
Plan-and-Execute 负责全局规划；
ReAct 负责单步执行。
```

可以理解为：

```text
Planner 决定任务步骤；
Executor 在每个步骤内部用 ReAct 调工具、看结果、继续执行；
Re-planner 根据执行结果调整后续步骤。
```

## 五、面试官可能追问

### 追问 1：CoT 是不是就等于规划能力？

不是。

CoT 只是最基础的规划方式，它让模型把推理过程显式写出来，但它只有一条推理链，没有多路径探索，也没有纠偏机制。

更完整的规划能力还包括：

```text
ToT 的多路径探索；
GoT 的中间结果复用；
Plan-and-Execute 的任务级规划和执行。
```

---

### 追问 2：ToT 相比 CoT 最大改进是什么？

ToT 最大改进是：

```text
从单条推理链变成多条推理路径搜索。
```

CoT 是一条路走到底；ToT 会生成多个候选思路，然后评估、剪枝，保留更好的路径继续深入。

所以 ToT 解决的是：

```text
CoT 走错方向后无法纠偏的问题。
```

---

### 追问 3：GoT 相比 ToT 又多解决了什么？

GoT 解决的是：

```text
ToT 不同分支之间中间结果不能复用的问题。
```

ToT 是树结构，分支之间相对独立；GoT 是图结构，可以让多个节点的结果汇聚到同一个后续节点。

所以 GoT 更适合表达复杂依赖关系。

---

### 追问 4：为什么工程里更常用 Plan-and-Execute？

因为真实 Agent 任务往往不是单纯推理题，而是需要：

```text
拆任务；
调用工具；
读取外部信息；
分析结果；
生成输出；
检查和修正。
```

Plan-and-Execute 能先建立全局视角，再逐步执行，并在执行过程中动态调整计划。

它比单纯 CoT 更适合工程中的复杂任务。

---

### 追问 5：Plan-and-Execute 有什么风险？

主要风险有三个：

```text
1. 初始计划可能不合理；
2. 执行结果可能和计划预期不一致；
3. 如果没有限制，重新规划可能陷入循环。
```

工程上通常需要：

```text
设置最大步骤数；
设置最大重试次数；
设置最大重新规划次数；
让每个 step 有明确完成条件；
记录每一步的执行状态和日志。
```

## 六、容易踩的坑

### 坑 1：把 CoT 等同于全部规划能力

错误说法：

```text
给 LLM 加规划能力就是在 prompt 里加一句“请一步步思考”。
```

正确理解：

```text
CoT 只是最基础的显式推理方式，不等于完整规划能力。
```

---

### 坑 2：只讲 ToT 更强，不讲成本

错误说法：

```text
ToT 比 CoT 更强，所以应该优先用 ToT。
```

正确理解：

```text
ToT 的效果可能更好，但调用成本也更高，通常是 CoT 的 3 到 5 倍。
```

---

### 坑 3：认为 GoT 很适合直接落地

错误说法：

```text
GoT 比 ToT 更强，所以生产系统应该用 GoT。
```

正确理解：

```text
GoT 表达能力强，但实现复杂，目前更多是学术思想，工程落地较少。
```

---

### 坑 4：Plan-and-Execute 只做 Plan，不做 Re-plan

错误做法：

```text
先生成计划，然后死板执行到底。
```

正确做法：

```text
每一步执行后都检查结果，必要时重新规划后续步骤。
```

---

## 七、记忆版总结

这篇文章可以压缩成一句话：

> LLM 的规划能力不是简单加一句 CoT，而是从 CoT 的单链推理，到 ToT 的多路径搜索，再到 GoT 的图结构复用；而工程里更常用的是 Plan-and-Execute，用 Planner 先生成计划，再由 Executor 执行，并通过 Re-planner 动态调整。

再进一步记：

```text
CoT：
- Chain of Thought
- 一条推理链
- 解决显式推理问题
- 缺点是单路径，走错容易一路错

ToT：
- Tree of Thoughts
- 多条推理路径
- 生成、评估、剪枝
- 解决 CoT 无法纠偏的问题
- 缺点是成本高，通常是 CoT 的 3 到 5 倍

GoT：
- Graph of Thoughts
- 图结构推理
- 支持中间结果复用和合并
- 表达能力强，但工程落地复杂

Plan-and-Execute：
- 先规划，再执行
- Planner 生成计划
- Executor 执行步骤
- Re-planner 动态调整
- 工程里更常用，适合复杂 Agent 任务
```

面试最重要的是记住：

```text
1. CoT 不等于规划能力的全部。
2. CoT、ToT、GoT 是从链到树再到图的演进。
3. ToT 要讲清楚成本，通常是 CoT 的 3 到 5 倍。
4. GoT 更偏学术思想，生产落地较少。
5. 工程里真正常用的是 Plan-and-Execute。
6. Plan-and-Execute 和 ReAct 可以组合：前者管全局规划，后者管单步执行。
```
