
## 一、面试标准答案

复杂任务做任务拆分，本质上是为了降低 LLM 一次性处理复杂任务时的认知负担。

我理解任务拆分主要解决三个问题：**上下文混乱、目标漂移、难以验证**。

因为 LLM 的 context window 是有限的，当一个任务太大时，中间状态会非常多，比如搜索结果、分析过程、临时结论、输出草稿都会堆在同一个上下文里。这样模型很容易忘记当前子目标，或者把不同阶段的任务混在一起，导致输出结构混乱、遗漏信息或者中途跑偏。

所以任务拆分的核心思想是：

> **把一个大任务拆成多个边界清晰的小任务，让 LLM 每一步只专注完成一件事。**

这样做有几个好处。

第一，**每一步更简单，准确率更高**。  
比如“写一份竞品分析报告”本身包含搜索资料、筛选信息、对比分析、生成报告等多个动作。如果一次性交给模型做，模型很容易混乱；拆开后，每一步只处理一个明确目标，出错概率会下降。

第二，**中间结果可以被单独检查和重试**。  
如果某一步搜索结果不完整，只需要重试搜索这一步，而不是从头重跑整个任务。这对工程系统非常重要。

第三，**拆分后可以分析依赖关系，做并行优化**。  
有些步骤必须串行执行，比如“先检索资料，再基于资料总结”；但有些步骤彼此独立，比如同时搜索竞品 A、竞品 B、竞品 C 的信息。对于独立步骤，可以并发执行，这样端到端延迟由关键路径决定，而不是所有步骤时间简单相加。

任务拆分主要有两种方式。

一种是 **静态拆分**，也就是开发者提前把流程写死，适合流程比较固定、业务规则比较明确的场景，比如固定的 RAG 流程、固定报告生成流程。它的优点是稳定、可控、好调试，缺点是灵活性差。

另一种是 **动态拆分**，也就是让 LLM 根据用户目标先生成执行计划，再一步步执行，本质上就是 Plan-and-Execute。它适合开放性强、任务形态变化大的复杂任务，比如深度调研、多文档分析、金融研报生成。优点是灵活，缺点是规划质量不稳定，需要配合校验、重试和 Replan 机制。

拆分时最关键的是控制粒度。任务不是拆得越细越好，也不是越粗越好。太粗会导致模型一次处理的信息太多，容易出错；太细会导致 LLM 调用次数变多、token 成本上升，而且步骤之间衔接可能变得生硬。

比较好的标准是 **原子操作**：

> **一个步骤只做一件边界清晰的事情，有明确输入、明确输出，并且能写出清晰的验收标准。**

比如“搜索竞品 A 的定价信息”就是一个比较原子的步骤；而“整理竞品分析”就太粗了，因为它里面包含搜索、筛选、对比、总结多个动作。

最后，拆分之后还要验证拆得好不好。我一般会看三个标准：**完备性、独立性、可验证性**。

- **完备性**：所有步骤加起来是否覆盖了原始任务的全部要求；
- **独立性**：每个步骤职责是否清晰，有没有重复或边界模糊；
- **可验证性**：每个步骤完成后是否能判断做得对不对，最好有验收标准。

如果执行过程中发现某一步太复杂，LLM 一次做不好，还可以进一步做 **自适应拆分**，也就是“做不好就继续拆”。如果执行结果导致原计划不再合理，还可以触发 **Replan**，重新调整后续计划。

所以完整回答可以总结为：

> **任务拆分不是简单把大任务切小，而是要结合静态拆分、动态拆分、依赖分析、并行执行、粒度控制和结果验证，最终让复杂任务更稳定、更可控、更容易调试，并且降低端到端延迟。**

## 三、为什么复杂任务要拆分？

复杂任务不拆分时，LLM 容易出现几个问题：

```text
1. 多个子任务混在一起
2. 中间状态太多，context 变乱
3. 当前子目标不清晰
4. 写着写着偏离原始目标
5. 某一步出错后无法单独修复
```

比如用户说：

```text
帮我写一份竞品分析报告。
```

这个任务看起来是一句话，其实至少包含：

```text
1. 搜索竞品信息
2. 筛选关键数据
3. 做功能对比
4. 分析优劣势
5. 总结结论和建议
6. 生成最终报告
```

如果一次性丢给 LLM，它会同时处理搜索、分析、写作、总结等多个目标，容易导致输出混乱。

拆分之后，每一步只做一件事：

```text
先搜集信息
再提炼关键点
再做对比
再写报告
最后校验
```

这样每一步的上下文更干净，模型更容易完成。

## 四、任务拆分的两种方式

## 1. 静态拆分

静态拆分就是开发者提前把流程设计好。

例如写技术博客，可以固定拆成：

```text
1. 搜索资料
2. 整理大纲
3. 分段撰写
4. 润色校对
```

它本质上更接近 Workflow。

### 优点

```text
1. 流程稳定
2. 行为可预测
3. 方便测试
4. 方便定位问题
5. 适合生产环境中流程固定的任务
```

### 缺点

```text
1. 灵活性不足
2. 遇到计划外情况容易卡住
3. 需要开发者提前设计所有步骤
```

### 适合场景

```text
1. 固定 RAG 流程
2. 固定报告模板生成
3. 固定审批流程
4. 固定客服流程
5. 文档处理流水线
```

---

## 2. 动态拆分

动态拆分就是把“怎么拆任务”交给 LLM。

也就是：

```text
用户目标
  ↓
LLM 先生成计划
  ↓
系统按计划执行
  ↓
最后汇总结果
```

这就是 Plan-and-Execute 的核心思想。

### 优点

```text
1. 灵活性更强
2. 可以根据不同任务生成不同计划
3. 适合开放性复杂任务
4. 不需要开发者提前穷举所有流程
```

### 缺点

```text
1. 规划质量不稳定
2. 初始计划错了，后续执行会被带偏
3. 需要额外的计划校验和重试机制
4. 成本和延迟比简单 Workflow 更高
```

### 适合场景

```text
1. 深度调研
2. 多文档分析
3. 金融研报生成
4. 竞品分析
5. 复杂问答
6. 开放式任务执行
```

## 五、任务拆分后最关键的优化：依赖分析和并行执行

这篇文章里一个很重要的点是：

> **任务拆分不是只为了提升准确率，还可以通过并行执行降低端到端延迟。**

拆分后，每个步骤之间不一定都有强依赖关系。

比如做竞品分析：

```text
1. 搜索竞品 A 信息
2. 搜索竞品 B 信息
3. 搜索竞品 C 信息
4. 汇总成对比表
5. 生成分析结论
```

其中：

```text
搜索竞品 A / B / C 信息
```

这三个步骤彼此独立，可以并行执行。

但是：

```text
汇总成对比表
```

必须等 A / B / C 的信息都搜完之后才能开始。

所以可以把步骤关系画成 DAG：

```text
搜索竞品 A ──┐
搜索竞品 B ──┼──> 汇总对比表 ──> 生成分析结论
搜索竞品 C ──┘
```

如果全部串行执行，总耗时是所有步骤相加。

如果能并行执行，总耗时主要由关键路径决定。

例如：

```text
搜索 A：3 秒
搜索 B：3 秒
搜索 C：3 秒
汇总：3 秒
生成结论：3 秒
```

串行执行：

```text
3 + 3 + 3 + 3 + 3 = 15 秒
```

并行执行：

```text
max(搜索 A, 搜索 B, 搜索 C) + 汇总 + 生成结论
= 3 + 3 + 3
= 9 秒
```

所以并行优化的核心不是减少每一步时间，而是：

> **缩短关键路径。**

## 六、任务拆分的粒度怎么控制？

任务不是拆得越细越好。

### 1. 拆得太粗的问题

如果拆得太粗，比如：

```text
整理竞品分析
```

这个步骤里面其实包含：

```text
1. 搜索竞品信息
2. 筛选关键信息
3. 做功能对比
4. 总结优劣势
```

这会导致模型一次性处理太多内容，仍然容易出错。

---

### 2. 拆得太细的问题

如果拆得太细，比如：

```text
1. 搜索竞品 A 官网
2. 搜索竞品 A 价格页
3. 搜索竞品 A 功能页
4. 搜索竞品 A 用户评价
5. 搜索竞品 A 版本更新
```

问题是：

```text
1. LLM 调用次数变多
2. token 消耗上升
3. 系统延迟变高
4. 步骤之间衔接变复杂
5. 最终汇总可能生硬
```

---

### 3. 合理粒度：原子操作

比较好的拆分标准是：

> **一个步骤只做一件独立的事，边界清晰，做完有明确输出。**

可以用一个判断方法：

> **这个步骤能不能写成一个清晰的函数签名？**

比如：

```python
search_competitor_pricing(competitor_name: str) -> PricingInfo
```

这个步骤就比较清晰。

但如果是：

```python
do_competitor_analysis(company: str) -> Report
```

这个函数太大，里面一定包含多个子任务，就应该继续拆。

---

## 七、自适应拆分：做不好就继续拆

这篇文章还有一个很重要的点：**自适应拆分**。

它的思想是：

> **不要一开始就把所有任务拆得特别细，而是在执行过程中根据效果决定要不要继续拆。**

流程可以理解为：

```text
先尝试执行当前任务
  ↓
如果执行效果好，继续下一步
  ↓
如果执行效果不好，把当前任务继续拆成更小子任务
  ↓
对子任务重复这个过程
```

比如：

```text
任务：写完整竞品分析报告
```

如果模型一次写不好，就拆成：

```text
1. 搜集竞品数据
2. 做功能对比
3. 写总结建议
```

如果“搜集竞品数据”这一步仍然太复杂，就继续拆：

```text
1. 搜集竞品 A 数据
2. 搜集竞品 B 数据
3. 搜集竞品 C 数据
```

这种方式的好处是：

```text
1. 简单任务不会被过度拆分
2. 复杂任务会自动拆得更细
3. 计算开销和任务实际难度更匹配
4. 只有做不好的节点才继续拆
```

可以把它理解成一棵递归展开的任务树：

```text
大任务
├── 子任务 1
├── 子任务 2
│   ├── 子任务 2.1
│   ├── 子任务 2.2
│   └── 子任务 2.3
└── 子任务 3
```

## 八、执行中的 Replan 机制

任务拆分后，并不是计划一旦生成就永远不变。

现实中执行某一步后，可能发现原计划不合理。

例如原计划是：

```text
1. 查询竞品 A 的定价
2. 查询竞品 B 的定价
3. 做定价对比
```

但第一步执行后发现：

```text
竞品 A 已经停止运营
```

这时继续做原来的定价对比就没意义了。

所以需要 Replan：

```text
执行一步
  ↓
检查当前结果是否影响剩余计划
  ↓
如果不影响，继续原计划
  ↓
如果影响，重新生成剩余步骤
```

Replan 的作用是：

> **让计划始终和最新执行结果保持一致，避免死守过时计划。**

但 Replan 也有成本，因为每次判断是否重规划都需要额外 LLM 调用。

所以工程上一般不会每一步都 Replan，而是设置触发条件：

```text
1. 当前步骤执行失败
2. 当前输出和预期差异很大
3. 工具返回异常
4. 发现关键前提不成立
5. 质量检查不通过
```

## 九、如何判断任务拆分是否合理？

这篇文章给了三个标准，非常适合面试回答。

## 1. 完备性

完备性指的是：

> **所有步骤加起来，是否覆盖了原始任务的全部要求。**

比如用户要求竞品分析包括：

```text
1. 市场份额
2. 产品功能对比
3. 定价策略
```

那么拆分结果中必须覆盖这三个维度。

如果少了“定价策略”，说明拆分不完备。

---

## 2. 独立性

独立性指的是：

> **每个步骤的职责边界是否清晰，有没有重复劳动。**

反例：

```text
步骤 1：搜索竞品 A 的产品功能
步骤 2：分析竞品 A 的核心能力
```

这两个步骤边界可能重叠，因为“产品功能”和“核心能力”很可能查到同一批内容。

更好的拆法是：

```text
步骤 1：搜索竞品 A 的全部公开资料
步骤 2：从资料中提取功能维度信息
步骤 3：基于功能维度做对比分析
```

这样每一步职责更清楚。

---

## 3. 可验证性

可验证性指的是：

> **每一步执行完之后，能不能判断它做得对不对。**

比如一个步骤叫：

```text
搜索竞品 A 的定价信息
```

最好给它配一个验收标准：

```text
输出必须包含：
1. 价格数字
2. 计费模式
3. 免费版/试用版信息
4. 信息来源
```

这样系统就可以自动检查：

```text
如果缺少价格数字，就触发重试；
如果缺少来源，就重新搜索；
如果计费模式不明确，就补充查询。
```

所以好的任务拆分应该是：

```text
步骤定义 + 验收标准
```

就像写代码时：

```text
函数实现 + 单元测试
```

## 十、结合金融分析 Agent 项目怎么说？

如果结合我的金融分析 Agent 项目，我会这样回答：

在金融分析 Agent 中，用户问题通常不是一个简单问答，而是一个复杂分析任务。

比如用户问：

```text
结合腾讯最近几个季度财报、相关新闻和历史研究记忆，分析它的业务趋势、增长原因和潜在风险。
```

这个任务如果一次性交给 LLM，很容易出现：

```text
1. 财报数据提取不完整
2. 多季度对比遗漏
3. 新闻和财报证据混在一起
4. 结论缺少证据支撑
5. 风险分析被弱化
```

所以我会先做任务拆分：

```text
1. 识别公司实体和时间范围
2. 检索相关季度财报
3. 提取收入、利润、毛利率、费用等核心指标
4. 做多季度趋势对比
5. 检索相关新闻和行业背景
6. 查询 Research Memory 中的历史研究结论
7. 分析增长原因
8. 总结潜在风险
9. 汇总成结构化中文研报
10. 对最终报告做质量检查
```

其中有些步骤可以并行：

```text
财报检索
新闻检索
Research Memory 查询
```

这三类信息来源彼此独立，可以同时跑。

但是：

```text
增长原因分析
风险总结
最终研报生成
```

这些步骤必须等前面的证据都准备好之后再执行。

所以整体可以设计成：

```text
识别公司和时间范围
  ↓
并行执行：
  ├── 财报 RAG 检索
  ├── 新闻搜索
  └── Research Memory 查询
  ↓
多季度指标对比
  ↓
增长原因分析
  ↓
风险总结
  ↓
结构化研报生成
  ↓
Reflection 质量检查
```

这样做的好处是：

```text
1. 每一步职责清晰
2. 中间结果可以单独验证
3. 某一步失败可以单独重试
4. 可并行步骤能降低端到端延迟
5. 最终输出更稳定、更可追溯
```

## 十一、面试官可能追问：为什么拆分后准确率会提升？

可以这样回答：

拆分后准确率提升，不只是因为“任务变简单了”，更本质的原因是降低了上下文混乱程度。

LLM 的 context window 虽然可以容纳很多 token，但它同时追踪多个子目标、中间结论和输出要求时，容易出现注意力分散。

一个大任务里可能同时包含搜索、分析、总结、格式化等多个动作，如果一次完成，模型会把这些状态混在一起，导致遗漏和跑偏。

拆分后，每一步只保留当前步骤需要的信息，模型只需要完成一个明确子目标，所以：

```text
1. 当前目标更清楚
2. 上下文更干净
3. 输出格式更容易控制
4. 中间结果可以检查
5. 错误可以局部重试
```

因此整体准确率和稳定性都会提升。

## 十二、面试官可能追问：静态拆分和动态拆分怎么选？

可以这样回答：

如果任务流程固定、业务规则明确，我会优先选择静态拆分。

比如固定的 PDF 解析、文档切块、向量入库、RAG 检索、答案生成，这些流程不需要每次让 LLM 重新规划，直接写成 Workflow 更稳定。

如果任务开放性强、用户输入变化大、开发者很难提前穷举所有路径，我会选择动态拆分，也就是让 LLM 先规划步骤，再执行。

简单说：

```text
流程固定、稳定性优先：静态拆分
任务开放、灵活性优先：动态拆分
```

实际项目中通常会混合使用：

```text
主链路用静态 Workflow
复杂分析节点用动态 Plan-and-Execute
关键输出前加 Reflection
```

---

## 十三、面试官可能追问：任务拆得越细越好吗？

可以这样回答：

不是。

任务拆得太粗，LLM 一步要处理太多事情，还是容易出错。

但拆得太细，也会带来问题：

```text
1. LLM 调用次数增加
2. token 成本上升
3. 端到端延迟变高
4. 中间结果汇总变复杂
5. 每一步缺少全局视角，衔接可能生硬
```

所以拆分要控制粒度。

我的标准是：

> **每一步应该是一个原子操作，只做一件边界清晰的事，有明确输入、明确输出，并且可以被验证。**

如果一个步骤可以写成清晰的函数签名，通常说明粒度比较合适。

---

## 十四、面试官可能追问：并行执行为什么能降低延迟？

可以这样回答：

任务拆分后，有些步骤之间没有依赖关系，这些步骤就可以并行执行。

并行执行降低的不是每一步本身的执行时间，而是降低整体任务的关键路径时间。

比如有三个独立搜索任务：

```text
搜索竞品 A：3 秒
搜索竞品 B：3 秒
搜索竞品 C：3 秒
```

如果串行执行，需要：

```text
3 + 3 + 3 = 9 秒
```

如果并行执行，只需要：

```text
max(3, 3, 3) = 3 秒
```

然后再进入后续汇总步骤。

所以拆分后要分析依赖关系，把任务组织成 DAG。没有依赖的节点并发跑，有依赖的节点等待父节点完成。

## 十五、面试官可能追问：如何判断拆分结果好不好？

可以这样回答：

我会用三个标准判断：**完备性、独立性、可验证性**。

第一是完备性，也就是所有步骤加起来是否覆盖原始任务的全部要求，不能遗漏用户要求的任何关键维度。

第二是独立性，也就是每个步骤的职责边界是否清楚，步骤之间不要重复做同一件事，也不要出现边界模糊。

第三是可验证性，也就是每个步骤执行完之后，是否能判断它做得对不对。最好在拆分时就为每一步定义验收标准，比如输出必须包含哪些字段、引用哪些证据、满足什么格式。

如果一个拆分结果同时满足这三点，就说明它比较适合工程落地。

## 十六、最终背诵版

复杂任务做任务拆分，本质上是为了降低 LLM 一次性处理复杂任务时的认知负担。

因为 LLM 的 context window 有限制，任务越大，中间状态越多，模型越容易在搜索、分析、写作、总结等多个子目标之间混乱，导致遗漏信息、结构混乱或者中途跑偏。

所以任务拆分的核心思想是：**把一个大任务拆成多个边界清晰的小任务，让 LLM 每一步只专注完成一件事。**

任务拆分主要有两种方式。

第一种是 **静态拆分**，也就是开发者提前把步骤写死，适合流程固定、规则明确的场景，比如固定 RAG 链路、固定文档处理流程。它的优点是稳定、可控、好调试，缺点是灵活性不足。

第二种是 **动态拆分**，也就是让 LLM 根据用户目标先生成执行计划，再按计划执行，本质上就是 Plan-and-Execute。它适合开放性强、流程变化大的复杂任务，比如深度调研、多文档分析、金融研报生成。它的优点是灵活，缺点是规划质量不稳定，需要配合校验、重试和 Replan。

拆分时要注意粒度，不是越细越好，也不是越粗越好。太粗会导致模型仍然一次处理太多信息，太细会增加 LLM 调用次数、token 成本和系统延迟。比较好的标准是原子操作：每一步只做一件边界清晰的事，有明确输入、明确输出，并且能被验证。

拆分之后还要做依赖分析，把步骤组织成 DAG。没有依赖的步骤可以并行执行，有依赖的步骤必须等待父节点完成。这样端到端延迟不再是所有步骤时间简单相加，而是由关键路径决定。

最后，要判断拆分是否合理，可以看三个标准：完备性、独立性、可验证性。完备性保证不遗漏原始任务要求；独立性保证步骤之间职责清晰、不重复；可验证性保证每一步执行完后可以自动检查、失败重试。

所以完整的任务拆分方案不只是“把大任务切小”，而是包括：

```text
1. 静态拆分或动态拆分
2. 合理控制拆分粒度
3. 分析步骤依赖关系
4. 对无依赖步骤并行执行
5. 给每一步设置验收标准
6. 必要时做自适应拆分和 Replan
```

这样才能真正提升复杂任务的稳定性、准确率、可调试性和执行效率。

---

## 十七、一句话总结

> **任务拆分的核心是把复杂任务拆成可执行、可验证、可并行的小步骤；静态拆分保证稳定，动态拆分保证灵活，依赖分析提升效率，验收标准保证质量。**
