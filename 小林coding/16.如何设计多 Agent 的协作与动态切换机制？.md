## 一、面试标准回答

如果面试官问：

> 多 Agent 系统里，各个 Agent 之间怎么协作？下一步该调用哪个 Agent，又是怎么动态切换的？

可以这样回答：

我理解多 Agent 系统里有两个核心问题：

```text
1. Agent 之间怎么传递信息？
2. 下一步应该切换到哪个 Agent？
```

第一个问题是 **协作机制**，第二个问题是 **路由机制**。

在协作机制上，常见方式主要有两种：**消息传递** 和 **共享状态**。

**消息传递** 可以理解成 Agent 之间发消息。一个 Agent 完成自己的任务后，把结果发送出去，下游 Agent 订阅或接收对应消息后继续处理。它的核心优势是 **解耦**，发送方不需要知道谁会消费这个结果，接收方也不一定要知道是谁发来的。它比较适合多个 Agent 独立运行、并行处理、互相之间依赖不强的场景。但缺点是需要消息队列或事件机制来维护通信链路，系统复杂度会更高。

**共享状态** 可以理解成多个 Agent 共同读写一个状态对象。比如系统维护一个全局 `State`，里面记录用户原始任务、当前进度、各 Agent 的中间产物和最终输出。每个 Agent 执行完后，把自己的结果写入 State，后续 Agent 再从 State 中读取。它的核心优势是 **直接**，适合步骤之间依赖关系很强、流程比较清晰的任务。比如 LangGraph 的设计就是这种思路，每个节点读取 State，并返回增量更新。

这两种方式怎么选，主要看 Agent 之间的依赖强不强：

```text
依赖强、流程清晰：优先共享状态。
依赖弱、需要解耦并行：优先消息传递。
```

第二个问题是动态切换，也就是 **路由**。路由一般由 Orchestrator 来做，决定下一步应该调用哪个 Agent。

路由策略也有两种：**静态路由** 和 **动态路由**。

**静态路由** 是提前写好规则，比如任务包含“搜索”就交给 Researcher Agent，代码生成完成后就交给 Reviewer Agent。它的优点是稳定、可预测、好调试，缺点是只能覆盖提前设计好的路径，遇到边缘情况时不够灵活。

**动态路由** 是让 LLM 根据当前任务状态、已完成步骤和可用 Agent 列表，实时判断下一步该调用谁。它的优点是灵活，可以处理未预料到的情况；缺点是每次路由都要多一次 LLM 调用，增加成本和延迟，而且 LLM 可能路由错，系统行为可预测性会下降。

所以工程上更稳健的做法不是全靠动态路由，而是 **静态路由和动态路由混合使用**：

```text
主流程用静态路由，保证稳定可控；
边缘情况或规则无法覆盖时，再交给 LLM 动态决策。
```

此外，还有一种更去中心化的切换方式叫 **Handoff**。它不是由中央 Orchestrator 决定下一步，而是由当前 Agent 自己判断任务完成后应该交给哪个 Agent。它适合 Agent 职责边界非常清晰、任务流向相对确定的场景。但如果 Agent 数量多、流程复杂，就容易出现交接混乱甚至循环，所以必须设计防循环机制，比如记录任务经过过哪些 Agent，并限制最大交接次数。

总结来说：

> 多 Agent 协作不是简单地“一个 Agent 做完传给下一个”，而是要根据依赖关系选择消息传递或共享状态；动态切换也不是全靠 LLM 决策，而是主流程静态路由保稳定，边缘情况动态路由补灵活，必要时再用 Handoff 做去中心化交接。

## 二、文章核心观点

这篇文章的核心观点是：

```text
多 Agent 系统不只是“多个 Agent 分工”，还必须设计清楚：
1. Agent 之间怎么传信息；
2. Agent 之间怎么切换；
3. 谁来决定下一步调用哪个 Agent。
```

最重要的两个关键词是：

```text
协作机制：消息传递 vs 共享状态
切换机制：静态路由 vs 动态路由
```

工程上的推荐做法是：

```text
通信方式按依赖关系选；
路由方式用静态 + 动态混合；
复杂系统优先 Orchestrator 统一调度。
```

## 三、关键知识点

### 1. 多 Agent 协作的三种模式

文章先提到多 Agent 协作常见有三类：

```text
1. 流水线模式
2. 层级模式
3. 协商模式
```

简单理解：

```text
流水线模式：
Agent 按固定顺序依次执行，前一个结果交给后一个。

层级模式：
Orchestrator 负责分配任务，Worker Agent 负责执行。

协商模式：
多个 Agent 之间互相沟通、辩论、达成一致。
```

实际工程中这三种不是互斥的，复杂系统里经常会混合使用。

---

### 2. 消息传递：核心是解耦

消息传递可以理解成：

```text
Agent A 完成任务后，把结果发出去；
Agent B 监听到相关消息后，再继续处理。
```

它的特点是：

```text
发送方不需要知道接收方是谁；
接收方不需要强依赖发送方；
多个 Agent 可以独立运行；
适合事件驱动、并行、解耦场景。
```

优点：

```text
解耦性强；
适合多个 Agent 并行协作；
系统扩展性较好。
```

缺点：

```text
需要消息队列或事件机制；
部署和调试成本更高；
消息顺序、失败重试、重复消费都要处理。
```

适合场景：

```text
Agent 之间依赖较弱；
多个 Agent 可以独立执行；
希望系统扩展性更强。
```

---

### 3. 共享状态：核心是直接

共享状态可以理解成：

```text
所有 Agent 都读写同一个 State。
```

例如：

```text
State = {
  用户原始任务,
  当前执行进度,
  搜索 Agent 的结果,
  分析 Agent 的结果,
  写作 Agent 的结果,
  错误信息,
  最终输出
}
```

每个 Agent 做完后，把结果写回 State，后续 Agent 直接读取。

它的特点是：

```text
信息传递直接；
流程更清晰；
适合强依赖的流水线任务；
LangGraph 就是这种思想。
```

优点：

```text
实现直观；
适合前后步骤依赖明确的任务；
便于统一记录任务状态。
```

缺点：

```text
多个 Agent 同时读写时容易出问题；
可能出现覆盖、脏数据、状态污染；
需要设计好 State 结构和写入规则。
```

适合场景：

```text
Agent 执行顺序明确；
前一步输出直接影响后一步；
需要统一管理任务状态。
```

---

### 4. 共享状态的设计要点

共享状态最容易出 bug，所以要注意三个点。

#### 1. 状态结构要分层

一般分成：

```text
全局状态：
所有 Agent 都需要的信息，比如用户任务、当前进度、最终输出。

局部状态：
单个 Agent 自己用的中间结果，比如候选文档、草稿代码、临时分析。
```

这样可以避免所有中间信息都污染全局上下文。

---

#### 2. 写入规则要明确

比较稳的方式是：

```text
只追加，不覆盖。
```

也就是每个 Agent 不直接修改已有字段，而是返回增量更新。

这样可以避免：

```text
Agent A 写入的结果被 Agent B 覆盖；
状态被意外改坏；
难以追踪历史变化。
```

---

#### 3. 错误状态也要写入

如果某个 Agent 执行失败，不能悄悄吞掉错误。

应该把错误写入 State：

```text
error_agent
error_message
error_step
retry_count
```

这样 Orchestrator 或后续 Agent 才能判断：

```text
是否重试；
是否换 Agent；
是否跳过；
是否终止任务。
```

---

### 5. 静态路由：稳定可控

静态路由就是提前写规则：

```text
搜索任务 → Researcher Agent
代码任务 → Coder Agent
代码完成 → Reviewer Agent
没有匹配规则 → Orchestrator 兜底
```

优点：

```text
稳定；
可预测；
成本低；
好调试；
适合主流程。
```

缺点：

```text
不够灵活；
只能覆盖预先设计好的路径；
遇到边缘情况可能不知道下一步怎么走。
```

一句话记忆：

```text
静态路由适合确定性主流程。
```

---

### 6. 动态路由：灵活但不可控

动态路由是让 LLM 根据当前状态决定下一步调用哪个 Agent。

典型输入包括：

```text
当前任务是什么；
已经完成了什么；
当前 State 里有什么信息；
可用 Agent 有哪些；
每个 Agent 的职责是什么。
```

LLM 输出：

```text
下一步应该调用哪个 Agent。
```

优点：

```text
灵活；
能处理边缘情况；
可以应对没有提前写死的任务路径。
```

缺点：

```text
每次路由都要多一次 LLM 调用；
增加成本和延迟；
LLM 可能选错 Agent；
系统行为可预测性下降。
```

一句话记忆：

```text
动态路由适合规则覆盖不到的边缘情况。
```

---

### 7. 工程推荐：静态路由 + 动态路由混合

文章最重要的工程结论是：

```text
不要全靠 LLM 动态路由。
```

更稳的做法是：

```text
主流程用静态路由；
规则无法覆盖时才用动态路由。
```

也就是：

```text
静态路由负责稳定性；
动态路由负责灵活性。
```

这样既能保证大部分任务可控，又能处理少数非标准场景。

---

### 8. Handoff：Agent 之间的接力棒

Handoff 是一种更去中心化的切换方式。

它的核心是：

```text
不是 Orchestrator 决定下一步；
而是当前 Agent 自己决定把任务交给谁。
```

可以类比成接力赛：

```text
当前 Agent 跑完自己这一棒；
自己判断下一棒应该交给哪个 Agent；
然后直接交接。
```

优点：

```text
没有中央调度瓶颈；
每个 Agent 更清楚自己的任务边界；
简单场景下更轻量。
```

缺点：

```text
缺少全局视角；
容易交接错误；
可能出现 Agent 之间来回踢皮球；
可能陷入死循环。
```

所以 Handoff 必须加防护：

```text
记录任务经过过哪些 Agent；
限制最大 Handoff 次数；
检测重复交接；
必要时回到 Orchestrator 兜底。
```

适合场景：

```text
Agent 数量不多；
职责边界非常清晰；
任务流向相对确定。
```

不适合场景：

```text
Agent 很多；
任务流向复杂；
需要全局调度；
错误代价较高。
```

## 四、工程上怎么用？

这篇文章的工程建议可以压缩成三条。

### 1. 通信方式按依赖关系选择

```text
如果是清晰流水线：
用共享状态。

如果是多个 Agent 独立并行：
用消息传递。
```

简单判断：

```text
前一步结果是否直接决定后一步？
是：共享状态。
否：消息传递。
```

---

### 2. 路由方式不要全靠 LLM

推荐：

```text
主流程：静态路由
边缘情况：动态路由
异常情况：fallback Agent 或 Orchestrator 兜底
```

不要这样：

```text
每一步都让 LLM 判断下一个 Agent。
```

因为这样会导致：

```text
成本高；
延迟高；
行为不可预测；
调试困难。
```

---

### 3. Handoff 只适合简单清晰的 Agent 网络

如果 Agent 数量少、职责边界清楚，可以用 Handoff。

如果任务复杂，优先用：

```text
Orchestrator 统一调度。
```

因为 Orchestrator 更容易保证：

```text
任务完整性；
执行顺序；
错误感知；
最终结果汇总。
```

## 五、面试官可能追问

### 追问 1：消息传递和共享状态有什么区别？

可以回答：

消息传递的核心是 **解耦**，Agent 完成任务后把结果发出去，下游 Agent 订阅消息后处理，发送方不需要知道接收方是谁。

共享状态的核心是 **直接**，多个 Agent 共同读写同一个 State，前一个 Agent 把结果写进去，后一个 Agent 直接读取。

选择标准是：

```text
依赖强、流程清晰：共享状态；
依赖弱、需要并行解耦：消息传递。
```

---

### 追问 2：共享状态怎么避免多个 Agent 互相覆盖？

可以回答：

主要靠三个设计：

```text
1. 状态分层：全局状态和局部状态分开。
2. 增量更新：每个 Agent 只追加自己的结果，不直接覆盖已有字段。
3. 错误显式记录：失败信息也写入 State，方便后续决策。
```

这样可以减少状态污染、覆盖和脏数据问题。

---

### 追问 3：静态路由和动态路由怎么选？

可以回答：

静态路由适合主流程，因为它稳定、可预测、成本低、好调试。

动态路由适合边缘情况，因为它灵活，能处理规则没覆盖到的路径。

工程上一般混合使用：

```text
确定性流程用静态路由；
规则兜不住时再交给 LLM 动态路由。
```

---

### 追问 4：为什么不能每一步都用 LLM 动态路由？

因为每一步动态路由都会增加一次 LLM 调用，带来：

```text
成本增加；
延迟增加；
路由错误风险；
系统行为不可预测；
调试困难。
```

所以动态路由不适合当作默认主流程，更适合处理规则覆盖不到的边缘情况。

---

### 追问 5：Handoff 和 Orchestrator 有什么区别？

Orchestrator 是中心化调度：

```text
由统一调度者决定下一步调用谁。
```

Handoff 是去中心化交接：

```text
由当前 Agent 自己决定把任务交给谁。
```

Handoff 更轻量，但缺少全局视角，容易出现循环交接。

所以：

```text
Agent 少、职责清楚：可以 Handoff；
Agent 多、流程复杂：优先 Orchestrator。
```

## 六、容易踩的坑

### 坑 1：把协作简单理解成“传结果”

错误说法：

```text
一个 Agent 做完，把结果传给下一个 Agent。
```

问题是：

```text
没有说明具体怎么传，也没有区分消息传递和共享状态。
```

正确回答：

```text
Agent 协作要区分消息传递和共享状态，两者适用场景不同。
```

---

### 坑 2：全靠 LLM 动态路由

错误做法：

```text
每一步都让 LLM 判断下一个 Agent。
```

问题是：

```text
成本高、延迟高、不可预测、容易路由错。
```

正确做法：

```text
主流程静态路由，边缘情况动态路由。
```

---

### 坑 3：共享状态没有写入规则

错误做法：

```text
所有 Agent 随便修改同一个 State。
```

问题是：

```text
容易互相覆盖、产生脏数据、状态不可追踪。
```

正确做法：

```text
状态分层，尽量增量追加，不随便覆盖。
```

---

### 坑 4：忽略错误状态

错误做法：

```text
Agent 失败后只返回空结果，或者直接吞掉异常。
```

正确做法：

```text
把错误信息写入 State，让 Orchestrator 或后续 Agent 能感知并处理。
```

---

### 坑 5：Handoff 不做防循环

错误做法：

```text
Agent 自己决定交给谁，但不记录交接路径。
```

问题是：

```text
可能 A 交给 B，B 交给 C，C 又交回 A，形成死循环。
```

正确做法：

```text
记录经过的 Agent，设置最大交接次数，重复交接时终止或回到 Orchestrator。
```

## 七、记忆版总结

这篇文章可以压缩成一句话：

> 多 Agent 协作机制主要解决“信息怎么传”和“下一步叫谁”的问题；信息传递上有消息传递和共享状态，路由切换上有静态路由和动态路由，工程上最稳的是主流程静态路由，边缘情况动态路由，复杂系统优先 Orchestrator 统一调度。

再进一步记：

```text
协作机制：
1. 消息传递
   - 核心是解耦
   - 适合依赖弱、并行、事件驱动场景

2. 共享状态
   - 核心是直接
   - 适合依赖强、流程清晰、流水线场景
   - 要注意状态分层、增量更新、错误写入

切换机制：
1. 静态路由
   - 规则写死
   - 稳定、低成本、好调试
   - 适合主流程

2. 动态路由
   - LLM 判断下一步调用谁
   - 灵活，但成本高、不可预测
   - 适合边缘情况

Handoff：
- 当前 Agent 自己决定交给谁
- 适合 Agent 少、职责清楚的场景
- 必须防循环

工程最佳实践：
主流程静态路由保稳定；
边缘情况动态路由保灵活；
复杂任务用 Orchestrator 统一调度；
简单清晰任务可以考虑 Handoff。
```

面试最重要的是记住：

```text
1. 协作不要只说“传结果”，要区分消息传递和共享状态。
2. 切换不要只说“LLM 动态决策”，要区分静态路由和动态路由。
3. 工程上不要全靠动态路由，主流程静态路由，边缘情况动态路由。
4. 共享状态要设计好 State 结构、写入规则和错误处理。
5. Handoff 可以用，但必须防止循环交接。
```
