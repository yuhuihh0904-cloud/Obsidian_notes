## 一、面试标准答案

Agent 常见的推理模式可以分成几类：**直接回答、CoT、ReAct、Plan-and-Execute**。

最简单的是 **直接回答**，也就是用户输入问题后，LLM 不显式展开中间过程，直接生成最终答案。这种方式适合简单问答，但遇到多步推理或者需要外部信息的任务时，容易出错。

**CoT，也就是 Chain of Thought，思维链**，是在回答前让模型先展开中间推理步骤，再给出最终答案。它的作用是把模型原本隐式的推理过程显式写出来，降低多步推理中的错误累积。但 CoT 的问题是，它本质上还是纯文本推理，不能主动调用工具，也拿不到实时数据。

**ReAct，全称是 Reasoning and Acting**，是在 CoT 的基础上加入了“行动”。它让模型按照：

```text
Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

这样的循环推进任务。

其中：

- **Thought** 是模型的推理；
- **Action** 是模型决定要调用的工具；
- **Observation** 是工具执行后返回的结果；
- **Final Answer** 表示模型认为任务完成，可以输出最终答案。

ReAct 的关键在于，它不是只让模型在脑子里推理，而是让模型一边推理，一边调用外部工具获取信息，再根据工具结果继续推理。因此它适合需要搜索、查数据库、调用 API、执行计算等任务。

但是这里有一个非常重要的点：**ReAct 的循环不是模型自己跑起来的，而是由代码框架驱动的。**

LLM 每次只负责根据当前历史输出下一步的 Thought 和 Action。代码负责解析模型输出，判断是否出现 Final Answer；如果没有，就执行对应工具，把工具返回结果作为 Observation 追加到历史中，然后再次调用 LLM。也就是说：

```text
模型负责推理和决策
代码负责循环、解析、执行工具、追加 Observation、控制终止条件
```

ReAct 的问题是，它本质上是“走一步看一步”的前向推理模式，缺少全局规划，所以复杂任务中容易出现两个问题：

1. **循环漂移**：做着做着偏离原始目标；
2. **错误传播**：早期某一步工具结果或判断错了，后续推理会被带偏。

为了解决 ReAct 缺少全局规划的问题，就出现了 **Plan-and-Execute**。

**Plan-and-Execute 的核心思想是：先规划，再执行。**

它把任务分成两个阶段：

1. **Planner 阶段**：先让 LLM 从全局视角拆解任务，生成结构化计划；
2. **Executor 阶段**：再按计划一步步执行，每一步内部可以继续使用 ReAct 调工具。

这样做的好处是：整体任务有全局路线，不容易像纯 ReAct 那样跑偏；同时每一步执行时仍然可以保留 ReAct 的灵活工具调用能力。

实际工程中常见的组合是：

```text
Plan-and-Execute 负责全局规划
ReAct 负责单步执行和工具调用
Workflow 负责整体流程控制
Tools 负责具体能力执行
```

如果任务简单、步骤少，直接用 ReAct 就够了；如果任务复杂、步骤多、依赖关系强，比如深度调研、多文档分析、金融研报生成，就更适合用 Plan-and-Execute，甚至在关键步骤上再叠加 Reflection 做质量检查。

## 二、这篇博客的核心观点

这篇博客主要讲了四个重点：

1. **为什么需要 Agent 推理模式**
2. **CoT 和 ReAct 的区别**
3. **ReAct 到底是怎么实现的**
4. **Plan-and-Execute 为什么能解决 ReAct 的部分问题**

其中最重要的面试点是：

> **ReAct 不是模型自己在循环，而是代码框架驱动循环。模型每次只输出下一步，代码负责解析、执行工具、追加 Observation，并再次调用模型。**

这个点非常容易被问到，也很容易答错。

很多人会说：

> ReAct 就是模型自己思考、自己调用工具、自己循环，直到完成任务。

这个说法不准确。

更准确的说法是：

> ReAct 的“智能决策”来自模型，但“循环机制”来自代码框架。模型不是一个会自动运行的程序，它每次只会根据输入生成一段文本或结构化工具调用。真正让 ReAct 一轮一轮跑起来的是外层代码。

## 三、为什么需要推理模式？

LLM 的本质是根据上下文一个 token 一个 token 地预测下一个 token。

对于简单问题，它可以直接回答。

但是对于复杂问题，如果让它直接给最终答案，中间推理过程就是隐式的，模型可能在某一步暗中出错，最后输出一个看似合理但其实错误的答案。

这有点像人做数学题：

```text
心算复杂乘法，容易出错；
把每一步写下来，错误率会下降。
```

所以推理模式的作用是：

> **把模型原本隐式的中间推理显式化，让后续生成建立在已经写出来的中间结果上，从而减少多步推理中的错误累积。**

这就是 CoT、ReAct、Plan-and-Execute 这些范式出现的原因。

---

## 四、CoT 是什么？

### 1. CoT 的定义

**CoT，全称 Chain of Thought，思维链。**

它的核心思想是：

> **让模型先写出中间推理步骤，再输出最终答案。**

比如在 Prompt 中加入：

```text
让我们一步一步思考。
```

模型就会倾向于先展开推理，再给结论。

---

### 2. CoT 为什么有效？

因为 LLM 是顺序生成的。

模型先生成的内容会进入上下文，成为后续生成的依据。

比如模型先写出：

```text
第一步：A 是 B 的 1.2 倍，所以 A > B。
```

那么下一步生成时，它就可以基于这条已经显式写出来的结论继续推理，而不是在隐式状态里“记住”这个中间结论。

所以 CoT 本质上是把推理过程变成上下文的一部分。

---

### 3. CoT 的两种形式

#### 3.1 Zero-shot CoT

Zero-shot CoT 就是不提供示例，只在 Prompt 中加一句：

```text
让我们一步一步思考。
```

优点是简单、零成本、即插即用。

缺点是输出格式和推理深度不稳定，模型可能有时候推得很细，有时候又跳步。

---

#### 3.2 Few-shot CoT

Few-shot CoT 是在 Prompt 中提供几个带完整推理过程的例子，让模型模仿这种格式。

优点是格式更稳定，适合对输出结构有要求的场景。

缺点是需要准备高质量示例，而且示例会占用 token。

---

### 4. CoT 的局限

CoT 的问题在于：

> **它只能做纯文本推理，不能和外部世界交互。**

也就是说，它不能：

- 搜索实时信息；
- 查询数据库；
- 调用 API；
- 执行代码；
- 使用计算器；
- 读取外部系统状态。

比如你问：

```text
现在苹果公司的市值是多少？
```

CoT 再怎么一步步推理，也无法拿到最新数据。

所以 CoT 适合模型内部知识和逻辑推理，但不适合需要外部信息或真实操作的任务。

这就引出了 ReAct。

---

## 五、ReAct 是什么？

### 1. ReAct 的定义

**ReAct = Reasoning + Acting。**

它的核心思想是：

> **把推理和行动交织在一起，让模型边思考、边调用工具、边根据工具结果继续推理。**

典型流程是：

```text
Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

---

### 2. ReAct 每个环节是什么意思？

#### 2.1 Thought

Thought 是模型的思考过程。

它负责分析当前任务状态，比如：

- 用户到底想要什么？
- 当前信息是否足够？
- 下一步应该做什么？
- 应该调用哪个工具？
- 是否已经可以输出最终答案？

---

#### 2.2 Action

Action 是模型决定采取的行动，通常是调用某个工具。

比如：

```text
Action: search
Action Input: 苹果公司 2024 年市值
```

Action 表示模型决定调用搜索工具，并传入搜索关键词。

---

#### 2.3 Observation

Observation 是工具执行后的返回结果。

比如：

```text
Observation: 苹果公司 2024 年市值约为 3.5 万亿美元
```

注意，Observation 不是模型自己生成的，而是系统调用工具后填回来的结果。

---

#### 2.4 Final Answer

Final Answer 表示模型认为任务完成，输出最终答案。

它也是 ReAct 循环的终止信号。

---

## 六、ReAct 和 CoT / Act-only 的区别

### 1. ReAct vs CoT

CoT 只有推理，没有行动。

```text
CoT：Thought → Thought → Thought → Final Answer
```

ReAct 有推理，也有工具调用。

```text
ReAct：Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

所以 CoT 适合内部推理任务，而 ReAct 更适合需要外部工具的任务。

比如：

- 最新新闻查询；
- 股票行情查询；
- 数据库检索；
- 文件读取；
- 代码执行；
- API 调用。

---

### 2. ReAct vs Act-only

Act-only 是只行动不推理。

```text
Action → Observation → Action → Observation → Final Answer
```

这种方式看起来效率高，但问题是中间没有 Thought 来解释为什么这么做。

一旦某一步工具结果不相关，后续动作就容易被带偏。

ReAct 的优势在于：

> **Thought 给 Action 提供方向，Action 给 Thought 提供事实反馈。**

也就是说：

```text
推理为行动提供方向
行动为推理提供外部事实
```

这就是 ReAct 比纯 CoT 和纯 Act-only 更适合 Agent 的原因。

---

## 七、ReAct 的具体实现原理

### 1. 最容易答错的点

面试中最容易答错的是：

> ReAct 的循环是不是模型自己在跑？

答案是：

> **不是。ReAct 的循环是代码框架驱动的，模型每次只负责输出下一步。**

LLM 本身不会自动进入循环。

它每次调用只会根据输入生成一段输出。

真正的 ReAct 循环由外层代码完成。

---

### 2. ReAct 的实现分工

可以这样理解：

```text
LLM 负责：
1. 读取用户问题和历史记录；
2. 生成 Thought；
3. 决定 Action；
4. 生成 Action Input；
5. 判断是否输出 Final Answer。

代码框架负责：
1. 构造 Prompt；
2. 保存历史记录；
3. 调用 LLM；
4. 解析 LLM 输出；
5. 判断是否出现 Final Answer；
6. 根据 Action 调用对应工具；
7. 把工具结果作为 Observation 追加回历史；
8. 控制最大轮数、超时和异常处理。
```

一句话总结：

> **模型负责想，代码负责跑。**

---

### 3. 经典 ReAct Prompt 格式

经典 ReAct 通常通过 Prompt 约束模型输出格式：

```text
你是一个 AI 助手，可以使用以下工具：

- search(query): 搜索互联网获取最新信息
- calculator(expr): 计算数学表达式
- query_database(sql): 查询数据库

回答时请严格按照以下格式：

Thought: 你的思考过程
Action: 工具名称
Action Input: 工具输入
Observation: 这里由系统填入工具返回结果

以上 Thought / Action / Observation 可以重复多轮。

当你确定可以回答时，请输出：

Final Answer: 最终答案
```

---

### 4. ReAct 的伪代码

```python
def react_agent(question: str, tools: dict, max_steps: int = 10):
    prompt = build_react_prompt(question, tools)
    history = []

    for step in range(max_steps):
        # 1. 调用 LLM，让它根据当前历史输出 Thought + Action 或 Final Answer
        response = llm.generate(prompt + "\n".join(history))

        # 2. 如果模型输出 Final Answer，说明任务完成
        if "Final Answer:" in response:
            return response.split("Final Answer:")[-1].strip()

        # 3. 解析模型输出中的 Action 和 Action Input
        action, action_input = parse_action(response)

        # 4. 执行工具
        if action in tools:
            observation = tools[action](action_input)
        else:
            observation = f"工具 {action} 不存在，请选择可用工具"

        # 5. 把模型输出和工具结果都追加到历史中
        history.append(response)
        history.append(f"Observation: {observation}")

    return "超过最大步数，任务未完成"
```

这段伪代码体现了 ReAct 最核心的机制：

```text
调用 LLM
  ↓
解析 Action
  ↓
执行工具
  ↓
追加 Observation
  ↓
再次调用 LLM
```

---

### 5. 现代 Function Calling 和 ReAct 的关系

早期 ReAct 主要靠文本格式解析：

```text
Action: search
Action Input: xxx
```

然后代码用正则或者字符串解析出工具名和参数。

现代 LLM 通常支持 **Function Calling / Tool Use**，模型可以直接输出结构化 JSON 工具调用。

比如：

```json
{
  "tool_name": "search",
  "arguments": {
    "query": "苹果公司 2024 年市值"
  }
}
```

这样实现更稳定，因为不用依赖模型严格输出 `Action: xxx` 这种文本格式。

但本质没有变：

> **ReAct 的核心仍然是 Thought → Action → Observation 的循环，只是 Action 从文本解析变成了结构化工具调用。**

---

## 八、ReAct 的优点

### 1. 能调用外部工具

ReAct 解决了 CoT 不能访问外部世界的问题。

它可以通过工具获取：

- 实时信息；
- 数据库内容；
- API 结果；
- 计算结果；
- 文件内容；
- 检索结果。

---

### 2. 推理过程更清晰

ReAct 显式保留 Thought、Action、Observation。

开发者可以看到：

- 模型为什么调用这个工具；
- 工具输入是什么；
- 工具返回了什么；
- 模型如何根据结果继续推理。

这比黑盒式直接回答更容易排查问题。

---

### 3. 适合开放式任务

ReAct 不要求开发者提前写死所有步骤。

模型可以根据当前 Observation 动态决定下一步。

所以它适合：

- 信息搜索；
- 多跳问答；
- 工具调用；
- 简单调研；
- 数据查询；
- API 编排。

---

## 九、ReAct 的局限

### 1. 循环漂移

ReAct 是“走一步看一步”。

每一轮都是根据当前历史临时决定下一步。

如果历史越来越长，模型可能被中间信息吸引，逐渐偏离原始目标。

例如用户原本问：

```text
分析苹果最近三年的营收变化趋势。
```

Agent 第一步查了苹果营收，第二步看到搜索结果里提到三星竞争，第三步可能就去查三星，最后偏离了“苹果三年营收趋势”这个目标。

这就是循环漂移。

---

### 2. 错误传播

ReAct 每一步都依赖之前的 Observation。

如果某一步工具结果错误，或者模型误解了工具结果，后续推理就会建立在错误基础上。

更麻烦的是，普通 ReAct 没有内置反思机制，不会自动检查：

```text
刚才这个 Observation 靠谱吗？
前面的推理有没有问题？
是否需要重新搜索验证？
```

所以早期错误可能一路传递到最终答案。

---

### 3. 成本和延迟增加

ReAct 是多轮调用：

```text
每一步都要调用 LLM
每一步都要带上历史上下文
每一步可能还要调用工具
```

所以步骤越多：

- token 消耗越多；
- 延迟越高；
- 工具调用成本越高；
- 上下文管理越困难。

---

### 4. 输出格式可能不稳定

如果使用经典文本解析版 ReAct，模型可能没有严格输出指定格式，导致解析失败。

比如模型可能输出：

```text
我觉得应该搜索一下苹果市值。
```

而不是：

```text
Action: search
Action Input: 苹果公司市值
```

所以现代工程中更推荐使用 Function Calling / Tool Use，让模型输出结构化工具调用。

---

## 十、Plan-and-Execute 是什么？

### 1. 为什么需要 Plan-and-Execute？

Plan-and-Execute 是为了解决 ReAct 缺少全局规划的问题。

ReAct 像是：

```text
边走边问路
```

每到一个路口，根据当前信息决定下一步。

Plan-and-Execute 像是：

```text
先看地图，再出发
```

先把整体路线规划清楚，再一步步执行。

---

### 2. Plan-and-Execute 的定义

Plan-and-Execute 的核心思想是：

> **先让 LLM 从全局视角制定任务计划，再按计划逐步执行。**

它通常分为两个阶段：

```text
Planner：生成计划
Executor：执行计划
```

---

### 3. Planner 阶段

Planner 负责把用户目标拆成多个具体步骤。

它只做规划，不执行工具。

比如用户问：

```text
请分析腾讯最近四个季度的财报变化，并总结增长原因和风险。
```

Planner 可能生成计划：

```text
1. 收集腾讯最近四个季度的财报数据；
2. 提取收入、利润、毛利率、费用等关键指标；
3. 对比四个季度的变化趋势；
4. 检索管理层解释和相关新闻；
5. 分析增长原因；
6. 总结潜在风险；
7. 输出结构化分析报告。
```

Planner 的关键是：

> **先从全局角度想清楚任务怎么做。**

---

### 4. Executor 阶段

Executor 拿着计划一步步执行。

每一步内部可以继续使用 ReAct。

也就是说：

```text
Plan-and-Execute 管整体路线
ReAct 管单步执行
```

比如执行第 1 步“收集最近四个季度财报数据”时，可以用 ReAct 调用：

- PDF 解析工具；
- RAG 检索工具；
- 数据库查询工具。

执行第 4 步“检索相关新闻”时，也可以用 ReAct 调用搜索工具。

---

### 5. 动态重规划

Plan-and-Execute 并不是计划生成后就完全写死。

更成熟的实现会加入 **动态重规划**。

也就是每执行完一步后，系统检查：

- 当前结果是否符合预期？
- 后续计划是否还合理？
- 是否需要增加步骤？
- 是否需要删除步骤？
- 是否需要调整顺序？

例如原计划是：

```text
1. 查询某个 API
2. 用 API 结果做分析
3. 输出报告
```

但执行时发现：

```text
API 已经下线
```

那么系统可以把当前结果和剩余任务重新交给 Planner，让它生成新的后续计划。

这就像导航遇到封路后重新规划路线。

## 十一、Plan-and-Execute 的伪代码

```python
def plan_and_execute(question: str, tools: dict):
    # 1. 规划阶段：让 LLM 生成完整计划
    plan = llm.generate(f"""
    请为以下任务制定一个分步执行计划：
    任务：{question}

    要求：
    1. 每一步都要具体、可执行；
    2. 标明每一步需要什么输入；
    3. 标明每一步预期输出什么。
    """)

    steps = parse_plan(plan)
    results = []

    # 2. 执行阶段：按计划执行
    for i, step in enumerate(steps):
        step_result = react_executor(
            task=step,
            tools=tools,
            context=f"整体计划共 {len(steps)} 步，当前是第 {i + 1} 步",
            previous_results=results
        )

        results.append(step_result)

        # 3. 动态重规划
        if need_replan(step, step_result, steps[i + 1:]):
            remaining_steps = llm.generate(f"""
            原计划：{steps}
            已完成步骤：{steps[:i + 1]}
            已有结果：{results}
            剩余步骤：{steps[i + 1:]}

            请判断剩余步骤是否需要调整。
            如果需要，请输出新的剩余步骤。
            """)
            steps = steps[:i + 1] + parse_plan(remaining_steps)

    # 4. 汇总结果
    return llm.generate(f"""
    请根据以下执行结果回答用户问题：
    用户问题：{question}
    执行结果：{results}
    """)
```

## 十二、ReAct 和 Plan-and-Execute 的区别

| 维度 | ReAct | Plan-and-Execute |
|---|---|---|
| 核心思想 | 边思考边行动 | 先规划再执行 |
| 类比 | 边走边问路 | 先看地图再出发 |
| 决策方式 | 每一步临时决策 | 先全局规划，再局部执行 |
| 是否有全局计划 | 没有 | 有 |
| 是否能调工具 | 可以 | 可以，执行阶段通常也用 ReAct |
| 优点 | 灵活、实现简单、适合工具调用 | 不容易跑偏，适合复杂任务 |
| 缺点 | 容易漂移、错误传播、上下文变长 | 多一次规划调用，简单任务可能有额外开销 |
| 适合场景 | 简单搜索、问答、短链路工具调用 | 深度调研、多步骤分析、多工具协同 |
| 成本 | 简单任务成本低，长任务成本高 | 规划有额外成本，但复杂任务更稳定 |

## 十三、什么时候用 ReAct？什么时候用 Plan-and-Execute？

### 1. 适合 ReAct 的场景

如果任务满足下面特点，可以优先用 ReAct：

- 任务步骤不多；
- 目标比较开放；
- 需要边查边判断；
- 主要是搜索、查库、计算等工具调用；
- 不需要复杂的全局规划。

例如：

```text
帮我查一下某家公司最近的一条新闻，并总结影响。
```

这种任务用 ReAct 就够了。

---

### 2. 适合 Plan-and-Execute 的场景

如果任务满足下面特点，更适合 Plan-and-Execute：

- 任务步骤多；
- 目标明确；
- 子任务之间有依赖关系；
- 需要全局统筹；
- 需要多工具协同；
- 容易因为中间信息偏离目标。

例如：

```text
请分析腾讯最近四个季度的财报变化，结合新闻和行业趋势，总结增长原因和风险。
```

这种任务最好先规划：

```text
先确定分析维度
再收集财报数据
再做季度对比
再检索新闻
再总结增长原因和风险
```

否则如果直接用 ReAct，很容易查着查着跑偏。

---

### 3. 实际工程中的推荐组合

真实工程里，两者经常混合使用：

```text
Plan-and-Execute：负责全局任务拆解
ReAct：负责每个子任务内部的工具调用
Reflection：负责关键结果的质量检查
Workflow：负责整体链路编排和边界控制
```

也就是：

```text
用户任务
  ↓
Workflow：判断任务类型
  ↓
Planner：生成任务计划
  ↓
Executor：逐步执行
      ↓
      每一步内部用 ReAct 调工具
  ↓
Reflection：检查结果质量
  ↓
Workflow：格式化输出
```

## 十四、结合金融分析 Agent 项目怎么回答？

如果结合我的金融分析 Agent 项目，我会这样说：

在我的金融分析 Agent 项目里，我不会把所有任务都交给纯 ReAct，因为金融分析任务往往是多步骤的，而且对证据和结构要求比较高。

比如用户问：

```text
分析腾讯最近几个季度财报变化，并总结增长原因和风险。
```

如果直接用 ReAct，它可能会先查一个季度的数据，然后看到新闻里提到游戏业务，就跑去查游戏新闻，接着又查竞品，最后偏离“多季度财报变化”这个主目标。

所以我更倾向于用 **Plan-and-Execute + ReAct** 的组合。

整体上，先让 Planner 拆解任务：

```text
1. 识别用户要分析的公司和时间范围；
2. 检索对应季度的财报 PDF；
3. 提取收入、利润、毛利率、费用等核心指标；
4. 对比不同季度的变化趋势；
5. 检索相关新闻和管理层解释；
6. 结合历史研究记忆补充背景；
7. 总结增长原因和风险；
8. 生成结构化中文研报。
```

然后每一步执行时，再用 ReAct 调用具体工具：

```text
RAG 检索工具
PDF 解析工具
Milvus 向量检索工具
Research Memory 查询工具
新闻搜索工具
行情查询工具
```

这样设计的好处是：

- **Planner 保证全局方向不跑偏**；
- **ReAct 保证每一步可以灵活调用工具**；
- **Workflow 保证主链路稳定可控**；
- **Tools 提供真实的数据和外部能力**。

所以在项目里，我会把 ReAct 理解成“局部执行器”，而不是整个系统的全部。复杂任务上层应该有 Plan-and-Execute 或 Workflow 控制。

## 十五、面试官可能追问：ReAct 的循环到底是谁驱动的？

可以这样回答：

ReAct 的循环不是模型自己驱动的，而是外层代码框架驱动的。

LLM 每次调用只会根据当前上下文输出一段内容，比如 Thought 和 Action，或者 Final Answer。它不会自己自动调用工具，也不会自己进入下一轮。

真正的循环过程是代码做的：

```text
1. 把用户问题和历史记录发给 LLM；
2. LLM 输出 Thought 和 Action；
3. 代码解析 Action 和 Action Input；
4. 代码调用对应工具；
5. 工具返回结果；
6. 代码把结果作为 Observation 追加到历史；
7. 代码再次调用 LLM；
8. 如果检测到 Final Answer，就结束循环。
```

所以 ReAct 的分工是：

```text
LLM 负责推理和决策
代码负责循环和执行
```

这个理解非常重要，因为 Agent 不是魔法，本质上是模型输出和工程控制逻辑组合起来的系统。

---

## 十六、面试官可能追问：ReAct 为什么比 CoT 更适合 Agent？

可以这样回答：

CoT 只能让模型把推理过程写出来，但它本质上还是纯文本推理，不能访问外部世界。

如果问题只依赖模型内部知识，CoT 是有帮助的。

但如果问题需要实时信息、数据库结果、API 调用或者计算结果，CoT 就不够了。

ReAct 在 CoT 的基础上加入了 Action 和 Observation，让模型可以在推理过程中调用工具，并把工具结果作为新的上下文继续推理。

所以 ReAct 的优势是：

```text
CoT 负责把推理显式化
Action 负责获取外部事实
Observation 负责把外部结果反馈给模型
```

这让 Agent 既能推理，又能和外部环境交互。

## 十七、面试官可能追问：ReAct 有什么问题？怎么优化？

可以这样回答：

ReAct 的主要问题有两个。

第一个是 **循环漂移**。因为 ReAct 每一步都是根据当前历史临时决策，没有全局计划约束，所以执行步骤一多，模型可能被中间信息带偏，逐渐偏离原始目标。

第二个是 **错误传播**。ReAct 每一步都依赖前面的 Observation，如果早期工具返回结果不准确，或者模型理解错了结果，后续推理会建立在错误基础上一路走偏。

优化方式主要有几个：

1. **设置最大步数和超时**，避免无限循环；
2. **使用结构化工具调用**，减少文本解析失败；
3. **在 Prompt 中持续保留用户原始目标**，降低漂移；
4. **引入 Plan-and-Execute**，先规划再执行；
5. **引入 Reflection**，对关键结果做自我检查；
6. **对工具结果做校验和引用追踪**，减少错误传播；
7. **用 Workflow 限制 Agent 的行动边界**，不要让它完全自由发挥。

其中最核心的优化是：

> **简单任务用 ReAct，复杂任务用 Plan-and-Execute 包住 ReAct。**

---

## 十八、面试官可能追问：Plan-and-Execute 为什么能缓解 ReAct 的漂移？

可以这样回答：

ReAct 容易漂移的根本原因是它没有全局计划，每一步都是根据当前历史临时决定下一步。

而 Plan-and-Execute 会先让模型从全局角度生成完整计划，明确任务要分成哪几步、每一步做什么、步骤之间有什么依赖关系。

这样 Executor 在执行每一步时，就知道自己处在整体计划中的哪个位置，也知道下一步应该做什么，不会完全被当前搜索结果或者中间信息牵着走。

所以 Plan-and-Execute 的作用是：

```text
用全局计划约束局部行动
```

它不是替代 ReAct，而是把 ReAct 放在更清晰的任务框架里，让 ReAct 负责每一步的工具调用，而不是让 ReAct 独自决定整个任务路径。

---

## 十九、面试官可能追问：Plan-and-Execute 的缺点是什么？

可以这样回答：

Plan-and-Execute 的缺点主要有三个。

第一，**有额外规划成本**。

它需要先调用一次 LLM 生成计划，如果任务本来一两步就能完成，那么规划阶段就是额外开销。

第二，**初始计划可能出错**。

如果 Planner 一开始拆解任务方向就错了，后续 Executor 可能会沿着错误计划执行。

第三，**链路更复杂**。

相比纯 ReAct，Plan-and-Execute 多了 Planner、Executor、计划解析、动态重规划等模块，工程实现和调试成本更高。

所以 Plan-and-Execute 不适合所有任务。

我的选型原则是：

```text
简单任务直接 ReAct
复杂任务使用 Plan-and-Execute
高质量任务再叠加 Reflection
生产系统用 Workflow 控制边界
```

## 二十、最终背诵版

Agent 的推理模式可以从简单到复杂分成直接回答、CoT、ReAct 和 Plan-and-Execute。

**直接回答** 是让 LLM 不展开中间过程，直接输出答案，适合简单问题，但复杂推理容易出错。

**CoT** 是让模型先写出中间推理步骤，再给最终答案。它的本质是把隐式推理显式化，让后续生成建立在已经写出来的中间结果上，从而降低多步推理错误。但 CoT 只能做纯文本推理，不能访问外部工具和实时数据。

**ReAct** 是在 CoT 基础上加入 Action，让模型按照 Thought、Action、Observation 的循环来完成任务。Thought 是模型推理，Action 是模型决定调用的工具，Observation 是工具返回结果，Final Answer 是最终答案。ReAct 的优势是既能推理，又能调用外部工具，适合搜索、查库、计算、API 调用等 Agent 场景。

但是 ReAct 的循环不是模型自己跑起来的，而是代码框架驱动的。模型每次只输出下一步 Thought 和 Action，代码负责解析 Action、执行工具、把工具结果作为 Observation 追加进历史，然后再次调用模型。如果模型输出 Final Answer，代码就结束循环。

ReAct 的问题是它缺少全局规划，容易出现循环漂移和错误传播。循环漂移是指步骤多了以后 Agent 可能偏离原始目标；错误传播是指早期某一步结果错了，后续推理都会被带偏。

为了解决这个问题，可以使用 **Plan-and-Execute**。它先由 Planner 生成完整执行计划，再由 Executor 按计划执行，每一步内部仍然可以用 ReAct 调工具。这样既有全局规划，不容易跑偏，又保留了 ReAct 的灵活工具调用能力。

实际项目中，我会根据任务复杂度选型：简单工具调用用 ReAct；复杂多步骤任务用 Plan-and-Execute；对输出质量要求高的任务再叠加 Reflection；生产环境中再用 Workflow 控制整体链路和边界。

---

## 二十一、一句话总结

> **CoT 让模型把推理写出来，ReAct 让模型边推理边调用工具，Plan-and-Execute 让模型先规划再执行；ReAct 的智能来自模型，但循环由代码驱动，复杂任务中最好用 Plan-and-Execute 包住 ReAct。**
