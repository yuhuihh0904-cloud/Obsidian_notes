## 一、面试标准回答

如果面试官问：

> 你了解 Multi-Agent 吗？说说它是什么，为什么要用？

可以这样回答：

我理解的 **Multi-Agent**，就是让多个 Agent 通过分工协作来共同完成一个复杂任务，而不是让一个 Agent 包揽所有事情。

它解决的核心问题主要有两个：

第一是 **context window 的硬限制**。单个 Agent 在处理复杂任务时，需要把用户需求、历史对话、工具调用结果、中间推理过程都放进上下文窗口里。一旦任务变长、资料变多，context 很容易被撑满，早期的重要信息就可能被截断，Agent 就会出现“遗忘”。

第二是 **单个 Agent 的专业度限制**。如果让一个 Agent 同时负责需求分析、信息搜索、代码实现、测试、总结等多个环节，它就像一个人同时扮演产品经理、开发、测试和文档工程师，每个角色都不够专注，输出质量也容易下降。

Multi-Agent 的核心思路是：

> 用团队协作代替单个 Agent 单打独斗。

每个 Agent 只负责一个相对明确的角色，比如搜索 Agent 负责检索信息，代码 Agent 负责实现代码，测试 Agent 负责验证结果，评审 Agent 负责检查质量。这样每个 Agent 的上下文更干净，职责更清晰，系统也更容易调试。

常见的 Multi-Agent 协作模式有三种：

1. **顺序流水线**：一个 Agent 做完后，把结果交给下一个 Agent。
2. **并行扇出**：多个 Worker Agent 同时处理不同子任务，最后由调度者汇总。
3. **辩论/评审模式**：多个 Agent 给出不同方案，再由裁判 Agent 或互评机制选择更优结果。

工程上常见的组织方式有两类：  
一种是 **中心化调度**，由 Orchestrator 负责拆任务、分发任务、收集结果；  
另一种是 **去中心化协作**，多个 Agent 之间直接通信、互相协调。

实际项目里更常用的是中心化方案，因为它的任务流更清晰，责任边界更明确，也更方便排查问题。

总结来说，Multi-Agent 的价值不只是“多个 AI 一起工作”，而是通过 **拆分上下文、角色专业化、并行执行和结果评审**，提升复杂任务处理能力、执行效率和输出质量。

## 二、这篇文章的核心主题

这篇文章主要回答的是：

> 什么是 Multi-Agent？为什么复杂任务不能总靠一个 Agent？多个 Agent 应该如何协作？

核心观点是：

> Multi-Agent 不是简单地堆多个大模型，而是把复杂任务拆成多个职责明确的子任务，让不同 Agent 分工协作，从而突破单 Agent 的 context 限制和专业能力限制。
