## 一、面试标准回答

如果面试官问：

> Agent 对话历史越来越长，context 快撑满了怎么办？记忆压缩有哪些方法？

可以这样回答：

Agent 的记忆压缩主要是为了解决两个问题：

1. **Context Window 空间有限**
2. **长上下文带来的 token 成本和推理延迟变高**

因为 LLM 本身并不是持续记住所有历史，而是每次调用时依赖传入的 `messages`。当对话越来越长时，如果不做压缩，历史消息、工具结果、中间推理和文档内容都会不断堆进 prompt，最终会导致 context 撑爆、重要信息被截断，或者每次调用成本越来越高。

常见的记忆压缩方法主要有四种：

第一种是 **滑动窗口**。  
它只保留最近 N 轮对话，把更早的历史直接丢掉。优点是实现简单、成本最低，不需要额外 LLM 调用；缺点是硬截断，可能把很早之前的重要决策也丢掉。

第二种是 **摘要压缩**。  
它不是直接丢掉旧历史，而是先用 LLM 把旧对话总结成一段摘要，再用摘要替代原始对话。这样可以保留历史主线和关键决策，但缺点是摘要本身可能丢失细节。

第三种是 **重要性过滤**。  
它不再按时间决定去留，而是给历史记录打重要性分数，只保留高价值内容。比如用户确认的需求、关键决策、工具结论会被保留，闲聊和重复内容会被丢弃。它解决的是“内容不等价，怎么挑”的问题。

第四种是 **结构化抽取**。  
它不再把所有历史都当成对话文本保存，而是把真正重要的信息抽成结构化字段，比如用户偏好、项目状态、已确认方案、预算、截止时间等。这样信息密度更高，后续使用时也更稳定，但需要提前设计字段，开发成本更高。

这四种方法不是互斥的，实际工程里通常会组合使用。比较常见的方案是：

> 最近几轮对话用滑动窗口保留原文，较早历史用摘要压缩，关键事实用结构化抽取保存，再通过重要性过滤控制哪些内容值得进入长期上下文。

另外还要区分 **记忆压缩** 和 **Prompt Caching**。记忆压缩是在信息层面决定哪些内容要保留、压缩或丢弃；Prompt Caching 是计算层面的优化，用来复用重复 prompt 前缀的计算结果，降低成本和延迟。它们是互补关系，不是替代关系。
