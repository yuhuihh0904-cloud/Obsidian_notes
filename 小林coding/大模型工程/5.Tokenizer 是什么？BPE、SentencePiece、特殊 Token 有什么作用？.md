
## 一、面试标准回答

面试官问：**Tokenizer 是什么？为什么大模型不能直接处理文字？BPE 和 SentencePiece 有什么区别？特殊 Token 又是干什么的？**

可以这样回答：

LLM 本质上只能处理数字张量，不能直接理解字符串。  
所以文本进入模型前，必须先经过 Tokenizer，把文本转换成 token id；模型输出后，再把 token id 解码回文本。

整个流程是：

~~~text
文本
  ↓
Tokenizer
  ↓
token id
  ↓
Embedding
  ↓
Transformer
  ↓
输出 token id
  ↓
Tokenizer 解码
  ↓
文本
~~~

Tokenizer 的核心目标是：

> **在“词表大小”和“序列长度”之间做平衡。**

如果按字符切分：

~~~text
优点：词表小
缺点：序列太长
~~~

如果按整词切分：

~~~text
优点：序列短
缺点：词表爆炸，未登录词 OOV 严重
~~~

所以现代 LLM 通常采用：

~~~text
子词（Subword）切分
~~~

也就是 BPE、WordPiece、SentencePiece 这类方法。

其中：

- **BPE**：不断合并高频字符对；
- **WordPiece**：类似 BPE，但根据概率收益选择合并；
- **SentencePiece**：直接在原始文本上训练，不依赖空格分词，更适合中文、多语言。

现在很多大模型实际使用的 tokenizer 都属于：

~~~text
BPE 系
SentencePiece 系
~~~

另外，大模型里还有很多特殊 Token，比如：

~~~text
<BOS>：句子开始
<EOS>：句子结束
<PAD>：padding
<UNK>：未知词
<CLS>：分类标记
<SEP>：分隔符
```

Chat 模型还会额外加入：

~~~text
<|system|>
<|user|>
<|assistant|>
~~~

这些特殊 Token 的作用是：

> **给模型提供结构化控制信号。**

比如告诉模型：

- 谁在说话；
- 一轮对话什么时候结束；
- 哪部分是 system prompt；
- 哪部分是用户问题；
- 哪部分需要预测。

一句话总结：

> **Tokenizer 负责把文本和 token id 相互转换；现代 LLM 通常用子词切分平衡词表和序列长度，而特殊 Token 本质上是给模型注入结构化控制信息。**

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **Tokenizer 不是简单“切词工具”，它决定了模型如何看待文本世界。**

同一句话：

~~~text
我喜欢Machine Learning
~~~

不同 tokenizer 可能切成：

~~~text
["我", "喜欢", "Machine", "Learning"]

或者

["我", "喜欢", "Mach", "ine", "Learn", "ing"]
```

不同切分会影响：

- 序列长度；
- 模型学习难度；
- 上下文窗口利用率；
- 推理速度；
- 多语言能力；
- 长文本成本。

所以 Tokenizer 是 LLM 很核心的一部分。

---

## 三、为什么模型不能直接处理文字？

神经网络本质上只能处理：

~~~text
向量
矩阵
张量
数字
~~~

不能直接处理：

~~~text
字符串
字符
单词
~~~

所以必须先把文本转成数字。

Tokenizer 的第一步就是：

~~~text
文本 -> token id
~~~

例如：

~~~text
"hello world"
```

可能变成：

~~~text
[15496, 995]
~~~

然后模型再把这些 id 查 embedding table：

~~~text
15496 -> 一个向量
995 -> 一个向量
~~~

最终输入 Transformer。

---

## 四、为什么不直接按字符切分？

例如：

~~~text
"Transformer"
```

按字符切：

~~~text
["T", "r", "a", "n", "s", ...]
```

优点：

- 词表很小；
- 不会出现 OOV 未知词。

但问题是：

> **序列太长。**

因为：

~~~text
Attention 复杂度 ~ O(N²)
~~~

token 数越多，训练和推理越贵。

比如：

~~~text
1000 个词
```

按字符切可能变成：

~~~text
5000+ token
```

成本会暴涨。

---

## 五、为什么不直接按整词切分？

按词切：

~~~text
["Transformer", "Tokenizer"]
```

优点：

- 序列短；
- 语义完整。

但问题是：

### 1. 词表爆炸

自然语言词汇组合无限。

如果全收录：

~~~text
英文词汇
专有名词
拼写错误
人名
代码变量
新网络词
```

词表会非常大。

---

### 2. OOV 问题严重

如果某个词没见过：

~~~text
ChatGPTPlusUltraProMax
```

模型就完全不会处理。

所以：

> **字符太细，词太粗。**

于是出现了：

~~~text
Subword（子词）
```

---

## 六、子词切分：现代 Tokenizer 核心思想

子词的核心思想是：

> **高频词整体保留，低频词拆成更小单位。**

例如：

~~~text
"playing"
```

可能切成：

~~~text
["play", "ing"]
```

而：

~~~text
"transformer"
```

可能整体保留。

这样：

- 高频词序列短；
- 生僻词也能拆开表示；
- OOV 大幅减少；
- 词表大小可控。

现代 LLM 基本都用：

~~~text
Subword Tokenizer
```

---

## 七、BPE：Byte Pair Encoding

BPE 最开始是数据压缩算法，后来被 NLP 使用。

它的核心思想是：

> **不断合并最常出现的字符对。**

例如：

初始词表：

~~~text
["l", "o", "w"]
```

如果：

~~~text
"lo"
```

经常一起出现，就合并：

~~~text
["lo", "w"]
```

如果：

~~~text
"low"
```

又经常出现，再继续合并：

~~~text
["low"]
```

训练过程本质是：

~~~text
从字符开始
不断统计高频 pair
不断 merge
```

最终形成子词词表。

---

## 八、WordPiece

WordPiece 和 BPE 很像。

区别在于：

- BPE 看出现频率；
- WordPiece 更偏向看概率收益。

典型模型：

~~~text
BERT
```

使用的就是 WordPiece。

---

## 九、SentencePiece

SentencePiece 是 Google 提出的方案。

它最大的特点是：

> **不依赖空格分词。**

传统英文 tokenizer 常假设：

~~~text
空格 = 单词边界
```

但中文、日文并没有天然空格。

SentencePiece 直接在原始文本上训练：

~~~text
原始字符串
  ↓
学习子词
```

所以特别适合：

- 中文；
- 日文；
- 多语言；
- 混合语言。

LLaMA、T5 等很多模型都用了 SentencePiece 系方案。

---

## 十、BPE 和 SentencePiece 的核心区别

| 对比项 | BPE | SentencePiece |
|---|---|---|
| 是否依赖空格 | 通常依赖 | 不依赖 |
| 训练对象 | 预分词文本 | 原始文本 |
| 多语言支持 | 一般 | 更好 |
| 中文支持 | 一般 | 更好 |
| 常见模型 | GPT 系早期 | LLaMA/T5 等 |

可以记：

~~~text
BPE：更像英文 NLP 传统路线
SentencePiece：更适合多语言 LLM
~~~

## 十一、特殊 Token 是什么？

除了普通文本 token，大模型还会有：

~~~text
<BOS>
<EOS>
<PAD>
<UNK>
<CLS>
<SEP>
```

这些特殊 Token 不表示自然语言词汇，而是：

> **给模型的控制信号。**

---

## 十二、常见特殊 Token

### 1. BOS：Begin Of Sequence

表示序列开始：

~~~text
<BOS> 你好
```

告诉模型：

~~~text
这是输入起点
```

---

### 2. EOS：End Of Sequence

表示序列结束：

~~~text
你好 <EOS>
```

模型生成到 EOS 时通常停止生成。

---

### 3. PAD：Padding

因为 batch 中不同句子长度不同：

~~~text
句子1：5 token
句子2：20 token
```

为了组成矩阵，需要补齐：

~~~text
[PADDING]
```

PAD token 通常会被 attention mask 掉。

---

### 4. UNK：Unknown

表示未知词。

不过现代 subword tokenizer 基本很少真正出现 OOV。

---

### 5. CLS：分类标记

BERT 常用：

~~~text
[CLS]
```

模型会把整个句子信息聚合到 CLS 向量，用于分类任务。

---

### 6. SEP：分隔符

例如：

~~~text
句子A [SEP] 句子B
```

告诉模型：

~~~text
这里是不同段落边界
```

---

## 十三、Chat 模型里的特殊 Token

Chat 模型通常还会有：

~~~text
<|system|>
<|user|>
<|assistant|>
```

例如：

~~~text
<|system|>
你是一个 AI 助手

<|user|>
你好

<|assistant|>
你好，请问有什么可以帮助你的？
```

这些 token 的作用是：

> **告诉模型当前是谁在说话。**

本质上：

> **Chat 其实只是“结构化 Prompt + 特殊 Token”。**

---

## 十四、为什么特殊 Token 很重要？

特殊 Token 本质上是：

> **模型控制接口。**

例如：

~~~text
<EOS>
```

决定什么时候停止生成。

~~~text
<|assistant|>
```

决定模型进入“助手回复模式”。

~~~text
<image>
```

决定后面输入的是视觉 token。

所以：

> **特殊 Token 是模型行为控制的重要入口。**

---

## 十五、Tokenizer 会影响什么？

Tokenizer 不只是切词工具，它会影响：

### 1. 序列长度

同一句话：

~~~text
不同 tokenizer
```

token 数可能差很多。

而：

~~~text
Attention 成本 ~ O(N²)
```

token 数越多，推理越贵。

---

### 2. 多语言能力

英文 tokenizer 不一定适合中文。

中文如果切得太碎：

~~~text
token 数暴涨
```

成本会明显增加。

---

### 3. 长文本能力

上下文窗口通常按：

~~~text
token 数
```

计算。

Tokenizer 越高效：

~~~text
同样窗口能装更多文本
```

---

### 4. 代码能力

代码 tokenizer 会影响：

- 缩进；
- 换行；
- 符号；
- 变量名。

很多代码模型会针对代码专门优化 tokenizer。

---

~~~mermaid
flowchart TD
    A[文本] --> B[Tokenizer]

    B --> C[token 切分]
    C --> D[token id]

    D --> E[Embedding]
    E --> F[Transformer]

    F --> G[输出 token id]
    G --> H[Tokenizer 解码]
    H --> I[文本]
~~~

---

## 十七、流程图：BPE 训练

~~~mermaid
flowchart TD
    A[初始字符词表] --> B[统计高频字符对]
    B --> C[合并最常出现 pair]
    C --> D[形成新的子词]
    D --> E[继续统计]
    E --> F[不断 merge]
    F --> G[得到最终 subword 词表]
~~~

## 十八、面试官可能继续追问

### 1. Tokenizer 为什么重要？

因为模型实际处理的是 token，不是文字。

Tokenizer 会影响：

~~~text
序列长度
训练成本
推理速度
多语言能力
上下文利用率
```

---

### 2. 为什么现代 LLM 都用 subword？

因为：

~~~text
字符切分：序列太长
整词切分：词表太大、OOV 严重
```

subword 在词表大小和序列长度之间取得平衡。

---

### 3. BPE 的核心思想是什么？

不断统计高频字符对，然后 merge。

最终形成高频词整体保留、低频词拆分的子词词表。

---

### 4. SentencePiece 为什么适合中文？

因为它不依赖空格。

它直接在原始文本上学习子词，更适合中文、日文、多语言场景。

---

### 5. 特殊 Token 的本质是什么？

本质是：

> **结构化控制信号。**

它告诉模型：

- 谁在说话；
- 哪段是 system prompt；
- 哪段结束；
- 哪部分不要 attention；
- 什么时候停止生成。

---

### 6. Chat 模型为什么需要 `<|user|>`、`<|assistant|>`？

因为模型本质上只是续写 token。

这些特殊 token 能帮助模型区分：

~~~text
用户输入
助手回复
system prompt
```

从而学会对话结构。
