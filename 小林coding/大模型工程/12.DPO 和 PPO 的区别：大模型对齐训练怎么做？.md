
## 一、面试标准回答

面试官问：**大模型的 DPO 和 PPO 有什么区别？**

可以这样回答：

DPO 和 PPO 都是大模型 SFT 之后常用的对齐方法，目标都是让模型输出更符合人类偏好。

SFT 解决的是：

~~~text
模型会不会按指令回答
~~~

但 SFT 本质上是模仿学习，它只告诉模型“怎么写一个合格答案”，并没有明确告诉模型：

~~~text
哪个回答更好
哪个回答更安全
哪个回答更符合人类偏好
~~~

所以 SFT 之后还需要偏好对齐。

PPO 是经典 RLHF 里的强化学习方案。它的流程是：

~~~text
先收集人类偏好排序数据
    ↓
训练 Reward Model 作为裁判
    ↓
主模型生成回答
    ↓
Reward Model 给回答打分
    ↓
用 PPO 优化主模型，让它更倾向于生成高分回答
~~~

PPO 训练时通常要同时维护 4 个模型：

~~~text
policy model：正在训练的主模型
reference model：冻结的 SFT 模型，用于 KL 约束
reward model：奖励模型，负责给回答打分
value model：价值模型，估计未来奖励，用于 RL 稳定训练
~~~

其中 KL 约束很关键，它限制 policy model 不要偏离 reference model 太远，防止模型为了骗 Reward Model 而 reward hacking。

DPO 是后来提出的简化方案。它不再单独训练 Reward Model，也不需要 PPO 这套强化学习流程，而是直接使用偏好对数据：

~~~text
prompt
chosen：人类更喜欢的回答
rejected：人类不喜欢的回答
~~~

DPO 的核心思想是：

> **让模型在 chosen 上的概率提升幅度，大于在 rejected 上的概率提升幅度。**

它只需要两个模型：

~~~text
policy model：正在训练的主模型
reference model：冻结的参考模型
~~~

所以 DPO 相比 PPO：

- 不需要 Reward Model；
- 不需要 Value Model；
- 不需要 PPO 强化学习训练；
- 工程更简单；
- 训练更稳定；
- GPU 成本更低。

可以用一个类比记：

> **PPO 是先训练一个裁判，再让选手根据裁判打分不断训练；DPO 是直接看比赛录像，告诉选手哪个动作更好、哪个动作更差。**

一句话总结：

> **PPO 是“奖励模型 + 强化学习”的对齐方案，DPO 是“偏好对 + 监督学习式优化”的对齐方案；两者目标一致，但 DPO 省掉了 Reward Model 和 PPO 流程，因此更简单、更稳定。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **DPO 和 PPO 的区别，不是简单的“DPO 更简单”，而是在线强化学习和偏好监督学习之间的差别。**

核心主线：

~~~text
SFT 之后模型会回答
但不知道哪种回答更好
    ↓
PPO：
先训练 Reward Model 作为裁判
再用 RL 优化主模型
    ↓
DPO：
绕过裁判
直接用 chosen / rejected 偏好对训练
    ↓
DPO 把复杂 RLHF 流程简化成监督学习式训练
~~~

最重要的一点：

> **PPO 需要 4 个模型，DPO 只需要 2 个模型。**

---

## 三、为什么 SFT 之后还需要对齐？

SFT 的训练数据一般是：

~~~text
用户指令 -> 标准答案
~~~

它让模型学会：

- 指令跟随；
- 对话格式；
- 基本回答风格；
- Markdown / JSON / 代码格式。

但 SFT 有一个局限：

> **它主要是模仿标准答案，却没有显式学习“哪个回答更好”。**

例如同一个问题：

~~~text
怎么学好 Python？
~~~

可能有多个回答：

~~~text
回答 A：很详细，有学习路线和代码练习
回答 B：很简短，只说多看教程
回答 C：带有错误建议
~~~

SFT 只能模仿某一种答案，但偏好对齐要解决的是：

~~~text
A 比 B 更好
B 比 C 更好
~~~

也就是：

> **让模型学会人类偏好排序。**

---

## 四、PPO：先训练裁判，再训练选手

PPO 全称：

~~~text
Proximal Policy Optimization
~~~

它是强化学习算法，在 RLHF 中用于优化大模型。

PPO 的整体流程可以理解成：

~~~text
人类偏好数据
    ↓
训练 Reward Model
    ↓
Policy Model 生成回答
    ↓
Reward Model 给回答打分
    ↓
PPO 根据奖励更新 Policy Model
~~~

类比：

~~~text
Reward Model：裁判
Policy Model：选手
PPO：训练选手的方法
~~~

---

## 五、PPO 第一步：训练 Reward Model

先收集人类偏好排序数据。

例如：

~~~text
prompt：解释什么是递归

回答 A：清晰，有例子
回答 B：太简略
回答 C：错误
~~~

人工标注偏好：

~~~text
A > B > C
~~~

然后训练 Reward Model，让它学会：

~~~text
输入：prompt + answer
输出：reward score
~~~

分数越高，代表回答越符合人类偏好。

Reward Model 的作用是：

> **代替人类，在后续训练中自动给模型回答打分。**

---

## 六、PPO 第二步：用奖励优化主模型

有了 Reward Model 后，就可以训练 Policy Model。

流程：

~~~text
Policy Model 生成回答
    ↓
Reward Model 打分
    ↓
PPO 根据 reward 更新 Policy Model
~~~

目标是：

> **让主模型以后更倾向于生成高 reward 的回答。**

但这里有一个风险：

> **模型可能学会钻 Reward Model 的空子。**

这叫：

~~~text
reward hacking
~~~

例如模型生成一些 Reward Model 打高分、但真实用户不喜欢的模式化回答。

---

## 七、KL 约束：防止模型跑偏

为了防止 Policy Model 为了高分偏离太远，PPO 中会引入 reference model。

reference model 通常是：

~~~text
SFT 模型的冻结副本
~~~

KL 约束的作用是：

> **限制正在训练的 policy model 不要偏离原来的 SFT 模型太远。**

可以理解成：

~~~text
Reward Model 是油门
KL 约束是刹车
~~~

模型可以朝高分方向优化，但不能失控。

---

## 八、PPO 的 4 模型架构

PPO 训练时通常需要维护 4 个模型：

~~~text
policy_model：
正在被优化的主模型

reference_model：
冻结的 SFT 模型副本，用于 KL 约束

reward_model：
奖励模型，负责给回答打分

value_model：
价值模型，用于估计未来奖励，辅助 PPO 计算优势函数
~~~

这也是 PPO 工程复杂的核心原因。

可以记：

~~~text
PPO = policy + reference + reward + value
~~~

---

## 九、PPO 的问题

### 1. 工程复杂

需要：

- 训练 Reward Model；
- 维护 4 个模型；
- 运行 PPO；
- 管理 KL；
- 调 RL 超参数。

---

### 2. 显存成本高

4 个模型同时存在，显存压力很大。

尤其是大模型对齐时，资源成本非常高。

---

### 3. 训练不稳定

强化学习本身不稳定，容易出现：

- reward hacking；
- 训练震荡；
- KL 失控；
- 超参数敏感；
- 收敛困难。

---

### 4. 门槛高

PPO 不只是普通 supervised training，而是完整 RL 系统。

所以通常只有资源充足、有 RL 工程能力的团队更适合做。

---

## 十、DPO：绕过裁判，直接看回放

DPO 全称：

~~~text
Direct Preference Optimization
~~~

直接偏好优化。

它的核心思想是：

> **不再显式训练 Reward Model，而是直接用偏好对训练模型。**

DPO 使用的数据格式：

~~~text
prompt
chosen：更好的回答
rejected：更差的回答
~~~

例如：

~~~json
{
  "prompt": "如何学好 Python？",
  "chosen": "建议先从基础语法入手，再通过小项目练习，并逐步学习常用库。",
  "rejected": "Python 很简单，随便看看教程就行。"
}
~~~

DPO 直接让模型学会：

~~~text
chosen 的概率应该高于 rejected
~~~

---

## 十一、DPO 的直觉

DPO 不是简单地让模型死记 chosen。

更准确地说，它比较的是：

~~~text
policy model 相对 reference model 的概率变化
~~~

直觉上：

~~~text
相对于 reference model，
policy model 应该更提高 chosen 的概率，
更降低 rejected 的概率。
~~~

也就是：

~~~text
chosen 的概率提升幅度 > rejected 的概率提升幅度
~~~

所以 DPO 仍然保留 reference model，用来作为基准。

---

## 十二、DPO 为什么不需要 Reward Model？

DPO 的关键来自数学推导：

> **带 KL 约束的 RLHF 目标，可以改写成偏好对上的监督学习损失。**

也就是说，Reward Model 的作用可以被：

~~~text
policy model 和 reference model 的概率比
~~~

替代。

所以 DPO 不需要显式训练 Reward Model。

它直接优化：

~~~text
模型对 chosen / rejected 的相对偏好
~~~

这就是文章里说的：

> **绕过裁判，直接看回放。**

---

## 十三、DPO 只需要两个模型

DPO 训练时只需要：

~~~text
policy_model：
正在训练的主模型

reference_model：
冻结的参考模型
~~~

不需要：

~~~text
reward_model
value_model
~~~

所以：

~~~text
PPO：4 个模型
DPO：2 个模型
~~~

这就是 DPO 工程上大幅简化的关键。

---

## 十四、DPO 的优点

### 1. 工程简单

DPO 可以用普通深度学习训练框架实现。

不需要复杂 RL 基础设施。

---

### 2. 训练稳定

DPO 更接近监督学习，训练曲线通常比 PPO 稳。

---

### 3. 成本低

少了 Reward Model 和 Value Model：

- 显存更低；
- 训练流程更短；
- 调参更简单。

---

### 4. 开源社区友好

大多数开源团队没有 OpenAI 级别的 RLHF 工程能力。

DPO 更适合：

- 快速迭代；
- 小团队；
- 单机 / 多卡训练；
- 偏好数据微调。

---

## 十五、DPO 的局限

DPO 也不是万能。

### 1. 受偏好数据分布限制

DPO 本质上是在已有 chosen / rejected 数据上做偏好学习。

如果偏好数据质量差，模型会学偏。

---

### 2. 探索能力弱于 PPO

PPO 可以让模型在线采样新回答，再根据 Reward Model 反馈优化。

DPO 更偏离线训练，主要依赖已有偏好对。

所以：

> **PPO 的探索能力更强，DPO 的工程稳定性更好。**

---

### 3. 对数据质量敏感

如果 chosen 和 rejected 差异不明显，或者标注质量不稳定，DPO 效果也会受影响。

---

## 十六、PPO 和 DPO 对比表

| 对比项 | PPO | DPO |
|---|---|---|
| 方法类型 | 强化学习 | 监督学习式偏好优化 |
| 是否需要 Reward Model | 需要 | 不需要 |
| 是否需要 Value Model | 需要 | 不需要 |
| 同时维护模型数 | 4 个 | 2 个 |
| 训练稳定性 | 较差，RL 不稳定 | 较好 |
| 工程复杂度 | 高 | 低 |
| 显存成本 | 高 | 低 |
| 探索能力 | 更强 | 较弱 |
| 适合团队 | 大厂、RL 工程能力强 | 开源社区、中小团队 |
| 核心类比 | 先训裁判再训选手 | 直接看回放学偏好 |

---

## 十七、GRPO：PPO 的改进版

GRPO 全称：

~~~text
Group Relative Policy Optimization
~~~

它可以理解为 PPO 的改进版本。

核心创新是：

> **砍掉 Value Model，用组内归一化代替价值模型。**

PPO 需要 Value Model 来估计：

~~~text
当前回答的预期奖励
~~~

GRPO 的做法是：

~~~text
对同一个 prompt 采样多个回答
    ↓
给每个回答打分
    ↓
用组内平均值和标准差归一化
    ↓
得到每个回答的相对优势
~~~

公式直觉：

~~~text
A_i = (r_i - mean(r_1 ... r_G)) / std(r_1 ... r_G)
~~~

也就是说：

> **不需要单独训练 Value Model，而是用同一组回答的相对分数作为优势估计。**

---

## 十八、GRPO 的意义

GRPO 把 PPO 的：

~~~text
policy + reference + reward + value
~~~

简化为：

~~~text
policy + reference + reward
~~~

如果是数学题、代码题这类可验证任务，还可以直接用对错作为 reward：

~~~text
正确：1
错误：0
~~~

这时甚至可以不训练复杂 Reward Model。

GRPO 适合：

- 数学推理；
- 代码生成；
- 可验证任务；
- 推理模型训练。

---

## 十九、什么时候选 PPO，什么时候选 DPO？

### 1. 选择 DPO

适合：

- 已有高质量偏好对数据；
- 希望快速迭代；
- GPU 资源有限；
- 没有复杂 RL 工程能力；
- 开源模型对齐；
- 想稳定提升回答偏好。

可以记：

> **大多数团队优先选 DPO。**

---

### 2. 选择 PPO

适合：

- 资源充足；
- 需要极致对齐效果；
- 有成熟 RLHF 工程能力；
- 需要模型探索偏好数据之外的新策略；
- 追求接近顶级闭源模型的对齐效果。

可以记：

> **PPO 上限高，但成本和风险也高。**

---

### 3. 选择 GRPO

适合：

- 数学；
- 代码；
- 逻辑推理；
- 有明确对错反馈的任务；
- 推理模型训练。

---

## 二十一、流程图：DPO 训练流程

~~~mermaid
flowchart TD
    A[偏好对数据] --> B[prompt + chosen + rejected]

    B --> C[Policy Model 计算 chosen/rejected 概率]
    B --> D[Reference Model 计算 chosen/rejected 概率]

    C --> E[比较相对概率提升]
    D --> E

    E --> F[提高 chosen 概率]
    E --> G[降低 rejected 概率]

    F --> H[更新 Policy Model]
    G --> H
~~~

## 二十二、结合项目怎么说

如果面试官问：

> **如果你要做模型对齐，会选 PPO 还是 DPO？**

可以这样回答：

如果是资源有限的项目，我会优先考虑 DPO。

因为 DPO 不需要单独训练 Reward Model，也不需要 PPO 这种复杂强化学习流程，只需要准备：

~~~text
prompt
chosen
rejected
~~~

这样的偏好对数据，就可以直接训练模型，让模型更偏向 chosen、远离 rejected。

这对中小团队和开源项目更现实，训练稳定性也更好。

但如果是大厂级别的对齐训练，有大量算力、成熟 RLHF 工程能力，并且希望模型能探索偏好数据之外的更优策略，那 PPO 或 GRPO 仍然有价值。

如果是数学、代码这类可验证任务，我会关注 GRPO，因为它可以通过同一个 prompt 下多回答的组内相对分数来估计优势，减少 Value Model 的成本，适合训练推理模型。

## 二十三、面试官可能继续追问

### 1. PPO 和 DPO 都解决什么问题？

都解决 SFT 之后的偏好对齐问题。

SFT 让模型会回答，PPO / DPO 让模型更符合人类偏好。

---

### 2. PPO 为什么需要 Reward Model？

因为 PPO 是强化学习，需要 reward 信号。

Reward Model 负责代替人类给模型回答打分。

---

### 3. PPO 为什么需要 Reference Model？

为了做 KL 约束。

它限制 policy model 不要偏离 SFT 模型太远，避免 reward hacking 和输出风格崩坏。

---

### 4. PPO 为什么需要 Value Model？

Value Model 用来估计未来奖励，帮助计算 advantage，让强化学习训练更稳定。

---

### 5. DPO 为什么不需要 Reward Model？

因为 DPO 将带 KL 约束的 RLHF 目标改写成偏好对上的监督学习损失。

它直接通过 chosen / rejected 的概率比学习偏好，不需要显式奖励模型。

---

### 6. DPO 为什么还需要 Reference Model？

Reference Model 是基准。

DPO 比较的是：

~~~text
policy 相对于 reference 对 chosen 的概率提升
policy 相对于 reference 对 rejected 的概率提升
~~~

目标是：

~~~text
chosen 的提升幅度 > rejected 的提升幅度
~~~

---

### 7. DPO 和 SFT 有什么区别？

SFT 数据是：

~~~text
prompt -> answer
~~~

目标是模仿标准答案。

DPO 数据是：

~~~text
prompt + chosen + rejected
~~~

目标是学习偏好排序。

---

### 8. DPO 比 PPO 一定更好吗？

不是。

DPO 更简单、更稳定、更省资源，但依赖偏好数据分布，探索能力弱。

PPO 更复杂、更不稳定，但理论上可以探索偏好数据之外的策略，上限可能更高。

---

这篇文章可以压缩成一句话：

> **PPO 是“先训练裁判，再用强化学习训练选手”；DPO 是“直接用偏好对告诉模型哪个回答更好”，两者目标都是对齐，但 DPO 省掉 Reward Model 和 PPO 流程，因此更简单稳定。**

~~~text
SFT：
让模型会回答
但不知道哪个回答更好

PPO：
policy model
reference model
reward model
value model
4 个模型
强化学习
上限高但复杂不稳定

DPO：
policy model
reference model
2 个模型
偏好对监督学习
简单稳定，开源常用

先训练裁判

~~~

> **PPO 和 DPO 的本质区别是：PPO 用 Reward Model + RL 在线优化，DPO 用 chosen/rejected 偏好对直接做监督式偏好优化。**
