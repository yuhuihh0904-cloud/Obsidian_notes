
## 一、面试标准回答

面试官问：**大模型微调的方案有哪些？LoRA、QLoRA、SFT、DPO 分别是什么关系？**

可以这样回答：

大模型微调不能把全量微调、LoRA、QLoRA、SFT、DPO 这些名词简单并列起来背。它们其实属于两个不同维度。

第一个维度是：

> **改哪些参数。**

这个维度回答的是：

~~~text
微调时到底更新模型的哪些权重？
~~~

常见方案有：

- **全量微调**：所有参数都更新；
- **LoRA**：冻结原模型，只训练少量低秩适配矩阵；
- **QLoRA**：把基础模型量化到 4-bit，再在量化模型上训练 LoRA。

第二个维度是：

> **学什么目标。**

这个维度回答的是：

~~~text
微调时希望模型学会什么行为？
~~~

常见目标有：

- **SFT**：让模型学会按指令回答；
- **DPO**：让模型学会偏好对齐，也就是好回答概率高于差回答。

这两个维度是正交的，可以组合：

~~~text
全量微调 + SFT
LoRA + SFT
QLoRA + SFT
LoRA + DPO
QLoRA + DPO
~~~

所以不能说 LoRA 和 SFT 是同一类方法。  
更准确地说：

> **LoRA / QLoRA 是“怎么改参数”，SFT / DPO 是“学什么目标”。**

另外，微调不是第一选择，而是最后手段。  
如果只是控制输出格式，优先用 Prompt + Few-shot；如果只是补充领域知识，优先用 RAG；如果只是控制风格，优先用 System Prompt 和示例。只有这些方法都不稳定，或者需要模型长期稳定掌握某种任务模式、领域表达、固定风格，或者想用小模型替代大模型降低推理成本时，才考虑微调。

实战选型上：

- 个人开发者或资源有限：优先 **QLoRA + SFT**；
- 有几张 A100 的团队：可以用 **LoRA + SFT**；
- 需要偏好对齐：在 SFT 后接 **LoRA + DPO**；
- 大厂追求最高效果：可以考虑 **全量微调 + SFT + RLHF / DPO**。

一句话总结：

> **微调要先判断该不该做，再从“改哪些参数”和“学什么目标”两个正交维度选组合；绝大多数项目优先考虑 QLoRA + SFT，而不是一上来全量微调。**

---

## 二、这篇文章的核心观点

这篇文章主要讲的是：

> **微调方案不能按名词罗列，而要按两个维度理解：改哪些参数，学什么目标。**

很多人会把这些名词并列起来：

~~~text
全量微调
LoRA
QLoRA
SFT
DPO
~~~

但这是错误的。

正确关系是：

~~~text
改哪些参数：
全量微调 / LoRA / QLoRA

学什么目标：
SFT / DPO
~~~

这两个维度可以组合。

例如：

~~~text
LoRA 可以做 SFT
LoRA 也可以做 DPO
QLoRA 可以做 SFT
QLoRA 也可以做 DPO
~~~

所以：

> **LoRA 是参数更新方式，SFT 是训练目标。**

---

## 三、前置判断：什么时候才需要微调？

微调不是第一选择。

当模型表现不好时，应该先判断问题类型。

### 1. 输出格式不稳定

比如希望模型输出 JSON。

优先尝试：

~~~text
Prompt 写清楚格式
Few-shot 给 3~5 个示例
结构化输出约束
~~~

不应该一上来微调。

---

### 2. 缺少领域知识

比如产品价格、政策条款、库存状态、合同原文。

优先使用：

~~~text
RAG
数据库
搜索工具
知识库检索
~~~

因为这类知识经常变化，不适合写进模型参数。

---

### 3. 风格不一致

比如公司客服语气、文案风格。

优先尝试：

~~~text
System Prompt
Few-shot 对话示例
风格模板
~~~

如果 Prompt 控制不稳定，再考虑微调。

---

### 4. 真正适合微调的场景

微调更适合：

- 长期稳定的输出风格；
- 特定任务格式；
- 内部术语表达；
- 特定领域问答模式；
- 用小模型替代大模型降低推理成本。

记一句话：

> **微调适合学习行为、格式、风格、任务模式，不适合存放频繁变化的事实知识。**

---

## 四、第一个维度：改哪些参数？

这个维度回答：

> **训练时到底更新模型的哪些权重？**

主要有三类。

---

## 五、全量微调：所有参数都改

全量微调就是：

~~~text
基础模型所有参数都参与训练
~~~

优点是：

- 表达自由度最高；
- 效果上限最高；
- 适合资源充足的大厂。

缺点也很明显：

- 显存成本极高；
- 训练成本高；
- 调参难度高；
- 容易灾难性遗忘；
- 基础模型升级后要重新训练。

例如一个 7B 模型，全量微调不仅要存权重，还要存：

~~~text
梯度
优化器状态
激活值
~~~

实际显存需求远高于模型本身大小。

所以：

> **全量微调效果上限高，但工程成本非常高。**

---

## 六、LoRA：只训练低秩适配矩阵

LoRA 全称：

~~~text
Low-Rank Adaptation
~~~

核心思想是：

> **冻结原模型权重，只训练额外加上的低秩矩阵。**

原来权重更新是：

~~~text
W -> W + ΔW
~~~

LoRA 不直接训练完整的 `ΔW`，而是把它分解成两个小矩阵：

~~~text
ΔW = B × A
~~~

其中 rank `r` 很小，比如：

~~~text
r = 8
r = 16
~~~

训练时只更新：

~~~text
A 和 B
~~~

原模型权重 `W` 保持冻结。

---

## 七、LoRA 的优点

LoRA 的优点：

- 可训练参数少；
- 显存需求低；
- 训练速度快；
- 不容易破坏原模型通用能力；
- 可以保存多个 adapter；
- 适合开源社区和中小团队。

可以类比：

> **全量微调像重写整本书，LoRA 像在书上贴便利贴。**

原书不变，只通过便利贴补充新能力。

---

## 八、QLoRA：量化基础模型 + LoRA

QLoRA 的核心思想是：

> **把基础模型量化到 4-bit，再在量化模型上训练 LoRA。**

流程是：

~~~text
基础模型 4-bit 量化
    ↓
冻结量化后的基础模型
    ↓
插入 LoRA adapter
    ↓
只训练 LoRA 参数
~~~

这样可以把显存需求进一步压低。

QLoRA 适合：

- 个人开发者；
- 单卡 4090；
- 资源受限团队；
- 7B / 13B 模型微调。

一句话：

> **LoRA 让微调变轻，QLoRA 让消费级 GPU 也能微调。**

---

## 九、三种参数更新方案对比

| 方案 | 更新参数 | 显存成本 | 效果上限 | 适合场景 |
|---|---|---|---|---|
| 全量微调 | 全部参数 | 最高 | 最高 | 大厂、强资源、追求极致效果 |
| LoRA | 少量 adapter 参数 | 中低 | 较高 | 中小团队、主流工业方案 |
| QLoRA | 4-bit 基座 + LoRA | 最低 | 接近 LoRA | 个人开发者、资源有限场景 |

记忆方式：

~~~text
全量微调：全改，贵但强
LoRA：少改，性价比高
QLoRA：量化后少改，最省显存
~~~

---

## 十、第二个维度：学什么目标？

这个维度回答：

> **微调到底希望模型学会什么？**

主要有：

~~~text
SFT
DPO
~~~

---

## 十一、SFT：学会按指令回答

SFT 全称：

~~~text
Supervised Fine-Tuning
~~~

监督微调。

SFT 的目标是：

> **让模型从“续写模式”变成“指令回答模式”。**

训练数据格式通常是：

~~~text
指令 -> 期望回答
~~~

例如：

~~~text
用户：请介绍一下北京
助手：北京是中国的首都，位于华北平原……
~~~

SFT 让模型学会：

- 看懂指令；
- 按要求回答；
- 控制输出格式；
- 使用对话格式；
- 生成更符合任务要求的回答。

重点：

> **SFT 是训练目标，不是参数更新方法。**

所以可以：

~~~text
全量微调做 SFT
LoRA 做 SFT
QLoRA 做 SFT
~~~

---

## 十二、SFT 数据质量很关键

SFT 不是数据越多越好。

更重要的是：

~~~text
高质量
高一致性
任务覆盖合理
格式稳定
答案可靠
~~~

少量高质量数据，往往比大量粗糙数据更有用。

如果 SFT 数据质量差，模型会学到：

- 错误格式；
- 错误知识；
- 啰嗦风格；
- 不稳定回答；
- 低质量推理模式。

所以：

> **SFT 的核心不是堆数据量，而是控制数据质量。**

---

## 十三、DPO：学会偏好对齐

DPO 全称：

~~~text
Direct Preference Optimization
~~~

直接偏好优化。

SFT 解决的是：

> **模型会不会按指令回答。**

但 SFT 之后还会有问题：

~~~text
回答太啰嗦
回答太冷淡
回答不够安全
回答风格不符合用户偏好
~~~

DPO 解决的是：

> **同一个问题下，哪种回答更受欢迎。**

DPO 数据通常是三元组：

~~~text
问题
好回答 chosen
差回答 rejected
~~~

训练目标是：

~~~text
提高好回答概率
降低差回答概率
~~~

---

## 十四、DPO 和 RLHF 的关系

早期偏好对齐常用 RLHF：

~~~text
收集偏好数据
    ↓
训练 Reward Model
    ↓
用 PPO 优化主模型
~~~

但 RLHF 工程复杂：

- 要训练奖励模型；
- 要跑 PPO；
- 训练不稳定；
- 调参困难；
- 成本高。

DPO 的优势是：

> **绕过 Reward Model 和 PPO，直接用偏好数据训练模型。**

所以 DPO 更适合开源社区和资源有限团队。

重点：

> **DPO 也是训练目标，不是参数更新方法。**

它可以搭配：

~~~text
全量微调
LoRA
QLoRA
~~~

工业界常见：

~~~text
LoRA + DPO
~~~

---

## 十五、SFT 和 DPO 的区别

| 对比项 | SFT | DPO |
|---|---|---|
| 学习目标 | 学会按指令回答 | 学会偏好对齐 |
| 数据格式 | 指令 + 标准回答 | 问题 + 好回答 + 差回答 |
| 解决问题 | 会不会回答 | 哪种回答更好 |
| 常用阶段 | 指令微调 | SFT 之后的偏好优化 |
| 常见组合 | QLoRA + SFT | LoRA + DPO |

可以记：

~~~text
SFT：教模型怎么答
DPO：教模型哪种答法更好
~~~

---

## 十六、两个维度的组合关系

完整理解应该是：

~~~text
参数更新方式：
全量微调 / LoRA / QLoRA

训练目标：
SFT / DPO
~~~

它们可以组合：

| 组合 | 含义 | 常见程度 |
|---|---|---|
| 全量微调 + SFT | 全部参数学习指令回答 | 大厂、资源充足 |
| LoRA + SFT | 用 adapter 学指令回答 | 常见 |
| QLoRA + SFT | 低显存学指令回答 | 最常见、性价比高 |
| LoRA + DPO | 用 adapter 做偏好对齐 | 常见 |
| QLoRA + DPO | 低显存做偏好对齐 | 资源受限可用 |

核心句：

> **LoRA 不是 SFT 的替代品，DPO 也不是 QLoRA 的替代品，它们属于不同维度。**

---

## 十七、实战选型怎么做？

### 1. 个人开发者 / 单卡资源

优先：

~~~text
QLoRA + SFT
~~~

适合：

- 单卡 4090；
- 7B / 13B 模型；
- 少量高质量指令数据；
- 领域任务适配。

---

### 2. 中小团队 / 有几张 A100

可以选择：

~~~text
LoRA + SFT
~~~

如果还需要偏好对齐：

~~~text
LoRA + SFT
    ↓
LoRA + DPO
~~~

---

### 3. 大厂 / 资源充足

可以选择：

~~~text
全量微调 + SFT
    ↓
RLHF / DPO
~~~

适合追求极致效果，但成本和工程复杂度都很高。

---

### 4. 需求只是补知识

不要优先微调。

应该用：

~~~text
RAG
数据库
搜索
知识库
~~~

因为知识经常变化，写进参数里维护成本很高。

---

## 十九、结合自己的项目怎么说

如果面试官问：

> **你在项目里什么时候会考虑微调？**

可以这样回答：

在我的项目里，我不会一开始就微调。

如果问题是模型缺少某些金融文档里的事实知识，比如某个季度财报数据、公告内容、行业报告结论，我会优先用 RAG，因为这些知识经常变化，放进向量库或数据库里更灵活。

如果问题是输出格式不稳定，比如报告结构、JSON 字段、摘要格式，我会先用 Prompt、Few-shot 和模板约束。

只有当这些方法都不稳定，比如我希望一个小模型长期稳定生成某种金融分析风格、掌握固定财报分析结构、理解一些内部术语表达，或者希望用 7B 小模型替代大模型降低推理成本时，我才会考虑微调。

方案上，我会优先考虑：

~~~text
QLoRA + SFT
~~~

因为性价比最高，资源需求低。  
如果 SFT 后模型已经会回答，但回答风格还不够符合偏好，比如太啰嗦、不够专业、不够符合金融分析表达，就可以继续做：

~~~text
LoRA / QLoRA + DPO
~~~

让模型学会偏好对齐。

## 二十、面试官可能继续追问

### 1. LoRA 和 SFT 是同一个层面的概念吗？

不是。

LoRA 是：

~~~text
改哪些参数
~~~

SFT 是：

~~~text
学什么目标
~~~

可以用 LoRA 做 SFT，也可以用全量微调做 SFT。

---

### 2. QLoRA 和 LoRA 的区别是什么？

LoRA 是冻结基础模型，只训练低秩 adapter。

QLoRA 是先把基础模型量化到 4-bit，再训练 LoRA adapter。

所以 QLoRA 更省显存，更适合消费级 GPU。

---

### 3. SFT 和 DPO 的区别是什么？

SFT 学的是：

~~~text
按指令回答
~~~

DPO 学的是：

~~~text
好回答比差回答更受偏好
~~~

简单说：

~~~text
SFT：让模型会答
DPO：让模型答得更符合偏好
~~~

---

### 4. 为什么微调不是第一选择？

因为微调成本高：

- 数据要高质量；
- 需要 GPU；
- 调参复杂；
- 容易过拟合；
- 可能灾难性遗忘；
- 基础模型升级后要重训。

所以能用 Prompt、Few-shot、RAG 解决的问题，就不要优先微调。

---

### 5. 为什么不建议用微调补充频繁变化的知识？

因为知识一变，模型参数里的旧知识就过期。

例如：

- 产品价格；
- 政策条款；
- 库存状态；
- 最新公告；
- 财报数据。

这些更适合放在：

~~~text
RAG
数据库
搜索系统
~~~

里实时查询。

---

### 6. 最常见的微调组合是什么？

资源有限时最常见：

~~~text
QLoRA + SFT
~~~

需要偏好对齐时：

~~~text
SFT -> DPO
~~~

大厂资源充足时：

~~~text
全量微调 + SFT + RLHF
~~~

---
