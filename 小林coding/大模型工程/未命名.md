# Temperature、Top-K、Top-P：解码参数到底怎么配？

## 一、面试标准回答

面试官问：**Temperature、Top-K、Top-P 分别控制什么？实际工程中应该怎么配？**

可以这样回答：

这三个参数本质上都在解决同一个问题：

> **模型生成时，如何从下一个 token 的概率分布中进行采样。**

但三者控制的维度不同。

### Temperature：控制概率分布锐度

Temperature 会重新调整概率分布。

~~~text
Temperature ↓
概率分布变尖锐
更倾向高概率 token

Temperature ↑
概率分布变平坦
低概率 token 更容易被选中
~~~

控制的是：

> **随机性（Creativity）**

---

### Top-K：控制候选数量

Top-K 会保留：

~~~text
概率最高的 K 个 token
```

其余全部丢弃。

例如：

~~~text
Top-K = 50
```

表示：

~~~text
只允许前 50 名参与抽奖
```

控制的是：

> **候选集合大小**

---

### Top-P：控制候选概率覆盖范围

Top-P（Nucleus Sampling）会：

~~~text
从高概率 token 开始累加
直到累计概率超过 P
```

例如：

~~~text
Top-P = 0.9
```

表示：

~~~text
保留累计概率达到 90% 的最小集合
```

控制的是：

> **候选集合质量**

---

工程上最常见配置：

~~~text
Temperature = 0.7
Top-P = 0.9

Top-K 不限制
```

因为：

> **Top-P 比 Top-K 更自适应。**

实际经验：

~~~text
代码生成：
Temperature=0

普通聊天：
Temperature=0.7
Top-P=0.9

创意写作：
Temperature=1.0~1.2
Top-P=0.95

数学推理：
Temperature=0
或 Self-Consistency
```

一句话总结：

> **Temperature 控制随机性，Top-K 控制候选数量，Top-P 控制候选质量；现代 LLM 通常优先使用 Temperature + Top-P。**

---

## 二、这篇文章的核心观点

很多人会误以为：

~~~text
Temperature
Top-K
Top-P
```

是三个独立参数。

实际上：

> **它们是在不同阶段作用于同一个概率分布。**

流程：

~~~text
模型输出 logits
      ↓
Temperature 调整分布
      ↓
Top-K / Top-P 截断候选集
      ↓
从候选集随机采样
```

因此：

~~~text
Temperature 控制随机程度

Top-K / Top-P 控制允许随机的范围
```

---

## 三、模型输出的是什么？

假设模型预测：

~~~text
我今天想吃 ___
```

得到：

| Token | 概率 |
|---------|---------|
| 火锅 | 0.40 |
| 烧烤 | 0.25 |
| 面条 | 0.15 |
| 米饭 | 0.10 |
| 苹果 | 0.05 |
| 飞机 | 0.0001 |

这就是：

~~~text
next token probability distribution
```

解码参数就是对这个分布进行处理。

---

## 四、Temperature：控制随机性

Temperature 作用于：

~~~text
softmax 之前
```

公式：

~~~text
P_i = softmax(logit_i / T)
```

其中：

~~~text
T = Temperature
```

---

### 1. Temperature < 1

例如：

~~~text
T = 0.3
```

会让：

~~~text
高概率更高
低概率更低
```

例如：

~~~text
0.40 → 0.70
0.25 → 0.20
0.15 → 0.07
```

分布更尖锐。

效果：

- 更稳定；
- 更保守；
- 更容易复现。

---

### 2. Temperature > 1

例如：

~~~text
T = 1.2
```

会让：

~~~text
高概率下降
低概率上升
```

例如：

~~~text
0.40 → 0.28
0.25 → 0.22
0.15 → 0.18
```

分布变平坦。

效果：

- 更多样；
- 更有创造力；
- 更容易跑偏。

---

### 3. Temperature = 0

严格数学上不存在。

工程实现通常：

~~~text
直接切换 Greedy
```

也就是：

~~~text
永远选最大概率 token
```

---

## 五、Top-K：固定人数入围

假设：

~~~text
Top-K = 3
```

原分布：

| Token | Prob |
|---------|---------|
| 火锅 | 0.40 |
| 烧烤 | 0.25 |
| 面条 | 0.15 |
| 米饭 | 0.10 |
| 苹果 | 0.05 |

处理后：

~~~text
保留：

火锅
烧烤
面条

删除：

米饭
苹果
```

然后重新归一化：

| Token | New Prob |
|---------|---------|
| 火锅 | 0.50 |
| 烧烤 | 0.31 |
| 面条 | 0.19 |

再进行采样。

---

### Top-K 的问题

固定 K 不够智能。

例如：

情况 A：

~~~text
0.90
0.05
0.03
0.01
```

其实保留前 2 个就够了。

---

情况 B：

~~~text
0.05
0.05
0.05
...
```

前 50 个都差不多。

此时：

~~~text
K=10
```

反而截断太狠。

所以：

> **Top-K 不够自适应。**

---

## 六、Top-P：动态人数入围

Top-P 又叫：

~~~text
Nucleus Sampling
```

规则：

~~~text
从高到低累加概率
直到累计概率 ≥ P
```

---

例如：

~~~text
Top-P = 0.9
```

分布：

| Token | Prob |
|---------|---------|
| 火锅 | 0.40 |
| 烧烤 | 0.25 |
| 面条 | 0.15 |
| 米饭 | 0.10 |
| 苹果 | 0.05 |

累计：

~~~text
火锅       0.40
火锅+烧烤 0.65
+面条      0.80
+米饭      0.90
```

达到：

~~~text
0.90
```

停止。

最终候选：

~~~text
火锅
烧烤
面条
米饭
```

苹果被删除。

---

## 七、为什么 Top-P 更流行？

因为它会根据分布自动调整。

### 情况一：模型非常确定

~~~text
0.95
0.03
0.01
```

Top-P=0.9：

~~~text
只保留第一个 token
```

非常稳定。

---

### 情况二：模型不确定

~~~text
0.10
0.09
0.08
0.08
...
```

Top-P=0.9：

~~~text
保留很多 token
```

允许更多探索。

因此：

> **Top-P 会根据模型信心自动调节候选集大小。**

---

## 八、Temperature 和 Top-P 的配合

现代 LLM 最常见组合：

~~~text
Temperature = 0.7
Top-P = 0.9
```

原因：

### Temperature

控制：

~~~text
随机程度
```

---

### Top-P

控制：

~~~text
随机范围
```

两者互补。

---

例如：

~~~text
Temperature = 1.5
```

本来可能抽到很离谱的 token。

但：

~~~text
Top-P = 0.9
```

把极低概率 token 截断了。

于是：

~~~text
有创造力
但不至于胡说八道
```

---

## 九、为什么很多模型默认不用 Top-K？

例如：

- GPT 系列；
- Claude；
- Gemini；
- DeepSeek；

很多 API 默认：

~~~text
Temperature
Top-P
```

但：

~~~text
Top-K 不启用
```

原因：

Top-P 已经具备：

~~~text
动态候选集
```

能力。

Top-K 提供的信息增量很小。

所以：

> **现代大模型通常优先 Temperature + Top-P。**

---

## 十、实际工程怎么配？

### 1. 代码生成

目标：

~~~text
正确性 > 多样性
```

推荐：

~~~text
Temperature=0

Top-P=1
```

直接 Greedy。

---

### 2. SQL 生成

目标：

~~~text
稳定
可复现
```

推荐：

~~~text
Temperature=0
```

---

### 3. JSON 输出

目标：

~~~text
格式正确
```

推荐：

~~~text
Temperature=0
```

---

### 4. RAG 问答

目标：

~~~text
忠于检索内容
```

推荐：

~~~text
Temperature=0.2~0.5
Top-P=0.9
```

避免幻觉。

---

### 5. 普通聊天

目标：

~~~text
自然
不死板
```

推荐
