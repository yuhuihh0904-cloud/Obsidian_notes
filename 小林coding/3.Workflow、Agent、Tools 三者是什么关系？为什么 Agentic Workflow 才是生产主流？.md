## 一、面试标准答案

我理解 **Tools、Agent、Workflow** 不是三个互相替代的概念，而是可以相互组合的三层结构。

**Tools 是最底层的能力单元**，本质上就是封装好的函数，比如搜索网页、查询数据库、调用 API、执行代码等。它只负责“执行”，本身没有决策能力，也不知道什么时候该被调用。

**Agent 是一个带有自主决策能力的系统**，通常由 LLM 作为大脑，结合 Tools、Memory、Prompt 等组件来完成任务。Agent 的核心特点是：它可以自己判断下一步该做什么、要不要调用工具、调用哪个工具、工具返回结果是否足够、什么时候停止。

**Workflow 是更上层的流程编排方式**，它把 LLM、Tools、Agent 等节点按照开发者预先设计好的流程组织起来。Workflow 里面的节点不一定都是 Agent，也可以是一次普通的 LLM 调用，或者一个 Tool。Workflow 的核心特点是：整体流程由开发者在代码里写死，比如先做意图识别，再做检索，再生成答案，流程可控、可预测、好调试。

所以三者最核心的区别可以用一句话概括：

> **Tools 不做决策，只负责执行；Agent 由 LLM 在运行时动态决策；Workflow 是开发者提前把流程决策写在代码里。**

---

## 二、从“谁决定下一步”来理解

这道题最重要的切入点是：

> **下一步该干什么，到底是谁决定的？**

### 1. Tools：没有决策能力，只负责执行

Tool 就像一个普通函数，只是多了一层给 LLM 看的描述信息，比如工具名称、工具作用、参数格式。

比如：

```python
web_search(query)
query_database(sql)
send_email(to, subject, body)
```

这些工具本身不会主动判断：

- 什么时候该搜索？
- 应该搜索什么关键词？
- 搜索完要不要继续搜索？
- 要不要调用别的工具？

它只是在被调用的时候完成具体功能。

所以 Tools 的定位是：

> **能力封装层。**

---

### 2. Agent：LLM 自己做动态决策

Agent 的核心是一个循环过程，可以理解为：

```text
Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

也就是：

1. LLM 先根据用户目标进行思考；
2. 判断是否需要调用工具；
3. 如果需要，就选择工具并生成参数；
4. 工具执行后，把结果返回给 LLM；
5. LLM 根据结果继续判断下一步；
6. 直到它认为任务完成，输出最终答案。

比如用户说：

```text
帮我调研一下某个公司的最新动态，并整理成报告。
```

Agent 可能会自己决定：

- 先调用搜索工具查公司新闻；
- 再查财报；
- 再查行业动态；
- 再对信息做总结；
- 如果信息不够，再继续搜索；
- 最后生成报告。

这个过程中，开发者没有提前写死每一步路径，而是由 LLM 在运行时决定。

所以 Agent 的优势是：

> **灵活，能处理路径不确定的复杂任务。**

但缺点也很明显：

- 执行路径不稳定；
- 同一个问题可能每次走不同流程；
- 调试困难；
- 成本和延迟不可控；
- 需要设置最大轮数、超时、token 预算等停止条件，防止死循环。

---

### 3. Workflow：开发者提前写好流程

Workflow 可以理解成一个确定性的业务流程编排。

比如一个客服系统：

```text
用户问题
   ↓
意图识别
   ↓
根据意图走不同分支
   ↓
知识库检索 / 订单查询 / 转人工
   ↓
生成回答
```

这里面：

- LLM 可以作为某个节点，比如做意图分类或者生成回答；
- Tool 也可以作为某个节点，比如查数据库、查订单、检索知识库；
- Agent 也可以作为某个节点，比如遇到复杂问题时交给 Agent 自主处理。

但关键是：

> **Workflow 的整体流程不是 LLM 自己决定的，而是开发者通过代码中的 if / else、状态机、DAG、流程图提前定义好的。**

所以 Workflow 的优势是：

- 流程清晰；
- 行为可预测；
- 便于调试；
- 成本可控；
- 适合生产环境。

缺点是：

- 灵活性不如 Agent；
- 需要开发者提前设计流程；
- 对预料之外的复杂输入处理能力有限。

## 三、三者对比表

| 维度 | Tools | Agent | Workflow |
|---|---|---|---|
| 本质 | 可调用函数 | 自主决策系统 | 流程编排框架 |
| 决策能力 | 没有，只负责执行 | 有，LLM 动态决策 | 没有自主决策，开发者提前写好 |
| 谁决定下一步 | 调用方决定 | LLM 决定 | 开发者代码决定 |
| 执行方式 | 被动等待调用 | Thought-Action-Observation 循环 | 按预设流程执行 |
| 灵活性 | 低 | 高 | 中等 |
| 可控性 | 高 | 较低 | 高 |
| 调试难度 | 低 | 高 | 低 |
| 成本控制 | 容易 | 较难 | 容易 |
| 适合场景 | 封装单一能力 | 路径不确定的复杂任务 | 流程相对固定的业务系统 |

## 四、三者怎么组合？

实际工程里，通常不会只用其中一种，而是三者组合使用。

比较主流的方式是：

> **用 Workflow 固定主流程骨架，在需要灵活判断的节点嵌入 Agent，其余确定性步骤直接使用 LLM 或 Tools。**

这就是所谓的 **Agentic Workflow**。

它的核心思想是：

```text
Workflow 负责整体可控
Agent 负责局部灵活
Tools 负责具体执行
```

---

## 五、Agentic Workflow 的典型结构

可以理解成下面这种结构：

```text
用户输入
  ↓
Workflow：任务入口和意图识别
  ↓
Workflow：判断任务类型
  ↓
固定流程节点：普通 LLM 调用 / RAG 检索 / 数据库查询
  ↓
复杂节点：交给 Agent 自主判断
  ↓
Agent：选择是否调用工具
  ↓
Tools：搜索、查库、调用 API、执行代码
  ↓
Agent：根据工具结果继续推理
  ↓
Workflow：统一汇总、校验、格式化输出
  ↓
最终结果
```

也就是说，Workflow 像一个“总导演”，控制整体流程；Agent 像一个“局部专家”，负责处理不确定的复杂问题；Tools 像“工具箱”，提供外部能力。

## 六、结合金融分析 Agent 项目的例子

如果结合一个金融分析 Agent 项目，可以这样理解：

```text
用户上传金融 PDF
   ↓
Workflow：PDF 解析与文本清洗
   ↓
Workflow：文档类型识别
   ↓
Workflow：文档切块并写入向量数据库
   ↓
Workflow：根据用户问题进行 RAG 检索
   ↓
Agent：判断是否需要调用行情工具、新闻搜索工具、历史研究记忆
   ↓
Tools：查询股价 / 搜索新闻 / 检索数据库 / 调用外部 API
   ↓
Workflow：整合证据并生成结构化研报
```

在这个架构里：

- **PDF 解析、清洗、切块、入库、检索** 这些步骤比较确定，适合放在 Workflow 里；
- **查询股价、搜索新闻、检索数据库、调用 API** 这些能力适合封装成 Tools；
- **是否需要继续搜索、是否需要补充行情数据、是否需要调用历史研究记忆** 这些判断比较灵活，适合交给 Agent。

这样设计的好处是：

> **既不像纯 Agent 那样不可控，也不像纯 Workflow 那样死板。**

## 七、为什么生产环境更偏向 Agentic Workflow？

生产环境里很少直接使用“完全自由的纯 Agent”，因为纯 Agent 虽然灵活，但是问题也很明显。

### 1. 纯 Agent 的问题

#### 1）路径不可控

同一个任务，Agent 每次可能调用不同工具，走不同路径。

比如同样是“分析一家公司”，它这次可能先搜新闻，下次可能先查财报，再下次可能先查行情。

这会导致线上行为不稳定。

#### 2）调试困难

如果结果出错，很难复现它当时为什么这么做。

因为它的中间推理路径、工具调用顺序、停止条件都可能受模型输出影响。

#### 3）成本不可控

Agent 可能多轮调用 LLM 和工具。

一旦它反复思考、反复搜索，就会增加：

- token 消耗；
- API 调用成本；
- 工具调用成本；
- 用户等待时间。

#### 4）稳定性不足

生产环境更看重稳定性和可预测性。

不能让模型完全自由发挥，否则容易出现：

- 工具乱调；
- 参数生成错误；
- 循环调用；
- 输出格式不稳定；
- 结果不可复现。

---

### 2. 纯 Workflow 的问题

但是完全写死的 Workflow 也有问题。

#### 1）灵活性不足

开发者不可能提前穷举所有用户输入。

如果用户问题稍微复杂一点，固定流程可能无法覆盖。

#### 2）复杂任务处理能力有限

比如用户问：

```text
这家公司最近业绩变化背后的原因是什么？结合财报、新闻和行业趋势分析。
```

这个问题可能需要：

- 查财报；
- 查历史数据；
- 搜索新闻；
- 看行业趋势；
- 判断哪些信息有用；
- 生成综合结论。

如果全部靠固定 Workflow 写死，流程会非常复杂。

#### 3）扩展成本高

每增加一种任务类型，都要手动加分支、加规则、改流程。

系统越做越复杂，维护成本会越来越高。

---

### 3. Agentic Workflow 的优势

Agentic Workflow 的思路就是折中：

> **确定的地方用 Workflow，不确定的地方用 Agent，具体能力用 Tools。**

这样可以同时获得三方面好处：

#### 1）主流程可控

核心链路由 Workflow 控制，比如：

```text
输入校验 → 检索 → 生成 → 校验 → 输出
```

这保证了系统不会完全失控。

#### 2）局部决策灵活

在复杂节点中交给 Agent，比如：

- 要不要搜索？
- 要不要调用数据库？
- 要不要继续追问？
- 要不要结合历史记忆？
- 哪些证据更重要？

这些不确定问题交给 Agent 更合适。

#### 3）工具能力可扩展

Tools 可以不断扩展，比如：

- 搜索工具；
- 数据库工具；
- 代码执行工具；
- 文件解析工具；
- 外部 API 工具；
- 向量检索工具。

Agent 不需要知道底层工具怎么实现，只需要根据工具描述选择调用。

#### 4）更适合工程落地

因为它兼顾了：

- Workflow 的稳定性；
- Agent 的灵活性；
- Tools 的扩展性。

所以在真实生产环境中，Agentic Workflow 往往比纯 Agent 更可靠，也比纯 Workflow 更智能。

## 八、面试官可能追问：为什么不用纯 Agent？

可以这样回答：

纯 Agent 的优点是灵活，但生产环境更关心稳定性、成本和可观测性。

如果所有步骤都交给 Agent 自己决定，那么它每次调用哪些工具、执行多少轮、什么时候停止都不完全稳定。这会带来成本不可控、延迟不可控、结果难复现、调试困难等问题。

所以真实项目中一般不会让 Agent 完全自由发挥，而是会用 Workflow 先固定主流程，比如输入解析、检索、生成、校验、输出这些步骤；只有在局部需要动态判断的时候，才交给 Agent 去做。

这样既能利用 Agent 的自主决策能力，又能通过 Workflow 限制它的行动边界。

---

## 九、面试官可能追问：为什么不用纯 Workflow？

可以这样回答：

纯 Workflow 的优点是稳定、清晰、好调试，但它的问题是灵活性不足。

因为用户输入是不确定的，尤其是在复杂任务中，系统很难提前把所有路径都写死。

比如用户让系统分析一家公司，可能需要查财报、查新闻、查行业信息、结合历史记忆，还要判断哪些信息是关键证据。如果全部用 Workflow 写死，会导致分支越来越多，维护成本越来越高。

所以更好的方式是在主流程上使用 Workflow，在复杂节点中嵌入 Agent，让 Agent 根据当前上下文动态决定是否调用工具、调用哪个工具、是否继续补充信息。

这样可以让系统既稳定，又有一定的自主处理复杂问题的能力。

---

## 十、面试官可能追问：Tools 和 Agent 的关系是什么？

可以这样回答：

Tools 是 Agent 的外部能力，Agent 是 Tools 的调度者。

Tool 本身只是一个函数，比如搜索、查库、调用 API、执行代码。它不会主动决定什么时候调用，也不会理解整个任务目标。

Agent 则由 LLM 驱动，它会根据用户目标和当前上下文判断是否需要调用工具。如果需要，它会选择合适的 Tool，并生成调用参数。Tool 执行后把结果返回给 Agent，Agent 再根据结果继续推理。

所以可以理解为：

```text
Tools 提供能力
Agent 负责决策和调度
```

没有 Tools，Agent 只能依赖模型内部知识，无法获取实时信息和外部系统数据；没有 Agent，Tools 只是孤立函数，需要开发者手动决定调用顺序。

---

## 十一、面试官可能追问：Workflow 和 Agent 的关系是什么？

可以这样回答：

Workflow 和 Agent 的主要区别在于控制流归属不同。

Workflow 的控制流由开发者提前定义，系统按照固定流程执行，比如先分类、再检索、再生成、再校验。

Agent 的控制流由 LLM 在运行时动态决定，它会根据当前任务状态判断下一步要不要调用工具、调用哪个工具、是否继续执行。

所以 Workflow 更适合确定性强、对稳定性要求高的业务流程；Agent 更适合开放性强、路径不确定的复杂任务。

实际工程中，两者可以结合：Workflow 负责主流程，Agent 作为 Workflow 中的一个节点，处理那些需要动态决策的部分。

---

## 十二、可以背诵的最终版

Tools、Agent、Workflow 的核心区别在于：**谁决定下一步该做什么**。

**Tools 是最底层的能力单元，本质上是封装好的函数，只负责执行，没有决策能力。**

**Agent 是由 LLM 驱动的自主决策系统，它可以根据目标动态判断下一步做什么、是否调用工具、调用哪个工具以及什么时候停止。**

**Workflow 是开发者提前定义好的流程编排，它把 LLM、Tools、Agent 等节点组织成一个稳定的业务流程，整体执行路径由代码控制，而不是由模型自由决定。**

实际生产中更主流的是 **Agentic Workflow**。也就是用 Workflow 固定主流程，保证系统稳定、可控、可调试；在局部复杂节点嵌入 Agent，让它处理动态决策；底层通过 Tools 提供搜索、数据库查询、API 调用等具体能力。

这样设计的好处是：既避免了纯 Agent 路径不可控、成本不可控、调试困难的问题，也避免了纯 Workflow 过于死板、难以处理复杂输入的问题。因此，Agentic Workflow 是目前更适合生产落地的 Agent 系统架构。

---

## 十三、一句话总结

> **Tools 是能力，Agent 是决策者，Workflow 是流程骨架；生产中最合理的方式是用 Workflow 控制主流程，用 Agent 处理局部不确定性，用 Tools 执行具体动作。**
