## 一、面试标准回答

如果面试官问：

> 请你介绍一下 AI Agent 的记忆机制，并说明在实际开发中应该如何设计记忆模块？

可以这样回答：

我理解 Agent 的记忆机制，本质上是为了解决两个问题：  
第一，**在一次复杂任务中保持上下文连续性**；  
第二，**在多次任务之间沉淀用户偏好、历史经验和重要知识**。

从工程实现上看，Agent 的记忆通常可以分成四层：

1. **感知记忆**：当前这次调用的原始输入，比如用户输入、上传文件、截图、工具返回的原始内容，生命周期最短。
2. **短期记忆**：当前任务中的上下文，一般就是 LLM 的 `messages` 列表，用来保存用户输入、模型回复、工具调用结果和任务状态。
3. **长期记忆**：跨任务持久化保存的信息，比如用户偏好、历史任务总结、项目背景、外部知识等，通常存储在向量数据库、关系数据库或 KV 存储中。
4. **实体记忆**：从对话中提取出来的结构化事实，比如用户偏好、项目配置、截止时间、预算、公司实体信息等，通常用结构化数据库保存，方便精确查询。

实际设计记忆模块时，核心不是简单地“把聊天记录存起来”，而是要回答三个问题：

- **存什么**：只存对未来任务有价值的信息，比如用户偏好、关键决策、任务结论、外部知识，不存无意义闲聊、中间推理过程和冗余工具日志。
- **怎么存**：非结构化文本适合用 Embedding + 向量数据库做语义检索；结构化偏好和状态字段适合用关系数据库或 KV 存储做精确查询；复杂实体关系可以引入知识图谱。
- **什么时候取**：任务开始前可以主动检索用户偏好和历史背景注入 system prompt；任务执行过程中，如果某一步需要历史知识，可以把“查记忆”封装成 Tool，让 Agent 按需检索。

完整流程可以概括为 **读 → 用 → 写**：

> 任务开始前先读长期记忆和实体记忆，执行过程中用短期记忆维护上下文，任务结束后把有价值的新信息写回长期记忆或实体记忆。

这样 Agent 才能从一个单次问答工具，变成一个能够长期理解用户、持续积累经验的智能助手。

## 二、这篇文章的核心主题

这篇文章主要回答的是：

> AI Agent 的记忆机制有哪些类型？在实际开发中应该如何设计记忆模块？

核心观点是：

> 记忆系统不是简单保存聊天记录，而是要根据不同信息的生命周期、结构化程度和使用方式，设计不同层次的存储与检索机制。

Agent 如果没有记忆，就会出现两个问题：

1. **当前任务不连贯**
   - 不知道上一步做了什么
   - 不知道工具已经返回了什么
   - 不知道当前任务执行到了哪个阶段

2. **跨任务无法积累**
   - 不记得用户偏好
   - 不记得历史项目背景
   - 不记得之前踩过的坑
   - 每次对话都像第一次见用户

所以，记忆是 Agent 从 **单次问答工具** 变成 **真正助手** 的关键。
