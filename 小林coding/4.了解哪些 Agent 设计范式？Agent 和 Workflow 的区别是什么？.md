## 一、面试标准答案

我了解的 Agent 设计范式主要有 **ReAct、Plan-and-Execute、Reflection / Reflexion** 这几类。

它们的核心区别在于：**Agent 是如何决定下一步行动的，以及是否具备规划、执行、反思这几个能力。**

**ReAct** 是最经典的 Agent 范式，全称是 **Reasoning + Acting**，也就是让模型在每一步执行前先思考，再决定要不要调用工具，然后根据工具返回结果继续下一轮推理。它的典型流程是：

```text
Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

**Plan-and-Execute** 是为了解决 ReAct “走一步看一步、缺少全局规划”的问题。它会先让模型生成一个完整计划，然后再逐步执行。执行过程中如果发现结果和预期不一致，还可以动态重规划。

**Reflection** 是在 Agent 执行之后增加自我评估环节，让模型检查当前结果是否满足要求。如果不满足，就重新执行或者调整策略。它的变体 **Reflexion** 会把失败原因和改进建议总结成“经验”，作为下一次尝试的上下文，有点像给 Agent 建立一个“错题本”。

至于 **Agent 和 Workflow 的区别**，我认为最核心的是：**谁决定下一步该做什么。**

**Workflow 是开发者提前写好的确定性流程**，比如第一步检索，第二步重排，第三步生成答案，整个控制流由代码决定。它的优点是稳定、可控、好测试、好调试，缺点是灵活性不足。

**Agent 是把下一步决策权交给 LLM**，模型可以根据当前目标和上下文动态决定是否调用工具、调用哪个工具、是否继续执行以及什么时候停止。它的优点是灵活，能处理开放性任务，缺点是路径不稳定、成本不可控、调试困难。

所以在生产环境中，通常不会优先使用完全自由的纯 Agent，而是更倾向于 **Agentic Workflow**：用 Workflow 固定主流程，在需要动态决策的节点中嵌入 Agent。这样既能保证整体链路稳定可控，又能在局部保留 Agent 的灵活性。

一句话总结就是：

> **ReAct 适合简单的工具调用任务，Plan-and-Execute 适合复杂任务规划，Reflection 适合高质量输出场景；生产环境里更推荐 Agentic Workflow，而不是完全自由的纯 Agent。**

---

## 二、Agent 和 Workflow 的本质区别

### 1. 核心区别：谁决定下一步？

Agent 和 Workflow 最核心的区别不是有没有用 LLM，也不是有没有用工具，而是：

> **下一步该做什么，到底是谁决定的？**

如果是开发者提前通过代码写死流程，那就是 **Workflow**。

如果是 LLM 在运行时根据上下文动态决定下一步，那就是 **Agent**。

---

### 2. Workflow：开发者决定流程

Workflow 可以理解成一个确定性的流程图。

比如一个 RAG 问答系统：

```text
用户问题
  ↓
向量检索
  ↓
Rerank 重排序
  ↓
构造 Prompt
  ↓
LLM 生成答案
  ↓
返回结果
```

对应代码大概是：

```python
def workflow_answer_question(user_query: str):
    # 第一步：固定做向量检索
    docs = vector_db.search(user_query, top_k=5)

    # 第二步：固定做 rerank
    reranked_docs = reranker.rank(user_query, docs)

    # 第三步：固定构造 prompt
    prompt = build_prompt(user_query, reranked_docs)

    # 第四步：固定调用 LLM 生成答案
    answer = llm.generate(prompt)

    return answer
```

这个流程里，LLM 只是其中一个节点。

它不负责决定：

- 要不要检索；
- 要不要重排；
- 要不要调用工具；
- 下一步走哪个分支。

这些都由开发者提前写死。

所以 Workflow 的特点是：

- **流程确定**
- **行为可预测**
- **方便测试**
- **方便调试**
- **成本和延迟更容易控制**

但它的问题是：

- **灵活性不足**
- **处理不了太多预料之外的情况**
- **复杂场景下分支会越来越多**

---

### 3. Agent：LLM 决定流程

Agent 则是把“下一步做什么”的决策权交给 LLM。

比如：

```python
def agent_answer_question(user_query: str):
    memory = []

    while True:
        action = llm.decide(user_query, history=memory)

        if action.type == "search":
            result = search_tool.run(action.query)
            memory.append(result)

        elif action.type == "calculate":
            result = calculator.run(action.expr)
            memory.append(result)

        elif action.type == "database_query":
            result = database_tool.run(action.sql)
            memory.append(result)

        elif action.type == "final_answer":
            return action.content
```

这里开发者没有写死每一步，而是让 LLM 自己判断：

- 要不要搜索；
- 要不要计算；
- 要不要查数据库；
- 要不要继续；
- 什么时候输出最终答案。

所以 Agent 的特点是：

- **灵活性强**
- **能处理开放性任务**
- **可以根据中间结果动态调整路径**
- **适合工具调用和复杂任务执行**

但它的问题也很明显：

- **路径不稳定**
- **同一个输入可能走出不同路径**
- **调试困难**
- **成本和延迟不可控**
- **可能出现循环调用或者无效工具调用**

---

## 三、Workflow 和 Agent 对比表

| 维度 | Workflow | Agent |
|---|---|---|
| 决策者 | 开发者 | LLM |
| 控制流 | 代码提前写死 | 运行时动态决定 |
| 确定性 | 高 | 低 |
| 灵活性 | 低 | 高 |
| 可控性 | 高 | 较低 |
| 调试难度 | 低 | 高 |
| 成本控制 | 容易 | 较难 |
| 适合场景 | 流程固定的业务系统 | 路径不确定的开放任务 |
| 典型例子 | RAG 固定链路、客服流程、审批流程 | 自动搜索、调研分析、复杂工具调用 |

## 四、常见 Agent 设计范式

## 1. ReAct：边思考边行动

### 1.1 ReAct 是什么？

**ReAct** 全称是 **Reasoning + Acting**。

它的核心思想是：

> **让模型先推理，再行动，然后根据行动结果继续推理。**

典型流程是：

```text
Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

也就是：

1. **Thought**：模型先分析当前任务；
2. **Action**：模型决定调用哪个工具，以及传什么参数；
3. **Observation**：工具返回执行结果；
4. 模型读取 Observation，再进入下一轮 Thought；
5. 直到模型认为信息足够，输出最终答案。

---

### 1.2 ReAct 的例子

用户问：

```text
帮我查一下英伟达最近的股价，并分析一下近期新闻。
```

ReAct Agent 可能会这样执行：

```text
Thought: 用户需要最新股价和近期新闻，我需要先查询股价。
Action: stock_price_tool("NVDA")
Observation: NVDA 当前股价为 xxx。

Thought: 已经拿到股价，但还需要近期新闻。
Action: news_search_tool("NVIDIA latest news")
Observation: 搜索到若干相关新闻。

Thought: 现在我有股价和新闻，可以综合分析。
Final Answer: ...
```

这个过程里，Agent 每一步都会根据当前信息决定下一步。

---

### 1.3 ReAct 的优点

ReAct 的优点主要有三个。

第一，**简单直接**。

它不需要提前生成完整计划，而是每一步根据当前状态做决策，工程实现相对简单。

第二，**适合工具调用**。

搜索、数据库查询、代码执行、计算器调用等场景，都很适合用 ReAct。

第三，**可解释性比普通 LLM 更强**。

因为 Thought / Action / Observation 的过程是显式的，开发者可以看到 Agent 是怎么一步步做决策的。

---

### 1.4 ReAct 的缺点

ReAct 最大的问题是：

> **它是“走一步看一步”的局部决策模式。**

也就是说，它每一轮只根据当前状态决定下一步，缺少全局规划。

所以当任务比较复杂、步骤很多、依赖关系很强时，ReAct 容易出现问题：

- 做到一半忘记最初目标；
- 在几个工具之间反复调用；
- 缺少全局任务拆分；
- 容易陷入局部最优；
- 多步复杂任务稳定性不足。

因此，ReAct 更适合：

- 步骤不太多的任务；
- 工具调用类任务；
- 信息查询类任务；
- 简单调研任务。

---

## 2. Plan-and-Execute：先规划，再执行

### 2.1 Plan-and-Execute 是什么？

**Plan-and-Execute** 是为了解决 ReAct 缺少全局规划的问题。

它的核心思想是：

> **先让模型生成完整计划，再按照计划一步步执行。**

典型流程是：

```text
用户目标
  ↓
Planner：生成任务计划
  ↓
Executor：逐步执行计划
  ↓
每步执行结果反馈给 Planner
  ↓
必要时动态重规划
  ↓
最终答案
```

---

### 2.2 Plan-and-Execute 的例子

用户问：

```text
帮我分析腾讯最近几个季度的财报变化，并总结增长原因和风险。
```

如果用 ReAct，它可能会边查边想，容易缺少整体结构。

如果用 Plan-and-Execute，Agent 会先规划：

```text
Plan:
1. 获取腾讯最近几个季度的财报数据；
2. 提取收入、利润、毛利率、费用等核心指标；
3. 对比不同季度的变化趋势；
4. 检索相关新闻和管理层解释；
5. 分析增长原因；
6. 总结潜在风险；
7. 输出结构化报告。
```

然后再逐步执行每一步。

---

### 2.3 动态重规划

Plan-and-Execute 不是“计划写完就死板执行”。

更成熟的实现通常会有 **动态重规划**。

也就是说，每执行完一步之后，会把结果反馈给规划器，让规划器判断：

- 当前结果是否符合预期？
- 后续计划是否还适用？
- 是否需要插入新步骤？
- 是否需要删除无效步骤？
- 是否需要改变执行顺序？

例如原计划是：

```text
1. 搜索竞品 A 的产品信息
2. 搜索竞品 B 的产品信息
3. 做竞品对比
```

但执行第一步时发现：

```text
竞品 A 刚刚发布了一个重大版本更新
```

那么规划器可能会动态调整为：

```text
1. 搜索竞品 A 的产品信息
2. 搜索竞品 A 最新版本更新细节
3. 搜索竞品 B 的产品信息
4. 分析 A 的更新对竞品格局的影响
5. 做竞品对比
```

所以 Plan-and-Execute 的关键不是“先写一个死计划”，而是：

> **先建立全局视角，再根据执行结果动态调整。**

---

### 2.4 Plan-and-Execute 的优点

Plan-and-Execute 的优点是：

- **适合复杂任务**
- **有全局规划能力**
- **任务结构更清晰**
- **执行过程更容易管理**
- **可以在执行前审核计划**
- **适合多步骤、有依赖关系的任务**

比如：

- 深度调研；
- 金融分析；
- 代码迁移；
- 项目规划；
- 多文档分析；
- 复杂报告生成。

---

### 2.5 Plan-and-Execute 的缺点

它的缺点也很明显：

第一，**成本更高**。

因为多了规划器调用，执行过程中还可能多次重规划。

第二，**延迟更高**。

相比 ReAct，它需要先生成计划，再逐步执行，整体链路更长。

第三，**初始计划错误会影响后续执行**。

如果一开始规划方向就错了，后面即使动态调整，也可能很难完全纠正。

所以 Plan-and-Execute 更适合：

> **复杂度高、步骤多、需要全局统筹的任务。**

---

## 3. Reflection：执行后自我评估

### 3.1 Reflection 是什么？

**Reflection** 指的是在 Agent 执行一步或者完成整个任务后，再让模型对结果进行检查和评估。

它的核心思想是：

> **先生成结果，再反思结果是否足够好，如果不好就修正。**

典型流程是：

```text
执行任务
  ↓
生成初版结果
  ↓
Reflection：评估结果质量
  ↓
如果通过：输出最终结果
  ↓
如果不通过：修改策略或重新执行
```

---

### 3.2 Reflection 不是简单调试

面试中容易踩的坑是：

> **把 Reflection 理解成调试工具。**

实际上，Reflection 是一种正式的 Agent 运行时机制。

它不是开发者事后 debug，而是系统流程中的一个节点。

也就是说，Agent 在运行过程中会主动进行自我检查，比如：

- 结果是否回答了用户问题？
- 是否缺少关键信息？
- 是否存在逻辑漏洞？
- 是否引用了不可靠证据？
- 是否需要再次调用工具？
- 是否需要重新生成？

---

### 3.3 Reflection 的例子

比如用户让 Agent 生成一份金融分析报告。

初版报告生成后，Reflection 节点可能检查：

```text
1. 是否覆盖收入、利润、费用、风险？
2. 是否有数据支撑？
3. 是否存在结论和证据不匹配？
4. 是否遗漏用户明确要求的部分？
5. 是否需要补充新闻或财报原文证据？
```

如果发现报告缺少风险分析，就返回修改建议：

```text
当前报告缺少风险部分，需要补充政策风险、行业竞争风险和利润率下降风险。
```

然后 Agent 再重新生成或者补充内容。

---

### 3.4 Reflexion：带经验总结的 Reflection

**Reflexion** 可以理解成 Reflection 的增强版。

普通 Reflection 只是告诉模型：

```text
这个结果不好，需要重做。
```

而 Reflexion 会进一步生成一段具体的失败原因和改进建议：

```text
本次失败原因：
1. 没有充分引用检索到的证据；
2. 对利润下降原因分析过于笼统；
3. 没有区分短期因素和长期因素。

下次改进：
1. 先列出关键证据；
2. 再根据证据归纳原因；
3. 最后单独输出风险部分。
```

这段总结会作为下一次尝试的上下文。

所以 Reflexion 像是给 Agent 建了一个“错题本”：

> **不是简单重做，而是带着失败经验重新做。**

---

### 3.5 Reflection 的优点

Reflection 的优点是：

- **提高输出质量**
- **减少明显错误**
- **适合高质量生成任务**
- **能提升复杂任务的可靠性**
- **适合代码生成、报告生成、文案写作等场景**

尤其是在一次生成很难做到完美的任务中，Reflection 很有价值。

---

### 3.6 Reflection 的缺点

Reflection 最大的代价是：

> **增加 token 消耗和延迟。**

因为每加一轮反思，通常就意味着额外调用一次或多次 LLM。

所以它不适合所有场景。

如果任务很简单，比如普通 FAQ 问答，就没必要加 Reflection。

但如果任务对质量要求高，比如：

- 代码生成；
- 研报生成；
- 法律文本分析；
- 多文档总结；
- 简历优化；
- 复杂推理任务；

那么可以考虑引入 Reflection。

---

## 五、三种 Agent 范式对比

| 范式 | 核心思想 | 优点 | 缺点 | 适合场景 |
|---|---|---|---|---|
| ReAct | 边思考边行动 | 简单、灵活、适合工具调用 | 缺少全局规划，容易走一步看一步 | 简单工具调用、信息查询、轻量任务 |
| Plan-and-Execute | 先规划再执行 | 有全局视角，适合复杂任务 | 成本高、延迟高、规划错会影响后续 | 深度调研、多步骤任务、复杂报告 |
| Reflection / Reflexion | 执行后自我评估和修正 | 输出质量更高，可靠性更强 | 增加 LLM 调用、token 和延迟成本 | 代码生成、高质量报告、复杂文案 |

---

## 六、三种范式怎么选型？

面试中可以从两个维度回答：

> **任务复杂度** 和 **质量要求**。

### 1. 任务简单、步骤不多：用 ReAct

如果任务主要是：

- 查信息；
- 调工具；
- 做简单判断；
- 步骤不超过几轮；

那么 ReAct 就够了。

比如：

```text
查询某只股票最新价格，并结合最近新闻做简短总结。
```

这种任务不需要复杂规划，用 ReAct 即可。

---

### 2. 任务复杂、步骤依赖强：用 Plan-and-Execute

如果任务需要：

- 多步骤拆解；
- 有明显依赖关系；
- 需要全局视角；
- 可能需要动态调整计划；

那么更适合 Plan-and-Execute。

比如：

```text
分析一家公司的连续四个季度财报，并总结业务趋势、增长原因和风险。
```

这种任务最好先规划，再执行。

---

### 3. 质量要求高：叠加 Reflection

如果任务不仅复杂，而且对结果质量要求很高，就可以在 ReAct 或 Plan-and-Execute 上叠加 Reflection。

比如：

```text
生成一份可以直接对外展示的金融研究报告。
```

这种场景不能只生成一次就结束，最好增加自我检查环节。

---

### 4. 实际工程中可以混合使用

这三种范式不是互斥的。

一个复杂系统可以这样组合：

```text
Plan-and-Execute：负责整体任务规划
  ↓
ReAct：负责每个步骤内部的工具调用
  ↓
Reflection：负责关键节点的质量检查
```

例如金融分析 Agent：

```text
用户提出复杂分析问题
  ↓
Plan：拆解分析步骤
  ↓
Execute Step 1：用 ReAct 检索财报数据
  ↓
Execute Step 2：用 ReAct 搜索相关新闻
  ↓
Execute Step 3：用 ReAct 查询历史研究记忆
  ↓
Reflection：检查证据是否充分、结论是否可靠
  ↓
生成最终研报
```

这种组合方式比单一范式更适合复杂生产系统。

---

## 七、为什么生产中不推荐一上来就用纯 Agent？

生产环境中，纯 Agent 最大的问题是：

> **不可控。**

纯 Agent 把控制流交给 LLM，虽然灵活，但会带来很多工程问题。

### 1. 行为不稳定

同样的输入，Agent 可能每次走不同路径。

这会导致线上系统难以预测。

---

### 2. 成本不可控

Agent 可能多次调用 LLM 和工具。

如果没有限制，它可能反复搜索、反复思考，导致：

- token 消耗增加；
- API 成本增加；
- 响应延迟变长。

---

### 3. 调试困难

Workflow 出错时，可以定位到某个固定节点。

但 Agent 出错时，很难判断：

- 是哪一步想错了；
- 是工具选错了；
- 是参数生成错了；
- 是 Observation 理解错了；
- 是停止条件判断错了。

---

### 4. 测试覆盖困难

Workflow 的路径是固定的，可以逐个节点测试。

Agent 的路径是动态的，可能输入稍微变化，执行路径就变了。

这使得测试覆盖率天然更难保证。

---

## 八、生产主流方案：Agentic Workflow

### 1. 什么是 Agentic Workflow？

**Agentic Workflow** 指的是：

> **整体用 Workflow 控制主流程，在局部需要动态决策的节点中嵌入 Agent。**

也就是：

```text
Workflow 负责整体稳定性
Agent 负责局部灵活性
Tools 负责具体执行能力
```

---

### 2. Agentic Workflow 的结构

典型结构如下：

```text
用户输入
  ↓
Workflow：输入解析
  ↓
Workflow：任务分类
  ↓
Workflow：固定链路处理
  ↓
Agent 节点：处理不确定任务
  ↓
Tools：搜索 / 查库 / API / 代码执行
  ↓
Workflow：结果校验
  ↓
Workflow：格式化输出
```

---

### 3. 为什么 Agentic Workflow 更适合生产？

因为它能兼顾两边的优点。

Workflow 提供：

- 稳定流程；
- 可观测链路；
- 可测试节点；
- 可控成本；
- 更容易上线维护。

Agent 提供：

- 动态决策；
- 工具选择能力；
- 复杂任务处理能力；
- 对未知输入的适应能力。

Tools 提供：

- 搜索能力；
- 数据库能力；
- API 能力；
- 代码执行能力；
- 外部系统交互能力。

所以 Agentic Workflow 的优势是：

> **确定的地方写死，不确定的地方交给 Agent。**

---

## 九、结合项目怎么回答？

如果结合我的金融分析 Agent 项目，可以这样回答：

在我的项目里，我不会把所有流程都交给 Agent 自由决策，而是采用更接近 **Agentic Workflow** 的设计。

比如用户上传金融 PDF 后，下面这些步骤是相对确定的，所以我用 Workflow 固定下来：

```text
PDF 解析
  ↓
文本清洗
  ↓
文档类型识别
  ↓
切块
  ↓
写入向量数据库
  ↓
根据用户问题做 RAG 检索
```

这些步骤不需要 Agent 自己决定，因为它们是稳定的数据处理链路。

但在问答和分析阶段，有些事情是不确定的，比如：

- 要不要调用股票行情工具；
- 要不要搜索最新新闻；
- 要不要检索历史研究记忆；
- 要不要跨多个季度财报对比；
- 当前证据是否足够支撑结论。

这些判断就适合交给 Agent 或 Agent-like 的逻辑去处理。

所以我会把项目设计成：

```text
Workflow：负责 PDF 解析、清洗、切块、入库、检索等确定性流程
  ↓
Agent：负责分析阶段的动态决策
  ↓
Tools：负责行情查询、新闻搜索、RAG 检索、历史记忆查询等具体能力
  ↓
Reflection：在生成研报前检查证据是否充分、结论是否可靠
```

这样做的好处是：

- 主链路稳定；
- 工具调用灵活；
- 分析结果可追溯；
- 成本和延迟更容易控制；
- 比纯 Agent 更适合生产落地。

---

## 十、面试官可能追问：ReAct、Plan-and-Execute、Reflection 有什么区别？

可以这样回答：

ReAct、Plan-and-Execute、Reflection 的区别主要在于它们解决的问题不同。

**ReAct 解决的是“模型如何边推理边调用工具”的问题。**  
它通过 Thought、Action、Observation 的循环，让模型每一步先思考，再行动，再根据工具结果继续思考。它适合简单的工具调用任务，但缺少全局规划。

**Plan-and-Execute 解决的是“复杂任务如何全局规划”的问题。**  
它先让模型生成完整计划，再逐步执行，并且可以根据执行结果动态重规划。它适合多步骤、有依赖关系的复杂任务，但成本和延迟更高。

**Reflection 解决的是“结果质量如何提升”的问题。**  
它在执行后增加自我评估环节，让模型检查结果是否满足要求，不满足就重试或修正。它适合代码生成、报告生成等质量要求高的任务，但会增加 token 消耗和延迟。

所以三者不是互斥关系，而是可以组合使用：

```text
Plan-and-Execute 负责整体规划
ReAct 负责单步执行和工具调用
Reflection 负责结果质量检查
```

---

## 十一、面试官可能追问：为什么 Reflection 会提升效果？代价是什么？

可以这样回答：

Reflection 能提升效果，是因为它给 Agent 增加了一层自我评估机制。

普通 Agent 生成结果后就直接输出，如果中间有遗漏、逻辑错误或者证据不足，系统本身不一定能发现。

Reflection 会让模型再检查一遍结果，比如：

- 有没有回答用户问题；
- 有没有遗漏关键要求；
- 证据是否充分；
- 结论是否和证据匹配；
- 输出格式是否符合要求。

如果检查不通过，就让 Agent 重新执行或者修改结果。

它的代价也很明显：**会增加 LLM 调用次数，所以 token 消耗和响应延迟都会上升。**

因此 Reflection 不适合所有任务，通常只在质量要求高的场景使用，比如代码生成、复杂报告、金融分析、法律文本总结等。

---

## 十二、面试官可能追问：Plan-and-Execute 和 ReAct 怎么选？

可以这样回答：

如果任务比较简单，步骤不多，主要是查信息或者调用工具，我会优先用 ReAct，因为它实现简单、响应快、灵活性够用。

如果任务比较复杂，需要先拆解多个步骤，而且步骤之间有依赖关系，我会用 Plan-and-Execute。比如多文档分析、深度调研、复杂报告生成，这些任务需要先有全局规划，否则 ReAct 走一步看一步，很容易中途偏离目标。

所以我的选型原则是：

```text
简单工具调用：ReAct
复杂多步骤任务：Plan-and-Execute
高质量输出任务：叠加 Reflection
生产落地系统：Agentic Workflow
```

---

## 十三、面试官可能追问：为什么说“能用 Workflow 就不要用 Agent”？

可以这样回答：

因为生产环境里，可控性通常比灵活性更重要。

如果一个任务可以用固定流程解决，比如意图识别、知识库检索、答案生成，那用 Workflow 更合适。Workflow 的路径稳定，容易测试，出了问题也能快速定位到具体节点。

但如果一上来就用纯 Agent，虽然看起来更智能，但它的执行路径由 LLM 动态决定，同样的输入可能走出不同路径，工具调用次数和 token 成本也不稳定，调试和测试都会更困难。

所以更合理的做法是：

> **先从简单 Workflow 开始，只有当某个节点确实需要动态决策时，再把这个节点升级成 Agent。**

这就是 Agentic Workflow 的思路。

---

## 十四、最终背诵版

我了解的 Agent 设计范式主要有三种：**ReAct、Plan-and-Execute 和 Reflection / Reflexion**。

**ReAct** 是最经典的工具调用范式，它通过 Thought、Action、Observation 的循环，让模型先思考、再行动、再根据工具返回结果继续推理。它实现简单，适合搜索、查库、计算等工具调用任务，但缺点是走一步看一步，缺少全局规划。

**Plan-and-Execute** 是为了解决 ReAct 缺少全局规划的问题。它先让模型生成完整计划，再逐步执行，并且可以根据执行结果动态重规划。它适合复杂多步骤任务，比如深度调研、多文档分析、金融报告生成，但成本和延迟会更高。

**Reflection** 是在执行后增加自我评估环节，让模型检查结果是否满足要求。如果不满足，就重新执行或者修改策略。它能提升输出质量，适合代码生成、研报生成等高质量场景，但代价是增加 token 消耗和响应延迟。Reflexion 则是在 Reflection 基础上进一步总结失败原因和改进建议，类似给 Agent 建立错题本。

至于 Agent 和 Workflow 的区别，核心在于：**谁决定下一步。**

Workflow 是开发者提前写好的确定性流程，控制流由代码决定，优点是稳定、可控、好测试、好调试；Agent 是由 LLM 在运行时动态决定下一步，优点是灵活，能处理开放性任务，但缺点是路径不稳定、成本不可控、调试困难。

因此生产环境里一般不会优先使用完全自由的纯 Agent，而是更推荐 **Agentic Workflow**：整体用 Workflow 框住主流程，在需要灵活判断的节点中嵌入 Agent。这样既能保证系统稳定可控，又能保留 Agent 在复杂局部任务上的动态决策能力。

---

## 十五、一句话总结

> **ReAct 解决工具调用，Plan-and-Execute 解决复杂规划，Reflection 解决质量提升；Agent 负责动态决策，Workflow 负责稳定流程，生产中最推荐的是 Agentic Workflow。**
