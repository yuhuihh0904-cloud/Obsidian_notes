## 一、面试标准回答

如果面试官问：

> 你们项目里的 Agent 记忆系统是怎么做的？短期记忆和长期记忆分别存在哪里？记忆的粒度是多少？怎么使用？

可以这样回答：

我理解 Agent 的记忆系统一般可以分成两层：**短期记忆**和**长期记忆**。

**短期记忆**主要用于维护当前任务的上下文状态，本质上就是每次调用 LLM 时传入的 `messages` 列表。它会保存当前用户输入、模型中间推理、工具调用结果、阶段性结论等信息，让 Agent 在一次任务执行过程中知道自己“做到哪一步了”。短期记忆通常存在内存中，任务结束后就会清空。

**长期记忆**用于跨任务保存重要信息，比如用户偏好、历史任务经验、项目背景、常见处理流程等。长期记忆不能简单依赖 SQL 的 `LIKE` 或关键词搜索，因为很多时候当前问题和历史记忆之间是语义相关，而不是关键词完全重合。所以工程上通常会使用 **Embedding + 向量数据库** 来实现，把记忆文本转成向量后存入向量库，使用时再把当前任务也转成向量，做语义相似度检索，把相关记忆取出来注入 prompt。

在记忆粒度上，我不会把每一句话都单独存，也不会把整段完整对话原封不动存进去。比较合理的粒度是按 **一次完整交互**、**一个独立知识点** 或 **一个关键事件** 来存。太细会导致记忆碎片化，检索时只召回一部分偏好；太粗又会导致一条记忆里混入大量无关信息，影响 LLM 判断。

整体使用流程是：**任务开始前先根据用户请求检索长期记忆，把相关记忆注入 system prompt；任务执行过程中依赖短期记忆维护上下文；任务结束后再判断本次任务中是否产生了值得沉淀的信息，如果有，就写入长期记忆。**

## 二、核心概念总结

这篇文章主要讲的是：

> Agent 为什么需要记忆系统，以及短期记忆和长期记忆在工程上应该如何设计。

如果没有记忆系统，Agent 每次对话都会像“第一次见用户”一样：

- 不记得用户之前说过的偏好
- 不记得之前做过的任务
- 不记得之前踩过的坑
- 不会随着使用次数增加而变得更懂用户

所以 Agent 记忆系统的作用是：

> 让 Agent 在一次任务中保持上下文连贯性，并且能够跨任务沉淀重要经验。

### 1. 短期记忆是什么？

**短期记忆就是当前任务里的上下文信息。**

在代码实现上，最常见的形式就是一个 `messages` 列表：

    {"role": "user", "content": "帮我分析这段代码"},

    {"role": "assistant", "content": "根据工具结果继续分析"}
