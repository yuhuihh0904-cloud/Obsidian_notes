## 一、面试标准回答

如果面试官问：**“什么是多路召回？为什么 RAG 系统里不能只用向量检索？”**

可以这样回答：

多路召回就是在 RAG 检索阶段，**同时使用多种检索方式召回候选文档，再把多路结果融合排序**，而不是只依赖单一路径。

之所以需要多路召回，是因为没有任何一种检索方式是万能的。**向量检索**擅长语义相似，比如同义词、近义表达、口语化问题，但它对产品型号、专有名词、数字、缩写这类精确词不够敏感；**BM25 关键词检索**正好相反，它擅长精确词匹配，但不理解语义，如果用户问题和文档表达没有词汇重叠，就可能召回不到。

所以生产中的 RAG 系统通常会做 **向量检索 + BM25 关键词检索** 的混合召回。如果用户提问方式很多变，还可以再加一路 **多 Query 扩展召回**，让 LLM 把用户问题改写成多个不同角度的 Query，分别检索后合并结果。

多路召回之后，因为不同检索方式的分数体系不一样，不能直接把向量相似度和 BM25 分数相加，所以通常使用 **RRF 倒数排名融合**。RRF 不看原始分数，只看每个文档在各路结果中的排名：一个文档在多路检索中都排名靠前，最终综合分就更高。

最后，RRF 融合结果通常还只是粗排候选集，如果对质量要求更高，还会接一个 **Rerank 精排模型**，把真正能回答用户问题的 chunk 排到最前面。

一句话总结：

> **多路召回的本质是用多种检索方式互补盲区：向量检索管语义，BM25 管精确词，多 Query 管表达角度差异，再用 RRF 融合结果，最后交给 Rerank 精排。**

---

## 二、文章核心结论

这篇文章主要讲的是：

> **RAG 中只靠单一路检索不够稳定，多路召回可以提升召回覆盖率和鲁棒性。**

单路召回最典型的问题是：

- 只用向量检索：语义匹配可以，但精确词、数字、型号容易漏。
- 只用 BM25：关键词命中可以，但同义词、口语化表达、语义相关内容容易漏。

所以多路召回的核心思想是：

> **不要指望一种检索方式解决所有问题，而是让不同检索方式各自发挥优势。**

---

## 三、为什么单路召回不够？

单路召回的问题在于：**每种检索方式都有天然短板。**

### 1. 向量检索的短板

向量检索依赖 Embedding 模型，把 Query 和文档转成向量，再看语义距离。

它适合：

- 同义词
- 近义表达
- 语义相似
- 口语化问题

但它不擅长：

- 产品型号
- 专有名词
- 数字
- 缩写
- 代码名

因为向量检索更关注整体语义，可能会弱化精确词差异。

---

### 2. BM25 的短板

BM25 依赖关键词匹配、词频和逆文档频率。

它适合：

- 精确词
- 产品型号
- 人名
- 数字
- 专有名词

但它不擅长：

- 同义词
- 近义表达
- 口语化问题
- 语义相关但没有词重叠的内容

所以向量检索和 BM25 的短板正好互补。

---

## 四、多路召回通常有哪些路？

常见的多路召回可以分成三路。

### 1. 向量检索 Dense Retrieval

作用：

> **负责语义召回。**

它通过 Embedding 模型把 Query 和 chunk 转成向量，然后在向量数据库里找相似度最高的 Top-K。

适合解决：

- 用户表达和文档表达不同
- 同义词、近义词问题
- 语义相似但字面不一致的问题

---

### 2. BM25 关键词检索 Sparse Retrieval

作用：

> **负责精确词召回。**

它基于倒排索引、词频和逆文档频率，对包含查询关键词的文档进行排序。

适合解决：

- 产品型号
- 专有名词
- 数字
- 代码符号
- 缩写

中文场景下，BM25 通常还需要先做分词，然后再建立索引和计算匹配分数。

---

### 3. 多 Query 扩展召回

作用：

> **负责覆盖不同提问角度。**

它会先用 LLM 把用户原始问题改写成多个不同版本，然后分别检索，再合并结果。

它解决的问题是：

> **用户提问角度和文档表述角度不一致。**

多 Query 扩展的优点是扩大召回范围，缺点是会增加 LLM 调用次数和检索成本。

---

## 五、多路结果怎么融合？

多路召回之后，每一路都会返回自己的 Top-K 结果。

问题是：

> **不同检索方式的分数不能直接比较。**

比如：

- 向量检索分数可能是余弦相似度。
- BM25 分数是基于词频统计的分数。
- 多 Query 每一路也有自己的排序。

这些分数不是同一个量纲，直接相加不可靠。

所以常用方法是：

> **RRF：Reciprocal Rank Fusion，倒数排名融合。**

RRF 的核心思想是：

> **不看原始分数，只看排名。**

可以简单理解为：

    RRF_score = Σ 1 / (k + rank)

其中：

- `rank`：文档在某一路召回结果中的排名。
- `k`：平滑参数，通常取 60。

如果一个文档在多路检索中都排名靠前，它的 RRF 综合分就会更高。

---

## 六、RRF 融合后还要不要 Rerank？

通常需要。

RRF 解决的是：

> **把多路召回结果合并成一个统一候选列表。**

但它本质上还是粗排。

它只能说明：

> **这个文档在多路召回中表现不错。**

但不能细粒度判断：

- 这个 chunk 是否真正回答了问题
- 内容是否完整
- 是否只是关键词重合
- 是否和用户意图真正相关

所以对质量要求高的 RAG 系统，一般会在 RRF 后接一个 **Rerank 精排模型**。

完整流程是：

    用户 Query
        ↓
    向量检索 + BM25 + 多 Query 扩展
        ↓
    RRF 融合排序
        ↓
    Rerank 精排
        ↓
    选出最终上下文
        ↓
    LLM 生成答案

---

## 七、什么时候需要多路召回？

不是所有场景都必须三路全上，要根据业务复杂度选择。

| 场景 | 推荐方案 |
|---|---|
| 普通知识库问答 | 向量检索即可起步 |
| 有大量产品型号、专有名词、数字 | 向量检索 + BM25 |
| 用户提问方式很多变 | 向量检索 + BM25 + 多 Query 扩展 |
| 对答案质量要求很高 | 多路召回 + RRF + Rerank |

简单说：

> **精确词多，就加 BM25；表达方式复杂，就加多 Query；质量要求高，就加 Rerank。**

## 八、核心记忆点

### 1. 多路召回解决的是单路检索的盲区问题

单路检索不稳定，因为每种检索方式都有短板。

---

### 2. 向量检索和 BM25 是最常见的互补组合

可以这样记：

    向量检索：负责语义相似
    BM25：负责关键词精确匹配

---

### 3. 多 Query 扩展是为了覆盖不同表达角度

它不是解决同义词那么简单，而是解决：

> **用户提问角度和文档表述角度不一致。**

---

### 4. RRF 是多路结果融合的常用方法

RRF 不看原始分数，只看排名，避免了不同检索分数量纲不一致的问题。

---

### 5. RRF 后通常还要接 Rerank

RRF 是粗排融合，Rerank 才负责进一步判断 chunk 是否真正相关。

---

## 九、一句话总结

多路召回就是在 RAG 检索阶段同时使用多种召回方式，**向量检索负责语义相似，BM25 负责精确词匹配，多 Query 扩展负责覆盖不同表达角度，再用 RRF 融合多路结果，并在需要时接 Rerank 精排，从而提高召回覆盖率和最终回答质量。**
