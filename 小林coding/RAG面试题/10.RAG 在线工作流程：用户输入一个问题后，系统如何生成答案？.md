## 一、面试标准回答

如果面试官问：**“你使用 RAG 给大模型一个输入，系统是怎样的工作流程？”**

可以这样回答：

RAG 的在线流程不是简单的“检索 + 生成”，而是一条完整的查询处理链路。

用户输入问题后，系统通常不会直接拿原始问题去检索，而是先做 **Query 预处理**，比如把口语化、带指代、上下文依赖的问题改写成一个独立、清晰、适合检索的问题。

然后系统会使用和离线建库时相同的 **Embedding 模型**，把改写后的 Query 转成向量，再去向量数据库中做 **ANN 近似最近邻搜索**，召回语义相关的文档片段。

在工程实践中，通常不会只依赖向量检索，还会结合 **BM25 / 全文检索** 做多路召回。向量检索擅长语义相似，BM25 擅长关键词、专有名词、产品型号等精确匹配，两路结果可以通过 **RRF** 等方式融合。

召回结果只是粗排，Top-K 中可能包含噪声，所以还需要经过 **Rerank 精排**。Rerank 模型会把 Query 和候选文档片段一起输入模型，重新判断相关性，最终筛选出 Top-3 到 Top-5 个高质量片段。

接着系统把筛选后的参考资料、用户问题和约束指令拼成 Prompt，例如要求大模型 **只能根据参考资料回答**，如果资料中没有答案就回答“无法根据现有资料回答”，不要自行编造。

最后，大模型基于 Prompt 生成答案，并尽量给出 **引用来源 / 溯源信息**，让用户知道答案依据了哪些文档片段，从而提高可信度并降低幻觉风险。

所以完整流程可以概括为：

**Query 预处理 → Query Embedding → 向量检索 + 多路召回 → Rerank 精排 → Prompt 拼装 → 大模型生成 + 溯源。**

---

## 二、这篇文章的核心主题

这篇文章主要讲的是：

> **RAG 系统在“在线查询阶段”到底是怎么工作的。**

也就是用户输入一个问题后，系统从接收问题到最终生成答案，中间完整经过哪些步骤。

很多人只知道 RAG 是：

> **检索 + 生成**

但这只是大框架。

真正工程中的 RAG 在线链路要复杂得多，至少包括：

1. **Query 预处理**
2. **Query Embedding**
3. **向量检索 + 多路召回**
4. **Rerank 精排**
5. **Prompt 拼装**
6. **大模型生成 + 溯源**

---

## 三、为什么不能直接把问题扔给大模型？

大模型本身存在两个问题：

### 1. 上下文窗口有限

知识库可能有大量文档，不可能把所有文档都塞进 Prompt。

所以 RAG 的核心思想是：

> **从大量知识中先找出最相关的少量片段，再让大模型基于这些片段回答。**

---

### 2. 大模型容易幻觉

如果没有外部知识依据，大模型只能依赖参数中的“记忆”回答。

这会带来两个问题：

- 知识可能过时
- 容易编造不存在的内容

所以 RAG 的作用是：

> **给大模型提供真实的参考资料，让它基于资料回答，而不是凭空发挥。**

---

## 四、RAG 在线流程总览

完整流程如下：

```text
用户问题
   ↓
Query 预处理 / Query Rewrite
   ↓
Query Embedding
   ↓
向量检索 + BM25 多路召回
   ↓
Rerank 精排
   ↓
Prompt 拼装
   ↓
LLM 生成答案
   ↓
引用溯源 / 返回结果
```


## 五、第一步：Query 预处理

### 1. 为什么需要 Query 预处理？

用户输入的问题通常不适合直接检索。

比如：

上次那个方案怎么样？

这个问题有两个问题：

1. **有指代**：“那个方案”到底是什么？
2. **依赖上下文**：离开历史对话后无法理解

如果直接把它拿去向量数据库中检索，效果会很差。

所以 Query 预处理的目标是：

> **把用户原始问题改写成一个独立、清晰、适合检索的问题。**

---

### 2. 常见 Query 预处理方式

#### 方式一：简单改写

把口语化问题改成正式问题。

例如：

```
原始问题：上次那个方案怎么样？改写后：请分析上一轮讨论中的 RAG 多路召回方案的优缺点。
```

这样的问题更适合检索。

---

#### 方式二：HyDE

HyDE 全称是：

> **Hypothetical Document Embeddings**

它的做法是：

> 先让大模型根据用户问题生成一个“假设答案”，再用这个假设答案去做向量检索。

听起来有点奇怪，因为我们不是用问题去搜，而是用“可能的答案”去搜。

但这样做有一个核心原因：

> **问题和答案的表达方式可能差异很大，而假设答案和真实文档内容的表达方式更接近。**

例如用户问：

```
RAG 为什么要做多路召回？
```

直接用这个问题检索，可能不如用下面这种假设答案检索效果好：

```
RAG 使用多路召回是为了结合向量检索和关键词检索的优势，提升召回覆盖率和准确率。
```

因为知识库里的文档通常更像“答案描述”，而不是“问题表达”。

---

#### 方式三：多角度扩写

把同一个问题扩展成多个不同表达，然后分别检索。

例如用户问：

```
RAG 怎么减少幻觉？
```

可以扩写成：

```
RAG 如何降低大模型幻觉？RAG 如何通过引用资料提高回答可信度？RAG 中 Prompt 约束如何避免模型编造？RAG 检索结果质量如何影响幻觉？
```

这样做的好处是：

> **提高召回覆盖面，避免单一 Query 表达不准导致漏召回。**

---

## 六、第二步：Query Embedding

Query 预处理完成后，要把 Query 转成向量。

这一步叫：

> **Query Embedding**

也就是使用 Embedding 模型，把文本映射到向量空间中。

---

### 关键点：必须使用和建库时相同的 Embedding 模型

这是一个很重要的工程细节。

离线建库时，文档片段已经被某个 Embedding 模型转成向量并存入向量数据库。

在线查询时，用户 Query 也必须用 **同一个 Embedding 模型** 转成向量。

原因是：

> **不同 Embedding 模型生成的向量不在同一个语义空间中，不能直接比较距离。**

---

### 举例理解

假设离线建库时用的是模型 A：

```
文档 chunk → 模型 A → 文档向量
```

在线查询时如果用了模型 B：

```
用户 Query → 模型 B → 查询向量
```

那就相当于：

```
文档向量在 A 坐标系查询向量在 B 坐标系
```

两个坐标系不同，计算余弦相似度没有意义。

甚至可能出现：

- 向量维度不同
- 相似语义在空间中的方向不同
- 距离计算完全失真

所以：

> **一旦更换 Embedding 模型，整个知识库通常需要重新向量化和重建索引。**

---

## 七、第三步：向量检索 + 多路召回

### 1. 向量检索

Query 被转成向量后，就可以去向量数据库中检索。

常见做法是：

> **ANN 近似最近邻搜索**

也就是在向量库中找到和 Query 向量最相似的 Top-K 个文档片段。

例如：

```
Query 向量   ↓向量数据库   ↓Top-20 相关 chunk
```

向量检索擅长的是：

> **语义相似。**

比如用户问：

```
怎么提升检索质量？
```

向量检索可能召回：

```
多路召回、Rerank、Query Rewrite 可以提升 RAG 检索效果。
```

即使原文没有完全相同的关键词，也能通过语义相似找到相关内容。

---

### 2. 为什么不能只用向量检索？

向量检索也有不足。

它对下面这些内容不一定敏感：

- 精确关键词
- 产品型号
- 人名
- 术语
- 股票代码
- 错误拼写
- 特定缩写

例如用户问：

```
LSTM 和 Transformer 的区别是什么？
```

向量检索可能找到“序列模型对比”这种语义相关内容。

但如果文档中明确出现了：

```
LSTMTransformer
```

这时 BM25 / 全文检索反而可能更准。

---

### 3. 多路召回

所以实际工程中通常使用：

```
向量检索 + BM25 / 全文检索
```

两路召回各有优势：

|召回方式|擅长内容|
|---|---|
|向量检索|语义相似、表达不同但意思接近|
|BM25 / 全文检索|关键词、专有名词、型号、精确匹配|

然后再把两路结果融合。

常见融合方式是：

> **RRF：Reciprocal Rank Fusion，互倒排名融合**

简单理解：

> **一个文档在多个召回通道中排名都靠前，那么它最终得分就更高。**

---

## 八、第四步：Rerank 精排

### 1. 为什么需要 Rerank？

前面的向量检索和 BM25 检索属于 **粗排召回**。

粗排的目标是：

> **先快速找出一批可能相关的候选文档。**

但 Top-K 结果中可能混入不相关内容。

例如召回 Top-20：

```
chunk1：高度相关chunk2：一般相关chunk3：不相关chunk4：只包含关键词但语义不相关...
```

这时就需要 Rerank 精排。

---

### 2. Rerank 是什么？

Rerank 的作用是：

> **对粗排召回的候选文档重新打分排序，筛掉噪声，保留最相关的片段。**

通常流程是：

```
Query + chunk1 → Rerank 模型 → 相关性分数Query + chunk2 → Rerank 模型 → 相关性分数Query + chunk3 → Rerank 模型 → 相关性分数...
```

最后保留：

```
Top-3 ~ Top-5 高质量 chunk
```

---

### 3. Rerank 为什么比向量检索更准？

向量检索通常是：

```
Query 单独编码chunk 单独编码然后计算向量距离
```

而 Rerank 常用的是 **Cross-Encoder** 结构：

```
[Query, chunk] 一起输入模型
```

它可以更细粒度地理解：

- Query 和 chunk 是否真正相关
- chunk 是否回答了问题
- chunk 中的信息是否完整
- 是否只是关键词重合但语义不相关

所以 Rerank 通常比单纯向量相似度更准确。

---

### 4. 为什么不直接用 Rerank 检索全库？

因为 Rerank 计算成本高。

如果知识库有百万级文档：

```
Query + 100 万个 chunk 分别输入模型打分
```

延迟完全不可接受。

所以工程上采用两阶段策略：

```
第一阶段：向量检索 / BM25 粗排，快速召回几十条第二阶段：Rerank 精排，从几十条里选几条最好的
```

这样可以同时兼顾：

- 速度
- 准确率
- 成本

---

## 九、第五步：Prompt 拼装

Rerank 后拿到高质量文档片段，接下来要把它们拼进 Prompt。

一个典型模板如下：

```
prompt = f"""你是一个专业助手，请根据以下参考资料回答用户的问题。如果参考资料中没有相关信息，请回答“根据现有资料无法回答”，不要自行猜测。参考资料：[1] {chunk_1}[2] {chunk_2}[3] {chunk_3}用户问题：{user_query}"""
```

---

### Prompt 拼装的核心目的

Prompt 拼装不是简单地把 chunk 丢进去，而是要明确约束大模型。

主要有三个目的：

---

### 1. 限制回答范围

```
请根据以下参考资料回答用户的问题
```

这句话的目的是：

> **让模型优先依赖检索到的资料，而不是自由发挥。**

---

### 2. 防止模型幻觉

```
如果参考资料中没有相关信息，请回答“根据现有资料无法回答”
```

这句话的目的是：

> **告诉模型资料不足时不要强行编答案。**

因为大模型天然倾向于“尽量回答用户问题”，即使资料中没有答案，也可能编一个看起来合理的回答。

---

### 3. 支持引用溯源

```
[1] chunk_1[2] chunk_2[3] chunk_3
```

给资料编号的目的是：

> **方便模型在回答中标注信息来源。**

例如：

```
根据资料[1]，RAG 在线流程首先需要对 Query 进行预处理。
```

这样用户可以知道答案依据来自哪里。

---

## 十、第六步：大模型生成 + 溯源

### 1. 大模型生成

大模型拿到最终 Prompt 后，会基于参考资料生成答案。

这一步是 RAG 的“生成阶段”。

但需要注意：

> **RAG 生成的答案不等于一定正确。**

即使给了参考资料，大模型仍然可能：

- 误读资料
- 过度总结
- 引入资料中没有的信息
- 混淆多个 chunk 的内容

所以还需要溯源。

---

### 2. 溯源是什么？

溯源就是让模型在答案中标注依据。

例如：

```
RAG 在线流程通常包括 Query 预处理、Query Embedding、召回、Rerank、Prompt 拼装和生成溯源几个步骤。[1]
```

这里的 `[1]` 就对应某个原始文档片段。

---

### 3. 为什么溯源很重要？

溯源的价值在于：

> **让用户可以验证答案是否真的来自资料，而不是模型自己编的。**

它可以提升：

- 答案可信度
- 可解释性
- 可审计性
- 用户信任感

尤其在金融、法律、医疗、企业知识库等场景中，溯源非常关键。

## 十二、结合项目怎么说？

如果结合自己的金融 RAG / FinReportAgent 项目，可以这样说：

在我的项目中，用户上传金融 PDF 后，系统会先把文档解析、清洗、切块并写入 Milvus 向量库。在线查询时，用户输入问题后，系统会先对 Query 进行必要的改写和上下文补全，然后使用和建库一致的 Embedding 模型生成 Query 向量，在 Milvus 中召回相关 chunk。

如果是当前文档问答，就优先检索当前文档；如果是同一公司多期财报对比，就可以基于实体 ID 检索同实体下的多个文档 chunk。召回结果经过筛选后，会和用户问题一起拼接成 Prompt，要求大模型基于证据回答，并在回答中尽量保留页码、文档来源等溯源信息。

这样做的好处是，大模型不是凭空分析金融问题，而是基于 PDF 原文证据进行回答，能够提高回答的可追溯性和可信度。

## 十三、重点对比：粗排和精排

|阶段|作用|特点|
|---|---|---|
|粗排召回|快速找出一批可能相关的文档|速度快，但可能有噪声|
|Rerank 精排|对候选文档重新排序，筛掉不相关内容|更准确，但计算成本更高|

简单理解：

> **粗排负责“别漏掉”，精排负责“选得准”。**

---

## 十四、重点对比：向量检索和 BM25

|检索方式|优点|缺点|
|---|---|---|
|向量检索|擅长语义相似，能处理同义表达|对专有名词、精确关键词不一定敏感|
|BM25|擅长关键词精确匹配|不理解深层语义|
|多路召回|结合两者优势，召回更全面|系统复杂度更高，需要融合排序|

## 十五、这篇文章的核心记忆点

### 1. RAG 在线流程不是简单的“检索 + 生成”

更完整的流程是：

```
Query 预处理 → Query Embedding → 多路召回 → Rerank → Prompt 拼装 → 生成溯源
```

---

### 2. Query 预处理决定检索上限

如果用户问题本身不清楚，后面检索和精排再强也救不回来。

所以要做：

- Query Rewrite
- HyDE
- 多角度扩写

---

### 3. Embedding 模型必须前后一致

离线建库和在线查询必须用同一个 Embedding 模型。

否则：

> **文档向量和 Query 向量不在同一个语义空间中，距离计算没有意义。**

---

### 4. 多路召回比单一路召回更稳

向量检索擅长语义，BM25 擅长关键词。

工程中常用：

```
向量检索 + BM25 + RRF 融合
```

---

### 5. Rerank 是提升检索质量的关键步骤

向量检索召回的是候选集，不一定都相关。

Rerank 可以从候选集中筛出真正高质量的片段。

---

### 6. Prompt 拼装要约束大模型

不能只把 chunk 丢给模型，而是要明确告诉它：

- 只能根据资料回答
- 资料没有就说不知道
- 尽量标注引用来源

---

### 7. 溯源是 RAG 可信度的关键

RAG 答案不是生成完就结束。

还需要让用户知道：

> **这句话依据的是哪份文档、哪个片段。**

这样才能降低幻觉风险，提高答案可信度。

---

## 十六、一句话总结

RAG 在线流程的本质是：

> **先把用户问题改写成适合检索的 Query，再通过向量检索和关键词检索召回相关文档，经过 Rerank 精排筛选高质量证据，最后把证据和问题拼成 Prompt 交给大模型生成可溯源答案。**
