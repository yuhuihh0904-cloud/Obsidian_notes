## 一、面试标准回答

如果面试官问：**“RAG 系统检索效果不好，你会从哪些方面优化？”**

可以这样回答：

RAG 检索优化不能只说“换 Embedding 模型”或者“调 chunk size”，更系统的回答应该从 **四层优化框架** 来讲：

1. **索引层优化**：解决“知识怎么存”的问题，核心是让文档既方便被检索，又能保留足够上下文。
2. **查询层优化**：解决“用户问题怎么转”的问题，通过 Query Rewrite、HyDE、多 Query 等方法，让用户问题更容易命中文档。
3. **召回层优化**：解决“从哪些路径找”的问题，用向量检索、BM25、多 Query 等多路召回方式互补盲区。
4. **重排序层优化**：解决“候选里谁最相关”的问题，用 Rerank 从粗召回结果中筛出最适合放进 Prompt 的 chunk。

我的理解是，RAG 的答案质量很大程度上取决于检索质量。因为 LLM 最终只能基于送进去的 context 回答，如果相关内容没有被召回，后面的 Prompt 和生成模型再强也很难弥补。

所以实际优化时，我一般会优先保证召回覆盖率，再提升排序精度。典型组合是：

> **Parent-Child Chunking 索引 + 向量检索和 BM25 多路召回 + Rerank 精排**

如果用户提问质量比较差，比如口语化、指代不清、表达角度和文档不一致，再额外加 Query Rewrite 或多 Query 扩展。

一句话总结：

> **RAG 检索优化要从索引层、查询层、召回层、重排序层系统分析：索引层保证知识存得合理，查询层保证问题问得对，召回层保证相关内容不漏，Rerank 保证最终进 Prompt 的内容足够准。**

---

## 二、文章核心结论

这篇文章的核心观点是：

> **RAG 检索优化不是零散调参，而是一个分层系统工程。**

可以从四个层次理解：

| 层次 | 解决的问题 | 典型方法 |
|---|---|---|
| **索引层** | 知识怎么存 | Parent-Child Chunking、摘要索引、多粒度索引 |
| **查询层** | 问题怎么转 | Query Rewrite、HyDE、Step-back、多 Query |
| **召回层** | 从哪些路径找 | 向量检索、BM25、多路召回、RRF |
| **重排序层** | 谁最相关 | Cross-Encoder Rerank |

---

## 三、为什么检索优化重要？

RAG 的核心是：

> **先检索相关知识，再让 LLM 基于知识回答。**

所以检索结果直接决定了系统上限。

如果相关 chunk 没被召回，LLM 就没有依据，只能猜或者编。

如果召回了一堆噪声 chunk，LLM 也容易被干扰，生成质量会下降。

因此：

> **生成层优化是锦上添花，检索层优化决定 RAG 的基础能力。**

---

## 四、第一层：索引层优化

索引层解决的是：

> **知识应该怎么切、怎么存，才能既方便检索，又方便 LLM 阅读。**

这里有一个核心矛盾：

| 粒度 | 优点 | 缺点 |
|---|---|---|
| **小 chunk** | 语义聚焦，检索更准 | 上下文不足，LLM 不容易读懂 |
| **大 chunk** | 上下文完整，LLM 更容易理解 | 语义被稀释，检索可能不准 |

所以索引优化的核心思路是：

> **Small-to-Big：小块检索，大块使用。**

典型方法是 **Parent-Child Chunking**：

1. 文档切成小的 child chunk，用来建立向量索引。
2. 每个 child chunk 关联一个更大的 parent chunk。
3. 检索时用 child chunk 精准匹配。
4. 命中后取出对应 parent chunk 给 LLM 阅读。

这样既能保证检索精度，又能保证上下文完整。

---

## 五、第二层：查询层优化

查询层解决的是：

> **用户问题和知识库文档表达方式不一致的问题。**

用户的问题可能是口语化、模糊、带指代的，而知识库通常是正式、书面化的。

所以需要在检索前优化 Query，让它更容易命中文档。

常见方法包括：

| 方法 | 作用 |
|---|---|
| **Query Rewrite** | 把口语化、模糊、有指代的问题改写清楚 |
| **多 Query 扩展** | 生成多个不同角度的问题，扩大召回面 |
| **HyDE** | 先生成假设答案，再用答案风格文本去检索 |
| **Step-back Prompting** | 把具体问题抽象成背景问题，先检索通用知识 |

查询层优化的重点是：

> **让 Query 更接近知识库中的表达方式。**

---

## 六、第三层：召回层优化

召回层解决的是：

> **只走一条检索路径容易漏召回的问题。**

单独使用向量检索或 BM25 都有短板：

| 检索方式 | 擅长 | 短板 |
|---|---|---|
| **向量检索** | 语义相似、同义表达 | 精确词、型号、数字可能漏 |
| **BM25** | 关键词、专有名词、精确匹配 | 不理解语义 |

所以工程中常用 **多路召回**：

> **向量检索 + BM25 + 多 Query 扩展**

多路召回后，需要把不同路径的结果融合。

常用方法是 **RRF**：

> **RRF 不看原始分数，只看排名。一个 chunk 在多路结果中排名都靠前，最终综合分就更高。**

RRF 的意义是解决不同检索分数量纲不一致的问题：

- 向量检索是相似度分数
- BM25 是词频统计分数
- 两者不能直接相加

所以用排名融合更稳定。

---

## 七、第四层：重排序层优化

多路召回之后，通常会得到 20～30 个候选 chunk。

这些候选里可能有噪声，不能全部塞给 LLM：

1. **token 成本高**
2. **上下文太长容易干扰模型**
3. **LLM 可能忽略中间内容，出现 Lost in the Middle**

所以需要 **Rerank 精排**。

Rerank 的作用是：

> **从粗召回候选中选出最相关的 3～5 个 chunk。**

它通常使用 **Cross-Encoder** 结构：

- 向量检索是 Bi-Encoder：Query 和 chunk 分开编码，再算相似度。
- Rerank 是 Cross-Encoder：把 Query 和 chunk 拼在一起，让模型直接判断相关性。

因此 Rerank 更准，但成本更高，所以适合用在小规模候选集上，而不是全库检索。

---

## 八、四层优化怎么组合？

实际落地时，不一定四层全部都上，要根据问题选择。

| 问题表现 | 优先优化 |
|---|---|
| chunk 太碎或上下文不完整 | 索引层：Parent-Child Chunking |
| 用户提问口语化、指代不清 | 查询层：Query Rewrite |
| 精确词或语义内容容易漏召回 | 召回层：向量检索 + BM25 |
| 召回结果多但噪声大 | 重排序层：Rerank |

比较推荐的通用组合是：

> **Parent-Child Chunking + 向量/BM25 多路召回 + Rerank**

如果用户问题质量差，再加：

> **Query Rewrite / 多 Query 扩展**

## 九、核心记忆点

### 1. 检索是 RAG 的能力上限

LLM 只能基于送进去的 context 回答，检索不到正确内容，后面很难补救。

---

### 2. RAG 检索优化要分层看

不要只说换模型、调参数，要从四层讲：

> **索引层、查询层、召回层、重排序层。**

---

### 3. 索引层解决“怎么存”

核心矛盾是：

> **小 chunk 检索准，大 chunk 上下文完整。**

解决思路是：

> **小块检索，大块使用。**

---

### 4. 查询层解决“怎么问”

通过 Query Rewrite、HyDE、多 Query 等方法，让用户问题更容易命中文档。

---

### 5. 召回层解决“怎么找”

向量检索和 BM25 互补，多路召回可以减少漏召回。

---

### 6. 重排序层解决“谁最相关”

Rerank 从粗召回结果中选出最适合进入 Prompt 的 chunk，是提升最终答案质量的重要步骤。

---

## 十、一句话总结

RAG 检索优化的核心不是单点调参，而是一个四层框架：**索引层保证知识存得合理，查询层保证问题表达适合检索，召回层通过多路检索减少漏召回，重排序层用 Rerank 筛出最相关的上下文，最终提升 RAG 答案质量。**
