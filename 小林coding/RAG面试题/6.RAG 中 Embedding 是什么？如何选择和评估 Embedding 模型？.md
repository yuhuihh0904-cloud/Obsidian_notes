## 一、面试标准回答

### 问题：RAG 里的 Embedding 是什么？你是怎么选择和评估 Embedding 模型的？

可以这样回答：

Embedding 在 RAG 里不是简单的“把文本变成向量”，更准确地说，它是把一段自然语言文本映射成一个固定长度的高维向量，并且希望这个向量能够表达文本的语义。

它最关键的性质是：

> **语义相近的文本，向量空间中的方向也接近；语义不相关的文本，向量距离会更远。**

所以 RAG 里的向量检索本质上不是关键词匹配，而是语义匹配。

比如：

```text
苹果手机怎么截图
iPhone 如何截屏
```

这两句话关键词不完全一样，但语义相近，Embedding 后的向量相似度应该很高。

而：

```text
苹果手机怎么截图
苹果汁怎么做
```

虽然都有“苹果”这个词，但语义完全不同，所以向量距离应该比较远。

在相似度计算上，一般常用 **余弦相似度**，因为它主要看两个向量的方向是否一致，而不是只看向量长度。文本长短、表达强度可能影响向量模长，但语义相似更关注方向是否接近。

选 Embedding 模型时，我不会只看排行榜，而会结合业务场景综合考虑几个维度：

**第一，看语言和业务场景。**  
如果知识库主要是中文，我会优先考虑中文效果好的开源模型，比如 BGE 系列或者 Qwen Embedding 系列；如果是中英文混合，可以考虑 bge-m3；如果是英文场景，也可以考虑 OpenAI、Voyage、Cohere 这类 API 模型。

**第二，看数据合规要求。**  
如果是企业内部文档、金融研报、合同、医疗数据这类敏感内容，数据不能出境，就应该优先选择可以本地部署的开源 Embedding 模型。

**第三，看向量维度和成本。**  
向量维度越高，表达能力通常更强，但存储空间、索引构建时间和检索成本也会增加。比如百万级 chunk，如果用 1024 维 float32 向量，单纯向量数据就大约需要 4GB；如果是 3072 维，成本会变成 3 倍。所以不能只追求高维度，要看检索效果和成本之间的平衡。

**第四，看最大输入长度。**  
Embedding 模型的最大输入长度会影响 chunk_size 的设计。如果 chunk 很长，但 Embedding 模型只能处理较短文本，就会截断内容，影响检索效果。

评估 Embedding 模型时，不能只看 MTEB 这种通用排行榜。排行榜可以作为初筛，但不能代表自己的业务效果。更合理的方式是在自己的业务数据上构造测试集。

比如准备几百条：

```text
用户问题 query + 正确答案所在 chunk
```

然后用不同 Embedding 模型去检索，看正确 chunk 是否出现在 Top-K 结果里。

常用指标是 **Hit@K**：

```text
Hit@5 = 0.8
```

意思是：80% 的问题，它们对应的正确 chunk 都出现在检索结果前 5 条里。

如果 Hit@K 很低，就说明模型在当前业务场景下召回效果不好，可能要换 Embedding 模型，也可能要优化 chunking、query rewrite、混合检索或 rerank。

所以总结来说：

> **Embedding 决定了 RAG 的语义召回能力。选型不能只看模型名气或排行榜，而要结合语言、业务场景、合规、维度、成本、输入长度，并在自己的业务数据上用 Hit@K 做检索评估。**

# 三、Embedding 是什么？

## 1. 基本定义

Embedding 是把文本转换成向量的过程。

比如输入一段文本：

```text
苹果手机怎么截图？
```

Embedding 模型会输出一个固定长度的数字列表，例如：

```text
[0.12, -0.31, 0.08, ..., 0.45]
```

如果这个模型是 1024 维，那么不管输入是 10 个字还是 500 个字，最后都会输出一个长度为 1024 的向量。

---

## 2. Embedding 的本质：语义压缩

Embedding 可以理解为一种 **语义压缩**。

它把自然语言中的语义信息压缩到一个高维向量里。

```text
自然语言文本
    ↓
Embedding 模型
    ↓
高维语义向量
```

这个向量不是给人看的，而是给检索系统计算相似度用的。

---

## 3. Embedding 最关键的性质

Embedding 最重要的性质是：

```text
语义相近的文本，向量距离近
语义不同的文本，向量距离远
```

例如：

```text
句子 A：苹果手机怎么截图？
句子 B：iPhone 如何截屏？
```

这两句话虽然关键词不完全一样，但语义相近，所以向量应该接近。

再比如：

```text
句子 A：苹果手机怎么截图？
句子 C：苹果汁怎么做？
```

这两句话都有“苹果”，但语义完全不同，所以向量应该距离较远。

---

## 4. RAG 为什么需要 Embedding？

RAG 的核心是：

```text
用户问题
    ↓
检索相关文档片段
    ↓
把文档片段交给 LLM 生成答案
```

问题在于，用户问题和文档片段不一定使用完全相同的词。

比如用户问：

```text
iPhone 怎么截屏？
```

文档里写的是：

```text
苹果手机截图方法如下。
```

如果只靠关键词匹配，可能匹配不到。

Embedding 可以从语义层面判断它们是同一个意思，所以能提高召回能力。

---

# 四、为什么常用余弦相似度？

## 1. 向量怎么判断相似？

文本变成向量后，就可以计算两个向量之间的相似度。

常见方法包括：

```text
余弦相似度
欧氏距离
点积
```

在文本语义检索里，余弦相似度很常见。

---

## 2. 余弦相似度看什么？

余弦相似度主要看两个向量的 **方向是否一致**。

可以理解成：

```text
两个向量方向越接近
    ↓
夹角越小
    ↓
余弦相似度越接近 1
    ↓
语义越相近
```

如果方向差别很大，说明语义可能不相关。

---

## 3. 为什么不用简单的直线距离？

因为向量的长度可能受到一些非语义因素影响，比如：

```text
文本长度
表达强度
模型输出尺度
```

如果直接比较欧氏距离，可能会把这些因素混进去。

而余弦相似度更关注方向，不太关心向量本身有多长。

所以可以记成一句话：

> **向量方向更像是在表达“语义指向”，余弦相似度就是看两段文本是否指向同一个意思。**

---

# 五、Embedding 和关键词检索的区别

## 1. 关键词检索看字面匹配

关键词检索更关注词是否出现。

比如用户问：

```text
iPhone 如何截屏？
```

如果文档里写的是：

```text
苹果手机怎么截图？
```

关键词检索可能效果不好，因为：

```text
iPhone ≠ 苹果手机
截屏 ≠ 截图
```

---

## 2. Embedding 检索看语义相似

Embedding 检索关注的是意思是否相近。

所以即使用词不同，只要语义接近，也可能被召回。

例如：

```text
iPhone 如何截屏？
苹果手机怎么截图？
```

这两个表达在语义上很接近，Embedding 检索更容易匹配。

---

## 3. 关键词检索也有优势

但是不能说 Embedding 一定完全替代关键词检索。

关键词检索在以下场景很有优势：

```text
精确名词
编号
代码
股票代码
合同条款编号
错误码
专有名词
```

例如：

```text
HTTP 404
AAPL
第 3.2.1 条
Q3-2025
```

这些内容关键词检索往往更稳。

所以实际 RAG 里经常会用：

```text
向量检索 + 关键词检索
```

也就是混合检索。

# 六、常见 Embedding 模型类型

## 1. OpenAI text-embedding 系列

OpenAI 的 text-embedding 系列调用方便，英文效果好，适合快速上手。

常见特点：

```text
API 调用方便
英文效果强
支持降维
不用自己部署
```

缺点是：

```text
有调用费用
数据需要发到外部 API
企业内部敏感数据可能有合规问题
中文场景不一定是最优选择
```

---

## 2. BGE 系列

BGE 是中文和多语言 RAG 场景里非常常见的开源 Embedding 模型系列。

常见选择：

```text
bge-large-zh-v1.5：
中文知识库经典选择，1024 维。

bge-m3：
多语言版本，支持中英混合、多语言检索，也支持多种检索模式。
```

BGE 的优点是：

```text
中文效果好
可以本地部署
数据不出境
适合企业私有知识库
```

---

## 3. Qwen Embedding 系列

Qwen Embedding 是近几年中文和多语言场景里很值得关注的新一代模型。

它的特点是：

```text
中文效果强
多语言能力好
可以本地部署
适合中文企业知识库和金融文档场景
```

如果是中文 RAG 新项目，可以把 Qwen Embedding 和 BGE 一起作为候选模型评估。

---

## 4. 其他 API 模型

还有一些国外 API Embedding 模型，比如：

```text
Voyage
Cohere
Gemini Embedding
```

它们在英文、多语言或长上下文场景里也有不错表现。

但在企业私有数据场景下，需要额外考虑：

```text
数据是否能出境
调用成本
延迟
稳定性
```

# 七、如何选择 Embedding 模型？

## 1. 维度一：语言场景

首先看知识库主要是什么语言。

### 中文知识库

如果主要是中文，可以优先考虑：

```text
BGE 中文模型
Qwen Embedding
```

例如：

```text
企业中文文档
中文财报
中文研报
中文客服知识库
中文产品手册
```

---

### 中英混合

如果知识库中既有中文，又有英文，比如技术文档、论文摘要、代码说明，可以考虑：

```text
bge-m3
多语言 Embedding 模型
```

---

### 英文为主

如果知识库以英文为主，可以考虑：

```text
OpenAI text-embedding
Voyage
Cohere
Gemini Embedding
```

---

## 2. 维度二：数据合规

如果知识库包含敏感数据，比如：

```text
企业内部文档
合同
财务数据
医疗数据
客户信息
未公开研报
```

就要考虑数据是否能发给外部 API。

如果不能出境，就应该选择：

```text
可本地部署的开源 Embedding 模型
```

例如：

```text
BGE
Qwen Embedding
```

---

## 3. 维度三：向量维度

向量维度会影响两个方面：

```text
语义表达能力
存储和检索成本
```

一般来说，维度越高，表达能力可能越强，但成本也越高。

---

## 4. 向量维度如何影响存储？

假设使用 float32 存储，每个维度占 4 字节。

如果有 100 万个 chunk：

### 1024 维

```text
1000000 × 1024 × 4 bytes
≈ 4GB
```

### 1536 维

```text
1000000 × 1536 × 4 bytes
≈ 6GB
```

### 3072 维

```text
1000000 × 3072 × 4 bytes
≈ 12GB
```

所以维度越高，向量库的存储成本和检索成本都会上升。

---

## 5. 维度不是越高越好

不能简单认为：

```text
维度越高，效果一定越好。
```

更合理的思路是：

```text
在自己的业务数据上评估
如果低维模型 Hit@K 已经足够高，就没必要盲目上更高维
```

因为高维会带来：

```text
更高存储成本
更慢检索速度
更高索引构建成本
```

---

## 6. 维度六：最大输入长度

Embedding 模型也有最大输入长度限制。

这会影响 chunk_size 的设计。

如果 chunk 太长，超过 Embedding 模型输入限制，可能会被截断。

一旦截断，chunk 后半部分信息就不会进入向量，检索效果会受影响。

所以选择 Embedding 模型时要确认：

```text
模型最大输入长度
自己的 chunk_size
文档平均长度
是否存在长 chunk
```

---

# 八、如何评估 Embedding 模型？

## 1. 不要只看通用排行榜

MTEB 这类排行榜是有参考价值的，但不能完全依赖。

原因是：

```text
排行榜使用的是通用数据集
你的业务数据可能是金融、医疗、法律、客服、代码
数据分布和排行榜数据不一样
```

所以排行榜第一，不一定在你的业务场景里第一。

---

## 2. 正确做法：在业务数据上评估

最可靠的方法是构造自己的评估集。

格式可以是：

```text
query + 正确答案 chunk
```

例如金融分析场景：

```text
query：
腾讯 Q3 广告业务增长原因是什么？

正确 chunk：
广告收入同比增长明显，主要原因是视频号商业化加速、AI 广告投放系统优化，以及小游戏广告需求提升。
```

然后用候选 Embedding 模型分别做检索，观察正确 chunk 能不能被召回。

---

## 3. 什么是 Hit@K？

Hit@K 是评估召回效果的常用指标。

它的含义是：

> **对于每个问题，如果正确答案 chunk 出现在检索结果前 K 条中，就算命中。**

例如：

```text
Hit@5 = 0.8
```

意思是：

```text
80% 的问题，它们对应的正确答案 chunk 出现在检索结果前 5 条里。
```

---

## 4. Hit@K 举例

假设有 10 个测试问题。

每个问题都有一个标准答案 chunk。

我们用 Embedding 模型检索 Top-5。

结果如下：

```text
问题 1：正确 chunk 在 Top-5 中，命中
问题 2：正确 chunk 在 Top-5 中，命中
问题 3：正确 chunk 不在 Top-5 中，未命中
问题 4：正确 chunk 在 Top-5 中，命中
问题 5：正确 chunk 在 Top-5 中，命中
问题 6：正确 chunk 不在 Top-5 中，未命中
问题 7：正确 chunk 在 Top-5 中，命中
问题 8：正确 chunk 在 Top-5 中，命中
问题 9：正确 chunk 在 Top-5 中，命中
问题 10：正确 chunk 在 Top-5 中，命中
```

命中 8 个，总共 10 个问题：

```text
Hit@5 = 8 / 10 = 0.8
```

说明这个模型在这批测试问题上，80% 的问题可以在前 5 条里找到正确 chunk。

---

## 5. Hit@1、Hit@5、Hit@10 怎么理解？

```text
Hit@1：
正确 chunk 是否排在第一名。
要求最高，反映最精准召回能力。

Hit@5：
正确 chunk 是否出现在前 5 条。
适合 RAG，因为通常会把 Top-3 或 Top-5 放进 Prompt。

Hit@10：
正确 chunk 是否出现在前 10 条。
更偏召回能力，适合后面还有 Rerank 的场景。
```

如果系统有 Rerank，可以更关注 Hit@10，因为只要正确 chunk 先被召回，后面还有机会通过 Rerank 排到前面。

---

## 6. Hit@K 低怎么办？

如果 Hit@K 比较低，不一定只是 Embedding 模型的问题。

可能原因包括：

```text
Embedding 模型不适合当前语言或领域
chunk 切得太大，语义太杂
chunk 切得太小，语义不完整
query 太口语化，没有改写
知识库文档质量差
专有名词、编号、表格不适合纯向量检索
```

对应优化方向：

```text
更换 Embedding 模型
优化 Chunking
加入 Query Rewrite
加入关键词检索 BM25
加入多路召回
加入 Rerank
补充 metadata 过滤
```

# 十、结合自己的金融分析 Agent 项目理解

在金融分析 Agent 项目中，Embedding 模型非常关键。

因为 RAG 要解决的问题是：

```text
用户问一个金融问题
系统要从财报 / 研报 / 公告 chunk 中找出相关证据
再让 LLM 基于证据回答
```

如果 Embedding 模型不适合中文金融文本，就可能出现：

```text
用户问广告业务增长原因
检索却召回游戏业务或者云业务

用户问经营现金流下降原因
检索却召回利润表摘要

用户问毛利率变化
检索不到对应财务指标解释
```

---

## 1. 项目中为什么适合中文 Embedding 模型？

金融 PDF 多数是中文财报、中文研报或中文公告。

所以 Embedding 模型应该优先考虑：

```text
中文语义理解能力
中文金融术语理解能力
本地部署能力
向量维度和 Milvus 存储成本
```

比如候选：

```text
bge-large-zh-v1.5
bge-m3
Qwen Embedding
```

---

## 2. 为什么不能只靠排行榜？

因为金融文档有很多特殊表达：

```text
同比增长
环比下降
毛利率
经营现金流
广告业务
云业务
金融科技服务
非 IFRS 净利润
```

通用排行榜不一定能代表模型对这些金融文本的检索能力。

所以需要在自己的项目数据上测试。

---

## 3. 可以怎么评估？

可以从真实财报里人工构造一批测试集。

例如：

```text
query：
腾讯 Q3 广告业务增长原因是什么？

gold chunk：
包含“视频号商业化、AI 广告投放系统优化、小游戏广告需求提升”的 chunk。
```

再比如：

```text
query：
腾讯 Q3 云业务增长受到哪些因素影响？

gold chunk：
包含“企业数字化需求、AI 云服务需求”的 chunk。
```

然后分别用不同 Embedding 模型检索 Top-5，看 gold chunk 是否命中。

## 4. 项目里可以这样回答面试官

如果面试官问：

```text
你项目里 Embedding 模型怎么选？
```

可以这样回答：

在我的金融分析 Agent 项目里，知识库主要是中文财报和中文研报，所以我不会直接默认用英文效果强的 Embedding 模型，而是优先考虑中文效果更好的开源模型，比如 BGE 或 Qwen Embedding。

选型时我主要看四点：

```text
中文金融文本召回效果

```

评估时我会从财报中构造一批 query 和正确答案 chunk，对不同模型跑 Hit@5。如果某个模型在自己的金融问答测试集上 Hit@5 更高，并且部署和成本可接受，就优先选择它。

这样回答比直接说“我用了某某模型，因为它效果好”更有说服力。

# 十一、面试官可能追问

## 追问 1：Embedding 为什么可以做语义检索？

可以这样回答：

因为 Embedding 模型会把文本映射到高维语义空间中。

在这个空间里，语义相近的文本向量方向更接近，余弦相似度更高；语义不相关的文本向量距离更远。

所以用户问题和文档 chunk 即使用词不同，只要意思相近，也能通过向量相似度被匹配到。

---

## 追问 2：为什么常用余弦相似度？

可以这样回答：

因为余弦相似度主要比较两个向量的方向，而不是向量长度。

文本向量的长度可能受到文本长短或模型输出尺度影响，而语义相似更关注两个向量是否指向相近方向。

所以在文本语义检索中，余弦相似度比较常见。

---

## 追问 3：Embedding 模型是不是维度越高越好？

可以这样回答：

不是。

维度高通常表达能力更强，但也会带来更高的存储成本、索引成本和检索延迟。

比如 100 万个 chunk，1024 维 float32 大约需要 4GB，3072 维就大约需要 12GB。

所以要在自己的业务数据上评估，如果低维模型 Hit@K 已经足够，就没必要盲目追求高维。

---

## 追问 4：为什么不能只看 MTEB 排行榜？

可以这样回答：

MTEB 是通用 Embedding 评测榜单，可以作为参考，但不能直接代表业务效果。

因为自己的数据可能是金融、法律、医疗、客服或者代码文档，和通用评测集分布不一样。

所以最终还是要在自己的业务数据上构造 query + gold chunk 测试集，用 Hit@K 评估真实召回效果。

---

## 追问 5：Hit@K 是什么？

可以这样回答：

Hit@K 表示正确答案 chunk 是否出现在检索结果前 K 条中。

比如 Hit@5 = 0.8，表示 80% 的问题，它们对应的正确 chunk 都被召回到了前 5 条结果里。

在 RAG 里，Hit@K 很重要，因为如果正确 chunk 没有被召回，后面的 LLM 再强也很难回答正确。

---

## 追问 6：Hit@K 低应该怎么办？

可以这样回答：

Hit@K 低不一定只怪 Embedding 模型。

我会从几个方向排查：

```text
1. 模型是否适合当前语言和领域
2. chunk 是否切得太大或太小
3. query 是否需要改写
4. 是否需要引入 BM25 关键词检索
5. 是否需要多路召回
6. 是否需要 Rerank
7. metadata 过滤是否做得合理
```

如果是中文金融场景，我会优先比较 BGE、Qwen Embedding 这类中文模型，然后结合 chunking 和 rerank 一起优化。

# 十二、记忆版总结

这篇文章可以记成一句话：

> **Embedding 决定了 RAG 能不能把“语义相关”的文档 chunk 召回出来。**

Embedding 的核心：

```text
不是简单文本变向量
而是语义相近 → 向量距离近
```

RAG 中的作用：

```text
用户问题 → 向量化
文档 chunk → 向量化
计算相似度 → 召回相关 chunk
```

选型看：

```text
语言场景：中文 / 英文 / 中英混合
业务领域：金融 / 医疗 / 法律 / 客服 / 代码
数据合规：是否能调用外部 API
向量维度：精度、存储、检索速度
最大输入长度：是否匹配 chunk_size
部署成本：本地部署还是 API
```

评估看：

```text
不要只看 MTEB 排行榜
要在自己的业务数据上构造 query + gold chunk
用 Hit@K 测试召回效果
```

```text
表示 80% 的问题，正确 chunk 出现在检索结果前 5 条里。
```
