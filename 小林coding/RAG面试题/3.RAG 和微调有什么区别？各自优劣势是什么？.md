## 一、面试标准回答

### 问题：相比直接微调 LLM，RAG 解决了什么问题？微调和 RAG 各自的优劣势是什么？

可以这样回答：

我理解 RAG 和微调解决的不是同一个层面的问题，它们不是谁完全替代谁的关系。

**微调 Fine-tuning 是修改模型参数**，通过用自己的数据继续训练模型，让模型在参数层面学习某种知识、任务模式、输出格式或者领域表达方式。

**RAG 是不修改模型参数**，而是在推理时先从外部知识库中检索相关资料，再把资料放进 Prompt，让模型基于这些资料生成答案。

所以两者的本质区别是：

```text
微调：改模型参数，把能力或知识写进模型里
RAG：不改模型参数，把外部知识在推理时临时注入 Prompt
```

如果是知识库问答、企业内部文档问答、财报分析、政策问答这类场景，我会优先选择 RAG。因为这些知识经常变化，而且需要答案可溯源。RAG 只需要更新知识库，不需要重新训练模型，成本更低，更新更快。

如果是想让模型学会某种固定输出格式、行业语气、任务风格，或者提升某个垂直领域的深层能力，那微调会更合适。因为微调能从模型行为层面改变它的回答方式。

两者各有优劣：

**微调的优势**是模型行为改变更彻底，推理时不需要额外检索，延迟低，适合定制输出风格和领域能力。  
**微调的缺点**是训练成本高、数据准备成本高、知识更新慢，而且答案不容易溯源。

**RAG 的优势**是知识更新方便、成本低、可接入私有知识库、答案可以追溯到原始文档。  
**RAG 的缺点**是多了一步检索，延迟会增加，并且系统效果强依赖检索质量。如果相关 chunk 没有被召回，后面的大模型再强也很难回答正确。

所以实际工程里不是简单二选一，而是经常组合使用：

> **微调解决“怎么说”，RAG 解决“说什么”。**

比如可以先微调模型，让它学会金融研报的输出格式、分析语气和专业表达；再用 RAG 检索最新财报、研报和公告，让模型基于最新资料生成答案。

这样既保证了回答风格和任务能力，又保证了知识的时效性和可溯源性。

## 二、这篇文章的核心观点

这篇文章主要回答：

> **RAG 和微调有什么区别？各自适合什么场景？**

文章最核心的观点是：

> **RAG 和微调不是二选一关系，而是解决不同层面问题的两种方案。**

可以简单记成：

```text
微调：解决“模型怎么说”
RAG：解决“模型说什么”
```

也就是说：

- 微调用来改变模型行为、输出风格、任务格式、领域表达方式。
- RAG 用来补充外部知识、私有知识、最新知识，并支持答案溯源。

---

## 三、先理解 LLM 的知识本质

大模型的知识主要是在训练阶段通过海量文本学习到的。

训练完成后，这些知识会被固化在模型参数里。

可以理解为：

```text
训练数据
   ↓
模型训练
   ↓
知识被压缩进模型参数
   ↓
训练完成后参数固定
```

这会带来一个问题：

> **模型训练完成之后，不会自动知道新知识，也不会天然知道企业私有数据。**

所以后面才会有两种补知识的思路：

```text
方案一：微调
继续训练模型，把知识或能力写进参数。

方案二：RAG
不改参数，回答时从外部知识库现查资料。
```

---

# 四、微调 Fine-tuning：把知识或能力写进模型参数

## 1. 微调是什么？

微调就是在已有大模型的基础上，用自己的数据继续训练模型。

它会修改模型参数，让模型在行为上发生变化。

可以理解为：

```text
基础模型
   ↓
领域数据 / 任务数据
   ↓
继续训练
   ↓
得到一个更适合特定场景的模型
```

---

## 2. 微调适合解决什么问题？

微调更适合解决下面这类问题：

### 1. 输出格式定制

比如希望模型稳定输出：

```text
一、核心结论
二、关键依据
三、风险提示
四、后续建议
```

如果只是靠 Prompt，模型有时可能不稳定。

微调可以让模型更稳定地学会这种格式。

---

### 2. 回答风格定制

比如希望模型回答更像：

- 金融分析师
- 法律顾问
- 医疗客服
- 代码审查助手
- 企业内部客服

这种“表达风格”和“语气习惯”适合通过微调强化。

---

### 3. 领域术语和任务模式学习

比如某个领域有大量专业术语、固定判断逻辑、固定分析范式。

微调可以让模型更熟悉这种领域表达方式。

---

## 3. 微调的优势

### 优势一：模型行为改变更彻底

微调不是临时把资料塞给模型，而是直接改模型参数。

所以它对模型输出风格、格式、术语使用的影响更深入。

---

### 优势二：推理时延迟更低

微调后的模型在推理时不需要额外检索。

流程就是：

```text
用户问题 → 微调后的模型 → 答案
```

而 RAG 需要：

```text
用户问题 → 检索 → 拼 Prompt → LLM → 答案
```

所以微调在推理链路上更短。

---

### 优势三：适合稳定任务

如果任务规则相对稳定、输出格式固定、知识变化不频繁，微调是比较合适的。

例如：

```text
固定格式客服回复
固定风格文案生成
固定结构报告生成
领域术语表达强化
```

---

## 4. 微调的缺点

### 缺点一：成本高

微调需要：

- 准备训练数据
- 清洗数据
- 标注数据
- 配置训练环境
- 消耗 GPU 资源
- 做模型评估

所以整体工程成本不低。

---

### 缺点二：知识更新慢

如果业务知识经常变化，微调会很麻烦。

例如：

```text
本周更新一批产品说明
下周更新一批政策规则
月底又发布新财报
```

如果每次知识变动都重新微调，成本非常高。

---

### 缺点三：不可溯源

微调后的模型回答来自模型参数。

如果模型给出一个结论，很难知道它到底依据了哪条训练数据。

比如模型回答：

```text
某公司广告业务同比增长 20%
```

你很难直接知道这个数字来自哪份文档、哪一页、哪一段。

这对金融、法律、医疗这类需要证据的场景是不够的。

---

# 五、RAG：不改参数，推理时现查知识

## 1. RAG 是什么？

RAG 的核心是不修改模型参数，而是把知识放在外部知识库中。

用户提问时，系统先检索相关资料，再把资料和问题一起交给大模型。

流程是：

```text
用户问题
   ↓
检索外部知识库
   ↓
召回相关 chunk
   ↓
拼接 Prompt
   ↓
LLM 基于资料生成答案
```

所以 RAG 更像是让模型做“开卷考试”。

模型本身不一定记得所有知识，但系统会在回答前把相关资料找出来给它看。

---

## 2. RAG 适合解决什么问题？

RAG 更适合解决下面这类问题：

### 1. 知识频繁变化

比如：

- 最新财报
- 最新政策
- 最新公告
- 产品文档更新
- 企业制度更新

这些知识变化频繁，不适合每次都微调模型。

RAG 只需要更新知识库即可。

---

### 2. 企业私有知识问答

比如：

- 公司内部文档
- 项目文档
- 客服知识库
- 合同模板
- 业务流程
- 研报和财报 PDF

这些内容通用大模型参数里没有。

RAG 可以把它们接入外部知识库，用户提问时动态检索。

---

### 3. 需要答案溯源

RAG 可以把回答和原始文档 chunk 绑定起来。

例如：

```text
答案依据：
- 文档名
- 页码
- 段落
- chunk 内容
```

这对严肃场景很重要。

---

## 3. RAG 的优势

### 优势一：知识更新方便

RAG 不需要重新训练模型。

只需要：

```text
新增文档 → 清洗切块 → 向量化 → 写入知识库
```

之后用户提问就可以检索到新知识。

---

### 优势二：实现成本低

相比微调，RAG 通常不需要训练大模型。

只需要：

- Embedding 模型
- 向量数据库
- 文档切块
- 检索模块
- Prompt 拼接

小团队也可以比较快落地。

---

### 优势三：答案可溯源

RAG 检索出来的 chunk 可以作为答案依据。

如果回答错了，可以回头看：

```text
是文档本身错了？
是 chunk 切得不好？
是检索没召回？
是 Rerank 排错了？
是 Prompt 没约束好？
```

这比微调更容易排查问题。

---

## 4. RAG 的缺点

### 缺点一：推理延迟更高

RAG 比普通 LLM 多了检索步骤。

流程更长：

```text
Query 改写 → 向量检索 → Rerank → Prompt 拼接 → LLM 生成
```

所以整体延迟会增加。

---

### 缺点二：效果受检索质量限制

RAG 的核心瓶颈是检索。

如果相关 chunk 没有被召回，后面的大模型再强也没有依据。

可以记成一句话：

> **RAG 的生成上限，被检索质量卡住。**

也就是说：

```text
没检索到正确资料
   ↓
Prompt 里没有答案依据
   ↓
LLM 很难回答正确
```

所以优化 RAG 时，不能只想着换更强的 LLM。

更关键的是优化：

- 文档切块
- Embedding 模型
- 向量检索
- 关键词检索
- 多路召回
- Rerank
- Query Rewrite

---

### 缺点三：复杂推理能力提升有限

RAG 只是把参考资料提供给模型。

如果模型本身对某个领域完全缺乏推理能力，只靠检索资料也不一定够。

比如复杂的医疗诊断、法律推理、财务建模分析，可能还需要微调或专门训练来增强模型的领域推理能力。

# 六、RAG 和微调的核心对比

| 对比维度 | 微调 Fine-tuning | RAG |
|---|---|---|
| 是否修改模型参数 | 修改 | 不修改 |
| 知识注入方式 | 训练时写入参数 | 推理时检索注入 Prompt |
| 知识更新 | 需要重新训练或继续训练 | 更新知识库即可 |
| 实现成本 | 高，需要数据和 GPU | 相对低，需要向量库和 Embedding |
| 推理延迟 | 低，无额外检索 | 较高，多一步检索 |
| 答案溯源 | 弱，不知道来自哪条训练数据 | 强，可以追溯到 chunk |
| 适合场景 | 输出格式、风格、领域能力 | 私有知识、动态知识、文档问答 |
| 主要瓶颈 | 数据质量、训练成本、灾难遗忘 | 检索质量、chunk 质量、上下文长度 |
| 典型问题 | “怎么说得更像专业分析师” | “应该基于哪些资料回答” |

# 七、为什么不能简单二选一？

这篇文章强调：

> **RAG 和微调不是谁更好，而是解决的问题不同。**

错误理解是：

```text
微调更高级，所以一定比 RAG 好
```

或者：

```text
RAG 成本低，所以微调没用了
```

这两种说法都不准确。

正确理解是：

```text
微调适合改变模型行为
RAG 适合补充外部知识
```

它们是互补关系。

---

## 1. 什么时候优先选 RAG？

如果需求是下面这些，优先考虑 RAG：

```text
需要回答企业内部文档
知识经常变化
需要引用来源
需要快速上线
不想训练模型
希望降低知识更新成本
```

例如：

- 财报问答
- 企业知识库问答
- 产品文档问答
- 政策法规问答
- 客服知识库问答
- 研报检索分析

---

## 2. 什么时候优先选微调？

如果需求是下面这些，优先考虑微调：

```text
需要稳定输出固定格式
需要特定行业语气
需要模型掌握某种任务模式
需要提升领域推理能力
知识本身比较稳定
```

例如：

- 固定格式报告生成
- 特定风格客服回复
- 医疗问答格式训练
- 法律文书生成风格
- 代码审查风格统一
- 金融研报表达范式学习

# 八、组合使用：微调解决“怎么说”，RAG 解决“说什么”

## 1. 为什么可以组合？

实际工程里，很多系统既需要模型会“查资料”，也需要模型会“按特定方式回答”。

单独用 RAG，可能出现：

```text
资料找到了，但回答格式不稳定
资料找到了，但表达不像专业分析师
资料找到了，但总结方式不符合业务要求
```

单独用微调，可能出现：

```text
格式学会了，但不知道最新知识
风格学会了，但没有私有文档依据
回答很专业，但无法溯源
```

所以组合使用是更合理的方案。

---

## 2. 组合方式

典型组合流程是：

```text
第一步：微调模型
让模型学会固定输出格式、行业语气、专业表达方式。

第二步：接入 RAG
让模型在回答时检索最新资料和私有文档。

第三步：生成答案
模型按照微调学到的方式，基于 RAG 检索到的资料回答。
```

可以概括为：

```text
微调：控制表达方式
RAG：提供事实依据
```

---

## 3. 用一句话理解组合方案

> **微调解决“怎么说”，RAG 解决“说什么”。**

也就是：

```text
怎么说：
回答格式、语气风格、行业表达、任务范式

说什么：
事实内容、最新知识、私有文档、证据来源
```

---

# 九、结合自己的金融分析 Agent 项目理解

在金融分析 Agent 项目中，可以这样解释 RAG 和微调的关系。

## 1. RAG 在项目里解决什么？

RAG 主要解决：

```text
模型不知道用户上传的财报和研报内容
模型不知道最新季度财务数据
模型回答金融问题需要原文证据
```

所以项目中会做：

```text
PDF 解析
   ↓
文档清洗
   ↓
Chunk 切割
   ↓
Embedding 向量化
   ↓
写入 Milvus
   ↓
用户提问时检索相关 chunk
   ↓
LLM 基于证据生成回答
```

这里 RAG 解决的是：

> **说什么：基于哪些财报、研报、公告内容回答。**

---

## 2. 微调在项目里可以解决什么？

如果后续给这个项目加入微调，微调更适合解决：

```text
让模型稳定输出研报格式
让模型回答更像金融分析师
让模型学会风险提示、财务分析、业务拆解的表达方式
让模型更稳定地区分“事实、推断、风险”
```

这里微调解决的是：

> **怎么说：用什么格式、什么语气、什么分析范式回答。**

---

## 3. 项目中的组合表达

面试时可以这样说：

在我的金融分析 Agent 项目中，我目前更侧重使用 RAG，因为金融文档和财报数据更新频繁，而且回答需要有证据来源。RAG 可以把 PDF 财报、研报和公告切块后存入 Milvus，用户提问时动态检索相关 chunk，让模型基于原文回答。

如果后续进一步优化，我会考虑结合微调。微调不负责记住每份财报的具体数字，而是让模型学会更稳定的金融分析格式和表达方式；RAG 则负责提供最新财报和研报证据。

也就是：

```text
RAG 提供事实依据
微调提供表达范式
```

这样比单独使用 RAG 或单独微调更适合金融分析场景。

# 十、常见误区

## 误区一：RAG 也是训练模型

错误说法：

```text
RAG 也是拿自己的数据训练一下模型。
```

正确说法：

```text
RAG 不训练模型，也不改模型参数。
RAG 是把文档构造成外部知识库，推理时检索相关内容注入 Prompt。
```

---

## 误区二：微调一定比 RAG 更好

错误说法：

```text
微调把知识写进模型，所以一定更好。
```

正确说法：

```text
如果知识频繁变化，微调成本很高。
而且微调后的答案不容易溯源。
这种场景 RAG 更合适。
```

---

## 误区三：RAG 可以完全替代微调

错误说法：

```text
RAG 成本低，所以以后不用微调了。
```

正确说法：

```text
RAG 主要解决外部知识注入问题。
微调主要解决模型行为、输出格式、领域能力问题。
两者解决的问题不同。
```

---

## 误区四：RAG 效果不好就换更强 LLM

错误说法：

```text
RAG 回答不好，换一个更强的大模型就行。
```

正确说法：

```text
RAG 的上限很大程度取决于检索质量。
如果正确 chunk 没有被召回，换更强 LLM 也很难回答正确。
应该优先检查 chunk、embedding、召回、rerank、query rewrite。
```

# 十一、面试官可能追问

## 追问 1：RAG 和微调最大的本质区别是什么？

可以这样回答：

最大的区别是是否修改模型参数。

微调会用训练数据继续训练模型，改变模型参数，让模型把某些知识、格式或行为模式学进去。

RAG 不改模型参数，而是把外部知识存在知识库里，用户提问时实时检索相关资料，再把资料注入 Prompt，让模型基于这些资料回答。

所以：

```text
微调是训练时注入
RAG 是推理时注入
```

---

## 追问 2：为什么知识库问答一般优先选 RAG？

可以这样回答：

因为知识库问答通常有三个特点：

1. 知识经常变化
2. 内容来自企业私有文档
3. 答案需要可溯源

如果用微调，每次文档更新都要重新训练，成本很高，而且回答也很难追溯到原始文档。

RAG 更适合这种场景，因为它只需要更新知识库，用户提问时动态检索，并且可以把答案关联到具体 chunk。

---

## 追问 3：RAG 的主要瓶颈是什么？

可以这样回答：

RAG 的主要瓶颈是检索质量。

因为 LLM 只能基于 Prompt 里提供的资料回答。如果相关 chunk 没有被召回，那么模型就没有正确依据。

所以 RAG 优化的重点通常不只是换更强的模型，而是优化检索链路，包括：

```text
文档切块
Embedding 模型
向量检索
关键词检索
多路召回
Rerank
Query Rewrite
```

---

## 追问 4：微调适合解决什么问题？

可以这样回答：

微调更适合解决模型行为层面的问题，比如：

```text
固定输出格式
特定语气风格
行业术语表达
任务范式学习
领域推理能力增强
```

如果知识本身比较稳定，而且希望模型稳定按照某种方式回答，那么微调更合适。

但如果是频繁变化的事实知识，尤其是企业文档和最新资料，就更适合 RAG。

---

## 追问 5：RAG 和微调怎么组合？

可以这样回答：

实际工程中可以先微调再接 RAG。

微调让模型学会固定输出格式、行业语气和任务范式；RAG 负责检索最新知识和私有文档。

比如在金融分析场景中：

```text
微调：
让模型学会研报式表达、风险提示、财务分析结构。

RAG：
检索最新财报、公告、研报原文，提供事实依据。
```

所以可以概括为：

```text
微调解决“怎么说”
RAG 解决“说什么”
```

# 十二、记忆版总结

这篇文章可以记成一句话：

> **RAG 和微调不是二选一，微调解决“怎么说”，RAG 解决“说什么”。**

核心区别：

```text
微调：
修改模型参数，训练时注入知识或行为。

RAG：
不修改模型参数，推理时检索外部知识注入 Prompt。
```

微调适合：

```text
输出格式
回答风格
行业语气
任务范式
稳定领域能力
```

RAG 适合：

```text
企业私有知识
最新知识
动态知识库
文档问答
答案溯源
```

选型原则：

```text
知识频繁变化、需要溯源 → 优先 RAG
输出格式和风格稳定定制 → 优先微调
既要专业表达又要最新知识 → 微调 + RAG 组合
```

最重要的一句话：

```text
微调负责让模型“会说”，RAG 负责让模型“有东西可说”。
```
