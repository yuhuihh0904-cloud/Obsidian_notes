## 一、面试标准回答

### 问题：RAG 中的文档是怎么存的？粒度是多大？Chunking 策略有哪些？

可以这样回答：

RAG 里的文档一般不会整篇直接存进向量数据库，而是会先经过 **文档解析、清洗、切块、向量化、入库** 这几个步骤。

一篇原始文档会被切成多个 **chunk**，每个 chunk 会单独向量化，然后作为一条记录存入向量库。每条记录通常包含三部分：

1. **向量 embedding**
   - 用于语义相似度检索。

2. **chunk 原文**
   - 检索命中后真正塞进 Prompt 给 LLM 阅读的内容。

3. **metadata 元数据**
   - 比如文档名、页码、章节标题、来源路径、时间、文档类型等，用于过滤和溯源。

所以可以理解为：

```text
向量负责“找到”
原文负责“阅读”
metadata 负责“定位和溯源”
```

文档不能整篇存进向量库，主要有两个原因。

第一，Embedding 模型有输入长度限制，长文档可能根本无法一次性编码。  
第二，就算模型支持长输入，把整篇文档压缩成一个向量，也会导致细节信息被平均掉，检索时无法精准定位到具体段落。

至于 chunk 粒度，没有固定答案。一般工程里可以从 **500～1000 token** 作为起点，但更重要的是根据文档类型选择切割策略。

常见的 Chunking 策略有几种：

**第一，固定大小切割。**  
按照固定 token 数或者字符数切，比如每 500 token 切一块。优点是简单、可控；缺点是容易切断句子或语义，所以通常会加 overlap，比如 chunk_size=500，overlap=100。

**第二，语义边界切割。**  
按照自然语义边界切，比如标题、段落、句子。相比固定长度，它能更好地保证 chunk 语义完整，适合文章、Markdown、HTML、技术文档这类结构比较清晰的内容。

**第三，特殊内容专项切割。**  
代码最好按函数、类切；表格最好整体保留，或者转成 Markdown 表格后作为完整 chunk 存储。因为代码和表格如果被硬切，很容易丢失上下文和结构含义。

**第四，父子切割 Parent-Child Chunking。**  
这是实际工程里很常用的一种高质量策略。它的核心思想是：

```text
小 chunk 用来检索，大 chunk 用来返回给 LLM 阅读。
```

也就是存储时同时维护细粒度子 chunk 和粗粒度父 chunk。检索时用小 chunk 精准命中相关位置，然后根据关联关系取出它对应的父 chunk，把更完整的上下文交给 LLM。

这样可以同时兼顾：

```text
检索精度：小 chunk 更聚焦
上下文完整性：父 chunk 信息更完整
```

缺点是存储和索引结构更复杂，存储量也会增加。

所以面试中可以总结为：

> **Chunking 的关键不是简单切成多大，而是在检索精度和上下文完整性之间做平衡。普通文本可以用固定大小加 overlap 兜底，结构化文档优先按标题和段落切，代码表格要专项处理，高质量 RAG 场景可以使用父子切割。**

## 二、这篇文章的核心内容

这篇文章主要讲：

> **RAG 中的文档为什么不能整篇存？应该怎么切？不同 Chunking 策略分别适合什么场景？**

核心观点可以概括为一句话：

> **Chunking 的本质是在“检索精准度”和“上下文完整性”之间做权衡。**

如果 chunk 太大：

```text
语义太杂
向量表示变得笼统
检索不够精准
Prompt 占用更大
```

如果 chunk 太小：

```text
语义不完整
上下文丢失
LLM 读不懂
检索噪音变多
```

所以 Chunking 不是简单地“切得越小越好”，也不是“越大越完整越好”，而是要结合文档类型和业务场景选择合适策略。

---

# 三、RAG 中的文档是怎么存的？

## 1. 文档不能直接整篇存入向量库

原始文档通常不能直接整篇存进向量库。

比如一篇 5000 字的文档，如果整体向量化成一个 embedding，会有两个问题。

---

## 2. 原因一：Embedding 模型有输入长度限制

Embedding 模型一次能处理的 token 数有限。

如果文档很长，比如：

```text
一份几十页 PDF
一篇很长的技术文档
一本产品手册
一份完整财报
```

这些内容通常不能直接整体送入 Embedding 模型。

所以需要先切成多个 chunk。

---

## 3. 原因二：整篇向量化会丢失细节

即使 Embedding 模型支持长输入，也不推荐把整篇文档压缩成一个向量。

因为一篇长文档里可能有很多主题：

```text
退款政策
配送时效
会员积分
售后流程
发票规则
```

如果整篇文档只变成一个向量，这个向量会变成整篇文档的“平均语义”。

当用户只问：

```text
退款政策是什么？
```

系统可能只能检索到整篇笼统文档，而无法精准定位到“退款政策”那一小段。

所以 RAG 中通常会把文档切成多个 chunk，每个 chunk 表示一个更聚焦的语义片段。

---

## 4. 向量库中每条 chunk 记录包含什么？

每条 chunk 记录一般包含三部分：

```text
chunk_id
embedding 向量
chunk 原文
metadata 元数据
```

可以理解成：

| 部分 | 作用 |
|---|---|
| embedding 向量 | 用于相似度检索 |
| chunk 原文 | 检索命中后给 LLM 阅读 |
| metadata | 记录来源、页码、章节、文档类型等 |

---

## 5. 为什么向量和原文都要存？

因为它们负责的事情不同。

```text
向量：给检索系统看的
原文：给 LLM 看的
```

向量本质是一串浮点数，LLM 不能直接基于这串数字生成回答。

真正生成答案时，LLM 读的是 chunk 原文。

所以完整流程是：

```text
用户问题
   ↓
问题向量化
   ↓
在向量库中检索相似 chunk
   ↓
取出命中的 chunk 原文
   ↓
拼接进 Prompt
   ↓
LLM 基于原文生成答案
```

---

# 四、Chunk 粒度怎么定？

## 1. Chunk 太大的问题

如果 chunk 太大，比如一个 chunk 包含 2000 token 甚至更多，会有这些问题：

```text
一个 chunk 中包含多个主题
embedding 表达变得笼统
检索命中不够精准
会占用更多上下文窗口
可能超过 Embedding 模型输入长度
```

举例：

```text
一个 chunk 里同时有：
- 公司收入情况
- 成本结构
- 广告业务
- 云业务
- 风险提示
```

用户问广告业务时，这个 chunk 可能相关，但里面大量内容其实和问题无关。

这样会降低检索精度，也会浪费 Prompt 空间。

---

## 2. Chunk 太小的问题

如果 chunk 太小，比如只有 50 token，也有问题：

```text
语义不完整
上下文断裂
代词指代不清楚
LLM 拿到后无法理解
检索噪音更多
```

比如一个 chunk 只有：

```text
同比增长 18%，主要受上述因素影响。
```

这句话单独拿出来没有意义。

因为不知道：

```text
谁同比增长？
上述因素是什么？
增长的是收入、利润还是用户数？
```

所以 chunk 太小虽然看起来检索更精准，但可能会破坏语义完整性。

---

## 3. 常见经验值

常见起点是：

```text
chunk_size = 500 ~ 1000 tokens
chunk_overlap = 50 ~ 150 tokens
```

但这只是经验值，不是固定答案。

真正工程里要结合：

```text
文档类型
段落结构
Embedding 模型长度限制
LLM 上下文窗口
业务问题粒度
是否需要溯源
是否需要高召回质量
```

---

# 五、策略一：固定大小切割 Fixed Size Chunking

## 1. 什么是固定大小切割？

固定大小切割就是按照固定字符数或 token 数切分文档。

例如：

```text
每 500 token 切成一个 chunk
```

这种方式最简单，容易实现，也容易控制 chunk 大小。

---

## 2. 固定大小切割的优点

优点是：

```text
实现简单
chunk 大小稳定
方便控制向量库记录数量
方便控制 Prompt 长度
适合作为兜底策略
```

---

## 3. 固定大小切割的缺点

缺点是：

```text
不理解语义边界
可能在句子中间切断
可能把一个完整段落拆开
可能把标题和正文切分到不同 chunk
```

比如原文是：

```text
退款申请需要在订单完成后的 7 天内提交，并且商品不能影响二次销售。
```

固定长度可能切成：

```text
退款申请需要在订单完成后的 7 天内提交，并且
```

和：

```text
商品不能影响二次销售。
```

两个 chunk 单独看都不完整。

---

## 4. 为什么要加 overlap？

为了缓解语义被截断的问题，固定大小切割通常会加 overlap。

例如：

```text
chunk_size = 500
overlap = 100
```

意思是：

```text
第一个 chunk：0 ~ 500 token
第二个 chunk：400 ~ 900 token
第三个 chunk：800 ~ 1300 token
```

相邻 chunk 之间保留一部分重叠内容。

这样即使边界切在一句话中间，也有更大概率让完整语义出现在某一个 chunk 中。

---

## 5. 适用场景

固定大小 + overlap 适合：

```text
纯文本
没有明显标题结构的文档
快速搭建 RAG 原型
作为其他切割策略的兜底方案
```

但它不是最优策略，只是最基础、最保底的方案。

---

# 六、策略二：语义边界切割 Semantic / Structure Based Chunking

## 1. 什么是语义边界切割？

语义边界切割就是按照文本天然边界切分。

常见边界包括：

```text
标题
章节
段落
句子
标点
```

核心思想是：

> **不要在语义中间硬切，而是在自然断点处切。**

---

## 2. 为什么语义边界切割更好？

因为一个 chunk 最好是语义完整的。

比如：

```text
一个段落通常围绕一个主题展开
一个章节通常对应一个完整小主题
一个句子至少表达一个完整意思
```

如果按这些边界切，每个 chunk 的独立理解能力会更强。

---

## 3. 常见做法：递归切割

实际工程里通常不会只按一种分隔符切，而是会设置一个优先级。

例如：

```text
优先按标题切
如果太大，再按段落切
如果还太大，再按句子切
如果还太大，再按标点切
最后再按固定长度切
```

可以理解为：

```text
标题 > 段落 > 句子 > 标点 > 固定长度
```

这样既尽量保留语义结构，又能保证 chunk 不超过大小限制。

---

## 4. 标题层级切割

对于 Markdown、HTML、技术文档、产品手册这类有明显标题层级的文档，可以优先按标题切。

比如：

```text
# 产品手册
## 退款政策
### 申请条件
### 申请流程
```

可以切成：

```text
chunk 1：退款政策 > 申请条件
chunk 2：退款政策 > 申请流程
```

同时在 metadata 里保存标题路径：

```text
metadata = {
  "section": "产品手册 > 退款政策 > 申请流程"
}
```

这样有两个好处：

```text
chunk 语义更完整
后续可以根据章节做过滤和溯源
```

---

## 5. 适用场景

语义边界切割适合：

```text
Markdown 文档
HTML 文档
技术博客
产品手册
有标题层级的知识库
段落结构清晰的文章
```

---

# 七、策略三：特殊内容专项处理

## 1. 为什么特殊内容不能通用切割？

普通文本可以按段落、句子切。

但代码和表格这类内容有特殊结构，如果硬切，很容易破坏含义。

---

## 2. 代码文件：按函数或类切割

代码应该尽量按：

```text
函数
类
模块
方法
```

这些结构切割。

因为一个函数通常是最小的逻辑完整单元。

如果把函数切断，比如参数定义在一个 chunk，函数主体在另一个 chunk，LLM 就很难理解这段代码到底做什么。

更好的做法是用语法解析工具。

比如 Python 可以用：

```text
AST
```

识别函数、类的边界。

---

## 3. 表格：尽量整块保留

表格不能简单按行切。

因为表格的每一行都依赖表头。

例如单独看到：

```text
2 小时
```

完全不知道含义。

但如果有表头：

```text
响应时间：2 小时
```

就能理解。

所以表格通常应该：

```text
整张表保留
或转成 Markdown 表格
或把表头补到每一行
```

这样才能保留结构含义。

---

# 八、策略四：父子切割 Parent-Child Chunking

## 1. 父子切割解决什么问题？

前面几种策略都有矛盾：

```text
chunk 小：检索精准，但上下文不完整
chunk 大：上下文完整，但检索不精准
```

父子切割就是为了解决这个矛盾。

它的核心思想是：

> **小块负责检索，大块负责阅读。**

也可以记成：

```text
检索时用放大镜
返回时给全景图
```

---

## 2. 父子切割怎么做？

存储时，系统维护两种 chunk：

```text
子 chunk：小块，比如 200 token
父 chunk：大块，比如 1000 token
```

子 chunk 用于向量检索，因为它语义更聚焦，更容易精准命中用户问题。

父 chunk 用于最终返回给 LLM，因为它包含更完整的上下文。

两者通过 ID 关联。

例如：

```text
parent_chunk_001
    ├── child_chunk_001_1
    ├── child_chunk_001_2
    ├── child_chunk_001_3
    └── child_chunk_001_4
```

---

## 3. 查询时的流程

父子切割的查询流程是：

```text
用户问题
   ↓
问题向量化
   ↓
用子 chunk 向量检索
   ↓
命中相关 child chunk
   ↓
根据 child 的 parent_id 找到父 chunk
   ↓
把父 chunk 原文放进 Prompt
   ↓
LLM 基于完整上下文回答
```

---

## 4. 为什么父子切割效果好？

因为它同时利用了小 chunk 和大 chunk 的优点。

| 对象 | 作用 | 优点 |
|---|---|---|
| 子 chunk | 检索 | 语义聚焦，命中更精准 |
| 父 chunk | 返回 | 上下文完整，LLM 更容易理解 |

所以父子切割适合对回答质量要求比较高的 RAG 场景。

---

## 5. 举个例子

假设原文里有一章：

```text
腾讯 2025 年 Q3 广告业务分析

本季度广告收入同比增长，主要受视频号商业化、AI 广告投放系统优化、小游戏广告需求提升等因素影响。
同时，宏观消费恢复不均衡也对部分广告主预算产生影响。
从长期看，广告业务仍然依赖流量生态、推荐算法和商业化效率提升。
```

如果子 chunk 是：

```text
视频号商业化、AI 广告投放系统优化、小游戏广告需求提升
```

用户问：

```text
腾讯 Q3 广告业务增长原因是什么？
```

子 chunk 很容易被精准检索到。

但如果只把这个子 chunk 给 LLM，信息可能不够完整。

父子切割会根据这个子 chunk 找到完整父 chunk：

```text
腾讯 2025 年 Q3 广告业务分析
本季度广告收入同比增长，主要受……
同时……
从长期看……
```

这样 LLM 得到的上下文更完整，回答质量更好。

---

## 6. 父子切割的代价

父子切割不是没有成本。

它的缺点是：

```text
存储结构更复杂
需要维护 child_id 和 parent_id
索引构建更复杂
存储量可能增加
检索后还要做 parent 映射
```

但在高质量 RAG 场景中，这个成本通常是值得的。

---

# 九、Late Chunking 简单理解

文章还提到一种新的思路：**Late Chunking**。

传统 Chunking 是：

```text
先切块
再对每个 chunk 单独做 Embedding
```

问题是：

```text
每个 chunk 编码时看不到全文上下文
跨 chunk 的指代、背景、对比关系可能丢失
```

Late Chunking 的思路是反过来：

```text
先把整篇文档送入支持长上下文的 Embedding 模型
得到 token 级别表示
再按照 chunk 边界做 pooling
得到每个 chunk 的向量
```

也就是：

```text
传统方式：先切后编码
Late Chunking：先编码后切
```

它的好处是：

```text
chunk 向量中保留了更多全文上下文信息
```

但缺点也明显：

```text
需要 Embedding 模型支持长上下文
计算和显存开销更大
实现复杂度更高
```

所以它属于更进阶的优化方案，不是最基础的 RAG 必选项。

# 十、几种 Chunking 策略对比

| 策略 | 适用文档 | 优点 | 缺点 |
|---|---|---|---|
| 固定大小 + overlap | 纯文本、无明显结构文档 | 简单、稳定、容易实现 | 可能切断语义 |
| 语义边界切割 | 段落清晰的文章 | 语义完整，检索质量更好 | chunk 大小不均 |
| 标题层级切割 | Markdown、HTML、技术文档 | 保留结构，metadata 清晰 | 依赖文档结构质量 |
| 代码按函数切割 | 代码文件 | 保留代码逻辑完整性 | 需要 AST 等解析工具 |
| 表格整块保留 | 表格、财报数据 | 保留表头和行列关系 | 表格过大时不好处理 |
| 父子切割 | 高质量 RAG 场景 | 检索精准 + 上下文完整 | 存储和索引更复杂 |
| Late Chunking | 长文档、高质量检索 | chunk 向量保留全文上下文 | 依赖长上下文 Embedding，开销大 |

---

# 十一、结合自己的金融分析 Agent 项目理解

在金融分析 Agent 项目里，Chunking 特别重要。

因为金融 PDF、财报、研报有几个特点：

```text
文档长
表格多
章节结构明显
财务指标密集
用户问题通常很细
```

如果 chunk 切得不好，就会出现：

```text
检索不到关键财务数据
检索到的 chunk 太泛
表格和解释被切开
答案没有足够上下文
```

所以可以这样设计：

## 1. 财报正文

财报正文可以优先按章节、段落切割。

例如：

```text
管理层讨论与分析
主营业务分析
风险因素
财务摘要
```

这些章节天然有语义边界。

---

## 2. 财务表格

财务表格不能简单按固定 token 切。

应该尽量：

```text
保留表头
保留指标名称
保留单位
保留年份 / 季度
转成 Markdown 表格
```

否则 LLM 看到单独数字时不知道它表示什么。

---

## 3. 高质量问答可以用父子切割

对于金融问答，父子切割很适合。

因为用户的问题通常很细：

```text
腾讯 Q3 广告收入增长原因是什么？
经营现金流变化原因是什么？
毛利率提升来自哪些业务？
```

这些问题需要小 chunk 精准命中。

但回答时又需要完整上下文，比如前后段落中的解释、风险、对比数据。

所以可以：

```text
子 chunk：用于检索具体指标、原因、业务描述
父 chunk：返回完整段落或完整小节
```

这样能同时保证：

```text
召回精准
回答有上下文
分析更完整
```

# 十二、面试官可能追问

## 追问 1：为什么不能把整篇文档直接向量化？

可以这样回答：

主要有两个原因。

第一，Embedding 模型有输入长度限制，长文档可能超过模型可处理范围。

第二，就算可以编码，整篇文档会被压缩成一个向量，细节语义会被平均掉。用户问的是某个具体段落时，系统只能召回整篇文档，无法精准定位。

所以 RAG 通常要把文档切成多个 chunk，每个 chunk 单独向量化入库。

---

## 追问 2：chunk 是不是越小越好？

可以这样回答：

不是。

chunk 小确实更聚焦，检索可能更精准，但如果太小，会导致语义不完整。

比如一个 chunk 里只有“同比增长 20%”，没有指标名称、时间范围和原因解释，LLM 根本无法理解。

所以 chunk 粒度要平衡：

```text
太大：检索不精准
太小：上下文不完整
```

一般可以从 500～1000 token 起步，再根据文档类型和效果调整。

---

## 追问 3：overlap 有什么作用？

可以这样回答：

overlap 是为了缓解切割边界破坏语义的问题。

固定大小切割可能把一句话或一段话切成两半。加 overlap 后，相邻 chunk 会共享一部分内容，使跨边界的语义更有可能完整出现在某一个 chunk 中。

比如：

```text
chunk_size = 500
overlap = 100
```

表示后一个 chunk 会重复前一个 chunk 末尾 100 token。

---

## 追问 4：父子切割为什么比普通切割好？

可以这样回答：

普通切割要在小 chunk 和大 chunk 之间做取舍。

小 chunk 检索准，但上下文少；大 chunk 上下文完整，但检索不准。

父子切割把两者分开：

```text
小 chunk 用于检索
大 chunk 用于返回
```

检索时用小 chunk 精准定位，生成时返回它对应的父 chunk，让 LLM 看到完整上下文。

所以它能兼顾检索精度和上下文完整性。

---

## 追问 5：代码和表格为什么不能按普通文本切？

可以这样回答：

因为代码和表格有强结构。

代码如果从函数中间切开，参数、逻辑和返回值可能分散到不同 chunk，语义就不完整。

表格每一行都依赖表头，如果只切出某一行或某个数字，模型不知道这个数字对应什么指标。

所以代码最好按函数或类切，表格要尽量整块保留，或者转成 Markdown 表格并保留表头。

---

# 十三、记忆版总结

这篇文章可以记成一句话：

> **Chunking 的核心不是切多小，而是在检索精准度和上下文完整性之间做平衡。**

RAG 文档入库流程：

```text
原始文档
   ↓
解析清洗
   ↓
切成多个 chunk
   ↓
每个 chunk 做 Embedding
   ↓
存入向量库
```

每条记录包含：

```text
embedding：负责检索
原文：负责给 LLM 阅读
metadata：负责过滤和溯源
```

为什么不能整篇存：

```text
1. Embedding 模型有长度限制
2. 整篇向量化会让细节语义被平均掉
```

chunk 粒度原则：

```text
太大：语义太杂，检索不准
太小：语义不完整，LLM 看不懂
```

常见策略：

```text
固定大小 + overlap：基础兜底
语义边界切割：按标题、段落、句子切
特殊内容处理：代码按函数切，表格整块保留
父子切割：小块检索，大块返回
Late Chunking：先整体编码，再按 chunk pooling
```

面试中最重要的一句话：

```text
普通文本用固定大小加 overlap 兜底；
结构化文档优先按标题和段落切；
代码和表格要专项处理；
高质量 RAG 场景可以用父子切割，兼顾检索精度和上下文完整性。
```
