## 一、面试标准回答

如果面试官问：**“Query Rewrite 里的多 Query 扩展是什么？为什么要这样做？”**

可以这样回答：

**多 Query 扩展**是 Query Rewrite 的一种方法，它不是只把用户原始问题改写成一个更好的问题，而是让 LLM 从多个角度生成 **3～5 个不同版本的 Query**，然后分别拿这些 Query 去检索，最后把多路检索结果合并去重。

它解决的核心问题是：**单个 Query 的表达角度太窄，可能覆盖不到知识库里的正确文档。**

因为用户提问和文档表述之间经常存在差异。用户可能只从一个角度问问题，但知识库里的相关内容可能分散在不同表达方式、不同术语、不同维度的文档中。如果只用一个 Query 检索，很容易漏召回。

多 Query 扩展的思路是：

> **不指望一个 Query 一次命中，而是生成多个不同角度的 Query，扩大召回面。只要其中一个 Query 和文档表达对上，就有机会把正确内容召回来。**

它的优点是可以提升召回覆盖率，特别适合用户提问风格多变、问题涉及多个维度、知识库表述差异较大的场景。

缺点是会增加额外开销，因为需要 LLM 生成多个 Query，并且每个 Query 都要执行一次检索。

一句话总结：

> **多 Query 扩展的本质是用多个不同角度的检索 Query 扩大召回面，降低单一 Query 表达不准导致漏召回的风险。**

---

## 二、为什么需要多 Query 扩展？

前面几种 Query Rewrite 方法，大多还是围绕 **一个问题** 进行处理：

- **直接改写**：把原问题改得更清晰
- **HyDE**：给原问题生成一个假设答案
- **Step-back**：把原问题抽象成背景问题

但有些问题只靠一个 Query 不够。

原因是：

> **用户问题可能只表达了一个角度，而知识库里的答案可能分散在多个角度里。**

这时，与其纠结“怎么把一个 Query 改得最好”，不如直接生成多个 Query，让它们分别覆盖不同表达方式。

---

## 三、多 Query 扩展的核心流程

多 Query 扩展的流程可以理解为：

    用户原始问题
        ↓
    LLM 生成 3～5 个不同角度的 Query
        ↓
    每个 Query 分别执行检索
        ↓
    合并所有检索结果
        ↓
    去重、排序
        ↓
    得到最终候选文档

它的关键不是“生成更多问题”本身，而是：

> **让不同 Query 覆盖不同检索角度。**

---

## 四、多 Query 扩展解决什么问题？

它主要解决的是：

> **单一 Query 覆盖不全的问题。**

比如一个用户问题可能同时涉及：

- 概念解释
- 原理分析
- 操作步骤
- 参数影响
- 优化方法
- 常见问题

如果只用一个 Query，可能只能召回其中一部分内容。

多 Query 扩展可以让系统从多个角度去搜，从而提高正确文档被召回的概率。

---

## 五、多 Query 扩展的优缺点

| 方面 | 说明 |
|---|---|
| **优点** | 扩大召回面，减少单个 Query 表达不准导致的漏召回 |
| **适合场景** | 用户提问风格多变、文档表述差异大、复杂问题、多维度问题 |
| **缺点** | 增加 LLM 调用成本和检索次数 |
| **核心风险** | Query 扩展过多可能引入噪声，需要后续合并、去重和 Rerank |

---

## 六、和其他 Query Rewrite 方法的区别

| 方法 | 核心作用 |
|---|---|
| **直接改写** | 让问题更清晰、更正式 |
| **HyDE** | 用假设答案弥合“问题-文档”的文体差异 |
| **Step-back** | 把具体问题抽象成背景问题，检索原理知识 |
| **多 Query 扩展** | 从多个角度生成 Query，扩大召回覆盖面 |

可以这样记：

> **直接改写是把一个 Query 改准；HyDE 是把问题变成答案风格；Step-back 是把问题往上抽象；多 Query 是把问题横向展开。**

---

## 七、核心记忆点

### 1. 多 Query 扩展不是为了生成答案

它生成的是多个检索 Query，不是最终回答。

---

### 2. 它解决的是召回覆盖率问题

核心目标是：

> **让正确文档更容易被召回。**

---

### 3. 它适合复杂问题和表达差异大的场景

如果知识库内容本身比较简单，或者用户 Query 已经很明确，就不一定需要多 Query 扩展。

---

### 4. 多 Query 后通常还需要结果融合和 Rerank

因为多个 Query 会带来更多候选文档，也可能引入噪声。

所以后面通常需要：

    多 Query 检索
        ↓
    合并去重
        ↓
    RRF / 其他融合排序
        ↓
    Rerank 精排

---

## 八、一句话总结

多 Query 扩展是 Query Rewrite 中用于提升召回覆盖率的方法：**它让 LLM 从多个角度生成多个检索 Query，分别检索后合并结果，从而降低单一 Query 角度太窄导致漏召回的风险，但代价是增加 LLM 调用和检索成本。**
