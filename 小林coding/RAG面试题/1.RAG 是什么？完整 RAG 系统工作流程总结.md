## 一、面试标准回答

### 问题：什么是 RAG？完整的 RAG 系统工作流程是什么？

可以这样回答：

RAG 全称是 **Retrieval-Augmented Generation，检索增强生成**。

我理解的 RAG，本质上是为了解决大模型自身知识受限的问题。因为大模型训练完成之后，参数里的知识基本就固定了，它不知道训练截止时间之后的新信息，也不知道企业内部的私有文档。如果直接问它这些内容，很容易回答不出来或者产生幻觉。

RAG 的做法不是重新训练模型，也不是把知识写进模型参数，而是在用户提问时，先从外部知识库中检索出和问题相关的内容，然后把这些内容作为上下文，连同用户问题一起交给大模型，让大模型基于检索到的资料生成答案。

所以 RAG 可以理解为给大模型加了一个“开卷考试”的能力：  
模型本身不一定记得所有知识，但只要我们在回答前把相关资料找出来放进 Prompt，它就可以基于这些资料进行理解、总结和回答。

一个完整的 RAG 系统一般分为两个阶段：

**第一是离线阶段，负责构建知识库。**

流程是：

1. **文档加载**：把 PDF、Word、网页、Markdown、数据库记录等原始资料读取进来。
2. **文档切割 Chunking**：把长文档切成多个语义相对完整的小片段，避免整篇文档过长，也避免信息被压缩得太粗。
3. **Embedding 向量化**：用 Embedding 模型把每个 chunk 转成向量，让文本可以通过语义相似度进行检索。
4. **向量入库**：把 chunk 原文、向量和元数据一起存入向量数据库，比如 Milvus、Chroma、Qdrant 等。

**第二是在线阶段，负责根据用户问题实时检索并生成答案。**

流程是：

1. **Query 处理 / Query Rewrite**：对用户问题进行改写，补全上下文，让问题更适合检索。
2. **向量检索 / 粗排**：把用户问题也转成向量，到向量数据库中召回语义相近的 Top-K 个 chunk。
3. **Rerank 精排**：用精排模型重新判断问题和候选 chunk 的相关性，把真正有用的内容排到前面。
4. **Prompt 拼接**：把用户问题和筛选后的文档片段组织成 Prompt。
5. **LLM 生成答案**：让大模型基于检索到的上下文生成最终回答。

RAG 的核心价值主要有两个：

第一，**知识可以热更新**。只需要更新外部知识库，不需要重新训练大模型。  
第二，**答案可以溯源**。回答可以对应到具体文档 chunk，方便验证和排查问题。

所以面试中不能只说 RAG 是“检索 + 生成”，而要说清楚它为什么存在、离线阶段做什么、在线阶段做什么，以及它相比微调的优势。

## 二、这篇文章的核心内容

这篇文章主要回答一个问题：

> **什么是 RAG？一个完整的 RAG 系统是怎么工作的？**

文章不是重点讲复杂优化，而是先把 RAG 的基本概念和主流程讲清楚。

核心可以概括为一句话：

> **RAG = 先从外部知识库中检索相关资料，再让大模型基于这些资料生成答案。**

---

## 三、为什么需要 RAG？

### 1. LLM 有“知识冻结”问题

大模型训练完成之后，它的知识主要来自训练数据。

这会带来两个问题：

1. **不知道最新信息**
   - 比如训练截止日期之后发生的新闻、政策、股价、产品变化等。

2. **不知道私有数据**
   - 比如公司内部文档、业务手册、财报、合同、知识库等。

如果直接让大模型回答这些问题，它要么答不上来，要么可能编造答案。

---

### 2. 为什么不直接微调？

理论上，可以通过微调让模型学习新知识。

但是微调有几个问题：

1. **成本高**
   - 需要准备数据、训练模型、验证效果。

2. **更新慢**
   - 每次知识变化，都可能需要重新训练或继续训练。

3. **知识写进参数后不容易修改**
   - 如果某条知识过期了，不好单独删除或替换。

所以对于经常变化的知识、企业私有知识、需要溯源的知识问答场景，RAG 通常比微调更适合。

---

## 四、RAG 和微调的本质区别

| 对比项 | RAG | 微调 |
|---|---|---|
| 知识存放位置 | 外部知识库 | 模型参数 |
| 是否修改模型参数 | 不修改 | 修改 |
| 更新成本 | 低，更新知识库即可 | 高，通常需要重新训练 |
| 适合场景 | 私有知识问答、企业文档问答、实时知识问答 | 改变模型风格、任务格式、领域能力 |
| 可溯源性 | 强，可以追踪到文档 chunk | 弱，答案来自模型参数 |

一句话总结：

> **微调是让模型“记住知识”，RAG 是让模型“查资料后回答”。**

---

## 五、完整 RAG 系统的两个阶段

RAG 系统一般分为：

1. **离线阶段：提前构建知识库**
2. **在线阶段：用户提问时检索并生成**

---

# 第一阶段：离线阶段

## 1. 文档加载

离线阶段的第一步是把原始文档读取进来。

常见数据来源包括：

- PDF
- Word
- Markdown
- HTML 网页
- 数据库记录
- 企业内部文档
- 接口返回数据

这一步的目标是：

> **把各种格式的数据统一转换成系统可以处理的文本。**

比如在项目中，PDF 文档通常需要先经过解析、清洗，去掉页眉页脚、目录、免责声明等无效内容。

---

## 2. 文档切割 Chunking

### 为什么要切割文档？

不能直接把整篇文档作为一个整体存入向量库，主要有两个原因。

### 原因一：Embedding 模型有输入长度限制

Embedding 模型一次能处理的 token 数是有限的。

如果一篇 PDF 有几万字，不能直接整体送入 Embedding 模型。

---

### 原因二：整篇文章向量化会丢失细节

如果把一整篇长文档压缩成一个向量，那么这个向量只能表示整篇文章的“平均语义”。

这样会导致很多细节被淹没。

比如用户问：

> “腾讯 Q3 广告业务增长的原因是什么？”

如果整篇财报只变成一个向量，那么这个问题对应的细节信息可能被大量无关内容稀释，检索效果会很差。

所以需要把文档切成多个 chunk，每个 chunk 表示一个更聚焦的语义片段。

---

## 3. Chunk 大小怎么设置？

Chunk 太大和太小都有问题。

### Chunk 太大

如果一个 chunk 有 2000 token 甚至更多：

- 信息太杂
- 一个 chunk 里可能包含多个主题
- 检索回来后不一定精准
- 会占用更多 Prompt 上下文窗口

### Chunk 太小

如果一个 chunk 只有几十个 token：

- 语义可能不完整
- 上下文容易被切断
- 模型看到的内容不够完整

### 常见做法

实践中常见设置是：

```text
chunk_size = 500 ~ 1000 tokens
chunk_overlap = 100 tokens 左右
```

其中 overlap 的作用是：

> **让相邻 chunk 有一定重叠，避免一句话或者一段完整语义被硬切断。**

---

## 4. Embedding 向量化

### Embedding 是什么？

Embedding 就是把文本转换成一个高维向量。

可以理解为：

> **把一段文字映射到一个语义坐标系中。**

语义相近的文本，在向量空间中的距离更近；语义不相关的文本，距离更远。

比如：

```text
“苹果手机怎么截图”
“iPhone 如何截屏”
```

这两句话关键词不完全一样，但意思接近，所以它们的向量距离应该比较近。

而：

```text
“苹果手机怎么截图”
“今天股票市场怎么样”
```

语义差异很大，所以向量距离应该比较远。

---

## 5. 向量数据库入库

完成 Embedding 后，需要把下面这些内容一起存入向量数据库：

1. **chunk 原文**
2. **chunk 对应的向量**
3. **元数据**
   - 文档 ID
   - 页码
   - 标题
   - 来源文件
   - 时间
   - 公司名
   - 文档类型等

常见向量数据库包括：

- Chroma
- Milvus
- Qdrant
- Weaviate

入库完成后，离线阶段就结束了。

---

# 第二阶段：在线阶段

## 1. Query 处理 / Query Rewrite

用户的问题往往不是天然适合检索的。

比如用户问：

```text
上次说的那个方案怎么样？
```

这个问题对人来说可能能理解，因为人知道“上次”和“那个方案”指什么。

但对检索系统来说，这句话没有明确关键词，也没有完整语义，很难检索。

所以实际工程中通常会做 Query Rewrite。

### Query Rewrite 的作用

Query Rewrite 的目标是：

> **把用户的口语化、上下文依赖强的问题，改写成更完整、更适合检索的问题。**

例如：

```text
用户原问题：
上次说的那个方案怎么样？

改写后：
请分析腾讯 2025 年 Q3 财报中广告业务增长方案的效果和风险。
```

这样检索系统才能更准确地找到相关 chunk。

---

## 2. 向量检索：粗排

Query 处理后，会把用户问题也转成向量。

然后到向量数据库中检索距离最近的 Top-K 个 chunk。

这个过程叫：

> **向量检索，也可以理解为粗排。**

粗排的特点是：

- 速度快
- 可以从大量 chunk 中快速召回一批候选内容
- 但不一定非常准确

原因是向量检索主要看向量距离，它不是真正逐字逐句理解问题和文档之间的关系。

所以召回结果中可能混入一些“看起来相似，但其实不相关”的 chunk。

---

## 3. Rerank：精排

为了解决粗排不够精准的问题，通常会增加 Rerank。

### Rerank 做什么？

Rerank 会对粗排召回的候选 chunk 重新排序。

它会更细致地判断：

> **用户问题和每个 chunk 到底有多相关。**

一般流程是：

```text
向量检索召回 Top-20
        ↓
Rerank 重新打分排序
        ↓
保留 Top-3 / Top-5
```

### 粗排和精排的区别

| 阶段 | 作用 | 特点 |
|---|---|---|
| 粗排 | 快速召回候选 chunk | 快，但可能不够准 |
| 精排 | 重新判断相关性并排序 | 更准，但更慢 |

可以这样理解：

> **粗排负责“先找一批可能相关的”，精排负责“从里面挑出真正相关的”。**

---

## 4. Prompt 拼接

拿到最终筛选后的 chunk 后，需要把它们和用户问题拼成 Prompt。

一个常见 Prompt 结构是：

```text
你是一个专业问答助手。
请只根据下面提供的资料回答用户问题。
如果资料中没有答案，请回答“资料中没有提到”。

【参考资料】
chunk 1
chunk 2
chunk 3

【用户问题】
xxx

【回答】
```

这里的关键点是：

> **要明确要求模型基于资料回答，不要自由发挥。**

这样可以降低大模型幻觉。

---

## 5. LLM 生成答案

最后，大模型根据 Prompt 中的用户问题和检索资料生成答案。

此时模型不是单纯依赖自己参数中的旧知识，而是基于外部知识库中检索出来的内容作答。

这也是 RAG 的核心：

> **检索负责找资料，生成负责组织语言和推理总结。**

# 七、这篇文章最重要的几个理解点

## 1. RAG 不是搜索引擎

很多人会把 RAG 理解成搜索。

但 RAG 不是单纯搜索。

搜索引擎只负责找到文档，而 RAG 还要让大模型基于文档生成答案。

所以 RAG 的完整结构是：

```text
检索 Retrieval + 增强 Augmented + 生成 Generation
```

---

## 2. RAG 不是微调

RAG 不会修改模型参数。

它只是把外部知识作为上下文放进 Prompt，让模型临时参考。

所以：

```text
微调：把知识写进模型参数
RAG：把知识放在外部知识库，回答时临时检索
```

---

## 3. Chunking 很关键

RAG 效果不好，很多时候不是大模型的问题，而是 chunk 切得不好。

常见问题包括：

1. chunk 太大，召回内容不精准
2. chunk 太小，语义不完整
3. 没有 overlap，导致上下文断裂
4. 切割方式不考虑标题、段落、表格结构

所以 Chunking 是 RAG 工程落地中非常关键的一环。

---

## 4. Embedding 决定语义检索能力

Embedding 的作用是把文本变成向量。

向量检索不是关键词匹配，而是语义匹配。

所以它可以处理：

```text
“iPhone 如何截屏”
```

和：

```text
“苹果手机怎么截图”
```

这种表达不同但意思相近的问题。

---

## 5. Rerank 是提升检索质量的重要手段

向量检索速度快，但可能不够准。

Rerank 速度慢一些，但判断更细。

所以实际系统中常见组合是：

```text
向量检索召回一批候选内容
        ↓
Rerank 重新排序
        ↓
只把最相关的内容交给 LLM
```

这样既兼顾速度，也提升准确率。

---

## 6. Prompt 要限制模型基于资料回答

如果只是把 chunk 扔给模型，不加约束，模型仍然可能自由发挥。

所以 Prompt 中通常要加类似约束：

```text
请只根据提供的资料回答。
如果资料中没有相关信息，请直接说不知道。
不要编造资料中没有出现的内容。
```

这样可以减少幻觉。

---

# 八、结合项目理解 RAG

如果放到自己的金融分析 Agent 项目里，可以这样理解：

```text
金融 PDF / 财报 / 研报
        ↓
PDF 解析和清洗
        ↓
按页或按段落切成 chunk
        ↓
Embedding 向量化
        ↓
写入 Milvus 向量数据库
        ↓
用户提问
        ↓
检索相关财报片段
        ↓
拼接 Prompt
        ↓
LLM 生成金融分析回答
```

在这个过程中：

- **RAG 存的是原始证据**
- **Research Memory 存的是长期研究结论**
- **Session Memory 存的是当前对话上下文**

三者作用不同：

| 模块 | 作用 |
|---|---|
| RAG | 从文档中找原始证据 |
| Session Memory | 记住当前对话上下文 |
| Research Memory | 复用历史研究结论和长期洞察 |

所以 RAG 不是简单“把 PDF 发给大模型”，而是：

> **先把 PDF 处理成可检索的知识库，再在用户提问时动态找证据，让模型基于证据回答。**

# 九、记忆版总结

RAG 可以记成一句话：

> **RAG 不是让模型记住知识，而是让模型回答前先查资料。**

完整流程记成两阶段：

```text
离线阶段：
文档加载 → 文档切割 → Embedding 向量化 → 向量数据库入库

在线阶段：
用户提问 → Query 改写 → 向量检索粗排 → Rerank 精排 → Prompt 拼接 → LLM 生成
```

RAG 的核心价值：

```text
1. 解决大模型知识冻结问题
2. 支持企业私有知识问答
3. 知识可以热更新
4. 答案可以溯源
5. 降低幻觉
```

面试时重点说清楚：

```text
RAG 解决什么问题？
RAG 和微调有什么区别？
RAG 离线阶段做什么？
RAG 在线阶段做什么？
Chunking、Embedding、Rerank 分别有什么作用？
```

只要这几个点说清楚，基本就能把“什么是 RAG”和“完整 RAG 流程”这个问题回答完整。
