## 一、面试标准回答

### 问题：RAG 里用的 Embedding 算法有哪些？你了解过几代演进？

可以这样回答：

Embedding 算法大致可以分成三代。

**第一代是静态词向量**，代表模型是 **Word2Vec、GloVe、FastText**。

这类方法的核心思想是把每个词映射成一个固定向量。比如 Word2Vec 会根据词的上下文共现关系来学习词向量，语义相近的词在向量空间中距离更近。它解决了早期 NLP 中“词怎么表示成数字”的问题。

但是第一代方法有一个很大的问题：**每个词只有一个固定向量，处理不了多义词**。

比如：

```text
我吃了一个苹果。
我买了一部苹果手机。
```

这两个句子里的“苹果”含义完全不同，但 Word2Vec 里“苹果”只有一个固定向量，所以无法根据上下文区分语义。因此第一代 Embedding 不适合现代 RAG 语义检索。

---

**第二代是上下文相关向量**，代表模型是 **ELMo、BERT**。

这类方法解决了第一代的多义词问题。它会根据上下文动态生成 token 向量。

比如同样是“苹果”：

```text
我吃了一个苹果。
我买了一部苹果手机。
```

在 BERT 里，这两个“苹果”的向量是不一样的，因为模型会结合前后文理解它到底是水果还是公司品牌。

但是 BERT 这类模型虽然语义理解强，却不适合直接做大规模 RAG 检索。原因是 BERT 原生更像一个 **Cross-Encoder**：如果要判断 query 和 doc 是否相关，通常需要把两段文本拼在一起输入模型，让模型整体判断相关性。

这会导致一个严重问题：

```text
用户 query + 第 1 个 chunk → BERT 判断相关性
用户 query + 第 2 个 chunk → BERT 判断相关性
用户 query + 第 3 个 chunk → BERT 判断相关性
...
用户 query + 第 100 万个 chunk → BERT 判断相关性
```

如果知识库有 100 万个 chunk，就要跑 100 万次模型前向传播，在线检索速度完全不可接受。

所以第二代模型解决了语义理解问题，但没有解决大规模检索效率问题。

---

**第三代是句子级对比学习 Embedding**，代表模型是 **SBERT、SimCSE、BGE、E5、Qwen Embedding**。

第三代模型专门为语义相似度和检索场景设计。它的核心是 **Bi-Encoder** 结构：

```text
query 单独编码成向量
document chunk 单独编码成向量
然后用余弦相似度计算相似度
```

这样文档 chunk 的向量可以提前离线算好，存进向量数据库。用户提问时，只需要实时计算一次 query 向量，再和向量库中的文档向量做相似度检索即可。

流程是：

```text
离线阶段：
所有文档 chunk → Embedding → 存入向量库

在线阶段：
用户 query → Embedding → 向量检索 → 返回相似 chunk
```

这就非常适合 RAG。

第三代模型一般通过对比学习训练，让语义相近的句子向量更接近，让语义不相关的句子向量更远。

比如 SimCSE 会把同一句话通过 dropout 得到两个不同表示，把它们当作正样本拉近；把同一个 batch 里的其他句子当作负样本推远。

BGE 这类模型则是在大规模中英文检索数据上训练出来的，更适合实际 RAG 场景，尤其是中文企业知识库和中文文档问答。

所以总结来说：

> **第一代解决“词怎么变向量”，但处理不了多义词；第二代解决“上下文语义理解”，但在线检索太慢；第三代解决“句子级语义检索”，可以离线建库、在线快速召回，是 RAG 的主流选择。**

如果在项目里选型，我会优先选择第三代 Embedding 模型。中文金融文档场景可以优先考虑 **BGE、Qwen Embedding**；英文或多语言场景可以考虑 **E5、bge-m3、OpenAI text-embedding**。最终不是只看模型名气，而是要在自己的业务数据上用 **Hit@K** 评估召回效果。

---

## 二、这篇文章的核心内容

这篇文章主要讲：

> **Embedding 算法从 Word2Vec 到 BERT，再到 SBERT / SimCSE / BGE，是如何一步步演进到适合 RAG 检索场景的。**

核心主线是：

```text
第一代：静态词向量
解决“词怎么表示成向量”
但处理不了多义词

第二代：上下文相关向量
解决“同一个词在不同语境中含义不同”
但大规模检索太慢

第三代：句子级对比学习 Embedding
解决“如何高效判断两段文本语义相似”
成为 RAG 的标配
```

这篇文章最重要的一句话是：

> **RAG 场景下真正常用的是第三代句子级 Embedding，因为它可以提前把文档向量算好，查询时只算一次 query 向量。** :contentReference[oaicite:0]{index=0}

# 三、为什么 Embedding 算法要一代代演进？

Embedding 的核心目标是：

> **让计算机能够用向量表示自然语言语义。**

比如用户问：

```text
苹果手机怎么截图？
```

文档里写的是：

```text
iPhone 如何截屏？
```

这两句话字面上没有完全相同的关键词，但语义是接近的。

如果只靠关键词匹配，可能匹配不到。

Embedding 要解决的就是：

```text
不同表达
相同语义
向量距离应该接近
```

但是不同阶段的 Embedding 方案能力不同。

每一代算法都是在补上一代的短板：

```text
Word2Vec / GloVe：
能把词变成向量，但无法理解上下文。

BERT / ELMo：
能根据上下文动态生成表示，但不适合大规模实时检索。

SBERT / SimCSE / BGE：
能把整句话独立编码成向量，适合语义检索和 RAG。
```

---

# 四、第一代：静态词向量

## 1. 代表模型

第一代 Embedding 的代表是：

```text
Word2Vec
GloVe
FastText
```

它们的共同特点是：

> **每个词对应一个固定向量。**

比如：

```text
苹果 → [0.12, -0.08, 0.35, ...]
手机 → [0.21, 0.13, -0.09, ...]
水果 → [-0.04, 0.42, 0.18, ...]
```

训练完成后，“苹果”这个词的向量就是固定的。

---

## 2. Word2Vec 的核心思想

Word2Vec 的核心思想是：

> **一个词的含义，可以通过它周围经常出现的词来表示。**

也就是所谓的：

```text
You shall know a word by the company it keeps.
```

如果两个词经常出现在相似上下文中，它们的向量就会接近。

比如：

```text
国王、王后、男人、女人
```

这些词之间会形成某种语义关系。

经典例子是：

```text
国王 - 男人 + 女人 ≈ 王后
```

这说明词向量在一定程度上学到了语义结构。

---

## 3. Word2Vec 的两种训练方式

Word2Vec 主要有两种训练方式：

```text
CBOW
Skip-gram
```

---

### 3.1 CBOW：用上下文预测中心词

CBOW 的思路是：

```text
周围词 → 预测中心词
```

比如句子：

```text
我 喜欢 吃 苹果
```

假设中心词是：

```text
吃
```

上下文是：

```text
我、喜欢、苹果
```

CBOW 就是根据周围词预测中心词。

流程可以理解为：

```text
我 + 喜欢 + 苹果
        ↓
预测
        ↓
吃
```

CBOW 的特点是：

```text
训练速度快
在大规模语料上收敛快
对高频词效果不错
```

---

### 3.2 Skip-gram：用中心词预测上下文

Skip-gram 的思路相反：

```text
中心词 → 预测周围词
```

比如中心词是：

```text
吃
```

它要预测周围词：

```text
我、喜欢、苹果
```

流程是：

```text
吃
 ↓
预测：我、喜欢、苹果
```

Skip-gram 的特点是：

```text
样本利用率更高
对低频词更友好
更常作为默认选择
```

因为一个中心词可以产生多个训练样本：

```text
吃 → 我
吃 → 喜欢
吃 → 苹果
```

---

## 4. GloVe 的核心思想

GloVe 的全称是：

```text
Global Vectors for Word Representation
```

它的核心思路是利用全局词共现矩阵。

Word2Vec 更像是通过局部上下文窗口学习词向量，而 GloVe 更强调：

```text
整个语料库中词与词之间的共现统计关系
```

比如它会统计：

```text
苹果 和 手机 共现多少次
苹果 和 水果 共现多少次
苹果 和 公司 共现多少次
```

然后通过矩阵分解学习词向量。

可以理解为：

```text
Word2Vec：更偏局部上下文预测
GloVe：更偏全局共现统计
```

---

## 5. FastText 的改进

Word2Vec 和 GloVe 都有一个问题：

> **遇到训练时没见过的新词，就无法得到合理向量。**

这叫 **未登录词 OOV** 问题。

FastText 的改进是：

> **把词拆成字符级 n-gram 子词，再用子词向量组合成词向量。**

比如：

```text
苹果手机
```

可以拆成：

```text
苹
果
手
机
苹果
手机
苹果手
果手机
```

每个子词片段都有自己的向量。

如果遇到一个训练集中没出现过的新词，只要它的子词片段出现过，就可以估算出这个新词的向量。

所以 FastText 对新词、专有名词、形态变化更友好。

---

## 6. 第一代的核心缺陷：静态

第一代最大问题是：

> **一个词永远只有一个向量，不管上下文是什么。**

比如：

```text
我吃了一个苹果。
苹果发布了新款手机。
```

在 Word2Vec 里，这两个“苹果”的向量完全一样。

但是第一个“苹果”是水果，第二个“苹果”是公司品牌。

第一代模型无法根据上下文动态区分。

所以它不适合现代 RAG 中的复杂语义检索。

---

# 五、第二代：上下文相关向量

## 1. 代表模型

第二代 Embedding 的代表是：

```text
ELMo
BERT
```

这一代的核心改进是：

> **同一个词在不同上下文中，会得到不同的向量。**

---

## 2. ELMo：基于双向 LSTM

ELMo 使用双向 LSTM 来建模上下文。

它会从两个方向读句子：

```text
从左到右
从右到左
```

然后把两个方向的隐藏状态拼接起来，作为某个词的上下文向量。

比如：

```text
我吃了一个苹果。
```

这里的“苹果”会受到“吃了”这个上下文影响，更偏水果语义。

而：

```text
苹果发布了新款手机。
```

这里的“苹果”会受到“发布”“手机”影响，更偏公司品牌语义。

---

## 3. BERT：基于 Transformer 的上下文表示

BERT 使用 Transformer 来建模上下文，并通过 Masked Language Model 进行预训练。

它的特点是：

```text
双向建模
上下文理解强
能生成 token 级别的动态向量
```

BERT 中通常会使用 `[CLS]` token 表示整句话。

输入：

```text
[CLS] 苹果 发布 了 新款 手机 [SEP]
```

BERT 会输出每个 token 的向量，以及 `[CLS]` 位置的句子表示。

---

## 4. 第二代解决了什么问题？

第二代解决的是第一代无法处理多义词的问题。

比如：

```text
句子 1：我吃了一个苹果。
句子 2：苹果发布了新款手机。
```

在 BERT 里，两个“苹果”的向量不同。

因为 BERT 会结合上下文判断：

```text
吃了一个苹果 → 水果
苹果发布手机 → 公司品牌
```

所以第二代比 Word2Vec 更有语义理解能力。

---

## 5. 为什么 BERT 不适合直接做 RAG 检索？

这是这篇文章非常重要的点。

很多人会误以为：

```text
BERT 语义理解强，所以直接用 BERT 做 RAG 检索就行。
```

但问题在于，BERT 原生更适合做 **Cross-Encoder**。

也就是：

```text
把 query 和 doc 拼在一起输入模型
让模型判断它们是否相关
```

例如：

```text
[CLS] 苹果手机怎么截图 [SEP] iPhone 截屏方法 [SEP]
```

BERT 可以充分建模 query 和 doc 之间的 token 交互，所以判断很准确。

但是它不能提前给每个 doc 单独算好一个固定向量，然后直接相似度检索。

---

## 6. Cross-Encoder 为什么慢？

假设知识库里有 100 万个 chunk。

用户问：

```text
苹果手机怎么截图？
```

如果用 BERT Cross-Encoder，需要这样做：

```text
query + chunk_1 → BERT → relevance score
query + chunk_2 → BERT → relevance score
query + chunk_3 → BERT → relevance score
...
query + chunk_1000000 → BERT → relevance score
```

也就是每次查询都要跑 100 万次 BERT。

这在在线系统里完全不可接受。

所以第二代模型的问题是：

```text
语义理解强
但不能高效大规模检索
```

这就引出了第三代句子级 Embedding。

---

# 六、第三代：句子级对比学习 Embedding

## 1. 代表模型

第三代代表是：

```text
SBERT
SimCSE
BGE
E5
Qwen Embedding
```

这一代是现代 RAG 中真正常用的 Embedding。

核心特点是：

```text
句子级向量
Bi-Encoder 结构
对比学习训练
支持离线建库
支持在线快速检索
```

---

## 2. 核心问题：能不能让句子独立生成向量？

BERT Cross-Encoder 准确，但太慢。

第三代要解决的问题是：

> **能不能让每个句子或 chunk 独立生成一个向量，然后直接用向量相似度比较？**

也就是：

```text
query → query_vector
chunk → chunk_vector

similarity(query_vector, chunk_vector)
```

这样就可以提前把所有文档向量算好。

---

## 3. Bi-Encoder 是什么？

Bi-Encoder 指的是：

```text
query 和 document 分别独立编码
```

流程是：

```text
query
  ↓
Encoder
  ↓
query 向量

document chunk
  ↓
Encoder
  ↓
doc 向量

query 向量 + doc 向量
  ↓
余弦相似度 / 点积
  ↓
相关性分数
```

图示：

```text
              ┌──────────────┐
query ───────▶│   Encoder     │──────▶ query_vector
              └──────────────┘

              ┌──────────────┐
chunk ───────▶│   Encoder     │──────▶ chunk_vector
              └──────────────┘

query_vector · chunk_vector
        ↓
相似度分数
```

其中 query 和 chunk 可以使用同一个编码器，也可以使用参数共享的双塔结构。

---

## 4. Bi-Encoder 为什么适合 RAG？

因为文档 chunk 的向量可以提前算好。

RAG 的典型流程是：

```text
离线阶段：
chunk_1 → embedding_1
chunk_2 → embedding_2
chunk_3 → embedding_3
...
chunk_N → embedding_N

把 embedding 全部存进向量数据库
```

在线阶段：

```text
用户 query → query_embedding
在向量数据库中做相似度搜索
返回 Top-K chunk
```

这样每次查询只需要：

```text
计算 1 次 query 向量
执行 1 次向量检索
```

而不是：

```text
query 和所有 chunk 两两拼接跑模型
```

所以速度可以从不可接受变成毫秒级或秒级。

---

## 5. Bi-Encoder 的代价

Bi-Encoder 也不是完美的。

它的缺点是：

> **query 和 doc 在编码阶段没有深度交互。**

Cross-Encoder 是：

```text
query 和 doc 拼在一起
每个 token 都可以互相 attention
```

所以判断更细。

Bi-Encoder 是：

```text
query 单独编码
doc 单独编码
最后只比较两个向量
```

因此在一些非常细微的相关性判断上，Bi-Encoder 的精度可能不如 Cross-Encoder。

但它换来了极大的速度优势。

所以 RAG 常用组合是：

```text
Bi-Encoder 负责召回
Cross-Encoder / Reranker 负责精排
```

也就是：

```text
先快召回一批候选 chunk
再用更慢但更准的模型重新排序
```

---

# 七、SBERT：把 BERT 改造成句子向量模型

## 1. SBERT 解决什么问题？

SBERT 全称是：

```text
Sentence-BERT
```

它的目标是：

> **让 BERT 可以高效生成句子级向量，用于语义相似度计算。**

SBERT 采用 Bi-Encoder 结构。

两个句子分别过 BERT，得到各自的句子向量，然后用余弦相似度比较。

---

## 2. SBERT 的基本流程

比如：

```text
query：
苹果手机怎么截图？

doc：
iPhone 截屏方法。
```

SBERT 会这样处理：

```text
苹果手机怎么截图？
        ↓
BERT Encoder
        ↓
query_vector

iPhone 截屏方法。
        ↓
BERT Encoder
        ↓
doc_vector

cosine_similarity(query_vector, doc_vector)
        ↓
相似度分数
```

这样就不需要把 query 和 doc 拼在一起跑模型。

---

## 3. SBERT 的优势

SBERT 的优势是：

```text
可以提前编码文档
在线查询速度快
适合语义检索
相比普通 BERT 更适合 RAG
```

---

## 4. SBERT 的局限

SBERT 的局限是：

```text
精细交互能力弱于 Cross-Encoder
对非常复杂的语义匹配可能不够准
```

所以在高质量 RAG 中，经常会用：

```text
SBERT / BGE 召回
Reranker 精排
```

---

# 八、SimCSE：用对比学习提升句子向量质量

## 1. SimCSE 是什么？

SimCSE 是一种句子 Embedding 方法。

它的核心思想是：

> **用对比学习让语义相近的句子向量靠近，让语义不相近的句子向量远离。**

---

## 2. 无监督 SimCSE 怎么做？

无监督 SimCSE 非常巧妙。

它把同一句话输入模型两次，因为 dropout 的随机性，两次会得到略有不同的向量。

比如同一句话：

```text
苹果手机怎么截图？
```

第一次经过模型：

```text
vector_a
```

第二次经过模型：

```text
vector_b
```

这两个向量来自同一句话，所以它们应该是正样本对。

训练目标是：

```text
把 vector_a 和 vector_b 拉近
把 batch 中其他句子的向量推远
```

---

## 3. SimCSE 的训练示意

假设一个 batch 中有 4 个句子：

```text
s1：苹果手机怎么截图？
s2：腾讯 Q3 广告业务增长原因是什么？
s3：如何申请企业客服服务？
s4：GPU 显存为什么会爆？
```

对 `s1` 做两次 dropout，得到：

```text
s1_a
s1_b
```

其中：

```text
s1_a 和 s1_b 是正样本
s2、s3、s4 是负样本
```

训练目标：

```text
sim(s1_a, s1_b) 变大
sim(s1_a, s2) 变小
sim(s1_a, s3) 变小
sim(s1_a, s4) 变小
```

---

## 4. SimCSE 解决了什么问题？

SimCSE 解决了 BERT 原生句子向量质量不好的问题。

BERT 原生句子向量有一个问题：

> **向量分布各向异性。**

可以理解为：

```text
很多句子向量挤在一个狭窄区域里
向量空间没有被充分利用
不同语义之间区分度不够
```

SimCSE 通过对比学习把语义空间“撑开”。

这样：

```text
相似句子更近
不相似句子更远
向量空间分布更均匀
```

检索效果就会更好。

---

# 九、BGE：面向检索优化的通用 Embedding

## 1. BGE 是什么？

BGE 全称是：

```text
BAAI General Embedding
```

是北京智源研究院推出的通用 Embedding 模型系列。

它是中文 RAG 场景中非常常见的开源 Embedding 选择。

---

## 2. BGE 的特点

BGE 的特点是：

```text
基于大规模中英文数据训练
面向检索任务优化
中文效果好
支持本地部署
支持 bi-encoder 和 reranker 形态
```

所以在中文企业知识库、中文文档问答、中文金融研报分析中经常会使用 BGE 系列。

---

## 3. BGE 为什么适合中文 RAG？

因为中文 RAG 场景有一些特点：

```text
中文表达变化多
专有名词多
企业文档风格复杂
很多数据不适合发给国外 API
```

BGE 可以本地部署，中文语义召回效果也比较好，因此很适合私有知识库场景。

---

# 十、E5、Qwen Embedding 和新趋势

## 1. E5

E5 是微软推出的一类 Embedding 模型，在英文和多语言检索场景中很常见。

它和 BGE 类似，也属于面向检索优化的第三代句子级 Embedding。

适合：

```text
英文文档
多语言检索
通用语义搜索
```

---

## 2. Qwen Embedding

Qwen Embedding 是近几年中文和多语言场景中很值得关注的模型。

它的优势是：

```text
中文能力强
多语言能力好
适合中文企业知识库
支持本地部署或 API 调用
```

在中文金融分析项目中，可以把 Qwen Embedding 和 BGE 一起作为候选模型，用业务数据做 Hit@K 评估。

---

## 3. 指令感知 Embedding

新一代 Embedding 里，一个重要趋势是 **指令感知 Embedding**。

它的意思是：

> **同一段文本，在不同检索任务下，可以根据指令生成不同的向量表示。**

比如同一段文档：

```text
腾讯广告收入增长主要来自视频号商业化和 AI 广告系统优化。
```

如果检索指令是：

```text
查找能回答用户问题的文档片段。
```

模型会更关注“答案相关性”。

如果检索指令是：

```text
查找语义相似的问题。
```

模型会更关注“问题之间是否相似”。

也就是说：

```text
不同检索意图
    ↓
不同向量表示
```

这比传统固定 Embedding 更灵活。

---

## 4. Matryoshka 表示学习

Matryoshka 表示学习可以理解为：

> **训练模型时让向量的前 N 维就包含有用语义。**

也就是说，一个 1024 维向量，不一定必须完整使用 1024 维。

可以截断成：

```text
1024 维
768 维
512 维
256 维
```

并且仍然保持一定检索效果。

它的价值是：

```text
灵活平衡精度和成本
降低向量存储空间
提升检索速度
```

比如百万级 chunk 场景下，如果 1024 维成本太高，可以尝试截断到 512 维，然后评估 Hit@K 是否还能接受。

---

## 5. 多模态 Embedding

多模态 Embedding 的目标是：

> **让文本、图片等不同模态进入同一个向量空间。**

例如：

```text
文本 query：
这张图展示了 RAG 的检索流程

图片：
一张 RAG 流程图
```

如果使用多模态 Embedding，它们可以映射到同一个语义空间。

这样 RAG 不仅能检索文本，还能检索：

```text
图片
图表
截图
流程图
PDF 中的图像内容
```

这对含有大量图表、截图、结构图的知识库很有价值。

---

# 十一、三代 Embedding 对比表

| 代际 | 代表模型 | 核心特点 | 主要问题 | RAG 适用性 |
|---|---|---|---|---|
| 第一代 | Word2Vec / GloVe / FastText | 静态词向量，每个词一个固定向量 | 无法处理多义词，词级而非句子级 | 基本不适合现代 RAG |
| 第二代 | ELMo / BERT | 上下文相关向量，同一个词在不同语境向量不同 | 检索时需要 query-doc 两两拼接，速度太慢 | 不适合实时大规模检索 |
| 第三代 | SBERT / SimCSE / BGE / E5 / Qwen Embedding | 句子级向量，Bi-Encoder，对比学习 | 精度弱于 Cross-Encoder，需要 Rerank 补强 | RAG 标配 |

# 十二、Bi-Encoder 和 Cross-Encoder 对比

## 1. Cross-Encoder

Cross-Encoder 的输入是：

```text
query + document
```

模型会把两段文本拼起来一起编码。

优点是：

```text
query 和 document 可以深度交互
相关性判断更准确
适合精排 Rerank
```

缺点是：

```text
不能提前编码所有文档
每次 query 都要和所有候选文档重新跑模型
大规模检索太慢
```

---

## 2. Bi-Encoder

Bi-Encoder 的输入是：

```text
query 单独编码
document 单独编码
```

优点是：

```text
文档向量可以提前计算
检索速度快
适合大规模召回
```

缺点是：

```text
query 和 document 编码阶段没有 token 级交互
精细判断能力弱于 Cross-Encoder
```

---

## 3. RAG 中常见组合

RAG 中通常这样用：

```text
Bi-Encoder：
负责从海量知识库中快速召回 Top-K。

Cross-Encoder / Reranker：
负责对 Top-K 候选结果重新精排。
```

流程是：

```text
用户 query
   ↓
Bi-Encoder 生成 query 向量
   ↓
向量数据库召回 Top-50
   ↓
Cross-Encoder / Reranker 精排
   ↓
保留 Top-5
   ↓
拼接 Prompt 给 LLM
```

这就是速度和精度的折中。

---

# 十三、结合自己的金融分析 Agent 项目理解

在金融分析 Agent 项目中，Embedding 模型主要负责：

```text
把用户问题和财报 / 研报 chunk 映射到同一个语义空间
从 Milvus 中召回相关证据 chunk
```

例如用户问：

```text
腾讯 Q3 广告业务增长原因是什么？
```

知识库里可能有 chunk 写的是：

```text
广告收入同比提升，主要受益于视频号商业化加速、AI 广告投放系统优化以及小游戏广告需求提升。
```

这两句话不一定完全关键词一致，但语义相关。

Embedding 模型要做的就是让它们的向量距离足够近，从而被召回。

---

## 1. 为什么不能用 Word2Vec？

Word2Vec 是词级静态向量，处理不了上下文，也不适合直接表示长 chunk。

金融文档里有大量多义词和上下文依赖表达，例如：

```text
收入
增长
广告
现金流
业务
该项
上述变化
```

这些词单独看没意义，必须结合上下文。

所以 Word2Vec 不适合这种 RAG 场景。

---

## 2. 为什么不直接用 BERT 做检索？

BERT 语义理解强，但如果直接用 Cross-Encoder 做检索，需要把用户问题和每个 chunk 拼接后分别跑模型。

如果 Milvus 里有几十万甚至上百万个 chunk，这种方式在线不可接受。

所以更合理的是：

```text
用 BGE / Qwen Embedding 这类 Bi-Encoder 先召回
必要时再用 Reranker 对候选 chunk 精排
```

---

## 3. 为什么适合 BGE / Qwen Embedding？

金融分析 Agent 的文档主要是：

```text
中文财报
中文研报
中文公告
```

所以应该优先选择中文检索效果好的第三代 Embedding 模型，比如：

```text
BGE
Qwen Embedding
bge-m3
```

这些模型可以本地部署，也比较适合私有金融文档场景。

---

## 4. 面试中可以这样结合项目说

可以这样回答：

在我的金融分析 Agent 项目中，我不会使用 Word2Vec 这类静态词向量，因为它是词级表示，处理不了中文金融文本中的上下文语义，也不适合长文档 chunk 检索。

BERT 虽然语义理解强，但原生 Cross-Encoder 检索需要把 query 和每个 chunk 两两拼接，在大规模知识库里延迟太高。

所以项目中更适合使用第三代句子级 Embedding，比如 BGE 或 Qwen Embedding。它们采用 Bi-Encoder 思路，可以提前把财报和研报 chunk 向量化存入 Milvus。用户提问时只需要算一次 query 向量，再做向量检索。

如果召回结果还不够精准，可以在 Top-K 候选上再加 Reranker，用更慢但更准的 Cross-Encoder 做精排。

总结就是：

```text
Bi-Encoder 负责快速召回
Reranker 负责精细排序
LLM 负责基于证据生成答案
```

# 十四、面试官可能追问

## 追问 1：Word2Vec 为什么不适合 RAG？

可以这样回答：

Word2Vec 是静态词向量，每个词只有一个固定向量，无法根据上下文变化。

比如“苹果手机”和“吃苹果”里的“苹果”含义不同，但 Word2Vec 里的向量相同。

而 RAG 检索的是句子或段落级 chunk，需要理解上下文和整段语义，所以 Word2Vec 不适合现代 RAG。

---

## 追问 2：BERT 语义理解强，为什么不用 BERT 直接做检索？

可以这样回答：

BERT 原生更适合 Cross-Encoder 方式，也就是把 query 和 document 拼在一起判断相关性。

这种方式精度高，但无法提前把所有文档向量算好。

如果知识库有 100 万个 chunk，每次查询都要把 query 和 100 万个 chunk 分别拼接后跑 100 万次 BERT，在线延迟不可接受。

所以 RAG 通常用 Bi-Encoder 模型做召回，再用 Cross-Encoder 做 Rerank。

---

## 追问 3：SBERT 和 BERT 的区别是什么？

可以这样回答：

BERT 原生主要输出 token 级上下文表示，如果要判断两句话是否相关，通常需要把两句话拼在一起输入模型。

SBERT 把 BERT 改造成了句子级向量模型，使用 Bi-Encoder 结构，让两个句子可以分别独立编码成向量，再用余弦相似度比较。

所以 SBERT 更适合大规模语义检索，而普通 BERT 更适合精细的文本对判断。

---

## 追问 4：SimCSE 的核心思想是什么？

可以这样回答：

SimCSE 的核心是对比学习。

无监督 SimCSE 会把同一句话输入模型两次，因为 dropout 不同得到两个略有差异的向量，把它们作为正样本拉近；同时把同一个 batch 里的其他句子作为负样本推远。

这样可以让语义相近的句子向量更近，不相关的句子向量更远，同时改善 BERT 原生句子向量各向异性的问题。

---

## 追问 5：Bi-Encoder 和 Cross-Encoder 怎么取舍？

可以这样回答：

Bi-Encoder 的优势是快，可以提前编码文档，适合从大规模知识库中召回候选 chunk；缺点是 query 和 doc 编码时没有深度交互，精度可能不如 Cross-Encoder。

Cross-Encoder 的优势是准，因为 query 和 doc 可以在模型内部充分交互；缺点是慢，不能对全库逐条计算。

所以实际 RAG 通常是：

```text
Bi-Encoder 召回 Top-K
Cross-Encoder / Reranker 精排 Top-K
```

这样兼顾速度和精度。

---

## 追问 6：第三代 Embedding 为什么是 RAG 标配？

可以这样回答：

因为第三代 Embedding 解决了 RAG 最核心的工程问题：

```text
既要语义相似度效果好
又要能在大规模知识库上快速检索
```

它使用 Bi-Encoder 独立编码 query 和 doc，文档向量可以离线算好并存进向量数据库。在线查询时只需要算一次 query 向量，再做向量相似度搜索。

所以它在性能和效果之间取得了比较好的平衡。

---

# 十五、记忆版总结

这篇文章可以记成一句话：

> **Embedding 算法的演进，就是从“词向量”走向“上下文表示”，再走向“适合大规模检索的句子向量”。**

三代演进：

```text
第一代：Word2Vec / GloVe / FastText
静态词向量。
解决词怎么表示成向量。
缺点是处理不了多义词。

第二代：ELMo / BERT
上下文相关向量。
解决同一个词在不同语境下含义不同的问题。
缺点是直接做检索需要 query-doc 两两拼接，速度太慢。

第三代：SBERT / SimCSE / BGE / E5 / Qwen Embedding
句子级对比学习 Embedding。
query 和 doc 分别独立编码。
文档向量可提前入库。
在线只算 query 向量。
是 RAG 的主流选择。
```

RAG 中的典型检索链路：

```text
离线：
文档 chunk → 第三代 Embedding → 向量库

在线：
用户 query → 同一个 Embedding 模型 → query 向量
query 向量和 chunk 向量做相似度检索
返回 Top-K chunk
```

最重要的面试表达：

```text
Word2Vec 解决了词表示，但处理不了多义词；
BERT 解决了上下文语义，但全库检索太慢；
SBERT / SimCSE / BGE 这类句子级 Embedding 用 Bi-Encoder 独立编码，
可以离线建库、在线快速检索，所以是 RAG 场景的标配。
```

项目中可以这样说：

```text
我的金融分析 Agent 更适合使用 BGE 或 Qwen Embedding 这类第三代模型，
把财报和研报 chunk 提前向量化存入 Milvus。
用户提问时只算一次 query embedding，再从 Milvus 召回相关证据。
如果召回结果需要进一步提升，可以加 Reranker 做精排。
```
