
## 一、面试标准回答

面试官问：**你们线上跑了 RAG，那你怎么衡量它的效果好不好？**

可以这样回答：

RAG 的效果评估不能只靠用户反馈或者人工抽查，而应该建立一套**分层的量化指标体系**。

我一般会从三层来看：

第一层是**检索层评估**，主要看系统有没有把正确的知识片段召回。常用指标是 **Hit@K** 和 **MRR**。  
**Hit@K** 关注的是：正确 chunk 有没有出现在 Top-K 结果里；**MRR** 关注的是：正确 chunk 排得够不够靠前。

第二层是**生成层评估**，主要看大模型基于检索内容生成的答案质量怎么样。常用的是 **RAGAs** 这一类评估框架，核心指标包括 **Faithfulness、Answer Relevancy、Context Recall、Context Precision**。  
**Faithfulness** 看答案有没有幻觉；**Answer Relevancy** 看答案有没有回答用户问题；**Context Recall** 看必要信息有没有被检索出来；**Context Precision** 看检索结果里的有效信息是不是排在前面。

第三层是**线上业务指标**，比如点踩率、追问率、转人工率、空回答率、会话解决率等。  
离线指标主要用于快速定位问题和辅助调参，线上指标才是最终验收标准。

所以我不会只看一个总分，而是会通过不同指标判断问题到底出在**检索召回、排序、上下文质量、Prompt 约束，还是最终用户体验**上，然后针对性优化。

---

# 二、这篇文章的核心观点

这篇文章主要讲的是：

> **RAG 系统不能靠感觉评估，而要用指标体系把“回答得好不好”拆成可量化、可定位、可优化的问题。**

很多人评估 RAG 时会说：

- 用户投诉了，说明效果不好；
- 人工抽查几十条，感觉还可以；
- 看模型回答顺不顺眼。

这些都不够系统。

因为 RAG 系统的问题可能出现在多个环节：

- 可能是**检索没找到正确内容**；
- 可能是**找到了但排得太靠后**；
- 可能是**召回了太多无关 chunk**；
- 可能是**LLM 没有忠实使用上下文**；
- 也可能是**Prompt 没约束好，导致答案跑题或幻觉**。

所以正确做法是：

> **把 RAG 评估拆成检索层、生成层、线上业务层三部分。**

---

# 三、为什么需要 RAG 评估？

RAG 不是搭完之后就结束了，它是一个会不断变化的系统。

比如：

- 知识库会更新；
- 文档切块策略可能会调整；
- Embedding 模型可能会替换；
- Rerank 策略可能会修改；
- Prompt 可能会迭代；
- 用户问题的分布也会变化。

如果没有评估体系，你就不知道：

- 改完之后效果是变好了还是变差了；
- 问题到底出在哪一层；
- 哪个优化方案真的有效；
- 是否出现了线上体验退化。

所以 RAG 评估的本质是：

> **把“感觉好不好用”变成一组可以持续追踪、可以横向对比、可以指导优化的客观数字。**

---

# 四、RAG 评估为什么难？

普通分类任务比较好评估，因为它通常有标准答案。

比如：

~~~text
预测结果 == 标准标签
~~~

但是 RAG 不一样，它有几个难点。

---

## 1. 答案是自然语言，没有唯一标准答案

同一个问题，大模型可能有很多种正确表达。

比如问：

~~~text
RAG 的作用是什么？
~~~

它可以回答：

~~~text
RAG 可以通过外部知识库增强大模型回答能力。
~~~

也可以回答：

~~~text
RAG 主要用于缓解大模型知识过时和幻觉问题。
~~~

这两个答案都可能是对的。

所以不能简单用字符串完全匹配来判断答案对不对。

---

## 2. 问题可能出在不同环节

RAG 是一个链路系统：

~~~text
用户问题 -> Query 改写 -> 检索 -> Rerank -> 拼接上下文 -> LLM 生成
~~~

最终答案错了，不一定是 LLM 的问题。

可能是：

- 检索没找到正确 chunk；
- 找到了但排名太靠后；
- 上下文里噪音太多；
- Prompt 没约束好；
- LLM 编造了上下文里没有的内容。

所以不能只看最终答案，要**分层评估**。

---

## 3. 人工评估成本高，难以持续

人工看几条样本可以做初步检查，但不能长期依赖。

因为：

- 样本太少，代表性不足；
- 人和人的评判标准不一致；
- 大规模测试成本很高；
- 每次系统改动都人工评估不现实。

所以需要自动化、指标化的评估体系。

---

# 五、第一层：检索层评估

检索层评估主要回答一个问题：

> **正确的 chunk 有没有被找出来？**

这里先不管 LLM 怎么回答，只单独评估检索效果。

一般需要准备一批测试数据：

~~~text
问题 + 正确 chunk ID
~~~

例如：

~~~text
问题：腾讯 2025 年 Q2 游戏业务增长原因是什么？
正确 chunk：chunk_1023、chunk_1045
~~~

然后看检索系统返回的 Top-K 结果里，有没有这些正确 chunk。

---

## 1. Hit@K：正确内容有没有被召回

**Hit@K** 的意思是：

> **在 Top-K 检索结果里，是否命中了正确 chunk。**

比如 Hit@5：

~~~text
问题：腾讯 Q2 游戏业务增长原因是什么？

检索 Top-5：
1. chunk_001
2. chunk_078
3. chunk_1023  ← 正确 chunk
4. chunk_344
5. chunk_567
~~~

因为正确 chunk 出现在 Top-5 里，所以这条样本算命中。

如果 100 个问题里，有 80 个问题的正确 chunk 都出现在 Top-5 中，那么：

~~~text
Hit@5 = 80 / 100 = 0.8
~~~

### Hit@K 关注什么？

Hit@K 关注的是：

> **找到了没有。**

它不关心正确 chunk 是排第 1，还是排第 5。

只要在 Top-K 里，就算命中。

---

## 2. MRR：正确内容排得够不够靠前

**MRR** 全称是 **Mean Reciprocal Rank**，中文可以理解为**平均倒数排名**。

它关注的是：

> **正确 chunk 在检索结果中排第几名。**

计算方式：

~~~text
如果正确 chunk 排第 1，得分 = 1 / 1 = 1
如果正确 chunk 排第 2，得分 = 1 / 2 = 0.5
如果正确 chunk 排第 3，得分 = 1 / 3 ≈ 0.33
如果正确 chunk 排第 5，得分 = 1 / 5 = 0.2
~~~

然后对所有问题求平均。

### MRR 关注什么？

MRR 关注的是：

> **多快找到。**

也就是说，正确 chunk 越靠前，MRR 越高。

---

## 3. Hit@K 和 MRR 的区别

这两个指标很容易混淆。

可以这样记：

| 指标 | 关注点 | 回答的问题 |
|---|---|---|
| Hit@K | 是否召回 | 正确内容有没有出现在 Top-K 里？ |
| MRR | 排名位置 | 正确内容排得够不够靠前？ |

举个例子：

~~~text
Hit@5 = 0.9
MRR = 0.3
~~~

这说明：

- 大部分问题的正确 chunk 都能被召回；
- 但是正确 chunk 经常排在第 4、第 5；
- 说明召回还可以，但排序效果不好；
- 优化方向应该是加强 Rerank。

---

# 六、第二层：生成层评估

检索层解决的是：

~~~text
有没有找到正确资料？
~~~

生成层解决的是：

~~~text
找到资料之后，LLM 有没有正确使用这些资料？
~~~

这部分常用 **RAGAs** 这类评估框架。

RAGAs 的核心思想是：

> **LLM-as-a-Judge，也就是让大模型作为裁判，对 RAG 输出结果自动打分。**

它可以降低人工评估成本，但也会带来额外的 LLM 调用成本。

---

# 七、RAGAs 的核心指标

## 1. Faithfulness：忠实度

**Faithfulness** 衡量的是：

> **答案中的内容，是否都能在检索到的上下文里找到依据。**

简单说就是看有没有幻觉。

例如：

~~~text
用户问：公司 Q2 营收是多少？

检索上下文：公司 Q2 营收为 100 亿元。

模型回答：公司 Q2 营收为 100 亿元，并且利润同比增长 50%。
~~~

如果上下文里没有“利润同比增长 50%”这个信息，那么这部分就是编造。

这种情况下 Faithfulness 会下降。

### 记忆方式

Faithfulness 问的是：

~~~text
你说的每句话，在资料里有出处吗？
~~~

它主要衡量：

~~~text
幻觉问题
~~~

---

## 2. Answer Relevancy：答案相关性

**Answer Relevancy** 衡量的是：

> **答案有没有回答用户真正问的问题。**

它和 Faithfulness 不一样。

一个答案可能完全有依据，但仍然跑题。

例如：

~~~text
用户问：北京今天的天气怎么样？

模型回答：北京是中国的首都，有悠久的历史文化。
~~~

这句话可能是事实，也可能有资料支撑，所以 Faithfulness 不一定低。

但是它没有回答天气问题，所以 Answer Relevancy 很低。

### 记忆方式

Answer Relevancy 问的是：

~~~text
你有没有回答用户问的那个问题？
~~~

它主要衡量：

~~~text
跑题问题
~~~

---

## 3. Context Recall：上下文召回率

**Context Recall** 衡量的是：

> **回答这个问题需要的信息，有多少被检索结果覆盖到了。**

它关注的是：

~~~text
该找的有没有找全
~~~

比如用户问：

~~~text
腾讯 Q2 游戏业务增长的原因是什么？
~~~

标准答案需要两个信息：

~~~text
1. 本土游戏收入增长
2. 国际游戏收入增长
~~~

但检索结果只找到了本土游戏收入增长，没有找到国际游戏收入增长。

那么 Context Recall 就不高。

### 记忆方式

Context Recall 问的是：

~~~text
回答问题需要的关键信息，检索结果里覆盖了多少？
~~~

它主要衡量：

~~~text
检索有没有漏召回
~~~

---

## 4. Context Precision：上下文精确率

**Context Precision** 衡量的是：

> **检索结果里的有用内容是否排在前面，噪音是否太多。**

它关注的是：

~~~text
找到的内容里面，有用的内容是不是靠前
~~~

比如检索返回了 10 个 chunk：

~~~text
1. 无关 chunk
2. 无关 chunk
3. 无关 chunk
4. 相关 chunk
5. 相关 chunk
...
~~~

虽然相关 chunk 被召回了，但排得比较靠后，前面噪音太多。

这种情况下 Context Precision 会低。

### 记忆方式

Context Precision 问的是：

~~~text
检索结果里有用内容多不多？有没有排在前面？
~~~

它主要衡量：

~~~text
上下文噪音问题
~~~

---

# 八、通过指标定位问题

这篇文章最重要的地方是：

> **不同指标低，代表不同环节有问题。**

---

## 1. Context Recall 低

说明：

~~~text
检索结果没有覆盖回答问题所需的信息。
~~~

也就是该找的没找全。

可能原因：

- Embedding 模型语义匹配能力不够；
- Chunking 策略不好，关键信息被切散；
- 只用单路向量检索，覆盖面不足；
- Query Rewrite 没做好，导致检索方向偏了。

优化方向：

- 换更强的 Embedding 模型；
- 调整 Chunk 大小和 overlap；
- 加入关键词检索、BM25、多路召回；
- 加 Query Rewrite 或 Multi-Query 扩展。

---

## 2. Context Precision 低

说明：

~~~text
检索结果里噪音太多，相关内容没有排在前面。
~~~

可能原因：

- Top-K 设置太大；
- 向量检索召回了很多语义相似但无关的内容；
- 没有 Rerank；
- Rerank 模型效果不好。

优化方向：

- 加强 Rerank；
- 减少最终送给 LLM 的 chunk 数量；
- 对 chunk 做质量过滤；
- 对检索结果做去重和重排。

---

## 3. Faithfulness 低

说明：

~~~text
LLM 编造了上下文里没有的信息。
~~~

也就是幻觉问题。

可能原因：

- Prompt 没有强约束；
- 检索上下文质量太差；
- LLM 在信息不足时强行回答；
- 没有引用校验机制。

优化方向：

- 在 Prompt 中要求“只能基于上下文回答”；
- 要求答案附带引用来源；
- 加入答案生成后的引用核查；
- 当上下文不足时，让模型回答“不知道”；
- 加检索质量门控，低质量上下文不进入生成阶段。

---

## 4. Answer Relevancy 低

说明：

~~~text
答案没有聚焦用户问题，出现跑题。
~~~

可能原因：

- Prompt 指令不够清晰；
- 上下文太长，LLM 注意力被干扰；
- Query 理解错了；
- 系统倾向于生成泛泛而谈的答案。

优化方向：

- Prompt 中明确要求“只回答用户问题本身”；
- 精简上下文；
- 强化 Query Rewrite；
- 在生成前明确用户意图。

---

# 九、核心指标对照表

| 指标 | 所属层级 | 衡量什么 | 低了说明什么 | 优化方向 |
|---|---|---|---|---|
| Hit@K | 检索层 | 正确 chunk 是否被召回 | Embedding 或 Chunking 有问题 | 换 Embedding、调 Chunking、多路召回 |
| MRR | 检索层 | 正确 chunk 是否排得靠前 | Rerank 效果差 | 加强 Rerank、优化排序 |
| Context Recall | 生成输入层 | 必要信息是否覆盖 | 该找的没找全 | 多路召回、Query Rewrite、优化切块 |
| Context Precision | 生成输入层 | 上下文噪音多不多 | 无关 chunk 太多 | Rerank、过滤、减少 Top-K |
| Faithfulness | 生成层 | 答案是否忠实于上下文 | LLM 幻觉 | Prompt 约束、引用核查、质量门控 |
| Answer Relevancy | 生成层 | 答案是否回答问题 | 答案跑题 | 优化 Prompt、明确用户意图 |
| 点踩率 / 追问率 / 转人工率 | 线上层 | 用户真实满意度 | 整体体验不好 | 结合日志回放，定位问题链路 |

---

# 十、线上指标：最终验收标准

离线指标可以帮助我们快速定位问题，但最终还是要看用户在线上的真实反馈。

常见线上指标包括：

---

## 1. 点踩率

~~~text
thumbs_down_rate
~~~

用户点踩说明这次回答让用户不满意。

它是最直接的负反馈指标。

---

## 2. 追问率

~~~text
followup_rate
~~~

如果用户在同一个问题后继续追问：

~~~text
你没回答我的问题
~~~

或者换一种方式继续问同一件事，说明上一次回答没有解决问题。

追问率高，通常意味着：

- 答案不完整；
- 答案跑题；
- 用户没有获得有效信息。

---

## 3. 转人工率

~~~text
escalation_rate
~~~

用户转人工，说明 RAG 系统没能解决问题。

但是要注意：

> **转人工率上升不一定永远是坏事。**

如果系统加了质量门控，在上下文不足时主动转人工，而不是胡编乱造，那么短期转人工率可能会上升，但整体风险反而降低。

---

## 4. 空回答率

~~~text
answer_empty_rate
~~~

系统回答：

~~~text
我不知道
~~~

或者：

~~~text
根据当前资料无法回答
~~~

的比例。

如果空回答率太高，说明：

- 知识库覆盖不足；
- 检索召回能力不足；
- 质量门控过于严格。

---

## 5. 会话解决率

~~~text
session_resolution_rate
~~~

这是最综合的线上指标。

它衡量的是：

~~~text
一次对话最终有没有解决用户问题。
~~~

这个指标最贴近真实用户体验。

---

# 十一、离线评估和线上评估的关系

离线评估和线上评估不是互相替代，而是互相配合。

完整闭环应该是：

~~~text
离线测试集评估
    ↓
定位问题
    ↓
优化检索 / Rerank / Prompt / 知识库
    ↓
上线灰度
    ↓
观察线上指标
    ↓
发现问题
    ↓
回放线上 bad case
    ↓
加入离线测试集
    ↓
继续迭代
~~~

也就是：

> **离线指标用于定位和调参，线上指标用于最终验收。**

---

# 十二、RAG 评估流程图

~~~mermaid
flowchart TD
    A[用户问题] --> B[检索系统]
    B --> C[Top-K Chunk]
    C --> D[Rerank 排序]
    D --> E[拼接上下文]
    E --> F[LLM 生成答案]
    F --> G[用户反馈]

    B --> H[检索层评估]
    H --> H1[Hit@K: 正确 chunk 是否召回]
    H --> H2[MRR: 正确 chunk 是否靠前]

    E --> I[上下文质量评估]
    I --> I1[Context Recall: 必要信息是否覆盖]
    I --> I2[Context Precision: 有用信息是否靠前]

    F --> J[生成层评估]
    J --> J1[Faithfulness: 是否忠实于上下文]
    J --> J2[Answer Relevancy: 是否回答用户问题]

    G --> K[线上指标]
    K --> K1[点踩率]
    K --> K2[追问率]
    K --> K3[转人工率]
    K --> K4[会话解决率]
~~~

# 十三、指标定位问题流程图

~~~mermaid
flowchart TD
    A[RAG 效果不好] --> B{Hit@K 低吗?}
    B -- 是 --> B1[检索没召回正确 chunk]
    B1 --> B2[优化 Embedding / Chunking / 多路召回]

    B -- 否 --> C{MRR 或 Context Precision 低吗?}
    C -- 是 --> C1[正确内容找到了但排序不好或噪音太多]
    C1 --> C2[加强 Rerank / 降低 Top-K / 过滤无关 chunk]

    C -- 否 --> D{Context Recall 低吗?}
    D -- 是 --> D1[回答所需信息没覆盖全]
    D1 --> D2[Query Rewrite / Multi-Query / 扩展知识库]

    D -- 否 --> E{Faithfulness 低吗?}
    E -- 是 --> E1[LLM 编造上下文没有的信息]
    E1 --> E2[Prompt 约束 / 引用核查 / 检索质量门控]

    E -- 否 --> F{Answer Relevancy 低吗?}
    F -- 是 --> F1[答案跑题]
    F1 --> F2[优化 Prompt / 明确用户意图 / 精简上下文]

    F -- 否 --> G[看线上指标和业务体验]
~~~

# 十四、结合项目怎么说

如果面试官继续追问：

> **你在自己的项目里会怎么做 RAG 评估？**

可以这样回答：

在我的金融文档 RAG 项目里，我会先构建一批金融问答测试集，比如从腾讯多个季度财报中整理出：

~~~text
问题 + 标准答案 + 相关页码 / chunk ID
~~~

然后分层评估。

检索层我会看：

- **Hit@K**：正确财报 chunk 有没有被召回；
- **MRR**：正确 chunk 是否排在前面。

如果 Hit@K 低，说明 Milvus 里的向量检索召回不足，可能要调整 chunk size、overlap、Embedding 模型，或者加入关键词检索做混合召回。

如果 Hit@K 高但 MRR 低，说明正确内容找到了但排得不够靠前，这时候应该加入 Rerank。

生成层我会看：

- **Faithfulness**：答案里的财务数据、同比增速、业务原因是否都能在原文中找到依据；
- **Answer Relevancy**：答案是否聚焦用户问题，而不是泛泛总结整份财报；
- **Context Recall**：回答一个问题需要的多个关键点是否都被检索出来；
- **Context Precision**：最终送入 LLM 的 chunk 里是否混入大量无关页，比如免责声明、目录、背景介绍。

线上或者 Demo 层面，我会记录：

- 用户是否继续追问同一个问题；
- 用户是否认为回答缺少依据；
- 回答是否触发“资料不足无法回答”；
- 用户是否满意最终结果。

这样就可以把 RAG 项目从“能跑起来”提升到“能量化评估和持续优化”。

# 十五、面试官可能继续追问

## 1. Hit@K 和 MRR 有什么区别？

Hit@K 关注的是正确内容有没有出现在 Top-K 结果里，它只看有没有命中，不关心排名。

MRR 关注的是正确内容排在第几名，排得越靠前分数越高。

所以：

- **Hit@K 衡量的是召回能力**；
- **MRR 更偏向衡量排序质量**。

---

## 2. 如果 Hit@K 很高，但答案还是错，说明什么？

说明检索层大概率不是主要问题。

正确 chunk 已经被召回了，但答案仍然错，可能是：

- 正确 chunk 排得太靠后；
- 上下文里噪音太多；
- LLM 没有忠实使用上下文；
- Prompt 没约束好；
- LLM 出现幻觉。

这时要重点看：

- **Context Precision**
- **Faithfulness**
- **Answer Relevancy**

---

## 3. 如果 Faithfulness 低，应该怎么优化？

Faithfulness 低说明答案中有很多内容在上下文里找不到依据，也就是幻觉较多。

优化方式包括：

- Prompt 中明确要求只能基于上下文回答；
- 要求模型输出引用来源；
- 做答案后的引用核查；
- 上下文不足时要求模型拒答；
- 加检索质量门控，避免低质量上下文进入生成阶段。

---

## 4. 如果 Context Recall 低，应该怎么优化？

Context Recall 低说明检索结果没有覆盖回答问题所需的完整信息。

优化方式包括：

- 换更强的 Embedding 模型；
- 优化 Chunking 策略；
- 增大 chunk overlap；
- 加入 BM25、关键词检索、多路召回；
- 使用 Query Rewrite 或 Multi-Query 扩展用户问题。

---

## 5. 为什么不能只看线上点踩率？

因为点踩率是滞后指标。

用户点踩时，说明错误答案已经影响了用户。

而且用户不点踩不代表答案好，可能用户直接放弃使用了。

所以线上指标要看，但不能只看线上指标。

更合理的做法是：

~~~text
离线指标用于提前发现问题和定位问题；
线上指标用于最终验证真实用户体验。
~~~

---

# 十六、记忆版总结

这篇文章可以压缩成一句话：

> **RAG 评估要分层：检索层看找没找到，生成层看答得对不对，线上层看用户满不满意。**

再进一步记：

~~~text
Hit@K：有没有找到
MRR：排得靠不靠前
Context Recall：该找的有没有找全
Context Precision：找出来的噪音多不多
Faithfulness：有没有幻觉
Answer Relevancy：有没有跑题
线上指标：用户到底满不满意
~~~

最终目标不是为了追求某个指标好看，而是为了：

~~~text
发现问题 -> 定位问题 -> 优化系统 -> 线上验证 -> 持续迭代
~~~
