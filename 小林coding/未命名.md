# 为什么有时候选择“手搓”Agent，而不是直接用成熟框架？

## 一、面试标准回答

如果面试官问：

> 你平时做 Agent 开发用什么框架？为什么有时候会选择手搓 Agent，而不是直接用 LangChain / LangGraph / LlamaIndex 这类成熟框架？

可以这样回答：

我不会简单地说框架不好，也不会盲目坚持所有东西都手写。我的理解是，**框架和手搓 Agent 的选择，取决于项目阶段和对控制权的要求**。

在 **POC 阶段**，我会倾向于使用框架。因为框架已经帮我们封装好了工具注册、ReAct Loop、Tool Calling、Memory、Tracing、Callback、RAG 接入等通用能力，可以快速跑通 Demo，验证想法是否成立。这个阶段最重要的是开发效率，而不是极致可控。

但是到了 **生产阶段**，如果核心 Agent 逻辑完全依赖框架，就会遇到几个问题：

第一是 **抽象层太多，调试困难**。  
比如一个工具参数解析错误，表面上是自己的代码报错，但真正调用链可能藏在框架内部几十层 stack trace 里。排查时不仅要看自己的业务代码，还要理解框架内部的 AgentExecutor、Callback、Runnable、Memory 等执行逻辑。

第二是 **版本升级可能带来 breaking change**。  
Agent 框架发展很快，接口变化、推荐写法变化、模块废弃都比较常见。如果核心链路强绑定框架，一次依赖升级可能导致线上逻辑不兼容。

第三是 **框架的通用性设计会带来隐性开销**。  
框架为了适配各种场景，内部可能做了很多通用处理，比如序列化、中间状态包装、Callback 分发、日志记录、Tracing 适配等。但在具体业务里，这些逻辑不一定都需要，高并发或低延迟场景下会变成额外成本。

所以我更认可的工程方案是：

> **核心 Agent Loop 手写，周边工具性能力借用框架。**

比如工具调用循环、messages 管理、错误重试、任务状态维护、记忆写入策略，这些决定 Agent 行为的核心逻辑，我希望自己完全掌控。  
而像文档解析、向量数据库客户端、Tracing 平台、评估工具这类周边能力，可以借助成熟框架或第三方工具，提高开发效率。

总结来说：

> 框架适合快速验证和复用通用能力；手搓适合生产阶段掌控核心链路。真正成熟的做法不是全盘否定框架，而是明确边界：核心逻辑自己掌控，非核心能力合理复用。

---

## 二、这篇文章的核心主题

这篇文章主要回答的是：

> 在 Agent 工程实践中，为什么有些场景下不直接用成熟框架，而是选择手搓 Agent？

核心观点是：

> 框架不是不好，手搓也不是天然更高级。关键在于：框架适合快速启动，但当项目进入生产、调试、性能优化和长期维护阶段时，核心链路是否可控就变得非常重要。

所以这篇文章真正讲的不是：

```text
框架 vs 手搓，谁更好？
```

而是：

```text
什么阶段适合框架？
什么阶段适合手搓？
哪些核心逻辑必须掌控？
哪些周边能力可以复用框架？
```

---

## 三、框架的价值是什么？

### 1. 框架解决了大量重复工作

从零搭建一个 Agent，通常需要自己实现很多通用能力。

比如：

```text
1. 定义工具格式
2. 让 LLM 理解工具名称、参数和用途
3. 解析 LLM 返回的工具调用结果
4. 执行工具函数
5. 把工具结果写回 messages
6. 维护多轮对话历史
7. 控制 Agent 最大执行轮数
8. 处理工具调用失败和重试
9. 接入向量数据库
10. 管理 Memory 和 Callback
```

这些事情几乎每个 Agent 项目都会遇到。

框架的价值就在于：

> 把这些通用样板代码封装起来，让开发者更快搭出一个能跑的 Agent。

---

### 2. 框架在 POC 阶段非常有价值

在项目早期，最重要的是：

```text
先跑通流程
先验证 idea
先确认用户需求是否成立
```

这个阶段用框架很合适。

比如 LangChain / LangGraph / LlamaIndex 这类框架可以快速提供：

```text
工具注册
Agent Loop
RAG 管道
Memory 接入
Tracing
Callback
文档加载器
向量检索封装
```

这样原本可能需要一两周写完的通用代码，用框架可能两三天就能搭起来。

所以面试时不能一上来就说：

```text
框架不好，所以我要手搓。
```

这会显得太绝对。

更好的说法是：

```text
框架在 POC 和快速验证阶段非常有价值，但生产阶段要重新评估核心链路是否需要自己掌控。
```

---

## 四、框架的问题什么时候开始出现？

框架的问题通常不是项目一开始就暴露，而是在项目逐渐复杂、进入线上、需要稳定性和性能时才会明显出现。

主要有三个痛点：

```text
1. 抽象层太多，调试困难
2. 版本升级带来 breaking change
3. 通用性设计带来隐性性能开销
```

---

## 五、痛点一：抽象层太多，调试困难

### 1. 问题表现

使用框架时，表面上自己的代码可能只有几十行。

例如：

```python
agent = create_openai_tools_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools)
result = executor.invoke({"input": "帮我查一下今天的天气"})
```

看起来非常简洁。

但是当线上出现问题时，比如：

```text
工具参数解析错误
工具没有被正确调用
messages 顺序不对
模型输出格式不符合预期
callback 触发时机异常
memory 没有正确写入
```

你就需要往框架内部追。

可能会看到很长的调用链：

```text
AgentExecutor
Runnable
CallbackManager
ToolExecutor
OutputParser
MemoryManager
...
```

这时候问题就变成：

> 你不知道 bug 出在自己的业务代码，还是框架内部封装逻辑，还是某个版本变化导致的行为差异。

---

### 2. 为什么这是生产环境的大问题？

生产环境排查 bug 最重要的是：

```text
快速定位问题
快速复现问题
快速修复问题
```

但框架抽象层太多时，排查路径会变长：

```text
用户反馈异常
   ↓
查看业务日志
   ↓
发现业务代码没有明显问题
   ↓
查看框架调用链
   ↓
阅读框架源码
   ↓
搜索 GitHub issue
   ↓
确认是框架行为、版本问题还是使用方式问题
```

这会显著增加认知负担。

---

### 3. 类比理解

框架就像一辆高度集成的现代汽车。

好处是：

```text
开起来舒服
功能很多
上手简单
```

但坏处是：

```text
一旦出问题，引擎盖打开之后全是复杂电子模块，
你很难一眼看出问题在哪里。
```

手搓 Agent 更像一辆结构简单的老车。

虽然没有那么多自动化能力，但：

```text
每根线、每个零件你都知道在哪里。
```

出问题时更容易排查。

---

## 六、痛点二：版本升级带来 breaking change

### 1. 问题表现

Agent 框架发展很快，版本变化也比较快。

可能会出现：

```text
某个接口被废弃
某个模块迁移到新包
某个参数含义变化
推荐写法从 AgentExecutor 变成 LangGraph
Memory 模块用法变化
Callback 机制变化
```

如果核心业务逻辑高度依赖框架，一次依赖升级可能导致整个 Agent 链路出问题。

---

### 2. 为什么这很危险？

生产系统最怕的是：

```text
业务逻辑没改，但依赖升级后行为变了。
```

比如：

```text
原来工具调用结果会自动写入 messages
升级后需要手动处理

原来 output parser 能兼容某种格式
升级后解析失败

原来 memory 自动注入 prompt
升级后注入顺序变了
```

这些变化可能不是语法层面的报错，而是行为层面的变化，更难发现。

---

### 3. 手搓为什么更稳定？

如果核心 Agent Loop 是自己写的：

```text
messages 怎么拼
工具什么时候调
结果怎么写回
失败怎么重试
最大轮数怎么控制
```

这些行为都不会因为第三方框架升级而突然改变。

只要底层 LLM SDK 接口相对稳定，核心逻辑就更可控。

---

## 七、痛点三：通用性设计带来隐性性能开销

### 1. 框架为什么会有额外开销？

框架是为了适配很多场景设计的。

所以它内部通常会加入大量通用逻辑，比如：

```text
中间状态包装
输入输出序列化
Callback 分发
Tracing 记录
Memory 自动处理
工具 schema 适配
错误包装
日志采集
兼容不同模型和不同工具类型
```

这些设计对框架来说是合理的，因为它要支持各种用法。

但对你的具体业务来说，可能很多逻辑并不需要。

---

### 2. 在小规模场景下问题不明显

如果只是做 Demo 或低频调用：

```text
多几十毫秒
多一些中间对象
多一点 token 包装
```

可能完全感受不到。

---

### 3. 在规模化场景下会变成真实成本

当流量上来后，这些隐性开销会累积。

比如：

```text
每次调用多 100ms
每天调用 10 万次
额外延迟就非常明显
```

或者：

```text
每次 prompt 多塞一些框架包装内容
每次工具调用多记录一堆 callback
每轮 Agent 多做几次无用状态转换
```

这些都会带来：

```text
延迟增加
token 成本增加
计算资源浪费
排查复杂度增加
```

---

## 八、手搓 Agent 的本质优势：完全掌控

文章强调，手搓 Agent 的核心优势可以概括成四个字：

```text
完全掌控
```

具体体现在三个方面：

```text
1. 链路透明，可观测性好
2. 精确裁剪，没有多余开销
3. 稳定可控，不受框架升级影响
```

---

## 九、优势一：链路透明，可观测性好

手搓 Agent 时，每一步都在自己代码里。

比如：

```text
什么时候调用 LLM
messages 长什么样
LLM 返回了什么
有没有 tool_calls
工具参数是什么
工具结果怎么写回 messages
什么时候结束循环
什么时候重试
```

这些都可以完全看到。

---

### 1. 手搓 Agent Loop 示例

```python
messages = [{"role": "system", "content": system_prompt}]
messages.append({"role": "user", "content": user_input})

for step in range(max_turns):
    # 1. 调用 LLM
    response = client.chat.completions.create(
        model=model_name,
        messages=messages,
        tools=tool_schemas
    )

    msg = response.choices[0].message
    messages.append(msg)

    # 2. 如果没有工具调用，说明任务完成
    if not msg.tool_calls:
        final_answer = msg.content
        break

    # 3. 执行工具调用
    for tool_call in msg.tool_calls:
        tool_name = tool_call.function.name
        tool_args = tool_call.function.arguments

        result = execute_tool(tool_name, tool_args)

        # 4. 把工具结果写回 messages
        messages.append({
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": result
        })

        # 5. 记录日志
        logger.info({
            "step": step,
            "tool_name": tool_name,
            "tool_args": tool_args,
            "tool_result": result
        })
```

这段逻辑虽然比框架代码长，但好处是：

```text
每一步都可见
每一步都能打日志
每一步都能加断点
每一步都能插入监控
```

---

### 2. 为什么这对线上很重要？

线上排查时最重要的问题通常是：

```text
模型到底看到了什么？
模型为什么选择这个工具？
工具参数是怎么生成的？
工具执行结果是什么？
结果有没有正确写回上下文？
为什么 Agent 没有停止？
为什么 Agent 提前停止？
```

如果是手搓 Agent，这些问题可以通过日志直接回答。

如果是高度封装框架，就可能需要进入框架内部追调用链。

---

## 十、优势二：精确裁剪，没有多余开销

手搓 Agent 的另一个优势是：

> 只写业务真正需要的逻辑。

比如你的项目只需要：

```text
一个简单的 Tool Calling Loop
最多执行 5 轮
工具失败重试 1 次
messages 只保留最近上下文
日志只记录工具名称、参数和结果摘要
```

那就只实现这些逻辑。

不需要为了适配所有场景而引入：

```text
复杂 callback 系统
多种 memory 类型
多种 agent 类型
多层 runnable 包装
不需要的 tracing adapter
复杂中间状态对象
```

这样系统会更轻。

---

## 十一、优势三：稳定可控，不受框架升级影响

核心逻辑手写后，系统行为由自己决定。

比如：

```text
工具调用格式
messages 维护方式
memory 注入顺序
异常处理策略
重试策略
终止条件
日志字段
```

这些都不会因为第三方框架升级而变化。

这对生产环境非常重要。

因为生产环境最怕：

```text
依赖版本升级后，核心行为悄悄变了。
```

---

## 十二、框架写 vs 手搓写：差别在哪里？

### 1. 框架写法

框架写法通常很短：

```python
from langchain.agents import AgentExecutor, create_openai_tools_agent

agent = create_openai_tools_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools)

result = executor.invoke({"input": "帮我查一下今天的天气"})
```

优点是：

```text
代码短
上手快
功能封装完整
适合快速验证
```

缺点是：

```text
核心循环在框架内部
messages 怎么维护不直观
工具调用细节不直观
调试时要读框架源码
```

---

### 2. 手搓写法

手搓写法代码更多，但逻辑更透明：

```python
messages = [{"role": "system", "content": system_prompt}]
messages.append({"role": "user", "content": user_input})

for i in range(max_turns):
    response = client.chat.completions.create(
        model="gpt-4",
        messages=messages,
        tools=tool_schemas
    )

    msg = response.choices[0].message
    messages.append(msg)

    if not msg.tool_calls:
        break

    for tc in msg.tool_calls:
        result = execute_tool(tc.function.name, tc.function.arguments)

        messages.append({
            "role": "tool",
            "tool_call_id": tc.id,
            "content": result
        })

        logger.info(f"工具 {tc.function.name} 返回: {result}")
```

优点是：

```text
每一步都看得见
容易打日志
容易定位问题
容易做业务定制
容易裁剪性能开销
```

缺点是：

```text
前期开发成本更高
需要自己处理边界情况
需要自己维护代码质量
不能直接享受框架的完整生态
```

---

## 十三、什么时候适合用框架？

适合用框架的场景主要有三类。

---

### 1. POC 阶段快速验证 idea

如果当前目标是：

```text
快速验证想法
快速搭 Demo
快速跑通 Agent 流程
快速给用户看效果
```

那框架非常适合。

因为这个阶段最重要的是速度。

例如：

```text
两天内搭一个能查新闻、调工具、回答问题的 Agent Demo
快速验证 RAG + Agent 是否能解决业务问题
快速尝试多种 Memory 或 Tool Calling 方式
```

这时不需要一开始就追求完全掌控。

---

### 2. 团队刚接触 Agent 开发

如果团队对 Agent 还不熟悉，框架可以帮助减少基础性错误。

比如：

```text
工具 schema 怎么写
tool_call 怎么解析
工具结果怎么回填
RAG 管道怎么搭
Memory 怎么接
Tracing 怎么看
```

框架可以提供一套成熟范式，让团队先理解 Agent 的完整流程。

---

### 3. 周边能力依赖框架生态

有些框架在周边生态上确实做得很好。

例如：

```text
文档解析器
向量库适配器
RAG pipeline
Tracing 工具
评估工具
数据加载器
Callback 系统
```

如果这些能力不是你的核心业务逻辑，可以直接借用。

例如：

```text
用 LlamaIndex 做文档解析和索引
用 LangSmith 做 tracing
用向量库官方 Python SDK 做检索
用框架封装好的 document loader 加快开发
```

---

## 十四、什么时候适合手搓？

适合手搓的场景主要有四类。

---

### 1. 准备上生产，稳定性成为核心诉求

生产环境最重要的是：

```text
稳定
可控
可排查
可恢复
```

如果 Agent 核心链路完全黑盒，线上出问题就很麻烦。

所以生产阶段更适合把核心逻辑手写。

---

### 2. 流量上来，性能和成本敏感

当调用量比较大时，每一点额外开销都会变成成本。

这时需要精确控制：

```text
每次 LLM 调用带多少上下文
每次工具调用是否必要
是否重复序列化
日志记录粒度
tracing 是否采样
memory 是否每轮都注入
```

框架的通用封装可能不够精细，手写更容易优化。

---

### 3. 业务逻辑高度定制

如果业务需求很特殊，框架的通用设计可能反而成为负担。

例如你的 Agent 需要：

```text
特殊的工具选择策略
特殊的 memory 注入顺序
特殊的 RAG 检索策略
特殊的错误恢复逻辑
特殊的多轮状态机
特殊的权限控制
特殊的审计日志格式
```

这时为了适配框架，可能要绕很多弯。

与其强行塞进框架，不如直接手写核心逻辑。

---

### 4. 团队需要高可观测性

如果你需要清楚记录：

```text
每轮 LLM 输入输出
工具调用参数
工具执行耗时
工具返回摘要
RAG 命中文档
memory 注入内容
异常栈
重试次数
最终停止原因
```

那手写会更直接。

因为你可以按照自己的业务需求设计日志和 tracing。

---

## 十五、最务实的方案：核心手写，周边借用

这篇文章最重要的工程结论是：

> 框架和手搓不是非此即彼，最常见也最务实的方案是：核心逻辑手写，周边工具借用。

---

### 1. 什么是核心逻辑？

Agent 的核心逻辑包括：

```text
1. Agent Loop
2. Tool Calling 流程
3. messages 管理
4. 任务状态维护
5. 错误处理和重试
6. Memory 读写策略
7. RAG 检索决策
8. 终止条件
9. 关键日志和监控
```

这些逻辑直接决定系统行为，最好自己掌控。

---

### 2. 什么是周边工具？

周边工具包括：

```text
1. 文档解析
2. 向量数据库客户端
3. Embedding 调用
4. Tracing 平台
5. 评估工具
6. 数据加载器
7. 文件格式转换
8. UI Demo 工具
```

这些能力不直接决定 Agent 的核心决策逻辑，可以借用框架或第三方库。

---

### 3. 类比理解

可以把 Agent 系统比作盖房子。

核心逻辑就像：

```text
地基
承重墙
房间布局
水电主线
```

这些必须自己清楚。

周边工具就像：

```text
门锁
插座
水龙头
灯具
家具
```

这些可以买现成的。

所以合理做法是：

```text
核心结构自己设计，标准零件直接采购。
```

## 十六、工程演进路径

一个真实项目常见的演进路线是：

```text
第一阶段：框架快速 POC
   ↓
第二阶段：发现核心链路调试困难
   ↓
第三阶段：把 Agent Loop / Tool Calling / Memory 管理替换成手写
   ↓
第四阶段：流量上来后继续裁剪性能敏感逻辑
   ↓
第五阶段：框架只保留周边工具能力
```

也就是：

```text
先借框架提速
再逐步拿回核心控制权
```

这比一开始就完全手搓，或者一直完全依赖框架，都更务实。

## 十七、判断信号：什么时候该从框架迁移到手搓？

可以用几个信号判断。

---

### 1. 你已经看不清框架内部做了什么

如果你调用了一个方法：

```python
executor.invoke(...)
```

但你说不清：

```text
它内部怎么拼 messages
怎么解析 tool_calls
怎么处理工具异常
怎么决定终止
怎么注入 memory
```

那就说明你对核心链路缺乏掌控。

---

### 2. 线上问题经常需要读框架源码

如果排查一个业务 bug 时，经常需要：

```text
看框架源码
翻 GitHub issue
查版本变更记录
绕过框架内部逻辑
```

那说明框架抽象已经变成负担。

---

### 3. 业务需求经常和框架默认设计冲突

例如你想改：

```text
工具选择规则
memory 插入位置
RAG 检索顺序
错误恢复策略
日志格式
状态管理方式
```

但框架默认封装不方便改。

这时可以考虑把核心逻辑拆出来手写。

---

### 4. 性能瓶颈来自框架内部开销

如果 profile 发现：

```text
LLM 调用本身不慢
工具执行也不慢
但框架中间包装、callback、序列化、日志耗时明显
```

那就说明需要更精确地裁剪核心逻辑。

---

## 十八、结合自己的 FinReportAgent 项目怎么回答？

如果结合自己的金融分析 Agent 项目，可以这样说：

在我的 FinReportAgent 项目里，我更倾向于采用：

```text
核心链路自己实现，周边能力合理借用
```

因为这个项目的核心不是简单调一个工具，而是要围绕金融 PDF 分析构建一条完整链路：

```text
PDF 文本提取
文档类型识别
RAG 检索
Session Memory
Research Memory
行情 / 新闻工具调用
结构化中文研报生成
```

其中真正需要掌控的是：

```text
1. Agent 如何组织 prompt
2. 用户问题如何进入 Session Memory
3. RAG 检索结果如何作为 evidence 注入
4. Research Memory 如何补充历史研究结论
5. 工具调用结果如何写回上下文
6. 最终研报如何基于证据生成
```

这些逻辑如果完全交给框架，调试起来会很困难。

比如模型回答质量不好时，我需要清楚知道：

```text
是 PDF chunk 没检索到？
是 Milvus 返回结果不相关？
是 Research Memory 注入太多干扰？
是 prompt 模板不适合当前文档类型？
还是 Agent 没有正确使用工具结果？
```

所以这些核心链路我更希望自己写清楚，并且在每一步加日志。

但是周边能力我不会重复造轮子。

比如：

```text
PyMuPDF 用来解析 PDF
Milvus 客户端用来管理向量检索
Chroma / SQLite 用来管理 Research Memory
Gradio 用来搭建 Demo UI
搜索 / 行情 API 用来获取外部信息
```

这些都是工具性能力，可以直接借用成熟库。

如果后续项目扩展，我也会优先保持核心 Agent Loop 可控，再按需引入 LangGraph 这类框架做流程编排，而不是让框架完全接管核心决策。

## 十九、面试官可能追问

### 追问 1：为什么不用成熟框架？是不是重复造轮子？

可以回答：

不是完全不用框架，也不是为了重复造轮子。

我的策略是：

```text
通用周边能力可以用框架；
核心业务链路尽量自己掌控。
```

因为 Agent 的核心循环、工具调用、消息管理、Memory 注入和错误处理，直接决定系统行为。如果这些完全藏在框架里，生产环境排查问题会很困难。

所以手搓的是核心链路，不是所有东西都从零实现。

---

### 追问 2：框架最大的价值是什么？

可以回答：

框架最大的价值是 **快速验证和复用通用能力**。

比如工具注册、Agent Loop、RAG 接入、Memory、Tracing、Callback 等，框架都提供了现成封装。在 POC 阶段，它可以显著降低开发成本，让团队快速跑通流程。

所以我认为框架适合早期验证，不适合在不了解内部机制的情况下直接承载核心生产链路。

---

### 追问 3：手搓 Agent 最大优势是什么？

可以回答：

手搓 Agent 最大优势是 **完全掌控**。

具体包括：

```text
1. 链路透明，每一步都知道发生了什么
2. 可观测性好，可以按业务需求打日志和监控
3. 性能可裁剪，只保留必要逻辑
4. 稳定性更高，不容易受框架升级影响
5. 定制能力强，方便适配复杂业务流程
```

---

### 追问 4：手搓 Agent 有什么缺点？

可以回答：

手搓也有明显缺点：

```text
1. 前期开发成本更高
2. 很多通用能力要自己实现
3. 对工程能力要求更高
4. 边界情况要自己处理
5. 不容易直接享受框架生态
```

所以我不会说手搓一定比框架好，而是生产阶段核心链路更适合手写，周边能力仍然可以复用框架。

---

### 追问 5：你会如何设计一个手搓 Agent Loop？

可以回答：

一个最小可用的 Tool Calling Agent Loop 包括：

```text
1. 初始化 system prompt 和 messages
2. 加入用户输入
3. 调用 LLM
4. 判断是否有 tool_calls
5. 如果有工具调用，解析工具名和参数
6. 执行对应工具
7. 把工具结果以 tool message 写回 messages
8. 继续下一轮 LLM 调用
9. 如果没有工具调用，返回最终答案
10. 设置最大轮数，避免死循环
```

伪代码如下：

```python
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_input}
]

for step in range(max_turns):
    response = llm.chat(
        messages=messages,
        tools=tool_schemas
    )

    assistant_msg = response.message
    messages.append assistant_msg

    if not assistant_msg.tool_calls:
        return assistant_msg.content

    for tool_call in assistant_msg.tool_calls:
        tool_result = execute_tool(
            name=tool_call.name,
            arguments=tool_call.arguments
        )

        messages.append({
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": tool_result
        })

return "任务未在最大轮数内完成"
```

核心是：

```text
LLM 决策 → 工具执行 → 结果回填 → 再次决策
```

这就是最基本的 Agent Loop。

## 二十、容易踩的坑

### 坑 1：一上来就否定框架

错误回答：

```text
框架都不好，所以我不用框架。
```

问题是：

```text
这会显得太绝对，也没有看到框架在 POC 阶段的价值。
```

正确回答：

```text
框架适合快速验证和复用通用能力，但生产阶段核心链路需要更高掌控权。
```

---

### 坑 2：只说手搓更灵活

错误回答：

```text
手搓更灵活，所以我手搓。
```

问题是：

```text
太泛泛，没有说清楚灵活体现在哪里。
```

正确回答：

```text
手搓的优势具体体现在：链路透明、可观测性好、性能可裁剪、稳定不受框架升级影响。
```

---

### 坑 3：把手搓理解成所有东西都自己写

错误理解：

```text
手搓 Agent 就是什么库都不用。
```

正确理解：

```text
手搓主要是指核心 Agent Loop 和业务链路自己掌控，周边工具仍然可以使用成熟库。
```

---

### 坑 4：不了解框架内部就重度依赖

错误做法：

```text
直接调用框架高级接口，但不知道内部做了什么。
```

问题是：

```text
Demo 阶段没问题，生产出问题时很难排查。
```

正确做法：

```text
使用框架前要知道它替你做了什么；如果核心链路无法解释，就不要盲目依赖。
```

---

### 坑 5：生产系统完全依赖框架默认行为

错误做法：

```text
工具调用、memory、重试、日志、状态管理全部用框架默认行为。
```

问题是：

```text
默认行为不一定适合你的业务，线上异常时也不好定位。
```

正确做法：

```text
核心行为要显式设计，关键链路要有日志和监控。
```

## 二十一、框架 vs 手搓对比表

| 对比项 | 框架方案 | 手搓方案 |
|---|---|---|
| **开发效率** | 高，适合快速验证 | 较低，需要自己实现 |
| **上手成本** | 低，封装好很多能力 | 较高，需要理解底层流程 |
| **调试难度** | 抽象层多，复杂问题难排查 | 链路透明，容易定位 |
| **可控性** | 受框架设计影响 | 完全掌控 |
| **性能优化** | 可能受框架封装限制 | 可以精确裁剪 |
| **版本稳定性** | 受框架升级影响 | 核心逻辑稳定 |
| **生态能力** | 丰富，工具多 | 需要自己集成 |
| **适用阶段** | POC / Demo / 快速验证 | 生产 / 高定制 / 性能敏感 |
| **最佳实践** | 不要盲目依赖 | 不要重复造所有轮子 |

## 二十二、记忆版总结

这篇文章可以浓缩成一句话：

> 框架适合快速验证，手搓适合掌控生产核心链路；真正成熟的工程方案不是二选一，而是核心 Agent Loop 自己写，周边工具性能力合理复用框架。

再进一步记：

```text
框架的价值：
1. 快速搭建 POC
2. 封装工具注册、Agent Loop、Memory、RAG 等通用能力
3. 降低早期开发成本
4. 适合团队快速理解 Agent 流程

框架的痛点：
1. 抽象层太多，调试困难
2. 版本升级可能带来 breaking change
3. 通用性设计带来隐性性能开销
4. 高度定制业务中可能反而不好改

手搓的优势：
1. 链路透明
2. 可观测性好
3. 性能可裁剪
4. 稳定可控
5. 定制能力强

什么时候用框架：
1. POC 阶段
2. 快速验证 idea
3. 团队刚接触 Agent
4. 周边生态能力成熟且非核心

什么时候手搓：
1. 准备上生产
2. 性能和成本敏感
3. 业务逻辑高度定制
4. 需要高可观测性和可调试性

最佳实践：
核心逻辑手写，周边能力借用。
```

面试最重要的是记住这三句话：

```text
1. 不要否定框架，框架在 POC 阶段非常有价值。
2. 手搓的核心价值是完全掌控，尤其是调试、性能、稳定性和定制能力。
3. 工程上最务实的方案是核心 Agent Loop 手写，周边工具性能力借用框架。
```
