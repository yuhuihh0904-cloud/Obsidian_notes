# 面试题：ReAct、Plan-and-Execute、Reflection 三种 Agent 范式有什么核心区别？实际项目中该如何选型？

## 一、面试标准答案

我理解 **ReAct、Plan-and-Execute、Reflection** 这三种范式并不是简单的并列替代关系，而是在解决 Agent 系统里的不同层次问题。

**ReAct** 解决的是 **单步灵活决策和工具调用** 的问题。它的核心流程是：

```text
Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

也就是 Agent 每一步先思考，再决定要不要调用工具，然后根据工具返回的 Observation 继续推理。它的特点是 **边想边干，走一步看一步**，灵活性很高，适合流程不固定、复杂度中等的任务，比如信息搜索、简单问答、客服机器人、工具调用助手。

**Plan-and-Execute** 解决的是 **复杂长任务容易跑偏** 的问题。它会先让 Planner 从全局角度生成完整计划，再让 Executor 按计划一步步执行。它的特点是 **先规划，再执行**，把“规划”和“执行”解耦开。相比 ReAct，它全局结构更清楚，不容易在长流程任务中偏离目标，适合深度调研、多文档分析、竞品分析报告、金融研报生成等复杂任务。

**Reflection** 解决的是 **输出质量不够高** 的问题。它不是一套独立的完整 Agent 流程，而是可以叠加在 ReAct 或 Plan-and-Execute 上的一层质量增强机制。它会在生成结果之后，让模型对结果进行自我检查、自我评估和自我修正。它适合对准确性、完整性、严谨性要求很高的场景，比如代码生成、正式报告、法律文书、金融分析等。

所以三者的核心区别可以总结为：

```text
ReAct：解决“怎么灵活地一步步做”
Plan-and-Execute：解决“复杂任务怎么不跑偏”
Reflection：解决“结果怎么做得更好”
```

实际项目中，选型主要看三个维度：

1. **任务复杂度**
2. **流程确定性**
3. **输出质量要求**

如果任务不复杂、流程不固定、需要实时根据工具结果调整，可以优先用 **ReAct**。

如果任务很长、步骤很多、容易跑偏、需要全局结构，就用 **Plan-and-Execute**。

如果输出要求高、不能有明显错误，就在 ReAct 或 Plan-and-Execute 基础上叠加 **Reflection**。

在生产项目里，常见组合是：

```text
Plan-and-Execute 负责全局规划
ReAct 负责每一步内部的工具调用
Reflection 负责最终结果质量检查
Workflow 负责整体链路控制
```

一句话总结就是：

> **简单任务用 ReAct，复杂长任务用 Plan-and-Execute，高质量场景再加 Reflection；不要为了炫技把所有机制都堆上去，工程上要够用就好。**

---

## 二、这篇博客的核心观点

这篇博客的核心不是单纯背三个概念，而是强调：

> **ReAct、Plan-and-Execute、Reflection 解决的是不同层次的问题，不能简单理解成谁比谁高级。**

很多人容易把三者看成并列关系：

```text
ReAct vs Plan-and-Execute vs Reflection
```

但更准确的理解应该是：

```text
ReAct：基础执行范式
Plan-and-Execute：复杂任务规划范式
Reflection：质量增强机制
```

也就是说：

- **ReAct 和 Plan-and-Execute** 更像是 Agent 的主流程范式；
- **Reflection** 不是独立主流程，而是可以挂在前两者上的检查机制；
- **动态 Replan** 是 Plan-and-Execute 的增强；
- **Reflexion** 是 Reflection 的增强。

这道题面试时最关键的是要说清楚：

> **Reflection 不是和 ReAct、Plan-and-Execute 完全同级的独立流程，而是给它们加的“检查修正 buff”。**

## 三、三个概念的定位

## 1. ReAct：基础款，边想边干

### 1.1 ReAct 是什么？

**ReAct = Reasoning + Acting**。

它的核心是：

> **让模型在每一步先推理，再行动，再根据行动结果继续推理。**

典型流程是：

```text
Thought → Action → Observation → Thought → Action → Observation → Final Answer
```

其中：

- **Thought**：模型思考当前任务该怎么做；
- **Action**：模型决定调用哪个工具；
- **Observation**：工具返回执行结果；
- **Final Answer**：模型认为信息足够，输出最终答案。

---

### 1.2 ReAct 的特点

ReAct 最大的特点是：

> **走一步看一步。**

它不会一开始就把完整计划全部制定出来，而是每一步都根据当前 Observation 实时调整。

比如用户问：

```text
帮我查一下某家公司最近的新闻，并总结对股价的影响。
```

ReAct 可能这样执行：

```text
Thought: 用户需要最近新闻，我应该先搜索新闻。
Action: search("某家公司 最近 新闻")
Observation: 搜索到几条新闻。

Thought: 现在有新闻了，但还需要股价信息。
Action: stock_price("某家公司")
Observation: 返回最近股价变化。

Thought: 新闻和股价都有了，可以综合分析。
Final Answer: ...
```

---

### 1.3 ReAct 的优点

ReAct 的优点是：

1. **实现简单**  
   不需要复杂规划模块，基本只要有 LLM、工具、循环控制就能跑起来。

2. **灵活性高**  
   每一步都可以根据最新 Observation 调整方向。

3. **适合工具调用**  
   搜索、查库、计算、调用 API 等场景非常适合 ReAct。

4. **逻辑透明**  
   Thought、Action、Observation 都可以记录下来，方便观察 Agent 的执行过程。

---

### 1.4 ReAct 的缺点

ReAct 的缺点也很明显：

1. **长任务容易跑偏**  
   因为没有全局计划，执行步骤一多，Agent 可能被中间信息带偏。

2. **容易陷入无效循环**  
   比如反复搜索、反复调用同一个工具，但得不到有效结果。

3. **错误容易传播**  
   如果某一步 Observation 错了，后续推理可能都建立在错误基础上。

4. **上下文越来越长**  
   每一轮都要带上历史记录，token 消耗会快速增加。

---

### 1.5 ReAct 适合什么场景？

ReAct 适合：

- 日常信息搜索；
- 简单问答助手；
- 客服机器人；
- 简单工具调用；
- 流程不固定但复杂度不高的任务；
- 新手入门 Agent 的第一种范式。

可以简单记：

```text
任务不复杂 + 流程不固定 + 需要灵活调整 = ReAct
```

---

## 2. Plan-and-Execute：复杂任务款，先规划再执行

### 2.1 Plan-and-Execute 是什么？

**Plan-and-Execute** 是为了解决 ReAct 长任务容易跑偏的问题。

它的核心思想是：

> **先让 Planner 制定完整计划，再让 Executor 按计划执行。**

典型流程是：

```text
用户目标
  ↓
Planner：拆解任务，生成完整计划
  ↓
Executor：按计划逐步执行
  ↓
汇总每一步结果
  ↓
生成最终答案
```

---

### 2.2 Plan-and-Execute 和 ReAct 的核心区别

ReAct 是：

```text
边规划边执行
走一步看一步
```

Plan-and-Execute 是：

```text
先完整规划
再分步执行
```

它把 ReAct 中混在一起的两件事拆开了：

```text
规划推理：想清楚整个任务怎么做
执行推理：把每一步具体做完
```

这就是所谓的 **规划和执行解耦**。

---

### 2.3 Plan-and-Execute 的例子

用户说：

```text
帮我写一份某公司竞品分析报告。
```

如果用 ReAct，Agent 可能一边搜索一边决定下一步，查到什么就分析什么，容易越走越散。

如果用 Plan-and-Execute，Planner 会先生成计划：

```text
1. 明确目标公司和竞品范围；
2. 收集目标公司的产品信息；
3. 收集主要竞品的产品信息；
4. 对比功能、价格、用户群体和商业模式；
5. 分析目标公司的优势和劣势；
6. 总结市场机会和风险；
7. 生成结构化竞品分析报告。
```

然后 Executor 按计划执行每一步。

这样整体方向更稳定，不容易偏离目标。

---

### 2.4 Plan-and-Execute 的优点

Plan-and-Execute 的优点是：

1. **全局结构清晰**  
   一开始就知道任务分几步，每一步做什么。

2. **长任务不容易跑偏**  
   因为有计划兜底，执行时不会完全被中间结果牵着走。

3. **适合复杂任务**  
   多步骤、多工具、多文档、多依赖关系的任务更适合这种方式。

4. **方便并行优化**  
   如果计划中的某些步骤没有依赖关系，可以并行执行，降低总耗时。

5. **可以做模型分层**  
   规划阶段用强模型，执行阶段用便宜小模型，降低整体成本。

---

### 2.5 强模型规划，弱模型执行

Plan-and-Execute 有一个工程上的好处：

> **Planner 和 Executor 可以用不同模型。**

比如：

```text
Planner：使用 GPT-4 / Claude / 强推理模型
Executor：使用便宜模型 / 小模型 / 开源 7B 模型
```

原因是：

- 规划阶段对推理能力要求高，需要强模型；
- 执行阶段每一步任务已经被拆得很具体，可以用较便宜模型完成；
- 执行阶段通常调用次数更多，所以用小模型能明显降低成本。

这也是 Plan-and-Execute 相比纯 ReAct 的一个隐藏优势。

因为 ReAct 每一步都是同一个模型同时负责“想”和“做”，很难做这种差异化模型分配。

---

### 2.6 Plan-and-Execute 的缺点

Plan-and-Execute 的缺点是：

1. **灵活性不如 ReAct**  
   因为一开始有计划，遇到计划外情况时可能不够灵活。

2. **实现更复杂**  
   需要维护 Planner、Executor、计划解析、结果汇总等模块。

3. **规划本身有额外成本**  
   即使任务很简单，也要先规划一次，可能不划算。

4. **初始计划可能出错**  
   如果 Planner 一开始拆错方向，后续执行也会受影响。

---

### 2.7 Plan-and-Execute 适合什么场景？

Plan-and-Execute 适合：

- 深度调研；
- 竞品分析报告；
- 金融研报生成；
- 多文档分析；
- 多步骤项目任务；
- 流程长、复杂度高、容易跑偏的任务。

可以简单记：

```text
任务复杂 + 流程长 + 需要全局结构 = Plan-and-Execute
```

---

## 3. Reflection：质量增强款，生成后检查修正

### 3.1 Reflection 是什么？

**Reflection** 是反思机制。

它的核心思想是：

> **结果生成后，不直接输出，而是先让模型检查结果是否达标，如果不达标就修正。**

典型流程是：

```text
生成结果
  ↓
评估结果
  ↓
发现问题
  ↓
修改结果 / 重新执行
  ↓
再次评估
  ↓
输出最终结果
```

---

### 3.2 Reflection 的定位

Reflection 最重要的定位是：

> **它不是一套独立的完整流程，而是可以叠加在 ReAct 或 Plan-and-Execute 之上的质量增强机制。**

也就是说，它不是这样：

```text
ReAct / Plan-and-Execute / Reflection 三选一
```

而是这样：

```text
ReAct + Reflection
Plan-and-Execute + Reflection
Plan-and-Execute + ReAct + Reflection
```

ReAct 和 Plan-and-Execute 解决的是：

```text
怎么把任务做完
```

Reflection 解决的是：

```text
怎么把结果做得更好
```

---

### 3.3 Reflection 的例子

比如 Agent 生成了一份金融分析报告。

Reflection 节点可以检查：

```text
1. 是否覆盖用户要求的所有问题？
2. 是否有足够证据支撑结论？
3. 是否存在逻辑跳跃？
4. 是否遗漏风险分析？
5. 是否把短期因素和长期因素混在一起？
6. 输出结构是否清晰？
```

如果发现问题：

```text
当前报告缺少风险分析，并且对利润变化原因解释过于笼统。
```

那么系统可以让 Agent 补充：

```text
请补充风险分析，并结合已有财报证据重新解释利润变化原因。
```

---

### 3.4 Reflection 的优点

Reflection 的优点是：

1. **提升输出质量**
2. **减少幻觉**
3. **减少逻辑漏洞**
4. **减少遗漏**
5. **适合高严谨性场景**

比如：

- 代码生成；
- 正式商业报告；
- 法律文书；
- 金融分析；
- 简历优化；
- 高质量长文生成。

---

### 3.5 Reflection 的缺点

Reflection 的代价也很明显：

1. **增加 LLM 调用次数**
2. **增加 token 消耗**
3. **增加响应延迟**
4. **可能陷入反复修改**
5. **需要设置最大反思轮数**

所以 Reflection 不是越多越好。

工程中一般会设置：

```text
最多反思 2 到 3 轮
```

否则容易出现：

```text
为了改而改
```

---

### 3.6 Reflection 适合什么场景？

Reflection 适合：

- 输出要求高；
- 错误代价大；
- 需要严谨检查；
- 需要减少幻觉；
- 需要完整性检查；
- 生成内容要直接交付用户或业务方。

可以简单记：

```text
质量要求高 + 错误代价大 = 加 Reflection
```

---

## 四、三种范式的核心对比

| 维度 | ReAct | Plan-and-Execute | Reflection |
|---|---|---|---|
| 核心定位 | 基础执行范式 | 复杂任务规划范式 | 质量增强机制 |
| 解决问题 | 单步灵活决策 | 长任务不跑偏 | 输出质量不达标 |
| 核心流程 | 边想边干 | 先规划再执行 | 生成后检查修正 |
| 是否独立流程 | 是 | 是 | 不是，通常叠加使用 |
| 是否提前规划 | 否 | 是 | 不负责规划 |
| 是否调用工具 | 常用于工具调用 | 执行阶段可调用工具 | 可检查工具结果和最终输出 |
| 灵活性 | 高 | 中等 | 取决于底层范式 |
| 可控性 | 中等 | 高 | 提升质量可控性 |
| 成本 | 中等，长任务较高 | 规划有额外成本，但长任务可能更省 | 额外增加 30% 到 100% 成本 |
| 适合场景 | 简单问答、搜索、工具调用 | 深度调研、多步骤复杂任务 | 代码、报告、法律、金融等高质量任务 |

## 五、进阶机制：动态 Replan 和 Reflexion

## 1. 动态 Replan

### 1.1 动态 Replan 解决什么问题？

动态 Replan 解决的是 Plan-and-Execute 的问题：

> **计划太僵硬，中途遇到意外怎么办？**

Plan-and-Execute 一开始会生成完整计划，但现实任务中经常会出现计划外情况。

比如原计划是：

```text
1. 收集竞品 A 信息
2. 收集竞品 B 信息
3. 对比 A 和 B
4. 生成报告
```

执行到第二步时发现：

```text
竞品 B 已经被收购，产品线发生重大变化
```

这时候如果还按原计划执行，结果就会不准确。

---

### 1.2 动态 Replan 怎么做？

动态 Replan 的做法是：

```text
每执行完一步
  ↓
把当前结果和剩余计划交给 Planner
  ↓
判断原计划是否还合理
  ↓
如果不合理，生成新的剩余计划
  ↓
继续执行
```

也就是：

```text
计划 → 执行一步 → 检查计划 → 必要时重规划 → 继续执行
```

---

### 1.3 动态 Replan 的优缺点

优点：

- 保留全局规划；
- 遇到意外可以调整；
- 比固定计划更灵活；
- 能降低计划失效带来的风险。

缺点：

- 每次 Replan 都要额外调用 LLM；
- token 成本增加；
- 系统复杂度上升。

---

## 2. Reflexion

### 2.1 Reflexion 是什么？

**Reflexion** 是 Reflection 的增强版。

普通 Reflection 是：

```text
做完检查，发现问题就改
```

Reflexion 是：

```text
做完检查，发现问题后，不只是改，还把失败原因总结成经验，存进记忆里，下次避免再犯
```

它像一个“错题本”。

---

### 2.2 Reflexion 的流程

典型流程是：

```text
执行任务
  ↓
检查结果
  ↓
发现失败
  ↓
总结失败原因
  ↓
写入记忆
  ↓
下一次执行时读取相关经验
  ↓
避免重复错误
```

比如代码生成任务中，Agent 第一次生成代码后测试失败。

Reflexion 会总结：

```text
失败原因：没有处理空数组边界情况。
改进建议：下次生成代码前先检查输入为空、长度为 1、重复元素等边界条件。
```

下一次再做类似题目时，这条经验会作为上下文输入给模型。

---

### 2.3 Reflection 和 Reflexion 的区别

| 维度 | Reflection | Reflexion |
|---|---|---|
| 核心动作 | 检查并修正当前结果 | 总结失败经验并写入记忆 |
| 作用范围 | 当前任务 | 当前任务 + 后续任务 |
| 是否需要记忆模块 | 不一定 | 通常需要 |
| 类比 | 做完检查一遍 | 写错题本 |
| 适合场景 | 提升当前输出质量 | 跨任务积累经验 |

---

## 六、Token 消耗与成本对比

### 1. ReAct 的 token 消耗

ReAct 的问题是：

> **每一轮调用 LLM 都要带上完整历史。**

假设一个任务需要 5 步，每步平均产生 2000 token。

那么 ReAct 的输入可能是：

```text
第 1 步：2000 token
第 2 步：4000 token
第 3 步：6000 token
第 4 步：8000 token
第 5 步：10000 token
```

总输入约为：

```text
2000 + 4000 + 6000 + 8000 + 10000 = 30000 token
```

所以 ReAct 在长任务中 token 消耗会越来越高。

---

### 2. Plan-and-Execute 的 token 消耗

Plan-and-Execute 的消耗主要在：

```text
规划阶段 + 每步执行阶段 + 汇总阶段
```

它不一定每一步都带完整历史，而是可以只带：

```text
当前步骤指令 + 前面结果摘要
```

所以长任务中可能比纯 ReAct 更省。

例如：

```text
规划阶段：3000 token
执行阶段：每步 1500 token，5 步共 7500 token
汇总阶段：4000 token
总计：约 14500 token
```

相比 ReAct 的 30000 token，可能节省一半左右。

---

### 3. Reflection 的 token 消耗

Reflection 会在原有基础上增加额外调用。

如果每个关键节点都反思一次，就意味着：

```text
至少多一次 LLM 调用
```

如果反思不通过还要重做，成本会继续增加。

所以 Reflection 的成本可能增加：

```text
30% 到 100% 甚至更多
```

因此 Reflection 要控制使用位置和轮数。

## 七、实际项目选型指南

## 1. 简单任务：用 ReAct

如果任务满足：

- 不复杂；
- 流程不固定；
- 需要实时调整；
- 主要是搜索、查库、调用工具；
- 步骤不会太长；

就用 ReAct。

例如：

```text
帮我查一下某家公司最近一条新闻，并总结影响。
```

选择：

```text
ReAct
```

原因：

```text
步骤少，实时工具调用为主，不需要复杂规划。
```

---

## 2. 复杂长任务：用 Plan-and-Execute

如果任务满足：

- 步骤很多；
- 容易跑偏；
- 需要整体结构；
- 有多个子任务；
- 需要多工具、多文档协同；

就用 Plan-and-Execute。

例如：

```text
帮我生成一份某公司的完整竞品分析报告。
```

选择：

```text
Plan-and-Execute
```

原因：

```text
任务复杂，需要先拆解整体计划，再逐步执行。
```

---

## 3. 执行中经常遇到意外：加动态 Replan

如果任务满足：

- 计划可能中途失效；
- 外部环境变化大；
- 执行结果可能改变后续步骤；
- 需要根据中间结果调整计划；

就在 Plan-and-Execute 基础上加动态 Replan。

例如：

```text
调研一个快速变化的行业，并生成趋势报告。
```

选择：

```text
Plan-and-Execute + Dynamic Replan
```

原因：

```text
先规划，但执行中要根据新信息调整计划。
```

---

## 4. 输出要求高：加 Reflection

如果任务满足：

- 不能明显出错；
- 需要严谨性；
- 需要完整性检查；
- 需要减少幻觉；
- 结果会直接交付；

就在基础范式上加 Reflection。

例如：

```text
生成一份可直接给业务方看的金融分析报告。
```

选择：

```text
Plan-and-Execute + Reflection
```

原因：

```text
既要结构清晰，又要结果质量高。
```

---

## 5. 需要跨任务积累经验：用 Reflexion

如果任务满足：

- 同类任务会反复出现；
- 失败经验有复用价值；
- 希望 Agent 下次少犯同类错误；
- 有记忆模块支持；

就用 Reflexion。

例如：

```text
代码生成 Agent 反复解决编程题，并根据测试失败经验改进。
```

选择：

```text
Reflection + Memory = Reflexion
```

原因：

```text
失败教训可以沉淀为长期经验。
```

---

## 八、选型口诀

可以直接背这个：

```text
简单任务用 ReAct；
复杂长任务用 Plan-and-Execute；
计划容易变化就加 Dynamic Replan；
输出要求高就加 Reflection；
需要积累经验就用 Reflexion；
工程上不要过度设计，够用就好。
```

## 九、实际生产中的混合架构

很多生产级 Agent 系统不是只用一种范式，而是组合使用。

典型结构是：

```text
用户输入
  ↓
Workflow：任务分类和边界控制
  ↓
Planner：生成全局计划
  ↓
Executor：按计划执行
      ↓
      每一步内部用 ReAct 调用工具
  ↓
Reflection：检查最终结果
  ↓
Workflow：格式化输出
```

也就是：

```text
Workflow 控制主流程
Plan-and-Execute 控制全局任务拆解
ReAct 控制单步工具调用
Reflection 控制输出质量
Tools 提供外部能力
```

这种架构适合复杂项目，比如：

- 金融分析 Agent；
- 企业知识库 Agent；
- 数据分析 Agent；
- 代码生成 Agent；
- 多文档报告生成 Agent。

---

## 十、结合金融分析 Agent 项目怎么回答？

如果结合我的金融分析 Agent 项目，我会这样说：

在金融分析 Agent 里，我不会一上来把所有机制都堆上去，而是根据任务复杂度选型。

如果用户只是问：

```text
这份财报里收入是多少？
```

这种问题比较简单，可以直接用 RAG 检索或者轻量 ReAct。

如果用户问：

```text
结合腾讯最近几个季度财报、相关新闻和历史研究记忆，分析它的业务趋势、增长原因和潜在风险。
```

这就是复杂长任务，更适合用 **Plan-and-Execute**。

Planner 可以先拆解任务：

```text
1. 识别公司和时间范围；
2. 检索相关季度财报；
3. 提取收入、利润、毛利率、费用等指标；
4. 对比多季度变化趋势；
5. 检索相关新闻；
6. 查询 Research Memory 中的历史研究结论；
7. 分析增长原因；
8. 总结风险；
9. 生成结构化研报。
```

Executor 执行每一步时，可以用 ReAct 调用工具：

```text
RAG 检索工具
Milvus 向量库
PDF 解析工具
新闻搜索工具
股票行情工具
Research Memory 查询工具
```

最后因为金融报告对准确性和证据要求比较高，可以再加一层 Reflection，检查：

```text
1. 是否引用了足够证据；
2. 是否覆盖收入、利润、费用、风险；
3. 结论是否和证据匹配；
4. 是否存在幻觉；
5. 输出结构是否符合研报格式。
```

这样设计的好处是：

```text
Plan-and-Execute 保证整体分析不跑偏；
ReAct 保证每一步可以灵活调用工具；
Reflection 保证最终报告质量；
Workflow 保证主链路稳定可控。
```

---

## 十一、面试官可能追问：Reflection 为什么不是独立流程？

可以这样回答：

Reflection 不是独立流程，因为它本身不负责完成任务的主链路。

ReAct 和 Plan-and-Execute 都是在回答：

```text
任务应该怎么一步步完成？
```

而 Reflection 回答的是：

```text
任务完成后，结果是否足够好？
```

所以 Reflection 更像是一个质量检查模块，可以挂在 ReAct 后面，也可以挂在 Plan-and-Execute 后面。

比如：

```text
ReAct + Reflection
Plan-and-Execute + Reflection
Plan-and-Execute + ReAct + Reflection
```

它不改变主流程的基本执行方式，只是在生成结果后加入评估和修正环节。

所以我不会把 Reflection 看成和 ReAct、Plan-and-Execute 完全同级的独立 Agent 架构，而是把它看成质量增强机制。

---

## 十二、面试官可能追问：ReAct 和 Plan-and-Execute 怎么选？

可以这样回答：

我会根据任务复杂度和流程长度来选。

如果任务比较简单，主要是搜索、查库、调用工具，而且步骤不多，我会用 ReAct。因为 ReAct 实现简单、灵活性高，足够解决轻量任务。

如果任务比较复杂，需要先拆解多个步骤，而且步骤之间有明显依赖关系，我会用 Plan-and-Execute。因为它先做全局规划，再逐步执行，不容易像 ReAct 一样走着走着偏离目标。

简单说：

```text
步骤少、流程不固定：ReAct
步骤多、容易跑偏：Plan-and-Execute
```

---

## 十三、面试官可能追问：为什么 Plan-and-Execute 可能比 ReAct 更省成本？

可以这样回答：

表面上看，Plan-and-Execute 多了一次规划调用，似乎更贵。

但在长任务中，它反而可能更省。

因为 ReAct 每一步通常都要带上完整历史，随着步骤增加，上下文越来越长，token 消耗会不断累积。

而 Plan-and-Execute 可以把任务先拆成多个子步骤，每一步执行时只带当前步骤指令和前面结果摘要，不一定要带完整推理历史，所以长任务中 token 消耗可能更低。

另外，Plan-and-Execute 可以做模型分层：

```text
强模型负责规划
弱模型负责执行
```

规划只调用少数几次，执行调用很多次。如果执行阶段使用便宜小模型，总体成本会明显下降。

所以对于复杂长任务，Plan-and-Execute 不一定比 ReAct 贵，甚至可能更省。

---

## 十四、面试官可能追问：Reflection 的代价是什么？怎么控制？

可以这样回答：

Reflection 的主要代价是增加 LLM 调用次数，因此会增加 token 消耗和响应延迟。

如果每个步骤都做 Reflection，系统会变得很慢，成本也会明显上升。而且如果没有限制，模型可能陷入反复修改，出现“为了改而改”的情况。

所以工程上要控制 Reflection 的使用：

```text
1. 只在关键节点或最终输出前使用；
2. 设置最大反思轮数，比如 2 到 3 轮；
3. 使用明确的检查标准；
4. 简单任务不加 Reflection；
5. 只有高质量、高风险任务才加 Reflection。
```

也就是说，Reflection 是质量增强工具，不是默认所有地方都要加。

---

## 十五、面试官可能追问：动态 Replan 和 Reflection 有什么区别？

可以这样回答：

动态 Replan 和 Reflection 都是在执行过程中做调整，但它们关注的问题不同。

**动态 Replan 关注的是计划是否还合理。**

它通常出现在 Plan-and-Execute 中，每执行完一步，就根据当前结果判断剩余计划要不要调整。

它解决的是：

```text
原计划失效了怎么办？
```

**Reflection 关注的是输出结果是否达标。**

它通常在某一步结果生成后或者最终答案生成后检查质量。

它解决的是：

```text
结果质量不够好怎么办？
```

所以区别是：

```text
动态 Replan 改的是计划；
Reflection 改的是结果。
```

---

## 十六、面试官可能追问：Reflexion 和 Reflection 有什么区别？

可以这样回答：

Reflection 是对当前结果做自我检查和修正。

Reflexion 则是在 Reflection 基础上增加了记忆机制。

普通 Reflection 只解决当前任务：

```text
这次结果不好，那就改。
```

Reflexion 还会总结失败原因：

```text
这次为什么错？
下次怎么避免？
```

然后把这段经验写入记忆，下次遇到类似任务时作为上下文提供给模型。

所以可以理解为：

```text
Reflection 是检查；
Reflexion 是检查 + 总结经验 + 写入记忆。
```

它特别适合代码生成这类有明确反馈信号的任务，因为代码可以运行测试，失败原因比较容易总结。

## 十七、最容易踩的坑

## 1. 把 Reflection 当成独立主流程

错误说法：

```text
ReAct、Plan-and-Execute、Reflection 是三种并列的 Agent 流程。
```

更准确说法：

```text
ReAct 和 Plan-and-Execute 是主流程范式；
Reflection 是可以叠加在它们上面的质量增强机制。
```

---

## 2. 以为 Plan-and-Execute 一定比 ReAct 高级

错误理解：

```text
Plan-and-Execute 比 ReAct 高级，所以都应该用 Plan-and-Execute。
```

更准确理解：

```text
简单任务用 ReAct 就够了；
复杂任务才需要 Plan-and-Execute；
选型看任务，不是看概念高级不高级。
```

---

## 3. 什么机制都往系统里堆

错误做法：

```text
ReAct + Plan-and-Execute + Reflection + Dynamic Replan + Reflexion 全部加上。
```

问题是：

```text
系统复杂
响应变慢
token 成本上升
调试困难
bug 更多
```

正确做法：

```text
先用最简单方案跑通，再根据真实问题逐步加机制。
```

## 十八、最终背诵版

ReAct、Plan-and-Execute、Reflection 这三种范式解决的是 Agent 系统中不同层次的问题。

**ReAct** 是基础的单步迭代范式，核心流程是 Thought、Action、Observation 循环。它的特点是边想边干，走一步看一步，每一步根据工具返回结果实时调整。它实现简单、灵活性高，适合信息搜索、简单问答、客服机器人、工具调用助手等复杂度不高、流程不固定的任务。但它缺少全局规划，长任务中容易跑偏，也容易出现无效循环和错误传播。

**Plan-and-Execute** 是复杂任务规划范式，核心是先规划再执行。它把规划和执行解耦，先由 Planner 生成完整计划，再由 Executor 按计划执行。相比 ReAct，它全局结构更清楚，适合长流程、多步骤、容易跑偏的任务，比如竞品分析、深度调研、多文档分析、金融研报生成。它的缺点是实现更复杂，简单任务上会有额外规划成本，遇到计划外情况时灵活性不如 ReAct。

**Reflection** 不是独立的完整流程，而是可以叠加在 ReAct 或 Plan-and-Execute 上的质量增强机制。它在结果生成后增加自我检查、自我评估和自我修正，用来减少幻觉、逻辑漏洞和细节遗漏。它适合代码生成、正式报告、金融分析、法律文书等高质量、高风险场景。代价是增加 LLM 调用次数、token 消耗和响应延迟，所以要控制反思轮数，不能滥用。

实际选型时，我会看三个维度：任务复杂度、流程确定性和输出质量要求。任务简单、流程不固定时用 ReAct；任务复杂、流程长、容易跑偏时用 Plan-and-Execute；输出要求高时在前两者基础上加 Reflection。如果执行过程中经常遇到计划外情况，可以加动态 Replan；如果需要跨任务沉淀失败经验，可以用 Reflexion。

工程上最重要的原则是：**先用简单方案跑通，再根据真实问题逐步增强，不要过度工程化。**

---

## 十九、一句话总结

> **ReAct 解决单步灵活执行，Plan-and-Execute 解决复杂任务规划，Reflection 解决输出质量检查；简单任务用 ReAct，复杂长任务用 Plan-and-Execute，高质量场景再加 Reflection，工程上够用就好。**
