
## 一、面试标准回答

如果面试官问：

> 你实际项目里是怎么做 Agent 架构选型的？什么时候用 Single-Agent，什么时候用 Multi-Agent？

可以这样回答：

我不会简单地用“任务复杂就上 Multi-Agent”来判断，因为“复杂”这个词太模糊，而且 Multi-Agent 本身也会带来额外的调度成本、通信成本、调试成本和 token 成本。

我的判断逻辑一般是两步。

第一步，先判断 **Single-Agent 能不能解决问题**。如果任务流程比较清晰、步骤数量适中、没有明显的专业角色拆分，也不需要并行执行，那么 Single-Agent 就够了。Single-Agent 的优势是架构简单、链路短、可控性高、调试方便，工程上更容易落地。

第二步，如果 Single-Agent 确实遇到瓶颈，再考虑 Multi-Agent。真正适合 Multi-Agent 的场景主要有三类：

1. **Context 快撑爆了**  
   单个 Agent 需要维护太多历史对话、工具结果、文档内容和中间状态，导致上下文变脏、变长，甚至遗忘早期信息。

2. **需要明显的专业分工**  
   比如一个任务同时需要搜索、分析、写作、代码实现、测试、事实核查等能力，如果都塞给一个 Agent，它就会身兼多职，输出质量不稳定。

3. **存在可以并行的独立子任务**  
   比如多篇文档分析、多家公司竞品调研、多个季度财报对比，这些任务可以拆给多个 Worker Agent 并行执行，最后统一汇总。

如果确定要用 Multi-Agent，我在工程里一般优先选择 **中心化 Orchestrator 架构**，也就是由一个总调度 Agent 负责任务拆分、任务分发、结果收集和最终汇总。相比去中心化 Agent 之间自由通信，中心化方案更可控、更好调试，也更适合生产环境。

所以我的整体选型原则是：

> 先用 Single-Agent 把系统跑起来，只有当出现 context 压力、专业分工需求或并行执行需求时，再渐进式拆出 Worker Agent，演进成中心化 Multi-Agent，而不是一开始就设计一个复杂的多 Agent 系统。
