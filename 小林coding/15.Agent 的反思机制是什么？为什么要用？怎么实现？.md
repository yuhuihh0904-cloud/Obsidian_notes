# Agent 的反思机制是什么？为什么要用？怎么实现？

## 一、面试标准回答

如果面试官问：

> Agent 的反思机制你了解吗？为什么要用反思？具体怎么实现？

可以这样回答：

我理解的 **Agent 反思机制**，不是简单地让模型“不满意就重新生成一次”，而是一个有结构的质量改进闭环。

它的核心流程是：

```text
生成 → 评估 → 改进
```

也就是 Agent 先生成一个初始结果，然后再让模型或另一个 Critic Agent 按照明确标准检查这个结果。如果检查发现问题，就把评估意见和原始任务一起传回去，让 Agent 有针对性地修改；如果检查结果已经足够好，就直接停止。

反思机制要解决的问题是：LLM 第一次生成的结果不一定可靠。它可能出现逻辑跳跃、遗漏要求、事实错误、表达含糊等问题。反思机制相当于给 Agent 加了一轮“自我检查”或“代码 Review”，让它在输出前先发现问题、修正问题，从而提升最终结果质量。

实现上一般需要两个 prompt。

第一个是 **评估 prompt**，让模型扮演检查者，按明确维度检查当前输出，比如：

```text
1. 是否有事实错误？
2. 是否有逻辑漏洞？
3. 是否遗漏用户要求？
4. 表达是否清晰？
```

这里有两个关键点：

第一，评估 prompt 必须给出明确检查维度，不能只写“请检查一下有没有问题”。如果不给维度，模型往往会泛泛地说“整体不错”，很难发现具体问题。

第二，必须设计 **PASS 机制**。也就是说，如果当前输出已经足够好，模型应该回复 `PASS`。这个机制很重要，因为如果不给模型一个停止出口，它可能会为了反思而反思，不断挑小毛病，甚至把原本正确的内容改错。

第二个是 **改进 prompt**，它需要同时传入三部分内容：

```text
原始任务
当前输出
评估意见
```

然后让模型根据评估意见定向修改。这里不能只给评估意见，也不能只给当前输出，否则模型很容易改偏。只有原始任务、当前输出和评估意见都在，模型才能知道“目标是什么、现在结果是什么、应该改哪里”。

工程上，反思可以分成两种粒度：

第一种是 **步骤级反思**，也就是每个工具调用或每个中间步骤后都检查一次。它适合步骤之间强依赖的任务，因为前一步错了，后面可能全错。优点是错误能早发现、早修正；缺点是每一步都要多一次或多几次 LLM 调用，成本和延迟很高。

第二种是 **任务级反思**，也就是整个任务完成后再做一次整体检查。它适合报告生成、总结、写作、分析类任务。优点是成本较低，而且能从整体视角发现前后矛盾、结构不清晰等问题；缺点是如果中间某一步很早就错了，可能到最后才发现，前面的执行已经浪费了。

在工程实践里，反思不是每一步都要开，而是应该用在关键节点，比如最终报告生成、复杂推理、代码生成后的检查、事实核查等质量要求高、错误代价大的地方。对于简单问答、格式转换、实时性要求高的场景，不一定值得加反思。

最后一定要设置最大反思轮次，一般 2 到 3 轮就够了，不能完全依赖模型自己判断什么时候停止。因为模型可能会陷入“为了改而改”的循环，每次都觉得还有优化空间，导致成本和延迟失控。

所以我会把 Agent 反思机制总结成一句话：

> Reflection 不是随机重试，而是通过明确评估标准、PASS 停止机制和定向改进 prompt，构建“生成 → 评估 → 改进”的闭环，在关键节点提升 Agent 输出质量。

## 二、文章核心观点

这篇文章的核心观点是：

> Agent 反思机制不是“重新生成一次”，而是一个结构化闭环：先生成结果，再按标准评估，最后根据评估意见定向改进。

最重要的区别是：

```text
随机重试：
重新生成一次，结果好不好靠运气。

反思机制：
先指出问题，再根据问题修改，改进方向明确。
```

所以反思机制的重点不是“多试几次”，而是：

```text
评估标准是否明确；
是否有 PASS 停止机制；
改进是否基于原始任务、当前输出和评估意见。
```

## 三、关键知识点

### 1. 反思机制的基本闭环

Reflection 的基本流程是：

```text
初始生成
↓
评估输出
↓
如果 PASS，直接结束
↓
如果不 PASS，根据评估意见改进
↓
再次评估
↓
直到 PASS 或达到最大轮次
```

可以类比为：

```text
写草稿 → 老师批改 → 根据批注修改 → 再检查
```

---

### 2. 评估 prompt 的两个关键设计

评估 prompt 不能只写：

```text
请检查一下这个输出有没有问题。
```

这样太泛，模型容易流于表面。

更好的评估 prompt 应该包含明确维度：

```text
请从以下维度评估：
1. 事实是否准确
2. 逻辑是否完整
3. 是否遗漏用户要求
4. 表达是否清晰
```

并且必须有：

```text
如果已经足够好，请回复 PASS。
```

`PASS` 机制的作用是：

```text
让模型知道什么时候该停；
避免为了反思而反思；
避免把正确内容越改越乱。
```

---

### 3. 改进 prompt 必须包含三样东西

改进时要同时给模型：

```text
1. 原始任务
2. 当前输出
3. 评估意见
```

原因是：

```text
只有原始任务：
不知道基于什么内容修改。

只有当前输出：
不知道哪里有问题。

只有评估意见：
容易偏离原始目标。
```

所以改进 prompt 的核心是：

```text
根据原始任务和评估意见，对当前输出做定向修改。
```

---

### 4. 步骤级反思 vs 任务级反思

| 类型 | 触发时机 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **步骤级反思** | 每个步骤后检查 | 错误早发现、早纠正 | 成本高、延迟大 | 步骤强依赖任务 |
| **任务级反思** | 整个任务完成后检查 | 成本较低，能看整体质量 | 中间错误发现较晚 | 报告、总结、写作、分析 |

简单记：

```text
步骤级反思：
适合“前一步错，后面全错”的任务。

任务级反思：
适合“最终整体质量更重要”的任务。
```

---

### 5. 多 Agent 互评

除了自己检查自己，还可以让一个独立的 **Critic Agent** 来评审输出。

流程是：

```text
Executor Agent 生成结果
↓
Critic Agent 审查结果
↓
Executor Agent 根据批注修改
↓
Critic Agent 再次确认
```

为什么多 Agent 互评通常更好？

因为同一个模型检查自己的输出时，容易沿用自己生成时的逻辑，产生“自洽偏见”。  
独立的 Critic Agent 职责就是找问题，视角更像代码 Review，更容易发现漏洞。

适合场景：

```text
代码生成后的测试检查；
重要报告的事实核查；
高质量内容生成；
错误代价较高的任务。
```

但它的代价也更高：

```text
多一个 Agent；
多一次或多次 LLM 调用；
系统链路更复杂。
```

---

### 6. Reflexion 和 LATS 简单了解即可

文章还提到两个进阶方向：

```text
Reflexion：
把反思得到的失败经验存入记忆，下次遇到类似任务时参考，避免重复犯错。

LATS：
把反思和树搜索结合，通过多路径探索 + 反思反馈来优化后续搜索。
```

这两个方向更偏学术或高级 Agent 设计，面试时知道它们的核心思想即可，不需要展开太细。

## 四、工程权衡：怎么用才合理？

反思机制有用，但不是免费的。

每一轮反思通常至少包含：

```text
一次评估调用
一次改进调用
```

所以如果做 3 轮反思，除了初始生成外，可能额外增加：

```text
3 次评估 + 3 次改进 = 6 次 LLM 调用
```

这会显著增加：

```text
token 成本；
响应延迟；
系统复杂度。
```

因此工程上更合理的做法是：

```text
不要每一步都反思；
只在关键节点启用；
设置最大轮次；
有 PASS 停止机制；
必要时才用 Critic Agent。
```

适合开反思的场景：

```text
最终报告生成；
复杂推理结果；
代码生成后的检查；
事实核查；
重要决策分析；
用户要求质量很高的内容。
```

不适合开反思的场景：

```text
简单问答；
格式转换；
低价值文本处理；
实时性要求很高的任务；
错误代价很低的任务。
```

## 五、面试官可能追问

### 追问 1：反思和重新生成有什么区别？

重新生成是随机重试，模型没有明确改进方向。

反思是结构化改进，先评估当前输出的问题，再根据评估意见定向修改。

区别可以概括为：

```text
重新生成：再来一次。
反思机制：先指出哪里错，再针对性修改。
```

---

### 追问 2：为什么评估 prompt 里一定要有 PASS？

因为没有 PASS，模型可能会为了反思而反思。

它每一轮都能找到一些小问题，然后不断修改，导致：

```text
成本增加；
延迟增加；
内容越改越乱；
甚至把原本正确的内容改错。
```

PASS 相当于给模型一个明确出口：

```text
如果已经够好，就停止。
```

---

### 追问 3：步骤级反思和任务级反思怎么选？

如果任务步骤之间强依赖，前一步错了后面会全错，就适合步骤级反思。

如果任务更关注最终整体质量，比如写报告、总结、润色，就适合任务级反思。

简单说：

```text
怕中间错误扩散，用步骤级反思；
怕最终质量不够，用任务级反思。
```

---

### 追问 4：为什么多 Agent 互评比自我反思更好？

因为自我反思时，生成者和评估者是同一个模型，容易对自己的逻辑产生自洽偏见。

独立 Critic Agent 的角色就是找问题，它没有生成阶段的心理包袱，更像代码 Review 中的审查者，所以更容易发现漏洞。

---

### 追问 5：反思机制最大的工程风险是什么？

最大的风险是成本和死循环。

每轮反思都会增加 LLM 调用，如果没有最大轮次限制，模型可能不断提出小修改，导致系统一直循环。

所以工程上必须设置：

```text
最大反思轮次；
最大 token 预算；
最大延迟限制；
PASS 停止机制。
```

通常最大反思轮次设置为：

```text
2 到 3 轮。
```

## 六、容易踩的坑

### 坑 1：把反思理解成重新生成

错误说法：

```text
反思就是输出不好就重新生成一次。
```

正确理解：

```text
反思是生成、评估、改进的结构化闭环，不是随机重试。
```

---

### 坑 2：评估 prompt 太泛

错误写法：

```text
请检查一下有没有问题。
```

正确写法：

```text
请从事实准确性、逻辑完整性、任务覆盖度、表达清晰度等维度检查。
```

---

### 坑 3：没有 PASS 机制

没有 PASS，模型容易无限挑毛病。

正确做法：

```text
如果输出已经足够好，请回复 PASS。
```

---

### 坑 4：每一步都加反思

反思不是越多越好。

每一步都反思会导致：

```text
调用次数暴增；
延迟变高；
成本变高；
系统变复杂。
```

正确做法：

```text
只在关键节点启用反思。
```

---

### 坑 5：不设置最大轮次

错误做法：

```text
让模型自己判断什么时候停止。
```

正确做法：

```text
硬性设置最大反思轮次，一般 2 到 3 轮。
```

## 七、记忆版总结

这篇文章可以压缩成一句话：

> Agent 反思机制不是重新生成，而是通过“生成 → 评估 → 改进”的闭环，让模型按照明确检查维度发现问题，并根据评估意见定向修改，从而提升关键节点的输出质量。

再进一步记：

```text
Reflection 核心流程：
生成 → 评估 → 改进

评估 prompt 关键点：
1. 要有明确检查维度
2. 要有 PASS 停止机制

改进 prompt 关键点：
必须包含：
1. 原始任务
2. 当前输出
3. 评估意见

两种粒度：
1. 步骤级反思：错误早发现，但成本高
2. 任务级反思：成本低，适合整体质量检查

多 Agent 互评：
独立 Critic Agent 比自我检查更客观，但成本更高

工程原则：
1. 不要每步都反思
2. 只在关键节点用
3. 最大轮次通常设 2 到 3 轮
4. 必须防止死循环
```

面试最重要的是记住：

```text
1. 反思不是随机重试，而是“生成 → 评估 → 改进”。
2. 评估 prompt 必须有明确检查维度和 PASS 机制。
3. 改进 prompt 必须同时包含原始任务、当前输出和评估意见。
4. 步骤级反思和任务级反思要按场景选择。
5. 工程上要控制成本和轮次，反思只适合关键节点。
```
