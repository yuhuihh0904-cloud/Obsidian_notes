# Ch2: CUDA 与数据并行计算简介

  

## 核心概念

  

| 概念 | 英文 | 说明 |

|------|------|------|

| 线程 | Thread | 最小执行单位，拥有 PC、寄存器、上下文 |

| 线程块 | Block | 一组可协作的线程（共享内存 + 同步），最大 1024 线程 |

| 网格 | Grid | 内核函数执行时的顶层线程组织结构，包含若干 Block |

| 内核 | Kernel | GPU 上执行的函数，用 `__global__` 声明，返回 void |

| SPMD | Single Program, Multiple Data | 所有线程执行同一段代码，但处理不同的数据 |

  

## 内置变量

  

| 变量 | 类型 | 含义 |

|------|------|------|

| `threadIdx.x/y/z` | `dim3` | 线程在 Block 内的索引（0 到 blockDim-1） |

| `blockIdx.x/y/z` | `dim3` | Block 在 Grid 内的索引（0 到 gridDim-1） |

| `blockDim.x/y/z` | `dim3` | Block 的维度（每维线程数） |

| `gridDim.x/y/z` | `dim3` | Grid 的维度（每维 Block 数） |

  

## 经典例子：向量加法

  

```cuda

__global__ void vecAddKernel(float* A_d, float* B_d, float* C_d, int N) {

int i = blockIdx.x * blockDim.x + threadIdx.x;

if (i < N) C_d[i] = A_d[i] + B_d[i];

}

  

// 调用方式

dim3 DimGrid(ceil(N/256.0), 1, 1);

dim3 DimBlock(256, 1, 1);

vecAddKernel<<<DimGrid, DimBlock>>>(A_d, B_d, C_d, N);

```

  

> **要点**：`i = blockIdx.x * blockDim.x + threadIdx.x` 是全局一维索引的标准公式

  

## 主机端内存管理 API

  

```cuda

// 分配

cudaMalloc((void**)&A_d, size);

// 主机→设备传输

cudaMemcpy(A_d, A, size, cudaMemcpyHostToDevice);

// 设备→主机传输

cudaMemcpy(C, C_d, size, cudaMemcpyDeviceToHost);

// 释放

cudaFree(A_d);

```

  

### 粗粒度线程分配

  

一个线程也可处理多个元素以减少线程总数：

  

```cuda

// 每个线程处理 2 个元素

vecAdd<<<ceil(N/(2*256.0)), 256>>>(...)

int i = blockIdx.x * (2*blockDim.x) + threadIdx.x;

if (i < N) C[i] = A[i] + B[i];

i = i + blockDim.x;

if (i < N) C[i] = A[i] + B[i];

```

  

| 模式 | Grid 维度 | 线程总数 |

|------|----------|---------|

| 1元素/线程 | ceil(N/256) | ceil(N/256)×256 |

| 2元素/线程 | ceil(N/512) | ceil(N/512)×256 |

  

> 粗粒度可以减少线程数，但增加了每个线程的工作量

  

### Kernel 调用是异步的

  

```cuda

// 使用 dim3 显式指定维度

dim3 DimGrid(N/256, 1, 1);

if (0 != (N % 256)) DimGrid.x++;

dim3 DimBlock(256, 1, 1);

vecAddKernel<<<DimGrid, DimBlock>>>(A_d, B_d, C_d, N);

```

  

**注意**：Kernel 调用从 CUDA 1.0 开始就是**异步**的！如需阻塞等待完成，需要显式调用 `cudaDeviceSynchronize()`

  

### 编译流程

  

NVCC 编译器 → PTX 中间代码 → 设备即时编译器 (JIT) → 机器码

  

## 函数限定符

  

| 限定符 | 执行位置 | 可调用者 | 返回值 |

|--------|---------|---------|--------|

| `__global__` | 设备 (GPU) | 主机 (CPU) | 必须 `void` |

# Ch3: 基于 Kernel 的数据并行执行模型

  

## Warp 调度机制

  

- **Warp** = 32 个线程，是 SM 调度和执行的**基本单位**

- 一个 Block 包含 `ceil(threads_per_block / 32)` 个 warp

- SM 以 warp 为单位进行**零开销切换调度**（zero-overhead warp scheduling）

- 同一 warp 中所有线程**执行同一条指令**

- 一个 warp 中的线程必须**步调一致**共同前进

  

## 控制分歧 (Control/Branch Divergence)

  

- 当 warp 中的线程走不同分支时发生

- GPU 使用**谓词执行**（predicated execution）：每个线程为每个路径计算一个真/假值

- 不同路径在 warp 内**串行执行**

  

### 避免分歧的技巧

  

让分支粒度对齐 warp 大小（32 的倍数）：

  

```cuda

// 有分歧——warp 内线程走不同路径

if (threadIdx.x > 2) { /* THEN */ } else { /* ELSE */ }

  

// 无分歧——整个 warp 走同一路径

if (threadIdx.x / WARP_SIZE > 2) { /* THEN */ } else { /* ELSE */ }

```

  

#### GPU 硬件资源对比（Compute Capabilities）

  

| 特性 | Kepler (GK107) | Maxwell (GM107) |

|------|---------------|-----------------|

| Shared Memory / SM | 16/48 KB | 64 KB |

| Register File / SM | 256 KB | 256 KB |

| Active Blocks / SM | 16 | 32 |

| Max Threads / SM | 2048 | 2048 |

  

> 不同 GPU 的 compute capability 不同，应使用 `cudaGetDeviceProperties()` 查询实际参数

  

### Block 粒度选择

  

假设 SM 最大 1,536 线程、最多 8 个 block：

  

| Block 大小 | 线程/Block | 可容纳 Block | 实际线程 | 利用率 |

|-----------|-----------|-------------|---------|-------|

| 8×8 (64) | 64 | 上限 8 | 512 | ❌低 |

| 16×16 (256) | 256 | 6 (符合上限) | 1536 | ✅满 |

| 32×32 (1024) | 1024 | 上限 1 | 1024 | ❌仅 2/3 |

  

> 关键：选择 block 大小需同时考虑 max threads/SM 和 max blocks/SM 的约束

  

## 2D Grid 处理图片示例

  

### 彩色转灰度

  

```cuda

__global__ void colorToGreyscaleConversion(unsigned char *grayImage,

unsigned char *rgbImage, int width, int height) {

int Col = threadIdx.x + blockIdx.x * blockDim.x;

int Row = threadIdx.y + blockIdx.y * blockDim.y;

if (Col < width && Row < height) {

int greyOffset = Row * width + Col;

int rgbOffset = 3 * greyOffset; // RGB 三通道

unsigned char r = rgbImage[rgbOffset];

unsigned char g = rgbImage[rgbOffset + 1];

unsigned char b = rgbImage[rgbOffset + 2];

grayImage[greyOffset] = 0.21f*r + 0.71f*g + 0.07f*b;

}

}

```

  

### 图像模糊 (边界处理)

  

```cuda

__global__ void blurKernel(unsigned char *in, unsigned char *out, int w, int h) {

int Col = blockIdx.x * blockDim.x + threadIdx.x;

int Row = blockIdx.y * blockDim.y + threadIdx.y;

if (Col < w && Row < h) {

int pixVal = 0, pixels = 0;

for (int blurRow = -BLUR_SIZE; blurRow <= BLUR_SIZE; ++blurRow) {

for (int blurCol = -BLUR_SIZE; blurCol <= BLUR_SIZE; ++blurCol) {

int curRow = Row + blurRow;

int curCol = Col + blurCol;

if (curRow > -1 && curRow < h && curCol > -1 && curCol < w) {

pixVal += in[curRow * w + curCol];

pixels++;

}

}

}

out[Row * w + Col] = (unsigned char)(pixVal / pixels);

}

}

```

  

## 浮点数非结合性问题

  

- IEEE 单精度浮点数**不满足结合律**

- 例如：`2⁻³⁰ + (1 - 1) ≠ (2⁻³⁰ + 1) - 1`

- 原因：单精度尾数只有 23 位，`2⁻³⁰ + 1` 中 `2⁻³⁰` 被截断丢失

- 并行归约（reduction）时需要注意，结果可能因计算顺序而不同

# Ch4: 内存模型

  

## CUDA 内存层次结构

  

| 内存类型 | 位置 | 作用域 | 生命周期 | 延迟 | 大小 |

|---------|------|-------|---------|------|------|

| 寄存器 Register | 片上 | 单线程 | 线程内 | ~1 cycle | 几十 KB/线程 |

| 共享内存 Shared Memory | 片上 | Block | Block 内 | ~5 cycles | 48-96 KB/SM |

| 全局内存 Global Memory | 片外 DRAM | Grid | 整个程序 | ~500 cycles | GB 级 |

| 常量内存 Constant Memory | 片外 (有缓存) | Grid | 整个程序 | ~5 cycles (缓存命中) | 64 KB |

  

## 变量声明与存储位置

  

| 变量声明 | 内存 | 作用域 | 生命周期 |

|---------|------|-------|---------|

| `int LocalVar;`（自动变量） | 寄存器 | 线程 | 线程内 |

| `__device__ __shared__ int SharedVar;` | 共享内存 | Block | Block 内 |

| `__device__ int GlobalVar;` | 全局内存 | Grid | 应用程序 |

| `__device__ __constant__ int ConstVar;` | 常量内存 | Grid | 应用程序 |

  

> **注意**：不加限定符的**自动变量**——基本类型和结构体在寄存器中，数组在全局内存中

  

## 简单矩阵乘法——基本内核

  

```cuda

__global__ void MatrixMulKernel(float* M, float* N, float* P, int Width) {

int Row = blockIdx.y * blockDim.y + threadIdx.y;

int Col = blockIdx.x * blockDim.x + threadIdx.x;

if ((Row < Width) && (Col < Width)) {

float Pvalue = 0;

for (int k = 0; k < Width; ++k)

Pvalue += M[Row * Width + k] * N[k * Width + Col];

P[Row * Width + Col] = Pvalue;

}

}

  

// 调用方式

dim3 dimGrid(ceil(Width/TILE_WIDTH), ceil(Width/TILE_WIDTH));

dim3 dimBlock(TILE_WIDTH, TILE_WIDTH);

MatrixMulKernel<<<dimGrid, dimBlock>>>(Md, Nd, Pd, Width);

```

  

## 性能瓶颈：内存带宽

  

- 每个 float 4 字节，每次乘加需读 2 个 float = 8 字节

- 每个乘加 = 2 FLOP

- 每 FLOP 需要 **4B 数据** → 4B/FLOP

- 假设 GPU：1,000 GFLOPS 算力 + 150 GB/s 带宽

- 受限于带宽：`150 GB/s ÷ 4B/FLOP = 37.5 GFLOPS`

- 算力利用率仅 **3.75%**

- **解决方案**：用共享内存复用数据，减少全局内存访问

# Ch5: 局部性与分块矩阵乘法

  

## 分块策略的核心思想

  

- **问题**：矩阵 M、N 很大放不进共享内存，从全局内存读取太慢

- **方案**：将 M 和 N 切成 tile → 逐 tile 加载到共享内存 → 计算 → 重复

  

### 全流程

  

1. 每个 Block 负责计算 P 的一个 TILE_WIDTH×TILE_WIDTH 子矩阵

2. 所有线程协作将 M 的 tile 和 N 的 tile 加载到共享内存

3. `__syncthreads()` 屏障等待全部加载完成

4. 从共享内存读取进行计算（而非全局内存）

5. `__syncthreads()` 屏障确保所有线程用完数据

6. 加载下一组 tile，重复直到完成

  

## 完整的分块矩阵乘法内核

  

```cuda

__global__ void MatrixMulKernel(float* M, float* N, float* P, int Width) {

__shared__ float subTileM[TILE_WIDTH][TILE_WIDTH];

__shared__ float subTileN[TILE_WIDTH][TILE_WIDTH];

  

int bx = blockIdx.x; int by = blockIdx.y;

int tx = threadIdx.x; int ty = threadIdx.y;

  

int Row = by * TILE_WIDTH + ty;

int Col = bx * TILE_WIDTH + tx;

float Pvalue = 0;

  

for (int m = 0; m < Width / TILE_WIDTH; ++m) {

// 协作加载 tile

subTileM[ty][tx] = M[Row * Width + m*TILE_WIDTH + tx];

subTileN[ty][tx] = N[(m*TILE_WIDTH + ty) * Width + Col];

__syncthreads();

  

for (int k = 0; k < TILE_WIDTH; ++k)

Pvalue += subTileM[ty][k] * subTileN[k][tx];

__syncthreads();

}

P[Row * Width + Col] = Pvalue;

}

```

  

## 性能改善分析

  

| Tile 大小 | 数据复用倍数 | 等价算力 | 说明 |

|-----------|------------|---------|------|

| 不分块 | 1× | 37.5 GFLOPS | 严重受限于带宽 |

| 16×16 | 16× | 600 GFLOPS | 瓶颈大幅缓解 |

| 32×32 | 32× | 1200 GFLOPS | 算力不再是瓶颈！ |

  

> 计算公式：`(150 GB/s / 4B) × 复用倍数 = 等效算力`

  

## __syncthreads() 的使用规则

  

- 仅对 Block 内线程有效

- Block 内**所有**线程必须执行到同一个 `__syncthreads()` 调用处

- 保证所有线程在 barrier 之前的全局/共享内存写操作已对其他线程可见

- 不保证 I/O（如 printf）的可见性

  

### 是否需要 barrier 的判断

  

```cuda

// 场景1：每个线程只读自己写的值——不需要 barrier

int tx = threadIdx.x, ty = threadIdx.y;

__shared__ float tile[TW][TW];

tile[ty][tx] = myNumber;

otherNumber = tile[ty][tx];

  

// 场景2：线程间交叉读取——需要 barrier

int tx = threadIdx.x, ty = threadIdx.y;

__shared__ float tile[TW][TW];

tile[ty][tx] = myNumber;

__syncthreads();

otherNumber = tile[tx][ty];

  

// 场景3：读完之后没有写——不需要 barrier

int tx = threadIdx.x, ty = threadIdx.y;

__shared__ float tile[TW][TW];

tile[ty][tx] = myNumber;

__syncthreads();

otherNumber = tile[tx][ty];

thirdNumber = tile[TW - tx - 1][ty];

// 没有线程修改 tile，不需要更多 barrier

  

// 场景4：完成读取后又有写——需要 barrier

int tx = threadIdx.x, ty = threadIdx.y;

__shared__ float tile[TW][TW];

tile[ty][tx] = myNumber;

__syncthreads();

otherNumber = tile[tx][ty];

thirdNumber = tile[TW - tx - 1][ty];

tile[ty][tx] = myNumber * 2.0; // 写操作需要 barrier 保护之前的读

```

# Ch6: 分块进阶——边界条件处理

  

## 问题：矩阵大小不一定是 Tile 宽度的整数倍

  

- 边界 Block 的线程可能会访问不存在的输入元素

- 解决方案：加载 tile 时检查边界，越界的元素填 0

  

## 修改后的分块矩阵乘法（支持任意大小）

  

### 修改 tile 循环计数

  

```cuda

// 原代码（假设整除）

for (int m = 0; m < Width / TILE_WIDTH; ++m)

  

// 修正后（向上取整）

for (int m = 0; m < (Width - 1) / TILE_WIDTH + 1; ++m)

```

  

### 修改 tile 加载代码（写 0 替代越界元素）

  

```cuda

// 加载 M 的 tile

if (Row < Width && m * TILE_WIDTH + tx < Width) {

subTileM[ty][tx] = M[Row * Width + m * TILE_WIDTH + tx];

} else {

subTileM[ty][tx] = 0;

}

  

// 加载 N 的 tile

if (m * TILE_WIDTH + ty < Width && Col < Width) {

subTileN[ty][tx] = N[(m * TILE_WIDTH + ty) * Width + Col];

} else {

subTileN[ty][tx] = 0;

}

```

  

### 修改最终写 P 的代码

  

```cuda

if (Row < Width && Col < Width) {

P[Row * Width + Col] = Pvalue;

}

```

  

> **关键洞察**：对越界的输入填 0 而不是跳过它们，可以使内部计算循环保持统一的代码路径，避免控制分歧

# Ch7: DRAM 带宽与内存访问合并

  

## DRAM 工作原理

  

- DRAM 位存储在电容上，通过晶体管连接到 bit line 读取

- 需要周期性刷新（约 50ms），所以是「动态」RAM

- 一条 bit line 连接约 1000 个存储单元

  

## Burst 模式（突发传输）

  

- 激活一行 row 后，同一行上的所有数据都会被读出到 sense amp

- 可以通过改变 multiplexer 的输入选择位实现「突发」传输

- **现代 DRAM 总是以 burst 模式访问**

- 如果访问的不是顺序地址，burst 字节会被丢弃

  

## 多 Bank 并行

  

- DRAM 被分为多个 bank

- 不同 bank 可以并行访问

- 交错访问可以掩盖 bank 的延迟

  

## 2D 数组的行主序线性化

  

在 C/CUDA 中：

```

M[i][j] → M[i * Width + j] （行主序）

```

  

## 内存访问合并 (Coalescing)

  

### 关键概念

  

- 同一 warp 中的线程访问相邻内存位置时，DRAM burst 可以完全利用

- 不合并访问时，burst 中的大部分字节被浪费

  

### 矩阵乘法中的访问模式

  

```cuda

// M[Row * Width + k]：通过行访问 —— 不合并

// N[k * Width + Col]：通过列访问 —— 合并

  

for (int k = 0; k < Width; ++k)

Pvalue += M[Row * Width + k] * N[k * Width + Col];

```

  

- 对 N 的访问是合并的：相邻线程访问相邻列

- 对 M 的访问是不合并的：相邻线程访问相隔 Width 的元素

  

### 分块实现了「角落转折」(Corner Turning)

  

- 加载 tile 到共享内存时将不合并的模式变为合并的模式

- 之后的计算从共享内存读取，不再依赖全局内存的访问模式

  

## 访问模式练习题

  

```cuda

int tx = blockIdx.x * blockDim.x + threadIdx.x;

int ty = blockIdx.y * blockDim.y + threadIdx.y;

  

// 好的模式：B[ty * Width + tx] — 相邻线程访问相邻列

B[ty * Width + tx] = 2 * A[ty * Width + tx];

  

// 坏的模式：B[tx * Width + ty] — 相邻线程访问间隔 Width 的列

B[tx * Width + ty] = 2 * A[tx * Width + ty];

```

  

> X 维度是线性化索引的最小维度，同一个 warp 的线程在 X 上相邻

# Ch8: 卷积、常量内存与缓存

  

## 卷积概念

  

- 卷积将一个数组转换为另一个数组

- 每个输出元素是附近输入元素的加权和

- 权重由**卷积掩码**（mask/filter/kernel）定义

  

### 1D 卷积

  

```

P[i] = Σ N[i + j - radius] × M[j] (j 从 0 到 mask_width-1)

```

  

### 边界处理

  

- 在数组的边界处，输入元素可能不存在（ghost elements）

- 通常填 0 （zero padding）

  

```cuda

__global__ void convolution_1D_basic_kernel(float *N, float *M, float *P,

int Mask_Width, int Width) {

int i = blockIdx.x * blockDim.x + threadIdx.x;

float Pvalue = 0;

int N_start_point = i - (Mask_Width / 2);

for (int j = 0; j < Mask_Width; j++) {

if (N_start_point + j >= 0 && N_start_point + j < Width) {

Pvalue += N[N_start_point + j] * M[j];

}

}

P[i] = Pvalue;

}

```

  

## 常量内存

  

- 掩码 M 是只读的，所有线程访问相同的掩码元素

- 适合放入**常量内存**（`__constant__`）

- 有专用缓存，命中时延迟约 5 cycles

- 大小 64 KB

  

### 使用常量内存

  

```cuda

// 在文件作用域声明

static __constant__ float Mc[MASK_WIDTH][MASK_WIDTH];

  

// 主机端复制

cudaMemcpyToSymbol(Mc, M, MASK_WIDTH * MASK_WIDTH * sizeof(float));

  

// 内核中直接访问

ConvolutionKernel<<<dimGrid, dimBlock>>>(Nd, Pd);

```

  

## 缓存 vs 共享内存

  

| 特性 | 缓存 (Cache) | 共享内存 (Scratchpad) |

|------|-------------|---------------------|

| 位置 | 片上 SRAM | 片上 SRAM |

| 控制 | 硬件自动管理 | 程序员手动管理 |

| 写入 | 需要写回策略 | 直接写入 |

| 适应性 | 自动适应访问模式 | 需要预见性设计 |

  

> 从 Volta 架构起，L1 缓存和共享内存共享同一物理 SRAM，可动态分配

  

## 内存局部性

  

- **空间局部性**：访问地址 X 后很快会访问 X+1

- **时间局部性**：访问地址 X 后很快会再次访问 X

- 缓存利用两种局部性提升性能

# Ch9: 分块卷积

  

## 1D 分块卷积的基本思想

  

- 将输入分成 tile，每个 tile 加上左右 halo 区域加载到共享内存

- halo 大小 = mask_radius = mask_width/2

  

### 输出并行 vs 输入并行

  

| 策略 | 每线程做什么 | 共享内存 tile 是否包含 halo | 优点 | 缺点 |

|------|------------|--------------------------|------|------|

| 输出并行 | 每个线程计算一个输出元素 | 需要包含 halo | 计算无分歧 | 加载有分歧 |

| 输入并行 | 每个线程加载一个输入元素 | 包含完整输入 tile | 加载无分歧 | 部分线程不计算输出 |

  

### 1D 卷积——输出并行（halo 从全局内存读）

  

```cuda

__global__ void convolution_1D_tiled_cache_kernel(float *N, float *P, int Width) {

__shared__ float tile[TILE_WIDTH];

int This_tile_start_point = blockIdx.x * blockDim.x;

int i = This_tile_start_point + threadIdx.x;

tile[threadIdx.x] = N[i];

__syncthreads();

  

int radius = MASK_WIDTH / 2;

int N_start_point = i - radius;

float Pvalue = 0;

for (int j = 0; j < MASK_WIDTH; j++) {

int N_index = N_start_point + j;

if ((N_index >= 0) && (N_index < Width)) {

int tile_index = N_index - This_tile_start_point;

if ((tile_index >= 0) && (tile_index < blockDim.x))

Pvalue += tile[tile_index] * Mc[j];

else

Pvalue += N[N_index] * Mc[j]; // 从全局内存读 halo

}

}

P[i] = Pvalue;

}

```

  

### 1D 卷积——输出并行（halo 也加载到共享内存）

  

```cuda

__global__ void convolution_1D_tiled_kernel(float *N, float *P, int Width) {

__shared__ float tile[TILE_SIZE + MASK_WIDTH - 1];

int radius = MASK_WIDTH / 2;

int i = blockIdx.x * blockDim.x + threadIdx.x;

int start = i - radius;

  

// 所有线程加载一个元素

if (0 <= start && Width > start) {

tile[threadIdx.x] = N[start];

} else {

tile[threadIdx.x] = 0.0f;

}

// 部分线程加载额外元素（halo 的另一侧）

if (MASK_WIDTH - 1 > threadIdx.x) {

start += TILE_SIZE;

if (Width > start) {

tile[threadIdx.x + TILE_SIZE] = N[start];

} else {

tile[threadIdx.x + TILE_SIZE] = 0.0f;

}

}

__syncthreads();

  

if (i < Width) {

float Pvalue = 0.0f;

for (int j = 0; MASK_WIDTH > j; j++) {

Pvalue += tile[threadIdx.x + j] * Mc[j];

}

P[i] = Pvalue;

}

}

```

  

## 2D 分块卷积（输入并行）

  

- Block 维度：`TILE_WIDTH + MASK_WIDTH - 1`

- Grid 划分基于输出 tile

- 每个线程加载一个输入元素（而非输出元素）

- 只有内部线程计算输出

  

```cuda

dim3 dimBlock(TILE_WIDTH + 4, TILE_WIDTH + 4, 1); // 5×5 mask 时+4

dim3 dimGrid(ceil(Width/(1.0*TILE_WIDTH)), ceil(Width/(1.0*TILE_WIDTH)), 1);

```

  

### 2D 卷积坐标转换

  

```cuda

int tx = threadIdx.x, ty = threadIdx.y;

int row_o = blockIdx.y * TILE_WIDTH + ty;

int col_o = blockIdx.x * TILE_WIDTH + tx;

int row_i = row_o - MASK_WIDTH / 2;

int col_i = col_o - MASK_WIDTH / 2;

```

  

### 各策略的 warp 利用率

  

| 策略 | mask 大小 | Block 大小 | 计算线程 | 活跃 warp | 利用率 |

|------|----------|-----------|---------|----------|-------|

| 输入并行 | 5×5 | 16×16 | 12×12=144 | 6个warp(24线程/warp) | 较高 |

| 输入并行 | 9×9 | 16×16 | 8×8=64 | 2个warp(16线程/warp) | 较低 |

| 输出并行 | 9×9 | 16×16 | 全部256 | 全部活跃 | 高 |

  

> 对于大 mask，输入并行导致大量线程不参与计算；输出并行则所有线程活跃，但需要更复杂的加载代码

# Ch10: 卷积分析与数据重用

  

## 带宽减少的计算

  

### 1D 分块卷积

  

- 加载的元素数：`TILE_WIDTH + MASK_WIDTH - 1`

- 替换的全局内存访问：`TILE_WIDTH × MASK_WIDTH`

- 带宽减少率：`(TILE_WIDTH × MASK_WIDTH) / (TILE_WIDTH + MASK_WIDTH - 1)`

  

| TILE_WIDTH | 256 | 128 | 64 | 32 | 16 |

|-----------|-----|-----|----|----|----|

| Mask=5 减少率 | 4.9 | 4.9 | 4.7 | 4.4 | 4.0 |

| Mask=9 减少率 | 8.7 | 8.5 | 8.0 | 7.2 | 6.0 |

  

### 2D 分块卷积

  

- 加载的元素数：`(TILE_WIDTH + MASK_WIDTH - 1)²`

- 替换的访问数：`TILE_WIDTH² × MASK_WIDTH²`

- 减少率：`(TILE_WIDTH² × MASK_WIDTH²) / (TILE_WIDTH + MASK_WIDTH - 1)²`

  

| TILE_WIDTH | 64 | 32 | 16 | 8 |

|-----------|-----|----|----|---|

| Mask=5 减少率 | 22.1 | 19.7 | 16 | 11.1 |

| Mask=9 减少率 | 64 | 51.8 | 36 | 20.3 |

  

## 需要多少数据重用以充分利用算力？

  

以不同年代 GPU 为例：

  

| GPU | 算力 | 带宽 | 每FLOP需数据 | 所需复用倍数 |

|-----|------|------|-------------|------------|

| ~2010 GPU | 1,000 GFLOPS | 150 GB/s | 4B/FLOP | 13.3× |

| ~2020 GPU | 5,000 GFLOPS | 192 GB/s | 4B/FLOP | 52.1× |

| H100 (2023) | 26,000 GFLOPS | 2,000 GB/s | 4B/FLOP | **26×** |

  

## 并发内存访问

  

- 每个线程加载 4B 到共享内存

- 2048 个线程只加载 8 KB

- 但共享内存通常有 64 KB 或更大

- 可以通过计算每个线程多个元素来利用剩余共享内存

  

## Ampere SM 内存架构

  

| 资源 | 大小 |

|------|------|

| Register File / SM | 256 KB |

| L1 Cache / Shared Memory | 192 KB (最大 164 KB 给共享内存) |

# Ch20: 异构计算集群与 CUDA Streams

  

## 背景：HPC 集群与 MPI

  

世界顶级超级计算机的每个节点（node）同时使用 CPU 和 GPU。

主流的集群编程接口是 **MPI（Message Passing Interface）**。

  

MPI 核心概念：

- **SPMD 模型**：所有进程执行同一个程序

- **分布式内存模型**：进程之间通过发送/接收消息来交换数据

- **MPI Rank**：每个进程的唯一编号（类比 `threadIdx`）

- **Communicator**：定义通信范围的进程组，默认是所有进程（`MPI_COMM_WORLD`）

  

### MPI 基本 API

  

| 函数 | 功能 |

|------|------|

| `MPI_Init()` | 初始化 MPI 运行时 |

| `MPI_Comm_rank(comm, &rank)` | 获取进程的 rank |

| `MPI_Comm_size(comm, &size)` | 获取总进程数 |

| `MPI_Send(buf, count, type, dest, tag, comm)` | 发送消息 |

| `MPI_Recv(buf, count, type, src, tag, comm, &status)` | 接收消息 |

| `MPI_Abort(comm, code)` | 异常终止 |

| `MPI_Finalize()` | 结束 MPI 运行时 |

  

### MPI 基本程序结构

  

```c

#include "mpi.h"

int main(int argc, char *argv[]) {

int pid, np;

MPI_Init(&argc, &argv);

MPI_Comm_rank(MPI_COMM_WORLD, &pid);

MPI_Comm_size(MPI_COMM_WORLD, &np);

if (np < 3) {

if (0 == pid) printf("Need 3 or more processes.\n");

MPI_Abort(MPI_COMM_WORLD, 1);

}

// 计算进程 vs 数据服务器进程

if (pid < np - 1)

compute_process(params); // 计算进程

else

data_server(params); // 数据服务器

MPI_Finalize();

return 0;

}

```

  

## 领域划分（Domain Partition）

  

3D 热传导 stencil 计算（Jacobi 迭代法）的并行化：

- 每个点的新值 = 自身 + 24 个邻居的加权和（25-point stencil）

- 将 3D 网格沿 z 方向划分成若干 partition 分配给不同 MPI 进程

- 每个进程需要 **halo cells**：来自邻居进程的 4 个 slice 边界数据

  

### 计算过程与 Halo 交换

  

```

Iteration 1: 计算 → 发送边界 → 接收 halo →

Iteration 2: 计算 → 发送边界 → 接收 halo →

...

Iteration N: 计算 → 发送边界 → 接收 halo

```

  

## CUDA Streams 与计算-通信重叠

  

### 朴素方案的问题

  

一个简单方案是：

1. 所有进程先计算整个 partition

2. 然后全部进行 MPI 通信交换 halo

3. 重复

  

**问题**：计算时网络闲置，通信时 GPU 闲置。

  

### 优化方案：分阶段 + 重叠

  

将每个进程的计算分为两个阶段：

  

**阶段 1**：计算边界 slice（邻进程需要的 halo 数据）

**阶段 2**（并行进行）：

- 通信：将阶段 1 算出的边界数据发给邻居

- 计算：完成 partition 内部剩余网格点的计算

  

如果内部计算时间 > 通信时间，通信延迟则可以被完全隐藏。

  

### CUDA Streams 基本操作

  

```cuda

cudaStream_t stream0, stream1;

cudaStreamCreate(&stream0);

cudaStreamCreate(&stream1);

  

// 在 stream0 上进行异步内存复制

cudaMemcpyAsync(dst0, src0, size, cudaMemcpyHostToDevice, stream0);

  

// 在 stream1 上启动 kernel

kernel<<<grid, block, 0, stream1>>>(args);

  

// 等待 stream 完成

cudaStreamSynchronize(stream0);

cudaStreamDestroy(stream0);

```

  

> 不同 stream 中的操作可以重叠执行！

  

### Pinned Memory（锁定内存）

  

为了实现异步传输，需要**可分页锁定内存**（pinned memory）：

  

```cuda

// 分配 pinned memory（不会被 OS 换出）

cudaHostAlloc((void**)&h_boundary, num_bytes, cudaHostAllocDefault);

```

  

为什么需要 Pinned Memory？

- 普通（pageable）内存可能被 OS 换出→ 无法异步 DMA

- Pinned 内存固定在物理内存中→ DMA 引擎可直接读写

- 使用 `cudaMemcpyAsync()` 配合 pinned memory 才能实现真正的异步传输

  

### CUDA-aware MPI

  

如果 MPI 库是 CUDA-aware，可以直接传递 GPU 指针：

  

```c

// CUDA-aware MPI：直接使用设备指针

float *d_buf;

cudaMalloc(&d_buf, size);

MPI_Send(d_buf, count, MPI_FLOAT, dest, tag, comm);

```

  

如果不是 CUDA-aware，需要手动中转：

  

```c

// 手动中转

cudaMemcpy(h_buf, d_buf, size, cudaMemcpyDeviceToHost);

MPI_Send(h_buf, count, MPI_FLOAT, dest, tag, comm);

MPI_Recv(h_buf, count, MPI_FLOAT, src, tag, comm, &status);

cudaMemcpy(d_buf, h_buf, size, cudaMemcpyHostToDevice);

```

# Ch21: CUDA 动态并行（Dynamic Parallelism）

  

## 概念

  

CUDA 动态并行允许 GPU kernel 在**设备端直接启动子 kernel**，不需要 CPU 介入。

  

### 等效模型

  

```

CPU 启动 parent_kernel

└─ parent_kernel 在 GPU 上运行

└─ thread 0 调用 child_kernel<<<…>>>→ 子 kernel 在 GPU 上运行

└─ 子 kernel 完成后，父 kernel 继续

└─ 父 kernel 结束

```

  

### 基本用法

  

```cuda

__global__ void child_kernel(int *data) {

data[threadIdx.x] *= 2;

}

  

__global__ void parent_kernel(int *data, int n) {

if (threadIdx.x == 0) {

// 从 GPU 上直接启动子 kernel

child_kernel<<<1, n>>>(data);

}

// 父 kernel 不需要显式等待子 kernel

// 父 kernel 结束时自动等待所有子 kernel 完成

}

```

  

## 同步语义

  

- 子 kernel 的启动对启动它的线程是**隐式同步**的

- 子 kernel 在父 kernel 结束时**自动隐式等待**

- 可使用 `cudaDeviceSynchronize()` 在父 kernel 中显式等待子 kernel

  

## 限制与使用建议

  

| 限制 | 说明 |

|------|------|

| 嵌套深度 | 理论 24 层，实际受资源限制（通常 2-3 层足够） |

| 资源消耗 | 每个子 kernel 消耗 SM 资源（寄存器、共享内存） |

| 同步开销 | 动态并行增加了调度复杂度 |

| 调试难度 | 比主机端启动更难 debug |

  

## 适用场景

  

- 递归算法（快速排序、树遍历）

- 自适应网格细化

- 负载不均衡的算法

- 树形数据结构的处理

- 需要 GPU 运行时动态决策计算规模的任务
