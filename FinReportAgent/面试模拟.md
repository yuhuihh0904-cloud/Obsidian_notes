# FinReportAgent 模拟面试题参考答案（修正版）

## 1. 请你先用 2 到 3 分钟介绍一下你的这个项目。重点讲清楚三件事：1. 这个项目想解决什么问题 2. 整体做了哪些核心功能 3. 你认为这个项目里最有亮点、最能体现你能力的部分是什么

### 参考回答

我做的这个项目叫 **FinReportAgent**，核心目标是让系统能够针对**不同类型的金融 PDF 文档**，比如公司季报、年报、行业周报、券商研报等，自动完成分析，并且支持用户围绕当前文档继续追问，而不是只能给一次性总结。

这个项目主要解决两个问题：

第一，**金融文档通常比较长，而且噪声很多**。像财报里会有免责声明、目录、公司介绍等低价值内容，如果直接把整篇文档丢给开源模型，模型很容易因为上下文窗口限制而抓不住重点。所以我对 PDF 提取和清洗做了优化，尽量把有限的上下文预算留给真正有价值的财务信息。

第二，**用户对金融文档的需求通常不是一次性总结，而是连续追问**。比如用户先看整体结论，再问某个业务条线、某个季度变化、某个风险点。如果系统没有记忆能力，也没有原文证据支撑，后续追问就很容易答偏或者答空。所以我在项目里加入了 `Session Memory`、`Research Memory` 和 `RAG` 三层机制。

整体上，这个项目主要做了三部分工作：

1. **PDF 提取与解析优化**  
   我对 PDF 文本提取做了清洗、压缩和高价值段落保留，提升了开源免费模型处理长文档的能力。最开始项目大概只能稳定处理十几页，后来优化后对常见季报、周报都能比较稳定处理，超长文档也支持手动指定页码范围分析。

2. **记忆模块设计**  
   我设计了两类记忆：
   - `Session Memory`：保存当前会话里的当前文档状态、最近问题和最近分析结果，服务连续追问。
   - `Research Memory`：面向研究对象实体做长期存储，比如某家公司或某个行业，把历史分析抽象成 `summary / insight / risk` 三类 card，并存到 SQLite 和 Chroma 里，支持跨文档、跨会话补充历史背景。

3. **RAG 模块接入**  
   我把 PDF 切分成 chunk 做向量检索，RAG 支持两种模式：  
   - 当前文档检索  
   - 同实体多文档检索  
   这样追问时既可以基于当前文档找原文依据，也可以在需要对比多个季度或多个历史文档时，按 `entity_id` 做跨文档检索。

我觉得这个项目里最能体现我能力的部分，主要是 **Research Memory 和 RAG 的结合设计**。

因为普通的 PDF 分析项目通常只能做到“上传一份文档，生成一个总结”，但我这个项目进一步做到了：
- 能围绕当前文档连续追问
- 能结合历史研究背景补充理解
- 能在回答里尽量基于原文证据，而不是只靠模型生成

尤其是在讨论同一家公司多个季度趋势、变化、风险演变时，这种设计的效果会比较明显。这个部分既涉及系统设计，也涉及存储、检索和 prompt 组织，所以我认为是项目里最有技术含量的部分。

## 2. 你刚才提到有 Session Memory、Research Memory 和 RAG 三部分。那你详细讲一下：1. 这三者分别解决什么问题 2. 为什么不能只用其中一种机制解决全部问题 3. 在你的项目里，用户上传一份 PDF 之后，到第一次追问之前，这三者分别在什么时候介入

### 参考回答

这三者在我的项目里分别解决的是三个不同层次的问题。

### 第一，Session Memory 解决的是“当前会话连续性”问题
它保存的是当前会话下和当前文档相关的状态，比如：
- 当前在分析哪份文档
- 当前文档的 `document_id`、`entity_id`、报告期等信息
- 最近用户问了什么
- 上一轮或最近几轮分析结果是什么

它的核心作用不是做长期存储，而是保证用户在追问时，模型知道“我们现在讨论的是哪份文档、上一轮已经得出了什么结论、这一轮问题是在承接什么上下文”。

### 第二，Research Memory 解决的是“跨文档、跨会话的历史研究背景”问题
它面向的是研究对象实体，比如某一家公司或者某一个行业，不是面向某一次会话。  
我会把历史分析结果抽象成 `summary / insight / risk` 三类 card，再存到 SQLite 和 Chroma 里。这样后续无论是在首轮 PDF 分析还是在追问阶段，系统都可以补充该实体的历史背景信息。

比如分析腾讯 2025Q4 时，Research Memory 可以补充腾讯之前几个季度的核心结论、重要风险点或业务趋势，这样模型在做趋势判断时不会只盯着当前文档。

### 第三，RAG 解决的是“原文证据检索”问题
RAG 存的是当前 PDF 切出来的 chunk 以及对应的文档元信息，比如 `document_id`、`entity_id`、报告期、页码等。  
追问时系统会根据 query 去检索最相关的原文片段，把这些片段作为证据注入 prompt。这样模型回答时不是只靠记忆或总结，而是尽量基于当前文档或相关历史文档的原始内容。

---

### 为什么不能只用一种机制解决全部问题？

因为这三者的职责边界不一样：

- **只用 Session Memory 不够**  
  它只能保证当前对话连续，但不能提供稳定的跨会话历史背景，也不能保证对具体数字、原文细节回答准确。

- **只用 Research Memory 不够**  
  它存的是抽象后的历史结论，不是原文证据，适合补充背景，但不能替代当前文档原文，也不适合承接当前会话中的逐轮追问。

- **只用 RAG 也不够**  
  RAG 能提供原文证据，但它并不天然知道上一轮已经讨论了什么，也不适合承担“历史结论压缩存储”这类任务。

所以我的理解是：
- `Session Memory` 负责上下文连续性
- `Research Memory` 负责历史背景补充
- `RAG` 负责证据可追溯性

这三者是互补关系，而不是替代关系。

---

### 在我的项目里，从用户上传 PDF 到第一次追问之前，这三者的介入时机是这样的：

#### 首轮 PDF 分析前
1. 先提取 PDF 文本，识别文档类型
2. 根据当前 PDF 解析出 `entity_id`、报告期等元信息
3. **RAG 先介入**：把当前 PDF 切 chunk 入库，供后续追问检索
4. **Research Memory 介入**：检索该 entity 的历史 card，拼成历史背景上下文，补充进首轮分析 prompt

#### 首轮分析完成后
5. 系统得到当前文档的分析结果
6. **Session Memory 介入**：把当前文档状态和本轮分析结果写进去，供后续追问承接上下文
7. 同时把本轮分析再归档到 `Research Memory` 中，形成新的长期研究 card

#### 到第一次追问时
8. 三者都会参与：
- Session Memory 提供当前会话状态
- Research Memory 提供历史背景
- RAG 提供原文证据

## 3. 假设用户上传的是腾讯连续几个季度的财报，然后追问：“和上季度相比，这一季度广告业务的变化说明了什么？这种趋势是短期波动还是持续改善？” 请你结合这个具体例子讲一下，系统在追问时会如何组织上下文。重点回答：1. Session Memory 在这个问题里具体提供什么 2. Research Memory 在这个问题里具体提供什么 3. RAG 在这个问题里具体提供什么 4. 如果 Research Memory 里的历史结论和当前文档原文不一致，你的系统以谁为准，为什么

### 参考回答

这个问题是一个比较典型的“趋势判断型追问”，它不是只问一个静态事实，而是在问：
- 当前季度和上季度相比发生了什么变化
- 这个变化应该怎么解释
- 它到底是短期波动还是持续改善

在我的项目里，追问时会把 `Session Memory`、`Research Memory` 和 `RAG` 三类上下文一起组织进 prompt，但三者提供的信息层次不一样。

### 1. Session Memory 提供的是当前会话里的承接关系
在这个例子里，Session Memory 主要会提供：
- 当前正在讨论的是腾讯这份财报
- 当前文档对应的 `document_id`、`entity_id`、报告期等信息
- 上一轮或最近几轮分析里已经形成的结论
- 用户之前是否已经问过广告业务、增长趋势、风险等内容

也就是说，它的作用是让模型知道：
**“我们不是第一次在谈腾讯，也不是第一次在谈这份文档，这一轮问题是在承接前面的讨论。”**

它解决的是追问不断层的问题，而不是提供原始事实依据。

### 2. Research Memory 提供的是腾讯这个 entity 的历史研究背景
在这个问题里，Research Memory 主要会补充：
- 腾讯之前几个季度广告业务相关的 `summary / insight / risk`
- 历史分析里对广告业务变化的结论
- 是否曾经出现过类似的趋势判断，比如“广告恢复”“广告承压”“商业化改善”等

它的作用是帮助模型从更长的历史视角去看这个问题。  
比如当前季度广告业务改善了，Research Memory 可能会提示：这并不是第一次改善，而是前几个季度已经出现恢复迹象，那模型就更容易判断这是“延续性改善”，而不是单季度偶然波动。

但它提供的是**历史结论和背景信息**，不是当前问题最核心的证据来源。

### 3. RAG 提供的是和当前问题最相关的原文证据
这个问题里，RAG 应该尽量检索出：
- 当前季度财报中关于广告业务的数据和原文表述
- 上一季度或历史相关季度财报中广告业务的数据和原文表述
- 如果开启的是同实体多文档检索，还可以把几个季度相关 chunk 一起捞出来

比如：
- 当前季度广告收入同比/环比数据
- 上季度广告收入数据
- 管理层对广告业务变化原因的描述
- 财报中是否提到行业恢复、视频号商业化、宏观广告投放修复等因素

这些内容会被整理进 prompt 里，成为模型判断“变化说明了什么”的主要依据。

### 4. 如果 Research Memory 和当前文档原文不一致，以谁为准？
在我的系统里，**必须以当前文档原文和 RAG 检索到的原文证据为准**。

原因是：
- Research Memory 存的是过去分析后的抽象结论，本质上是辅助背景
- RAG 提供的是当前问题对应的原始文档证据，更接近事实来源
- 趋势类问题尤其容易因为历史总结滞后而产生偏差，所以不能让历史 card 替代当前文档原文

所以系统设计上，我会让：
- `Research Memory` 只用于比较、趋势解释和补充背景
- `RAG` 优先提供当前问题的证据
- 一旦二者冲突，以当前文档原文为准

## 4. 为什么这个追问不能只靠 Research Memory，也不能只靠 Session Memory，还必须要有 RAG？你从下面三个角度来回答：1. 如果只有 Session Memory，会缺什么 2. 如果只有 Research Memory，会缺什么 3. RAG 在这个场景下为什么是不可替代的

### 参考回答

这个场景里，RAG 是不可替代的，因为这个问题本质上不是单纯承接上下文，而是要做**带证据的趋势判断**。

### 1. 如果只有 Session Memory，会缺什么？
如果只有 Session Memory，系统能知道：
- 我们前面讨论过腾讯
- 用户上一轮问了什么
- 上一轮得出的结论是什么

这样追问流程是能连续下去的，但问题在于：
**Session Memory 不保证细节准确，也不提供稳定的原文依据。**

比如用户问：
- 广告业务这一季度和上季度具体差了多少
- 财报原文对变化原因是怎么说的
- 这个判断对应的是哪一页哪一段内容

这些都不是 Session Memory 的强项。  
它能让模型“知道在聊什么”，但不能让模型“准确引用依据”。

### 2. 如果只有 Research Memory，会缺什么？
如果只有 Research Memory，系统可以拿到腾讯历史上的一些总结性 card，比如：
- 广告业务恢复
- 风险点变化
- 历史判断是改善还是承压

但问题是：
**Research Memory 是历史结论层，不是当前问题的证据层。**

它的问题主要有两个：
- 不能很好承接当前会话里一轮接一轮的追问过程
- 不能替代当前季度和上季度财报里的原始数据、原始表述

也就是说，它可以帮助模型理解背景，但如果单靠它，很容易把问题回答成“泛泛而谈的历史总结”，而不是“基于当前文档的具体分析”。

### 3. 为什么 RAG 在这个场景下不可替代？
因为这类问题问的是：
- 具体业务线
- 具体季度变化
- 具体趋势判断

这就要求系统必须回到原文里找依据。  
RAG 在这里的价值是：

- 把当前季度和上季度最相关的原文片段检索出来
- 让模型基于这些原文做判断
- 降低模型凭印象或凭总结回答的风险
- 让回答更可解释、更可追溯

所以我会把这三个模块的定位总结成一句话：

- `Session Memory` 解决上下文连续性
- `Research Memory` 解决历史背景补充
- `RAG` 解决证据可追溯性

在讨论季度变化、趋势改善、业务判断这类问题时，**RAG 提供的原文证据是必须的**，因为没有它，回答很容易流于抽象和不准确。

## 5. 你在项目里同时用了 SQLite 和 Chroma 来做 Research Memory。为什么要用两套存储，而不是只用一种？你回答时重点讲这几个点：1. SQLite 存什么 2. Chroma 存什么 3. 为什么不能只用 SQLite 4. 为什么不能只用 Chroma 5. 你实际的检索链路是怎么走的

### 参考回答

我在 `Research Memory` 里同时使用 SQLite 和 Chroma，核心原因是：
**我希望同时兼顾“完整结构化存储”和“语义检索能力”。**

### 1. SQLite 存什么？
SQLite 存的是完整的结构化研究数据，包括：
- `entity` 层信息，比如公司名、ticker、aliases、tags
- `analysis_record`，也就是某次分析的来源文档、报告期、摘要等
- `memory_card` 的完整内容，包括：
  - card_id
  - card_type
  - content
  - importance
  - tags
  - source_pages
  - source_file
  - doc_type
  - ticker
  - created_at
  等完整字段

也就是说，SQLite 是我的**主存储和完整归档层**。

### 2. Chroma 存什么？
Chroma 主要存的是适合做语义召回的 card 信息。  
不会把 SQLite 里所有结构化字段都塞进去，而是保留最重要的部分，比如：
- card_id
- entity_id
- content
- 部分必要 metadata

这样当用户 query 到来时，我可以先通过 Chroma 做向量相似度检索，快速找到最相关的 card。

### 3. 为什么不能只用 SQLite？
因为 SQLite 更适合结构化存储，不擅长语义召回。  
如果只用 SQLite，我可以按字段过滤、按时间排序，但对于用户这种自然语言问题，比如：

- “腾讯广告业务趋势”
- “历史风险变化”
- “游戏业务承压的原因”

SQLite 并不能很好处理“语义相似”这种检索需求。

所以如果只用 SQLite，Research Memory 很难做到自然语言驱动的历史卡片召回。

### 4. 为什么不能只用 Chroma？
因为 Chroma 更适合做轻量的向量召回，但不适合承担完整研究档案的主存储职责。  
如果只用 Chroma，会有两个问题：

第一，很多完整字段不适合或不方便都塞进向量库里，尤其是复杂结构化信息。  
第二，我后续还需要对 research memory 做调试、可视化、按表查看、做归档管理，这些用 SQLite 更自然。

所以我的设计是：
- Chroma 负责“找”
- SQLite 负责“存”和“取完整信息”

### 5. 实际检索链路怎么走？
我的实际链路是：

1. 用户发起 query
2. 先到 Chroma 里做语义检索
3. 返回最相关的一批 `card_id`
4. 再根据 `card_id` 去 SQLite 里读取完整 card 信息
5. 最后把这些完整的 card 组织成 prompt 上下文

所以这个链路可以概括成一句话：

**先用 Chroma 找相关卡片，再用 SQLite 取完整卡片。**

这样设计的好处是：
- 检索更灵活
- 存储更完整
- 两者各做自己擅长的事情

## 6. 你刚才说 Research Memory 会把内容抽成 summary / insight / risk 三类 card。那你为什么要这样设计？为什么不是直接把整篇历史分析全文存下来，追问时直接检索全文？你重点讲这几个方面：1. 三类 card 各自想表达什么信息 2. 这样拆分相比“整篇分析全文存储”有什么好处 3. 对后续检索和 prompt 注入有什么帮助 4. 这样做有没有代价或者局限

### 参考回答

我把 `Research Memory` 设计成 `summary / insight / risk` 三类 card，而不是直接存整篇历史分析，主要是因为它的定位不是“历史全文仓库”，而是**面向 entity 的轻量化长期研究记忆**。

### 1. 三类 card 各自表达什么？
- **summary**：存某次分析最核心的总结结论，相当于浓缩版整体判断
- **insight**：存关键发现、积极变化、值得关注的亮点
- **risk**：存风险点、负面因素或未来可能持续影响判断的问题

这三类 card 不是为了完全复刻原始分析，而是为了让系统在后续检索时，能从不同维度补充对同一 entity 的理解。

### 2. 为什么不直接存全文？
如果直接存整篇历史分析，有几个明显问题：

#### 第一，容易挤占 prompt 上下文
Research Memory 的定位只是补充背景，如果把整篇历史分析都拿来参与检索和注入，很容易把上下文窗口浪费在大量冗余内容上。

#### 第二，信息混合严重，容易污染召回
整篇分析里往往会同时包含：
- 总结
- 正面信息
- 风险信息
- 各种细节展开

如果直接全文存储并检索，召回结果常常是多种信息混在一起，既不够精炼，也不利于模型快速抓重点。

#### 第三，Research Memory 面向的是 entity，不是单篇文档
这个模块的目标是沉淀“某个公司/行业的长期研究背景”，不是把每篇文档全文原样保留给后续再读一遍。  
如果直接存全文，它更像文档仓库，而不是研究记忆层。

### 3. 对后续检索和 prompt 注入有什么帮助？
把历史分析拆成 card 之后，有两个直接好处：

#### 一是检索更干净
用户问趋势、问风险、问亮点时，更容易召回到语义纯度更高的内容，而不是一篇很长、信息混杂的历史全文。

#### 二是 prompt 注入更轻量
系统可以只注入少量高价值 card，而不是整篇历史报告。  
这样能让 Research Memory 真正扮演“背景补充”的角色，而不会喧宾夺主。

### 4. 这样做有没有代价或局限？
有。

#### 第一，抽象之后会丢失部分细节
Card 是压缩后的信息，不可能保留整篇分析的所有上下文细节，所以它不适合作为精确事实的最终依据。

#### 第二，依赖抽取质量
如果 summary、insight、risk 抽取质量不高，后续召回效果也会受影响。

#### 第三，它不能替代原文证据
这也是为什么我还需要 RAG。Research Memory 只能提供历史背景，不能替代当前问题对应的原文 chunk。

所以这套设计本质上是一个取舍：
**用信息压缩和结构化，换取更高质量的长期背景检索能力。**

## 7. 你说一开始只能处理大约 15 页，后来优化到能处理 50 页左右。你具体做了哪些优化？这些优化分别解决了什么问题？你回答时尽量讲清楚：1. 最开始的瓶颈是什么 2. 你做了哪些具体优化 3. 这些优化为什么有效 4. 为什么最后不是无限扩大，而是大概停在 50 页左右

### 参考回答

我最开始做 PDF 分析时，最大的瓶颈其实不是“提取不到文本”，而是：
**提取出来的文本质量不高、噪声太多，而且开源免费模型的上下文能力有限。**

也就是说，问题不是单纯页数，而是：
- PDF 里有很多低价值内容
- 有效财务信息被淹没
- 模型上下文预算有限
- 文本过长时直接塞进 prompt 会明显影响分析质量

### 1. 最开始的瓶颈是什么？
最开始项目大概只能稳定处理十几页，主要有三个原因：

#### 第一，原始 PDF 文本噪声很多
像财报目录、免责声明、公司简介、重复页眉页脚这些内容很多，直接提取后会浪费大量字符预算。

#### 第二，模型窗口有限
我用的是开源免费模型，这类模型在长文本场景下不如大模型稳定，所以一旦输入太长，分析质量会明显下降。

#### 第三，重要信息没有被优先保留
直接把前若干页文本丢进去，并不能保证这些页刚好都是最有价值的信息。

### 2. 我做了哪些具体优化？
我主要做了四类优化：

#### （1）基础文本清洗
包括：
- 去掉多余空格和空行
- 清理明显无效格式字符
- 尽量减少页面噪声

#### （2）去掉低价值内容
比如：
- 目录
- 免责声明
- 公司背景介绍等不直接影响财务判断的内容

这一步的目标是：把字符预算尽量留给真正有分析价值的段落。

#### （3）支持用户手动指定页码范围
对于超长 PDF，尤其是大型年报，我允许用户手动输入起始页和结束页。  
这样用户可以主动跳过目录、前言、附录等低价值区域，直接分析重点部分。

#### （4）超长文本时做高价值段落保留
如果清洗后的文本还是超过 `max_char`，我会按段落打分，优先保留高价值段落。  
这个机制本质上是一个“保底方案”，保证在实在放不下全文时，系统优先保留更有可能包含财务结论、业务变化、风险提示的内容。

### 3. 这些优化为什么有效？
因为这些优化不是盲目扩展页数，而是在做两件事：

#### 第一，提高输入文本的“信息密度”
让同样的上下文预算里，真正有价值的信息占比更高。

#### 第二，降低模型被噪声干扰的概率
对于开源模型来说，输入越干净，结果通常越稳定。  
所以与其简单追求页数，不如提升单位字符里的有效信息量。

### 4. 为什么最后不是无限扩大，而是大概停在 50 页左右？
因为这个项目的定位不是“无限长文档处理系统”，而是**面向金融分析场景、尽量在开源模型约束下做到实用可用**。

实际场景里：
- 大多数季报、周报，本项目已经可以比较轻松处理
- 真正特别长的，主要是部分年报
- 对这种超长文档，我更倾向于让用户指定页码范围，而不是强行让模型一次吞下所有内容

所以最后停在一个比较实用的范围，是一个工程取舍：
**在模型能力、效果稳定性和系统复杂度之间做平衡，而不是一味追求页数上限。**

## 8. 你刚才提到“如果清洗后还是超过 max_char，就按段落价值打分，优先保留高价值段落”。那你为什么没有选择更常见的方案，比如：直接把整篇 PDF 切 chunk 做 map-reduce 式总结，或者干脆完全依赖 RAG，首轮分析时也不直接塞大段正文？你当时为什么选择了现在这套方案？请你从下面几个角度讲：1. 你当时面对的实际约束是什么 2. 你为什么没有走更复杂的长文档方案 3. 你现在这套方案适合什么场景，不适合什么场景 4. 如果后面模型能力更强或者资源更够，你会怎么继续改

### 参考回答

我当时没有直接上更复杂的长文档方案，主要是基于项目定位和实际约束做的选择，而不是单纯觉得那种方案不好。

### 1. 我当时面对的实际约束是什么？
首先，这个项目最初的目标是做一个**面向金融 PDF 的实用分析助手**，而不是一个完整的超长文档处理框架。  
同时我当时用的是开源免费模型，模型能力、稳定性和成本都有限，所以需要优先做“够用、稳定、容易落地”的方案。

另外，从输入分布看，项目主要面对的是：
- 公司季报
- 行业周报
- 券商研报

这类文档虽然不短，但大部分并没有长到必须用非常重的 map-reduce 体系去处理。  
真正特别长的，更多是一些大型公司年报。

### 2. 为什么没有直接走更复杂的长文档方案？
主要有三个原因：

#### 第一，复杂方案会显著增加链路复杂度
像 map-reduce 式总结，需要：
- 先切 chunk
- 分块生成局部摘要
- 再做汇总
- 可能还要做多轮 prompt 设计

这会增加工程复杂度，也会让输出更难稳定控制。

#### 第二，首轮分析场景需要较强的整体感
金融文档分析不是纯问答，有时候需要模型对整份文档形成一个相对完整的判断。  
如果从一开始就完全碎片化处理，可能会损失整体语境，影响首轮报告的连贯性。

#### 第三，项目演进顺序的原因
这个 PDF 提取和压缩模块是项目比较早期做的，当时还没有把 RAG 纳入整体设计。  
所以我当时优先做的是：让首轮分析在有限上下文里尽量保留高质量信息，而不是一开始就引入更复杂的多阶段长文档方案。

### 3. 我现在这套方案适合什么场景，不适合什么场景？
#### 适合的场景
- 常见公司季报
- 行业周报
- 中等长度券商研报
- 需要快速得到一份整体分析结论的场景

#### 不太适合的场景
- 极长年报
- 需要覆盖整本文档全部细节的深度研究
- 需要极强全书级别信息完整性的场景

所以我现在这套方案更像是：
**在金融分析场景里，用较轻的方式解决大部分常见文档的首轮分析问题。**

### 4. 如果后面模型更强、资源更够，我会怎么改？
如果后面模型能力更强或者资源更充足，我会考虑两条升级方向：

#### 第一，首轮分析引入分块总结或层级式摘要
对超长年报，可以考虑：
- 分章节总结
- 再做全局汇总
- 让超长文本分析更加系统化

#### 第二，进一步强化首轮分析和 RAG 的结合
目前 RAG 更多是在追问阶段发挥作用。  
如果后面要继续优化，也可以考虑让首轮分析阶段就更强地结合检索机制，而不是主要依赖压缩后的正文输入。

所以我现在这套方案，本质上是一个阶段性最优解：
**优先保证实用性和稳定性，而不是一开始就把系统做得很重。**

## 9. 你这个项目里，RAG 和 Research Memory 都会涉及“跨文档”的信息。那你怎么避免把不同报告期、不同文档的信息混在一起，导致回答串掉？你重点讲：1. 你在数据组织上做了哪些区分 2. RAG 里“当前文档检索”和“同实体多文档检索”是怎么区分的 3. 在回答追问时，系统怎么尽量提醒模型区分来源文件和报告期 4. 这个问题在你项目里有没有真实出现过，后来是怎么处理的

### 参考回答

这个问题其实是我项目里非常关键的一点，因为金融文档里“同一家公司多个季度的信息混在一起”会直接影响回答质量，尤其是在做趋势比较和跨季度分析的时候。

### 1. 在数据组织上，我做了哪些区分？
无论是 `Research Memory` 还是 `RAG`，我在存储时都会尽量带上文档背景信息，比如：
- `document_id`
- `entity_id`
- `report_period`
- `report_year`
- `report_type`
- `source_file`
- 页码信息等

这样做的目的，就是让系统在检索和组织上下文时，不是只拿到一段“内容”，而是知道：
**这段内容属于哪份文档、哪个报告期、哪个研究对象。**

### 2. RAG 里“当前文档检索”和“同实体多文档检索”怎么区分？
我的 RAG 支持两种模式：

#### （1）当前文档检索
这个模式会优先基于 `document_id` 过滤，只在当前上传的这份文档里找相关 chunk。  
适合回答：
- 当前文档里某个指标是多少
- 当前文档里某个业务怎么描述的
- 当前文档有没有提到某个风险

#### （2）同实体多文档检索
这个模式会基于 `entity_id` 来检索同一个研究对象下的多份历史文档。  
适合回答：
- 和上季度相比有什么变化
- 某个趋势是不是持续改善
- 历史几个季度里某个风险是如何演化的

也就是说：
- `document_id` 用来防止当前文档问答跑偏
- `entity_id` 用来支持历史对比和跨文档趋势分析

### 3. 在回答追问时，系统怎么提醒模型区分来源和报告期？
这部分我做了两层处理：

#### 第一层：检索结果本身带来源信息
RAG 检索出来的 chunk 会保留：
- `document_id`
- `report_period`
- `source_file`
- 页码等信息

#### 第二层：prompt 明确要求模型区分来源
在追问 prompt 里，我会明确要求模型：
- 如果 RAG 证据来自同一 entity 下的多份文档，回答时必须区分来源文件和报告期
- 如有页码，尽量在回答中体现依据
- 如果历史结论和当前文档原文冲突，以当前文档原文为准

这样做的目的，是在 prompt 层再次提醒模型不要把不同季度的信息混写在一起。

### 4. 这个问题在项目里有没有真实出现过？后来怎么处理的？
真实测试里这个问题确实出现过，尤其是在我用腾讯多季度财报做测试时。

一开始我的 chunk 切分和召回策略比较简单，容易出现两个问题：
- 多个季度的相关信息没有同时被有效召回
- 或者虽然是同一个 entity，但某些季度的信息刚好没进召回结果，导致比较结论不完整

后来我做了几件事去缓解这个问题：

1. 在 RAG 存储中显式保留报告期、document_id 等元信息  
2. 在检索时增加模式区分，明确当前文档检索和同实体多文档检索的使用场景  
3. 在 prompt 里强调区分来源文件和报告期  
4. 在调试页里直接查看 chunk 和检索结果，确认是否真的检索到了想要的季度内容

所以这个问题不是我事后想出来的，而是在真实测试中遇到过，然后逐步通过“元信息约束 + 模式区分 + prompt 提醒 + 调试验证”来改善的。

## 10. 你怎么验证你的 Session Memory、Research Memory 和 RAG 这三部分，是真的对回答有帮助，而不是只是“看起来接上了”？你可以从下面几个方面讲：1. 你做过哪些测试场景 2. 你怎么区分某次回答变好，是因为当前文档本身就有答案，还是因为 memory / RAG 真起作用了 3. 你在测试过程中发现过哪些问题 4. 后来又做了哪些调整

### 参考回答

我在这个项目里是做过比较系统的验证的，而且不是等全部做完才测，而是**每加一个模块就单独验证一次**。  
因为像 `Session Memory`、`Research Memory`、`RAG` 这种东西，最怕的就是“代码接上了，但实际对回答没帮助”，所以我比较重视可验证性。

### 1. 我做过哪些测试场景？
我主要用的是**同一家公司多个季度财报**来做测试，比如腾讯 2025 年不同季度的财报。  
之所以选这种数据，是因为它特别适合验证：

- Session Memory：能不能承接前后追问
- Research Memory：能不能提供跨会话、跨文档的历史背景
- RAG：能不能召回当前问题对应的原文证据

另外我也会针对每个模块设计一些比较有针对性的问题，而不是只问泛泛的总结。

### 2. 我怎么区分回答变好，到底是文档本身有答案，还是 memory / RAG 真起作用了？
这个我主要通过“模块拆开验证”和“场景对比”来判断。

#### Session Memory 的验证
我会在同一份文档上连续追问，比如：
- 先问整体结论
- 再基于上一轮回答追问某个业务点
- 再继续问某个风险或某个数字

如果系统能明显承接上一轮上下文，说明 Session Memory 起作用了；  
如果不加它，模型通常会更容易丢失上下文，或者回答变得泛化。

#### Research Memory 的验证
我会看：
- 在首轮分析或跨文档追问时，是否真的能注入历史背景
- 重启会话后，系统是否还能给出和同一 entity 历史研究相关的补充结论

这部分如果只是当前文档本身有答案，一般不会出现“跨会话后仍能补充历史背景”的效果，所以比较容易区分。

#### RAG 的验证
RAG 的验证会更直接一些，我会观察：
- 检索结果里有没有命中与 query 最相关的原文 chunk
- 回答里是否真的引用了当前文档或历史文档中的具体依据
- 没有命中时，回答是否明显变虚

也就是说，RAG 的验证更看“证据是否可追溯”，而不仅仅是回答表面上像不像对。

### 3. 我在测试过程中发现过哪些问题？
发现过几个比较典型的问题：

#### 第一，某些季度信息没有被正确召回
尤其是多季度对比时，如果 chunk 切得不理想，或者召回预算不够，就可能只命中两个季度，漏掉其他季度信息。

#### 第二，历史信息和当前文档容易混淆
如果没有足够明确的文档元信息和 prompt 约束，模型可能会把不同季度的结论混写。

#### 第三，Research Memory 过长或过杂会干扰当前分析
如果历史上下文过多，反而会让模型把重点从当前文档拉走。

### 4. 后来我做了哪些调整？
针对这些问题，我后面做了几类调整：

#### （1）增强调试能力
我专门做了调试页，分别查看：
- SQLite 里的 entity、analysis_record、memory_card
- Chroma 里的 card
- Milvus 里的 RAG chunk 和检索结果

这样我能直接看到：
- 数据到底有没有写进去
- 写进去的内容是什么
- 检索出来的是不是我预期的内容

#### （2）加强元信息控制
在 Research Memory 和 RAG 中都尽量保留 `entity_id`、`document_id`、`report_period` 等字段，降低跨文档信息混淆的风险。

#### （3）调整 RAG 策略和 chunk 组织
在真实测试中发现召回不完整后，我会结合调试页看 chunk 是否切得合理、某些季度内容是否根本没进入前几名召回结果。

#### （4）限制 Research Memory 的角色
我明确把它定位成“历史背景补充”，不让它替代当前文档原文证据，避免它喧宾夺主。

所以我对这个项目的验证思路不是“只看结果像不像”，而是：
**既看最终回答，也看存储层、检索层和 prompt 层是不是都按预期工作。**
