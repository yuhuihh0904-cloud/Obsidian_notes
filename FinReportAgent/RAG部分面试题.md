# RAG 模块 40 题标准答案

## 1. 你先整体介绍一下你这个项目里的 RAG 模块，核心职责是什么？

我这个项目里的 RAG 模块，本质上是一个“原文证据检索层”。  
它的核心职责不是替代 LLM 思考，而是在用户追问或者做跨季度分析时，根据当前 query 从上传过的 PDF 原文里召回最相关的片段，作为证据上下文补充给 LLM。

在我的实现里，RAG 不只支持当前文档检索，还支持同一研究对象下的跨文档检索，也就是 `current_document` 和 `entity_corpus` 两种模式。  
前者适合当前财报内部的细节问答，后者适合同一家公司多季度趋势对比。

它解决的核心问题是：LLM 自身上下文有限，不可能稳定记住所有历史原文内容，尤其在多文档、多季度场景下，更需要依赖 RAG 按 query 动态召回真实证据，提升回答的准确性和可解释性。

---

## 2. 你的 RAG 数据流是怎样的？请从 PDF 提取结果开始，一直讲到最后 prompt 中的证据上下文是怎么拼出来的。

我的 RAG 数据流可以分成两段：入库链路和检索链路。

入库时，先拿到 PDF 提取模块输出的 `cleaned_page_map`，然后基于文件名、文档类型和清洗后的页面文本生成稳定的 `document_id`。  
接着按页做固定窗口切块，每个 chunk 会带上 `document_id`、`entity_id`、`report_period`、`page_number` 等元数据。  
之后调用 embedding 模型把 chunk 文本转成向量，再把“向量 + 元数据 + 原文内容”一起写入 Milvus。

检索时，先对用户 query 做 query embedding，然后根据检索模式决定是按 `document_id` 过滤，还是按 `entity_id` 做跨文档召回。  
Milvus 返回相关 chunks 之后，我不会原样全塞给 LLM，而是再做去重、截断、格式化，最终整理成 `RAG Evidence Context`，再拼进 prompt 里。

这样做的好处是，既能控制 token，又能保留来源、页码、报告期这些关键信息。

---

## 3. 你这套 RAG 为什么要拆成 `chunking.py`、`embedding.py`、`rag_store.py`、`rag_service.py` 这几层？如果全写在一个文件里会有什么问题？

我把 RAG 拆层，主要是为了让不同职责稳定解耦。

- `chunking.py` 只负责把页面文本切成可入库的 `DocumentChunk`
- `embedding.py` 只负责向量化，并且通过 `EmbeddingProvider` 协议预留替换空间
- `rag_store.py` 只封装 Milvus 的 schema、插入、删除、检索和过滤表达式
- `rag_service.py` 负责把这些能力串起来，对上层暴露更高层接口

这样做有三个好处：

第一，某一层改动不会牵动全部逻辑。  
第二，后续替换 embedding 模型或者向量库时成本更低。  
第三，上层业务不需要理解 Milvus 或 embedding 细节，只调用 service 即可。

如果全写在一个文件里，随着降级逻辑、schema 校验、多检索模式、evidence formatting 不断增加，代码会迅速变得臃肿、耦合严重，也不利于维护和扩展。

---

## 4. `RagService` 在你的架构里扮演什么角色？为什么不让上层直接操作 Milvus？

`RagService` 在我这里相当于 RAG 子系统的统一编排层。  
它把切块、embedding、Milvus 入库、query 检索、evidence formatting 这些分散能力整合成两个更高层的能力：

- `index_pdf_result()`：负责把 PDF 提取结果写进向量库
- `build_evidence_context()`：负责从 query 直接生成可拼接到 prompt 的证据上下文

不让上层直接操作 Milvus，主要是因为底层细节很多，比如：

- collection 是否可用
- schema 是否匹配
- embedding provider 是否就绪
- query 为空怎么处理
- `entity_corpus` 缺少 `entity_id` 时如何回退
- 检索结果怎么压缩进 prompt

如果让业务层直接碰这些细节，代码会很脆弱。  
放到 `RagService` 里统一处理后，上层只需要关心“要不要用 RAG”和“用哪种检索模式”，整体鲁棒性会更好。

---

## 5. 你目前的 chunk 策略是什么？为什么是“按页 + 固定窗口字符切块”，而不是直接全文滑窗？

我当前的 chunk 策略是“按页遍历 + 固定窗口字符切块”，也就是在每一页内部按照 `chunk_size` 和 `chunk_overlap` 做滑窗，而不是拿整篇文档做全局滑窗。

这样设计主要有两个原因：

第一，金融财报天然带有比较清晰的页级结构，页码本身就是回答时的重要引用信息。  
第二，如果直接全文滑窗，很容易把两个逻辑上完全不相关的部分拼到一个 chunk 里，比如前面是业务概述，后面是风险提示，这会污染 embedding 语义。

所以我这里优先选择“页内切块”，本质上是用一点召回细粒度的损失，换取页码可追踪性和业务语义边界更稳定的好处。

---

## 6. 你的 `chunk_size=1200`、`chunk_overlap=200` 这两个参数是怎么考虑的？如果 overlap 太小或者太大，会分别带来什么问题？

这两个参数本质上是在召回粒度、语义完整性和 prompt 预算之间做平衡。

`chunk_size` 不能太大，否则一个 chunk 里会混入太多无关内容，而且多文档召回时很容易把 prompt 塞满；  
但也不能太小，否则一个财务指标的上下文会被切碎，embedding 语义会变差。

`chunk_overlap` 是为了减少边界截断问题。比如一段重要描述正好跨 chunk 时，适当 overlap 能保留连续性。

如果 overlap 太大，会造成 chunk 数量激增、内容重复严重，还会让检索结果更容易命中相似的重复片段。  
如果 overlap 太小，又会导致上下文断裂。

所以 1200/200 不是理论最优值，而是结合实际测试，在召回质量和 prompt 成本之间取的一个工程折中。

---

## 7. 你为什么强调“优先保留稳定的 page_number 信息”？页码信息在后续回答阶段具体有什么价值？

我会强调保留 `page_number`，因为在财报分析场景里，页码本身就是证据定位的一部分。  
RAG 的价值不只是把信息找回来，更重要的是告诉模型和用户“这段信息来自哪里”。

页码的价值主要有三层：

第一，增强回答的可解释性。  
第二，降低模型编造细节的风险。  
第三，方便用户回到原 PDF 做人工核查，尤其是做多季度对比时，可以快速定位到具体原文。

对于金融分析这种偏严肃场景，能不能把来源和页码讲清楚，影响的不只是体验，还影响结果的可信度。

---

## 8. 你觉得你当前这套 chunk 方案最大的缺点是什么？如果后续优化，你会优先改哪里？

我当前 chunk 方案最大的缺点是粒度还比较粗，因为本质上还是页内固定窗口切块，不是段落级或语义级切分。

这会带来两个后果：

第一，单个 chunk 里信息密度不够纯，容易混入不那么相关的内容。  
第二，当问题涉及多文档、多季度对比时，即使召回阶段能命中多个文档，后续 evidence context 也可能因为 chunk 太长而超出预算，导致部分重要证据没法真正进入 prompt。

我自己测试里就遇到过这个问题：查询涉及三个季度时，召回本身没有问题，但前两个季度的 chunk 太长，把 evidence window 塞满了，第三个季度虽然被召回，却没有成功拼进 prompt，最终模型看不到那部分内容。

如果后续优化，我会优先做两件事：

- 把切块从纯字符窗口升级成更细粒度的段落或语义切分
- 适当减小 chunk_size，并结合更精细的 evidence budget 分配策略

---

## 9. 你的 `document_id` 为什么要做成“稳定 ID”？你是怎么生成的？这样设计解决了什么问题？

我给文档做稳定 `document_id`，主要是为了解决“同一份文档重复上传或重复处理时，系统怎么识别它还是原来那份文档”的问题。

在我的实现里，`document_id` 不是随机生成的，而是基于文件名、文档类型和清洗后的页面文本内容做哈希生成。  
也就是说，同一份文档只要内容不变，生成出来的 `document_id` 就稳定。

这样做有两个直接价值：

第一，可以在入库前通过 `has_document(document_id)` 判断这份文档是不是已经入过库，避免重复写入。  
第二，在 `current_document` 检索模式下，`document_id` 也是一个关键的 metadata filter，用来把检索范围限制在当前文档。

---

## 10. `chunk_id` 为什么不是简单用自增 id，而是用 `document_id + page_number + chunk_index + content` 做 hash？

我没有给 chunk 用简单自增 id，而是用 `document_id`、`page_number`、`chunk_index` 和 `content` 组合后做哈希生成 `chunk_id`。

这样做的核心目的是让 chunk 的身份是“内容相关且稳定”的，而不是某次运行时临时分配的。

如果只用自增 id，那么同一份文档重新处理一次，chunk 的编号可能就变了，不利于去重、排查问题和保持幂等性。  
现在这种 hash 方式能让同一个 chunk 在多次运行下保持相对稳定，也方便后续在向量库里把它作为主键来存储和检索。

---

## 11. `DocumentChunk` 和 `RetrievedChunk` 为什么要分开建模？它们分别代表哪个阶段的数据？

我把 `DocumentChunk` 和 `RetrievedChunk` 分开建模，是因为它们代表的是 RAG 流程里的两个不同阶段。

`DocumentChunk` 是入库前的数据结构，重点是把一段原文和它对应的文档元数据组织好，便于 embedding 和写入 Milvus。  
`RetrievedChunk` 是检索阶段的数据结构，除了原始字段外，还会额外带上 `score`，表示这段内容和 query 的相似度。

分开建模的好处是语义更清晰，也避免把只在检索阶段才有的字段混进入库模型里，后续维护起来更干净。

---

## 12. 你在 chunk 元数据里保存了 `entity_id`、`report_period`、`report_year`、`report_type`，这些字段为什么重要？如果没有这些字段，你的多文档检索会有什么损失？

这些元数据对我的 RAG 很重要，因为它们不仅是展示信息，更是检索和推理时的上下文约束。

其中 `entity_id` 是 `entity_corpus` 模式的核心过滤条件，没有它就没法把检索范围限制到同一个研究对象。  
`report_period`、`report_year`、`report_type` 则是用来区分同一实体下不同报告来源和时间背景的。

如果没有这些字段，会有两个损失：

第一，多文档检索时无法准确约束范围。  
第二，模型即使召回了多条 chunk，也很难在回答时正确区分“这是哪个季度”“这是年报还是季报”。

所以这些字段既服务于检索过滤，也服务于回答阶段的来源区分。

---

## 13. 你的 embedding 层为什么要先抽象出一个 `EmbeddingProvider` 协议，而不是直接把 OpenAI 调用写死？

我先抽象了一个 `EmbeddingProvider` 协议，而不是把 OpenAI 兼容接口写死在业务里，主要是为了做依赖解耦。

对 `RagService` 来说，它只需要两个能力：

- 能批量向量化文本
- 能向量化 query

它并不关心底层到底是 OpenAI 兼容接口、别的云服务，还是本地 embedding 模型。

这样做的好处是后续更容易替换实现。  
比如我想从现在的 OpenAI 兼容 embedding 切到其他模型时，只需要新增一个 provider，而不用改 RAG 主流程代码。

---

## 14. `OpenAIEmbeddingProvider` 的 `is_ready` 机制有什么意义？为什么你要允许 RAG“自动降级”而不是直接报错退出？

`is_ready` 的作用是把“embedding provider 当前能不能正常提供向量化能力”变成一个显式状态，而不是等到真正发请求时才报错。

比如这些情况都可以在初始化阶段直接判断出来：

- 模型名没配
- openai SDK 没装
- embedding 配置不完整

允许自动降级，是因为 RAG 在我的项目里是增强能力，不是唯一能力。  
如果因为 embedding 配置缺失或者 Milvus 不可用，就让整个 Agent 直接崩掉，用户体验会很差。

现在的做法是：RAG 不可用时，返回“已自动降级”的结果，上层仍然可以继续靠当前文档文本、session memory 或 research memory 完成回答。  
这样系统整体鲁棒性更强。

---

## 15. 你的 `embed_texts()` 为什么要做 batch？如果不做 batch，在真实项目里会有什么问题？

`embed_texts()` 做 batch，主要是为了提高向量化阶段的吞吐效率。

因为一个 PDF 切完之后往往不是 1 个 chunk，而是一批 chunk。  
如果每个 chunk 都单独发一次 embedding 请求，网络往返次数会很多，整体延迟会明显变高。

做 batch 的好处有三个：

第一，减少接口调用次数。  
第二，提升吞吐和整体处理效率。  
第三，降低用户上传文档后的等待时间。

所以 batch 不是可选优化，而是实际工程里很常见的基础做法。

---

## 16. 为什么 `embed_query()` 最后还是复用了 `embed_texts([query])`？这种实现方式有什么好处？

我把 `embed_query()` 和 `embed_texts()` 分开暴露，是为了让接口语义更清晰：

- 一个面向单条查询
- 一个面向批量文本

但在实现上，`embed_query()` 直接复用 `embed_texts([query])`，是为了避免维护两套几乎相同的向量化逻辑。

这样做的好处是：

第一，query 和 chunk 文本走的是同一套底层 embedding 调用流程，行为更一致。  
第二，如果以后要改预处理逻辑，只要改一处即可。  
第三，代码更短、更稳定，不容易出现两套逻辑不一致的问题。

---

## 17. 如果 embedding 模型换了，最容易出什么线上问题？你现在的代码里做了什么保护？

如果 embedding 模型换了，最容易出的问题就是向量维度不一致。

因为 Milvus collection 在创建时，向量字段的维度已经固定了。  
如果新模型生成的 embedding 维度和旧 collection 不一致，就不能直接写进去，否则整个向量检索都会失效。

代码里做了两层保护：

第一，先校验本次 embedding 结果内部维度是否统一。  
第二，再检查它是否和已有 collection 的向量维度匹配。

一旦不一致，就直接抛异常，要求保持一致。  
这一步非常关键，因为维度不匹配不是“效果变差”，而是底层存储结构都对不上，不能凑合用。

---

## 18. 你为什么选 Milvus 做这个 RAG store？它在你的项目里承担什么职责？

我选 Milvus，主要是因为它更适合做一个相对独立、工程化一些的 RAG 检索层。

相比我在 research memory 里用的 Chroma，Milvus 在这些方面更完整：

- collection schema
- 索引参数
- 过滤表达式
- 向量检索能力
- 工程化配置

在我的项目里，Milvus 的职责不是做 embedding，而是负责存储 chunk 的向量和元数据，并在 query 来的时候执行向量相似度检索，再结合 metadata filter 返回相关原文片段。

也就是说，embedding 由外部 provider 生成，Milvus 负责“向量库”这一层。

---

## 19. 你这个 `MilvusRagStore` 除了 search 以外，还封装了哪些能力？为什么这些能力不能散落在业务代码里？

`MilvusRagStore` 除了 `search()`，还封装了很多底层能力，比如：

- collection 初始化
- schema 校验
- 向量维度校验
- chunk 插入
- 按 document 删除
- 判断文档是否已存在
- 统计 chunk 数量
- 列出某个 entity 下的 document_id
- 把业务层 filters 转成 Milvus 表达式

这些能力不能散落在业务代码里，主要是因为它们都属于“向量库适配层”的职责。

如果散在业务逻辑里，上层就要知道 collection 什么时候创建、维度怎么校验、过滤表达式怎么写，代码会很乱，而且一旦 Milvus 细节变化，上层业务也要一起改。

封装进 `rag_store` 后，上层只需要说“我要插入”或“我要搜索”，底层细节统一收口。

## 20. 你在建表时为什么把 `chunk_id` 设成主键，同时把 `document_id`、`entity_id`、`page_number`、`report_period` 这些字段也存进去？

这些字段的作用是不一样的。

`chunk_id` 是 chunk 级别的唯一身份，用来做主键和稳定标识。  
`document_id` 和 `entity_id` 是检索阶段最重要的过滤条件，分别对应当前文档检索和按实体跨文档检索。  
`page_number`、`report_period`、`report_year`、`report_type` 则属于文档背景信息，主要是为了让模型和用户知道这段证据来自哪一页、哪个报告期、什么类型的文档。

也就是说，这些字段既服务于“检索范围约束”，也服务于“回答时的来源解释”。  
如果不把这些元数据一起存入向量库，后面就很难做精确过滤和可解释回答。

---

## 21. 你的索引参数为什么是 `IVF_FLAT + COSINE`？`nlist`、`nprobe` 分别影响什么？

我这里用的是 `IVF_FLAT + COSINE`。

`COSINE` 指的是相似度度量方式，也就是看 query 向量和 chunk 向量方向有多接近，这比较适合 embedding 检索。  
`IVF_FLAT` 是一种近似检索索引。可以把它理解成先把向量空间粗略分桶，再去相关桶里找最相近的向量。

这里：

- `nlist` 决定分多少个桶，值越大，桶分得越细
- `nprobe` 决定查询时会探测多少个桶，值越大，召回更充分，但速度会更慢

所以这几个参数本质上是在“检索速度”和“召回质量”之间做折中。  
对我这个项目来说，数据量还不算特别大，所以我更偏向用稳妥参数，优先保证召回效果足够好。

---

## 22. `_build_filter_expr()` 的作用是什么？为什么向量检索还需要 metadata filter？

`_build_filter_expr()` 的作用，是把业务层传进来的 `document_id`、`entity_id` 这类过滤条件，转换成 Milvus 可以执行的表达式字符串。

这样 Milvus 在做向量检索时，不只是“看谁和 query 最像”，还会先把检索范围限制在某个文档或某个实体下。

之所以需要 metadata filter，是因为我的 RAG 不是一个全库无差别问答系统，而是有两种明确模式：

- 当前文档检索
- 同实体跨文档检索

如果不加这些过滤条件，系统就会退化成纯 query 相似度匹配，可能把别的文档里语义接近但业务无关的 chunk 召回出来。

---

## 23. 你的 `search()` 返回的 score 在这里代表什么？这个值能不能直接当“答案可信度”来理解？为什么？

这里的 score 代表的是 query 向量和 chunk 向量之间的相似度分数，本质上反映的是“这段文本和问题在语义空间里有多接近”。

但它不能直接当成“答案可信度”。

因为相似度高，只能说明这段文本大概率在讨论相关主题，不代表它一定包含问题的精确答案，更不代表模型基于它生成的最终回答就一定正确。

所以在我的项目里，score 更像“召回排序依据”，而不是“最终答案置信度”。

---

## 24. 你的项目里为什么要区分 `current_document` 和 `entity_corpus` 两种 retrieval mode？它们分别适合什么场景？

我区分 `current_document` 和 `entity_corpus`，是因为这两类问题的检索范围不一样。

`current_document` 适合当前这份文档内部的细节问答，比如某一页怎么描述现金流变化。  
`entity_corpus` 适合同一个研究对象的跨文档问题，比如同一家公司多季度利润率趋势如何。

如果不区分这两种模式，系统要么检索范围太小，回答不了跨文档问题；  
要么检索范围太大，当前文档问题又容易被其他文档干扰。

所以这本质上是按问题类型选择检索范围。

---

## 25. 你后来为什么要加 `entity_corpus` 的“分文档召回 + 合并排序”逻辑？它到底解决了什么问题？

我后来加“分文档召回 + 合并排序”，是为了解决 `entity_corpus` 模式下一个很实际的问题：

如果直接在同一个 entity 的所有文档上做全局 top-k，很容易出现某一个季度的多个 chunk 因为和 query 更像，把 top-k 全占满，其他季度一点都进不来。

但在多季度趋势分析里，我真正需要的是“每个相关文档都至少提供一点证据”，而不是只让一个季度特别强。

所以我先按 document_id 分别召回，每个文档保底拿若干条，再把这些候选结果合并、去重、按 score 重排。  
这样能显著提升多文档问题下的覆盖度。

---

## 26. 为什么 `entity_corpus_per_doc_top_k` 默认不是很大？这个参数如果调得过大，会有什么副作用？

`entity_corpus_per_doc_top_k` 默认不会设很大，是因为它的目标不是让每个文档拿很多结果，而是先保证“每个文档至少有代表性证据进入候选池”。

如果这个值设太大，多文档场景下候选 chunk 会迅速膨胀，后面即使合并排序了，也可能因为 evidence budget 有限，导致 prompt 里装不下，反而浪费上下文空间。

所以一般设成 1 或 2 就够了，重点是做覆盖，而不是做单文档深挖。

---

## 27. `_merge_entity_corpus_hits()` 为什么还要再做一次去重和重排？既然每个文档都已经搜过了，为什么还会重复？

这里的去重和重排有两个目的。

第一，虽然 `chunk_id` 是唯一的，但在多文档召回、分文档合并、以及 overlap 切块的场景下，候选结果里仍然可能出现内容高度重复的 chunk。  
比如同一页相邻 chunk 因为 overlap 带有大量重复内容，或者同一段话在不同文档版本里都出现了。

第二，分文档召回之后，每个文档先拿到的是“本地 top-k”，但这些结果放到一起后，仍然需要在全局范围内再按 score 做一次统一排序，最后只保留最值得拼进 prompt 的那几条。

所以这里的去重不是因为主键设计失败，而是因为检索后的候选池里仍然会有“内容层面的重复”和“全局排序”的需要。

---

## 28. 如果用户在 `entity_corpus` 模式下没传 `entity_id`，你的代码为什么要回退到当前文档检索，而不是直接报错？

因为从用户视角看，没传 `entity_id` 不一定是一个需要终止流程的致命错误，很多时候只是上层没把信息补全。

这种情况下，如果直接报错，会让整个检索链路中断；  
而回退到 `current_document`，至少还能保住当前文档内的检索能力。

所以这个设计本质上是“优雅降级”：优先保证系统还能给出尽可能有用的结果，而不是因为一个条件缺失就完全失败。

---

## 29. 你为什么不把检索出来的 chunk 原封不动全塞进 prompt，而是要做 `format_evidence_context()`？

检索结果不能原样塞进 prompt，主要有三个原因。

第一，Milvus 返回的是结构化检索结果，不是 LLM 最容易理解的自然语言证据块，需要整理成统一的 evidence context 文本。  
第二，我希望模型回答时带上来源和页码，所以必须把 `source_file`、`report_period`、`page_number` 这些 metadata 一起组织进去。  
第三，prompt 是有预算的，不能把所有命中的 chunk 全塞进去，所以还要做截断、去重和压缩。

因此 `format_evidence_context()` 其实是在做“把检索结果转成 LLM 可消费证据”的最后一步。

---

## 30. `evidence_max_chars` 这个预算控制解决了什么问题？如果没有它，会发生什么？

`evidence_max_chars` 解决的是 prompt 预算分配问题。

RAG 虽然能召回很多原文，但真正能送进 LLM 的上下文是有限的。  
如果不设这个预算，最直接的问题是 prompt 可能过长，甚至超出模型上下文上限。  
再进一步，即使没超上限，RAG 也可能挤占掉 session memory、research memory、系统指令这些同样重要的上下文。

在我的项目里，RAG 只是一个证据补充层，不应该独占 prompt，所以必须用 `evidence_max_chars` 控制它最多占多少空间。

---

## 31. 你在 evidence context 里为什么要保留 `source_file`、`document_id`、`report_period`、`report_type`、`page_number`、`chunk_index` 这些信息？

因为模型最后不是只要“有内容”，而是要“知道这段内容来自哪里”。

在我的项目里，很多问题是跨季度、跨文档的。  
如果 evidence context 里只有原文片段，没有来源文件、报告期、页码这些信息，模型很容易把不同季度的内容混在一起。

把这些字段一起放进去，相当于给每段证据都加了标签。  
模型在生成答案时就更容易区分这是哪份文档、哪个报告期、哪一页的内容，也更方便用户回查原文。

---

## 32. 你这里为什么还要对 retrieved content 再做一次 normalize 和 dedupe？不是入库前已经清洗过了吗？

这里再做一次 normalize，是因为入库前清洗和拼 prompt 前清洗的目标不完全一样。

前面的 normalize 主要是为了生成更干净的 embedding 输入，减少噪声空白对向量化的影响。  
后面的 normalize 则是为了让最终喂给 LLM 的 evidence context 更规整，减少多余空白和换行，节省字符预算。

而 dedupe 的目的，是避免把重复或高度相似的 chunk 一起拼进 prompt，浪费本来就有限的 evidence budget。  
尤其在 overlap 切块和多文档合并场景里，这一步非常有必要。

所以可以这样理解：

- 前面的 normalize：面向 embedding 输入
- 后面的 normalize：面向 prompt 输入
- dedupe：面向证据冗余控制

---

## 33. 你的 RAG 在哪些情况下会自动降级？为什么你认为这种设计比直接抛异常更合理？

我的 RAG 在这些情况下会自动降级：

- RAG 总开关关闭
- Milvus 当前不可用
- embedding provider 未就绪
- query 为空
- query embedding 为空
- `entity_corpus` 缺少必要条件时无法正常检索

之所以设计成自动降级，而不是直接抛异常，是因为 RAG 在我的项目里是增强能力，不是唯一能力。  
如果只因为某个依赖没准备好，就让整个 Agent 崩掉，用户体验会很差。

自动降级的价值在于：即使 RAG 不可用，系统仍然可以依赖当前文档文本、session memory、research memory 等其他模块继续完成回答，整体鲁棒性更强。

---

## 34. `index_pdf_result()` 为什么要先检查 `has_document(document_id)`？这和 overwrite 机制分别解决什么问题？

先检查 `has_document(document_id)`，主要是为了避免同一份文档被重复写入向量库。

因为我的 `document_id` 是稳定生成的，所以如果用户重复上传同一份文档，系统可以识别出它已经存在，然后直接跳过入库，避免重复数据堆积。

而 overwrite 机制解决的是另一类问题：  
如果我明确希望“用这次处理结果覆盖旧数据”，比如文档内容更新了，或者我要重新切块、重新 embedding，这时就可以先删除旧文档对应的 chunk，再把新的写进去。

所以两者的区别是：

- `has_document()`：解决重复入库问题
- `overwrite`：解决主动重建或更新问题

---

## 35. 如果 Milvus collection 的 schema 和当前代码版本不一致，你的系统是怎么处理的？为什么必须强校验？

如果 Milvus collection 的 schema 和当前代码版本不一致，我的系统会直接报错，而不是继续凑合运行。

这是因为我的 RAG 很依赖一些明确字段，比如：

- `chunk_id`
- `document_id`
- `entity_id`
- `report_period`
- `report_year`
- `report_type`
- `page_number`
- `chunk_index`
- `content`
- `created_at`
- 向量字段

如果 schema 里缺了这些字段，哪怕检索不报错，返回结果也可能不完整，尤其会直接影响多文档 entity 级检索和 evidence formatting。

所以这里必须强校验。  
它不是“代码洁癖”，而是为了保证检索语义和数据结构的一致性。  
如果 schema 对不上，最合理的做法就是提示删除并重建 collection，而不是默默带着隐患继续跑。

---

## 36. 如果新 embedding 模型的维度和旧 collection 不一致，为什么不能“凑合着继续用”？

不能凑合用，因为 Milvus collection 的向量字段维度是在 schema 里固定定义好的。

如果新 embedding 模型输出的向量维度和旧 collection 不一致，会出现两个问题：

第一，底层存储结构不匹配，数据根本没法正确写入。  
第二，即使强行混在一起，也无法保证检索的数学意义成立，整个相似度搜索都会失效。

所以这不是“效果差一点”的问题，而是系统从底层就不兼容。  
正确做法只能是保持 embedding 维度与 collection 一致，必要时重建 collection。

---

## 37. 你觉得你这套 RAG 当前最大的瓶颈是什么：切块策略、top-k、embedding 质量、还是 prompt budget？为什么？

如果只选一个，我认为当前最大的瓶颈是“切块策略和 prompt budget 叠加造成的 evidence packing 问题”。

原因是：

- embedding 和 Milvus 检索本身已经能召回到相关 chunk
- 真正的问题往往出现在“召回后怎么把多文档证据高效塞进 prompt”

尤其在多季度对比场景下，如果 chunk 太粗，即使 top-k 命中了多个季度，最终 evidence context 也可能因为字符预算有限，只装下前几个 chunk，后面的关键信息进不去，模型就等于没看到。

所以从当前项目阶段来看，最值得继续优化的不是先换 embedding，而是让切块更细、证据压缩更高效。

---

## 38. 如果让你继续优化这套 RAG，但要求“最小改动”，你会怎么做？

如果要求最小改动，我不会一下子大改架构，而会优先做三件事：

第一，优化 chunk 切分粒度。  
在保留现有按页逻辑的前提下，尝试更细一些的段落级切分，或者适当减小 `chunk_size`。

第二，优化 evidence packing。  
比如多文档场景下，不只是按全局分数截断，而是尽量保证每个相关文档都至少保留一条高价值证据。

第三，继续调 `top_k`、`entity_corpus_per_doc_top_k`、`evidence_max_chars` 这些参数。  
这些属于低成本但效果可能很明显的调优点。

也就是说，我的思路不是先做大重构，而是先围绕“多文档证据怎么更高效进入 prompt”做最小代价优化。

---

## 39. 如果面试官质疑：‘你这个 RAG 不是标准知识库问答，更像财报片段检索’，你会怎么回应？

我会说这恰恰是我这个项目的业务特征，而不是缺点。

标准知识库问答更强调“知识库覆盖面”和“通用问答能力”，而我的场景是财报分析，重点不是开放域问答，而是让模型在用户追问时，能够基于上传的原始金融文档做有来源的证据检索和回答。

所以从系统设计上看，我更关注的是：

- 当前文档和同实体历史文档的范围控制
- 证据片段的可解释性
- 来源、报告期、页码的保留
- 多季度比较下的召回覆盖度

换句话说，我这个 RAG 不是在做通用百科问答，而是在做“面向金融文档分析的证据增强型检索”。  
它的评价标准也不是开放域知识覆盖，而是能不能稳定支持多文档比较、追问和原文回溯。

---

## 40. 你认为这套 RAG 在你的 FinReportAgent 项目里，最有价值的项目亮点是什么？

我认为这套 RAG 最有价值的项目亮点有三个。

第一，不只是当前文档检索，还支持 `entity_corpus` 的跨文档按实体检索。  
这让系统可以回答同一家公司多季度趋势变化这类问题，而不是只能做单篇文档问答。

第二，我不是简单做全局 top-k，而是针对多文档场景专门设计了“分文档召回 + 合并排序”策略。  
这解决了一个很实际的问题：全局 top-k 容易被某一个季度挤满，导致其他季度没有证据进入 prompt。

第三，我把 RAG 做成了一个可解释的证据层，而不是只返回一堆相似文本。  
最终 evidence context 里会明确保留来源文件、报告期、页码、chunk 索引，模型回答时可以尽量带出处，这对金融分析这种偏严肃场景非常重要。

所以我认为它最大的亮点不是“用了向量库”，而是它确实围绕财报分析场景做了针对性的检索设计和工程权衡。
