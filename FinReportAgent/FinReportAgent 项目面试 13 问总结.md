## 1. 项目整体介绍

请你用 1 分钟左右介绍一下这个 FinReportAgent 项目：

- 核心功能有哪些？

### 标准回答

我这个项目是一个面向金融文档分析的 Agent 助手，主要处理公司财报、券商研报、行业报告等 PDF 文档。用户上传 PDF 后，系统会自动进行文本抽取、清洗、切块，并根据文档类型生成结构化中文分析报告。报告生成后，用户还可以继续追问，追问既可以基于当前文档，也可以基于同一公司历史文档进行跨文档检索，最后报告支持 Markdown 格式导出。

第一是 **PDF 预处理模块**，解决原始 PDF 文本噪声多、格式混乱、上下文过长的问题，通过文本清洗、页码控制和 chunk 切分提升后续分析质量。

第二是 **记忆与检索模块**，我设计了 Session Memory、Research Memory 和 RAG 原文检索三层上下文。Session Memory 保留当前对话上下文，Research Memory 存储历史分析沉淀，RAG 负责检索 PDF 原文证据。

第三是 **Agent 工具调用模块**，通过 Function Calling / ReAct 方式接入新闻搜索、行情查询等工具，让模型在分析过程中可以按需获取外部信息。

相比普通 PDF 总结工具，这个项目不只是单文档总结，而是支持当前文档问答、历史研究复用和同实体跨文档证据检索，能让金融分析更连续，也更有依据。

### 评价

- 技术模块：PDF 处理、Memory、RAG、Function Calling；
- 差异化亮点：不是普通 PDF 摘要，而是支持历史研究复用和跨文档分析。

## 2. PDF 处理模块

### 面试问题

金融 PDF 直接提取文本会遇到哪些问题？你在项目里具体做了哪些清洗和切分策略？为什么这些策略能提升后续 LLM 分析效果？

### 标准回答

金融 PDF 直接抽取文本主要有几个问题。

第一是噪声很多，比如页眉页脚、目录、免责声明、公司简介、重复标题等，这些内容信息密度低，但会占用上下文窗口。

第二是格式比较乱，可能存在多余空格、换行错乱、表格文本顺序异常等问题。

第三是长文档容易超出模型上下文，如果不做切分，会影响后续检索和生成质量。

我在项目里主要做了三类处理。

首先是基础清洗，比如去除多余空格、合并连续空行、过滤页眉页脚和免责声明等低价值内容。

其次是按页抽取文本，并保留文档 ID、页码等元数据，然后在页内使用固定窗口进行 chunk 切分，窗口大小大概是 1200 字符，overlap 设置为 200。这样既能保证 chunk 不会太长，也能通过 overlap 缓解跨页或跨段落的信息断裂问题。

另外，对于特别长的文档，如果清洗后的字符数仍然超过预算，我会按段落做一个简单的信息价值评分，优先保留包含财务指标、同比环比、营收利润、现金流、风险因素、管理层讨论，以及具体数字和百分比的段落。

这些策略的核心目的，是提高进入向量库的 chunk 质量。因为 RAG 后续召回的不是整篇 PDF，而是具体 chunk。如果 chunk 里混入太多免责声明、背景介绍或者重复页眉，就会降低 embedding 表达质量，导致检索结果不准，最终影响 LLM 的分析质量。

### 评价

这一题回答得比较扎实，体现了真实项目经验。

你答到了：

- 低价值信息过滤；
- 页眉页脚、免责声明；
- 空格、换行清洗；
- chunk size = 1200；
- overlap = 200；
- 段落打分；
- 高价值数字保留；
- chunk 质量会影响 RAG 召回。

### 需要优化的点

“按页切 + 滑动窗口切分”容易被追问，所以最好说得更严谨：

> 先按页抽取并保留页码元数据，再在页内按固定窗口切分。

### 面试金句

**PDF 清洗不是为了让文本更好看，而是为了提高进入向量库的 chunk 质量；chunk 质量直接决定后续 RAG 召回质量。**

## 3. RAG 和 Research Memory 的区别

### 面试问题

RAG 原文检索和 Research Memory 听起来都像是在检索历史信息，你怎么区分它们？为什么不只用一个向量库解决所有问题？

### 标准回答

我是这样区分 RAG 和 Research Memory 的：**RAG 检索的是 PDF 原文证据，Research Memory 检索的是历史分析沉淀。**

RAG 主要面向文档原文。PDF 清洗和切块后，我会把 chunk 连同 `document_id`、`entity_id`、页码、报告期等元数据写入 Milvus。用户追问时，RAG 可以分为两种模式：一种是当前文档检索，回答当前 PDF 里的问题；另一种是同实体跨文档检索，比如检索同一家公司多个季度的财报内容，方便做趋势对比。RAG 返回的是原始文本片段，主要作用是提供可追溯的证据。

Research Memory 则是分析完成后的长期记忆。每次生成报告后，系统会从分析结果里抽取 Summary、Insight、Risk 三类 Memory Card，并绑定 `entity_id`、`document_id`、`report_period` 等元数据，向量部分存到 Chroma，结构化元数据存到 SQLite。后续再分析同一家公司时，可以把相关 Memory Card 召回进 prompt，补充历史背景、风险变化和过往分析结论。

我没有把两者简单合并到一个向量库里，是因为它们的粒度和用途不同。RAG 是细粒度原文证据，更适合回答“报告里具体说了什么”；Research Memory 是高层语义沉淀，更适合回答“这家公司历史上有什么趋势和风险”。如果混在一起，容易导致召回结果粒度混乱，也会影响答案的可追溯性。

### 评价

这一题是 13 个问题里答得比较好的一题。

你的关键表达是：

> Research Memory 是历史实体记忆，RAG 是文档原文召回。

这说明你能区分两类检索系统的职责。

### 需要优化的点

一定要主动回答 **为什么不只用一个向量库**：

- RAG 和 Memory 粒度不同；
- RAG 是原文证据，Memory 是分析沉淀；
- RAG 用于可追溯回答，Memory 用于背景增强；
- 混在一起容易召回粒度混乱。

### 面试金句

**RAG 检索的是原文证据，Research Memory 检索的是历史分析沉淀；前者保证可追溯，后者增强连续分析能力。**

## 4. Research Memory 的生成和防幻觉

### 面试问题

Research Memory 中 Summary / Insight / Risk 三类 Memory Card 是怎么生成的？它们分别解决什么问题？怎么避免错误信息或幻觉内容长期保存？

### 标准回答

Research Memory 是在单篇文档分析完成之后生成的。系统先基于 PDF 原文生成结构化分析报告，然后再用一个较小的 LLM 对报告内容做二次抽取，生成多张 Memory Card。每张 card 都会绑定 `entity_id`、`document_id`、`report_period`、card 类型等元数据，方便后续按公司和报告期召回。

我主要设计了三类 card。

第一类是 **Summary Card**，用来记录某个章节或某个时期的核心结论，比如收入增长放缓、利润率改善、现金流承压等。它不会存储特别细的原始数值，而是存储高层趋势。

第二类是 **Insight Card**，用来记录积极信号或者有价值的业务洞察，比如某个业务线增长较快、成本结构优化、战略方向变化等。

第三类是 **Risk Card**，用来记录潜在风险，比如利润下滑、监管压力、行业竞争加剧、现金流压力等。

为了避免错误信息长期进入记忆，我做了几层控制。

第一，生成 card 的 prompt 会明确要求模型只能基于当前报告内容抽取，不允许引入外部猜测。

第二，card 会绑定来源文档、报告期等元数据，后续召回时可以知道它来自哪篇文档。

第三，Research Memory 只作为背景增强，不作为最终事实依据；如果用户问具体财务数据，仍然优先通过 RAG 检索原文 chunk。

第四，召回排序时会考虑相似度、实体匹配和时间衰减，较旧的 card 权重会降低，避免过期信息长期污染上下文。

所以我对 Research Memory 的定位不是“事实数据库”，而是“历史分析背景库”。关键事实还是以 RAG 原文证据为准。

### 评价

这一题回答中等偏好。

优点是你能讲出：

- card 的生成时机；
- Summary / Insight / Risk 三类 card；
- 小模型二次总结；
- prompt 限制；
- 遗忘因子 / 时间衰减。

但“防幻觉”部分需要加强。

### 需要优化的点

只靠 prompt 和遗忘因子不够。因为：

- prompt 不能完全防止幻觉；
- 遗忘因子只能缓解过期信息，不能处理刚生成的错误信息。

所以一定要补充：

- card 绑定来源文档；
- 关键事实回查 RAG；
- Memory 只做背景，不做最终事实依据；
- Memory 和 RAG 冲突时，以 RAG 为准。

### 面试金句

**Research Memory 不是事实数据库，而是历史分析背景库；关键事实必须回到 RAG 原文证据中验证。**

## 5. Memory 和 RAG 冲突怎么办

### 面试问题

假设 Research Memory 里记录了“公司利润率持续改善”，但当前文档 RAG 检索到的原文显示“本季度利润率明显下降”，系统应该怎么处理？如何避免 LLM 被旧记忆误导？

### 标准回答

如果 Research Memory 和 RAG 发生冲突，我会以 RAG 检索到的当前文档原文为准。因为 RAG 返回的是带有 `document_id`、`page_num`、`report_period` 等元数据的原始证据，而 Research Memory 是历史分析沉淀，本身存在时效性，不能作为最终事实来源。

在设计上我会做几层控制。

第一，在 prompt 中把 RAG 和 Research Memory 分成两个区域，明确 RAG 是 primary evidence，Memory 只是 auxiliary background。

第二，Research Memory 召回时会加入时间衰减，较旧的 card 权重更低，尤其是金融指标、经营趋势类信息，过旧内容只能作为历史背景。

第三，回答时要求模型优先引用 RAG 的页码和文档信息。如果 Memory 与当前原文不一致，需要明确说明这是历史结论与当前报告之间的变化。

比如 Memory 里记录过去利润率持续改善，但当前季度 RAG 原文显示利润率下降，那么系统不应该继续说利润率改善，而应该回答：历史上利润率曾改善，但当前报告显示本季度利润率下降，说明趋势可能出现反转。这样既利用了历史记忆，又不会被旧记忆误导。

### 评价

这一题你答对了核心原则：

> 以 RAG 为准。

但回答略短，最好补充系统层面的设计。

### 需要优化的点

不要只说“prompt 里规定”，还要说：

- prompt 明确分区；
- RAG 是 primary evidence；
- Memory 是 auxiliary background；
- Memory 加时间衰减；
- 冲突时不要简单忽略 Memory，而是描述为“历史趋势发生变化”。

### 面试金句

**当 Memory 和 RAG 冲突时，以当前 RAG 原文证据为准；Memory 只能作为历史背景，不能覆盖当前事实。**

## 6. 跨文档检索执行链路

### 面试问题

假设用户上传了腾讯 Q1、Q2、Q3 三份财报，然后问：“腾讯最近几个季度的广告业务变化趋势怎么样？”系统从用户提问到最终回答会经过哪些步骤？

### 标准回答

如果用户上传了腾讯 Q1、Q2、Q3 三份财报，然后问“腾讯最近几个季度的广告业务变化趋势怎么样”，系统首先会对 query 做解析，识别出实体是腾讯，同时识别出子主题是广告业务。实体识别后会做归一化，比如把“腾讯”“腾讯控股”“Tencent”统一映射到同一个 `entity_id`。

在 Research Memory 层面，系统会用这个 `entity_id` 作为过滤条件，到 Chroma 中做语义检索，召回和腾讯广告业务相关的 Summary、Insight、Risk card，拿到 `card_id` 后再去 SQLite 读取完整 card 内容和结构化元数据，比如 `document_id`、`report_period`、card 类型等。

在 RAG 层面，系统会同样用 `entity_id` 限定检索范围，然后根据“广告业务变化趋势”这个 query 去 Milvus 里检索相关原文 chunk。因为这个问题是多季度趋势分析，所以不能只按相似度取 top-k，否则可能召回结果都来自某一个季度。我会在 rerank 或上下文组装阶段考虑文档覆盖，尽量保证 Q1、Q2、Q3 每个报告期都有相关 chunk 进入 prompt，同时保留 `document_id`、页码和报告期信息。

最后在 prompt 组装时，我会把 RAG 原文证据作为主证据，把 Research Memory 作为历史背景，并要求模型按报告期进行对比回答，比如分别总结 Q1、Q2、Q3 广告业务的表现，再归纳整体趋势。如果上下文超过限制，会优先保留包含广告收入、同比环比、管理层解释等高价值信息的段落。

- Chroma 召回 card_id；
- SQLite 取 card；
- RAG 按实体过滤；
- 多季度检索；
- prompt 窗口限制；
- rerank 保证文档覆盖；
- 段落打分压缩。

### 需要优化的点

要补充 **实体归一化** 和 **主题识别**：

- “腾讯”“腾讯控股”“Tencent”“00700.HK”统一成同一个 `entity_id`；
- “广告业务”作为 topic，用于语义召回和关键词增强。

### 面试金句

**跨文档检索不是简单扩大 top-k，而是要先做实体过滤，再做主题召回，最后在 rerank 和 prompt 组装阶段保证时间维度覆盖。**

## 7. 多文档检索的召回偏置问题

### 面试问题

如果用户问 Q1、Q2、Q3 的趋势，但向量检索 top-k 结果大部分都来自 Q3，因为 Q3 文档和 query 最相似，你怎么保证 Q1、Q2 也能被召回？

### 标准回答

这个问题本质上是多文档检索里的召回偏置。如果只按全局 top-k 检索，确实可能因为 Q3 和 query 最相似，导致 top-k 大部分都来自 Q3，从而让 Q1、Q2 没有证据进入 prompt。

我的设计是引入一个 `at_least_per_doc_topk` 机制。对于趋势类问题，系统会先按照 `document_id` 或 `report_period` 分组检索，保证每个文档至少召回少量 chunk，比如每个季度先召回 1 到 2 个候选 chunk。这个 k 不会设置太大，它主要是兜底，目的是保证时间维度覆盖。

然后在 rerank 阶段再做全局排序，同时设置最低相关性阈值。如果某个季度召回的 chunk 和用户 query 相关性实在太低，比如完全没有提到广告业务，就不会为了覆盖而强行塞进 prompt。最后在 prompt 预算有限的情况下，会优先保留每个季度最相关的 chunk，再根据相似度和信息密度补充其他高分 chunk。

这样既能避免 top-k 被单个季度垄断，又能避免低质量证据污染上下文。

### 评价

这是 13 个问题里回答比较亮眼的一题。

你的 `at_least_per_doc_topk` 设计很工程化，而且你还提到了最低相关性阈值，说明你知道不能为了覆盖而强行塞无关内容。

### 需要优化的点

可以把流程说得更完整：

1. 按文档 / 报告期分组召回；
2. 每个文档至少保留少量候选 chunk；
3. 全局 rerank；
4. 设置最低相关性阈值；
5. prompt 预算不足时优先保留每个季度最相关 chunk。

### 面试金句

**趋势类问题要优化的是时间覆盖率，不能只优化语义相似度；但时间覆盖也要受相关性阈值约束，不能强行召回无关证据。**

## 8. Agent 工具调用设计

### 面试问题

你的 Agent 是怎么判断什么时候需要调用工具的？比如用户问“帮我分析腾讯最新的投资价值”，系统怎么决定是直接基于 PDF 回答，还是调用行情、新闻搜索这些外部工具？

### 标准回答

我的设计里，工具调用不是默认强制执行，而是由 Agent 根据用户问题和当前上下文是否足够来决定。系统会先基于当前文档、Research Memory 和 RAG 检索结果构造上下文。如果这些信息已经能回答用户问题，比如用户只是问当前 PDF 里的某个财务指标，就不需要调用外部工具。

但如果用户的问题涉及实时性或外部信息，比如问“腾讯最新的投资价值”“最近有没有重大新闻”“当前股价表现如何”，或者当前 PDF 的报告期比较旧、RAG 召回结果不足，Agent 就会倾向于调用新闻搜索、行情查询等工具。

工具调用完成后，返回结果会作为 Observation 加入当前上下文，再由 LLM 综合 PDF 原文证据、历史 Research Memory 和外部工具结果生成最终回答。这样既避免了每次都调用工具带来的成本，也能在需要实时信息时补充 PDF 之外的信息。

这一题回答方向正确。

你答到了：

- 先看 RAG / Memory 是否足够；
- 如果不确定或需要实时信息，再调用工具；
- 新闻、行情工具用于补充外部信息。

| 问题涉及外部事件 | 监管影响 / 行业变化 | 新闻工具 |

## 9. ReAct 和 Function Calling 的区别

### 面试问题

你这个项目里如果说是基于 ReAct / Function Calling 的 Agent，那你怎么理解 ReAct 和 Function Calling 的区别？它们分别解决什么问题？

### 标准回答

我理解 Function Calling 和 ReAct 是两个不同层级的概念。

Function Calling 更偏底层能力，它解决的是 LLM 如何以结构化方式表达工具调用意图。模型本身不会真正执行工具，它只是输出要调用哪个函数以及对应参数，真正执行函数的是 Agent 框架或业务代码。

ReAct 则是一种 Agent 工作范式，它强调 Reasoning 和 Acting 交替进行。也就是模型先根据当前问题进行 Thought，判断是否需要调用工具；如果需要，就执行 Action；工具返回结果作为 Observation 再进入上下文；模型再基于新的观察继续推理，直到生成最终答案。

在我的项目里，Function Calling 主要提供稳定的工具调用接口，比如新闻搜索、行情查询、RAG 检索等；ReAct 则负责组织整个分析流程，让 Agent 能够根据用户问题、PDF 内容、检索结果和工具返回结果，逐步完成金融分析任务。

所以可以简单理解为：**Function Calling 解决“怎么调用工具”，ReAct 解决“什么时候调用工具以及调用后如何继续推理”。**

### 评价

这一题回答不错。

你答到了：

- Function Calling 是底层工具调用；
- ReAct 是 Agent 范式；
- LLM 不是真正执行工具，而是输出调用意图；
- ReAct 是 Thought → Action → Observation 循环。

### 需要优化的点

补充一点：

> ReAct 不一定必须依赖 Function Calling，也可以通过文本 Action 解析实现；Function Calling 只是更稳定、更结构化的工具调用接口。

### 面试金句

**Function Calling 是工具调用接口，ReAct 是推理-行动循环。前者解决工具调用的结构化和可靠性，后者解决复杂任务中的多步决策和动态调整。**

## 10. Agent 完整执行链路

### 面试问题

假设用户上传一份腾讯财报，然后输入：“结合这份财报和近期市场信息，帮我分析腾讯当前的投资价值。”请按完整链路讲一下：从用户输入开始，到 Agent 生成最终 Markdown 报告，中间经过哪些模块？每一步输入输出是什么？

### 标准回答

可以，我会按五个阶段来讲：文档处理、检索增强、工具调用、报告生成和记忆写回。

第一阶段是 **文档处理**。用户上传腾讯财报后，系统会先用 PDF 解析模块提取文本，然后做清洗，比如去除页眉页脚、免责声明、重复空格和低价值段落，得到 `clean_text`。之后按页保留页码信息，并在页内做 chunk 切分，每个 chunk 会绑定 `document_id`、`entity_id`、`page_num`、`report_period` 等元数据，然后写入 Milvus。

第二阶段是 **检索增强**。系统会解析用户 query，识别实体是腾讯，主题是投资价值，同时因为问题要求结合当前财报，会从 Milvus 中检索当前文档里与收入、利润、现金流、业务增长、风险因素、管理层展望相关的原文 chunk。另一方面，系统也会用 `entity_id` 过滤 Research Memory，在 Chroma 里召回相关 Memory Card 的 `card_id`，再到 SQLite 里取出完整的 Summary、Insight、Risk card，并整理成适合放入 prompt 的自然语言背景。

第三阶段是 **工具调用**。因为用户明确说要结合“近期市场信息”，所以 Agent 会调用外部工具，比如新闻搜索工具和行情查询工具，获取腾讯近期新闻、市场变化或股价表现。工具返回结果会作为 Observation 加入当前上下文。

第四阶段是 **prompt 组装和报告生成**。系统会把 RAG 原文证据作为 primary evidence，把 Research Memory 作为 historical background，把新闻和行情作为 external context，然后按照投资分析报告模板组织 prompt，要求模型从基本面、业务表现、风险因素、近期市场变化和投资价值判断几个方面生成 Markdown 报告。

第五阶段是 **记忆写回和导出**。报告生成后，系统会从报告中抽取 Summary、Insight、Risk 三类 Memory Card，绑定当前 `entity_id` 和 `document_id` 后写入 Chroma 和 SQLite，方便后续同一家公司跨文档分析复用。最后用户可以把生成的 Markdown 报告下载到本地。

### 评价

这一题整体链路完整。

你能讲出：

~~~text
PDF 清洗
→ chunk 入库
→ Research Memory 召回
→ RAG 召回
→ 外部工具搜索
→ prompt 组装
→ LLM 生成报告
→ Memory 写回
→ Markdown 下载
~~~

这说明你对系统主流程比较熟。

### 需要优化的点

真实面试里不要说：

> 因为前面讲过，所以这里省略。

面试官问完整链路，就是想看你能不能重新组织系统流程。

建议开头固定为：

> 我按五个阶段讲：文档处理、检索增强、工具调用、报告生成、记忆写回。

### 面试金句

**完整链路可以概括为：文档处理负责把 PDF 变成高质量证据，检索增强负责找出相关上下文，工具调用负责补充外部实时信息，LLM 负责生成结构化报告，最后 Memory 写回支持后续连续分析。**

## 11. Prompt 组装策略和上下文裁剪

### 面试问题

RAG、Research Memory、外部工具结果、Session Memory 都可能很长，你是怎么组织 prompt 的？不同类型信息的优先级是什么？如果上下文超长怎么裁剪？

### 标准回答

我在 prompt 组装时会把不同信息明确分区，避免所有上下文混在一起。整体会包括 system prompt、用户当前 query、RAG 原文证据、外部工具结果、Session Memory 和 Research Memory。

在优先级上，我会把系统规则和用户当前问题放在最前面，然后是 RAG 原文证据，因为金融分析里最重要的是当前文档的事实依据。其次是外部工具返回的新闻、行情等结果，特别是用户问到“最新”“近期市场信息”时，这些工具结果也属于重要证据。Session Memory 主要保证当前对话连贯性，但不能覆盖 RAG 事实。Research Memory 是历史背景，只作为辅助参考。工具调用的中间日志优先级最低，一般只保留最终结果，不保留完整过程。

如果上下文超长，我会分层裁剪。

第一，Session Memory 只保留最近几轮对话，较早内容做摘要压缩。

第二，Research Memory 只保留和当前实体、主题最相关的 card，并做时间衰减。

第三，工具调用部分只保留最终 observation，比如新闻标题、摘要、时间和来源，不保留调用日志。

第四，RAG 不会随意摘要成新结论，因为它是主证据，但会通过 rerank、去重、相关性阈值、按报告期覆盖和高价值段落筛选来控制长度。

总体原则是：**事实证据优先保留，历史背景和对话上下文按需压缩，工具日志尽量裁剪。**

### 评价

这一题你有工程意识，但原回答里优先级有问题。

你原来认为：

~~~text
Session Memory > RAG > Research Memory > Function Calling
~~~

这个在“对话连贯性”上能理解，但在金融事实分析里不够准确。

更推荐的优先级是：

~~~text
System Prompt / 用户当前 Query
> RAG 原文证据
> 外部工具最终结果
> Session Memory
> Research Memory
> 工具中间日志
~~~

### 需要优化的点

注意区分：

- **工具最终结果**：属于外部证据，优先级较高；
- **工具调用日志**：属于过程信息，优先级最低。

另外，RAG 不是完全不处理，而是：

> 不随意摘要成新结论，但可以通过 rerank、去重、相关性阈值、高价值段落筛选控制长度。

### 面试金句

**Prompt 里的信息不是越多越好，而是要分清证据、背景、对话历史和过程日志。金融分析场景下，RAG 原文和外部工具结果是事实证据，Session Memory 和 Research Memory 只能辅助，不能覆盖证据。**

## 13. 项目不足和后续优化
