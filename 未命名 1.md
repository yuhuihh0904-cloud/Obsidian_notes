## 1.3 什么是异构编程？

异构编程指的是在一个程序中使用不同类型的计算设备协同完成任务，例如：

- CPU：适合复杂控制逻辑、串行任务、操作系统交互。
- GPU：适合大规模数据并行计算。
- NPU / TPU：适合神经网络推理和训练中的矩阵计算。

可以这样回答：

> 异构编程是指利用 CPU、GPU、NPU、TPU 等不同计算设备各自的硬件优势，将程序中的不同部分映射到合适的设备上执行。通常 CPU 负责控制流和串行逻辑，GPU 负责大规模数据并行计算。

---

## 1.4 这章考试可能怎么问？

### 问法 1：什么是 heterogeneous programming？

答：

> 异构编程是指在同一个应用中使用不同类型的处理器或计算设备协同工作，例如 CPU 与 GPU。不同设备具有不同的计算能力、内存结构和适合的任务类型，因此程序员需要根据任务特征决定哪些代码在 CPU 上执行，哪些代码在 GPU 上执行。

### 问法 2：为什么课程先从 GPU 开始？

答：

> 因为 GPU 是目前最典型、最成熟的大规模数据并行设备。通过学习 GPU 编程，可以理解异构编程中的核心问题，例如 host/device 协作、线程组织、内存层次、数据传输和性能优化。这些思想也可以扩展到 NPU、TPU 等设备。

# 2.2 Thread：计算的基本单位

对应页码：p3

## 2.2.1 什么是线程？

线程可以理解为程序执行的一个基本单位。一个线程通常包括：

- Program：正在执行的程序。
- PC：Program Counter，程序计数器，记录当前执行到哪条指令。
- Context：执行上下文。
- Memory：线程可访问的内存。
- Registers：寄存器状态。

在 CUDA 中，一个 kernel 会启动大量线程，每个线程执行同一段 kernel 代码，但处理不同的数据。

---

## 2.2.2 Multiple Threads 和 Many Threads

CPU 程序也可以有多线程，但 GPU 更强调 massive threads，也就是大量线程。

区别：

| 类型 | 特点 |
|---|---|
| CPU 多线程 | 线程数量相对较少，每个线程较重，适合复杂控制逻辑 |
| GPU many threads | 线程数量非常多，每个线程较轻，适合大规模数据并行 |

# 2.3 数据并行 Data Parallelism

对应页码：p4-p5

## 2.3.1 图像灰度转换例子

PPT 用图像转黑白作为数据并行的例子：

```text
for each pixel {
    pixel = gsConvert(pixel)
}
```

每个像素的输出只依赖自己的输入像素，不依赖其他像素。

所以它具有数据并行性：

```text
输入像素 I[0] -> 输出像素 O[0]
输入像素 I[1] -> 输出像素 O[1]
输入像素 I[2] -> 输出像素 O[2]
...
```

每个像素之间互不影响，因此可以让不同线程同时处理不同像素。

---

## 2.3.2 数据并行的判断标准

一个任务适合数据并行，通常满足：

1. 有大量类似的数据元素。
2. 每个数据元素执行相同或类似操作。
3. 不同数据元素之间依赖较少。
4. 可以把不同元素分配给不同线程。

典型例子：

- 图像灰度转换。
- 向量加法。
- 矩阵运算。
- 张量计算。
- 卷积操作。
- 大规模数组处理。

# 2.4 CUDA / OpenCL 执行模型

对应页码：p6

## 2.4.1 Host + Device 模型

CUDA 程序通常由两部分组成：

| 部分 | 运行位置 | 作用 |
|---|---|---|
| Host code | CPU | 串行控制逻辑、内存分配、数据传输、kernel 启动 |
| Device code | GPU | 大规模并行计算 |

PPT 中的结构：

```text
Serial Code(host)
    ↓
Parallel Kernel(device)
    ↓
Serial Code(host)
    ↓
Parallel Kernel(device)
```

也就是说，一个 CUDA 程序不是全部都在 GPU 上跑，而是 CPU 和 GPU 协同执行。

---

## 2.4.2 Kernel 是什么？

Kernel 是在 GPU 上并行执行的函数。

在 CUDA 中，启动 kernel 的语法是：

~~~cpp
KernelA<<<nBlk, nTid>>>(args);
~~~

其中：

| 参数 | 含义 |
|---|---|
| `nBlk` | block 数量 |
| `nTid` | 每个 block 中的 thread 数量 |
| `args` | 传给 kernel 的参数 |

---

# 2.5 CUDA Kernel 与 SPMD 模型

对应页码：p7-p9

## 2.5.1 Kernel executes as grid of threads

CUDA kernel 启动后，不是只执行一次，而是由很多线程共同执行。

每个线程：

- 执行同一个 kernel 代码。
- 有自己唯一的编号。
- 根据编号处理不同的数据。
- 可以根据编号做不同控制判断。

---

## 2.5.2 SPMD 模型

SPMD = Single Program Multiple Data。

中文可以翻译为：

> 单程序，多数据。

含义：

```text
所有线程执行同一份程序代码，
但是每个线程处理不同的数据。
```

例如向量加法：

~~~cpp
int i = threadIdx.x;
C[i] = A[i] + B[i];
~~~

每个线程执行的代码都一样，但是：

```text
线程 0 处理 C[0]
线程 1 处理 C[1]
线程 2 处理 C[2]
...
```

---

## 2.5.3 SPMD 和 SIMD 的区别

虽然 PPT 这里主要讲 SPMD，但考试可能会问对比。

| 模型 | 含义 | 特点 |
|---|---|---|
| SIMD | Single Instruction Multiple Data | 一条指令同时作用于多个数据 |
| SPMD | Single Program Multiple Data | 多个线程执行同一程序，但每个线程有自己的控制流和数据 |

CUDA 更常从程序员视角称为 SPMD，因为程序员写的是一个 kernel 程序，由多个线程执行。

# 2.6 CUDA 线程层次结构

对应页码：p8-p15

这是 Lecture 2 最重要的内容。

## 2.6.1 CUDA 层次

CUDA kernel 的逻辑执行结构是：

```text
Kernel
  -> Grid
      -> Thread Block
          -> Thread
```

PPT 中说明：

- 一个 kernel 由一个 grid 执行。
- 一个 grid 是 thread blocks 的三维数组。
- 一个 thread block 是 threads 的三维数组。
- 每个 thread 有唯一标识符。

---

## 2.6.2 gridDim

对应页码：p10

`gridDim` 表示 grid 中 block 的数量。

它有三个维度：

~~~cpp
gridDim.x
gridDim.y
gridDim.z
~~~

例如 PPT 中的图：

```text
gridDim.x = 8
gridDim.y = 3
gridDim.z = 2
```

如果是一维 grid：

```text
只用 gridDim.x
gridDim.y = 1
gridDim.z = 1
```

如果是二维 grid：

```text
用 gridDim.x 和 gridDim.y
gridDim.z = 1
```

---

## 2.6.3 blockIdx

对应页码：p11

`blockIdx` 表示当前 block 在 grid 中的位置。

它也有三个维度：

~~~cpp
blockIdx.x
blockIdx.y
blockIdx.z
~~~

取值范围：

~~~cpp
blockIdx.x: 0 到 gridDim.x - 1
blockIdx.y: 0 到 gridDim.y - 1
blockIdx.z: 0 到 gridDim.z - 1
~~~

---

## 2.6.4 blockDim

对应页码：p12

`blockDim` 表示每个 block 中 thread 的数量。

三个维度：

~~~cpp
blockDim.x
blockDim.y
blockDim.z
~~~

例如：

```text
blockDim.x = 5
blockDim.y = 4
blockDim.z = 3
```

那么一个 block 中线程总数为：

```text
5 * 4 * 3 = 60
```

如果是一维 block：

```text
只用 blockDim.x
blockDim.y = 1
blockDim.z = 1
```

---

## 2.6.5 threadIdx

对应页码：p13

`threadIdx` 表示当前 thread 在 block 内的位置。

三个维度：

~~~cpp
threadIdx.x
threadIdx.y
threadIdx.z
~~~

取值范围：

~~~cpp
threadIdx.x: 0 到 blockDim.x - 1
threadIdx.y: 0 到 blockDim.y - 1
threadIdx.z: 0 到 blockDim.z - 1
~~~

注意：

```text
threadIdx 只在一个 block 内唯一，
不同 block 中可以有相同的 threadIdx。
```

例如：

```text
Block 0 中有 threadIdx.x = 0
Block 1 中也有 threadIdx.x = 0
```

所以要得到全局线程编号，必须结合 `blockIdx`。

# 2.7 一维全局线程编号

对应页码：p14

这是本章最重要公式之一。

## 2.7.1 公式

~~~cpp
int i = blockIdx.x * blockDim.x + threadIdx.x;
~~~

含义：

```text
当前线程处理全局数组中的第 i 个元素。
```

解释：

| 部分 | 含义 |
|---|---|
| `blockIdx.x` | 当前是第几个 block |
| `blockDim.x` | 每个 block 有多少线程 |
| `blockIdx.x * blockDim.x` | 当前 block 之前已经有多少线程 |
| `threadIdx.x` | 当前线程在本 block 内的编号 |
| 总和 | 当前线程的全局编号 |

---

## 2.7.2 例子

假设：

```text
blockDim.x = 256
blockIdx.x = 3
threadIdx.x = 10
```

那么：

```text
i = 3 * 256 + 10 = 778
```

这个线程处理数组下标 778。

---

## 2.7.3 为什么要用全局编号？

因为实际数据通常是一个全局数组：

~~~cpp
C[i] = A[i] + B[i];
~~~

如果只用 `threadIdx.x`，不同 block 会访问相同下标，造成重复计算。

---

# 2.8 Thread Blocks：可扩展协作

对应页码：p14-p15

## 2.8.1 block 内线程可以协作

同一个 block 内的线程可以通过以下方式协作：

- shared memory
- atomic operations
- barrier synchronization

这些内容后续章节会展开。

---

## 2.8.2 不同 block 之间协作较少

不同 block 之间相对独立。

原因：

1. GPU 可以把不同 block 调度到不同 SM 上。
2. 不同 block 的执行顺序通常不保证。
3. block 之间不能直接同步，除非 kernel 结束。

因此 CUDA 的基本设计思想是：

```text
block 内可以紧密协作，
block 间尽量独立。
```

---

# 2.9 blockIdx 和 threadIdx 对多维数据的意义

对应页码：p15

PPT 强调 thread block 和 thread 的组织结构可以简化多维数据的内存寻址。

适用场景：

- Image processing
- Vectors
- Matrices
- Tensors
- Solving PDEs on volumes

也就是说：

| 数据类型 | 推荐线程组织 |
|---|---|
| 一维数组 | 一维 grid + 一维 block |
| 图像 | 二维 grid + 二维 block |
| 矩阵 | 二维 grid + 二维 block |
| 体数据 | 三维 grid + 三维 block |

# 2.10 向量加法 Vector Addition

对应页码：p16-p18

## 2.10.1 概念视图

向量加法：

```text
C = A + B
```

对于每个元素：

```text
C[i] = A[i] + B[i]
```

每个 `C[i]` 的计算互相独立，所以可以并行。

---

## 2.10.2 传统 C 代码

对应页码：p17

~~~cpp
void vecAdd(float* A, float* B, float* C, int N)
{
    for (int i = 0; i < N; i++)
        C[i] = A[i] + B[i];
}
~~~

特点：

```text
一个 CPU 线程通过 for 循环依次计算所有元素。
```

---

## 2.10.3 CUDA 版本的基本思路

对应页码：p18

CUDA 版本不是简单把 for 循环搬到 GPU，而是要经过 host/device 数据管理：

1. 在 device 上分配 A、B、C。
2. 把 A、B 从 host 拷贝到 device。
3. 启动 kernel，在 GPU 上计算 C。
4. 把 C 从 device 拷贝回 host。
5. 释放 device 内存。

# 2.13 Vector Addition Kernel

对应页码：p24-p28

## 2.13.1 Kernel 代码

~~~cpp
__global__
void vecAddKernel(float* A_d, float* B_d, float* C_d, int N)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < N)
        C_d[i] = A_d[i] + B_d[i];
}
~~~

---

## 2.13.2 每个线程做什么？

每个线程只做一个元素的加法：

```text
线程 i：C[i] = A[i] + B[i]
```

因此原来 CPU 中的 for 循环被大量 GPU 线程替代。

---

## 2.13.3 为什么需要 `if (i < N)`？

因为通常线程总数会向上取整。

例如：

```text
N = 1000
blockDim.x = 256
gridDim.x = ceil(1000 / 256) = 4
总线程数 = 4 * 256 = 1024
```

但是数组只有 1000 个元素。

所以编号 1000 到 1023 的线程不能访问数组，否则越界。

因此需要：

~~~cpp
if (i < N)
~~~

# 2.15 CUDA Function Declarations

对应页码：p29

CUDA 中函数有不同限定符。

| 限定符 | 从哪里调用 | 在哪里执行 | 含义 |
|---|---|---|---|
| `__host__` | Host | Host | 普通 CPU 函数 |
| `__device__` | Device | Device | GPU 上调用、GPU 上执行 |
| `__global__` | Host | Device | kernel 函数，由 CPU 启动，在 GPU 执行 |

---

## 2.15.1 `__global__`

Kernel 函数必须使用：

~~~cpp
__global__
void kernel(...)
~~~

特点：

- 从 host 调用。
- 在 device 上执行。
- 返回值通常是 `void`。
- 使用 `<<< >>>` 启动。

---

## 2.15.2 `__device__`

`__device__` 函数：

- 在 GPU 上执行。
- 只能由 device code 调用。
- 常用于 kernel 内部的辅助函数。

---

## 2.15.3 `__host__`

`__host__` 是普通 CPU 函数。

不写时默认就是 host 函数。


例子：

```text
blockIdx.x = 2
```

## 题型 2：计算 gridDim

如果：

```text
N = 1000
block size = 256
```

那么：

```text
gridDim.x = ceil(1000 / 256) = 4
```

总线程数：

```text
4 * 256 = 1024
```

有效线程：

```text
1000
```

无效线程：

```text
24
```

## 题型 3：解释 Host / Device 流程

# 3.3 Coarser Grains：一个线程处理多个元素

对应页码：p5-p6

## 3.3.1 为什么要让一个线程处理多个元素？

之前是一线程处理一个元素：

```text
thread i -> C[i]
```

但是也可以让一个线程处理多个元素，比如两个元素。

这样叫做更粗粒度的线程分配。

---

## 3.3.2 两个元素每线程的映射

启动方式：

~~~cpp
vecAdd<<<ceil(N / (2 * 256.0)), 256>>>(...);
~~~

第一个元素：

~~~cpp
int i = blockIdx.x * (2 * blockDim.x) + threadIdx.x;

~~~

第二个元素：

~~~cpp
i = i + blockDim.x;

    C[i] = A[i] + B[i];
~~~

为什么是：

~~~cpp
blockIdx.x * (2 * blockDim.x)
~~~

因为每个 block 不再处理 `blockDim.x` 个元素，而是处理：

```text
2 * blockDim.x
```

个元素。

# 3.5 图像模糊 Image Blurring

对应页码：p14-p18

## 3.5.1 图像模糊的思想

对应页码：p14-p15

每个输出像素等于周围像素的平均值。

例如 `BLUR_SIZE = 1` 时，通常会看当前像素周围：

```text
3 × 3 区域
```

如果 `BLUR_SIZE = 5`，则范围更大。

---

## 3.5.2 blurKernel 核心代码

对应页码：p16-p18

~~~cpp
__global__
void blurKernel(unsigned char *in, unsigned char *out, int w, int h)
{
    int Col = blockIdx.x * blockDim.x + threadIdx.x;
    int Row = blockIdx.y * blockDim.y + threadIdx.y;

    if (Col < w && Row < h) {
        int pixVal = 0;
        int pixels = 0;

        for (int blurRow = -BLUR_SIZE; blurRow <= BLUR_SIZE; ++blurRow) {
            for (int blurCol = -BLUR_SIZE; blurCol <= BLUR_SIZE; ++blurCol) {
                int curRow = Row + blurRow;
                int curCol = Col + blurCol;

                if (curRow >= 0 && curRow < h && curCol >= 0 && curCol < w) {
                    pixVal += in[curRow * w + curCol];
                    pixels++;
                }
            }
        }

        out[Row * w + Col] = (unsigned char)(pixVal / pixels);
    }
}
~~~

---

## 3.5.3 为什么 kernel 里可以有循环？

CUDA kernel 不是只能写一条语句。

每个线程可以执行：

- if 判断
- for 循环
- 局部变量
- 算术运算
- 内存访问

区别是：

```text
这些代码会被很多线程同时执行。
```

## 3.5.4 边界条件

对应页码：p17

边缘像素周围没有完整邻域。

例如左上角像素，它的上方和左方不存在像素。

所以需要判断：

~~~cpp

~~~

这可以避免越界访问。

# 3.9 Thread Scheduling 和 Warp

对应页码：p23-p24

## 3.9.1 Warp 是什么？

CUDA 中线程调度的基本单位是 warp。

一个 warp 通常包含 32 个线程。

但是 PPT 提醒：

```text
warp size may not always be 32
```

所以考试中如果需要严谨，可以说：

> 通常 NVIDIA GPU 中一个 warp 是 32 个线程，但具体大小依赖硬件。

---

## 3.9.2 SM 如何调度 warp？

SM 会把 block 中的线程分成多个 warp。

然后 warp scheduler 选择可以执行的 warp。

如果某个 warp 正在等待 global memory 数据，那么 SM 可以切换到另一个 ready warp 执行。

这就是 latency hiding。

---

## 3.9.3 Zero-overhead warp scheduling

对应页码：p24

PPT 中提到：

```text
SM implements zero-overhead warp scheduling.
```

含义：

```text
SM 可以快速在不同 warp 之间切换，不需要像 CPU 线程切换那样很高的上下文切换开销。
```

---

## 3.9.4 Latency tolerance

GPU 通过大量线程和 warp 来隐藏延迟。

例如：

```text
Warp A 等待 global memory
    ↓
SM 不空等
    ↓
切换执行 Warp B
    ↓
Warp B 等待时再执行 Warp C
```

所以 GPU 不是靠单个线程很快，而是靠大量线程隐藏内存访问延迟。

---

# 3.10 Branch Divergence 分支发散

对应页码：p25-p27

这是 Lecture 3 最重要的性能概念之一。

## 3.10.1 什么是 branch divergence？

当同一个 warp 内的线程走不同控制路径时，就发生 branch divergence。

例如：

~~~cpp
if (threadIdx.x > 2) {
    // THEN path
} else {
    // ELSE path
}
~~~

如果一个 warp 里有的线程满足条件，有的不满足，那么它们不能完全并行执行两个分支。

GPU 会：

```text
先执行 THEN path，同时屏蔽 ELSE path 的线程；
再执行 ELSE path，同时屏蔽 THEN path 的线程。
```

这会导致执行串行化，降低性能。

---

## 3.10.2 为什么 branch divergence 影响性能？

因为 warp 内线程是以类似 SIMD 的方式一起执行指令。

如果一个 warp 内线程执行不同路径：

```text
原本一次可以执行完
现在需要分多次执行
```

有效并行度下降。

---

## 3.10.3 predicated execution

PPT 提到 GPUs use predicated execution。

意思是每个线程有一个条件标志，用于决定该线程是否真正执行当前指令。

在分支发散时，不满足条件的线程会被 temporarily disabled。

---

## 3.10.4 如何减少 branch divergence？

对应页码：p27

PPT 建议：

```text
Try to make branch granularity a multiple of warp size.
```

也就是说，让同一个 warp 内的线程尽量走同一个分支。

例如：

~~~cpp
if (threadIdx.x / WARP_SIZE > 2) {
    // THEN path
} else {
    // ELSE path
}
~~~

这样判断粒度按 warp 分组，而不是单个 thread 分散判断，可以减少同一 warp 内部分歧。

---

# 3.11 Block Granularity Considerations

对应页码：p28

## 3.11.1 问题背景

PPT 问：

```text
对于 colorToGreyscaleConversion，应该用 8×8、16×16 还是 32×32 blocks？
```

假设：

```text
每个 SM 最多 1536 threads
最多 8 blocks
```

---

## 3.11.2 如何分析 block size？

三种 block：

| block 大小 | 每个 block 线程数 |
|---|---|
| 8 × 8 | 64 |
| 16 × 16 | 256 |
| 32 × 32 | 1024 |

如果每个 SM 最大 1536 threads：

| block 大小 | 每 SM 最多能放几个 block | 理论线程数 |
|---|---|---|
| 8×8 | 受最多 8 blocks 限制，8 个 block | 8 × 64 = 512 |
| 16×16 | 最多 6 个 block | 6 × 256 = 1536 |
| 32×32 | 最多 1 个 block | 1024 |

所以从 occupancy 角度，16×16 可能更好。

---

## 3.11.3 考试答题思路

如果问选择 block size，应从以下角度分析：

1. 每个 block 的线程数是否超过硬件限制。
2. 每个 SM 能容纳多少 block。
3. 每个 SM 能容纳多少线程。
4. occupancy 是否高。
5. 数据维度是否匹配。
6. 是否有边界浪费。
7. 是否有 shared memory/register 限制。

# 3.12 Lecture 3 Problem Solving

对应页码：p30-p32

## 3.12.1 题型：SM 资源与 block 配置

问题类型：

```text
一个 SM 最多 1536 threads，最多 4 thread blocks。
给若干 block 配置，问哪个可以让 SM 达到最大线程数。
```

解题方法：

```text
每个 block 线程数 × 同时驻留 block 数 <= 1536
同时驻留 block 数 <= 4
```

例如：

| block 线程数 | 最多 block 数 | SM 上线程数 |
|---|---|---|
| 128 | 4 | 512 |
| 256 | 4 | 1024 |
| 512 | 3 | 1536 |
| 1024 | 1 | 1024 |

所以如果目标是满 1536，512-thread block 是可能的。

---

## 3.12.2 题型：最少需要多少线程才能充分利用 GPU

问题类型：

```text
GPU 有 8 个 SM，每个 SM 有 16 个 SP。
如果每个线程执行一次操作，问至少多少线程才能完全利用？
```

基础思路：

```text
至少要有足够线程覆盖所有 SM 和 SP。
```

但真实 CUDA 还涉及 warp、block、occupancy、latency hiding，所以考试要看题目具体问的是“同时执行资源”还是“隐藏延迟所需线程”。

## 3.12.3 题型：kernel 执行结果

对应页码：p32

示例：

~~~cpp
__global__ void do_work(int q, int *A) {
    int result = 0;
    if (q < 5)
        result = threadIdx.x;
    A[threadIdx.x] = result;
}
~~~

分析：

- 如果 `q < 5`，则每个线程写入自己的 `threadIdx.x`。
- 如果 `q >= 5`，则每个线程写入 0。
- 这里没有使用 `blockIdx`，如果多个 block 同时执行，会有不同 block 的线程写同一个 `A[threadIdx.x]`，可能产生覆盖。

# 4.4 Registers vs Memory

## 4.4.1 Registers

寄存器特点：

- 很快。
- 约 1 cycle。
- 数量少。
- CPU 中可能有数百个。

# 4.5 CUDA Programmer View of Memories

对应页码：p8

这是 Lecture 4 最重要知识点之一。

## 4.5.1 CUDA 内存层次

CUDA 线程可以访问：

| 内存类型 | 访问权限 | 作用域 | 大致速度 |
|---|---|---|---|
| Register | 每个线程私有读写 | thread | 约 1 cycle |
| Shared Memory | block 内线程共享读写 | block | 约 5 cycles |
| Global Memory | grid/application 范围读写 | 所有线程可见 | 约 500 cycles |
| Constant Memory | 只读 | grid/application | 有 cache 时约 5 cycles |

---

## 4.5.2 Register

特点：

- 每个 thread 私有。
- 速度最快。
- 存储局部变量。
- 其他线程不能访问。

例如：

~~~cpp
int x;
float sum;
~~~

一般会放在 register 中。

---

## 4.5.3 Shared Memory

特点：

- 一个 block 内的所有线程共享。
- 不同 block 之间不能共享。
- 速度比 global memory 快得多。
- 常用于数据复用和 block 内协作。

典型用途：

```text
矩阵乘法 tiling
卷积中的邻域数据缓存
block 内 reduction
```

---

## 4.5.4 Global Memory

特点：

- 所有线程都可以访问。
- host 可以通过 cudaMemcpy 与其交换数据。
- 容量大。
- 速度慢。
- 是性能瓶颈的常见来源。

---

## 4.5.5 Constant Memory

特点：

- device 上的只读内存。
- 所有线程可读。
- 有缓存时速度快。
- 适合存放不会改变的常量参数。

---

# 4.6 CUDA Variable Type Qualifiers

对应页码：p9

## 4.6.1 变量限定符表

| 变量声明 | 存储位置 | 作用域 | 生命周期 |
|---|---|---|---|
| `int LocalVar;` | register | thread | thread |
| `__device__ __shared__ int SharedVar;` | shared memory | block | block |
| `__device__ int GlobalVar;` | global memory | application | application |
| `__device__ __constant__ int ConstantVar;` | constant memory | application | application |

---

## 4.6.2 自动变量

没有限定符的自动变量：

```text
primitive types 和 structures 通常放在 registers。
per-thread arrays 可能放在 global memory。
```

原因是数组可能太大或需要可索引访问，不一定能完全放入 register。

---

## 4.6.3 `__device__`

PPT 提到：

```text
__device__ optional with __shared__ or __constant__
not allowed by itself within functions
```

理解：

- 全局 device 变量可以用 `__device__`。
- shared 或 constant 变量也可以配合 `__device__`。
- 在函数内部不能单独用 `__device__` 声明普通局部变量。

# 4.7 Matrix Multiplication：矩阵乘法

对应页码：p10-p13

## 4.7.1 问题定义

给定两个矩阵：

```text
M: Width × Width
N: Width × Width
```

计算：

```text
P = M × N
```

其中：

```text
P 也是 Width × Width
```

---

## 4.7.2 数学公式

对应页码：p11

```text
P[i][j] = Σ M[i][k] * N[k][j]
```

其中 k 从 0 到 Width - 1。

---

## 4.7.3 图形理解

计算 P 中某个元素 `P[i][j]`：

1. 取 M 的第 i 行。
2. 取 N 的第 j 列。
3. 对应元素相乘。
4. 把乘积加起来。

---

## 4.7.4 CPU 串行代码

对应页码：p12

~~~cpp
void CPUMatMul(float* M, float* N, float* P, int Width)
{
    for (int i = 0; i < Width; ++i)
        for (int j = 0; j < Width; ++j) {
            float sum = 0;
            for (int k = 0; k < Width; ++k) {
                float a = M[i * Width + k];
                float b = N[k * Width + j];
                sum += a * b;
            }
            P[i * Width + j] = sum;
        }
}
~~~

---

## 4.7.5 哪些循环可以并行？

对应页码：p13

可以先并行化外层两个循环：

```text
for i
    for j
```

也就是说：

```text
每个线程计算一个 P[i][j]
```

内层 k 循环暂时仍由单个线程完成。

# 4.8 Floating-Point is Not Associative

对应页码：p14-p15

这是一个重要概念题。

## 4.8.1 数学上的结合律

数学中：

```text
a + (b + c) = (a + b) + c
```

但浮点数中不一定成立。

---

## 4.8.2 PPT 例子

问题 1：

```text
2^-30 + (1 - 1)
```

先算：

```text
1 - 1 = 0
```

所以结果是：

```text
2^-30
```

问题 2：

```text
(2^-30 + 1) - 1
```

在单精度浮点数中，`2^-30` 太小，加到 1 上时可能丢失。

所以：

```text
2^-30 + 1 ≈ 1
```

再减 1：

```text
1 - 1 = 0
```

因此：

```text
2^-30 + (1 - 1) != (2^-30 + 1) - 1
```

---

## 4.8.3 为什么这和并行有关？

并行求和可能改变加法顺序。

例如 reduction 中：

```text
串行顺序：(((a + b) + c) + d)
并行顺序：(a + b) + (c + d)
```

如果是整数，通常结果一样。

如果是浮点数，结果可能略有不同。

---

## 4.8.4 考试标准中文答案

> 浮点数加法不满足严格结合律，因为浮点数精度有限，进行加法时需要对齐指数，小数部分可能被舍入或丢失。因此改变加法顺序可能导致不同的舍入误差。并行 reduction 会改变求和顺序，所以浮点计算结果可能与串行程序不完全一致。

---

## 4.9.1 为什么用二维线程组织？

因为输出矩阵 P 是二维的。


所以线程也组织成二维：

```text
Row -> 行
Col -> 列
```

```text

```

```text
2 × 2 的子矩阵
```

如果：

```text
TILE_WIDTH = 16

```

```text

```

# 4.10 Matrix Multiplication Kernel

对应页码：p21-p25

这是 Lecture 4 代码重点。

## 4.10.1 计算 Row 和 Col

对应页码：p22

~~~cpp
int Row = blockIdx.y * blockDim.y + threadIdx.y;
int Col = blockIdx.x * blockDim.x + threadIdx.x;
~~~

注意：

```text
Row 用 y 方向。
Col 用 x 方向。
```

---

## 4.10.2 完整 kernel

对应页码：p25

~~~cpp
__global__
void MatrixMulKernel(float* d_M, float* d_N, float* d_P, int Width)
{
    int Row = blockIdx.y * blockDim.y + threadIdx.y;
    int Col = blockIdx.x * blockDim.x + threadIdx.x;

    if (Row < Width && Col < Width) {
        float Pvalue = 0;

        for (int k = 0; k < Width; ++k) {
            Pvalue += d_M[Row * Width + k] * d_N[k * Width + Col];
        }

        d_P[Row * Width + Col] = Pvalue;
    }
}
~~~

---

## 4.10.3 三个关键下标

| 表达式 | 含义 |
|---|---|
| `d_M[Row * Width + k]` | M 的第 Row 行，第 k 列 |
| `d_N[k * Width + Col]` | N 的第 k 行，第 Col 列 |
| `d_P[Row * Width + Col]` | P 的第 Row 行，第 Col 列 |

---

## 4.10.4 为什么需要边界判断？

如果 `Width` 不是 `TILE_WIDTH` 的整数倍，grid 会向上取整。

例如：

```text
Width = 10
TILE_WIDTH = 4
grid = ceil(10 / 4) = 3
覆盖范围 = 3 * 4 = 12
```

线程会覆盖到 Row 或 Col 为 10、11 的位置，但矩阵只有 0 到 9。

所以需要：

~~~cpp
if (Row < Width && Col < Width)
~~~

# 4.11 Global Memory 性能瓶颈

对应页码：p26-p30

这是 Lecture 4 最重要的性能分析部分。

## 4.11.1 简单矩阵乘法的问题

对应页码：p26

简单 kernel 的思路是：

```text
把 CPU 三重循环中的外层 i、j 循环换成线程索引。
```

但是问题是：

```text
每个线程都大量访问 global memory。
```

global memory 很慢，所以性能受限。

---

## 4.11.2 哪里访问了 global memory？

对应页码：p27

在矩阵乘法 kernel 中：

~~~cpp
Pvalue += d_M[Row * Width + k] * d_N[k * Width + Col];
d_P[Row * Width + Col] = Pvalue;
~~~

访问：

| 访问 | 类型 |
|---|---|
| 读取 `d_M[...]` | global memory read |
| 读取 `d_N[...]` | global memory read |
| 写入 `d_P[...]` | global memory write |

---

## 4.11.3 每个 FLOP 需要多少数据？

对应页码：p28

PPT 说每个线程访问 M 和 N：

```text
M 一个 float = 4B
N 一个 float = 4B
```

一次乘加相关的数据读取大约是：

```text
8B per pair
```

如果每次只做少量计算，却要从 global memory 读大量数据，就会受到 memory bandwidth 限制。

---

## 4.11.4 性能上限分析

PPT 提到，如果受内存带宽限制，简单实现的 GFLOPs 会比较低。

核心思想：

```text
计算单元可能很强，
但如果数据喂不进去，
GPU 仍然跑不快。
```

这就是 memory bandwidth bottleneck。

---

## 4.11.5 如何优化？

对应页码：p30

PPT 给出的方向：

```text
Reuse memory accesses.
```

也就是：

```text
让从 global memory 读入的数据被多个线程重复使用。
```

典型方法：

```text
shared memory tiling
```

虽然这章还没有正式展开 shared memory 矩阵乘法，但它已经说明了优化动机。

# 4.12 Lecture 4 Problem Solving

对应页码：p32-p34

## 4.12.1 题型：二维矩阵需要多少 block？

对应页码：p32

问题类型：

```text
输入矩阵 250 行，60 列。
使用 16×16 thread blocks。
问需要多少 blocks？
```

解法：

```text
grid.x = ceil(columns / 16) = ceil(60 / 16) = 4
grid.y = ceil(rows / 16) = ceil(250 / 16) = 16
```

所以：

```text
总 block 数 = 4 × 16 = 64
```

注意：

```text
x 通常对应列 columns，
y 通常对应行 rows。
```

---

## 4.12.2 题型：同一 block 内线程共享数据用什么内存？

对应页码：p33

如果题目说：

```text
某些线程需要读取同一个 thread block 内其他线程写入的数据。
```

最合适的是：

```text
shared memory
```

原因：

- shared memory 对同一 block 内线程可见。
- 比 global memory 快。
- 适合 block 内通信和协作。

答题：

> 应使用 shared memory。因为 shared memory 的作用域是 thread block，同一 block 内的线程都可以访问它，适合在线程之间共享数据。同时它比 global memory 延迟更低，性能更好。

---

## 4.12.3 题型：多个 block 写同一位置结果不确定

对应页码：p34

示例：

~~~cpp
__global__ void kernel(char *dst) {
    dst[0] = blockIdx.x;
}
~~~

如果启动多个 block，每个 block 都写：

~~~cpp
dst[0]
~~~

则会发生写冲突。

最终 `dst[0]` 的值可能是任何一个 block 的 `blockIdx.x`。

原因：

```text
不同 block 的执行顺序不确定。
```

如果有多个 block：

```text
blockIdx.x = 0, 1, 2, ...
```

最终值取决于最后写入的是哪个 block。

# 6. 高频考试作答模板

## 6.1 解释 CUDA Host / Device 执行流程

> CUDA 程序采用 host/device 异构执行模型。Host 通常指 CPU，负责执行串行控制逻辑、分配 device memory、将数据从 host 拷贝到 device、启动 kernel、将结果从 device 拷贝回 host 并释放资源。Device 通常指 GPU，负责执行大规模并行 kernel。Kernel 启动后会产生大量线程，这些线程执行同一段程序代码，但根据自己的 blockIdx 和 threadIdx 处理不同数据。

---

## 6.2 解释为什么需要 `if (i < N)`

> CUDA kernel 启动的线程数量通常按 block 大小向上取整，因此总线程数可能大于实际数据元素个数。如果不做边界检查，编号超过 N 的线程会访问数组越界，导致错误。因此需要使用 `if (i < N)` 保证只有有效线程访问数据。

---

## 6.3 解释 SPMD

> SPMD 表示 Single Program Multiple Data，即单程序多数据。在 CUDA 中，kernel 中所有线程执行同一段程序代码，但每个线程拥有不同的 threadIdx 和 blockIdx，因此可以处理不同的数据元素。向量加法中，每个线程执行同样的加法代码，但线程 i 处理 A[i]、B[i]、C[i]。

---

## 6.4 解释 Branch Divergence

> Branch divergence 是指同一个 warp 内的线程执行不同控制流路径。例如部分线程进入 if 分支，另一部分线程进入 else 分支。由于 warp 内线程通常以同一指令流执行，分支发散会导致不同路径被串行执行，从而降低并行效率。减少 branch divergence 的方法是让分支条件尽量按照 warp 粒度划分，使同一个 warp 内线程尽量走相同路径。

---

## 6.5 解释 CUDA 内存层次

> CUDA 中不同内存具有不同作用域和性能。Register 是每个线程私有的，速度最快；shared memory 被同一个 block 内线程共享，适合 block 内协作；global memory 对所有线程可见，容量大但延迟高；constant memory 是只读内存，适合存放常量并可通过缓存加速。CUDA 程序优化的重要方向是减少 global memory 访问，并利用 shared memory 进行数据复用。

---

## 6.6 解释矩阵乘法如何并行化

> 矩阵乘法中，每个输出元素 P[i][j] 可以独立计算，它等于 M 的第 i 行与 N 的第 j 列对应元素乘积之和。因此可以让每个 CUDA 线程负责计算一个 P[i][j]。线程通过 blockIdx 和 threadIdx 计算自己的 Row 和 Col，然后在 kernel 内通过 k 循环完成点积计算。通常使用二维 grid 和二维 block 来匹配矩阵的二维结构。

---

## 6.7 解释 Global Memory 为什么是瓶颈

> Global memory 容量大但访问延迟高，通常需要几百个 cycle。简单矩阵乘法中，每个线程都反复从 global memory 读取 M 和 N 的元素，而每次读取只进行少量计算，导致计算单元可能等待数据，性能受 memory bandwidth 限制。优化思路是增加数据复用，例如把 global memory 中的数据加载到 shared memory，使同一个 block 内多个线程重复使用这些数据。

---

~~~cpp
    int greyOffset = Row * width + Col;
    int rgbOffset = 3 * greyOffset;

    unsigned char r = rgbImage[rgbOffset];
    unsigned char g = rgbImage[rgbOffset + 1];
    unsigned char b = rgbImage[rgbOffset + 2];

    grayImage[greyOffset] = 0.21f * r + 0.71f * g + 0.07f * b;
}
~~~

### Blur

~~~cpp
int Col = blockIdx.x * blockDim.x + threadIdx.x;
int Row = blockIdx.y * blockDim.y + threadIdx.y;

if (Col < w && Row < h) {
    int pixVal = 0;
    int pixels = 0;

    for (int blurRow = -BLUR_SIZE; blurRow <= BLUR_SIZE; blurRow++) {
        for (int blurCol = -BLUR_SIZE; blurCol <= BLUR_SIZE; blurCol++) {
            int curRow = Row + blurRow;
            int curCol = Col + blurCol;

            if (curRow >= 0 && curRow < h && curCol >= 0 && curCol < w) {
                pixVal += in[curRow * w + curCol];
                pixels++;
            }
        }
    }

    out[Row * w + Col] = pixVal / pixels;
}
~~~

### Matrix Multiplication

~~~cpp
int Row = blockIdx.y * blockDim.y + threadIdx.y;
int Col = blockIdx.x * blockDim.x + threadIdx.x;

if (Row < Width && Col < Width) {
    float Pvalue = 0;

    for (int k = 0; k < Width; k++) {
        Pvalue += d_M[Row * Width + k] * d_N[k * Width + Col];

    d_P[Row * Width + Col] = Pvalue;
~~~

# 9. 一句话总结前四章

前四章主要讲 CUDA 异构并行编程的基础：CPU 作为 host 负责控制和数据管理，GPU 作为 device 执行 kernel；kernel 由 grid、block、thread 组织，线程通过 blockIdx 和 threadIdx 映射到数组、图像、矩阵等数据；GPU 通过大量 warp 和线程隐藏延迟，但 branch divergence 会降低效率；CUDA 内存分为 register、shared、global、constant，其中 global memory 慢，性能优化的核心是减少 global memory 访问并提高数据复用。
