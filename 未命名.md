# 医疗大模型微调项目面试题库

> 适用项目：基于 LLaMA-Factory 的 Qwen3.5-4B 医疗问答微调，训练路线为 DoRA SFT -> DPO，评测包含 Base / SFT / DPO 三组 BLEU、ROUGE 对比。
>
> 面试表达原则：先讲目标，再讲数据，再讲训练，再讲评测，再讲风险和改进。每个问题尽量落到你自己的项目数字上。

## 0. 30 秒项目介绍

**Q1：请你简单介绍一下这个项目。**

**答：**
我做的是一个中文医疗问答大模型微调项目。官方参考目录是 Qwen2.5-7B 医疗微调模板，但我实际在服务器上使用的是本地 Qwen3.5-4B 基座模型，基于 LLaMA-Factory 完成了两阶段训练：第一阶段用医疗问答数据做 SFT，让模型学习医疗领域问答格式；第二阶段在 SFT checkpoint 基础上做 DPO，用 chosen/rejected 偏好对进一步优化回答质量。训练采用参数高效微调，不做全参更新，而是用 LoRA/DoRA adapter，并开启 Unsloth 加速。最后我对 Base、SFT、DPO 三个版本在验证集和测试集上做了生成式评测，测试集 BLEU-4 从 0.64 提升到 SFT 的 5.76，DPO 后进一步到 6.47，ROUGE-L 从 2.54 提升到 14.74，再到 15.52。

**追问点：**
- 官方模板是 Qwen2.5-7B，但你实际跑的是 Qwen3.5-4B。
- 你的核心路线是 SFT -> DPO。
- 你的微调方法是 LoRA/DoRA，不是全参微调。
- 你的有效性证据是 Base/SFT/DPO 的评测对比。

## 1. 项目背景与技术选型

**Q2：为什么要做医疗领域微调？通用大模型不能直接用吗？**

**答：**
通用大模型可以回答一些医学常识，但它的回答风格、术语密度、领域覆盖和安全提示不一定符合医疗问答场景。医疗问答往往有固定的表达方式，比如症状、病因、检查、治疗、注意事项等。微调的目标不是让模型从零学习医学知识，而是让它更贴近医疗问答数据分布，提高中文医疗问答的一致性和可用性。

**Q3：这个项目为什么不是从零预训练？**

**答：**
从零预训练需要海量语料、巨大算力和长时间训练，不适合个人或小团队项目。我这个项目的目标是领域适配，所以选择在已有 Qwen3.5-4B 基座上做后训练。这样能复用基座模型已有语言能力和知识，再用医疗问答数据进行领域增强。

**Q4：为什么选择 Qwen 系列？**

**答：**
Qwen 系列对中文支持较好，开源生态成熟，LLaMA-Factory 对 Qwen 模板、tokenizer、chat template 都支持比较完整。医疗问答主要是中文，所以中文能力和工程兼容性是关键。

**Q5：为什么实际用了 Qwen3.5-4B，而不是官方模板里的 Qwen2.5-7B？**

**答：**
官方项目是参考模板，提供的是 Qwen2.5-7B 的训练方案。我实际服务器上的基座模型是 Qwen3.5-4B，所以我把官方的两阶段医疗微调流程迁移到了 4B 基座上。这样显存压力更低，训练速度更可控，也适合作为完整项目闭环。

**Q6：你这个项目的创新点或亮点是什么？**

**答：**
主要有四点：第一，采用两阶段训练，先 SFT 学领域问答，再 DPO 做偏好对齐；第二，使用 DoRA 这种比普通 LoRA 表达能力更强的参数高效微调方法；第三，开启 Unsloth 进行训练加速和显存优化；第四，做了 Base、SFT、DPO 三组对照评测，不只是训练完模型，而是量化证明微调有效。

**Q7：这个项目适合写在简历上吗？一句话怎么写？**

**答：**
可以写成：基于 LLaMA-Factory 对 Qwen3.5-4B 进行中文医疗问答领域微调，采用 DoRA SFT + DPO 两阶段后训练，并通过 BLEU/ROUGE 对 Base、SFT、DPO 进行对照评估，测试集 BLEU-4 从 0.64 提升至 6.47，ROUGE-L 从 2.54 提升至 15.52。

## 2. 数据构造与数据格式

**Q8：你的训练数据是什么格式？**

**答：**
主要是 Alpaca 格式，每条数据包含 `instruction`、`input`、`output`。在我的医疗问答数据里，`instruction` 是用户问题，`input` 大多数为空，`output` 是参考回答。例如：

```json
{"instruction": "慢性支气管炎宜食什么？", "input": "", "output": "鸭肝；莲子；葵花子仁；白果（干）"}
```

**Q9：为什么要用 Alpaca 格式？**

**答：**
因为 LLaMA-Factory 对 Alpaca 格式支持很成熟，数据字段可以通过 `dataset_info.json` 映射到 prompt/query/response，训练时框架会按模型模板自动拼接成 chat 格式。这样可以减少手写数据处理代码的风险。

**Q10：`dataset_info.json` 的作用是什么？**

**答：**
它是 LLaMA-Factory 的数据集注册文件，告诉框架每个数据集文件在哪里，以及哪些字段对应 prompt、query、response。我的 `medical_sft` 对应 SFT 数据，字段映射是 instruction -> prompt，input -> query，output -> response；`medical_dpo` 还额外声明了 `ranking: true`，并映射 chosen/rejected。

**Q11：SFT 数据和 DPO 数据有什么区别？**

**答：**
SFT 数据是单个标准答案，格式是 question -> answer，用来让模型模仿高质量回答。DPO 数据是偏好对，格式是 question -> chosen/rejected，其中 chosen 是更优回答，rejected 是较差回答。SFT 学“怎么答”，DPO 学“哪个回答更好”。

**Q12：DPO 的 rejected 是怎么来的？**

**答：**
官方模板中的脚本会把原始高质量回答作为 chosen，然后通过截断原回答、添加模糊结尾等方式构造 rejected。例如把完整医学回答截断到 30%-50%，再加上“具体情况因人而异，建议咨询医生”之类的模糊句子。这样可以低成本构造偏好对。

**Q13：自动构造 rejected 有什么问题？**

**答：**
优点是成本低，能快速跑通 DPO 流程；缺点是偏好信号比较简单，模型可能只学到“回答更长更完整更好”，而不一定学到真正的医学准确性、安全性和逻辑性。更好的做法是人工标注偏好，或者用专家规则、LLM-as-judge、人审结合构造更可靠的偏好数据。

**Q14：医疗数据要做哪些清洗？**

**答：**
至少包括：去空样本、去重复、过滤乱码和异常长度、统一字段格式、去除明显广告和无关内容、检查敏感隐私信息、避免错误医学建议、控制回答长度分布。医疗场景还要重点关注不安全回答，比如乱开药、绝对化诊断、忽视急症等。

**Q15：为什么要划分 train、valid、test？**

**答：**
train 用于训练参数，valid 用于训练过程中的验证和调参，test 用于最终泛化评估。测试集不能参与训练和调参，否则评估结果会被污染。

**Q16：你的数据规模是多少？**

**答：**
当前下载回来的根目录数据里，`train_zh_0.json` 约 1949972 行，`valid_zh_0.json` 500 行，`test_zh_0.json` 500 行。实际 LLaMA-Factory 训练使用的是注册在 `fine_tuning_data` 下的 `medical_sft.jsonl` 和 `medical_dpo.jsonl`，但这两个中间文件没有随服务器工作目录完整拷回。因此面试中我会明确说：我能确认的是当前保留了原始/评估数据、训练配置、adapter 产物、训练日志和评估结果。

**Q17：如果面试官问“你到底训练了多少条数据”，怎么答？**

**答：**
不要硬编。可以说：官方脚本默认构造 20000 条 SFT 和 5000 条 DPO，但当前本地没有完整保存服务器上的中间 jsonl 文件。从 trainer log 看，SFT 训练 891 steps、3 epochs，DPO 训练 189 steps、1 epoch，说明实际训练规模可以根据 batch size 和梯度累积反推为万级 SFT、千级 DPO。这个地方我会如实说明数据文件迁移不完整，但训练日志和评估结果是完整保留下来的。

**Q18：医疗问答数据的 label 是否一定唯一？BLEU/ROUGE 合理吗？**

**答：**
不一定唯一。医疗问答可以有多种正确表达方式，所以 BLEU/ROUGE 只能衡量和参考答案的字面重合程度，不能完全代表医学正确性。它们适合作为自动化初筛和横向对比指标，但还需要人工评测、医学事实性评测、安全性评测配合。

## 3. SFT 监督微调

**Q19：什么是 SFT？**

**答：**
SFT 是 Supervised Fine-Tuning，监督微调。它用人工或高质量数据构造的 prompt-answer 样本训练模型，让模型在给定指令时生成目标答案。本质上还是自回归语言模型的 next token prediction，只是 loss 通常只作用在 assistant 回答部分。

**Q20：SFT 解决什么问题？**

**答：**
SFT 主要解决“模型会不会按照指令回答”和“回答风格是否符合目标场景”的问题。对于这个项目，SFT 让通用 Qwen3.5-4B 更贴近中文医疗问答格式。

**Q21：SFT 的输入输出怎么拼接？**

**答：**
LLaMA-Factory 根据 `template: qwen` 把 instruction/input/output 拼成 Qwen 的 chat template。大致形式是 system message + user message + assistant answer，训练时模型看到 user 之前的上下文，并学习 assistant 部分的 token。

**Q22：SFT 训练目标是什么？**

**答：**
最常见目标是最小化参考回答 token 的交叉熵 loss。给定上下文 x 和目标回答 y，模型最大化 P(y|x)，也就是让正确回答序列的负对数似然最小。

**Q23：你的 SFT 配置有哪些关键超参？**

**答：**
我的 SFT 配置：`stage: sft`，`finetuning_type: lora`，`lora_rank: 16`，`lora_alpha: 32`，`lora_dropout: 0.05`，`lora_target: all`，`use_dora: true`，`use_unsloth: true`，`cutoff_len: 2048`，学习率 `1e-4`，训练 `3 epochs`，bf16，per-device batch size 为 4，gradient accumulation 为 8。

**Q24：为什么 cutoff_len 设为 2048？**

**答：**
医疗问答回答可能比较长，1024 可能截断关键信息，2048 能保留更多上下文和答案内容。但 cutoff 越大显存和训练时间越高，所以 2048 是一个效果和成本的折中。

**Q25：为什么 SFT 学习率是 1e-4？**

**答：**
LoRA/DoRA 只训练 adapter 参数，训练参数量远小于全参微调，所以学习率通常可以比全参微调大一些。`1e-4` 是 LoRA SFT 中比较常见的量级；如果 loss 不稳定或过拟合，可以降低到 `5e-5` 或 `2e-5`。

**Q26：SFT 训练结果怎么看？**

**答：**
我的 SFT 训练 3 个 epoch，总 step 891，train loss 约 0.3145，eval loss 约 2.5858，训练耗时约 4 小时 43 分钟。train loss 下降说明模型在拟合训练数据；eval loss 是验证集上的生成概率损失，但最终效果还要看生成评测和样例质量。

**Q27：为什么训练 loss 很低但 eval loss 比较高？**

**答：**
可能有几个原因：训练集和验证集分布不同；医疗答案表达多样，参考答案不唯一；模型对训练集拟合较强；验证集回答更长或更难。对生成任务不能只看 loss，还要结合 BLEU/ROUGE、人工评测和具体样例。

**Q28：SFT 有什么风险？**

**答：**
数据质量差会让模型学到错误医学建议；数据风格单一会导致回答模板化；训练过度会遗忘基座模型的通用能力；医疗数据中如果有隐私信息，还会带来合规风险。

## 4. LoRA、DoRA、QLoRA、Unsloth

**Q29：什么是 LoRA？**

**答：**
LoRA 是 Low-Rank Adaptation。它不直接更新原始大模型权重 W，而是在某些线性层旁边训练一个低秩增量矩阵 ΔW。ΔW 被分解成 A 和 B 两个小矩阵，通常 ΔW = B A。训练时冻结原模型，只训练 A/B，因此显存和参数量大幅降低。

**Q30：LoRA 为什么有效？**

**答：**
LoRA 的假设是：大模型在下游任务适配时，权重变化具有低内在秩，不需要更新完整的高维矩阵。用低秩矩阵近似更新方向，就能以较少参数达到不错效果。

**Q31：LoRA 的 rank 是什么？你这里为什么是 16？**

**答：**
rank 决定低秩矩阵的瓶颈维度，也决定可训练参数量和表达能力。rank 越大，表达能力越强，但显存和过拟合风险也更高。`r=16` 是领域 SFT 中常见折中，适合 4B/7B 模型的中等规模微调。

**Q32：LoRA alpha 是什么？为什么 alpha=32？**

**答：**
alpha 是 LoRA 更新的缩放系数，实际缩放通常和 `alpha/r` 有关。我这里 r=16，alpha=32，也就是缩放比例约 2，是 DoRA/LoRA 里常见设置，可以让 adapter 更新有足够强度。

**Q33：lora_dropout 有什么用？**

**答：**
dropout 用来正则化 adapter，减少过拟合。医疗问答数据可能有重复和模板化内容，`0.05` 的 dropout 可以让训练更稳一点。

**Q34：`lora_target: all` 是什么意思？**

**答：**
表示尽可能对模型中的主要线性层都加 LoRA adapter，而不是只加 q_proj/v_proj。这样表达能力更强，但训练参数和显存也会增加。

**Q35：什么是 DoRA？和 LoRA 有什么区别？**

**答：**
DoRA 是 Weight-Decomposed Low-Rank Adaptation。普通 LoRA 只学习低秩增量；DoRA 把预训练权重分解成 magnitude 和 direction 两部分，对方向部分使用 LoRA 进行更新，同时单独学习幅度信息。它更接近全参微调的更新方式，因此通常比普通 LoRA 表达能力更好、训练更稳定。

**Q36：DoRA 会增加推理成本吗？**

**答：**
训练阶段会稍微复杂，但 adapter 合并后通常不会明显增加额外推理成本。实际部署时可以选择加载 base + adapter，也可以把 adapter merge 到基座权重中。

**Q37：QLoRA 是什么？你的项目用了 QLoRA 吗？**

**答：**
QLoRA 是把基座模型量化到 4-bit，再在量化模型上训练 LoRA adapter，以进一步降低显存。我的实际 SFT 配置开启了 DoRA 和 Unsloth，但没有使用 4-bit QLoRA；官方目录里有 qdora 备选配置。

**Q38：LoRA、DoRA、QLoRA 怎么选？**

**答：**
显存够时可以用 LoRA/DoRA，效果和稳定性更好；显存紧张时用 QLoRA/QDoRA；如果追求更高效果且有充足算力，可以考虑全参微调，但成本很高。我的项目在效果和成本之间选择 DoRA。

**Q39：Unsloth 是什么？**

**答：**
Unsloth 是一个训练加速和显存优化工具，通过优化底层 kernel、attention、反向传播等过程，让 LoRA/QLoRA 训练更快、更省显存。我的配置里 `use_unsloth: true`。

**Q40：参数高效微调和全参微调的区别？**

**答：**
全参微调更新模型全部权重，表达能力强但显存、算力、存储成本高，也更容易灾难性遗忘。参数高效微调冻结基座，只训练少量 adapter，成本低、易切换、多任务管理方便，但上限可能不如全参微调。

## 5. DPO、RLHF、偏好对齐

**Q41：什么是 DPO？**

**答：**
DPO 是 Direct Preference Optimization，直接偏好优化。它用同一个 prompt 下的 chosen/rejected 回答对训练模型，让模型提高 chosen 的相对概率、降低 rejected 的相对概率。它不需要单独训练 reward model，也不需要 PPO 那种复杂强化学习流程。

**Q42：DPO 和 SFT 的区别？**

**答：**
SFT 是单答案模仿学习，只告诉模型“这个答案是对的”；DPO 是成对偏好学习，告诉模型“这个答案比另一个答案更好”。SFT 让模型学会任务，DPO 让模型偏向更符合人类偏好的输出。

**Q43：DPO 和 RLHF/PPO 的区别？**

**答：**
传统 RLHF 通常先训练 reward model，再用 PPO 优化策略模型，流程复杂、训练不稳定、成本高。DPO 把偏好优化转化为一个直接的分类式目标，用 chosen/rejected 对直接更新语言模型，工程上更简单稳定。

**Q44：DPO 的 beta 参数是什么？你的项目为什么是 0.1？**

**答：**
beta 控制偏好优化强度，也可以理解为模型偏离 reference model 的程度。beta 大，模型更激进地追求 chosen；beta 小，模型更保守，避免偏离原模型太多。`0.1` 是常见默认值，适合作为稳定起点。

**Q45：DPO 里的 reference model 是什么？**

**答：**
reference model 通常是 DPO 开始前的模型，比如 SFT 后的模型。DPO 训练会鼓励新模型相比 reference model 更偏好 chosen，同时通过隐式 KL 约束避免偏离太大。

**Q46：你的 DPO 是接在哪个 checkpoint 上做的？**

**答：**
配置里 `adapter_name_or_path` 指向 SFT 的 `checkpoint-600`，说明 DPO 是在 SFT checkpoint-600 基础上继续训练得到的。

**Q47：DPO 训练结果怎么看？**

**答：**
我的 DPO 训练 1 epoch，总 step 189，train loss 约 0.00839，eval loss 约 0.00271，eval rewards accuracy 为 1.0。rewards accuracy 高说明在验证偏好对上模型基本能把 chosen 排在 rejected 前面。但这不等于真实医疗能力完美，因为 rejected 是自动构造的。

**Q48：为什么 DPO 后 BLEU/ROUGE 提升不算特别巨大？**

**答：**
合理。SFT 已经完成主要领域适配，DPO 主要做偏好和回答质量微调，所以指标通常是在 SFT 基础上小幅提升。我的测试集 BLEU-4 从 5.76 到 6.47，ROUGE-L 从 14.74 到 15.52，说明有增益但不是颠覆性变化。

**Q49：DPO 有什么风险？**

**答：**
偏好数据质量决定上限。如果 rejected 太弱，模型只学会区分很明显的差答案；如果偏好数据有偏见，模型会放大偏见；如果 beta 或学习率不合适，可能导致模型过度偏离 SFT 模型，出现回答重复、变啰嗦或安全性下降。

## 6. 训练配置、资源与日志

**Q50：你用的训练框架是什么？**

**答：**
LLaMA-Factory。它封装了模型加载、数据模板、LoRA/DoRA、SFT、DPO、评测和导出流程，适合快速完成大模型后训练实验。

**Q51：为什么用 bf16？**

**答：**
bf16 相比 fp32 显存更低、速度更快，同时数值范围比 fp16 更大，训练大模型更稳定。前提是 GPU 支持 bf16。

**Q52：gradient accumulation 是什么？**

**答：**
梯度累积是在显存放不下大 batch 时，把多个小 batch 的梯度累积起来再更新一次参数。有效 batch size = per_device_train_batch_size × gradient_accumulation_steps × GPU 数量。我的 SFT 配置单卡下是 4 × 8 = 32。

**Q53：为什么 DPO batch size 比 SFT 小？**

**答：**
DPO 每个样本包含 chosen 和 rejected 两个回答，需要同时计算两条序列的概率，显存占用比 SFT 更高，所以 batch size 通常更小。我的 DPO per-device batch size 是 1，gradient accumulation 是 8。

**Q54：学习率调度用了什么？**

**答：**
配置里使用 cosine scheduler，并设置 warmup ratio。warmup 可以避免训练初期学习率过大导致不稳定，cosine 后期逐渐降低学习率，有助于收敛。

**Q55：save_steps 和 eval_steps 有什么作用？**

**答：**
save_steps 定期保存 checkpoint，方便恢复训练和选择中间模型；eval_steps 定期在验证集上评估，观察是否过拟合或训练崩掉。SFT 中 save/eval 都是 200 steps，DPO 中 save 100、eval 50。

**Q56：为什么最终评估 SFT 用 checkpoint-600，而不是最终 checkpoint-891？**

**答：**
配置和评估文件显示评估使用的是 checkpoint-600。这可能是因为中间 checkpoint 在验证表现、稳定性或时间安排上更合适，也可能是训练时选择它作为 DPO 起点。面试中可以说：我保留了 checkpoint-600 和 checkpoint-891，当前评估闭环主要围绕 checkpoint-600；后续可以补充最终 checkpoint 的对比。

**Q57：adapter_model.safetensors 是什么？**

**答：**
这是训练得到的 LoRA/DoRA adapter 权重，不是完整模型权重。它需要和基座模型一起加载，或者通过 merge 导出成完整模型。

**Q58：如果线上部署，是加载 adapter 还是 merge？**

**答：**
如果要频繁切换多个领域 adapter，可以加载 base + adapter；如果只部署一个最终模型，merge 后推理更简单，也更容易接入 vLLM 等服务。

## 7. 评测指标：BLEU 与 ROUGE

**Q59：BLEU 是什么？**

**答：**
BLEU 是机器翻译里常用的自动评价指标，后来也常用于文本生成。它主要看生成答案和参考答案之间的 n-gram precision，即生成文本里有多少 n-gram 能在参考答案中匹配，同时加入 brevity penalty，避免模型只生成很短但高精度的片段。

**Q60：BLEU-4 是什么意思？**

**答：**
BLEU-4 通常综合 1-gram、2-gram、3-gram、4-gram 的匹配精度，使用几何平均并乘以长度惩罚。4-gram 越高，说明生成答案和参考答案在较长连续片段上的重合越高。

**Q61：BLEU 高代表什么？**

**答：**
代表生成文本和参考答案在词面 n-gram 上更接近。对于你的项目，DPO 测试集 BLEU-4 从 Base 的 0.64 提升到 6.47，说明微调后回答更贴近医疗问答参考答案。

**Q62：BLEU 有什么局限？**

**答：**
它偏重字面重合，不理解语义。两个医学上都正确但表达不同的答案，BLEU 可能很低；一个字面相似但医学上错误的答案，BLEU 也可能不低。因此医疗场景不能只看 BLEU。

**Q63：ROUGE 是什么？**

**答：**
ROUGE 是文本摘要和生成任务常用指标，偏重 recall，即参考答案中的内容有多少被生成答案覆盖。它适合看模型是否覆盖了参考回答中的关键信息。

**Q64：ROUGE-1 是什么？**

**答：**
ROUGE-1 看 unigram，也就是单字/词级别的重合。它反映生成答案覆盖了多少基础词汇信息。

**Q65：ROUGE-2 是什么？**

**答：**
ROUGE-2 看 bigram，也就是连续两个 token 的重合。它比 ROUGE-1 更严格，能反映局部短语结构是否接近。

**Q66：ROUGE-L 是什么？**

**答：**
ROUGE-L 基于最长公共子序列 LCS，不要求完全连续，但要求顺序一致。它能衡量生成答案和参考答案在整体句子结构上的相似性。

**Q67：你的评测结果说明什么？**

**答：**
测试集上 Base 的 BLEU-4 是 0.642、ROUGE-L 是 2.543；SFT 后 BLEU-4 到 5.762、ROUGE-L 到 14.745；DPO 后 BLEU-4 到 6.465、ROUGE-L 到 15.522。这说明 SFT 带来主要提升，DPO 在 SFT 基础上继续改善回答与参考答案的一致性。

**Q68：为什么 Base 指标这么低？**

**答：**
因为 Base 模型没有针对这个医疗问答格式训练，生成可能偏通用、偏英文、偏推理过程，和参考答案字面差异大。我们在样例里也看到 Base 可能输出很长英文 `<think>`，这会严重拉低 BLEU/ROUGE。

**Q69：为什么自动指标不能完全代表医疗问答质量？**

**答：**
医疗问答要看事实正确性、风险提示、是否建议及时就医、是否避免诊断绝对化、是否避免乱开药。BLEU/ROUGE 只看字面匹配，不能判断安全性和医学真实性。

**Q70：你会如何设计更完整的医疗评测？**

**答：**
我会分三层：第一层是自动指标，包括 BLEU、ROUGE、BERTScore、平均长度、重复率；第二层是 LLM-as-judge，从准确性、完整性、安全性、可读性评分；第三层是人工/专家评审，重点检查高风险问题，比如急症、用药、孕妇儿童、慢病管理等。

## 8. 结果分析与项目反思

**Q71：SFT 和 DPO 哪个提升更明显？**

**答：**
SFT 更明显。测试集 BLEU-4 从 0.64 到 5.76，ROUGE-L 从 2.54 到 14.74，说明领域适配主要来自 SFT。DPO 是在 SFT 基础上进一步优化，BLEU-4 到 6.47，ROUGE-L 到 15.52，属于增量提升。

**Q72：DPO 后有没有可能带来负面效果？**

**答：**
有。样例中可以看到 DPO 有时会出现重复表达，说明偏好优化如果数据构造简单，可能让模型过度偏向某些模板。后续要看重复率、人工质量和安全性，而不能只看 BLEU/ROUGE。

**Q73：你这个项目最大的不足是什么？**

**答：**
第一，DPO rejected 是自动构造，偏好信号不够真实；第二，评测主要是 BLEU/ROUGE，缺少医学专家评估；第三，当前本地下载包没有完整保留服务器训练时的中间 jsonl 文件；第四，没有做多模型横向对比，比如 LoRA vs DoRA、SFT final vs checkpoint-600、是否 merge 后推理评测。

**Q74：如果继续优化，你会做什么？**

**答：**
我会补四件事：一是补全数据版本管理，保存训练数据 hash 和采样脚本；二是引入人工或 LLM 辅助偏好标注，提升 DPO 数据质量；三是增加医疗安全评测集和人工评审；四是做消融实验，比如 LoRA vs DoRA、不同 rank、不同 checkpoint、SFT-only vs SFT+DPO。

**Q75：如何证明不是“背答案”？**

**答：**
可以从数据划分、测试集隔离、去重、泛化样例、人工评测几个方面证明。当前我有独立 test 集 500 条和 Base/SFT/DPO 对比，但后续更严谨的做法是对 train/test 做相似度去重，并构造新的人工问题集验证泛化。

**Q76：医疗问答项目如何处理安全边界？**

**答：**
应该在回答中避免绝对诊断和处方化建议，对严重症状提示及时就医；对药物剂量、儿童、孕妇、慢病等高风险场景要更保守；系统层面可以加入安全 prompt、拒答策略、规则后处理和人工审核。

## 9. 推理与部署

**Q77：训练完怎么推理？**

**答：**
有两种方式：一种是 transformers 加载基座模型和 adapter；另一种是把 adapter merge 到 base 后，用 vLLM 做高性能推理。官方模板里也提供了 chat.py 思路。

**Q78：为什么要 merge adapter？**

**答：**
merge 后模型就是完整权重，部署简单，推理时不需要额外加载 adapter 逻辑，也更容易接入部分推理框架。但缺点是每个 adapter 都要导出一份完整模型，存储更大。

**Q79：vLLM 有什么优势？**

**答：**
vLLM 通过 PagedAttention 等机制提高吞吐，适合多用户并发推理。训练阶段用 LLaMA-Factory，部署阶段可以用 vLLM，是比较常见的工程组合。

**Q80：推理时 temperature、top_p 怎么调？**

**答：**
医疗问答一般不希望太发散，所以 temperature 可以低一些，比如 0.1-0.7，top_p 也不要太高。需要事实稳定性时偏低，需要表达丰富时稍高。医疗场景优先稳定和安全。

**Q81：为什么 base 模型会输出 `<think>`？**

**答：**
可能是基座模型或模板带有推理模型风格，未微调前它不一定遵循目标中文医疗回答格式。SFT 后模型学习到数据集的直接回答风格，所以 `<think>` 现象减少。

## 10. Transformer 与大模型基础八股

**Q82：Transformer 的核心结构是什么？**

**答：**
核心是 self-attention + feed-forward network + residual connection + normalization。Decoder-only LLM 使用因果 mask，只能看当前位置之前的 token，用自回归方式逐 token 生成。

**Q83：Attention 公式是什么？**

**答：**
Attention(Q,K,V)=softmax(QK^T / sqrt(d_k))V。Q 表示查询，K 表示键，V 表示值。缩放因子 sqrt(d_k) 用于防止点积过大导致 softmax 梯度变小。

**Q84：Multi-Head Attention 为什么要多头？**

**答：**
多头可以让模型在不同子空间关注不同关系，比如实体、语法、长距离依赖等。多个头并行计算后拼接，再通过线性层融合。

**Q85：什么是 causal mask？**

**答：**
因果 mask 用于自回归生成，保证当前位置不能看到未来 token，避免信息泄露。

**Q86：什么是 KV Cache？**

**答：**
生成时，每一步都会用到历史 token 的 K/V。KV Cache 把历史 K/V 缓存起来，下一步只计算新 token 的 Q/K/V，避免重复计算，从而显著加速解码。

**Q87：为什么大模型推理越长越慢？**

**答：**
因为每生成一个新 token，都要和历史上下文做 attention，KV Cache 虽然避免重复算 K/V，但 attention 仍然要访问越来越长的历史缓存，显存和带宽压力会增加。

**Q88：RoPE 是什么？**

**答：**
RoPE 是旋转位置编码，通过对 Q/K 向量施加与位置相关的旋转，让模型感知 token 的相对位置信息。很多 LLaMA/Qwen 类模型都使用 RoPE。

**Q89：PreNorm 和 PostNorm 区别？**

**答：**
PreNorm 是在 attention/FFN 前做 normalization，训练深层模型更稳定；PostNorm 是在残差之后做 normalization，早期 Transformer 常用但深层训练更难。

**Q90：大模型预训练目标是什么？**

**答：**
Decoder-only LLM 通常用 causal language modeling，也就是根据前文预测下一个 token，最大化训练语料的似然。

## 11. RAG、微调、Agent 与幻觉治理

**Q91：RAG 和微调有什么区别？**

**答：**
微调用来改变模型参数，让模型学会领域风格、任务格式或稳定能力；RAG 不改参数，而是在推理时检索外部知识，适合知识更新快、需要可追溯来源的场景。医疗知识更新和安全要求高，真实应用里通常会结合 RAG。

**Q92：你的医疗项目为什么没有用 RAG？**

**答：**
这个项目目标是后训练闭环，重点验证 SFT + DPO 对医疗问答风格和指标的提升。RAG 是后续部署增强方向，可以接入医学指南、药品说明书、医院知识库，提高事实性和可追溯性。

**Q93：如果加 RAG，你会怎么设计？**

**答：**
流程是：医学文档清洗 -> chunk 切分 -> embedding 向量化 -> 向量库检索 -> rerank -> 拼接证据到 prompt -> 生成回答 -> 引用来源和安全校验。医疗场景还要做权限、版本、来源可信度管理。

**Q94：RAG 的 chunk 怎么切？**

**答：**
可以按标题、段落、语义边界切分，并设置 overlap。医学指南类文档适合保留章节标题和小节路径，避免单个 chunk 丢失上下文。

**Q95：如何缓解大模型幻觉？**

**答：**
可以从数据、检索、提示、解码、评测、后处理多层做：高质量 SFT/DPO 数据，RAG 引入可靠证据，低 temperature，要求引用来源，对低置信问题拒答，加入事实性评测和人工审核。

**Q96：LLM-as-judge 有什么问题？**

**答：**
它成本低、可扩展，但也会有偏见、位置偏差、长度偏好、模型自信错误等问题。最好用多模型评审、随机打乱答案顺序、人工抽检来校准。

## 12. 高频开放题与自救话术

**Q97：如果面试官说“BLEU/ROUGE 提升不大”，怎么答？**

**答：**
我会承认自动指标只是一个角度。这里最重要的是相对提升：Base 到 SFT 提升明显，说明领域适配有效；DPO 在 SFT 基础上继续提升，说明偏好优化有帮助。但医疗问答不能只看 BLEU/ROUGE，后续必须加入事实性、安全性和人工评测。

**Q98：如果问“你的 DPO accuracy 是 1.0，是不是过拟合？”**

**答：**
有可能，因为 rejected 是自动构造的，难度偏低，模型很容易区分 chosen 和 rejected。所以我不会把它解释成模型真实偏好能力完美，只说明它在当前偏好验证集上学到了区分信号。后续需要更难的人工偏好对验证。

**Q99：如果问“这个项目难点在哪里？”**

**答：**
难点不在把命令跑起来，而在完整闭环：数据格式适配、SFT 和 DPO 的参数选择、adapter 路径管理、训练日志解释、Base/SFT/DPO 对照评估，以及对医疗场景安全风险的认识。

**Q100：如果问“你做了哪些消融实验？”**

**答：**
目前做了 Base、SFT、DPO 三组对照，这是训练阶段上的消融；还没系统做 LoRA vs DoRA、不同 rank、不同学习率、不同 checkpoint 的消融。后续我会优先补 checkpoint-600 vs checkpoint-891，以及 LoRA vs DoRA 对比。

**Q101：如果问“你如何排查训练效果不好？”**

**答：**
我会先看数据格式是否正确、template 是否匹配、loss 是否只算 assistant 部分、训练集是否有大量脏数据；再看学习率、batch、cutoff_len、rank 是否合适；然后抽样看生成结果，最后对比 Base/SFT/DPO 指标和人工样例。

**Q102：如果问“为什么你的评估 stage 还是 sft，但评估 DPO 模型？”**

**答：**
在 LLaMA-Factory 里生成式预测通常用 `stage: sft` 加 `do_predict: true` 的方式跑，并通过 `adapter_name_or_path` 加载不同 adapter。这里的 stage 不是说重新做 SFT，而是复用生成预测流程评估 DPO adapter。

**Q103：如果问“医疗模型能不能上线给患者用？”**

**答：**
不能直接上线用于诊断或治疗决策。当前项目是技术验证，证明领域微调有效。真实上线还需要医学专家评审、风险分级、合规审核、免责声明、人工兜底、日志追踪和持续评测。

**Q104：如果问“你从这个项目学到了什么？”**

**答：**
我学到大模型项目不是只训练一个 adapter，而是要形成工程闭环：数据版本、训练配置、checkpoint、评估、样例分析和风险边界都要说得清楚。尤其医疗场景，模型效果不能只看自动指标，还要重视安全性和事实性。
