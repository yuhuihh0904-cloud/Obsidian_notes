# 1.2 回顾 1D Convolution 和 2D Convolution

对应页码：p3-p5

## 1.2.1 1D Convolution

一维卷积中，每个输出元素是输入数组附近元素的加权和。

如果：

```text
MASK_WIDTH = 5
radius = MASK_WIDTH / 2 = 2
```

那么：

```text
P[i] = N[i-2] * M[0]
     + N[i-1] * M[1]
     + N[i]   * M[2]
     + N[i+1] * M[3]
     + N[i+2] * M[4]
```

边界处访问不到的元素称为：

```text
ghost elements / halo elements
```

通常可以视为 0。

---

## 1.2.2 2D Convolution

二维卷积中，每个输出像素由输入图像中一个邻域窗口加权求和得到。

例如 `5 × 5 mask`：

```text
每个 P[row][col] 需要访问 N 中以它为中心的 5 × 5 区域。
```

公式：

```text
P[row][col] = ΣΣ N[row + i][col + j] * M[i][j]
```

其中 `M` 是 mask/filter/kernel。

---

# 1.3 Convolution 是否 memory-limited？

对应页码：p5

PPT 明确指出：1D 和 2D convolution 都是 memory-limited。

---

## 1.3.1 1D 情况

每个输出元素需要：

```text
2 * MASK_WIDTH loads
```

这里包括：

```text
从 N 读取 MASK_WIDTH 个输入
从 M 读取 MASK_WIDTH 个 mask
```

同时有：

```text
2 * MASK_WIDTH floating-point operations
```

即每个输入元素做一次乘法和一次加法。

所以：

```text
计算量和访存量同阶，容易受 memory bandwidth 限制。
```

---

## 1.3.2 2D 情况

每个输出元素需要：

```text
2 * MASK_WIDTH^2 loads
```

包括：

```text
N 中 MASK_WIDTH^2 个输入
M 中 MASK_WIDTH^2 个 mask 权重
```

并且需要：

```text
2 * MASK_WIDTH^2 floating-point operations
```

所以 2D convolution 也是 memory-limited。

---

# 1.4 Tiled 1D Convolution 基本思想

对应页码：p6-p10

## 1.4.1 为什么要 tiling？

卷积中相邻输出元素会使用大量相同输入元素。

例如 `MASK_WIDTH = 5`：

```text
P[4] 用 N[2], N[3], N[4], N[5], N[6]
P[5] 用 N[3], N[4], N[5], N[6], N[7]
P[6] 用 N[4], N[5], N[6], N[7], N[8]
```

可以看到：

```text
N[4], N[5], N[6] 会被多个输出元素重复使用。
```

所以可以把输入 tile 放到 shared memory 中，避免每个线程都从 global memory 重新读。

---

## 1.4.2 Output tile 和 Input tile 的区别

这是 Lecture 9 的重点。

### Output tile

output tile 是一个 block 要计算的输出范围。

例如：

```text
TILE_SIZE = 4
```

那么一个 block 可能计算：

```text
P[4], P[5], P[6], P[7]
```

---

### Input tile

为了计算这些输出，不只需要：

```text
N[4], N[5], N[6], N[7]
```

还需要左右 halo：

```text
左 halo：N[2], N[3]
右 halo：N[8], N[9]
```

所以 input tile 更大：

```text
input tile size = TILE_SIZE + MASK_WIDTH - 1
```

如果：

```text
TILE_SIZE = 4
MASK_WIDTH = 5
```

则：

```text
input tile size = 4 + 5 - 1 = 8
```

---

## 1.4.3 halo 是什么？

halo 是为了计算 tile 边界输出而额外需要的输入元素。

对于 1D：

```text
left halo：output tile 左边额外需要的输入
right halo：output tile 右边额外需要的输入
```

对于 2D：

```text
halo 是 output tile 周围的一圈输入区域。
```

对于 3D：

```text
halo 是 output tile 周围的一层输入体素区域。
```

---

# 1.5 是否把 halo 也放进 shared memory？

对应页码：p10-p14

PPT 讨论了两个方案：

1. halo 直接从 global memory 读。
2. halo 也加载到 shared memory。

---

# 1.6 方案一：halo 从 global memory 读

对应页码：p11-p12

## 1.6.1 思路

只把 output tile 对应的主体输入放入 shared memory：

```text
shared memory 只存 tile 内部数据
halo 不存
```

计算时：

```text
如果访问的是 tile 内部，就从 shared memory 读；
如果访问的是 halo，就从 global memory 读。
```

---

## 1.6.2 优点

```text
节省 shared memory。
```

因为 halo 不加载到 shared memory 中。

PPT 说：

```text
optimize reuse of shared memory
```

原因是 halo 的复用次数相对较少。

---

## 1.6.3 缺点

缺点很重要：

### 缺点 1：Branch divergence

因为线程计算时要判断：

```text
当前访问位置是否在 shared memory tile 内？
```

如果在：

```text
从 shared memory 读
```

如果不在：

```text
从 global memory 读
```

同一个 warp 内不同线程可能走不同路径，造成 branch divergence。

---

### 缺点 2：halo 太窄，不适合 memory burst

halo 区域宽度通常只有：

```text
MASK_WIDTH / 2
```

很窄。

如果单独从 global memory 读取 halo，可能不能充分利用 DRAM burst 和 coalescing。

PPT 提到：

```text
Halo too narrow to fill a memory burst.
```

---

## 1.6.4 代码结构

对应页码：p12

~~~cpp
__global__
void convolution_1D_tiled_cache_kernel(float *N, float *P, int Width)
{
    __shared__ float tile[TILE_WIDTH];

    int This_tile_start_point = blockIdx.x * blockDim.x;
    int i = This_tile_start_point + threadIdx.x;

    tile[threadIdx.x] = N[i]; // PPT 此处说明缺少边界检查
    __syncthreads();

    int radius = MASK_WIDTH / 2;
    int N_start_point = i - radius;

    float Pvalue = 0;

    for (int j = 0; j < MASK_WIDTH; j++) {
        int N_index = N_start_point + j;

        if ((N_index >= 0) && (N_index < Width)) {
            int tile_index = N_index - This_tile_start_point;

            if ((tile_index >= 0) && (tile_index < blockDim.x))
                Pvalue += tile[tile_index] * Mc[j];
            else
                Pvalue += N[N_index] * Mc[j];
        }
    }

    P[i] = Pvalue;
}
~~~

注意：

```text
这个版本中 mask Mc 通常放在 constant memory。
```

# 1.7 方案二：halo 也加载到 shared memory

对应页码：p13-p18

PPT 认为这是更好的答案。

---

## 1.7.1 优点

1. global memory load 更容易 coalesce。
2. 计算阶段不需要在 shared/global 之间分支。
3. 计算阶段没有 branch divergence。
4. halo 也在 shared memory 中，访存更快。

---

## 1.7.2 缺点

1. 需要更多 shared memory。
2. 部分线程需要加载多个元素。
3. 加载阶段可能产生 branch divergence。

不过 PPT 强调：

```text
加载阶段的分支一般比计算阶段的分支更容易接受。
```

因为计算阶段要执行 MASK_WIDTH 次循环，如果每次循环都有分支，代价更大。

---

## 1.7.3 shared memory 大小

如果 output tile 大小为：

```text
TILE_SIZE
```

mask 宽度为：

```text
MASK_WIDTH
```

那么 input tile 大小为：

```text
TILE_SIZE + MASK_WIDTH - 1
```

shared memory 声明：

~~~cpp
__shared__ float tile[TILE_SIZE + MASK_WIDTH - 1];
~~~

---

## 1.7.4 变量含义

对应页码：p14-p16

~~~cpp
int radius = MASK_WIDTH / 2;
int i = blockIdx.x * blockDim.x + threadIdx.x;
int start = i - radius;
~~~

| 变量 | 含义 |
|---|---|
| `radius` | halo 半径 |
| `i` | 当前线程对应的输出元素 P[i] |
| `start` | 当前线程要加载到 shared memory 的输入 N 的起始位置 |
| `tile[threadIdx.x]` | shared memory 中对应位置 |
| `tile[threadIdx.x + TILE_SIZE]` | 由部分线程额外加载的右侧 halo |

---

## 1.7.5 加载 input tile

对应页码：p16

第一部分：所有线程加载一个元素。

~~~cpp
if (0 <= start && Width > start) {
    tile[threadIdx.x] = N[start];
} else {
    tile[threadIdx.x] = 0.0f;
}
~~~

这里处理左边 ghost：

```text
如果 start < 0，说明访问到了数组左边界外，填 0。
```

第二部分：前 `MASK_WIDTH - 1` 个线程额外加载右侧 halo。

~~~cpp
if (MASK_WIDTH - 1 > threadIdx.x) {
    start += TILE_SIZE;

    if (Width > start) {
        tile[threadIdx.x + TILE_SIZE] = N[start];
    } else {
        tile[threadIdx.x + TILE_SIZE] = 0.0f;
    }
}
~~~

注意：

```text
只有部分线程需要做第二次 load。
```

最后同步：

~~~cpp
__syncthreads();
~~~

必须放在所有 if 外面，保证 block 内线程都执行到。

---

## 1.7.6 计算输出

对应页码：p17

~~~cpp
if (i < Width) {
    float Pvalue = 0;

    for (int j = 0; j < MASK_WIDTH; j++) {
        Pvalue += tile[threadIdx.x + j] * Mc[j];
    }

    P[i] = Pvalue;
}
~~~

这里没有再判断 halo 是否在 shared memory 中，因为已经全部加载进来了。

---

## 1.7.7 完整 1D tiled convolution kernel

整理后的考试版：

~~~cpp
__global__
void convolution_1D_tiled_kernel(float *N, float *P, int Width)
{
    __shared__ float tile[TILE_SIZE + MASK_WIDTH - 1];

    int radius = MASK_WIDTH / 2;
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    int start = i - radius;

    if (start >= 0 && start < Width)
        tile[threadIdx.x] = N[start];
    else
        tile[threadIdx.x] = 0.0f;

    if (threadIdx.x < MASK_WIDTH - 1) {
        int halo_index = start + TILE_SIZE;

        if (halo_index < Width)
            tile[threadIdx.x + TILE_SIZE] = N[halo_index];
        else
            tile[threadIdx.x + TILE_SIZE] = 0.0f;
    }

    __syncthreads();

    if (i < Width) {
        float Pvalue = 0.0f;

        for (int j = 0; j < MASK_WIDTH; j++) {
            Pvalue += tile[threadIdx.x + j] * Mc[j];
        }

        P[i] = Pvalue;
    }
}
~~~

# 1.8 第三种思路：并行化 tile loading

对应页码：p19-p23

前面两个方案的共同点是：

```text
一个线程主要负责计算一个 output element。
```

PPT 接着问：

```text
这是不是唯一选择？
```

答案：不是。

也可以：

```text
每个线程负责加载 input tile 的一个元素；
只有一部分线程参与计算 output。
```

---

## 1.8.1 Input parallelism

这种方式也可以叫 input parallelism。

特点：

```text
thread block 的大小匹配 input tile，而不是 output tile。
```

也就是说：

```text
线程数 = input tile 大小
```

其中 input tile 包含：

```text
output tile + halo
```

---

## 1.8.2 优点

对应页码：p20

1. 加载阶段没有 branch divergence。
2. 每个线程加载一个 input element。
3. 避免窄 halo global access。
4. global memory load 更规则。

---

## 1.8.3 缺点

1. 不是所有线程都计算输出。
2. 计算阶段有 branch divergence。
3. halo 区域对应的线程只负责 load，不负责 compute。
4. 线程利用率下降。

PPT 表述：

```text
Advantage:
No branch divergence for load high latency.
Avoid narrow global access.

Disadvantage:
Branch divergence for compute low latency.
```

# 1.9 2D tiled convolution

对应页码：p21-p24 及后续

## 1.9.1 2D input tile 和 output tile

二维卷积中，如果 output tile 大小是：

```text
TILE_WIDTH × TILE_WIDTH
```

mask 大小是：

```text
MASK_WIDTH × MASK_WIDTH
```

那么 input tile 大小是：

```text
(TILE_WIDTH + MASK_WIDTH - 1) × (TILE_WIDTH + MASK_WIDTH - 1)
```

因为上下左右都需要 halo。

---

## 1.9.2 output tile 坐标

PPT 中给出 output 坐标：

~~~cpp
col_o = blockIdx.x * TILE_WIDTH + threadIdx.x;
row_o = blockIdx.y * TILE_WIDTH + threadIdx.y;
~~~

含义：

```text
当前线程对应的输出位置是 P[row_o][col_o]
```

如果采用 input parallelism，thread block 比 output tile 大，那么不是所有 threadIdx 都对应有效输出。

---

## 1.9.3 input tile 坐标

输入 tile 坐标一般要向左上偏移 radius：

~~~cpp
int radius = MASK_WIDTH / 2;

int col_i = blockIdx.x * TILE_WIDTH + threadIdx.x - radius;
int row_i = blockIdx.y * TILE_WIDTH + threadIdx.y - radius;
~~~

其中：

```text
(row_i, col_i) 是要从 N 中加载到 shared memory 的输入坐标。
```

---

## 1.9.4 2D tiled convolution 代码思路

~~~cpp
#define IN_TILE_WIDTH  (TILE_WIDTH + MASK_WIDTH - 1)

__global__
void convolution_2D_tiled_kernel(float *N, float *P, int Width, int Height)
{
    __shared__ float tile[IN_TILE_WIDTH][IN_TILE_WIDTH];

    int radius = MASK_WIDTH / 2;

    int tx = threadIdx.x;
    int ty = threadIdx.y;

    int row_o = blockIdx.y * TILE_WIDTH + ty;
    int col_o = blockIdx.x * TILE_WIDTH + tx;

    int row_i = row_o - radius;
    int col_i = col_o - radius;

    if (row_i >= 0 && row_i < Height && col_i >= 0 && col_i < Width)
        tile[ty][tx] = N[row_i * Width + col_i];
    else
        tile[ty][tx] = 0.0f;

    __syncthreads();

    if (ty < TILE_WIDTH && tx < TILE_WIDTH &&
        row_o < Height && col_o < Width) {

        float Pvalue = 0.0f;

        for (int i = 0; i < MASK_WIDTH; i++) {
            for (int j = 0; j < MASK_WIDTH; j++) {
                Pvalue += tile[ty + i][tx + j] *
                          Mc[i * MASK_WIDTH + j];
            }
        }

        P[row_o * Width + col_o] = Pvalue;
    }
}
~~~

注意：

```text
这个代码是思路版，真实写法要确保 blockDim.x/y 等于 input tile 大小或处理多次加载。
```

# 1.10 Lecture 9 三种 input tile loading 风格总结

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| halo 从 global memory 读 | shared memory 只放 output tile 对应输入 | shared memory 用得少 | 计算时分支多，halo 访问不 coalesced |
| halo 也放 shared memory | shared memory 放完整 input tile | 计算阶段无 shared/global 分支，coalescing 更好 | 部分线程多加载，shared memory 更多 |
| input parallelism | 每个线程加载 input tile 一个元素，部分线程计算输出 | 加载阶段规则、无高延迟分支 | 计算阶段部分线程不工作，有 divergence |

# 2.2 Stencil Algorithms

对应页码：p3

## 2.2.1 什么是 stencil？

Stencil algorithm 是一种数值计算模式：

```text
按照固定邻域模式更新数组元素。
```

卷积就是一种 stencil。

---

## 2.2.2 例子

PPT 给了多个例子：

1. 2D convolutional kernel，9-point 2D stencil。
2. Nearest neighbor lattice Dirac operator，5-point 2D stencil。
3. Finite difference stencil，13-point 3D stencil。
4. Wilson-Dslash operator，4D stencil。

可以这样理解：

```text
只要每个输出点依赖周围固定模式的邻居点，就可以看作 stencil。
```

# 2.3 1D tiled convolution 带宽收益分析

对应页码：p4-p12

## 2.3.1 小例子

对应页码：p4

假设：

```text
TILE_WIDTH = 8
MASK_WIDTH = 5
```

一个 block 计算 8 个输出：

```text
P[8] 到 P[15]
```

为了计算这些输出，需要加载 input tile：

```text
8 + (5 - 1) = 12 个输入元素
```

即：

```text
TILE_WIDTH + MASK_WIDTH - 1
```

---

## 2.3.2 不使用 tiling 时的 global memory access

每个输出需要：

```text
MASK_WIDTH = 5
```

个输入元素。

8 个输出需要：

```text
8 × 5 = 40
```

次对 N 的 global memory access。

---

## 2.3.3 使用 tiling 后的 global memory access

使用 shared memory tiling 后，一个 block 只需要把 input tile 加载一次：

```text
8 + 5 - 1 = 12
```

次 global memory load。

之后的 40 次访问都变为 shared memory access。

---

## 2.3.4 带宽减少比例

对应页码：p6

```text
reduction = 40 / 12 = 3.3
```

也就是说，对 N 的 global memory access 理想情况下减少约 3.3 倍。

---

## 2.3.5 1D 一般公式

对应页码：p7

一般情况下：

```text
加载元素数 = TILE_WIDTH + MASK_WIDTH - 1
```

原始 global memory access：

```text
TILE_WIDTH × MASK_WIDTH
```

所以：

```text
Bandwidth reduction =
(TILE_WIDTH × MASK_WIDTH) / (TILE_WIDTH + MASK_WIDTH - 1)
```

---

## 2.3.6 从 reuse 次数看

对应页码：p8-p9

对于 `TILE_WIDTH = 8, MASK_WIDTH = 5`：

```text
边缘输入复用少
中间输入复用多
```

复用次数大致是：

```text
1, 2, 3, 4, 5, 5, 5, 5, 4, 3, 2, 1
```

总和：

```text
40
```

因为这些 shared memory 中的输入元素总共服务了 40 次原本的 global memory 访问。

---

## 2.3.7 边界 tile

对应页码：p10-p11

对于最左边或最右边的 boundary tile，ghost elements 不是真实输入。

例如左边界：

```text
P[0] 只使用 N[0], N[1], N[2]
P[1] 使用 N[0], N[1], N[2], N[3]
P[2] 使用 N[0], N[1], N[2], N[3], N[4]
```

所以边界 tile 的访问比例和中间 tile 不完全一样。

PPT 例子：

```text
TILE_WIDTH = 8
MASK_WIDTH = 5
boundary tile load = 10
计算访问 = 37
reduction = 37 / 10 = 3.7
```

---

## 2.3.8 1D bandwidth reduction 表

对应页码：p12

PPT 给出的趋势：

```text
TILE_WIDTH 越大，reduction 越接近 MASK_WIDTH。
MASK_WIDTH 越大，潜在复用越多。
```

例如：

| TILE_WIDTH | Mask=5 reduction | Mask=9 reduction |
|---|---|---|
| 16 | 4.0 | 6.0 |
| 32 | 4.4 | 7.2 |
| 64 | 4.7 | 8.0 |
| 128 | 4.9 | 8.5 |
| 256 | 4.9 | 8.7 |

结论：

```text
1D convolution 中，TILE_WIDTH 很大时，带宽减少比例趋近于 MASK_WIDTH。
```

# 2.4 2D tiled convolution 带宽收益分析

对应页码：p13-p16

## 2.4.1 例子

假设：

```text
Output tile = 8 × 8
MASK_WIDTH = 5
```

那么 input tile 大小是：

```text
(8 + 5 - 1) × (8 + 5 - 1)
= 12 × 12
= 144
```

所以 tiled 版本需要：

```text
144 次 global memory load
```

---

## 2.4.2 不使用 tiling 时

每个输出元素需要访问：

```text
5 × 5 = 25
```

个输入元素。

输出 tile 有：

```text
8 × 8 = 64
```

个输出。

所以原始 global memory access：

```text
8 × 8 × 25 = 1600
```

---

## 2.4.3 2D 带宽减少比例

对应页码：p14

```text
reduction = 1600 / 144 = 11.1
```

所以 2D convolution 的 tiling 收益通常比 1D 更明显。

---

## 2.4.4 2D 一般公式

对应页码：p15

加载 input tile 的元素数：

```text
(TILE_WIDTH + MASK_WIDTH - 1)^2
```

原始 global memory access：

```text
TILE_WIDTH^2 × MASK_WIDTH^2
```

所以：

```text
Bandwidth reduction =
(TILE_WIDTH^2 × MASK_WIDTH^2) /
(TILE_WIDTH + MASK_WIDTH - 1)^2
```

---

## 2.4.5 2D bandwidth reduction 表

对应页码：p16

趋势：

```text
TILE_WIDTH 越大，reduction 越大；
MASK_WIDTH 越大，reduction 也越大。
```

例如 PPT 给出：

| TILE_WIDTH | Mask=5 reduction | Mask=9 reduction |
|---|---|---|
| 8 | 11.1 | 20.3 |
| 16 | 16 | 36 |
| 32 | 19.7 | 51.8 |
| 64 | 22.1 | 64 |

# 2.5 为什么 2D tiling 收益更大？

因为 2D 中每个输入元素可以被二维邻域内多个输出重复使用。

1D 的最大复用大约是：

```text
MASK_WIDTH
```

2D 的最大复用大约是：

```text
MASK_WIDTH^2
```

所以 2D convolution 更适合 shared memory tiling。

| 层次 | 问题 |
|---|---|
| Register file | highly banked，但也可能有 bank conflict |
| Shared memory | highly banked，也可能有 bank conflict |

# 2.7 Problem Solving 题型

对应页码：p29-p32

## 2.7.1 题型 1：边界 block 的 global memory reduction

题目类型：

```text
2D tiled convolution，output parallelism，多次加载 input tile。
mask = 5×5，output tile = 16×16，input image = 32×32。
问 block(0,0) global memory access reduction。
```

解题思路：

1. 先判断 block(0,0) 在左上角，是 boundary tile。
2. mask radius = 2。
3. 不完整 halo 处是 ghost，不需要真实 global load。
4. 统计计算阶段实际使用的输入访问数。
5. 统计实际加载到 shared memory 的 input tile 元素数。
6. 用：

```text
reduction = 原始 global 访问数 / tiled global load 数
```

---

## 2.7.2 题型 2：input parallelism 下 warp divergence

题目类型：

```text
input parallelism，30×30 output tile，3×3 mask，因此 input tile 是 32×32。
问一个 block 中多少 warp 有 control divergence？
```

分析：

```text
block size = 32×32 = 1024 threads
一共有 1024 / 32 = 32 个 warp
```

所有线程都参与 load。

但只有满足：

```text
tx < 30 && ty < 30
```

的线程参与 compute。

所以：

```text
有些 warp 中部分线程 compute，部分线程不 compute，这就产生 divergence。
```

PPT 中结论：

```text
只有 1 个 warp 没有 control divergence。
```

对于大图内部 tile，可能有 2 个 warp 没有 divergence。

---

## 2.7.3 题型 3：Byte-to-FLOP ratio

题目类型：

```text
A40 GPU：FP32 peak = 37.4 TFLOPS
memory bandwidth = 696 GB/s
问要充分利用计算能力，byte-to-FLOP ratio 应为多少？
```

计算：

```text
696 GB/s / 37.4 TFLOPS = 0.019 B/FLOP
```

换句话说：

```text
每从 global memory 读取 1 byte，需要做约 53.7 个 FLOP
```

才能充分利用计算资源。

结论：

```text
如果算法每读取 1 byte 只做很少计算，就会 memory-bound。
```

# 3.2 Reduction 是什么？

对应页码：p5-p8

## 3.2.1 基本定义

Reduction 是把一组输入值归约成一个输出值。

例如：

```text
sum:  a0 + a1 + a2 + ... + an
max:  max(a0, a1, ..., an)
min:  min(a0, a1, ..., an)
product: a0 * a1 * ... * an
```

---

## 3.2.2 reduction 的操作要求

用于 reduction 的二元操作通常需要：

1. associative，结合律。
2. commutative，交换律。
3. identity value，单位元。

例如：

| 操作 | 单位元 |
|---|---|
| sum | 0 |
| product | 1 |
| min | 最大可能值 |
| max | 最小可能值 |

---

## 3.2.3 为什么需要结合律和交换律？

因为并行 reduction 会改变操作顺序。

例如：

```text
串行 sum:
(((a+b)+c)+d)

并行 sum:
(a+b) + (c+d)
```

如果操作不满足结合律和交换律，改变顺序可能导致结果不同。

注意：

```text
浮点数加法严格来说不满足结合律。
```

所以浮点 reduction 可能和串行结果有细微差别。

---

# 3.3 Sequential Reduction

对应页码：p9

顺序 reduction：

~~~cpp
result = identity;

for each value in input:
    result = result op value;
~~~

时间复杂度：

```text
O(N)
```

工作量：

```text
N - 1 次二元操作
```

---

# 3.4 Parallel Reduction Tree

对应页码：p10-p14

## 3.4.1 Tree 思想

并行 reduction 使用树形结构。

例如 max reduction：

```text
输入：3 1 7 0 4 1 6 3

第 1 层：
max(3,1)=3
max(7,0)=7
max(4,1)=4
max(6,3)=6

第 2 层：
max(3,7)=7
max(4,6)=6

第 3 层：
max(7,6)=7
```

最终结果：

```text
7
```

---

## 3.4.2 时间步数

如果有足够资源，树形 reduction 需要：

```text
log2(N)
```

步。

例如：

```text
N = 1,000,000
log2(N) ≈ 20
```

所以理论上非常快。

---

## 3.4.3 Work-efficient

对应页码：p12

虽然 parallel reduction 有很多线程，但总操作次数仍然是：

```text
N - 1
```

和顺序算法一样。

所以它是 work-efficient。

---

## 3.4.4 Diminishing Parallelism

对应页码：p13-p14

Reduction 的并行度每一层都会减半：

```text
第一步：N/2 个操作
第二步：N/4 个操作
第三步：N/8 个操作
...
最后：1 个操作
```

这叫：

```text
diminishing parallelism
```

问题：

```text
后期活跃线程越来越少，硬件利用率下降。
```

---

# 3.5 CUDA 中的 Reduction 策略

对应页码：p15-p17

## 3.5.1 基本策略

PPT 给出的 CUDA 策略：

```text
N 个值在 global memory 中。
每个 thread block 有 M 个线程。
每个 block 处理 2M 个输入值。
在 shared memory 中把 2M 个值 reduce 成 1 个值。
每个 block 写回一个 partial result 到 global memory。
之后再继续 reduce 这些 partial result。
```

---

## 3.5.2 为什么每个 block 处理 2M 个值？

如果 block 有 M 个线程，每个线程一开始可以加载两个值并合并：

```text
thread t 加载 input[2t] 和 input[2t+1]
```

这样可以：

1. 增加每个线程的工作。
2. 减少后续 shared memory 归约元素数。
3. 更好利用 global memory bandwidth。

---

## 3.5.3 CUDA reduction algorithm

对应页码：p16

步骤：

```text
1. Read block of 2M values into shared memory.
2. For each of log(2M) steps:
   - each active thread combines two values
   - writes result to shared memory
   - active threads halve
3. Write final result back to global memory.
```

---

# 3.6 Naive Data Mapping

对应页码：p17-p20

## 3.6.1 简单映射

每个线程开始处理两个相邻位置：

```text
Thread 0: input[0], input[1]
Thread 1: input[2], input[3]
Thread 2: input[4], input[5]
...
```

第一步写回偶数下标。

之后：

```text
stride 翻倍
活跃线程减半
```

最终结果在：

```text
partialSum[0]
```

---

## 3.6.2 reduction steps 代码

对应页码：p20

PPT 中 naive 写法大致是：

~~~cpp
for (unsigned int stride = 1;
     stride <= blockDim.x;
     stride *= 2)
{
    __syncthreads();

    if (t % stride == 0)
        partialSum[2 * t] += partialSum[2 * t + stride];
}
~~~

注意：

```text
这段代码是为了说明 reduction step，不一定是最佳实现。
```

---

# 3.7 为什么需要 `__syncthreads()`？

对应页码：p20-p22

每一轮 reduction 都依赖上一轮结果。

如果没有同步，可能出现：

```text
某些线程还没写完 partialSum
另一些线程已经进入下一轮并读取 partialSum
```

结果就可能错误。

所以每一轮开始前需要：

~~~cpp
__syncthreads();
~~~

作用：

```text
保证上一轮所有 partial sum 都已经生成，下一轮才能使用。
```

---

# 3.8 没有同步会怎样？

对应页码：p22-p23

PPT 给了错误例子：

```text
某个 partial sum 还没有算完
其他线程就使用了旧值
导致最终结果错误
```

所以 reduction 中 shared memory 迭代更新时，barrier synchronization 是必要的。

---

# 3.9 Naive reduction 的性能问题

虽然 PPT 前半段先讲 naive mapping，但考试通常会问它的问题。

## 3.9.1 Control divergence

naive reduction 中：

~~~cpp
if (t % stride == 0)
~~~

随着 stride 变化，warp 内只有部分线程活跃，容易造成 branch divergence。

---

## 3.9.2 Modulo 运算成本

~~~cpp
t % stride
~~~

取模操作本身也比简单比较更贵。

---

## 3.9.3 Shared memory bank conflict

不同 stride 下，线程访问 shared memory 的模式可能导致 bank conflict。

---

## 3.9.4 Global memory access

需要尽量让最初从 global memory 读取数据时 coalesced。

# 3.10 更常见的优化版 reduction 思路

考试可写这个版本，结构更清晰：

~~~cpp
__global__
void reduce_sum_kernel(float *input, float *partial, int N)
{
    extern __shared__ float sdata[];

    unsigned int t = threadIdx.x;
    unsigned int i = blockIdx.x * blockDim.x * 2 + threadIdx.x;

    float x = 0.0f;

    if (i < N)
        x += input[i];

    if (i + blockDim.x < N)
        x += input[i + blockDim.x];

    sdata[t] = x;

    __syncthreads();

    for (unsigned int stride = blockDim.x / 2;
         stride > 0;
         stride >>= 1)
    {
        if (t < stride) {
            sdata[t] += sdata[t + stride];
        }

        __syncthreads();
    }

    if (t == 0)
        partial[blockIdx.x] = sdata[0];
}
~~~

---

## 3.10.1 优化点

| 优化点 | 作用 |
|---|---|
| 每线程加载两个元素 | 减少初始元素数，提高工作量 |
| 使用 shared memory | block 内快速归约 |
| `t < stride` | 比取模更简单 |
| stride 每轮减半 | 活跃线程连续，减少 divergence |
| block 输出一个 partial sum | 多 block 并行处理大数组 |

---

# 3.11 多 block reduction

一个 kernel 中每个 block 得到一个 partial result：

```text
partial[blockIdx.x]
```

如果 partial 数量仍然大于 1，需要：

```text
再次启动 kernel
```

继续归约。

原因：

```text
CUDA block 之间不能在一个 kernel 内全局同步。
```

所以常见做法：

```text
kernel 1：N 个数 -> 多个 block partial result
kernel 2：partial result -> 更少 result
...
直到剩下一个值
```

# 3.12 Lecture 11 高频答题模板

## 3.12.1 什么是 reduction？

> Reduction 是把一组输入值通过某个二元操作合并成一个输出值的并行计算模式，例如 sum、max、min、product。这个操作通常需要满足结合律、交换律，并具有单位元，这样才能安全地改变计算顺序并并行执行。

---

## 3.12.2 为什么 reduction 可以并行？

> 因为 reduction 的操作满足结合律和交换律，输入数据可以被分成多个子集并行处理。每个线程或线程块先计算局部结果，再通过树形结构逐层合并，最终得到全局结果。

---

## 3.12.3 为什么 reduction 需要 `__syncthreads()`？

> 在 shared memory reduction 中，每一轮的计算依赖上一轮写入 shared memory 的 partial sum。如果没有 `__syncthreads()`，某些线程可能在其他线程尚未完成写入时就读取数据，导致读取旧值或未完成值，从而产生错误结果。因此每一轮之间需要同步。

---

## 3.12.4 reduction 的性能问题是什么？

> Reduction 的并行度会随着树的层次逐步减半，后期活跃线程很少，硬件利用率下降。naive reduction 还可能有 branch divergence、取模操作开销、shared memory bank conflict 和 global memory 非合并访问等问题。因此 GPU reduction 需要通过连续线程参与、shared memory、coalesced loading 和多阶段 kernel 等方式优化。

# 4.3 Read-Modify-Write 和 Data Race

对应页码：p5-p9

## 4.3.1 普通自增不是原子操作

例如：

~~~cpp
Mem[x] = Mem[x] + 1;
~~~

看起来是一句代码，但实际包括三步：

```text
Old = Mem[x]
New = Old + 1
Mem[x] = New
```

这叫：

```text
read-modify-write
```

---

## 4.3.2 两个线程同时执行会怎样？

假设初始：

```text
Mem[x] = 0
```

Thread 1 和 Thread 2 都执行：

```text
Mem[x]++
```

可能结果有两种：

### 正确情况

```text
Thread 1 先完成，Mem[x] = 1
Thread 2 再执行，Mem[x] = 2
最终 Mem[x] = 2
```

### 错误情况

```text
Thread 1 读到 0
Thread 2 也读到 0
Thread 1 写 1
Thread 2 也写 1
最终 Mem[x] = 1
```

这就是 data race。

# 4.4 Atomic Operation 的作用

对应页码：p10-p17

## 4.4.1 Atomic prevents interleaving

Atomic operation 保证：

```text
一个线程的 read-modify-write 操作执行期间，
其他线程不能插入访问同一个地址。
```

也就是说，对同一个地址的 atomic 操作看起来像串行执行。

---

## 4.4.2 Atomicity 是相对的

对应页码：p13

PPT 强调：

```text
Atomicity is always with respect to something.
```

意思是：

```text
原子性总是相对于某些操作或某个地址而言。
```

两个代码段如果相互原子，则软件上看起来它们没有交错执行。

---

## 4.4.3 Atomic 不保证顺序

对应页码：p14

Atomic operation 保证的是：

```text
不会交错导致数据破坏。
```

但不保证：

```text
哪个线程先执行。
```

所以 atomic 只能保证正确性，不保证确定顺序。

---

## 4.4.4 硬件如何实现 atomic？

对应页码：p15-p16

许多 ISA 提供同步原语，例如：

- bit test and set
- compare and swap / exchange
- swap / exchange
- fetch and increment / add

硬件保证：

```text
当一个线程对某地址执行 atomic 操作时，
其他线程访问该地址必须等待。
```

因此：

```text
对同一地址的 atomic 操作会串行化。
```

## 4.5.2 atomicAdd

它可以作用于：

```text
global memory 或 shared memory 中的地址。
```

unsigned int atomicAdd(unsigned int* address, unsigned int val);

~~~

其中 float atomicAdd 需要一定 compute capability 支持。

# 4.6 CAS：Compare And Swap

对应页码：p20-p21

## 4.6.1 atomicCAS

CAS 的逻辑：

~~~cpp
bool atomicCAS(int *address, int old, int new)
{
    if (*address != old)
        return false;

    *address = new;
    return true;
}
~~~

真实硬件中，这整个比较和写入过程是 atomic 的。

---

## 4.6.2 用 CAS 实现 atomicAdd

对应页码：p21

~~~cpp
int atomicAdd(int* address, int value)
{
    bool done = false;
    int old_v;

    while (!done) {
        old_v = *address;
        done = atomicCAS(address, old_v, old_v + value);
    }

    return old_v;
}
~~~

逻辑：

```text
先读取旧值 old_v
尝试把 old_v 替换成 old_v + value
如果期间没有其他线程修改成功，则 CAS 成功
如果失败，说明别人改了，重新尝试
```

# 4.7 Histogramming

对应页码：p22-p29

## 4.7.1 Histogram 是什么？

Histogram 用于统计数据集中不同值出现的次数。

例如：

```text
输入：一段文本
输出：每个字母出现次数
```

或者：

```text
输入：图像像素值
输出：0-255 每个灰度值出现次数
```

---

## 4.7.2 histogram 的基本算法

对于每个输入元素：

```text
bin = input[i]
histo[bin]++
```

并行化时，不同线程可能更新同一个 bin。

例如多个线程同时看到字母 `M`：

```text
atomicAdd(&histo['M'], 1)
```

---

## 4.7.3 为什么 histogram 需要 atomic？

因为多个输入值可能属于同一个 bin。

如果多个线程同时执行：

~~~cpp
histo[bin]++;
~~~

就会发生 read-modify-write data race。

所以需要：

~~~cpp
atomicAdd(&histo[bin], 1);
~~~

---

## 4.7.4 基本 histogram kernel

考试可写：

~~~cpp
__global__
void histo_kernel(unsigned char *buffer,
                  long size,
                  unsigned int *histo)
{
    int i = threadIdx.x + blockIdx.x * blockDim.x;
    int stride = blockDim.x * gridDim.x;

    while (i < size) {
        atomicAdd(&(histo[buffer[i]]), 1);
        i += stride;
    }
}
~~~

解释：

| 代码 | 含义 |
|---|---|
| `buffer[i]` | 当前数据值 |
| `histo[buffer[i]]` | 对应 bin |
| `atomicAdd` | 原子地给 bin 加 1 |
| `stride` | grid-stride loop，保证线程覆盖整个输入 |

---

# 4.8 Atomic 操作的性能问题

对应页码：p35-p41

## 4.8.1 Atomic on DRAM 很慢

PPT 提到 atomic operation on DRAM：

1. 以 read 开始，几百 cycles。
2. 以 write 结束，几百 cycles。
3. 整个期间其他线程不能访问该位置。

所以：

```text
atomic operation 会串行化同一地址上的访问。
```

---

## 4.8.2 Contention

如果很多线程更新同一个 bin：

```text
atomicAdd(&histo[0], 1)
```

它们会排队执行。

这种竞争叫：

```text
contention
```

contention 越严重，性能越差。

---

## 4.8.3 Atomic throughput 受限

因为 atomic 不是普通 load/store，它需要：

```text
read-modify-write
保证互斥
处理冲突
```

所以吞吐远低于普通内存访问。

对应页码：p39-p41

PPT 提到两类硬件改进：

## 4.9.1 L2 cache atomic

如果 atomic 可以在 L2 cache 中完成：

```text
比 DRAM atomic 快。
```

但如果多个线程访问同一地址：

```text
仍然会串行化。
```

---

## 4.9.2 Shared memory atomic

shared memory atomic 延迟更低：

```text
Very short latency, but still serialized.
```

特点：

| 类型 | 作用域 | 性能 |
|---|---|---|
| global memory atomic | 所有 block 共享 | 慢 |
| shared memory atomic | block 内私有 | 快 |

但 shared memory atomic 只能在 block 内使用，所以需要算法层面的改造。

# 4.10 Privatization 私有化

对应页码：p42-p45

这是 Lecture 13 最重要优化。

---

## 4.10.1 什么是 privatization？

Privatization 指：

```text
不要让所有线程直接更新一个全局 histo。
而是为每个 block 创建一个 private histogram。
每个 block 先在 shared memory 中统计自己的 histo。
最后再把 block-private histo 合并到 global histo。
```

---

## 4.10.2 为什么 privatization 有用？

原始版本：

```text
所有 block、所有线程竞争 global histo[bin]
```

私有化后：

```text
每个 block 只竞争自己的 shared histo_private[bin]
```

优点：

1. shared memory atomic 更快。
2. global atomic 数量大幅减少。
3. contention 分散到多个 block-private histogram。
4. 更容易提高吞吐。

---

## 4.10.3 private histogram 初始化

对应页码：p42

~~~cpp
__shared__ unsigned int histo_private[256];

if (threadIdx.x < 256)
    histo_private[threadIdx.x] = 0;

__syncthreads();
~~~

注意：

```text
必须同步，确保 private histo 全部清零后再统计。
```

---

## 4.10.4 构建 private histogram

对应页码：p43

~~~cpp
int i = threadIdx.x + blockIdx.x * blockDim.x;
int stride = blockDim.x * gridDim.x;

while (i < size) {
    atomicAdd(&(histo_private[buffer[i]]), 1);
    i += stride;
}
~~~

注意：

```text
这里 atomicAdd 作用在 shared memory 中。
```

---

## 4.10.5 合并到 global histogram

对应页码：p44

~~~cpp
__syncthreads();

if (threadIdx.x < 256) {
    atomicAdd(&(histo[threadIdx.x]),
              histo_private[threadIdx.x]);
}
~~~

这里仍然需要 global atomic，因为多个 block 会合并到同一个 global histo bin。

但是 global atomic 次数从：

```text
每个输入元素一次
```

减少到：

```text
每个 block 每个 bin 一次
```

例如 256 bins：

```text
每个 block 最多 256 次 global atomic
```

---

## 4.10.6 完整 privatized histogram kernel

~~~cpp
__global__
void histo_kernel_private(unsigned char *buffer,
                          long size,
                          unsigned int *histo)
{
    __shared__ unsigned int histo_private[256];

    if (threadIdx.x < 256)
        histo_private[threadIdx.x] = 0;

    __syncthreads();

    int i = threadIdx.x + blockIdx.x * blockDim.x;
    int stride = blockDim.x * gridDim.x;

    while (i < size) {
        atomicAdd(&(histo_private[buffer[i]]), 1);
        i += stride;
    }

    __syncthreads();

    if (threadIdx.x < 256) {
        atomicAdd(&(histo[threadIdx.x]),
                  histo_private[threadIdx.x]);
    }
}
~~~

# 4.11 Privatization 的适用条件

对应页码：p45

PPT 总结：

1. Privatization 是并行程序中非常常用的技术。
2. 操作需要满足 associative 和 commutative。
3. histogram add 满足结合律和交换律。
4. histogram 大小需要比较小，能放进 shared memory。
5. 如果 histogram 太大，就不适合直接 block-private shared memory。

---

# 4.12 如果 histogram 太大怎么办？

PPT 留了问题，但考试可以这样回答：

```text
如果 histogram 太大，无法完全放入 shared memory，可以：
1. 分批处理 bins；
2. 使用更粗粒度的 privatization，比如 per-block 部分 bins；
3. 使用 global memory 中的多份 private histogram；
4. 先局部统计再多阶段 merge；
5. 根据输入分布减少热点 bin contention。
```

# 4.13 Lecture 13 高频答题模板

## 4.13.1 为什么普通 `x++` 在并行中不安全？

> `x++` 实际上包含 read、modify、write 三步。如果两个线程同时读取同一个旧值，并分别计算新值后写回，就可能发生 lost update。例如初始值为 0，两个线程都执行加 1，最终结果可能是 1 而不是 2。因此并行更新共享位置时需要 atomic operation。

---

## 4.13.2 什么是 atomic operation？

> Atomic operation 是一种不可被其他相关操作交错打断的操作。对于同一个内存地址，atomic read-modify-write 会被硬件保证为一个整体执行，其他线程必须等待该操作完成。因此 atomic 可以避免 data race，但不能保证线程执行的相对顺序。

---

## 4.13.3 为什么 atomic operation 慢？

> Atomic operation 需要完成 read-modify-write，并保证同一地址上的操作互斥。如果多个线程同时更新同一地址，它们会被串行化。对于 global memory 或 DRAM 上的 atomic，还需要经历较高的读写延迟，因此会显著降低 memory throughput。冲突越严重，性能越差。

---

## 4.13.4 histogram 为什么需要 atomic？

> Histogram 中每个输入元素根据其值更新对应 bin。多个线程可能同时处理到相同值并更新同一个 bin，因此会对同一个 `histo[bin]` 执行自增操作。如果没有 atomic，会发生 lost update。使用 `atomicAdd(&histo[bin], 1)` 可以保证每次计数都被正确累加。

---

## 4.13.5 privatization 如何优化 histogram？

> Privatization 为每个 thread block 创建一个 private histogram，通常放在 shared memory 中。线程先使用 shared memory atomic 更新本 block 的 private histogram，最后再由每个 block 将 private histogram 合并到 global histogram。这样可以把大量 global atomic 转换为较快的 shared memory atomic，并显著减少 global atomic 的次数，从而提升性能。

---

## 6.1 1D convolution：halo 加载进 shared memory

~~~cpp
__global__
void convolution_1D_tiled_kernel(float *N, float *P, int Width)
{
    __shared__ float tile[TILE_SIZE + MASK_WIDTH - 1];

    int radius = MASK_WIDTH / 2;
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    int start = i - radius;

    if (start >= 0 && start < Width)
        tile[threadIdx.x] = N[start];
    else
        tile[threadIdx.x] = 0.0f;

    if (threadIdx.x < MASK_WIDTH - 1) {
        int halo_index = start + TILE_SIZE;

        if (halo_index < Width)
            tile[threadIdx.x + TILE_SIZE] = N[halo_index];
        else
            tile[threadIdx.x + TILE_SIZE] = 0.0f;
    }

    __syncthreads();

    if (i < Width) {
        float Pvalue = 0.0f;

        for (int j = 0; j < MASK_WIDTH; j++) {
            Pvalue += tile[threadIdx.x + j] * Mc[j];
        }

        P[i] = Pvalue;
    }
}
~~~

## 6.3 CUDA reduction kernel

~~~cpp
__global__
void reduce_sum_kernel(float *input, float *partial, int N)
{
    extern __shared__ float sdata[];

    unsigned int t = threadIdx.x;
    unsigned int i = blockIdx.x * blockDim.x * 2 + threadIdx.x;

    float x = 0.0f;

    if (i < N)
        x += input[i];

    if (i + blockDim.x < N)
        x += input[i + blockDim.x];

    sdata[t] = x;

    __syncthreads();

    for (unsigned int stride = blockDim.x / 2;
         stride > 0;
         stride >>= 1)
    {
        if (t < stride) {
            sdata[t] += sdata[t + stride];
        }

        __syncthreads();
    }

    if (t == 0)
        partial[blockIdx.x] = sdata[0];
}
~~~

# 7. 最可能考试题型

## 7.1 解释 output tile 和 input tile 的区别

> Output tile 是一个 thread block 负责计算的输出区域。Input tile 是计算该 output tile 所需要加载的输入区域。对于卷积来说，由于每个输出元素依赖周围邻域，input tile 通常比 output tile 更大，需要额外包含 halo 区域。若 output tile 宽度为 `TILE_WIDTH`，mask 宽度为 `MASK_WIDTH`，则 1D input tile 大小为 `TILE_WIDTH + MASK_WIDTH - 1`，2D input tile 大小为 `(TILE_WIDTH + MASK_WIDTH - 1)^2`。

---

## 7.2 解释为什么 tiled convolution 能减少 global memory access

> 卷积中相邻输出元素会重复使用大量相同输入元素。普通实现中，每个线程都从 global memory 读取自己需要的邻域数据，导致重复访存。Tiled convolution 将一个 block 所需的 input tile 加载到 shared memory 中，之后多个线程从 shared memory 复用这些输入数据，从而把大量 global memory 访问转换为 shared memory 访问，减少 global memory bandwidth 压力。

---

## 7.3 计算 1D tiled convolution reduction

题目：

```text
TILE_WIDTH = 8
MASK_WIDTH = 5
```

答案：

```text
input tile size = 8 + 5 - 1 = 12
原始 global access = 8 × 5 = 40
reduction = 40 / 12 = 3.3
```

---

## 7.4 计算 2D tiled convolution reduction

题目：

```text
output tile = 8 × 8
MASK_WIDTH = 5
```

答案：

```text
input tile = (8 + 5 - 1)^2 = 12^2 = 144
原始访问 = 8^2 × 5^2 = 64 × 25 = 1600
reduction = 1600 / 144 = 11.1
```

---

## 7.5 解释 reduction 为什么需要结合律和交换律

> 并行 reduction 会改变运算顺序，例如把串行的线性累加变成树形合并。如果操作满足结合律和交换律，则改变顺序不会影响结果，因此可以安全地并行化。sum、max、min 等通常适合 reduction。需要注意的是，浮点加法由于舍入误差严格来说不完全满足结合律，因此并行浮点 reduction 结果可能和串行结果有细微差别。

---

## 7.6 解释 atomic 和 reduction 的区别

| 对比 | Reduction | Atomic |
|---|---|---|
| 目标 | 多个值合并成一个值 | 多线程安全更新共享位置 |
| 常见操作 | sum、max、min | atomicAdd、atomicCAS |
| 性能 | 通常更高，可用 shared memory 优化 | 同地址会串行化，冲突大时慢 |
| 适用 | 可分块、可归约的问题 | 更新位置由数据动态决定的问题 |
| 例子 | 数组求和 | histogram bin 更新 |

---

## 7.7 为什么 histogram 不适合直接用 reduction？

> Histogram 中每个输入元素更新哪个 bin 取决于数据值，多个线程可能动态地更新同一个 bin。它不是简单地把所有输入归约成一个值，而是归约到多个 bin 中。直接并行更新会产生 data race，因此基本实现需要 atomicAdd。为了优化性能，可以使用 privatization，为每个 block 建立 private histogram，最后再合并到 global histogram。
