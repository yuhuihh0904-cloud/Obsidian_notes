## 1. 这篇博客主要讲什么？

这篇博客继续围绕 **BlockManager / BlockAllocator** 展开。

上一篇我们知道：

```text
Scheduler 负责决定哪些请求进入本轮推理
        ↓
BlockManager 负责判断有没有 KV Cache block
        ↓
BlockAllocator 负责真正分配物理 block
        ↓
block_tables 维护 seq_id -> physical blocks 的映射
```

这篇博客进一步讲：

> **如果两个请求的 prompt 前缀相同，vLLM 能不能直接复用前一个请求已经计算好的 KV Cache？**

答案是可以，这就是：

```text
Prefix Caching
```

它的目标是：

> **让相同 prefix 的 token 不再重复 prefill 计算，而是直接复用之前缓存下来的 KV Cache block。**

例如两个请求：

```text
请求 1：
你是一个专业金融分析师，请分析腾讯 2025 年 Q1 财报……

请求 2：
你是一个专业金融分析师，请分析腾讯 2025 年 Q2 财报……
```

它们开头这段：

```text
你是一个专业金融分析师，请分析腾讯……
```

很可能是相同的。如果这部分 token 对应的 KV Cache 已经算过，第二个请求就可以复用。

---

## 2. 它和上一篇 BlockManager 的关系

上一篇讲的是普通 block 分配：

```text
来了一个 seq
  ↓
算出它需要几个 block
  ↓
从 free_blocks 里 pop 几个物理 block
  ↓
写入 block_tables
```

这篇讲的是带缓存复用的 block 分配：

```text
来了一个 seq
  ↓
先算每个 block 的 hash
  ↓
看看这个 hash 对应的 KV Cache block 以前有没有算过
  ↓
如果有：直接复用旧 block
  ↓
如果没有：再分配新 block
```

所以区别是：

| 分配器 | 是否支持 Prefix Caching | 核心行为 |
|---|---|---|
| `NaiveBlockAllocator` / `UncachedBlockAllocator` | 不支持 | 来一个请求就正常分配新 block |
| `PrefixCachingBlockAllocator` / `CachedBlockAllocator` | 支持 | 先查缓存，能复用就复用，不能复用才分配 |

文章也明确提到，`SelfAttnBlockSpaceManager` 下面维护 `BlockAllocator` 和 `block_tables`，其中 `BlockAllocator` 又分为普通 allocator 和支持 prefix caching 的 allocator。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

# 3. Prefix Caching 要解决什么问题？

## 3.1 普通 prefill 的浪费

LLM 推理可以分成两阶段：

```text
prefill 阶段：
    一次性处理 prompt，计算 prompt 所有 token 的 KV Cache

decode 阶段：
    每次生成一个新 token，并追加 KV Cache
```

如果很多请求有相同 system prompt，比如：

```text
你是一个乐于助人的 AI 助手……
你是一个专业金融分析师……
你是一个代码审查助手……
```

那么每个请求都重新 prefill 这些相同 token，就很浪费。

---

## 3.2 Prefix Caching 的核心思想

Prefix Caching 的思想是：

```text
相同 prefix
  ↓
对应相同的 KV Cache
  ↓
可以直接复用之前算好的 KV Cache block
```

官方文档说，vLLM 采用的是 **hash-based approach**：对每个 KV cache block 做 hash，hash 的输入包括当前 block 中的 token，以及它前面的 prefix 信息。([docs.vllm.ai](https://docs.vllm.ai/en/stable/design/prefix_caching/))

你可以把它理解成：

```text
不是直接比较一大串 token
而是给每个 block 算一个“身份证”
如果两个 block 的身份证一样，就认为可以复用
```

---

# 4. Prefix Caching 的关键：block hash

## 4.1 为什么要算 hash？

假设：

```text
block_size = 4

请求 A：
[A, B, C, D] [E, F, G, H]

请求 B：
[A, B, C, D] [E, F, G, X]
```

第一个 block：

```text
[A, B, C, D]
```

完全一样，所以可以复用。

第二个 block：

```text
请求 A: [E, F, G, H]
请求 B: [E, F, G, X]
```

不一样，所以不能复用。

那么 vLLM 怎么知道“这个 block 之前算过没有”？

答案就是：

```text
给每个完整 block 计算 content_hash
然后用 hash -> physical block 建立映射
```

---

## 4.2 hash 不能只看当前 block

这是这篇博客最重要的点之一。

vLLM 的 block hash 不是只看当前 block 的 token，而是看：

```text
当前 block token
+
前面所有 prefix 的 hash
+
extra_hash
```

官方文档给出的 components 包括：**Parent hash value**、**Block tokens** 和 **Extra hashes**，其中 extra hashes 可以包含 LoRA ID、多模态输入 hash、cache salt 等信息。([docs.vllm.ai](https://docs.vllm.ai/en/stable/design/prefix_caching/))

也就是说：

```text
当前 block 的 hash = hash(前一个 block 的 hash + 当前 block tokens + extra_hash)
```

对应代码思想大概是：

```python
def hash_block_tokens(
    is_first_block: bool,
    prev_block_hash: Optional[int],
    cur_block_token_ids: List[int],
    extra_hash: Optional[int] = None,
) -> int:
    if is_first_block and prev_block_hash is None:
        prev_block_hash = _none_hash

    return hash((
        is_first_block,
        prev_block_hash,
        *cur_block_token_ids,
        extra_hash,
    ))
```

文章中也展示了类似逻辑：`hash_block_tokens` 会用 `is_first_block`、`prev_block_hash`、当前 block 的 token ids 和 `extra_hash` 来计算 block hash。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

## 4.3 为什么 hash 要包含前面的 prefix？

因为 KV Cache 和 **位置** 有关。

举个例子：

```text
序列 A：
[A, B, C, D] [E, F, G, H]

序列 B：
[X, X, X, X] [E, F, G, H]
```

第二个 block 的 token 都是：

```text
[E, F, G, H]
```

但是它们前面的上下文不同：

```text
序列 A 前面是 [A, B, C, D]
序列 B 前面是 [X, X, X, X]
```

由于 attention 计算和位置编码都依赖前文，所以这两个 `[E, F, G, H]` 的 KV Cache **不能直接复用**。

所以 vLLM 的 hash 必须包含：

```text
当前 block token
+
前面 prefix 的 hash
```

这样才能保证：

```text
只有真正相同 prefix 下的 block 才会命中缓存
```

这也是文章中强调“vLLM 的 hash 是 block-level 的，并且考虑当前 block 及其之前所有 block 的 token 值”的原因。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

# 5. PrefixCachingBlockAllocator 的核心数据结构

## 5.1 `_cached_blocks`

这是最核心的缓存表：

```python
self._cached_blocks: Dict[PrefixHash, BlockId] = {}
```

含义是：

```text
block_hash -> physical_block_id
```

也就是说：

```text
某段 prefix 的 hash
  ↓
对应哪个物理 KV Cache block
```

文章代码中也说明，`_cached_blocks` 是 prefix hash 到 block index 的映射，并且所有有 prefix hash 的 block 都在这个 dict 里，即使它们的 refcount 是 0。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

## 5.2 `_hashless_allocator`

不是所有 block 一开始都有 hash。

因为 vLLM 只缓存 **完整 block**。

如果一个 block 还没满，比如：

```text
block_size = 16
当前 block 只有 7 个 token
```

这个 block 暂时不能作为稳定 prefix cache。

官方文档也明确说：

> **We only cache full blocks.**  
> 也就是只缓存完整 block。([docs.vllm.ai](https://docs.vllm.ai/en/stable/design/prefix_caching/))

所以这些还没有 hash 的 block，会先通过：

```python
_hashless_allocator
```

来管理。

你可以理解成：

```text
_hashless_allocator：
    管理那些暂时还不能作为 prefix cache 的普通 block
```

---

## 5.3 `evictor`

Prefix Caching 有一个很关键的问题：

```text
如果一个请求结束了，它用过的 KV Cache block 要不要马上释放？
```

普通 allocator 可能会直接释放。

但是 prefix caching 不能马上丢掉，因为这个 block 将来可能还能被别的请求复用。

所以 vLLM 会把一些 refcount 变成 0 的 cached block 放进一个 **evictor** 里。

你可以把 evictor 理解成：

```text
缓存冷宫 / 候选淘汰队列
```

它里面的 block 当前没人用：

```text
ref_count = 0
```

但是 KV Cache 内容还在。

如果后面有请求命中这个 block 的 hash，就可以把它重新拿出来复用。

如果 GPU block 不够了，就从 evictor 里按策略淘汰一个 block，真正回收空间。

文章中提到 `PrefixCachingBlockAllocator` 内部会创建 `evictor`，用于在内存压力较高时处理 computed blocks。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

## 5.4 `refcounter`

`refcounter` 记录每个物理 block 当前被多少个 sequence 使用。

```text
ref_count > 0：
    当前有请求正在使用这个 block

ref_count = 0：
    当前没人用，但如果它是 cached block，可能还在 evictor 中等待复用
```

这和上一篇讲的 `ref_count` 是一脉相承的。

---

## 5.5 `BlockPool`

`BlockPool` 是 block 对象池。

文章代码中说，会预先创建一定数量的 block 对象，避免频繁创建 Python 对象。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

你可以理解成：

```text
物理 block 是 GPU / CPU KV Cache 的实际存储单位
BlockPool 是 Python 层用于管理这些 block 对象的池子
```

---

# 6. 三种 Block：NullBlock、NaiveBlock、PrefixCachingBlock

文章第二部分讲了几种 block。

## 6.1 NullBlock

`NullBlock` 是一个占位 block。

它一般和 sliding window 相关。

当某些历史 token 的 KV Cache 被滑动窗口丢弃时，逻辑上可能还需要一个 block 占位，但是它不能被修改。

文章代码中说明，`NullBlock` 用作因为 sliding window 被丢弃的 KV cache blocks 的 placeholder，并且它包装一个普通 block，同时阻止自身被修改。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

可以理解为：

```text
NullBlock：
    逻辑上占个位置
    但不是真的可写 KV Cache block
```

---

## 6.2 NaiveBlock

`NaiveBlock` 是普通 block。

它不支持 prefix caching。

特点是：

```text
只管存 token ids
只管 append token
必要时处理 copy-on-write
不维护 prefix hash
```

---

## 6.3 PrefixCachingBlock

`PrefixCachingBlock` 是这篇文章的重点。

它比普通 block 多了几个概念：

```text
token_ids
prev_block
content_hash
extra_hash
computed
```

其中最关键的是：

```text
prev_block + token_ids -> content_hash
```

它通过 `prev_block` 串起来，让当前 block 的 hash 能包含前面 prefix 的信息。

---

# 7. Prefill 阶段：Prefix Caching 如何分配物理块？

## 7.1 整体流程

prefill 阶段对应的是：

```text
waiting 队列中的新请求
  ↓
准备处理整段 prompt
  ↓
需要为 prompt 的 KV Cache 分配 block
```

上一篇普通 BlockManager 的逻辑是：

```text
can_allocate
  ↓
allocate
  ↓
从 free_blocks 里分配新 block
```

这篇带 Prefix Caching 后，逻辑变成：

```text
can_allocate
  ↓
allocate
  ↓
把 prompt 切成多个 block
  ↓
对每个完整 block 计算 hash
  ↓
用 hash 去 _cached_blocks / evictor 中查找
  ↓
命中：复用旧 block
  ↓
未命中：分配新 block
```

文章中也说，从 waiting 队列调度 seq_group 时，会先调用 `can_allocate(seq_group)` 判断 GPU 是否有足够空间，然后调用 `_allocate(seq_group)` 真正分配物理块；当 allocator 类型不同，实际 allocate 方法也不同。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

## 7.2 主线流程图：prefill 阶段

```text
新请求进入 waiting 队列
        │
        ▼
Scheduler._schedule_prefills()
        │
        ▼
block_manager.can_allocate(seq_group)
        │
        ├── 不够：继续等待 / 放弃
        │
        └── 足够
              │
              ▼
block_manager.allocate(seq_group)
              │
              ▼
把 prompt token 按 block_size 切块
              │
              ▼
对每个完整 block 计算 content_hash
              │
              ▼
PrefixCachingBlockAllocator 查 _cached_blocks
              │
              ├── hash 命中
              │      │
              │      ▼
              │   复用已有 physical block
              │   ref_count + 1
              │
              └── hash 未命中
                     │
                     ▼
                  分配新 physical block
                  写入 _cached_blocks
              │
              ▼
更新 block_tables:
seq_id -> [physical_block_0, physical_block_1, ...]
              │
              ▼
prefill forward
              │
              ▼
真正写入缺失部分的 KV Cache
```

---

## 7.3 为什么只缓存 full block？

因为只有 full block 的 hash 才稳定。

假设：

```text
block_size = 16
当前 block 只有 5 个 token
```

后面 decode 还会继续追加 token。

如果现在就把它作为 prefix cache，后面内容变化后 hash 也会变化，管理会很麻烦。

所以 vLLM 的策略是：

```text
只有 block 满了，才计算稳定 hash
只有有稳定 hash，才进入 prefix cache
```

官方文档明确说 vLLM only caches full blocks。([docs.vllm.ai](https://docs.vllm.ai/en/stable/design/prefix_caching/))

---

# 8. LRUEvictor：缓存 block 如何被保留和淘汰？

## 8.1 普通释放 vs Prefix Cache 释放

普通 block 的释放逻辑大概是：

```text
请求结束
  ↓
ref_count 减 1
  ↓
如果 ref_count = 0
  ↓
直接放回 free_blocks
```

但是 Prefix Caching 不能这么简单。

因为：

```text
ref_count = 0
```

只代表当前没人用，不代表这个 block 没价值。

它可能是：

```text
某个常用 system prompt 的 KV Cache
```

未来很可能再次被用到。

所以 Prefix Caching 的释放逻辑更像：

```text
请求结束
  ↓
block ref_count 变成 0
  ↓
如果它有 content_hash
  ↓
不要马上清空内容
  ↓
放入 evictor
  ↓
等待未来复用，或者在内存不足时被淘汰
```

---

## 8.2 evictor 的作用

evictor 管的是：

```text
当前没有请求引用
但仍然保留 KV Cache 内容的 cached block
```

如果后续请求命中 hash：

```text
新请求 block_hash == evictor 中某个 block_hash
  ↓
把这个 block 从 evictor 中拿出来
  ↓
ref_count + 1
  ↓
重新变成正在使用的 block
```

如果 GPU block 不够：

```text
需要新 block
  ↓
free block 不够
  ↓
从 evictor 里挑一个 ref_count = 0 的 cached block
  ↓
移除 _cached_blocks 映射
  ↓
这个 block 被真正复用为新内容
```

文章代码里 `_maybe_allocate_evicted_block_id` 的逻辑就是：当 hashless allocator 没有可用 block 时，从 evictor 中取出一个 refcount 为 0 的 block，并从 `_cached_blocks` 中删除原来的 hash 映射。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

## 8.3 evictor 的记忆图

```text
请求 A 结束
        │
        ▼
A 的某些 block ref_count = 0
        │
        ▼
这些 block 没有马上清空
        │
        ▼
进入 evictor
        │
        ├── 未来请求命中 hash
        │       │
        │       ▼
        │    重新复用
        │
        └── GPU 空间不足
                │
                ▼
             被淘汰
             删除 _cached_blocks 映射
             block 空间用于新内容
```

# 9. Decode 阶段：Prefix Caching 如何工作？

## 9.1 Decode 阶段的难点

decode 阶段和 prefill 不一样。

prefill 是一次处理一整段 prompt：

```text
prompt token 很多
可以一次切成多个完整 block
```

decode 是每次生成一个 token：

```text
每次只追加 1 个 token
```

所以 decode 阶段的问题是：

```text
新 token 不断追加
什么时候形成一个完整 block？
完整 block 形成后，能不能加入 prefix cache？
如果发现已有相同 hash 的 block，能不能复用？
```

文章中也强调，decode 阶段的 prefix caching 不是频繁地每个 token 都复用，而是当物理块满了以后，再对这个 block 计算一次有意义的 block-level hash。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

## 9.2 decode 阶段整体流程

文章给的入口是：

```python
self._append_slots(seq_group, blocks_to_copy, enable_chunking)
```

然后对 seq_group 中每个 seq 调用：

```python
self.block_manager.append_slots(seq, num_lookahead_slots)
```

对应代码中，`append_slots` 会拿到当前 seq 的 `block_table`，然后调用 `block_table.append_token_ids(...)` 追加新 token，并通过 `clear_copy_on_writes()` 返回 copy-on-write 产生的 block 拷贝任务。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

## 9.3 主线流程图：decode 阶段

```text
seq_group 已经在 running 队列
        │
        ▼
上一轮 forward 生成了新 token
        │
        ▼
Scheduler._schedule_running()
        │
        ▼
block_manager.can_append_slots(seq_group)
        │
        ├── 不够：可能触发 preemption / swap
        │
        └── 足够
              │
              ▼
scheduler._append_slots(seq_group)
              │
              ▼
for seq in seq_group:
    block_manager.append_slots(seq)
              │
              ▼
block_table.append_token_ids(...)
              │
              ├── 当前 block 没满
              │       │
              │       ▼
              │    继续追加 token
              │    暂时不形成稳定 prefix cache
              │
              ├── 当前 block 满了
              │       │
              │       ▼
              │    计算 content_hash
              │       │
              │       ▼
              │    查询 _cached_blocks
              │       │
              │       ├── hash 命中：复用旧 block
              │       └── hash 未命中：当前 block 晋升为 immutable cached block
              │
              └── 发生共享写入
                      │
                      ▼
                   copy-on-write
                   返回 blocks_to_copy
```

# 10. Copy-on-Write 和 Prefix Caching 的关系

上一篇你已经知道：

```text
多个 seq 共享同一个 block
如果其中一个 seq 要写入
就不能直接改共享 block
需要 copy-on-write
```

在 Prefix Caching 中，这个逻辑仍然存在。

尤其在 parallel sampling / beam search 中：

```text
一个 prompt 分裂成多个输出序列
它们共享 prompt block
但是 decode 出来的 token 不一样
```

例如：

```text
prompt block:
[A, B, C, D]

seq0 生成：
E, F, G, H...

seq1 生成：
E, F, G, X...
```

前面的 prompt block 可以共享。

但一旦两个 seq 要往同一个最后 block 写不同 token，就必须：

```text
copy-on-write
```

文章的 decode 示例中也提到，当某个 block 被两个子序列逻辑块引用时，`ref_count = 2`，所以需要拷贝一份，然后两个子序列分别继续 decode。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

---

# 11. `promote_to_immutable_block`：block 满了之后发生什么？

这是 decode 阶段的一个关键函数。

当一个 mutable block 被填满后，它就可以变成 immutable block。

为什么？

因为：

```text
block 已经满了
内容不会再继续 append
content_hash 稳定了
可以作为 prefix cache 被未来请求复用
```

函数名：

```python
promote_to_immutable_block(block)
```

它的核心逻辑是：

```text
当前 block 满了
        │
        ▼
计算 content_hash
        │
        ▼
查看 _cached_blocks 中有没有相同 hash
        │
        ├── 没有
        │      │
        │      ▼
        │   把当前 block 加入 _cached_blocks
        │   当前 block 成为 cached block
        │
        └── 有
               │
               ▼
            说明之前已经有完全相同 prefix 的 block
            释放当前 block 的引用
            当前逻辑 block 指向已有 cached block
            cached block ref_count + 1
```

文章代码中也说明，如果 `block.content_hash` 不在 `_cached_blocks` 中，就把当前 block 作为 cached block；如果已经存在相同 content hash，就复用已有 cached block，并把当前 block 映射替换成已有 block id。([zhuanlan.zhihu.com](https://zhuanlan.zhihu.com/p/1914966014971586393))

# 12. Prefill 和 Decode 中 Prefix Caching 的区别

| 阶段 | 特点 | Prefix Caching 行为 |
|---|---|---|
| **prefill** | 一次处理完整 prompt | prompt 可以切成多个 full block，直接逐块算 hash 并查缓存 |
| **decode** | 每次只生成一个 token | token 逐步追加，只有 block 满了之后才计算稳定 hash |
| **prefill** | 命中缓存收益很大 | 可以跳过一大段 prefix 的重复计算 |
| **decode** | 更偏向渐进积累 | 满 block 后才可能进入 cache 或复用已有 cache |

简单说：

```text
prefill 阶段：
    重点是“新请求进来时能不能复用历史 prompt block”

decode 阶段：
    重点是“生成过程中形成的新完整 block 能不能加入缓存，或者复用已有缓存”
```

# 13. 用一个例子串起来

假设：

```text
block_size = 4
```

现在请求 A 的 prompt 是：

```text
[A, B, C, D, E, F, G, H, I, J]
```

切 block：

```text
block0 = [A, B, C, D]
block1 = [E, F, G, H]
block2 = [I, J]          # 不满
```

## 13.1 请求 A 第一次来

```text
block0 满了
  ↓
计算 hash0 = hash([A,B,C,D])
  ↓
_cached_blocks 没有
  ↓
分配新物理 block 0
  ↓
_cached_blocks[hash0] = block0_id
```

```text
block1 满了
  ↓
计算 hash1 = hash(hash0 + [E,F,G,H])
  ↓
_cached_blocks 没有
  ↓
分配新物理 block 1
  ↓
_cached_blocks[hash1] = block1_id
```

```text
block2 不满
  ↓
暂时不进入 prefix cache
```

此时：

```text
请求 A:
block_tables[A] = [物理 block0, 物理 block1, 物理 block2]
```

---

## 13.2 请求 A 结束

请求 A 结束后：

```text
物理 block0 ref_count -> 0
物理 block1 ref_count -> 0
```

但是它们是 cached block，所以不一定马上清空。

它们会进入 evictor：

```text
evictor = [block0, block1]
_cached_blocks 仍然保留：
hash0 -> block0
hash1 -> block1
```

---

## 13.3 请求 B 来了

请求 B 的 prompt：

```text
[A, B, C, D, E, F, G, H, X, Y]
```

切 block：

```text
block0 = [A, B, C, D]
block1 = [E, F, G, H]
block2 = [X, Y]
```

计算 hash：

```text
hash0 命中
  ↓
复用物理 block0

hash1 命中
  ↓
复用物理 block1

block2 不满
  ↓
分配新的 hashless block
```

于是请求 B 的 block table 变成：

```text
block_tables[B] = [复用 block0, 复用 block1, 新 block2]
```

这意味着：

```text
[A, B, C, D, E, F, G, H]
```

这 8 个 token 的 KV Cache 不需要重新计算。

# 14. 为什么 Prefix Caching 只能复用 prefix，不能随便复用中间片段？

这个点很重要。

假设两个请求：

```text
请求 A：
[A, B, C, D] [E, F, G, H]

请求 B：
[X, Y, Z, W] [E, F, G, H]
```

虽然第二个 block token 都是：

```text
[E, F, G, H]
```

但是不能复用。

原因是：

```text
KV Cache 不只由当前 token 决定
还由它前面的上下文和位置决定
```

所以 vLLM 的 hash 必须带上 parent hash：

```text
block1_hash = hash(block0_hash + block1_tokens)
```

这样只有真正拥有相同前缀的 block 才会命中。

这就是为什么它叫：

```text
Prefix Caching
```

而不是：

```text
任意片段 Caching
```

# 16. 和上一篇 BlockManager 的知识怎么打通？

上一篇你理解的是：

```text
block_tables:
seq_id -> List[PhysicalTokenBlock]
```

这篇要在它上面加一层：

```text
_cached_blocks:
content_hash -> physical_block_id
```

所以完整关系变成：

```text
seq_id
  ↓
block_tables
  ↓
physical_block_id
  ↑
_cached_blocks
  ↑
content_hash
```

换句话说：

```text
block_tables：
    当前请求正在用哪些物理 block

_cached_blocks：
    哪些 hash 对应的物理 block 可以被未来复用
```

它们的关系是：

```text
block_tables 负责“当前请求怎么用 block”
_cached_blocks 负责“未来请求怎么复用 block”
evictor 负责“没人用但还值得保留的 block 怎么淘汰”
```

# 17. 这篇博客的核心结论

## 17.1 Prefix Caching 本质上是 KV Cache 的跨请求复用

普通 KV Cache 只在一个请求内部复用：

```text
decode 第 t 步复用前 t-1 个 token 的 KV Cache
```

Prefix Caching 则进一步做到跨请求复用：

```text
请求 B 复用请求 A 之前算好的 prefix KV Cache
```

---

## 17.2 vLLM 用 block-level hash 实现 Prefix Caching

它不是 token-level cache，而是 block-level cache。

```text
一个 block 满了
  ↓
计算 hash
  ↓
进入缓存表
```

所以它的粒度是：

```text
KV cache block
```

而不是：

```text
单个 token
```

---

## 17.3 hash 必须包含 parent hash

这是为了保证：

```text
只有相同 prefix 下的 block 才能复用
```

否则两个不同上下文里的相同 token block 可能被错误复用。

---

## 17.4 ref_count = 0 不代表马上删除

在 Prefix Caching 中：

```text
ref_count = 0
```

只代表当前没有请求使用。

如果这个 block 是 cached block，它可能会被放入 evictor 中保留，等待未来复用。

---

## 17.5 full block 才能成为稳定缓存

因为 partial block 还会继续 append token，所以 hash 不稳定。

因此：

```text
未满 block：
    先作为 mutable / hashless block 管理

满 block：
    计算 content_hash
    promote_to_immutable_block
    进入 _cached_blocks
```

# 18. 最后给你一个记忆版总结

这篇博客可以这样记：

```text
上一篇 BlockManager：
    管理 KV Cache block 放在哪里

这一篇 Prefix Caching：
    管理哪些 KV Cache block 可以被复用
```

再具体一点：

```text
PrefixCachingBlockAllocator：
    支持 prefix cache 的物理块分配器

_cached_blocks：
    content_hash -> physical_block_id

content_hash：
    parent_hash + 当前 block tokens + extra_hash

evictor：
    保存 ref_count = 0 但未来可能复用的 cached block

promote_to_immutable_block：
    block 满了以后，把它变成可缓存、可复用的 immutable block

copy-on-write：
    多个 seq 共享 block 时，如果有人要写，就复制一份再写
```

一句话总结：

> **Prefix Caching 是 vLLM 在 BlockManager 之上做的一层 KV Cache 复用机制。它通过给完整 block 计算包含 prefix 信息的 hash，把相同 prefix 对应的 KV Cache block 缓存起来。后续请求如果命中相同 hash，就可以直接复用旧 block，减少 prefill 重复计算，提高吞吐。**
